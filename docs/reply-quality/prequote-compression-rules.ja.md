# prequote 圧縮ルール（reply-only）

更新日: 2026-04-11

## 目的

- 見積り相談の返信で、説明しすぎて buyer を疲れさせる崩れを止める
- `短くする` ことではなく、`主質問に必要な情報だけを先に出す` ことを固定する
- 購入率と自然さを落とさず、`読んで返している感じ` を保つ

## 前提

- 対象は `prequote` の返信だけ
- `phase / service / scope / forbidden_moves` の骨格は残す
- 削るのは、主質問より先に出ている補足・例外・内部判断の説明

## 基本原則

### 1. 主質問に先に答える

- 価格質問なら、価格の考え方を先に答える
- 対応可否なら、可否を先に答える
- 納期質問なら、今返せる見立てを先に答える

禁止:
- 主質問の前にサービス説明を長く置く
- 主質問の前に別原因や追加料金の話を広げる

### 2. 1通1主論点

- その返信で解く不安を1つに決める
- 他の説明は、誤解防止に必要な最小限だけ残す

禁止:
- 価格質問に、サービス哲学・境界説明・代替案を全部入れる
- 可否質問に、先回りの例外説明を重ねる

### 3. 理由は1つまで

- 主回答の理由は1つで止める
- 2つ目の理由が必要な時は、次アクションに関係するものだけ残す

禁止:
- 同じ意味の理由を2文続ける
- `そのため` `つまり` `その場合` で同じ話を言い換える

### 4. 次アクションは1つだけ明示する

- 最後の文では、buyer に見える次アクションを1つだけ置く
- `何を返すか` `何を教えてほしいか` を明確にする

禁止:
- 最後に別の不安説明を足す
- `進め方 / ご案内 / ご相談` を同時に並べる

### 5. 内部判断をそのまま説明しない

- `主要1フロー` `同一原因1件` `範囲内 / 範囲外` などの内部判断は、buyer に必要な形に言い換える
- ルールそのものより、buyer が知りたい結論を優先する

禁止:
- 判断ロジックの途中経過をそのまま出す
- `今回は` `まず` `前提になります` で内部の段階判断を見せる

### 6. prequote は薄く、購入後は厚く

- 購入前は、判断に必要なことだけ返す
- 詳細な背景説明や厚い補足は、聞かれた時か購入後に回す

禁止:
- buyer が聞いていない詳細説明を先に入れる
- 購入後向けの丁寧な説明量を prequote に持ち込む

## 残すべき核

- 主質問への直答
- 今回の対象
- 価格 / 範囲 / phase の事実
- 次アクション

## 削る候補

- 同じ意味の説明2文目
- 相手が聞いていない境界説明
- 内部判断の説明
- `前提です` `形になります` `ご案内します` の重ね書き
- `まず` `今回は` `そのため` で増えた補足文

## 圧縮してよいかの判断

次の4つが残るなら削ってよい。

1. 相手の主質問に答えている
2. buyer が次に何をすればよいか分かる
3. 価格や範囲の事実が消えていない
4. phase が崩れていない

## よくある崩れ

### 価格不安

悪い例:
- 価格説明のあとに、サービス説明と代替案を重ねる

良い形:
- 何に対する料金か
- 何が返るか
- 必要なら次に何を教えてほしいか

### handoff の価値質問

悪い例:
- 整理サービスの説明を長くして、安い代替案への答えが後ろへずれる

良い形:
- 費用を抑える方法
- 代替案があるなら何か
- 何を答えてもらえれば案内できるか

### 複数要素の相談

悪い例:
- 先に `3件` `追加料金` `別フロー` を決め打ちする

良い形:
- 同じ流れとして見られるかを先に確認する
- 機械的に件数化しない
