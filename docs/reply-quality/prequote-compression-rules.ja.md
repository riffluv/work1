# prequote 圧縮ルール（reply-only）

更新日: 2026-04-11

## 目的

- 見積り相談の返信で、説明しすぎて buyer を疲れさせる崩れを止める
- `短くする` ことではなく、`主質問に必要な情報だけを先に出す` ことを固定する
- 購入率と自然さを落とさず、`読んで返している感じ` を保つ

## 前提

- 対象は `prequote` の返信だけ
- `phase / service / scope / forbidden_moves` の骨格は残す
- 削るのは、主質問より先に出ている補足・例外・内部判断の説明

## 基本原則

### 1. 主質問に先に答える

- 価格質問なら、価格の考え方を先に答える
- 対応可否なら、可否を先に答える
- 納期質問なら、今返せる見立てを先に答える
- `15,000円で対応できますか` `15,000円で済みますか` のような直球の価格質問には、`進められます / まだ言い切れないので1点確認します / その料金では受けません` の3択へ寄せる
- 極短文の `直せますか` `見てもらえますか` には、最初の2文以内で `対応できます` `対象です` `今の情報ではまだ言い切れないので1点だけ確認します` のどれかを返す
- 主質問に答えた直後の次アクションは、`こちらが次に欲しい情報` ではなく、`buyer が次にやる判断` に合わせる
- buyer が価格固定・復旧時間・原因不明時の料金/返金・保証・対象可否・コードか設定かのような `サービス仕様` を聞いている時は、汎用の不具合確認テンプレートへ流さず、仕様上いま答えられる事実を先に返す
  - `何時間で復旧できますか`: 復旧時間は原因確認前に約束できないことと、一次結果の目安を分けて返す
  - `簡単でも15,000円ですか`: 固定価格であることと、料金に含む範囲を先に返す
  - `原因不明なら返金ですか`: 原因不明のまま正式納品へ進めないことと、返金額はこの時点で断定しないことを分けて返す
  - `保証期間はありますか`: 固定保証期間の有無と、同じ原因が出た場合の扱いを分けて返す

禁止:
- 主質問の前にサービス説明を長く置く
- 主質問の前に別原因や追加料金の話を広げる
- buyer の `対応できますか / 済みますか` に対して、`見積もり判断できます` のような内部寄りの返しへずらす
- buyer の `直せますか` に対して、`対象候補です` のような内部寄りの入口語で返す
- buyer がまだ判断段階にいるターンで、同じ返信の中でいきなりスクショ・ログ・ファイル要求まで進める

### 2. 1通1主論点

- その返信で解く不安を1つに決める
- 他の説明は、誤解防止に必要な最小限だけ残す
- 価格不安が入っていても、buyer が出した技術情報や観測事実を1点は拾う

禁止:
- 価格質問に、サービス哲学・境界説明・代替案を全部入れる
- 可否質問に、先回りの例外説明を重ねる
- 相手が書いた観測事実を1点も拾わず、いきなり提案や判定へ飛ぶ
- 価格不安に引っ張られて、buyer が出した技術情報を完全に捨てる

### 3. 理由は1つまで

- 主回答の理由は1つで止める
- 2つ目の理由が必要な時は、次アクションに関係するものだけ残す

禁止:
- 同じ意味の理由を2文続ける
- `そのため` `つまり` `その場合` で同じ話を言い換える

### 4. 次アクションは1つだけ明示する

- 最後の文では、buyer に見える次アクションを1つだけ置く
- `何を返すか` `何を教えてほしいか` を明確にする

禁止:
- 最後に別の不安説明を足す
- `進め方 / ご案内 / ご相談` を同時に並べる

### 5. 内部判断をそのまま説明しない

- `主要1フロー` `同一原因1件` `範囲内 / 範囲外` などの内部判断は、buyer に必要な形に言い換える
- ルールそのものより、buyer が知りたい結論を優先する
- `対象候補です` が消えても、`見られる可能性があります` のような内部確信度報告へ変形して再発しないようにする
- 直答できる場面では、`可能性があります` より `対応できます` `まず見ます` `プラン変更まわりの不具合として見ていけます` のように、buyer に見える名詞で返す

禁止:
- 判断ロジックの途中経過をそのまま出す
- `今回は` `まず` `前提になります` で内部の段階判断を見せる
- `〜に合います` `〜の進め方になります` のように、scope判定の結論をそのまま本文に出す
- buyer が対応可否を聞いているのに、`見られる可能性があります` `対象になりそうです` のような確信度報告へ逃がす
- `この不具合として` `この内容として` `この症状として` のように、指示語だけで scope を受ける

### 6. prequote は薄く、購入後は厚く

- 購入前は、判断に必要なことだけ返す
- 詳細な背景説明や厚い補足は、聞かれた時か購入後に回す

禁止:
- buyer が聞いていない詳細説明を先に入れる
- 購入後向けの丁寧な説明量を prequote に持ち込む
- buyer が聞いていない内部メリット説明（`着手しやすくなります` `後で楽になります` など）を足す

## 残すべき核

- 主質問への直答
- 今回の対象
- 価格 / 範囲 / phase の事実
- 次アクション

## 削る候補

- 同じ意味の説明2文目
- 相手が聞いていない境界説明
- 内部判断の説明
- `前提です` `形になります` `ご案内します` の重ね書き
- `まず` `今回は` `そのため` で増えた補足文

## 圧縮してよいかの判断

次の4つが残るなら削ってよい。

1. 相手の主質問に答えている
2. buyer が次に何をすればよいか分かる
3. 価格や範囲の事実が消えていない
4. phase が崩れていない

## よくある崩れ

### 価格不安

悪い例:
- 価格説明のあとに、サービス説明と代替案を重ねる

良い形:
- 何に対する料金か
- 何が返るか
- 必要なら次に何を教えてほしいか

### handoff の価値質問

悪い例:
- 整理サービスの説明を長くして、安い代替案への答えが後ろへずれる

良い形:
- 費用を抑える方法
- 代替案があるなら何か
- 何を答えてもらえれば案内できるか

### 複数要素の相談

悪い例:
- 先に `3件` `追加料金` `別フロー` を決め打ちする

良い形:
- 同じ流れとして見られるかを先に確認する
- 機械的に件数化しない
