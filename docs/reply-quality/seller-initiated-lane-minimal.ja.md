# seller-initiated lane 最小メモ

最終更新: 2026-04-20

## 1. 位置づけ
- これは `#R` の代替ではない。
- `#R` は buyer の文章に返す返信レーンとして維持する。
- これは、**こちら起点で buyer に送る文面** を扱う最小レーンの考え方を整理するメモ。
- 既存資産として `#P` があるため、ゼロから別タグを乱立させず、**こちら起点文面の入口として `#P` を使う** 前提で考える。

## 2. 何を扱うか
- 着手報告
- 途中報告
- 追加情報依頼
- 言い忘れ補足
- 実装完了報告
- 納品前確認

## 3. 何を扱わないか
- buyer の主質問に返す返信
- 価格・スコープ・サービス選択の入口判定
- 納品物本文そのもの

## 4. 外部知見から固定してよい原則
### 4.1 短く、早く、定期的に出す
- こちら起点の文面は長文化しない。
- 特に途中報告は、`短く / 早く / 次回時刻つき` を優先する。
- 外向けは 1〜3 文でも成立する。

### 4.2 外向けは `今の状態 / 次の一手 / 必要なら次回時刻`
- buyer 向けに最優先で出すのは次の3つ。
  1. 今どこまで確認できているか
  2. 次に何をするか
  3. 必要なら次にいつ返すか
- 技術的な内部詳細は削り、buyer に必要な現在地を優先する。
- ただし `次回時刻` を強く出すのは、主に `着手報告 / 途中報告 / 納品前確認`。
- `追加情報依頼` や `言い忘れ補足` では、buyer の返答待ちや補足共有が主目的なら、無理に seller 側の次回時刻を入れない。

### 4.3 unknown は隠さない
- 原因や結論がまだ出ていない時は、曖昧にごまかさず `まだ断定していません` を使ってよい。
- ただし unknown だけで終わらせず、`次に何を見るか` を必ず続ける。

### 4.4 impact-directed にする
- buyer 向け文面は、内部実装語より「相手に見える状態」を優先する。
- 例:
  - `Webhook の 401 を調査中` より
  - `決済後の反映が止まる箇所を確認中`

### 4.5 single source of truth を保つ
- buyer 向けの進捗共有は、トークルームを正本にする。
- 別チャネル前提の説明にしない。

### 4.6 update は progress / risk / blocker / ask に絞る
- こちら起点文面は毎回すべてを盛り込まない。
- 次の4要素のうち、必要なものだけを出す。
  - `progress`
  - `newly uncovered risk`
  - `blocker`
  - `clear ask`

## 5. 固定しすぎない方がいい部分
- 2行目で何を「今見ている点」として表現するか
- 完了報告で何を成果として前面に出すか
- 追加情報依頼で、どの情報が本当に blocking か

これらは Codex の実装理解と case 状態に委ねる。  
seller-initiated lane は、事実抽出を厳しく固定しすぎない。

## 6. 最小の骨格
### 着手報告
1. 受領・着手したこと
2. 最初に見る箇所
3. 次回時刻

### 途中報告
1. 今の状態
2. 現在見ている箇所 / まだ断定していないこと
3. 次の一手 + 次回時刻

### 追加情報依頼
1. 何を確認したいか
2. なぜ必要か
3. 値そのものではなく、必要最小限の共有でよいこと

### 言い忘れ補足
1. 先ほどの補足
2. 何を追加で共有するか
3. buyer に必要な判断があるか

### 実装完了報告
1. 何が終わったか
2. どういう状態になったか
3. buyer に確認してほしい点

### 納品前確認
1. 変更点の要約
2. buyer 側で見てほしい確認ポイント
3. 問題なければ正式納品へ進むこと

### 正式納品の順番
- 納品系の seller-initiated 文面は、原則として次の順で出す。
  1. `実装完了報告`
  2. `納品前確認`
  3. `正式納品案内`
- いきなり `正式納品案内` へ飛ばさず、buyer が確認できる1通を先に挟く。
- buyer が確認後に質問してきた場合は、`#P` ではなく `#R` でその質問に直答する。

## 7. 既存 `#P` の扱い
- `#P` は seller-initiated lane の入口として使う。
- デフォルトの `#P` は `途中報告` として扱う。
- 目的を明示する時は `#P 着手報告` `#P 追加情報依頼` `#P 完了報告` `#P 納品前確認` のように用途を足す。
- buyer の文章に返す時は `#R`、こちら起点で送る時は `#P` に分ける。
- 短文テンプレを引く時は `docs/coconala-message-templates-short.ja.md` を参照してよいが、このメモを正本にして seller-initiated の型を判断する。
- 送信用文面の最終自然化は、AGENTS.md に従って `japanese-chat-natural-ja` を通す。

## 8. まず半手動でやる
- まずは `#P` を入口にして、`#P 補足` `#P 完了報告` `#P 納品前確認` のように都度作らせる。
- 細かい validator や renderer は増やしすぎず、Codex の文脈理解を残す。
- buyer が会話を閉じようとしていて、新しい情報や次の予告がない時は無理に返さない。返すなら、次の連絡時刻や次アクションを1文だけ添えて閉じる。

## 9. 今の時点での最小着手順
1. `#P` をこちら起点文面の入口として維持する
2. 最初に強化するのは次の3類型
   - 途中報告
   - 実装完了報告
   - 納品前確認
3. 次点で
   - 着手報告
   - 追加情報依頼
   - 言い忘れ補足
4. `#R` 本体には混ぜない

## 10. 一文で言うと
- こちら起点文面は必要だが、今は `#P` を入口にした半手動の軽い運用から始め、固定しすぎず buyer に見える現在地と次の一手を返す。
