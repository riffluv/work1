# 返信失敗 QA分類（reply-only）

更新日: 2026-04-26

## 目的

- 5件バッチ監査や実案件レビューで出た指摘を、毎回その場の感想で終わらせず、固定ラベルで戻せるようにする
- `どこが変か` だけでなく、`どの層を直すべきか` を即座に判断できるようにする
- 日本語の好みと、返信OSとして危険な崩れを分ける

## 使い方

- 1件の返信につき、まず `主ラベル` を1つ付ける
- 必要なら `副ラベル` を1つまで付ける
- 指摘を書くときは、`症状 / なぜまずいか / どこへ戻すか` の3点で残す
- 同じ崩れが2回以上出たら、単発修正で終わらせず `skill / schema / self-check / NG / gold` のどこへ戻すかを決める

## 最小 drift taxonomy（外部監査・batch監査で優先して使うタグ）

まずは以下の6タグを優先して使う。
この6つは、xhigh / Claude / 人手監査の結果を横断で比較しやすくするための共通タグとする。

- `nonanswer`
  - 主質問に最初に答えず、説明から入っている
- `redundant_ask`
  - 相手が既に書いた情報を聞き直している
- `premature_progress`
  - 未同意段階で、確認・着手・送付依頼など次工程へ進みすぎている
- `hidden_rule`
  - 公開正本にない価格条件・例外条件・成立条件を返信で足している
- `oversell`
  - 聞かれていないサービス説明・営業説明を盛っている
- `ai_tone`
  - 説明書っぽい、抽象語が多い、bot感がある

### phase 補助ラベル

- `phase_answer_gap`
  - 文面は安全に見えるが、今の phase で buyer が次に取れる行動が見えない
  - 特に `quote_sent / delivered / closed` で、可能な操作・不可な操作・代替導線・次アクションのどれかが抜けている
  - 生成 rule ではなく監査レンズとして使う。毎回説明を増やすためではなく、`確認します` で実務導線がぼやける時だけ拾う
- `transaction_model_gap`
  - 文面単体は安全でも、主質問・phase 上できること・確認材料と実作業・料金/返金/無料対応・次アクションが一本の取引構造としてつながっていない
  - 特に `closed` 後で、材料確認とコード修正、無料/通常料金、見積り/新規依頼の導線が断片的に並ぶ時に使う
  - 生成 rule や validator ではなく reviewer 専用の監査レンズ。単なる文体の好みや接続語の違和感には使わない

### discovery 補助ラベル

- `unnamed_discomfort`
  - 既存ラベルにまだ当てはまらないが、実務返信として buyer が詰まりそう・逃げに見えそう・商売上弱そうなどの違和感がある
  - 監査で最大1〜2件だけ挙げる。実務リスクを説明できない好み差は扱わない
  - 生成 rule ではなく発見用レンズとして使う。単発では rule 化せず、観察メモに残して、再発してから名前付け・gold・validator・rule 返却を検討する

## 既存 QA ラベルとの対応

- `QA-01 主質問ズレ` → 主に `nonanswer`
- `QA-02 余計情報差し込み` → 主に `oversell`
- `QA-03 phase漏れ` → 主に `premature_progress` / `phase_answer_gap`
- `QA-04 内部語漏れ` → `ai_tone` または `hidden_rule`
- `QA-05 意味接続破綻` → 文脈次第で `nonanswer` / `ai_tone`
- `QA-06 テンプレ臭` → 主に `ai_tone`
- `QA-07 温度感ズレ` → 補助観点。必要なら `ai_tone` の注記で扱う
- `未命名の違和感` → `unnamed_discomfort`。ただし、実務リスクを説明できる時だけ観察メモとして扱う

運用上は、
- まず 6タグで大分類する
- 必要なら既存 QA ラベルで細分類する
の順にする。

## 記録フォーマット

```text
case: CASE 03
main_label: QA-03 phase漏れ
sub_label: QA-05 意味接続破綻
symptom: 購入前なのに「確認できます」と書いており、確認済み/確認開始に読める
impact: フェーズ誤認を起こし、buyer の期待値が上がる
return_to: prequote policy / reply skill / self-check
```

## ラベル一覧

### QA-01 主質問ズレ

定義:
- 相手の主質問に最初に答えず、別論点から入っている

典型例:
- `お願いできますか` に対して、先に価格や別原因リスクの説明を始める
- `税込みですか` に対して、制度説明だけで終わる

見分け方:
- 最初の answer-bearing section が `primary_question_id` に答えていない

主な戻し先:
- `intake-router`
- `reply_contract`
- `self-check`

### QA-02 余計情報差し込み

定義:
- 内容自体は正しいが、相手がまだ聞いていない情報を先頭へ差し込んでいる

典型例:
- 価格を聞かれていないのに、基本料金を先頭で出す
- 別原因・追加料金の可能性を、主質問より先に説明する

見分け方:
- `聞かれたことに返せていない` というより、`聞かれていないことを前に出している`

主な戻し先:
- `forbidden_moves`
- `reply skill`
- `schema`

### QA-03 phase漏れ

定義:
- 現在の state に合わない時制・動詞・約束を出している

典型例:
- `prequote` なのに `確認を進めます` `コードを見ます` と書く
- `purchased` なのに `対応できるか判断します` と見積もり相談へ戻る

見分け方:
- その文を読んだ buyer が、今どの段階か誤認しうる

主な戻し先:
- `prequote commitment policy`
- `state guidance`
- `self-check`

### QA-04 内部語漏れ

定義:
- 内部の評価軸・運用都合・分類語が、そのまま外向け文面へ漏れている

典型例:
- `今回は作業量ではなく範囲で見ています`
- `対象候補です` を説明なしで使う
- `切り分けたいです` が buyer への返答より内部処理の都合に見える

見分け方:
- 社内メモなら自然だが、buyer 向けだと急に事務的・説明不足に見える

主な戻し先:
- `japanese naturalizer`
- `NG表現`
- `gold replies`

### QA-05 意味接続破綻

定義:
- 別軸の情報を同じ文でつなぎ、因果があるように読ませてしまう

典型例:
- `急ぎの状況とのこと、この内容なら15,000円で対応できます`
- `お問い合わせが増えている状況とのこと、この内容なら不具合1件として対応できます`

見分け方:
- 前半と後半の意味関係が弱いのに、読点で一文接続されている

主な戻し先:
- `NG表現`
- `self-check`
- `naturalizer`

### QA-06 テンプレ臭

定義:
- 意味は通るが、同じ型・同じ語尾・同じ質問数が続き、自動送信感が出ている

典型例:
- 毎回 `この2点だけお願いします`
- `ご連絡します` `進めます` `対応可能です` が連続する
- 冒頭が毎回ほぼ同じ

見分け方:
- 単体では問題が薄いが、5件並べると機械感が強い

主な戻し先:
- `gold replies`
- `naturalizer`
- `review process`

### QA-07 温度感ズレ

定義:
- 過剰共感か、逆に冷たさが出ており、その場面の実務温度と合っていない

典型例:
- 強い不安に対して、完全に無受領で手続きだけ返す
- 毎回重い共感を入れて、要件が薄くなる

見分け方:
- `言っていることは合っているのに、受け手の体感だけがズレる`

主な戻し先:
- `temperature plan`
- `gold replies`
- `review rubric`

## 境界の見方

- `QA-01 主質問ズレ` と `QA-02 余計情報差し込み` は近い
- 違いは、主質問へ答えていないのが主問題なら `QA-01`
- 主質問へ一応答えているが、余計情報を前に出しすぎているなら `QA-02`

- `QA-04 内部語漏れ` と `QA-06 テンプレ臭` も近い
- 内部処理語が漏れているなら `QA-04`
- 内部語ではないが、型が硬直しているなら `QA-06`

- `QA-05 意味接続破綻` は、日本語の自然さだけでなく誤読リスクがある
- 単なる好みではなく、優先度高めで潰す

## 修正先の原則

- router の誤り:
  - 主質問や explicit question の抽出ミス
- contract / schema の誤り:
  - 何を先に答えるか、何を禁止するかの指定不足
- skill の誤り:
  - phase や service ルールの反映不足
- naturalizer の誤り:
  - 日本語としての不自然さ、内部語感、テンプレ臭
- gold / NG の誤り:
  - 古い良い例が残っている、再発パターンが未登録

## この分類で固定したいこと

- 監査コメントは、まずこの7分類へ落としてから考える
- `なんか変` のまま終わらせない
- 好みの違いだけでは恒久反映しない
- 恒久反映するのは、再発性があり、戻し先が明確なものだけにする
