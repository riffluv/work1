# self-check L1 最小 Universal Check

## 目的
- `/home/hr-hm/Project/work/docs/coconala-reply-self-check.ja.md` から、サービス固有知識を必要としない `L1: Universal` だけを抜いた最小チェックを作る
- `service-pack` がなくても回せる、runtime の最低限の品質 gate にする
- 今後、他サービスへ返信OSを移植する時の共通土台にする

## このチェックが見ないもの
- 価格の正しさ
- スコープ境界
- hard no の具体中身
- bugfix / handoff の route 固有ルール
- 納品物や公開状態

上記は `L3` なので、service-pack 側で持つ。

## このチェックが見るもの
- 主質問に答えているか
- 会話の順番が自然か
- buyer の既出情報を無駄に聞き直していないか
- 断る時でも buyer が止まらないか
- multi-turn で約束や保留が崩れていないか
- 日本語として再発癖が出ていないか

## 最小 18 項目

### 1. 主質問に先に答えているか
- 最初の answer-bearing section で、buyer の主質問に答えているか
- 別論点や制度説明から始めていないか

### 2. buyer の判断段階を飛ばしていないか
- buyer がまだ価格や可否を判断する段階なのに、こちらの情報回収フェーズへ前のめりに進めていないか
- buyer が希望した進め方を断る時に、同意なしで次の ask へ直行していないか

### 3. 既出情報を聞き直していないか
- buyer がすでに出した観測事実や技術情報を、そのままもう一度聞いていないか
- 必要なら差分だけを聞けているか

### 4. buyer の語彙や事実を勝手に言い換えていないか
- buyer が使った名詞や現象を、こちらの推測で別の具体語に差し替えていないか
- 相手の文章そのものを評価する入り方になっていないか

### 5. 次アクションが1つだけ見えるか
- 返信後に buyer が「で、次に何をすればいいの？」で止まらないか
- ask や next action が複数に割れていないか

### 6. 断る時でも代替か行き先が見えるか
- 不可だけで終わらず、代替の進め方か次の動きが見えるか
- boundary やお断り寄りの返信で窓口を閉じすぎていないか

### 7. 不安や怒りへの受けが slot-fill になっていないか
- buyer が使っていない感情語を足していないか
- 深すぎる共感や長い受け止めで本題を遅らせていないか
- 受け止めの直後に `ですが` で打ち消していないか

### 8. 説明が防御や言い訳に寄っていないか
- 時間・原因・範囲の保留で、防御的な理由説明を重ねていないか
- 先にこちらが返せることや次の判断を置けているか

### 9. multi-turn で promise を回収しているか
- `promised_next_action` を次のターンで履行しているか
- 前ターンの保留をそのまま引き延ばしていないか

### 10. unresolved thread を宙に浮かせていないか
- secondary symptom や補足論点の行き先を明示しないまま close / delivery へ進んでいないか
- `後で見る` のまま終わっていないか

### 11. 再発癖の日本語になっていないか
- internal 語
- 空語
- ban 語
- 同じ結論の繰り返し
- 相手文のオウム返し

### 12. buyer の語を不必要に固く・狭くしていないか
- buyer が使った語を、必要なく一段硬い語へ変換していないか
- buyer が使った症状語を、こちらの都合で別の語へずらしていないか
- 技術語だけ公式名へ置き換えて、buyer の呼び方を消していないか

### 13. 直答をメタ説明や否定先置きで弱めていないか
- `先にお答えすると` `まず結論だけ言うと` のようなメタ説明を挟んでいないか
- `どうなりますか` `どう進みますか` への返答が、否定から始まっていないか
- Yes を返したあとに、条件・例外・否定を1文へ重ねすぎていないか

### 14. 質問設計が軽く具体的か
- 長い番号案内や選択メニューに逃げず、1問で済む確認へ圧縮できているか
- `情報` `もの` のような広い依頼ではなく、相手がそのまま出せる具体物で聞けているか
- 1点で足りる確認を、2点以上に膨らませていないか

### 15. 自分都合や願望表現が漏れていないか
- `確認したいです` `進めたいです` `見たいです` のような提供者願望が残っていないか
- `返しやすいです` `判断しやすいです` のような内部事情で結んでいないか
- `まず確認します。` 単独で始めて、購入後会話のような距離感になっていないか

### 16. 開始と終了が機械テンプレに寄りすぎていないか
- 冒頭の受領が `もちろんです。` や `はい、` に寄りすぎていないか
- 同一バッチで書き出しや closing が固定しすぎていないか
- ASK ブロックが `1つだけ教えてください` の同型に寄りすぎていないか

### 17. 相手の文章自体を評価していないか
- `この文面だけだと` `今の書き方だと` のように、相手の説明不足を評価する入り方になっていないか
- 相手が書いた内容と矛盾する質問へ戻っていないか
- 相手の質問を引用して、直前に答えた結論を言い直していないか

### 18. 返信全体が1本の線でつながっているか
- 途中で論点や温度感が不自然に飛んでいないか
- 1通としても、multi-turn としても、buyer が前に進んでいるか

## send / revise / escalate

### send
- 12項目で major fail なし

### revise
- 文面修正で直せる軽微なズレがある

### escalate-to-system
- 同型が再発した
- Writer / Reviewer / state のどこに戻すか切れる

## major fail
- 主質問未回答
- buyer の判断段階を飛ばした進行
- 既出情報の聞き直し
- promise 未回収
- unresolved thread の放置

## minor fail
- slot-fill
- opening / closing の単調さ
- 技術語の粒度ズレ
- 受け止めの薄さ
- オウム返し
- buyer 語彙の不要な言い換え
- 質問の膨らみ
- 提供者願望の漏れ

## 使い方
- `service-pack` を読む前に、この 12 項目だけでまず粗い品質を見る
- その後、必要なら `L2` と `L3` の validator を重ねる
- 新サービス導入時の最初の返信監査は、まずこの L1 だけで回してよい

## 位置づけ
- これは `self-check` の置き換えではない
- `self-check` のうち universal な芯だけを抜いた最小版
- 最終形は
  - `L1 minimal universal`
  - `L2 domain checks`
  - `L3 asset validators`
  の3段構成を想定する
