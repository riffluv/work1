# self-check residual から L2 へ戻す候補

## 目的
- `/home/hr-hm/Project/work/docs/coconala-reply-self-check.ja.md` の `compat-only residual` に残る項目のうち、実際には `L2: Domain` として扱えるものを抽出する
- monolith の residual を減らし、phase 運用や受託返信の型を `L2` 側へ寄せる

## 判断基準
- service 固有知識なしで成立する
- 受託返信としての進め方や phase 運用に関わる
- buyer の判断段階や温度への寄せ方として再利用できる

## 先に L2 へ戻せる候補

### 1. 書き出し / closing / route 運用
- route に合った書き出しになっているか
- 見積もり相談なのに、返信構造を毎回同じ5段で機械的に組んでいないか
- `問題なければ` の closing が固定しすぎていないか
- `切り分けたうえで対応します` が機械的な標準文になっていないか
- `2時間以内` を毎回固定フッターとして入れていないか
- 時刻コミットの言い回しが同型に寄りすぎていないか

### 2. 進捗 / urgency / 温度
- `急ぎ` `至急` を本文で直接受けているか
- 急ぎ・不安・負担感が強い場面で、価格を安心材料として先出ししすぎていないか
- 不安や遠慮を1行差し出した buyer に、必要な時だけ短い受け止めを置けているか
- 追加報告への返答で、持ち込まれた事象を主語にできているか

### 3. 断り方 / service mismatch
- `service_mismatch` の断りが `対象外です` だけで冷たく閉じていないか
- 完全に対象外でも、必要なら再相談や別の行き先を1文だけ残せているか
- 外部共有を止める時に、代替の共有方法まで同じ返信で示せているか

### 4. prequote / ask の運用
- 高確度案件で、習慣的な情報回収に寄っていないか
- 高確度・単一質問なのに、適合説明を後から足して長くしていないか
- `まずは次の3点だけお願いします` のような進行中テンプレを prequote に持ち込んでいないか
- `もの` `資料` `ファイル` のような広い ask に寄っていないか

### 5. ことばの強さと buyer の温度
- `可能性が高いです` のような確信度表現が会話として強すぎないか
- 対応可否で、その場で受けられる内容なら `対応できます` `まず見ます` へ寄せられているか
- `前回に続いてのご相談` のような分類ラベルを、自然な受領文へ落とせているか
- `流用` のような内部寄り語を buyer が読める語へ言い換えられているか

### 6. buyer 主語の維持
- `この不具合として` のような指示語だけの受けに戻っていないか
- `ここは優先して見ます` のような曖昧な指示語を、そのまま残していないか
- `決済とは関係ない` と buyer が切っている時に、こちらが勝手に広い scope へ戻していないか

## まだ residual に残すもの
- `.envやAPIキーの値は不要です` の機械適用
- `.env` 注意文の結語
- スクショ vs テキストの扱い
- prequote ask の bugfix 固有最小性
- `non_engineer` 向けの技術語粒度
- `15,000円で進められます` の同型
- `Stripe不使用ケース` の扱い

理由:
- これらは `L3` や asset 参照と混ざりやすく、今は domain だけで切ると雑になるため
