## 目的
- `/home/hr-hm/Project/work/docs/coconala-reply-self-check.ja.md` の `compat-only residual` に残っている項目のうち、実際には `L1: Universal` として扱えるものを先に抽出する
- monolith からいきなり削らず、まず「どれが universal か」を固定する

## 判断基準
- service-pack がなくても判定できる
- bugfix / handoff どちらでも再発しうる
- 会話品質や自然さの問題として説明できる
- runtime の最小 gate に置いた方が再利用しやすい

## 先に L1 へ戻せる候補

### 1. 質問設計
- 長い番号案内や選択メニューに逃げず、1問で済む確認へ圧縮できているか
- 質問数は 0〜3点の範囲に収まっているか。3点を機械的な標準にしていないか
- 質問リストの直後が `いただければ` だけで終わっていないか。動詞まで書けているか
- 相手が書いた内容と矛盾する質問になっていないか
- 診断上必要な質問と、テンプレ都合の先回りを取り違えていないか
- `情報` だけで依頼せず、相手がそのまま出せる具体物で聞けているか
- 2点テンプレを優先して、1点で足りる確認を増やしていないか
- 根本的な分岐が1つだけなのに、2問に増やしていないか

### 2. buyer 語彙 / 事実保持
- buyer が使った名詞を、こちらの推測で別の具体語へ差し替えていないか
- 相手が使った語を、必要なく一段硬い語へ変換していないか
- 相手が使った動詞や症状語を、こちらの都合で別の語へずらしていないか
- 技術語だけ公式名へ置き換えて、相手が使った呼び方を消していないか
- `崩れる` を便利語として使って、相手の具体的な症状語を潰していないか
- `この不具合として` のように、指示語だけで対象を受けていないか

### 3. 直答と文構造
- `どうなりますか` `どう進みますか` に、否定から答え始めていないか
- 納期・保証・見通しで、`確定できません` `約束できません` を先頭に置いていないか
- 専門家の判断を返す場面で、推量に逃げず判断そのものを返しているか
- `先にお答えすると` `まず結論だけ言うと` のようなメタ説明を挟んでいないか
- 相手の質問を引用して、前文の結論をもう一度言い直していないか
- Yes を返したあとに条件・例外・否定を1文へ重ねすぎていないか

### 4. slot-fill / 書き出し / closing の単調さ
- 冒頭の肯定や受領が `もちろんです。` に寄りすぎていないか
- 冒頭の直答が `はい、` に寄りすぎていないか
- 同一バッチで書き出し・末尾が固定しすぎていないか
- ASK ブロックが `1つだけ教えてください` の同型に寄りすぎていないか
- `問題なければ` の closing が固定しすぎていないか
- `切り分けたうえで対応します` が標準文になっていないか

### 5. 内部語 / 評価語 / bot語
- `共有` `流れ` `進め方` などの PM 語彙を buyer 向けに落とせる場面でそのまま残していないか
- `公開サービス` `この中で` `範囲で入る` のような内部スコープ語を前に出していないか
- `前回に続いてのご相談` のような分類ラベルが受領文に混ざっていないか
- `流用` のような内部寄り業務語が残っていないか
- `判断しやすいです` `見やすいです` のように内部事情で結んでいないか

### 6. 自分都合 / 願望表現
- `返しやすいです` `確認したいです` のような提供者視点の語が残っていないか
- `見たいです` `返したいです` のような願望変形が残っていないか
- `まず確認します。` 単独で始めて購入後会話のような距離感になっていないか

## まだ residual に残すもの
- `2時間以内` など時刻コミットの揺れ
- `.env` 注意文の後の結語
- route に合った書き出し
- purchased / closed での Drive / GitHub / URL 共有差し戻し
- bugfix prequote ask の最小性

理由:
- これらは `L2` や `L3` と混ざりやすく、先に universal 化すると境界が崩れやすいため

## 抽出優先順位
1. buyer 語彙 / 事実保持
2. 直答と文構造
3. 質問設計
4. 自分都合 / 願望表現
5. slot-fill / 単調さ
6. 内部語 / 評価語
