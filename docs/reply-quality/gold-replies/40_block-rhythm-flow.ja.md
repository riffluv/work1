# Gold Reply 40 — block rhythm flow

## 使いどころ

- 内容は正しいが、処理文・安全説明・条件文が同じリズムで並び、契約説明の塊に見える時。
- `conversation_flow_naturalness` の中でも、段落の見やすさと会話の流れの両立を確認したい時。
- closed 後、返金、無料対応、成功保証などの高リスク場面で、自然化しすぎると誤読が増えそうな時。

## 出力方針

- 句点数や段落数だけで判断しない。
- つなげるのは、同じ役割・同じ約束レベルの文だけにする。
- 滑らかにすると保証・返金・無料対応・closed 後作業の promise が強くなる文は、分けたままにする。
- 判断は `fix_recommended` / `acceptable_as_is` / `unsafe_to_smooth` に分ける。
- 自然化で price / scope / phase / secret / public-private / payment route を変えない。

## 1. purchased 受領後: 主質問に先に答えてから確認へ入る

```text
ログとスクショありがとうございます。届いています。

届いている内容でこちらから確認に入ります。いま追加で準備いただくものはありません。
必要なものが出た場合は、その時点でこちらからお伝えします。

本日18:00までに、現時点で見えている点と次に見る箇所をお伝えします。
```

### 守ること

- `届いています` を言ったあと、受領確認を繰り返さない。
- buyer の「追加で用意するものはあるか」へ先に答える。
- 追加依頼が必要な場合の導線は残す。
- 時刻コミットを入れる場合は、送信時刻と矛盾しないようにする。

## 2. closed 後: 送ってよい -> 届いた範囲で見る -> 実作業は別相談

```text
ご連絡ありがとうございます。
またStripeエラーが出ている件で、前回の修正との関係を確認したいということですね。

まずはこのメッセージ上で、今出ているエラー内容やログ、スクショを送ってください。
届いた範囲で、前回の修正と関係がありそうかを確認します。

前回トークルームは閉じているため、ここで修正作業や修正済みファイルの返却までは進めません。
実作業が必要そうな場合は、対応方法と費用を先にご相談します。
```

### 守ること

- `このメッセージ上で送ってよいか` へ先に答える。
- closed 後でも、概要確認と実作業を分ける。
- 無料対応・返金・旧トークルーム継続を約束しない。
- 読みやすさのために、関係確認と修正作業開始を一息でつなげない。

## 3. closed 後の返金・無料圧: 判断に必要な順序を前に出す

```text
ご連絡ありがとうございます。
またStripeエラーが出ているとのこと、ご不便な状況かと思います。

返金か新規相談かは、前回の修正との関係を見てからの判断になります。
まずはこのメッセージ上で、今出ているエラー内容やログ、スクショを送ってください。

届いた範囲で前回の修正と関係がありそうかを確認します。
前回トークルームは閉じているため、この場で修正作業や修正済みファイルの返却までは進めません。
```

### 守ること

- `返金してほしい` `無料で直してほしい` をそのまま復唱しない。
- 返金・無料・新規相談の論点自体は消さない。
- `断定できません` の連続より、何を見れば判断できるかを先に置く。

## reviewer 観点

- 返信が buyer の質問量・温度・リスク量に対して重すぎないか。
- safety boundary が外向け本文にチェックリストのように並びすぎていないか。
- 同じ役割・同じ約束レベルの文だけ自然につながっているか。
- 滑らかにすると誤読が増える箇所を、無理に自然化していないか。
- `fix_recommended` / `acceptable_as_is` / `unsafe_to_smooth` を分けて判定できているか。

## 避けること

- 句点数や段落数で機械判定する。
- 高リスク境界を短文化のために削る。
- 成功保証不可と成果物返却を条件なしでつなげる。
- closed 後の関係確認と実作業開始を同じ流れに見せる。
- gold 文面をテンプレとして丸ごと使う。
