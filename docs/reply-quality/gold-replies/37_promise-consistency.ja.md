# Gold Reply 37 — promise consistency

## 使いどころ

- 1文ずつは正しいが、前後をつなげて読むと未約束の成果物・作業・料金・納期を約束しているように見える時
- 成功保証、原因未確定、購入前着手、closed 後作業、secret 値、複数症状、追加料金などで、留保・不可・条件付き回答と後段の導線を分けたい時
- `promise_consistency` を hard fail にせず、最小差分で buyer の誤解を減らしたい時

## 出力方針

- 固定してよい約束は弱めない。価格、対応可能な入口、購入後に原因確認へ入れることは明確に言う。
- 未確認の成果だけ条件を付ける。成功保証、修正完了、原因断定、追加作業、無料対応は無条件に見せない。
- `修正済みファイル` 自体は ban しない。未確認の状態で、直前の留保を上書きする成果物表現にしない。
- `できるか分かりません` に寄せすぎず、約束レベルを分ける。

## 1. 成功保証を聞かれた時: 原因確認と修正済みファイル返却の promise level を分ける

```text
ご相談ありがとうございます。

確実に直ることまでは、コードとエラー内容を見る前にはお約束できません。

ただ、Stripe決済は成功しているのに注文が作られない症状であれば、15,000円の不具合修正として、まず原因確認から進められます。修正できる箇所が特定できた場合は、修正済みファイルをお返しします。

ご購入後にログと関連コードを確認し、必要な情報を確認できた時点で、一次結果は48時間以内を目安にお返しします。
急ぎの状況も踏まえて進めるので、この内容で問題なければそのままご購入ください。
```

### 守ること

- `確実に直ることは約束できません` の直後に、`修正済みファイルの返却まで進められます` と無条件に並べない。
- `原因確認` は購入後に始められる作業、`修正済みファイル返却` は修正箇所が特定できた場合の成果物として分ける。
- 成功保証を避けるために、15,000円・不具合修正として見られる入口まで弱めない。

## 2. 複数症状: 同じ原因なら含む、別原因なら止めて相談を導線まで残す

```text
ご相談ありがとうございます。

注文が作られない件と領収書メールが届かない件は、同じ原因かどうかを先に確認します。

まずは「注文が作られない」症状を起点に確認します。領収書メールの件も同じ決済後処理につながっていれば、同じ範囲で見ます。

別原因と分かった場合は、その時点で追加対応にするかどうかと費用を先にご相談します。
この前提でよければ、その内容でご購入ください。
```

### 守ること

- `同じ原因なら見ます` と言ったあと、最後の購入導線で片方の症状を宙に浮かせない。
- buyer が優先順位を明示していない時に、`一番困っている` と断定しない。
- 複数症状を無条件に15,000円で全部見ると読ませない。

## 3. 購入前原因確認: 断ったあとに確認開始へ戻らない

```text
ご連絡ありがとうございます。

購入前にGitHubへ招待いただいて、原因だけ先に確認する形では進めていません。
原因確認や作業は、お支払い完了後に始めます。

見積り提案の内容で問題なければ、お支払い完了後にトークルームで該当コードやログを送ってください。
```

### 守ること

- `購入前に原因確認しません` と言ったあと、`ログを見て原因を確認します` と書かない。
- 購入後に送ってほしい材料は、`お支払い完了後に` を付けて phase を固定する。

## reviewer 観点

- 前段で `できない / 未確定 / 条件付き / 購入後 / 値不要 / 先に相談` と言ったことを、後段が実質的に上書きしていないか。
- 成功保証・原因断定・修正済みファイル返却・購入前作業・closed 後作業・secret 値・追加料金・本番反映の promise level が分かれているか。
- hard fail にするのは、実際に保証断定、phase drift、secret 値要求、payment route 事故、closed 後作業 promise などが出た場合だけか。

## 避けること

- `対応できます` `進められます` `修正済みファイル` を blanket NG にする。
- 成功保証回避のために、固定価格や対応可能な入口まで曖昧にする。
- promise の問題を、単なる日本語自然化や句点の好みに回収する。
