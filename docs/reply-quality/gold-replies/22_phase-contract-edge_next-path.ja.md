# Gold Reply 22 — phase edge で buyer を迷子にしない

## 使いどころ

- `quote_sent` で、入金前に先に見てほしいと言われた時
- `delivered` で、承諾後や差し戻しについて聞かれた時
- `closed` で、旧トークルーム継続やおひねり追加作業を聞かれた時
- 文面は安全でも、buyer が「で、次に何をすればいいの？」で止まりそうな時

## 1. quote_sent / 入金前作業開始

```text
ご連絡ありがとうございます。
内容に問題ないとのこと、ありがとうございます。

申し訳ありませんが、支払い前の段階では、原因確認などの実作業は開始していません。
ご購入・入金完了後にトークルームが開いてから、正式に確認を始めます。

お支払いの準備ができましたら、そのままご購入ください。
```

## 2. delivered / 承諾後に気づいた場合

```text
ご連絡ありがとうございます。
大丈夫そうとのこと、確認しました。

承諾後はクローズ手続きに進むため、その後はこのトークルーム内でのやり取りができなくなります。
気になる点が少しでもある場合は、承諾前にこのトークルーム内でお伝えください。

同じ箇所に不具合が残っている可能性があれば、先に確認します。
承諾後に気づいた場合は、まずメッセージ上で前回対応とのつながりを確認します。実作業が必要な場合は、見積り提案または新規依頼としてご案内します。
```

## 3. delivered / 差し戻し・承諾判断

```text
ご連絡ありがとうございます。

本番で一部確認できていない状態なら、無理に承諾しなくて大丈夫です。
気になる点がある場合は、承諾前にこのトークルーム内でお伝えください。

画面上で差し戻しを選べる状態であれば、差し戻しで送っていただいて大丈夫です。
まず確認できていない箇所を見て、承諾いただいて問題ないかをお返しします。
```

## 4. closed / おひねり追加修正

```text
ご連絡ありがとうございます。

前回のトークルームは閉じているため、おひねりでそのまま追加修正を進めることはできません。

追加で直したい内容を、このメッセージで送ってください。
まず内容を確認し、実作業が必要な場合は新しく見積り提案としてご案内します。
```

## この型で守っていること

- `quote_sent` を購入・入金前として扱い、実作業開始へ滑らせない
- `delivered` を `closed` と混同せず、承諾前の確認導線を出す
- 承諾後・closed 後は、旧トークルーム内で続ける前提にしない
- `closed` 後のおひねり・追加購入を導線にしない
- 不可だけで終わらず、buyer が次に取れる行動を1つ見せる

## 使い方

- そのままテンプレート化せず、近い phase edge の gold として見る。
- `phase_answer_gap` が出た時に、不可・代替・次導線の置き方だけを参照する。
- 毎回説明を増やす目的では使わない。
