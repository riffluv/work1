# Gold Reply 30 — normal boundary routing

## 使いどころ

- 普通寄りの見積り相談や購入後相談だが、決済サービス名、実行環境、本番切替、返金操作などの境界が混ざる時
- `この不具合なら15,000円` の通常受付だけでは、scope や取引上の境界が曖昧になる時
- buyer が具体的な技術情報を出しているのに、情報不足テンプレや聞き直しへ戻りそうな時

## 出力方針

- まず buyer の主質問へ答える。
- 非Stripe決済は、決済サービス全体ではなく、Next.js 側の Webhook / API 受信処理として見られる範囲に閉じる。
- 本番だけの timeout やテストから本番切替は、buyer が出した環境差を1文で拾い、最初に見る箇所を具体化する。
- 返金操作は、手順案内と返金実行代行を分ける。カード情報や個人情報は求めない。

## 1. 非Stripe決済 / Webhook受信処理に閉じる

```text
ご相談ありがとうございます。

PayPayオンライン決済のWebhook通知が来ていない件ですね。

PayPay側そのものの仕様調査ではなく、Next.js側のWebhook/API受信処理として確認できる内容であれば見られます。
まずは通知を受けるAPIの場所と、表示されているエラーやログの有無を分かる範囲で送ってください。確認後、15,000円の範囲で進められるかをお返しします。
```

### 守ること

- `PayPay全体を見ます` のように広げない。
- 決済事業者側の仕様調査やアカウント操作を引き受ける文にしない。
- Next.js 側の Webhook / API 受信処理として見られるか、という入口にする。

## 2. 本番のみ API timeout

```text
ご相談ありがとうございます。

本番だけ500で、Vercelのログに FUNCTION_INVOCATION_TIMEOUT が出ている状態ですね。
ローカルとプレビューでは数秒で返ってくるとのことなので、Stripe関連APIが本番環境でtimeoutしている原因を確認する形になります。

この不具合なら15,000円で対応できます。

まずはVercel側の実行時間制限と、Stripe APIの応答時間や呼び出し経路が本番でどう変わっているかを確認します。

この内容で進める場合は、そのままご購入ください。必要情報がそろい次第、一次結果は48時間以内を目安にお返しします。
```

### 守ること

- buyer がすでに書いた `本番だけ` `Vercelログ` `FUNCTION_INVOCATION_TIMEOUT` を聞き直さない。
- `今の情報だけでは判断できません` を重ねない。
- まず見る箇所を、Vercel の実行時間制限と API 呼び出し経路へ絞る。

## 3. テスト環境から本番切替後の不具合

```text
ご相談ありがとうございます。

テスト環境では動いていて、APIキーをliveに変え、Webhookエンドポイントも本番用にした後に動かなくなった状態ですね。
エラーメッセージが出ていないとのことなので、本番キーとWebhook endpointの設定反映、処理が無言で止まっている箇所を確認します。

この不具合なら15,000円で対応できます。
APIキーやWebhook secretの値そのものは不要です。確認に必要な場合も、キー名や設定箇所が分かれば大丈夫です。

この内容で進める場合は、そのままご購入ください。必要情報がそろい次第、一次結果は48時間以内を目安にお返しします。
```

### 守ること

- `APIキーを送ってください` `Webhook secretを送ってください` と言わない。
- `エラーなし` を見落とさず、無言で止まっている箇所として扱う。
- テスト環境と本番環境の差分を確認軸にする。

## 4. 購入後 / 二重引き落としと返金操作

```text
ありがとうございます。
二重引き落としの件、確認しました。

今回の決済不具合と同じ原因で起きている内容であれば、このまま今回の件として確認します。
別の原因と分かった場合は、追加対応が必要かどうかを先にご相談します。

返金処理の操作については、原因の切り分け結果と合わせて、Stripe上で確認できる範囲の手順をお伝えします。
こちらで返金実行を代行するわけではありません。

まず今回の件とつながっているかを確認するので、二重に引き落とされた決済の日時・金額・決済IDなど、分かる範囲の情報を送ってください。カード情報や個人情報は不要です。

本日16:55までに、現時点の確認結果をお返しします。
```

### 守ること

- 二重引き落としを即座に今回範囲内と断定しない。
- Stripe上の返金操作手順の説明と、返金実行代行を分ける。
- カード情報、個人情報、生の秘密情報を求めない。
- `決済ID` は Stripe 上で確認できる ID という意味で使う。

## 避けること

- 非Stripe決済を、決済事業者側の仕様調査まで広げる
- 本番だけの timeout で、既出の環境差を聞き直す
- テスト/本番切替で、APIキーや secret の値を求める
- 返金操作質問に、返金実行まで代行するような文を出す
- 二重引き落としで、カード情報や個人情報を求める

## reviewer 観点

- buyer の質問は、単なる不具合受付ではなく scope 境界や作業境界を含んでいないか。
- 決済サービス名、環境差、APIキー、返金操作などの境界情報を拾えているか。
- `見られる範囲` と `引き受けない範囲` が buyer に分かる言葉で分かれているか。
- request_minimality が守られ、不要な秘密情報や個人情報を求めていないか。
