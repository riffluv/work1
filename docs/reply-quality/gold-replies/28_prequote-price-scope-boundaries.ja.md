# Gold Reply 28 — prequote price / scope boundaries

## 使いどころ

- 普通の `prequote` だが、buyer の不信感、疲弊、値引き希望、ついで修正、機能追加要求が混ざる時
- `この不具合なら15,000円` の旧テンプレへ戻すと、buyer の主質問や温度を拾えない時
- `同じ原因なら見る / 別原因なら相談` と `新機能追加は別` を取り違えそうな時

## 出力方針

- まず buyer の主質問へ答える。
- 価格や範囲を出す前に、buyer が書いた不安・状況・症状を1つだけ拾う。
- 値引き希望には、価格を崩さず、購入前にできる範囲確認を代替として出す。
- `ついでに数行` が新機能追加なら、同一原因/別原因の切り分けに流さず、今回の不具合修正とは分ける。

## 1. 外注不信 / 手動対応限界

```text
ご相談ありがとうございます。

2週間手動対応が続いていて、かなり負担が大きい状況ですね。
有料プランを購入してもアカウントが Free のままになる不具合として、15,000円で対応できます。

Stripeサポートからアプリ側と言われていて、Webhook周りも疑われているとのことなので、まずは決済完了後にアカウントのプラン更新がどこで止まっているかを確認します。

この内容で進める場合は、そのままご購入いただいて大丈夫です。必要情報がそろい次第、一次結果は48時間以内を目安にお返しします。
```

### 守ること

- 前の出品者や外注先を批判しない。
- `大変でしたね` のようなカウンセリング調に広げない。
- 事実ベースで疲弊を受け、すぐ確認対象へ戻す。

## 2. 間欠不具合 / 前回修正後の再発

```text
ご相談ありがとうございます。

決済完了後にDB更新が走らないことがある、という間欠的な不具合ですね。
毎回ではない分、原因の切り分けに工夫が必要になる場合がありますが、この不具合なら15,000円で対応できます。

まずは決済完了からDB更新までの処理で、成功する場合と失敗する場合の違いがどこにあるかを確認します。

この内容で進める場合は、そのままご購入いただいて大丈夫です。必要情報がそろい次第、一次結果は48時間以内を目安にお返しします。
```

### 守ること

- `毎回ではない` を技術的な切り分け難度として拾う。
- 前の出品者への不信は、競合批判に広げない。

## 3. ついで修正 / 同じ原因の決めつけ

```text
ご相談ありがとうございます。

決済反映バグと Apple Pay ボタン非表示の2件ですね。
同じ決済画面まわりでも、同じ原因かどうかは確認しないと判断できません。

同じ原因でつながっていれば15,000円の範囲で見ますが、別原因の場合は両方を同時に直す前提にはなりません。
まずは一番困っている方を1件として確認します。決済反映バグを優先する場合は、この内容でご購入いただいて大丈夫です。
```

### 守ること

- buyer の `原因は一緒だと思う` をそのまま採用しない。
- ただし冷たく切らず、同じ原因なら範囲内に残す余地を示す。

## 4. 値引き / お試し時間課金希望

```text
ご相談ありがとうございます。

お試し料金や時間単位での対応は行っていません。
このサービスは、15,000円で不具合1件の原因確認から修正済みファイルの返却まで進める形です。

相性確認という意味では、購入前にこのまま具体的な症状を送っていただければ、対応範囲に入りそうかを先にお返しします。
その確認だけで費用が発生することはありません。
```

### 守ること

- `5,000円で1時間だけ` に合わせて価格を崩さない。
- 代替は `購入前の対応範囲確認` に留める。
- 購入前にコード調査や具体修正手順までは出さない。

## 5. 購入後 / ついでの新機能追加

```text
ありがとうございます。
`evt_8888` の件、確認しました。

Customer ID をDBに残す件は、機能追加にあたるため、今回の不具合修正の範囲には含まれません。
今回は依頼いただいた不具合1件の修正に集中して進めます。

必要な場合は、今回の不具合修正とは分けて進め方をご相談します。

本日15:12までに、現時点の確認結果をお返しします。
```

### 守ること

- `ついでに数行` を軽く見ない。
- 新機能追加を `同じ原因 / 別原因` の切り分けへ流さない。
- 今回の不具合修正の進捗を止めず、追加要望は分ける。

## 避けること

- `内容ありがとうございます` だけで受けて、buyer の不信感や条件を拾わない
- 値引き希望に対して、通常の不具合受付テンプレートで返す
- `ついでに` `数行` を、同じ原因なら範囲内かもしれない話として扱う
- 追加機能を `別原因` と呼ぶ
- 前の出品者・外注先を否定する

## reviewer 観点

- buyer は不具合そのものではなく、価格・相性・過去の不信・追加要望を聞いていないか。
- `不具合1件` の中に残せる話か、`新機能追加` として分ける話か。
- 価格を崩さず、buyer が次に何を送ればよいかが見えているか。
