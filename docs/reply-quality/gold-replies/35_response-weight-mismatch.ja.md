# Gold Reply 35 — response weight mismatch

## 使いどころ

- 安全境界は守れているが、buyer の文量・温度・質問数に対して返信が重く見える時
- 状態受け、価格、scope、phase、closed 後導線、報告時刻が全部並び、担当者の会話ではなく安全条件の列挙に見える時
- `短くする` ではなく、必要な境界だけを外向けに残して自然な重さへ戻したい時

## 出力方針

- 文字数だけで判断しない。
- closed / 返金 / 保証 / 秘密情報 / 外部共有 / 複数症状 / 追加料金不安では、buyer 文が短くても必要境界を残す。
- 通常 prequote や短い yes/no 質問では、価格・次アクション・必要な1境界だけに圧縮する。
- 状態受けの直後に価格へ飛ばず、必要なら `今回は今の状態を基準に確認します` のような判断基準を1文置く。
- `はい` は全面禁止しない。先頭で yes/no に答える時は自然。文中で方向を出した後の単独 `はい` は避ける。

## 1. 他者修正失敗 + 不信: 状態受けから価格へ飛ばさない

```text
ご相談ありがとうございます。
一度「直った」と言われた後に再発しているので、同じ流れにならないか気になる状況だと思います。

今回は前回の修正結果を前提にせず、今のコードとログを基準に確認します。
Stripe決済後の注文反映が止まる不具合として、15,000円で対応できます。

15,000円内で直し切れないと分かった場合は、追加作業へ進まず、その時点でご説明します。
進める場合は、そのままご購入ください。
```

### 守ること

- 前任者批判をしない。
- 状態受けの直後に価格へ飛ばず、今回どう見るかを1文置く。
- `その点も含めて` だけで毎回つながない。必要なら具体的な判断基準へ置き換える。

## 2. purchased 進捗 + 短い返答希望: 最小構成にする

```text
お待たせしてすみません。
いまは決済後の反映処理がどこで止まっているかを確認しています。
まだ原因は断定していません。
本日06:06までに、分かっている点だけ短くお返しします。
```

### 守ること

- `長い説明じゃなくていい` と言われたら、説明を増やさない。
- 現在地、未確定点、次回時刻だけで足りるなら、それ以上広げない。
- 追加症状テンプレへ流さない。

## 3. delivered 承諾確認: `はい` が自然な短文

```text
はい、あと1回確認してから承諾で大丈夫です。
確認中に気になる点があれば、承諾前にこのトークルームで送ってください。
```

### 守ること

- buyer が yes/no を聞いている時は、先頭の `はい` は自然に使える。
- 承諾前に伝える導線だけを残し、closed 後の説明を盛りすぎない。

## 4. closed 通常再相談: 文中の `はい` を次アクションへ統合する

```text
ご連絡ありがとうございます。
前回とは別の内容かもしれないとのことなので、まずはこのメッセージで症状を送ってください。

確認だけで済む範囲はメッセージ上で確認します。
修正作業が必要な場合は、旧トークルームの続きではなく、ココナラ上で進め方と費用の有無を先にご相談します。

領収書メールが届かない操作の流れをそのまま送ってください。
```

### 守ること

- `まず今回の症状から確認します。はい、送って大丈夫です。` のように、文中で会話をリセットしない。
- closed 後なので、確認材料と修正作業の導線は残す。
- 報告時刻は、必要なら入れる。短い再相談では必須にしない。

## 5. 通常 prequote: 軽くてよい

```text
ご相談ありがとうございます。
Stripe側では成功していて、Next.js側のDB更新だけ止まっている不具合として、15,000円で対応できます。
進める場合は、そのままご購入ください。
ご購入後は、Webhook受信後にDB更新処理へ進めているかを確認します。
```

### 守ること

- buyer が復旧時間や保証を聞いていないなら、48時間目安や completion gate を毎回出さない。
- prequote では `ご購入後は` を添えて、作業開始感を抑える。

## 避けること

- buyer 文が短いという理由だけで、closed / 返金 / 保証 / 秘密情報の境界を削る。
- `はい` を全面禁止する。
- `その点も含めて` を固定接続句として毎回使う。
- 価格・phase・次アクションが必要な場面で、自然さを優先して消す。
- safety atom を全部並べて、buyer の主質問より運用説明を目立たせる。

## reviewer 観点

- buyer の文量・温度・質問数に対して、返信が契約説明化していないか。
- ただし、必要な safety boundary を削る指摘になっていないか。
- 状態受けから価格・scope・phase 導線へ移る時、今回の見方や判断基準が1文あるか。
- 短い yes/no 質問に、長い phase 説明を足していないか。
- closed / refund / guarantee / secret など高リスク場面では、短くしても必要境界が残っているか。
