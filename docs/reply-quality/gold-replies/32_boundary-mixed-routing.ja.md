# Gold Reply 32 — boundary mixed routing

## 使いどころ

- buyer が「これは不具合か仕様か」「整理と修正どちらが先か」など、相談の種類そのものを迷っている時
- 2サイト、別決済、別プロジェクトなどが混ざり、1件15,000円の範囲を広げすぎる危険がある時
- purchased / delivered / closed で、原因の技術分類と取引上の扱いが混ざって buyer が不安になっている時

## 出力方針

- まず buyer の迷いに答える。単なる不具合受付テンプレへ流さない。
- `整理` や `全体把握` の話が出ても、外向けには公開中の bugfix 範囲で言えることに閉じる。
- 複数サイト・別決済・別プロジェクトは、同じ症状に見えても1件に広げない。
- 技術的な原因の場所と、取引上の追加料金判定を混同しない。
- closed 後は、前回トークルームの続きとして作業できるように書かない。

## 1. bugfix 先行か整理先行か

```text
ご相談ありがとうございます。

この場合は、まず全体整理より customer.subscription.deleted のイベント処理が失敗する件から先に対応するのがよさそうです。

コード全体を先に整理し直す前提ではなく、修正に必要な範囲を確認します。

15,000円内で直し切れないと分かった場合は、追加作業へ進まず、そこでご説明します。

この内容で進める場合は、ご購入後に発生時のログや関係ファイルを送ってください。
```

### 守ること

- 未公開の整理サービスや別価格導線へ流さない。
- `コード全体を見ないと無理です` と広げすぎない。
- まず止まっている不具合を起点にし、必要範囲だけを見る。

## 2. 仕様か不具合か分からない相談

```text
ご相談ありがとうございます。

Stripe決済後のメールが30分ほど遅れて届く状態ですね。
「仕様か不具合か」をこの時点で断定することはできませんが、メール送信処理がどこで遅延しているかの原因確認は15,000円で対応できます。

原因を確認した結果、仕様上の制約や設計上の待ち時間だった場合は、その内容をお伝えしたうえで進め方をご相談します。

まずは決済完了後にメール送信が実行されるタイミングと、30分遅れている間に処理がどこで止まっているかを確認します。
この内容で進める場合は、そのままご購入ください。
```

### 守ること

- `不具合です` と断定しない。
- 外注先の「仕様です」を否定しない。
- 原因確認として受けられる範囲と、仕様上の制約だった場合の扱いを分ける。

## 3. 2サイト / 別決済 / 金額感

```text
ご相談ありがとうございます。
まず範囲を整理します。

2サイト分をまとめて15,000円で進める前提ではありません。
まずは一番困っている1サイト・1症状を、不具合1件として確認します。

Stripe側は不具合1件として15,000円で確認できます。
Square側もNext.js側のWebhook/API受信処理として見られる内容であれば確認できる可能性はありますが、別サイト・別決済なので別件扱いです。

両方を見る場合は、少なくとも2件分として30,000円が目安になる可能性があります。
まずは優先したいサイトと症状を1つ送ってください。対応範囲に入りそうかをお返しします。
```

### 守ること

- 2サイトを1件15,000円へ広げない。
- 非Stripe側は、決済事業者全体ではなく Next.js 側の Webhook / API 受信処理として見られる範囲に閉じる。
- buyer が金額を聞いている場合は、曖昧にせず金額感を出す。

## 4. purchased / 原因の場所と追加料金

```text
ありがとうございます。
追加料金のご不安、確認しました。

原因がフロント側だったとしても、今回依頼いただいた症状の原因として出てきたものであれば、それだけで別件扱いにはしません。
「原因の場所」と「追加料金になるか」は別の判断です。

今の時点では、追加料金になるとはまだ決まっていません。
勝手に追加料金が発生することはありません。

いまは、今回の症状とつながる内容として進められるかを確認しています。

本日19:07までに、修正方針をお返しします。
```

### 守ること

- `フロント側 = 別原因 = 追加料金` と短絡しない。
- 技術分類と取引上の扱いを分けて説明する。
- 追加料金は自動発生しないと先に伝える。

## 5. closed / 別プロジェクト同種バグ + 割引要求

```text
ご連絡ありがとうございます。

別プロジェクトの場合は、前回の修正範囲とは別件になります。
原因が似ていても、コードや環境が違うため、確認なしにそのままコピペ修正できるとはお約束できません。

内容を見て5,000円や10,000円に下げる形では受けておらず、この不具合修正サービスは15,000円固定です。

進める場合は、ココナラ上で新しいご依頼として進める形になります。前回のトークルームの続きとして作業を進めることはできません。
```

### 守ること

- `前回と同じ原因っぽいので安くします` と書かない。
- `コピペで直せます` と約束しない。
- closed 後に前回トークルームの続きとして作業するように書かない。
- 値引き可否だけで終えず、別件・固定価格・新しい導線を一息で示す。

## 6. delivered / 承諾を少し待ちたい

```text
大丈夫です。
承諾を少し待ちたいとのこと、承知しました。

確認が終わってからご判断いただければ問題ありません。
もし確認中に気になる点が出てきた場合は、承諾前にこのトークルーム内で送ってください。
```

### 守ること

- delivered を closed と混同して、メッセージ上の新規導線へ戻しすぎない。
- 承諾前に気になる点を送れる導線を出す。
- 自動承諾や差し戻し回数を、状況不明のまま外向けに言い切らない。

## 避けること

- `仕様か不具合か` の質問に、情報不足テンプレで返す
- `整理した方がよいか` から未公開サービスや別価格へ誘導する
- 2サイト・別決済・別プロジェクトを1件15,000円に広げる
- 原因の技術分類だけで追加料金を決める
- closed 後に旧トークルーム継続や割引修正を前提にする

## reviewer 観点

- buyer の質問は「不具合受付」ではなく、相談種類・仕様判断・取引導線・金額感を聞いていないか。
- `原因の場所` と `取引上の範囲` が混ざっていないか。
- 複数サイト、別決済、別プロジェクトで、1件15,000円の範囲を広げすぎていないか。
- closed 後は、前回範囲・無料/割引期待・旧トークルーム継続を作っていないか。
