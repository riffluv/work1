# Gold Reply 44 — quote_sent purchase CTA / materials order

## 使いどころ

- 見積り提案後、buyer がまだ購入判断中で、`直りそうなら購入したい` `支払い前に少し見てほしい` と言っている時。
- 購入前診断・GitHub確認・ログ先見を止めつつ、症状自体は対応範囲だと伝えたい時。
- 購入後の材料案内が先に出ることで、buyer がまだ同意していないのに買う前提で進んでいるように見える時。

## 出力方針

- まず、支払い前に原因確認・コード確認・GitHub確認を進めないことを明確にする。
- 次に、症状自体が対応範囲かを答える。
- その前提でよければ購入へ進める、と条件付きで CTA を置く。
- 購入後の材料案内は CTA の後に置く。
- secret 値を含めない注意は、材料案内の近くに置く。

## 1. quote_sent / GitHub先見 / 条件付き購入

```text
ご連絡ありがとうございます。

お支払い前は、GitHub上でコードを確認したり、原因の方向性を先に出したりする対応はしていません。
その判断は、お支払い完了後にログやコードを確認してから行います。

症状としては、本番でだけ起きていてローカルでは再現しない場合でも、Next.js / Stripeまわりの不具合として対応できます。

この前提で問題なければ、見積り提案の内容でお支払いへ進んでください。

お支払い完了後は、GitHub招待ではなく、関係ファイルやログをZIPまたは添付でトークルームに共有してください。
.env、APIキー、Webhook Secret、DB接続文字列などの値は含めないでください。
```

### 守ること

- `直りそうなら購入したい` への答えを落とさない。
- 購入後の材料案内を、購入判断の前に出しすぎない。
- `そのままご購入ください` が押し込みに見える時は、`この前提で問題なければ` を置く。
- prequote / quote_sent の標準 CTA は、buyer に判断余地が残る `この内容で問題なければ、そのままご購入ください` を優先する。`この内容で進める場合は` は、こちらが進行前提を強く置く響きが出やすいので、購入意思が強い時以外は使いすぎない。
- GitHub招待は作業面にしない。購入後もトークルーム添付・ZIPへ戻す。

## 2. 軽い共有タイミング確認との違い

buyer が単に `支払い後にまとめて送る形でよいですか` と聞いているだけなら、通常フローを先に出してよい。

```text
スクショやログは、お支払い完了後にトークルームでまとめて共有してください。
共有いただいた内容をもとに、原因確認から進めます。
```

ただし、`支払い前に見て、直りそうなら購入` のように購入前判断を求めている場合は、不可表明を省かない。

## 3. prequote / スクショだけ先見 / 見積り提案前との違い

buyer がまだ見積り提案前で、`支払い前にログだけ見て、直せそうなら購入したい` と言っている場合は、`見積り提案の内容で問題なければ` を使わない。

```text
ご相談ありがとうございます。

お支払い前は、ログやスクショを見て原因確認まで進めることはできません。

症状としては、Stripe決済後にアプリ側へ反映されない不具合なので、このサービスの対象になり得ます。

購入後に原因確認から進める形で問題なければ、15,000円で見積り提案します。
```

## 4. prequote / 購入意思あり / 支払い前に少しだけ見てほしい

buyer が購入意思を出していても、支払い前にファイルやログを見て原因確認することはしない。いきなり不可表明から入らず、短い受けを置いてから境界を伝える。

```text
ご相談ありがとうございます。

お支払い前は、ファイルやログを見て原因確認まで進めることはできません。

症状としては、決済後に会員状態が更新されない不具合なので、このサービスの対象になり得ます。

購入後に原因確認から進める形で問題なければ、15,000円で見積り提案します。
お支払い完了後に、関係しそうなファイルやログをトークルームで共有してください。
```

### 守ること

- 購入意思がある相手でも、支払い前診断は受けない。
- いきなり不可表明から入らず、短い中立的な受けを置く。
- buyer の `直りそうなら` `1つだけ` などの口語を貼り戻さない。
- 症状は必要な核だけ拾う。
- 購入後の材料案内は短く置く。

### prequote で守ること

- `見積り提案の内容で` は出さない。
- 購入後の材料案内を厚くしすぎない。buyer が購入判断中なら、まず対応可否と見積り提案へ進む前提を返す。
- `お支払い完了後にトークルームで...` の詳細案内は、quote_sent または buyer が材料共有を強く聞いている時に寄せる。

## reviewer 観点

- buyer がまだ購入判断中なのに、購入後の材料指示が先に出ていないか。
- 購入前診断不可を出したあと、対応範囲と購入判断の前提へつながっているか。
- CTA が buyer の温度に対して強すぎないか。
- 材料案内が phase と合っているか。支払い前作業開始に読めないか。
- `見積り提案の内容で` が quote_sent の state または buyer 文の明確な証拠なしに出ていないか。

## 避けること

- `ご購入ください` を blanket NG にすること。
- 軽い手順確認まで拒否先行にすること。
- 購入前診断を断るために、症状自体の対応可否まで曖昧にすること。
- 購入前のスクショ・症状ベースの見立てまで否定すること。購入前に止めるのは、コードやログを読んだ原因確認・修正可否の確定。
