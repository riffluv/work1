# Gold Reply 29 — price / guarantee / refund risk

## 使いどころ

- buyer が不具合の詳細ではなく、価格・保証・返金・キャンセル・復旧時間・保証期間を聞いている時
- `この不具合なら15,000円` の不具合受付テンプレへ流すと、主質問に答えられない時
- ココナラの手続き、completion gate、承諾前確認を分けて説明する必要がある時

## 出力方針

- 価格・返金・保証などのサービス構造質問には、まず仕様上いま答えられることを返す。
- 返金やキャンセルは、この場で一律に断定しない。
- ただし、原因や修正方針につながらず修正済みファイルを返せない状態で、一方的に正式納品へ進めないことは明確にする。
- 復旧時間と一次結果の目安を混同させない。
- fixed warranty を約束せず、同じ原因なら追加料金なしで確認する線を残す。

## 1. 固定価格 / 他社比較質問

```text
ご相談ありがとうございます。

難易度や作業時間にかかわらず、15,000円の固定価格です。
5分で原因が分かる場合でも、修正済みファイルの返却まで含めて15,000円になります。

他サービスとの価格比較については、こちらから安い高いの断定はしません。
このサービスでは、不具合1件について原因確認から修正、修正済みファイルの返却、確認手順の共有まで含めた料金です。

具体的な症状があれば、対応範囲に入りそうかをこのメッセージ上でお返しします。
```

### 守ること

- 難易度や作業時間で価格を揺らさない。
- 他サービス比較に出品者側から優劣をつけない。
- 15,000円に含む範囲を、調査時間ではなく「原因確認から修正済みファイル返却まで」として説明する。

## 2. 返金 / 途中キャンセル質問

```text
ご相談ありがとうございます。

返金や途中キャンセルについては、この場で一律に断定することはできません。
ココナラの手続きと、その時点の進み具合に沿って確認します。

原因や修正方針につながらず、修正済みファイルを返せない状態のまま、一方的に正式納品へ進めることはありません。
その場合は、分かった範囲をお伝えしたうえで、キャンセルを含めてご相談します。

「修正結果が気に入らない」場合については、依頼時の症状に対応できているかを確認する形になります。
修正内容に疑問がある場合は、承諾前にトークルーム内でお伝えください。

具体的な症状があれば、対応範囲に入りそうかを先にお返しします。
```

### 守ること

- `全額返金します` `返金できません` のように、手続き前に断定しない。
- `気に入らない` を好みの問題として広げず、依頼時の症状に対応できているかへ戻す。
- 承諾前にトークルームで伝えてもらう導線を出す。

## 3. 今日中復旧 / 設定戻し不明

```text
ご相談ありがとうございます。
お急ぎの状況は承知しました。

Webhookの設定を変更してから決済が止まり、元の設定が分からなくなっている状態ですね。
この不具合なら15,000円で対応できます。

今日中に復旧できるかは、原因を確認してからでないとお約束できません。
購入後は、まずWebhookの設定状態と、決済が止まっている箇所を優先して確認します。

この内容で進める場合は、そのままご購入いただいて大丈夫です。
必要情報がそろい次第、一次結果は48時間以内を目安にお返しします。これは復旧時間の保証ではなく、最初の確認結果の目安です。
```

### 守ること

- `今日中に直せます` と約束しない。
- `48時間以内` を復旧保証に見せない。
- 緊急感は受けつつ、まず確認する対象を具体化する。

## 4. 保証期間 / 再発時追加料金

```text
ありがとうございます。
保証期間と追加料金のご不安、確認しました。

固定の保証期間としてはお約束していません。
ただ、同じ問題が出た場合に、すぐ追加料金になると決まっているわけではありません。

同じ問題が出た場合は、まず今回の修正とつながる内容かを確認します。
同じ原因であれば、追加料金なしで確認します。別の原因と分かった場合も、勝手に追加料金が発生することはなく、進める前にご相談します。

本日15:39までに、現時点の確認結果をお返しします。
```

### 守ること

- 固定保証期間を作らない。
- `同じ問題` と `同じ原因` を混同しない。
- 別原因でも、勝手に追加料金が発生しないことを明確にする。

## 避けること

- 価格・返金・保証質問に、通常の不具合受付テンプレートで返す
- `この不具合なら15,000円` を、不具合が提示されていない価格構造質問へ使う
- 復旧時間と一次結果の目安を混同させる
- 返金可否やキャンセル結果を、ココナラ手続き前に断定する
- fixed warranty を作る

## reviewer 観点

- buyer は不具合内容ではなく、価格構造・返金・保証・復旧時間を聞いていないか。
- その場合、最初の answer-bearing section が仕様質問への答えになっているか。
- ココナラ手続き、completion gate、承諾前確認、追加料金自動発生なしが混ざらず分けられているか。
