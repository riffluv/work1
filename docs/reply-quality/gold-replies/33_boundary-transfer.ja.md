# Gold Reply 33 — boundary transfer

## 使いどころ

- buyer が「不具合修正で頼むべきか、先に整理すべきか」と迷っている時
- 不具合は出ていないが、動いている状態に漠然と不安がある時
- 複数症状と予算提示が同時に出て、1件15,000円の範囲を広げすぎる危険がある時
- quote_sent 後に Stripe 以外の処理名や追加ファイルが出てきた時
- purchased で保証期間や同じ問題の追加料金を聞かれた時
- closed 後にリピート割引や別箇所の新規相談が来た時

## 出力方針

- 未公開サービスへ流さず、公開中の `bugfix-15000` で言える範囲に閉じる。
- 不具合がない時は、無理に購入させず「具体的な不具合が出てからでよい」と返す。
- buyer の怒りや不信は事実ベースで1文だけ受け、過剰な謝罪や責任承認へ広げない。
- 複数症状は、同じ原因なら1件、別原因なら2件扱いの可能性を buyer の予算に接続する。
- quote_sent では `トークルーム内` と書かず、`ココナラ上` や `見積り相談上` に留める。
- 保証期間は、固定保証の有無、同じ原因の場合、別原因の場合を分ける。

## 1. 不具合なし・漠然不安

```text
ご相談ありがとうございます。
変ではありません。

ただ、このサービスは具体的な不具合が出ている状態を前提にしています。
「動いているが、なんとなく不安」という段階であれば、無理に購入を急がなくて大丈夫です。

決済が通らない、エラーが出る、表示がおかしいなど、気になる画面や操作があれば、それを1件の確認対象にできるか見立てます。
```

### 守ること

- 不具合がない状態で無理に購入を促さない。
- 未公開の整理サービスや別価格へ誘導しない。
- `変ではありません` の後に、bugfix サービスの前提を短く示す。

## 2. 怒り気味 buyer + bugfix / 整理判断

```text
ご相談ありがとうございます。
3ヶ月ほど決済まわりが動かない状態が続いているとのことなので、かなりお困りだと思います。

まずは決済が動くようにすることを優先して、コード全体の整理よりも不具合修正から入るのがよさそうです。

コードが何をしているか分からない状態でも、決済まわりの処理に絞って確認します。先に全体整理を前提にはしません。
15,000円内で直し切れないと分かった場合は、追加作業へ進まず、そこでご説明します。

この内容で進める場合は、そのままご購入ください。
```

### 守ること

- 怒りや不信を無視して、冷静な buyer と同一文面にしない。
- `大変申し訳ございません` のように過失認定へ寄せない。
- 整理が必要そうに見えても、外向けにはまず止まっている不具合を起点にする。

## 3. 複数症状 + 予算提示

```text
ご相談ありがとうございます。
ページの読み込み遅延と、Stripe決済後のWebhook不達の2つですね。

2つの症状が同じ原因で起きている場合は、不具合1件として15,000円の範囲で対応できる可能性があります。
別原因だった場合は2件扱いになる可能性があるため、30,000円の予算内で両方見られるかを先に切り分けます。
ページ速度改善が不具合修正の範囲を超える場合も、その時点で切り分けます。

その場合も、勝手に料金が増えたり、そのまま追加作業へ進むことはありません。
まずは2つが同じ原因でつながっていそうかを確認し、進め方をお返しします。
```

### 守ること

- buyer の予算提示を無視しない。
- 2症状を自動で1件扱いにしない。
- ページ速度改善が純粋なパフォーマンス改善へ広がる場合は、bugfix 範囲と分ける。

## 4. quote_sent 後の Stripe 以外の処理 + 追加ファイル

```text
ご連絡ありがとうございます。
提案後に変更したい点が出てきたとのことですね。

Stripeに関係する範囲であれば、SendGrid側の処理も今回の件として一緒に確認できる可能性があります。
Stripe以外の別の原因まで広がる場合だけ、その時点で切り分けてお返しします。

送り忘れていたファイルは、秘密情報を含めず、ココナラ上で送ってください。
```

### 守ること

- SendGrid など別サービス名が出ても、Stripe に関係する処理かどうかで切る。
- quote_sent で `トークルーム内` と書かない。
- 追加ファイルは受けても、生の秘密情報は送らせない。

## 5. purchased の保証期間 / 同じ問題

```text
ありがとうございます。
保証期間と追加料金のご不安、確認しました。

固定の保証期間としてはお約束していません。

ただ、同じ問題が出た場合に、すぐ追加料金になると決まっているわけではありません。
まずは今回の修正とつながる内容かを確認します。

同じ原因であれば、追加料金なしで確認します。
別の原因と分かった場合も、勝手に追加料金が発生することはなく、進める前にご相談します。

本日19:48までに、現時点の確認結果をお返しします。
```

### 守ること

- `追加料金になるとはまだ決まっていません` だけで終えない。
- 固定保証期間の有無を先に答える。
- 同じ原因の場合と別原因の場合を分ける。

## 6. closed 後のリピート割引要求

```text
ご連絡ありがとうございます。
またご相談いただけてありがたいです。

割引前提でのご案内はしておらず、この不具合修正サービスは15,000円固定です。今回の内容を見て見積りをお返しすることはできます。

トークルームは閉じているので、今回見たい不具合の内容が分かれば見積りの入口を整えられます。
今回見たい不具合の内容をそのまま送ってください。

送っていただいた内容を確認して、本日20:00までに見立てをお返しします。
```

### 守ること

- リピーターでも割引前提にしない。
- closed 後に前回トークルームの続きとして作業するように書かない。
- 値引き可否だけで終えず、今回の不具合内容を送る導線まで出す。

## 避けること

- 不具合がない不安に、購入を急がせる
- 怒り気味 buyer へ冷静な buyer と同一テンプレートで返す
- `30,000円まで出せます` を無視して、1件15,000円だけで終える
- quote_sent で `トークルーム内` と書く
- 保証期間質問に、追加料金テンプレだけで返す
- closed 後のリピート割引を、前回範囲や無料対応に寄せる

## reviewer 観点

- buyer の相談は「不具合そのもの」ではなく、どの入口で頼むか、今買うべきか、割引されるか、保証されるかを聞いていないか。
- 怒りや不信がある時に、事実ベースの受け止めを1文だけ置けているか。
- quote_sent / purchased / closed の phase 語彙が混ざっていないか。
- 予算提示や保証期間質問に、主質問へ直接答えずテンプレで流していないか。
