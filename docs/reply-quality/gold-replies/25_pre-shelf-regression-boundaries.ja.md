# Gold Reply 25 — pre-shelf regression boundaries

## 使いどころ

- batch-12〜17 の棚卸し後に、再発性が高いが本文へ常時入れると重くなる境界を見る時
- `quote_sent` の追加料金/キャンセル不安
- `prequote` の追加費用・返金・予算上限不安
- 技術文脈の `キャンセル` を含む不具合1件確認
- 購入前の `対処法だけ教えて` への線引き
- `delivered` の承諾後不安

## completion gate の内部判断モデル

この gold は、外向け文面を長くするためではなく、返信前に次の取引判断を固定するために使う。

- 15,000円で修正完了まで進める前提で受ける
- 別原因・別フロー・重い修正が見つかったら、勝手に追加作業へ進まない
- 15,000円内で完了できないと分かったら、そこで止めて説明する
- buyer が追加費用を望まないなら、未完成のまま正式納品へ押し切らない
- キャンセル/返金は断定せず、作業状況とココナラ上の手続きに沿って相談する
- 軽微で吸収できるなら、内部判断で基本料金内対応にする余地は残す

外向けには、必要な部分だけを短く出す。
標準は `15,000円の範囲で進める前提です。超える場合は止めて説明します。勝手に料金が増えたり、そのまま追加作業へ進むことはありません。` までで足りることが多い。

## 1. quote_sent / 追加料金とキャンセル不安

```text
ご連絡ありがとうございます。
金額が増えるのが不安という点を先に整理します。

今回の見積もりは15,000円の範囲で進める前提です。
確認の結果、別原因が複数あり、この金額内では修正完了まで進められないと分かった場合は、そこで止めてご説明します。

勝手に料金が増えたり、そのまま追加作業へ進むことはありません。

その場合は、キャンセル扱いを含めて、ココナラ上の手続きに沿ってご相談します。

この前提でよければ、そのままご購入ください。
```

## 2. prequote / 技術的なキャンセル語と1件可否

```text
ご相談ありがとうございます。

Customer Portal からのサブスクキャンセル時に、Webhook イベントの順序で DB がおかしくなる件ですね。

この内容なら、Stripe Customer Portal のキャンセル処理に関する不具合1件として見られそうです。
customer.subscription.updated と customer.subscription.deleted の順序や、DB更新の扱いを確認する形になります。

15,000円で進める場合は、そのままご購入ください。
ご購入後に該当Webhookのログと、DB更新まわりの関係ファイルを確認し、一次結果は48時間以内を目安にお返しします。
```

## 3. prequote / 追加費用が怖い・2件になったら不安

```text
ご相談ありがとうございます。
追加費用が不安という点を先に整理します。

2つの症状が同じ原因で起きている場合は、不具合1件として15,000円の範囲で見られる可能性があります。
別原因だった場合は、両方をこの金額内で直し切れるとは限りません。

その場合も、勝手に料金が増えたり、そのまま追加作業へ進むことはありません。
この金額内では修正完了まで進められないと分かった時点で止めてご説明します。

今書いていただいた2つを起点に、1件として見られそうかを先にお返しします。
```

## 4. prequote / 原因不明・返金・無駄にならないか

```text
ご相談ありがとうございます。
無駄にならないかという点を先に整理します。

15,000円は調査だけの料金ではなく、原因確認から修正、確認手順、修正済みファイルの返却まで進める前提です。

確認の結果、この金額内では修正完了まで進められないと分かった場合は、未完成のまま正式納品へは進めず、そこで止めて状況をご説明します。

勝手に料金が増えたり、そのまま追加作業へ進むことはありません。

返金やキャンセル扱いをこの時点で断定することはできませんが、必要になった場合はココナラ上の手続きに沿ってご相談します。

具体的な症状がまだであれば、分かる範囲で送ってください。対応範囲に入りそうかを先に見立てます。
```

## 5. prequote / 全部見て全部直したい・予算上限あり

```text
ご相談ありがとうございます。
予算内で全部見て全部直せるか、という点を先に整理します。

今の時点では、3万円以内で全体を直し切れるとは断定できません。
このサービスで進める場合は、まず直したい不具合を1件に絞り、15,000円の範囲で確認します。

見ていく中で別原因が複数あり、この金額内では修正完了まで進められないと分かった場合は、そこで止めてご説明します。
勝手に料金が増えたり、そのまま追加作業へ進むことはありません。

まず一番困っている症状を送ってください。そこから対応範囲を見立てます。
```

## 6. prequote / 3本あるが1本だけ不具合

```text
ご相談ありがとうございます。
3本分の料金になるのか、という点を先に整理します。

症状が出ているのが1本だけであれば、まずそのAPIを不具合1件として15,000円の範囲で確認します。
念のため3本すべてを同じ深さで確認する前提ではありません。

見ていく中で別のAPIも直さないと修正完了まで進められないと分かった場合は、そこで止めてご説明します。
勝手に3件分の料金にしたり、そのまま追加作業へ進むことはありません。

まずは、レスポンスが返らなくなる1本のAPIと、発生時の状況を送ってください。そこから1件として見られそうかをお返しします。
```

## 7. prequote / 対処法だけ教えてほしい

```text
ご相談ありがとうございます。

購入前に具体的な修正手順や、コード上の直し方まではお伝えしていません。

Webhook署名検証エラーであれば、対応範囲に入る可能性は高いです。
購入前は、症状・エラー内容・環境から見立てまでお返しできます。

具体的な原因確認と修正は、ご購入後にコードとログを確認して進めます。
この内容で進める場合は、そのままご購入ください。
```

## 8. delivered / 承諾後に同じ箇所で問題が出た場合

```text
ご連絡ありがとうございます。
軽く見た範囲では大丈夫そうとのこと、確認しました。

承諾前であれば、気になる点はこのトークルーム内で送ってください。
合意内容と違う点があるか確認します。

まだ確認できていない箇所がある場合は、無理に承諾しなくて大丈夫です。

承諾後もメッセージで状況確認はできますが、トークルームは閉じるため、そのまま修正作業を続ける前提にはなりません。
同じ箇所に不具合が残っている可能性がある場合は、内容を確認したうえで進め方をご相談します。
```

## この型で守っていること

- `quote_sent` では、購入前にやめられること、自動増額しないこと、未完成のまま進めないこと、キャンセル可否を断定しないことを分ける
- `prequote` の予算上限・複数原因・返金不安では、購入を止める方向に押し戻さず、15,000円内で進める前提と、超える場合に止める導線を先に出す
- 技術文脈の `キャンセル` を、ココナラ取引キャンセルと混同しない
- 購入前の具体手順要求は断りつつ、見立てまでは返す
- `delivered` は `closed` ではないが、承諾後は closed に進むため、承諾前確認と承諾後の再相談を分ける

## 使い方

- そのままテンプレート化しない。
- `transaction_model_gap` が出そうな時に、支払い状態、作業開始条件、追加相談、承諾後導線の順序だけ合わせる。
- 近い closed 後ケースは Gold 23/24 を優先して見る。
