# Gold Reply 24 — transaction model gap を埋める

## 使いどころ

- 日本語は成立しているのに、buyer が「結局、どの順番で進むのか」で止まりそうな時
- 追加症状、購入なしの修正助言、クローズ後の次回相談など、料金・作業開始・見積り導線が絡む時
- `phase_answer_gap` より一段深く、取引構造そのものが曖昧な時

## 出力方針

- この gold は、長文テンプレートではなく取引順序の anchor として使う
- `transaction_model_gap` は内部で厚く見て、外向け本文には buyer が聞いた順序だけを短く出す
- 既定は `直答 1〜2文 -> 必要な境界 1〜2文 -> 次アクション 1文`
- `同一原因 / 別原因 / 追加料金 / closed後作業不可 / 見積り導線` を、buyer が聞いていないのに全部並べない

## 1. purchased / 追加症状と追加料金不安

```text
ありがとうございます。

管理画面の注文一覧も読み込みが遅い件、確認しました。
今回の決済エラーと同じ原因であれば、今回の件として一緒に確認します。

別の原因と分かった場合だけ、追加対応が必要かどうかを先にご相談します。勝手に追加料金が発生することはありません。

まずは今の決済エラーとのつながりを確認します。
読み込みが遅い画面やエラー表示があれば送ってください。まだ出せなければ、先に今の本件を進めます。
```

## 2. closed / 購入なしのコード修正助言

```text
ご連絡ありがとうございます。

前回と同じエラーかどうかは、まずこのメッセージ上でエラー内容を確認できます。
ただし、コードのどこをどう直すかを具体的にお伝えすることは、修正作業の一部になります。

トークルームは閉じているため、購入なしでそのまま具体的な修正指示まで進めることはできません。
まずは今出ているエラー内容やログを送ってください。

前回の修正とつながる内容かを見たうえで、作業が必要な場合は、ココナラ上でどう進めるかと費用が発生するかを先にご相談します。
```

## 3. closed / 次回相談はメッセージ先か購入先か

```text
先日はありがとうございました。
次の流れとしては、まずこのメッセージで症状の概要を送ってください。

内容を確認して、対応できるかと次の進め方をお返しします。
ご購入は、見積り提案を確認いただいてからで大丈夫です。

今出ている決済エラーの内容を送ってください。
```

## この型で守っていること

- 追加症状では、同じ原因なら今回内、別原因なら先に相談、追加料金自動発生なしを一息で返す
- `closed` 後の「文章で直し方だけ教える」は、ファイル返却がなくても修正作業寄りとして扱う
- 次回相談では、先に購入させず、メッセージ上の症状概要 -> 見積り提案 -> 購入の順にする
- 「確認します」だけで逃げず、支払い・作業・見積りの順番を buyer に見せる

## 使い方

- そのままテンプレート化せず、取引構造の順番だけを合わせる。
- 近い型が Gold 22 / 23 にある場合は、そちらを優先して読む。
- `transaction_model_gap` が出た時に、料金・作業開始・成果物返却・見積り提案のどこが曖昧かを確認する。
- 本文が長くなった時は、`minimal_outward` として次のような1文へ圧縮できないか確認する。
  - `同じ原因なら今回の件として見ます。別原因と分かった場合だけ、作業前に先にご相談します。`
  - `購入なしで、コードを見て具体的な直し方を返すところまでは進めていません。`
  - `まずこのメッセージで症状の概要を送ってください。見積り提案に進むかは、その後にご案内します。`
