# Gold Reply 36 — conversation flow naturalness

## 使いどころ

- 内容は正しいが、短い断定文が続いてマニュアル的・AI的に見える時
- 価格、納期、scope、phase、secret などの安全境界を残したまま、会話として読みやすくしたい時
- `やさしくする` ではなく、buyer が答え・条件・次アクションを迷わず読めるようにしたい時

## 出力方針

- 句点「。」そのものを減らす目的で直さない。
- つなぐのは、同じ役割で強く関係する文だけにする。
- 金額、scope、phase、secret、payment route、public/private の判断は変えない。
- 固定価格を `想定しています` に弱めない。
- 最後は、buyer または seller の次アクションが見える形にする。

## 1. prequote 料金 + 納期: 固定価格は弱めず、納期条件だけつなぐ

```text
ご相談ありがとうございます。
Stripe決済後に注文が作られない不具合であれば、金額は15,000円です。

原因確認から修正済みファイルの返却まで、この料金内で進めます。日数はサービス上3日が目安ですが、コードの状態やエラー内容によって前後します。

ご購入後にコードとエラー内容を確認し、より正確な見通しをお返しします。
この内容で進める場合は、そのままご購入ください。
```

### 守ること

- `15,000円です` / `15,000円で進められます` のように固定価格を前半に残す。
- `想定しています` へ弱めない。
- 納期目安と変動条件だけを1文でつなぐ。
- `必要情報がそろい次第` ではなく、確認対象を具体化する。

## 1.5. prequote 原因不明相談: 先頭で相談可否に直答する

```text
ご相談ありがとうございます。

原因が分からない状態でも相談できます。Stripe決済後に注文が作られない不具合として、15,000円で進められます。

ご購入後にコードとエラー内容を見て、どこで止まっているかを確認します。確認できた時点で、一次結果は48時間以内を目安にお返しします。

この内容で進める場合は、そのままご購入ください。
```

### 守ること

- `原因が分からない状態でも相談できます` と、主質問へ先に答える。
- 原因不明を理由に handoff や整理へ逃がさない。
- `まず必要なところから確認します` だけで始めず、buyer の不安に対する可否を明確にする。
- `必要情報がそろい次第` のような待機条件だけで閉じず、コードとエラー内容を見ることを具体化する。

## 2. purchased 受領確認: 受け取ったものと次の確認行動を分ける

```text
ログとスクショありがとうございます。

昨日のログとスクショは届いています。この後はこちらで内容を確認し、追加で必要なものが出た場合だけお伝えします。
いまの時点で、追加で準備いただくものはありません。
```

### 守ること

- `確認しました` の直後に `はい、確認できています` を重ねない。
- 受領確認は1回で済ませる。
- 次に seller が何をするかを添える。

## 3. 追加情報依頼: なぜ必要か、何を送るか、送った後どうなるか

```text
このまま原因確認を進めるために、あと3点だけお願いします。

- エラー画面のスクリーンショット
- 該当箇所のコード
- どの操作で再現するか

お送りいただいた内容を確認したうえで、修正方針をお返しします。
```

### 守ること

- 質問票だけを出さない。
- secret 値は求めない。
- buyer が送った後に何が起きるかを1文で示す。

## 4. scope out: 確認したこと、範囲外、必要時の相談を順に置く

```text
ご希望の内容は確認しました。
ただ、今回の料金内で対応できるのは既存コード上の不具合修正までです。

新しい機能追加や画面全体の作り直しは、この範囲には含めていません。
必要であれば、進める前に追加対応として対応可否と金額を確認します。
```

### 守ること

- scope 外を「別料金で対応します」と直結しない。
- 進める前に相談する形にする。
- live で未公開サービスの導線を出さない。

## 5. 進捗報告: 完了したこと、今見ていること、次回時刻

```text
ありがとうございます。
お待たせしてすみません。先に現状だけお伝えします。

現時点では、決済後の反映処理がどこで止まっているかを見ている段階です。まだ原因は断定していません。

本日18:00までに、確認できている点だけ短くお返しします。
```

### 守ること

- `確認しました / 確認します / 確認結果` を密集させない。
- 現在地と未完了点を分ける。
- 次回連絡の時刻や条件を明確にする。

## 6. 納品後の補足: 承諾前の軽い説明

```text
ご連絡ありがとうございます。
確認手順が少し分かりにくかったとのこと、承知しました。

承諾前に、確認するポイントをもう少し分かりやすく補足できます。専門用語を減らしてこちらで整理します。
追加で特に知りたい箇所があれば、そのまま送ってください。
```

### 守ること

- 納品後 phase では、確認してほしい対象を簡潔に出す。
- 返金・保証・無料対応を足さない。
- closed 後の再相談文と混ぜない。

## 7. closed 後の再相談: 旧トークルーム継続にしない

```text
このたびはご依頼ありがとうございました。

クローズ後に新しい不具合や追加のご相談がある場合は、まず症状と該当箇所をメッセージでお送りください。
内容を確認したうえで、新規のお見積りとして対応可否をお返しします。
```

### 守ること

- closed 後に `このトークルームでそのまま` と言わない。
- 旧トークルームのおひねり継続や無料修正を約束しない。
- まず症状の概要を受け、実作業は新規見積りとして分ける。

## 7.5. closed 後の関係確認: 送ってよいかへ先に答える

```text
ご連絡ありがとうございます。
クローズ後のご相談ですね。

ログやスクショは、このメッセージで送っていただいて大丈夫です。届いた範囲で、前回の修正と関係がありそうかを見立てます。

トークルームは閉じているため、この場で修正作業や修正済みファイルの返却までは進めません。実作業が必要そうな場合は、見積り提案または新規依頼として対応方法をご相談します。
```

### 守ること

- `まずは内容を確認します` のような汎用処理語で始めない。
- `ログやスクショを送ってください` だけを切り出さず、buyer の「このメッセージで送ってよいか」に答える。
- `閉じているため` の境界説明を、依頼者の次アクションより前に出しすぎない。
- `見立てをお返しします` を重ねず、届いた範囲で見ることと実作業の分離を1回で示す。

## 避けること

- 句点の有無だけで自然・不自然を判定する。
- 短文を全部つなげる。
- 価格や scope を弱めて会話っぽくする。
- `ご案内します` を汎用締めにする。
- `やさしくする` ために約束や支払い導線を増やす。
- Deep Research の例文を、phase / scope / public-private の確認なしでそのまま使う。

## reviewer 観点

- 主質問への直答が前半にあるか。
- 条件・理由・次アクションが、buyer に見える順で並んでいるか。
- 短文断定が3文以上続く場合、意味を変えずに1箇所だけ自然化できるか。
- 自然化によって price / scope / phase / secret / payment route が揺れていないか。
- 好み差を hard fail にしていないか。
