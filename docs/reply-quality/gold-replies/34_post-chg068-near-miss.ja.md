# Gold Reply 34 — post CHG-068 near miss

## 使いどころ

- CHG-068 後に、非Stripe scope / 緊急復旧 / 仕様と不具合 / 返金期待 / 複数症状料金 / purchased 複数話題 / closed 疲弊が混ざる時
- r0 で一部は安定し始めたが、複合ケースではまだ主質問の取り落としが起きやすい時
- 「確認します」だけでは buyer が料金・返金・外部連絡・作業導線で迷う時

## 出力方針

- 非Stripe決済は、決済サービス全体ではなく `Next.js側のWebhook/API受信処理` に閉じて答える。
- 緊急復旧では、`対応可否 / 今日中保証不可 / 48時間は一次結果の目安` を分ける。
- 仕様か不具合か分からない時は断定せず、原因確認は受け、仕様上の制約なら内容を伝えて相談する。
- 返金期待が混ざる時は、返金を断定せず、ココナラ上の手続き・作業状況・completion gate で答える。
- 複数症状は、2つ目を無視しない。同じ原因なら1件、別原因なら2件扱いの可能性を先に示す。
- purchased で複数話題が混ざる時は、進捗 / 別件 / 外部連絡を分けて全部答える。
- closed 後は、メッセージ上の確認と修正作業の正式導線を分ける。

## 1. 仕様/不具合 + 返金 + 他者修正失敗

```text
ご相談ありがとうございます。
2回頼んでも直らず、今回も本当に直るのか不安に感じる状況かと思います。
仕様か不具合かも判断しづらい状態ですね。

「仕様か不具合か」をこの時点で断定することはできませんが、原因確認は15,000円で対応できます。
原因を確認した結果、仕様上の制約や設計上の待ち時間だった場合は、その内容をお伝えしたうえで対応をご相談します。

返金については、この場で一律に断定することはできません。
修正対象ではない、または修正方針につながらないと分かった場合は、分かった内容と作業状況を整理し、ココナラ上の手続きに沿って進め方をご相談します。

修正済みファイルを返せない状態のまま、一方的に正式納品へ進めることはありません。
この内容で進める場合は、そのままご購入ください。
```

### 守ること

- 他者修正失敗への不信は事実ベースで受ける。
- `次に頼む先を決めるのも不安` のような抽象的な共感にせず、`今回も本当に直るのか不安` のように buyer の判断不安へ寄せる。
- 仕様だった場合の返金を約束しない。
- `仕様か不具合か` を seller が先に断定しない。
- completion gate は出すが、契約説明だけにしない。

## 2. 複数症状 + 料金 + 急ぎ

```text
ご相談ありがとうございます。
お急ぎの状況は承知しました。

メール不達と、管理画面の注文一覧が空になる件の2つですね。

同じ原因で起きている場合は、不具合1件として15,000円の範囲で対応できる可能性があります。
別原因だった場合は2件扱いになる可能性がありますが、その場合も勝手に料金が増えたり、そのまま追加作業へ進むことはありません。

まずは2つが同じ原因でつながっていそうかを確認し、進め方をお返しします。
この内容で進める場合は、そのままご購入ください。必要情報がそろい次第、一次結果は48時間以内を目安にお返しします。
```

### 守ること

- 1つ目の症状だけ拾って `1件として確認` に流さない。
- 2件分料金かという質問に答える。
- 急ぎの温度は受けるが、復旧や完了時間は約束しない。

## 3. purchased 複数話題混在

```text
ご連絡ありがとうございます。
お待たせしてすみません。3点確認します。

まず進捗ですが、いまは今回の対象処理でどこまで更新が進んでいるかを確認しています。まだ原因は断定していません。

「退会できない」件については、今回の依頼と同じ原因でつながる内容かを急ぎで確認します。
別原因だと分かった場合は、追加対応が必要かどうかを先にご相談します。

Slackへの招待は不要です。やり取りはこのトークルーム内で進めます。

本日04:33までに、状況を整理してお送りします。
```

### 守ること

- 進捗だけ返して、別件と外部連絡を落とさない。
- `原因の切り分け中です` の同義反復で終えない。
- purchased のトークルームでは Slack など外部連絡へ逃がさない。

## 4. 秘密情報を実際に送られた時

```text
ご連絡ありがとうございます。
共有いただいた秘密情報の件、承知しました。

.env やログイン情報、接続URLなどの値は扱いません。
以後はキー名や設定箇所だけで大丈夫です。実際の値や接続URLは送らないでください。
該当メッセージを編集や削除できる状態なら、先にそうしてください。
必要なら、キーやパスワードは後で再設定しておくのが安全です。

本日04:33までに、現時点の確認結果をお返しします。
```

### 守ること

- `キー名やURLだけ` と書かない。接続URLは秘密情報として扱う。
- 送られた値を分析対象にしない。
- 削除・再設定の検討まで案内する。

## 5. closed 疲弊 + 無料期待

```text
ご連絡ありがとうございます。
同じような症状がまた出ていて、前回分との関係や費用面も気になる状況かと思います。

すぐに無料対応・追加費用のどちらかを決めるのではなく、まず今回の症状が前回の修正と同じ原因につながるものかを確認します。

トークルームは閉じていますが、いま出ている画面と同じ操作で起きるかが分かれば、前の件とのつながりを確認できます。
今の症状が出ている画面スクショと、同じ操作で起きているかを送ってください。

確認だけで済む範囲はメッセージ上で確認します。修正作業が必要な場合は、旧トークルームの続きではなく、ココナラ上で進め方をご相談します。

本日04:45までに、見立てをお返しします。
```

### 守ること

- 無料対応を約束しない。
- 即15,000円請求にも寄せない。
- closed 後は、確認材料と修正作業の導線を分ける。
- 疲弊・費用不満が明示されている時は、症状だけを `確認しました` で受けず、状態を事実ベースで1文だけ受ける。
- ただし、謝罪・過失認定・返金断定・無料対応約束には広げない。

## 避けること

- 複数質問のうち、返金・料金・Slack・秘密情報など事故りやすい問いを落とす
- 接続URLや秘密値の再共有を促す
- purchased の進捗を `切り分け中です` だけで終える
- closed 後に旧トークルーム継続を前提にする
- 仕様/不具合を確認前に断定する

## reviewer 観点

- 複合質問で、buyer の明示質問を全て回収しているか。
- 料金・返金・外部連絡・秘密情報・closed 後導線のいずれかを「確認します」に逃がしていないか。
- `見られる` 系が続いて硬くなっていないか。`確認します / 対応できます / つながりを確認します` に直せるか。
- URL と接続URLを混同していないか。接続URLは秘密情報として扱えているか。
- buyer が怒り・疲弊・不安・焦り・不信・困惑・遠慮・無料/返金不満を明示している時、症状・料金・手順だけを受けて状態シグナルを落としていないか。
