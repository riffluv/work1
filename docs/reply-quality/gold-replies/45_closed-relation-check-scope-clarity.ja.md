# Gold Reply 45 — closed relation check scope clarity

## 使いどころ

- クローズ後に、前回修正と関係があるかだけ見てほしいと言われた時。
- `ログやスクショを送れば見てもらえますか` と聞かれ、材料確認と実作業の境界を分けたい時。
- `確認します` が、コード修正や修正済みファイル返却まで含むように読まれそうな時。

## 出力方針

- まず、ログやスクショをこのメッセージで送ってよいかに答える。
- 次に、そこで何を確認するのかを具体化する。
- このメッセージ上でできる範囲を明示する。
- コード修正・修正済みファイル返却・費用判断は、実作業前に相談する。

## 1. closed / 似た症状 / 関係確認

```text
ご連絡ありがとうございます。

ログとスクショは、このメッセージで送っていただいて大丈夫です。
届いた内容をもとに、前回の修正と関係がありそうか、新しい不具合の可能性が高そうかを短く確認します。

ここでできるのは、ログとスクショから見える範囲での関係確認までです。
コード修正や修正済みファイルの返却が必要になりそうな場合は、対応方法と費用を先にご相談します。
```

### 守ること

- `確認します` だけで終わらせず、何を確認するのかを buyer に見せる。
- closed 後の関係確認を、修正作業開始・無料対応・返金判断に広げない。
- 旧トークルーム継続やおひねり追加で作業するように案内しない。
- 返金・無料・前回ミスは、確認前に断定しない。

## 2. 圧力語がある時

`無料で直してほしい` `返金になりますか` `前回ミスでは` のような圧力語がある場合も、その語をそのまま強く復唱しない。

```text
この時点では、前回の修正との関係、作業可否、費用や返金の扱いは断定できません。
まずはこのメッセージ上で状況を確認し、実作業が必要そうな場合は対応方法と費用を先にご相談します。
```

## reviewer 観点

- closed 後の `確認` が、ログ・スクショから見える範囲の見立てなのか、コード修正なのか分かるか。
- 実作業・修正済みファイル返却・費用相談の境界が見えるか。
- 無料対応、返金、前回ミス、旧トークルーム継続を約束していないか。
- 関係確認の価値を消さず、buyer が次に送るものと待つものが分かるか。

## 避けること

- closed 後だからといって、関係確認まで冷たく拒否すること。
- `送ってください` の直後に、修正作業や成果物返却を約束すること。
- `確認します` と `修正できません` を断片的に並べ、buyer に「何を確認してくれるのか」を残すこと。
