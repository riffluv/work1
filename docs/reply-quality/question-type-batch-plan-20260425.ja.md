# question-type batch plan（2026-04-25）

## 目的

`#RE` の次段階では、buyer が不具合そのものではなく、価格・対象・納期・保証などのサービス仕様を聞いているケースを重点観察する。

このメモは生成ルールではない。`batch_manifest` / learning-log / reviewer 監査で使う観察ラベルとして扱う。

## 背景

batch-03 では、buyer が次のような質問をしているのに、現行出力が汎用の不具合対応テンプレートへ流れた。

- 何時間で復旧できますか？
- 5分で分かっても15,000円ですか？
- こういうのも対象ですか？
- コード問題ですか？設定問題ですか？
- 保証期間はありますか？

この問題の本質は、返信文の自然さではなく `主質問タイプの取り違え`。
ただし、現時点で質問タイプ router を新設したり、renderer を刷新したりしない。まず 1〜3 batch で狭く再発確認する。

## buyer 質問タイプ

P0:

- `サービス仕様質問型`: 仕様上の答えを先に返す
- `価格不安型`: 15,000円固定・追加料金自動発生なしを先に返す
- `対象可否型`: 対象に近い / 対象外 / 判断に必要な条件を先に返す
- `納期・復旧時間型`: 断定可否と回答可能な範囲を先に返す
- `原因不明・正式納品可否型`: 原因不明のまま正式納品へ進めない条件を先に返す

P1:

- `秘密情報・共有方法型`: 秘密値不要・ZIP寄せを先に返す
- `本番反映・直接操作型`: 本番デプロイ代行や直接操作はしないと先に返す
- `保証・再発型`: 同一原因か別原因かで扱いが変わると先に返す
- `購入後進捗型`: 現在の確認状態・次に見る点を先に返す
- `納品後追加確認型`: 同一原因なら範囲内、別原因なら追加相談と先に返す

P2:

- `不具合相談型`: 対象範囲か、必要情報は何かを先に返す
- `原因カテゴリ・自力修正可否型`: 現時点で断定せず、切り分け方を先に返す

P3:

- `未公開サービス混線型`: 外向けには bugfix-15000 の範囲内外だけで答える

## rule に戻す条件

次を満たした時だけ、rule / renderer / validator への反映を検討する。

- 同じ質問タイプで 2 batch 以上、または同 batch 内で 2件以上 nonanswer が出る
- buyer に誤解を与える危険が高い
- 戻し先が `service_facts / rule / validator` のどれかに明確に切れる
- 副作用が低い
- 既存 facts にあるのに返信へ出ていない

## reviewer_prompt に留める条件

- 意味は合っているが自然さが落ちる
- 文脈依存が大きい
- まだ再発証拠が弱い
- rule 化すると短い自然文まで壊す
- 監査者の目で `buyer の主質問に最初に答えているか` を見れば足りる

## やらないこと

- 質問タイプ router を大規模追加しない
- renderer を刷新しない
- rule を大量追加しない
- `一息で言う` を絶対 rule にしない
- すべての返信を `はい、` で始める rule を作らない
- 納期を固定断定しない
- 保証期間を新設しない
- handoff-25000 を外向け導線に混ぜない
- AI tone を validator 化しない
- batch 件数を急に増やさない
- FAQ 文面をそのまま長く貼る返信を増やさない
