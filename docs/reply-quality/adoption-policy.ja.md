# 返信改善の採用ポリシー（reply-only）

更新日: 2026-04-11

## 目的

- 監査で見つかった違和感を、その場の勢いで毎回 rules に昇格させない
- モグラたたき型の調整を避け、再発性のある改善だけを正本へ戻す
- `何を直すか` だけでなく、`今は保留して観察するか` を判断できるようにする

## 基本方針

- 問題を見つけたら、まず直すのではなく分類する
- 恒久反映するのは、再発性があり、戻し先が明確で、副作用が小さいものだけにする
- 1回の改善では、原則として1論点だけ動かす
- 修正後は、5件または10件のバッチで再発と副作用を見る

## 採用フロー

1. 問題を `QA分類` に落とす
2. `guidance-scoping` で適用域を切る
3. `即反映 / 観察保留 / 却下` のどれかを決める
4. 反映するなら、戻し先を1つに絞る
5. バッチで再確認する

## 判定

### 即反映

条件:
- 再発性が高い
- buyer の誤解や不信に直結する
- 副作用が小さい
- 直し方が明確

代表例:
- `phase漏れ`
- 主質問より先に価格を出す
- 別原因・追加料金の先出し
- 内部語漏れ
- 意味接続破綻
- 謝意と緊急性を同文で束ねる

### 観察保留

条件:
- 意味は通る
- 単発の違和感か、再発癖かがまだ不明
- 直すと別ケースで副作用が出る可能性がある

代表例:
- closing の細かい温度差
- 語尾の微差
- `自然` `よさそう` のような近い表現の選好
- 1ケースだけで見えた言い回しの好み

運用:
- 同種の指摘が2回以上出たら再評価する

### 却下

条件:
- 1回限りの好み
- 既存ルールと衝突する
- 実務ノイズの方が増える
- 相手文や route が特殊すぎて一般化に向かない

## 戻し先の原則

### router / schema

使う:
- 主質問ズレ
- forbidden_moves の付与漏れ
- ask の出しすぎ
- state 判定のズレ

原則:
- 骨格の問題を prose 側だけで直さない

### policy / service facts

使う:
- prequote の約束範囲
- 公開/非公開 service の境界
- 価格や scope の説明

原則:
- ルールの出典が service や platform にあるなら、先にそちらへ戻す

### naturalizer

使う:
- 冒頭構文
- 初回対外文の距離感
- 口語すぎる表現
- 謝意、受け止め、closing の自然化

原則:
- 意味を変えずに直せる日本語問題は、まずここに戻す

### NG表現

使う:
- 明らかに再発しやすい悪い言い回し
- 一致または近似で拾いやすい表現

原則:
- 構文ルールの代わりに何でも NG へ入れない

### gold replies

使う:
- 良い温度感の参考例
- 近い場面で再利用したい骨格

原則:
- 旧ルールと競合する例は残さない

## 一度に動かす量

- 原則として1回の反映で1論点
- 同時に複数動かすのは、次の場合だけ
  - 同じ root cause で、同じ層に収まる
  - 片方だけ入れると意味が崩れる

避ける:
- 1つのケースを見て、router / policy / naturalizer / gold を一気に全部いじること

## 昇格条件

恒久反映してよい条件:
- 2回以上の再発がある
- 1回でも高リスク事故につながる
- 直し先が説明できる
- バッチで副作用が出ていない

## このポリシーで止めたいこと

- 指摘が出るたびに rules を増やす
- 気になる言い回しを全部 NG へ入れる
- 骨格問題を prose の言い換えだけで隠す
- 逆に、日本語の距離感問題を router で解こうとする

## 実務メモ

- あなたの単発指摘は貴重だが、恒久反映は別判定にする
- 人間レビューは `温度感 / 商談感 / 変な距離感` の発見に強い
- Codex 側は `分類 / 戻し先 / 副作用判定` に強い
- したがって、発見は広く受け、反映は狭く行う

## 固定したい運用

- `見つけたら直す` ではなく、`見つけたら分類して、昇格条件を満たしたら直す`
- まず `QA分類`
- 次に `適用域`
- その次に `採用判定`
- 最後に `戻し先`

この順番を守ることで、改善は積み上がるがノイズは増えにくくなる。
