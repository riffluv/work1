# 外部調査観察メモ（2026-04-25）

## 位置づけ

このメモは Deep Research の外部調査結果を、`#RE` で使える観察候補へ変換したもの。

生成ルールではない。外部調査結果をそのまま `rule / renderer / validator` へ入れない。
次の Codex は、ここを `#RE` の batch archetype と reviewer 観点の材料として読む。

## 結論

大きな再設計は不要。

現行の骨格を維持し、`#RE -> reviewer監査 -> 再発だけ最小 rule へ戻す` の運用を続ける。
外部調査の価値は、骨格変更ではなく、見落としやすい相談パターンを stock / batch に足すことにある。

## #RE に追加する観察候補

- `主質問直答の空白`
  - 最初の1〜2文で、buyer の主質問に答えているか。
- `一息回答の分断`
  - 可否・価格・範囲・次方針を、本来ひと続きで返す場面で不自然に分けていないか。
- `仕様質問のテンプレ流し`
  - 価格、納期、保証、1件定義、原因不明時の扱いを聞かれているのに、不具合ヒアリングへ逃げていないか。
- `保留だけ返して終わる弱さ`
  - `確認します` だけで終わらず、今言えること・返答時期・必要な追加入力の最小単位があるか。
- `既出情報の聞き直し`
  - buyer がすでに書いた症状・環境・不安を聞き直していないか。
- `内部語・一般論化・同義反復`
  - `切り分け` `整理して返す` `対応範囲として見る` などが、buyer 向けの結果語へ変換されているか。
- `事実上書き・仮説先走り`
  - buyer が書いていない原因、固有名詞、環境名、既知障害らしき断定を持ち込んでいないか。
- `安全境界の返し方`
  - 秘密情報、外部連絡、外部決済、本番環境、直接操作、GitHub共有に対して、禁止だけでなく安全な代替共有方法まで返せているか。

## 次の batch への使い方

### bugfix-05-boundary-secure-share

優先して見る。

- `.env と APIキーを送ればよいですか`
- `GitHubに招待した方が早いですか`
- `本番環境に入って直してもらえますか`
- `Vercel / Stripe の権限を渡してよいですか`
- `通話で説明できますか`
- `ZIPだと何を入れればよいですか`

見る観点:

- 禁止事項だけで冷たく終わっていないか
- `値は不要 / キー名のみ / ZIP / 本番反映は依頼者側 / テキストで代替` が自然に出ているか
- handoff-25000 の名前・価格・導線が外向けに混ざっていないか
- ココナラ外の連絡・決済・共有へ誘導していないか

### bugfix-06-poststate-aftercare

次点で見る。

- `進捗どうですか`
- `まだ直っていないように見えます`
- `納品後に同じエラーが出ました`
- `別のページでも似たエラーが出ています`
- `このログも追加で見てもらえますか`
- `原因が分からない場合は納品になりますか`

見る観点:

- 購入後 / 納品後なのに prequote へ戻っていないか
- `確認します` だけでなく、現在地・次に見る点・返答時期があるか
- 同一原因 / 別原因 / 追加相談の境界を勝手に断定していないか

## 反映判断

すぐ `rule` に戻してよいのは、次の条件を満たすものだけ。

- 同型再発がある
- buyer に誤解や不利益が出る
- 既存 service facts に答えがある
- 戻し先が明確
- 副作用が小さい

`reviewer_prompt` または `learning-log` に留めるもの:

- 自然さや語感の問題
- 文脈依存が大きいもの
- 1回限りの好み差
- rule 化すると短い自然文まで壊すもの

## やらないこと

- 新しい大規模 router を足さない
- renderer をケース別に増やさない
- `AIっぽさ` を hard-fail validator にしない
- 単語単位の NG リストを増やしすぎない
- `一息で言う` を絶対 rule にしない
- すべてを Yes/No で始める rule を作らない
- 返信を常に短くする rule を作らない
- 外部調査の一般論を service facts より上位に置かない
- handoff-25000 の外向け導線を混ぜない

## 次回 Codex へのメモ

Deep Research の調査結果は、現行骨格を補強している。

読むべき結論は、`大きな再設計は不要`、`#RE の観察候補を増やす`、`再発だけ最小 rule に戻す` の3点。
外部調査は、設計変更の理由ではなく、次にどの stock / archetype を回すかの材料として扱う。
