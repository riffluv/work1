# self-check L2 最小 Domain Check

## 目的
- `/home/hr-hm/Project/work/docs/coconala-reply-self-check.ja.md` から、サービス固有知識を使わずに成立する `L2: Domain` だけを抜く
- `L1` の会話品質と `L3` の service-pack validator の間にある、受託返信としての運び方を固定する
- ココナラ実務を前提にしつつ、別サービスへ移しても再利用できる「返信運用の型」を残す

## このチェックが見ないもの
- 固定価格の中身
- どの hard no を持つか
- bugfix / handoff の route 固有 must_say
- 公開 / 非公開サービスの具体名
- 納品物の具体名

上記は `L3` なので service-pack で持つ。

## このチェックが見るもの
- phase に合った進み方か
- buyer の判断サイクルに合っているか
- 断る / 保留 / 継続の返し方が会話として自然か
- 進捗共有や再発対応が停滞しないか
- delivered / closed で防御や言い訳に寄っていないか

## 最小 15 項目

### 1. prequote で重い ask に進みすぎていないか
- 見積り相談なのに、購入後レベルの解析依頼や開始報告になっていないか
- ask は「見積り判断や可否判断に必要な最小確認」に止まっているか

### 2. quote_sent で prequote に戻っていないか
- 結論に答えた後、generic ask や受付文面へ戻っていないか
- 今の見積り前提で進めてよいか、追加相談が必要かを先に返せているか

### 3. purchased / delivered / closed で受付窓口に戻っていないか
- 購入後なのに `対応できるか判断します` のような受付文面へ戻っていないか
- 現在見ている対象、未確定点、次に返すことが見えるか

### 4. buyer の判断サイクルを前に進めているか
- buyer がまだ判断段階なら、判断に必要な結論や材料を先に返しているか
- buyer がすでに購入済みなら、進行・確認・分岐の返しに寄せているか

### 5. 保留の出し方が自然か
- `確認します` で止めず、何を確認して何を返すかまで見えるか
- 同じ保留表現を続けていないか

### 6. 断る時に講義調や窓口閉じになっていないか
- 不可の理由だけで終わらず、代替の進め方か次の動きが見えるか
- `その代わり` のような不自然な補償語に逃げていないか

### 7. 進捗共有が停滞感を出していないか
- `今見ている点` `次に確認する点` `必要ならこちらから依頼する` のどれかが入っているか
- buyer が待つだけでよいのか、何か出す必要があるのかが見えるか

### 8. delivered / closed で防御や言い訳に寄っていないか
- 価値不満や軽いクレームに、まず改善可能な事実と次の扱いを返せているか
- 品質フィードバックを `承知しました` だけで流していないか

### 9. 再発 / 別件 / 追加症状の行き先を見せているか
- `同じ原因なら続ける / 別なら相談` だけで曖昧に止めず、どのタイミングで判断して返すかが見えるか
- purchased / delivered / closed で secondary thread を宙に浮かせていないか

### 10. 回答の強さと buyer の温度が合っているか
- angry / anxious / mixed でも、共感過多や冷たさに振れていないか
- 2通目以降も感情応答を繰り返して、要件が薄くなっていないか

### 11. route に合った書き出しと closing になっているか
- route や phase に合わない書き出しへ戻っていないか
- closing が毎回同じ定型で固定されていないか
- 時刻コミットや進行コミットを、固定フッターのように機械適用していないか

### 12. 高確度案件で無駄な情報回収に戻っていないか
- 高確度・単一質問なのに、習慣的な情報回収へ戻っていないか
- すでに相手が済ませた適合説明や進め方説明を、後から足して長くしていないか
- prequote で `まずは次の3点だけ` のような進行中テンプレへ滑っていないか

### 13. service mismatch やお断り時の閉じ方が自然か
- `対象外です` だけで冷たく閉じていないか
- 必要なら再相談の余地や、次の行き先を1文で示せているか
- お断り後に buyer の同意なしで ask へ直行していないか

### 14. urgency と進行感の返し方が自然か
- `急ぎ` `至急` を、説明の中へ埋めず直接受けているか
- 急ぎや不安が強い場面で、安心材料として価格を先出ししすぎていないか
- 価格より先に、止まっている症状・今返せること・次に返すことを置けているか

### 15. buyer が持ち込んだ事象を主語にできているか
- 追加報告や別件を、こちらの反応ではなく buyer が持ち込んだ事象を主語にして返せているか
- `前回に続いてのご相談` のような分類ラベルより、自然な受領文へ落とせているか
- `この不具合として` のような指示語受けに戻っていないか

## send / revise / escalate

### send
- 10項目で major fail なし

### revise
- 文面修正で直せる phase / tone / hold のズレがある

### escalate-to-system
- 同型の phase ズレや hold 停滞が再発した
- Router / state / reviewer へ戻すべきと切れる

## major fail
- prequote で購入後 ask へ進みすぎる
- quote_sent で prequote に戻る
- purchased / delivered / closed で受付文面に戻る
- 保留が不自然に長引く
- delivered / closed で防御や言い訳が前面に出る

## minor fail
- opening / closing の単調さ
- 進捗共有の粒度不足
- 断り方の抽象さ
- buyer の判断段階への寄り不足
- 高確度案件での過剰確認
- urgency への寄り不足
- route に合わない書き出し

## 使い方
- `L1` を通した後に、phase と進行感を見る第2 gate として使う
- `L3` の service-pack validator を見る前に、会話運びとして破綻していないかを確認する
- 新サービス導入時でも、価格や scope をまだ固める前にこの `L2` はそのまま使える

## 位置づけ
- `L1`
  - universal な会話品質
- `L2`
  - 受託返信としての進め方
- `L3`
  - サービス固有知識と境界

## 運用上の目安
- 返信監査は
  1. `L1`
  2. `L2`
  3. `L3`
  の順で見る
- `L1` と `L2` を通って `L3` で落ちるなら、配置の問題
- `L1` か `L2` で落ちるなら、service-pack ではなく runtime の問題
