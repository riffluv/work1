# Pro Analysis Queue 2026-04-30

## 位置づけ

ChatGPT Pro に後でまとめて相談する候補を保持する一時キュー。

すぐに Pro へ投げるためのプロンプトではない。`#RE` / `#BR` / human audit で実例を数件ため、再発性と副作用を見てから、まとめて分析依頼へ変換する。

## 現在の優先候補

### 1. `promise_consistency`

返信内で、前段の留保・不可・条件付き回答を、後段の成果物・納期・次アクションが上書きして見えないか。

代表例:

```text
確実に直ることまではお約束できません。
ただ、原因確認から修正済みファイルの返却まで進められます。
```

現状の方針:

- `promise-consistency-lens-20260429.ja.md` と Gold 37 で運用中。
- すぐ hard validator 化しない。
- `#RE` で数 batch 回し、どの subtype が再発するかを見る。

Pro に聞きたいこと:

- 正式 lens としてどこまで固定すべきか。
- `fix recommended` と `preference` の境界。
- lint / reviewer prompt / gold / skill のどこへ戻すべきか。

### 2. `conditional_scope_clarity`

「同一原因なら範囲内」「別原因なら相談」「修正できる箇所が特定できた場合は返却」のような条件付き scope を、buyer が誤解しない形で出せているか。

現状の方針:

- `promise_consistency` と近いが、成果物 promise より scope 条件の明瞭さに寄る。
- 補助 lens として観察し、単独正式化はまだ保留。

Pro に聞きたいこと:

- `promise_consistency` の subtype に含めるべきか、別 lens として分けるべきか。
- bugfix-15000 の公開返信で、条件をどこまで短く出すのが自然か。

### 3. `ack_to_answer_bridge`

相手の不安・状況を受けたあと、回答本体へ急に飛ばず、自然な橋を置けているか。

代表例:

```text
別の方でも原因が分からなかったとのことなので、まずは必ず直せるかが気になる状況だと思います。
コードとエラー内容を見る前に、必ず直せるとはお約束できません。
```

違和感:

- 内容は正しいが、受け止め文から制限回答へ少し急に飛ぶ。
- 内部パーツが外に出たように見える場合がある。

現状の方針:

- hard fail ではない。
- `conversation_flow_naturalness` の subpoint として観察する。
- 無理に接続語を増やして冗長にしない。

Pro に聞きたいこと:

- 日本語ビジネスチャットで、どの程度の橋渡しが必要か。
- `ack_to_answer_bridge` を独立 lens にする価値があるか。
- 何を gold 化し、何を好みに留めるべきか。

### 4. `permission_benefit_alignment`

`大丈夫です` などの許可・安心表現が、相手の負担を軽くしているのか、売り手側が相手の支払い・作業を許可しているように見えないか。

代表例:

```text
ログや関連コードは、お支払い完了後で大丈夫です。
```

違和感:

- 支払い後にログを送るのは buyer 側の負担であり、seller 側が「それで大丈夫」と許可しているように見える。
- 支払い前に送るか、支払い後に送るかだけの違いで、buyer の実務負担が減っていない場面では、`大丈夫です` が幼く・軽く見えやすい。
- `大丈夫です` 自体は便利で自然な場面も多いので、blanket ban はしない。

自然な使い方:

```text
値そのものは送らなくて大丈夫です。
原因が分からない状態でも相談できます。
```

置き換え候補:

```text
ログや関連コードは、お支払い完了後にトークルームで共有してください。
ログや関連コードは、お支払い完了後に共有いただければ、そこから確認に入ります。
スクショやログは、お支払い完了後にトークルームで共有いただければ進められます。
```

現状の方針:

- `jp_business_native_naturalness` の subpoint として観察する。
- 支払い・購入・相手に作業してもらう場面の `大丈夫です` は要注意。
- 相手の実務負担または心理負担を実際に減らす `大丈夫です` は残してよい。
- 単なる手順案内・材料送付タイミング・支払い導線では、`大丈夫です` より具体的な手順文を優先する。

Pro に聞きたいこと:

- この観点を正式 lens にする価値があるか。
- `大丈夫です` の hard rule / soft lens / lint 候補の線引き。
- 支払い導線で自然な標準表現。

### 5. `unnecessary_refusal_frame`

協力的な buyer に対して、必要以上に否定から入っていないか。

代表例:

```text
支払い前にスクショを送っていただいて、こちらで確認を先に始める形ではありません。
```

違和感:

- buyer は強い要求ではなく、単に「先に送るべきか」を確認しているだけの場合がある。
- その場面で `〜する形ではありません` や `先に確認はしません` から入ると、正しいが冷たく、防御的に読まれやすい。
- phase boundary は必要だが、外向けには拒否より手順を先に置いた方が自然な場面がある。

置き換え候補:

```text
スクショやログは、お支払い完了後にトークルームで共有いただければ進められます。
今の段階で先に送っていただく必要はありません。
```

現状の方針:

- `conversation_flow_naturalness` / `jp_business_native_naturalness` の subpoint として観察する。
- 強い要求・規約違反・secret値・外部誘導などは明確に止める。
- 協力的な材料共有タイミングの確認では、否定より手順案内を優先する。
- phase / payment route を弱めるためには使わない。

Pro に聞きたいこと:

- `permission_benefit_alignment` と同じ lens に含めるべきか、別 lens として分けるべきか。
- hard rule / soft lens / gold 候補の線引き。
- 支払い前材料共有の標準文として、どの表現が最も安全で自然か。

### 6. `agency_alignment`

返信文の中で、誰が何をするのか、誰に向けて許可・依頼・案内しているのかがズレていないか。

代表例:

```text
他の方で原因が分からなかった状態でも、Stripe決済後に注文が作られない不具合として15,000円で相談できます。
```

違和感:

- buyer の主質問は `依頼できますか？` なのに、返信が `相談できます` だと、返信者本人が受けるのではなく第三者へ相談できるようにも読める。
- 文法としては成立するが、seller / buyer / 第三者の行為主体が曖昧になり、`いや、相談する相手はあなたでは？` という日本語ネイティブの違和感が出る。
- `確認できます` `進められます` `相談できます` `大丈夫です` のような主語が省略されやすい表現で、対応主体・利益の向き・依頼先がずれることがある。

修正例:

```text
他の方で原因が分からなかった状態でも、Stripe決済後に注文が作られない不具合として、15,000円でご依頼いただけます。
```

現状の方針:

- hard fail ではなく soft lens として観察する。
- ただし、料金・作業開始・納品・返金・対応可否など、取引判断に関わる箇所で主体がズレる場合は必須修正寄りに扱う。
- `相談できます` を blanket NG にはしない。buyer が「相談してよいか」を聞いている場合は自然に使える。
- buyer の主質問が `依頼できますか` `対応できますか` `見てもらえますか` の場合、返答は `ご依頼いただけます` `対応できます` `確認できます` のように、行為主体と依頼先がずれない表現へ寄せる。

Pro に聞きたいこと:

- `agency_alignment` を独立 lens にする価値があるか。
- `permission_benefit_alignment` と同じ上位カテゴリに含めるべきか、別 lens として分けるべきか。
- hard fail / fix recommended / preference の線引き。
- lint 化できる表面パターンがあるか、それとも gold / reviewer prompt / human audit に留めるべきか。
- 日本語ビジネスチャットで、行為主体ズレが信頼感に与える影響。

### 7. `material_selection_burden`

buyer が「コードが分からない」「AIで作った」「どのファイルを送ればいいか分からない」と言っているのに、関係ファイルの選別や原因候補判断を buyer 側へ戻していないか。

代表例:

```text
どのファイルが関係するか迷う場合は、分かる範囲で送っていただければ、こちらで必要な範囲を見ながら不足分をお伝えします。
```

違和感:

- ガチ素人の buyer は `分かる範囲` 自体が分からない。
- `関係しそうなファイル` の選別を依頼すると、相談した理由そのものを buyer に返しているように読める。
- このサービスでは、コードが分からない / AIコード / StripeやWebhookやDBの関係が分からない buyer が多いため、材料選別負担の戻し方は実務品質に直結する。

修正例:

```text
ご購入後は、エラーログや画面スクショに加えて、可能であればコード一式をZIPでトークルームに送ってください。
どのファイルが関係するか分からなくても、こちらで必要な範囲を見ながら確認します。

APIキーやWebhook Secret、.envの値そのものは送らなくて大丈夫です。
コード一式に含まれそうな場合は、該当部分を伏せるか、.envファイルを外した状態で送ってください。
```

現状の方針:

- `buyer_burden_alignment` の subtype として観察する。
- 非エンジニア / AIコード / ファイル不明が明示されている時は、ファイル選別を buyer に返さず、コード一式 ZIP + secret 除外へ寄せる。
- エンジニア寄り buyer や、関係ファイルが明確な場面では、毎回コード一式 ZIP を要求しない。
- ZIP一式に寄せる時は、secret 値除外を必ず近くに置く。

Pro に聞きたいこと:

- `buyer_burden_alignment` の subtype で十分か、独立 lens 化すべきか。
- 「コード一式 ZIP」を出す条件と、関係ファイル指定に留める条件。
- secret safety と材料選別負担軽減を、どの順序・文量で出すのが自然か。

### 8. `fee_anxiety_intensity` / `unresolved_outcome_clarity`

料金・未解決時・返金/キャンセル不安の強さに応じて、説明量を変えられているか。

代表例:

```text
もし原因や修正方針につながらない状態だと分かった場合は、
原因不明のまま一方的に正式納品へ進めることはありません。
その時点で分かった範囲、不足している情報、今後の進め方を整理してご相談します。
```

違和感:

- 安全ではあるが、buyer が本当に知りたい `結局料金はどうなるのか` への直答が弱く見える。
- 逆に、毎回 `勝手に追加料金を発生させません` `ココナラ上の手続きに沿って相談します` まで出すと、軽い相談でも契約説明っぽくなり、返信が重くなる。
- ココナラでは正式納品・承諾/差し戻し・自動承諾・キャンセル/返金手続きが絡むため、seller が `返金になります` `料金はかかりません` `承諾しなければ成立しません` のように断定すると危険。

出し分け候補:

```text
軽い確認:
原因や修正方針につながらない場合は、その時点で状況を説明して、次の進め方をご相談します。

未解決時を明示している:
もし原因や修正方針につながらない状態だと分かった場合は、そこで止めて、分かった範囲と不足している情報をご説明します。
勝手に追加料金を発生させたり、原因不明のまま正式納品へ進めたりはしません。

返金・キャンセル・追加料金を強く気にしている:
その後の進め方は、作業状況に応じてココナラ上の手続きに沿ってご相談します。
```

公式仕様確認後の補足:

- `ココナラ上の手続きに沿ってご相談します` は曖昧で、運営に相談する意味にも読まれる。
- 原因不明・修正不可が実際に見えた場合の内部記憶は、`トークルームで分かった範囲・不足情報・修正できない理由を説明し、続けて確認するか、取引をキャンセル方向にするかを購入者と話し合う` が正確。
- キャンセルする場合は、原則として出品者からトークルーム上でキャンセルリクエストを送信し、buyer の合意または一定時間内の差し戻しなしでキャンセル成立、返金処理へ進む。
- ただし購入前の外向け返信で `キャンセルできます` `返金されます` を前面に出すと、まだ依頼前なのに終了導線を強く見せ、商売上もノイズになりやすい。
- 標準文型は `原因不明のまま修正完了扱いにはせず、トークルームで状況をご説明して進め方をご相談します` 程度がよさそう。

現状の方針:

- hard rule / lint にはまだしない。
- `料金不安がある = 毎回厚く説明` ではなく、buyer の不安強度に応じて説明量を調整する。
- `返金` `キャンセル` `無料` `追加料金` `原因不明の場合` が明示された時だけ、料金・正式納品・ココナラ手続きへの説明を一段厚くする。
- 軽い `原因不明でも見てもらえますか` では、短い保留説明に留める。

Pro に聞きたいこと:

- `fee_anxiety_intensity` を独立 lens にする価値があるか、`promise_consistency` / `conditional_scope_clarity` の subtype に留めるべきか。
- ココナラの正式納品・承諾/差し戻し・キャンセル/返金手続きを、外向け返信でどの深さまで触れるべきか。
- `勝手に追加料金を発生させない` `原因不明のまま正式納品へ進めない` `トークルームで状況説明して進め方を相談` の3要素を、どの不安強度で出すべきか。
- `キャンセルリクエスト` や `返金` を購入前返信に出すべき条件はどこか。明示された時だけか、修正不可時の質問が強い時だけか。
- 料金不安への回答で、buyer に安心感を出しつつ、返金/無料/キャンセル保証に読ませない標準文型。

### 9. `purchase_cta_strength_calibration`

購入前・見積り前・見積り提案後の返信で、購入導線の強さが buyer の温度や迷いに合っているか。

代表例:

```text
この内容で進める場合は、そのままご購入ください。
```

違和感:

- buyer が `支払い前に少し見てもらえますか` `直りそうなら購入したいです` のように、まだ迷いや条件付き購入を出している場面で、`そのままご購入ください` と閉じると押し込み感が出る。
- 意味としては購入導線だが、ココナラの見積り相談では buyer の不安を整理してから購入判断へ移る温度感が重要。
- `購入してください` を強く出しすぎると、事故ではないが seller 側が急かしている印象になりやすい。

置き換え候補:

```text
購入後に原因確認から進める流れで問題なければ、ご購入ください。

購入後に原因確認から進める形でよければ、そのままご購入いただけます。

購入前にできる範囲は、対応可否と進め方のご案内までになります。
購入後に原因確認から進める形でよければ、ご購入ください。
```

現状の方針:

- hard rule / lint にはしない。
- buyer が購入意思を明確に出している場合は、`この内容で問題なければ、そのままご購入ください` は許容。
- buyer が迷い・条件付き購入・支払い前確認を出している場合は、CTA を一段弱め、`購入後に〜する流れでよければ` のように条件と進め方を先に置く。
- 購入導線を弱めすぎて、主質問への回答や次アクションが消えないようにする。

Pro に聞きたいこと:

- ココナラ見積り相談で、購入導線の温度感をどう設計すべきか。
- `ご購入ください` / `ご購入いただけます` / `購入後に〜する形でよければ` の使い分け。
- 事故らず、かつ印象が良い購入前・見積り前・quote_sent の標準文型。
- `sales pressure` として soft lens 化すべきか、`positive_flow_before_refusal` / `phase_contract` の subtype に留めるべきか。

### 10. `purchase_before_materials_order`

購入前 / 見積り提案後で buyer が迷い・条件付き購入を出している場面で、購入後の材料案内を出す順番が自然か。

代表例:

```text
お支払い完了後は、GitHub招待ではなく、関係ファイルやログをZIPまたは添付でトークルームに共有してください。
.env、APIキー、Webhook Secret、DB接続文字列などの値は含めないでください。
購入後に原因確認から進める形でよければ、ご購入いただけます。
```

違和感:

- buyer はまだ `直りそうなら購入したい` と条件付きで迷っている。
- その段階で購入後の細かい材料共有を先に出すと、まだ購入していない相手に `購入後の指示` をしているように見える。
- 内容は正しくても、商談の順序として `買う前提` に見えやすい。

望ましい順序:

```text
1. 購入前に先見・原因判断はしない
2. ただし症状自体は対応範囲
3. この前提でよければ支払いへ
4. 支払い後の共有方法を案内
5. secret 値は含めない
```

修正例:

```text
購入前にGitHubを確認して、直りそうかどうかや原因の切り分けだけを先に判断することはしていません。
その判断は、お支払い完了後にログやコードを確認してから行う内容になります。

ただ、本番でだけ起きていてローカルでは再現しない症状でも、Next.js / Stripeまわりの不具合として対応できます。

この前提で問題なければ、見積り提案の内容でお支払いへ進んでください。
お支払い完了後は、GitHub招待ではなく、関係ファイルやログをZIPまたは添付でトークルームに共有してください。

.env、APIキー、Webhook Secret、DB接続文字列などの値は含めないでください。
```

現状の方針:

- すぐ hard rule にはしない。
- `purchase_cta_strength_calibration` とセットで観察する。
- buyer がすでに明確に購入意思を出している場合は、購入後材料案内まで1通に入れてよい。
- buyer が迷い・条件付き購入・支払い前確認を出している場合は、購入判断の前提と CTA を先に置き、材料共有はその後に置く。
- 1ラリーで親切に全部案内しようとして、購入前の相手に購入後の指示を先出ししない。

Pro に聞きたいこと:

- `purchase_before_materials_order` は `purchase_cta_strength_calibration` の subtype で十分か、独立観察軸にすべきか。
- ココナラの quote_sent / prequote で、購入後材料案内をどこまで1通に含めるべきか。
- `買う前提` に見えない順序と、1ラリーで判断材料を十分出す順序のバランス。
- GitHub招待・支払い前診断・secret 除外のような複数 boundary が同時にある時の標準順序。

### 11. `deictic_agency_alignment`

`こちら` / `そちら` / `ご自身` / 主語省略の使い分けで、buyer 側の作業と seller 側の作業が逆転していないか。

代表例:

```text
ご購入後に送るファイルは、無理にこちらで選び切らなくて大丈夫です。
```

違和感:

- `こちら` は seller 側を指すため、buyer が送るファイルを `こちらで選び切らなくて` と書くと主体が逆転する。
- 本来は `buyer がファイルを無理に絞らなくてよい`、`seller が必要範囲を見る` という2つの役割がある。
- `material_selection_burden` の自然化で、`選び切る必要はありません` を避けようとした結果、指示語が入り主体ズレが発生した可能性がある。
- 普通に freeform で書けば起きにくいが、guard / naturalization の言い換えが入ると起こり得る副作用として扱う。

修正例:

```text
ご購入後に送るファイルは、無理に絞らなくて大丈夫です。
どこが関係しているか分からない場合は、スクショに加えて、可能であればコード一式をZIPで共有してください。

ファイルは無理に絞らなくて大丈夫です。
必要な範囲はこちらで確認します。
```

現状の方針:

- `agency_alignment` の subtype として扱う。
- `こちら` は seller 側、`そちら` は buyer 側だが、`そちら` は距離が出るため多用しない。
- buyer の作業は、できれば `無理に絞らなくて大丈夫です` のように主語を消す。
- seller の作業は、`こちらで確認します` のように明示してよい。
- buyer の作業と seller の作業が混ざる時は、1文に詰めず文を分ける。

Pro に聞きたいこと:

- `deictic_agency_alignment` を `agency_alignment` の明示 subtype として正式化すべきか。
- 日本語ビジネスチャットで `こちら` / `そちら` / 主語省略をどう安全に扱うべきか。
- material selection / buyer burden naturalization と衝突しない標準文型。
- 指示語ズレを lint で拾えるか、それとも reviewer prompt / gold に留めるべきか。

### 12. `overhelpful_interpretation`

相手の軽い不安・難しさ・迷いを、こちらが勝手に原因確定して、過剰に親切な言い方へ変換していないか。

代表例:

```text
確認手順も、専門用語を減らして見やすい形にします。
```

違和感:

- buyer は `確認手順も少し難しく感じています` と言っただけで、`専門用語が多い` `専門用語をなくしてほしい` とは言っていない。
- seller 側が `専門用語を減らす` と言い切ると、相手の理解不足をこちらが勝手に認定したように見える。
- 日本語では、よかれと思った `分かりやすくします` `かみ砕きます` `簡単にします` が、文脈によっては嫌味・皮肉・見下しに読まれることがある。
- 相手が弱みを見せている時ほど、親切のつもりの表現が攻撃的に見えないように注意が必要。

修正例:

```text
確認手順も、見る場所と順番が分かる形にして補足します。

反映するファイルと、その後に確認する画面を分けてお伝えします。
```

現状の方針:

- hard rule / lint にはしない。
- `相手が理解できていない理由` を勝手に決めない。
- `専門用語を減らす` `かみ砕く` `簡単にする` のような相手評価に近い言い方より、`見る場所と順番` `確認する箇所` `反映するファイル` のように、作業対象へ寄せる。
- 相手が明示的に `専門用語が分からない` `初心者向けに言い換えてほしい` と言っている場合は、専門用語を減らす方針を出してよい。

Pro に聞きたいこと:

- `overhelpful_interpretation` を `business_native_naturalness` の subtype として正式観察すべきか。
- 日本語ビジネスチャットで、親切が皮肉・見下しに読まれる典型パターン。
- `分かりやすくします` `専門用語を減らします` `かみ砕きます` の安全な使い方と避けるべき場面。
- 相手の不安を尊重しつつ、相手の能力を勝手に評価しない標準文型。

### 13. `relation_check_scope_clarity`

closed 後や納品後の再相談で、`関係確認` と `実作業 / コード修正 / 修正済みファイル返却` の違いが buyer に分かる形で書けているか。

代表例:

```text
届いた範囲で、前回の修正と関係がありそうかを確認します。

前回のトークルームは閉じているため、この場で修正作業や修正済みファイルの返却までは進めません。
```

違和感:

- buyer から見ると、`確認してくれるのに修正作業やファイル返却はできないなら、確認とは何をしてくれるのか` が曖昧になる。
- closed 後の安全境界として `実作業不可` を入れること自体は正しいが、`確認` の中身が見立てなのか、コードを開く作業なのか、修正作業なのかが曖昧だと不信感が出る。
- 近年の安全境界強化で、`関係確認` と `実作業不可` が部品として並び、接続が弱くなる副作用がある。

修正例:

```text
ログとスクショは、このメッセージで送っていただいて大丈夫です。
届いた内容をもとに、前回の修正と関係がありそうか、新しい不具合の可能性が高そうかを短く確認します。

ここでできるのは、ログとスクショから見える範囲での関係確認までです。
コード修正や修正済みファイルの返却が必要になりそうな場合は、前回のトークルームは閉じているため、対応方法と費用を先にご相談します。
```

現状の方針:

- hard rule / lint にはまだしない。
- closed 後の `確認` は、外向けには `ログとスクショから見える範囲での関係確認 / 見立て` と明確にする。
- `実作業` は、コード修正・修正済みファイル返却・新しい納品物作成などと分ける。
- `確認できます` だけで終えず、何を確認するのか、どこまでがこのメッセージ上で可能なのかを短く添える。
- 安全境界を入れる時も、buyer の主質問 `関係あるかだけ見てもらえるか` への答えを消さない。

Pro に聞きたいこと:

- `relation_check_scope_clarity` を closed 後専用 lens として正式化すべきか。
- `見立てとしての確認` と `作業としての確認` を、日本語ビジネスチャットでどう分けるのが自然か。
- closed 後にメッセージ上で受けてよい確認範囲と、新規依頼/見積りへ戻すべき実作業の標準文型。
- 安全境界を説明しつつ、buyer に `結局何をしてくれるのか` と感じさせない文の順序。

### 14. `acceptance_redundancy`

依頼可否・対応可否をすでに答えたあとに、`問題ありません` / `大丈夫です` などの許可・安心語を重ねて、同じ意味を二重に言っていないか。

代表例:

```text
15,000円でご依頼いただけます。
原因が分からない状態でも問題ありません。
```

違和感:

- `15,000円でご依頼いただけます` の時点で、依頼可否・許容範囲はすでに答えている。
- 直後に `問題ありません` を重ねると、意味は通るが、許可・安心を二重に出して少し冗長になる。
- buyer が本当に知りたいのは「受けられるか」だけでなく、「原因不明やAI生成コードでも、こちらがどう確認するか」なので、後段は許可語より seller action へ寄せた方が自然。

修正例:

```text
15,000円でご依頼いただけます。
決済後にサイト側で購入済みにならない不具合として対応できます。

原因が分からない状態でも、確認に必要な範囲はこちらで見ます。
AIで作った部分があっても、同じように確認します。
```

または、少し短く:

```text
15,000円でご依頼いただけます。
決済後にサイト側で購入済みにならない不具合として対応できます。

原因が分からない状態でも、AIで作った部分があっても、確認に必要な範囲はこちらで見ます。
```

現状の方針:

- `問題ありません` / `大丈夫です` 自体は NG にしない。
- buyer が `問題ありませんか？` と聞いている場面や、secret 値不要・材料不足など buyer の負担を下げる場面では自然に使ってよい。
- 依頼可否を `ご依頼いただけます` / `対応できます` で答えた直後は、後段で許可語を重ねず、`こちらで何を見るか` `必要な範囲はこちらで確認する` `不足があればこちらから伝える` へ変換する。
- `agency_alignment` / `permission_benefit_alignment` / `context_anchor_granularity` と近いが、主な問題は「主体ズレ」ではなく「受けられる返答の重複」。

Pro に聞きたいこと:

- `acceptance_redundancy` を独立候補として見るべきか、`conversation_flow_naturalness` または `agency_alignment` の subtype に含めるべきか。
- `ご依頼いただけます` の後に `問題ありません` / `大丈夫です` を重ねても自然な条件と、不自然になる条件。
- 依頼可否回答後に、許可語ではなく seller action へ移す標準文型。
- lint ではなく reviewer prompt / gold / writer-brief に留めるべきか。

### 15. `boundary_phrase_maturity`

支払い前不可・closed後不可・範囲外など、できないことを伝える時に、buyer の口語や圧力語をそのまま貼り戻さず、成熟した業務文の境界表現に変換できているか。

代表例:

```text
支払い前にAPIファイルやログを確認して、直せそうかだけ先に判断する形では進めていません。
```

違和感:

- `直せそうかだけ先に` は buyer の口語をそのまま拾いすぎており、業務文として少し幼く見える。
- `〜する形では進めていません` は内部運用説明っぽく、相手にルールを突きつける感じが出やすい。
- 境界自体は必要だが、言い方が稚拙だと bot 感・拒否感が強くなる。

修正例:

```text
お支払い前は、ファイルやログを見て原因確認まで進めることはできません。
```

または、少し丁寧に:

```text
お支払い前の段階では、ファイルやログを拝見して個別に原因確認を行うことはしていません。
```

現状の方針:

- 支払い前診断・closed後作業・範囲外作業を曖昧にしない。
- ただし不可表明では、buyer の口語や圧力語をそのまま復唱せず、短く成熟した境界文へ変換する。
- `positive_flow_before_refusal` は順序の問題、`boundary_phrase_maturity` は不可表明の言葉の成熟度として分ける。
- hard rule / lint にはまだしない。writer-brief / naturalizer / reviewer prompt の観察候補に留める。

Pro に聞きたいこと:

- `boundary_phrase_maturity` を独立 lens として見るべきか、`conversation_flow_naturalness` / `negative_frame_non_echo` / `positive_flow_before_refusal` の subtype にすべきか。
- 支払い前不可・closed後不可・範囲外を、日本語ビジネスチャットとして成熟して伝える標準文型。
- `〜する形では進めていません` `〜だけ先に` のような表現をどこまで避けるべきか。
- 境界を弱めず、拒否感や幼さだけを減らす実装先は writer-brief / skill / gold / reviewer prompt のどこがよいか。

### 16. `opening_progression_pressure`

冒頭の進行語・条件語が、buyer の慎重さや相談温度より前に出ていないか。

代表例:

```text
この場合は、まず「決済後に注文データが作られない」不具合を1件として、15,000円でご依頼いただけます。
```

違和感:

- `この場合は` は誤りではないが、少し判定文・メール文寄りに見える。
- `まずは` は分かりやすい一方で、buyer が慎重に範囲確認をしている場面では、seller 側が進行を切っているような前のめり感が出る場合がある。
- 冒頭は返信全体の印象を決めるため、`この場合は` / `まずは` / `その前提で` / 無接続直答のどれが自然かは慎重に見たい。

修正候補:

```text
「決済後に注文データが作られない」不具合を1件として、15,000円でご依頼いただけます。
```

または、少し説明を残す場合:

```text
優先されている「決済後に注文データが作られない」不具合を1件として、15,000円でご依頼いただけます。
```

現状の方針:

- `この場合は` / `まずは` を禁止しない。場面によって自然。
- buyer が範囲・料金・追加費用を慎重に確認している場面では、進行語が seller 側の前のめりに見えないかを見る。
- 無接続で主質問に直答した方が自然な場面がある。
- ただし、無接続にしすぎて文脈が欠ける場合もあるため、まだ hard rule / skill 反映はしない。

外部調査メモ:

- 日本語の業務チャットでは、結論や次アクションを先に置くこと自体は読みやすさに寄与する。
- 一方で、短く切りすぎると冷たく見えやすく、最低限の受け止めや文脈接続は必要。
- `まずは` はビジネス語として不自然ではないが、「最初の手順」「こちらが進行を切る」ニュアンスを持つため、buyer が料金・範囲・追加費用を慎重に確認している冒頭では前のめりに見える場合がある。
- この観点は単語NGではなく、`buyer の温度` / `購入意思` / `主質問の種類` / `こちらが次に促してよい段階か` を見て判断する。

Pro に聞きたいこと:

- `opening_progression_pressure` を `commercial_temperature` の subtype として見るべきか。
- 日本語ビジネスチャットで、冒頭の `この場合は` / `まずは` / 無接続直答をどう使い分けるのが自然か。
- buyer が慎重に料金・範囲を聞いている場面で、seller の商売感・進行圧を抑えつつ、主質問へ直答する標準文型。
- この観点を writer-brief に入れるべきか、Proレビュー用の観察メモに留めるべきか。

### 17. `action_closure` / `next_action_clarity`

buyer の不安を下げる表現のあとに、具体的な次アクションが閉じているか。

代表例:

```text
どのファイルが必要か分からない場合も、無理に絞らなくて大丈夫です。
```

違和感:

- buyer の負担を下げる意図は正しいが、「では結局、何を送ればいいのか」が残る。
- `大丈夫です` / `問題ありません` / `無理に〜しなくて大丈夫です` だけで終わると、安心はあるが行動が閉じない。
- 非エンジニア buyer やファイル選定に不安がある buyer ほど、負担軽減だけではなく安全な既定入力を示す必要がある。

修正候補:

```text
お支払い完了後に、スクショ・ログとあわせて、プロジェクト一式をZIPでトークルームへ共有してください。
どのファイルが関係するか分からない場合でも、その形で大丈夫です。
```

または:

```text
関係するファイルを事前に絞り込めない場合は、プロジェクト一式をZIPで共有してください。
不要なファイルが混ざっていても、必要な範囲はこちらで確認します。
```

現状の方針:

- `大丈夫です` / `問題ありません` を禁止しない。buyer の負担軽減として自然な場面はある。
- ただし、負担軽減の表現だけで終えず、buyer が次に取る行動を具体的に閉じる。
- 特に「どのファイルが必要か分からない」「関係ファイルを選べない」「コードが分からない」場面では、safe default action を出す。
- hard rule / lint にはまだしない。`buyer_material_literacy` / `material_selection_burden` / `agency_alignment` の subtype か、独立 soft lens かを Pro に確認する。

Pro に聞きたいこと:

- `action_closure` / `next_action_clarity` を独立 lens として扱うべきか、`buyer_material_literacy` または `material_selection_burden` の subtype にすべきか。
- 日本語ビジネスチャットで、安心表現のあとに次アクションを閉じる標準文型。
- `無理に絞らなくて大丈夫です` のような負担軽減表現を残してよい条件と、具体行動まで書くべき条件。
- writer-brief / bugfix skill / gold / reviewer prompt のどこに戻すのが最もノイズになりにくいか。

### 18. `boundary_summary_overfit` / `boundary_layer_separation`

支払い前不可・GitHub不可・外部共有不可などの境界説明で、buyer の希望条件や症状を1文に抱え込みすぎて、拒否文そのものが入力要約のようになっていないか。

代表例:

```text
お支払い前は、GitHubに招待いただいてコードを確認し、原因の方向性や修正可否を判断することはできません。
ただ、本番だけで起きていてローカルでは再現しない Next.js / Stripe まわりの不具合でも、このサービスの対象として対応できます。
```

違和感:

- 1文目が `GitHub招待` / `コード確認` / `原因方向性` / `修正可否` をまとめて拒否しており、buyer 文を圧縮した拒否の塊に見える。
- 2文目の `ただ` は、前文の不可表明と後文の症状対応可否を無理につないでいる。両者は逆接ではなく、別レイヤの説明。
- `GitHub関連の境界` と `症状として対応できるか` と `購入後にどう判断するか` は、本文上で分けた方が自然。

修正候補:

```text
お支払い前は、GitHub上でコードを確認したり、原因の見立てを先に出したりする対応はしていません。

症状としては、このサービスで扱える内容です。
本番だけで起きていてローカルでは再現しない場合も、お支払い完了後にログと関係ファイルを確認しながら原因を見ます。
```

現状の方針:

- `boundary_phrase_maturity` は不可表明の言葉の成熟度を見る。
- `context_anchor_granularity` は相手文の長い症状要約を避ける。
- 今回の候補はその交差領域で、特に「不可表明の中に相手文の条件を抱え込みすぎる」問題を見る。
- GitHub / 支払い前診断 / 外部共有の境界は弱めない。ただし、境界説明と症状への対応可否を1文・1接続詞で無理につながない。
- hard rule / lint にはまだしない。Pro に、既存 lens の subtype で足りるか確認する。

Pro に聞きたいこと:

- `boundary_summary_overfit` / `boundary_layer_separation` を独立候補として見るべきか、`boundary_phrase_maturity` と `context_anchor_granularity` の subtype に留めるべきか。
- 支払い前診断不可・GitHub作業面不可・外部共有不可を、相手文の条件を抱え込みすぎずに伝える標準文型。
- `ただ` `一方で` などで境界説明と対応可否をつなぐべき場面と、段落を分けるべき場面。
- writer-brief / bugfix skill / gold / reviewer prompt のどこへ戻すと、境界を弱めずに bot 感だけを減らせるか。

### 19. `context_anchor_over-reconstruction`

冒頭では改善されていても、回答本文の中で buyer の症状を再構成しすぎて、入力を分析して整形したように見えていないか。

代表例:

```text
決済後に完了画面へ戻るのにDBへ注文が作られない不具合として、原因確認から進めます。
```

違和感:

- 事実としては正しいが、buyer がすでに説明した症状を `完了画面へ戻るのにDBへ注文が作られない` と再構成しており、少し分析文・bot 文に見える。
- `context_anchor_granularity` は冒頭要約だけでなく、本文中の answer-bearing section にも効かせる必要があるかもしれない。
- 症状を全部拾わなくても、今回の回答に必要な核は `注文データが作られない` で足りる場合がある。

修正候補:

```text
ご購入後は、まず注文データが作られない原因を確認します。
修正できる箇所が特定できた場合は、修正済みファイルと確認手順をお返しします。
```

または:

```text
ご購入後は、注文データが作られない原因を中心に確認します。
修正できる箇所が特定できた場合は、修正済みファイルと確認手順をお返しします。
```

現状の方針:

- 症状をまったく拾わない返信にはしない。
- ただし、buyer がすでに説明した技術症状を、本文内で毎回きれいに組み直さない。
- 主質問への答えに必要な核だけ拾い、すぐ seller action へ進む。
- `context_anchor_granularity` の subtype として扱うのが自然そうだが、Pro に確認する。

Pro に聞きたいこと:

- `context_anchor_over-reconstruction` を `context_anchor_granularity` の subtype として扱えば足りるか。
- 冒頭ではなく回答本文中で、症状をどこまで再構成してよいか。
- `症状として対応できます` と `原因を確認します` の間に、どれくらい症状の核を残すのが自然か。

### 20. `core_adapter_style_boundary`

ココナラ向けの自然化・導線調整を、将来 SaaS 化する汎用返信コアへどこまで入れるべきか。

背景:

- `別原因と分かった場合は、進める前に対応方法と費用を先にご相談します。` は汎用的には安全だが、ココナラの見積り相談では少し重く見える場面がある。
- ココナラでは `別原因と分かった場合は、その時点で追加対応にするかどうかと費用を先にご相談します。` の方が自然な場面があった。
- これは単なる言い換えではなく、`core` / `platform adapter` / `service adapter` / `style setting` / `user customization` の分離問題。
- 今後、ココナラで鍛えた自然化が汎用コアにも効く一方、ココナラ固有の語感や導線が core に入りすぎると、将来の SaaS 展開でノイズになる可能性がある。

暫定分類:

- core:
  - 勝手に追加作業へ進まない。
  - 追加料金・返金・無料対応・作業開始を未確認で断定しない。
  - buyer の次アクションを迷わせない。
- platform adapter:
  - ココナラでは `見積り提案` / `トークルーム` / `正式納品` / `承諾` / `おひねり` / `追加対応` のような語彙と導線を使い分ける。
- service adapter:
  - `15,000円` / `不具合1件` / `同一原因` / `修正済みファイル返却` / `secret値除外` など、bugfix-15000 固有の条件。
- style setting:
  - 事務的 / やわらかめ / 短め / かなり丁寧 / チャット寄り / メール寄り。
- user customization:
  - CTA の強さ、締め方、担当者名、会社ごとの言い回し、追加費用説明の温度感。

Pro に聞きたいこと:

- 返信OSを将来 SaaS 化する場合、現在ココナラで調整している文体・導線・温度感を `core` / `platform adapter` / `service adapter` / `style setting` / `user customization` のどこへ分けるべきか。
- ココナラ最適化が汎用 core へ入りすぎて、将来の業種展開でノイズになるリスクはあるか。
- 逆に、汎用 core にすべき構造を Coconala adapter に閉じ込めていないか。
- ユーザーがカスタマイズできるべき項目と、ユーザーに触らせない hard guard / core guard の線引き。
- `core: 勝手に追加作業へ進まない` / `platform adapter: ココナラでは追加対応にするかどうかと費用` / `style: 事務的・やわらかめ・短め` のような階層案は妥当か。
- 今後 #R で違和感を見つけた時、`core` / `adapter` / `style` / `preference` / `case_fix` のどこへ戻すべきかの判定基準。

現状の方針:

- 今は Coconala で実用するための文体調整を続けてよい。
- ただし、変更ログや Pro メモには `core / adapter / style` の分類を残す。
- ココナラ語をそのまま汎用 core に入れず、構造名へ抽象化してから戻す。
