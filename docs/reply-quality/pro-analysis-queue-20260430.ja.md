# Pro Analysis Queue 2026-04-30

## 位置づけ

ChatGPT Pro に後でまとめて相談する候補を保持する一時キュー。

すぐに Pro へ投げるためのプロンプトではない。`#RE` / `#BR` / human audit で実例を数件ため、再発性と副作用を見てから、まとめて分析依頼へ変換する。

## 現在の優先候補

### 1. `promise_consistency`

返信内で、前段の留保・不可・条件付き回答を、後段の成果物・納期・次アクションが上書きして見えないか。

代表例:

```text
確実に直ることまではお約束できません。
ただ、原因確認から修正済みファイルの返却まで進められます。
```

現状の方針:

- `promise-consistency-lens-20260429.ja.md` と Gold 37 で運用中。
- すぐ hard validator 化しない。
- `#RE` で数 batch 回し、どの subtype が再発するかを見る。

Pro に聞きたいこと:

- 正式 lens としてどこまで固定すべきか。
- `fix recommended` と `preference` の境界。
- lint / reviewer prompt / gold / skill のどこへ戻すべきか。

### 2. `conditional_scope_clarity`

「同一原因なら範囲内」「別原因なら相談」「修正できる箇所が特定できた場合は返却」のような条件付き scope を、buyer が誤解しない形で出せているか。

現状の方針:

- `promise_consistency` と近いが、成果物 promise より scope 条件の明瞭さに寄る。
- 補助 lens として観察し、単独正式化はまだ保留。

Pro に聞きたいこと:

- `promise_consistency` の subtype に含めるべきか、別 lens として分けるべきか。
- bugfix-15000 の公開返信で、条件をどこまで短く出すのが自然か。

### 3. `ack_to_answer_bridge`

相手の不安・状況を受けたあと、回答本体へ急に飛ばず、自然な橋を置けているか。

代表例:

```text
別の方でも原因が分からなかったとのことなので、まずは必ず直せるかが気になる状況だと思います。
コードとエラー内容を見る前に、必ず直せるとはお約束できません。
```

違和感:

- 内容は正しいが、受け止め文から制限回答へ少し急に飛ぶ。
- 内部パーツが外に出たように見える場合がある。

現状の方針:

- hard fail ではない。
- `conversation_flow_naturalness` の subpoint として観察する。
- 無理に接続語を増やして冗長にしない。

Pro に聞きたいこと:

- 日本語ビジネスチャットで、どの程度の橋渡しが必要か。
- `ack_to_answer_bridge` を独立 lens にする価値があるか。
- 何を gold 化し、何を好みに留めるべきか。

### 4. `permission_benefit_alignment`

`大丈夫です` などの許可・安心表現が、相手の負担を軽くしているのか、売り手側が相手の支払い・作業を許可しているように見えないか。

代表例:

```text
ログや関連コードは、お支払い完了後で大丈夫です。
```

違和感:

- 支払い後にログを送るのは buyer 側の負担であり、seller 側が「それで大丈夫」と許可しているように見える。
- 支払い前に送るか、支払い後に送るかだけの違いで、buyer の実務負担が減っていない場面では、`大丈夫です` が幼く・軽く見えやすい。
- `大丈夫です` 自体は便利で自然な場面も多いので、blanket ban はしない。

自然な使い方:

```text
値そのものは送らなくて大丈夫です。
原因が分からない状態でも相談できます。
```

置き換え候補:

```text
ログや関連コードは、お支払い完了後にトークルームで共有してください。
ログや関連コードは、お支払い完了後に共有いただければ、そこから確認に入ります。
スクショやログは、お支払い完了後にトークルームで共有いただければ進められます。
```

現状の方針:

- `jp_business_native_naturalness` の subpoint として観察する。
- 支払い・購入・相手に作業してもらう場面の `大丈夫です` は要注意。
- 相手の実務負担または心理負担を実際に減らす `大丈夫です` は残してよい。
- 単なる手順案内・材料送付タイミング・支払い導線では、`大丈夫です` より具体的な手順文を優先する。

Pro に聞きたいこと:

- この観点を正式 lens にする価値があるか。
- `大丈夫です` の hard rule / soft lens / lint 候補の線引き。
- 支払い導線で自然な標準表現。

### 5. `unnecessary_refusal_frame`

協力的な buyer に対して、必要以上に否定から入っていないか。

代表例:

```text
支払い前にスクショを送っていただいて、こちらで確認を先に始める形ではありません。
```

違和感:

- buyer は強い要求ではなく、単に「先に送るべきか」を確認しているだけの場合がある。
- その場面で `〜する形ではありません` や `先に確認はしません` から入ると、正しいが冷たく、防御的に読まれやすい。
- phase boundary は必要だが、外向けには拒否より手順を先に置いた方が自然な場面がある。

置き換え候補:

```text
スクショやログは、お支払い完了後にトークルームで共有いただければ進められます。
今の段階で先に送っていただく必要はありません。
```

現状の方針:

- `conversation_flow_naturalness` / `jp_business_native_naturalness` の subpoint として観察する。
- 強い要求・規約違反・secret値・外部誘導などは明確に止める。
- 協力的な材料共有タイミングの確認では、否定より手順案内を優先する。
- phase / payment route を弱めるためには使わない。

Pro に聞きたいこと:

- `permission_benefit_alignment` と同じ lens に含めるべきか、別 lens として分けるべきか。
- hard rule / soft lens / gold 候補の線引き。
- 支払い前材料共有の標準文として、どの表現が最も安全で自然か。

### 6. `agency_alignment`

返信文の中で、誰が何をするのか、誰に向けて許可・依頼・案内しているのかがズレていないか。

代表例:

```text
他の方で原因が分からなかった状態でも、Stripe決済後に注文が作られない不具合として15,000円で相談できます。
```

違和感:

- buyer の主質問は `依頼できますか？` なのに、返信が `相談できます` だと、返信者本人が受けるのではなく第三者へ相談できるようにも読める。
- 文法としては成立するが、seller / buyer / 第三者の行為主体が曖昧になり、`いや、相談する相手はあなたでは？` という日本語ネイティブの違和感が出る。
- `確認できます` `進められます` `相談できます` `大丈夫です` のような主語が省略されやすい表現で、対応主体・利益の向き・依頼先がずれることがある。

修正例:

```text
他の方で原因が分からなかった状態でも、Stripe決済後に注文が作られない不具合として、15,000円でご依頼いただけます。
```

現状の方針:

- hard fail ではなく soft lens として観察する。
- ただし、料金・作業開始・納品・返金・対応可否など、取引判断に関わる箇所で主体がズレる場合は必須修正寄りに扱う。
- `相談できます` を blanket NG にはしない。buyer が「相談してよいか」を聞いている場合は自然に使える。
- buyer の主質問が `依頼できますか` `対応できますか` `見てもらえますか` の場合、返答は `ご依頼いただけます` `対応できます` `確認できます` のように、行為主体と依頼先がずれない表現へ寄せる。

Pro に聞きたいこと:

- `agency_alignment` を独立 lens にする価値があるか。
- `permission_benefit_alignment` と同じ上位カテゴリに含めるべきか、別 lens として分けるべきか。
- hard fail / fix recommended / preference の線引き。
- lint 化できる表面パターンがあるか、それとも gold / reviewer prompt / human audit に留めるべきか。
- 日本語ビジネスチャットで、行為主体ズレが信頼感に与える影響。

## まだやらないこと

- `大丈夫です` の全面禁止。
- `promise_consistency` の hard validator 化。
- `ack_to_answer_bridge` を全返信に強制すること。
- `unnecessary_refusal_frame` を理由に、必要な不可表明まで消すこと。
- 支払い導線の自然化を理由に、料金・scope・phase・作業開始条件を弱めること。
- 1回限りの好みを skill / ng-expressions へ戻すこと。
