# Stripe実装（既存Next.js向け・UIデザイン除く）出品骨子（将来案）

更新日: 2026-02-24
ステータス: 未公開ドラフト（現行15,000円の不具合修正サービスで実績獲得後に検討）

## 目的

- 将来の追加出品として、`Stripe実装（既存Next.js向け）` を安全に出すための骨子を先に固定する。
- 「作れるか」ではなく「どこまで含めるか」を明確にし、スコープ事故を防ぐ。
- 現行の不具合修正サービス（15,000円）と役割を分ける。

## このサービスの立ち位置（将来）

- 現行サービス: 不具合診断・修正（入口）
- 将来追加サービス: Stripe実装（既存アプリへの組み込み）

分ける理由:
- 不具合修正と新規実装では、見積り時の確認項目と工数の振れ幅が大きく異なるため
- UI調整ループが混ざると、工数が読みにくくなるため

## 公開条件（Go条件）

以下を満たしたら公開を検討する。

1. 現行サービスで1〜3件を納品完了
2. 可能なら評価を1件以上取得
3. 見積り〜納品の運用が安定（返信速度 / スコープ判定 / 納品文）
4. Stripe実装の見積り項目をテンプレ化できている

## 対象案件（サービスの前提）

- 対象: `既存のNext.jsアプリ` への Stripe Checkout / Webhook の実装・組み込み
- 前提: 依頼者側で要件（何を売るか、決済後に何を反映するか）が定義されている
- 進行: テキストのみ（ココナラのメッセージ）

## 依頼内容の大分類（想定どおりの2分類）

1. 既存画面の指定箇所に決済導線を追加する
2. 新規ページを作成して決済導線を追加する

補足（工数を決める本命）:
- 実際の工数は、UIの置き場所より `バックエンド連携の深さ` で大きく変わる

## 工数を左右する主な軸（見積り判定の核心）

1. 決済種別
- 単発決済（payment）
- 定期課金（subscription）

2. Webhook後の処理
- ログ記録のみ
- DB反映（注文/会員状態更新）
- 既存ロジック連携（権限付与、メール、在庫等）

3. 画面範囲
- 既存画面の導線追加
- 新規ページ作成（簡易UI）

4. 管理機能
- Billing Portal 要/不要（定期課金時）

## 含める範囲（初期案）

- Stripe Checkout セッション作成
- 決済導線の実装（既存箇所追加 or 新規ページ1枚の簡易UI）
- success / cancel 画面（簡易）
- Webhook受信エンドポイント実装（署名検証）
- 受信確認または合意済みの反映処理（範囲固定）
- テスト手順メモ（Stripe CLI / テスト決済）
- 検証結果メモ（可能な範囲）

## 含めない範囲（明記必須）

- 細かいUIデザイン作成 / デザイン提案
- ピクセル単位の見た目調整、複数回のデザイン修正
- 本番デプロイ代行
- 依頼者リポジトリへの直接 push（契約明記がない限り）
- 仕様未確定の新規開発一式
- 周辺業務の大改修（会員管理、メール配信、管理画面刷新など）

## 納品物（将来案）

- 実装内容の要約メモ（何を追加/変更したか）
- 修正差分（patch / diff）または修正済みZIP
- 検証手順メモ（テスト決済、Webhook確認手順）
- 必要に応じて本番反映時の注意点（キー設定、Webhook URL確認）

## 見積り必須（推奨）

このサービスは、出品価格を固定しすぎず `見積り相談前提` で運用する。

理由:
- 依頼者アプリの構成差が大きい
- UI位置より、Webhook後の反映処理で工数が大きく変わる
- 定期課金/Billing Portalの有無で作業範囲が変わる

## 見積り前に回収する情報（必須）

1. 期待する動作（何を売るか / 決済後に何が起きるべきか）
2. 現在のアプリ構成（Next.js / 認証 / DB / デプロイ先）
3. 実装したい範囲（単発 or 定期、Webhook後の反映内容）
4. 導線位置（既存画面の指定箇所 / 新規ページ）
5. テスト環境の有無（Stripeテスト環境で確認可能か）

## スコープ事故を防ぐための判定ルール（将来サービス用）

- 「決済ボタン追加」と「決済後の業務反映」は別論点として切り分ける
- `どの画面` `どのAPI` `どのWebhookイベント` を対象にするかを見積り時に固定する
- UI調整は「軽微調整まで」。デザイン作成は範囲外
- 追加要件が出たら作業を止めて、別見積りまたは追加提案に切り替える

### FAQ骨子（将来）

Q: 新規デザインの作成もお願いできますか？
A: 本サービスは実装中心のため、細かいUIデザイン作成・デザイン提案は対象外です。既存デザインに合わせた最低限の実装・軽微調整まで対応します。

Q: 単発決済と定期課金のどちらも対応できますか？
A: 対応可能です。定期課金は Billing Portal の要否やWebhook後の処理内容で工数が変わるため、見積り時に範囲を固定します。

Q: 決済ボタンだけではなく、DB反映までお願いできますか？
A: 可能です。ただし工数差が大きいため、「どのイベントで、どのデータを、どこへ反映するか」を見積り時に明確化して対応します。

Q: 本番デプロイもお願いできますか？
A: 標準範囲外です。必要な設定手順と確認項目をお渡しし、最終反映は依頼者様環境でお願いします。

Q: 依頼者リポジトリへ直接pushできますか？
A: 契約に明記がない限り対応しません。修正差分（patch/diff）または修正済みZIPで納品します。
