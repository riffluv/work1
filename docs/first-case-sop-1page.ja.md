# 初案件 受注〜納品 1ページ手順

更新日: 2026-03-28

## 1. 見積り相談
- `場 + #R か #A + 相手文` で始める
- 購入前は `coconala` mode で進める
- 見積り初回は、必要な範囲だけ質問する
- 基本は `期待挙動 / 実際の挙動 / 環境` の最小3点で見立てる
- 購入前に重い解析へ踏み込まない

## 2. 購入直後
- `#S #R` または `#S #A` を使う
- `scripts/case-open.sh` で case を開始する
- `runtime/mode.txt` は `implementation` になる
- 以後の正本は `ops/cases/open/{case_id}/README.md`

## 3. 実装
- 期待挙動 / 実際の挙動 / 再現手順 / エラーログ / スタックを揃える
- 実装判断は `case README + コード + ログ` で進める
- 返信 docs は必要な時だけ戻る
- 大きい判断は `#M` で残す

## 4. 途中報告
- 購入後の進捗共有は `#P` を使う
- `#P` は3行で返す
- 未確認の原因断定はしない
- どこまでは正常で、今どこを見ているかを優先する

## 5. 納品準備
- 納品準備に入る時は `scripts/case-phase.sh --phase delivery`
- 主モードを `delivery` へ切り替える
- 合意済みスコープを再掲し、納品物を整える
- 納品物は `00_結論と確認方法.md` とコード納品物を標準にする

## 6. 正式納品
- 正式納品前に確認文を出す
- 正式納品では、何が壊れていたか / 何を変えたか / どう確認するか を明記する
- 最終確認は依頼者環境で行ってもらう

## 7. クローズ
- `#C` で案件を閉じる
- `scripts/case-close.sh` で `ops/cases/closed/{case_id}/` へ移動する
- `ops/case-log.csv` を更新する
- 次回に効く再発防止メモを短く残す
