# handoff-25000 納品物テンプレ

更新日: 2026-03-21

## 目的
- 非技術者は `何が分かったか / 何から見ればよいか` を最初に理解できるようにする。
- 技術者は `対象フロー / 関連ファイル / 危険箇所 / 次の着手順` をそのまま引き継げるようにする。
- 25,000円の納品物として、抽象語だけで終わらない実務メモにする。

## 品質基準
- 本番納品物の温度感と具体度は、次の内部品質本丸2点を基準にする。
  - `/home/hr-hm/Project/work/docs/internal-quality-samples/handoff-25000/00_結論と要点.md`
  - `/home/hr-hm/Project/work/docs/internal-quality-samples/handoff-25000/01_決済完了から会員状態更新_引き継ぎメモ.md`
- ただし、本丸の文言をそのまま流用するのではなく、実案件の対象フロー・一次証跡・実ファイル名に合わせて書き換える。
- 基準にするのは次の3点。
  - 非エンジニアでも一読で意味が取れる温度感
  - 危険箇所に `なぜ危ないか / 先に見るファイル / 起きやすい事故` があること
  - 次の着手順に `目的 / 最初に開くファイル / 確認すること / 次に進む目安` があること

## 標準納品物
1. `00_結論と要点.md`
2. `01_[対象フロー名]_引き継ぎメモ.md`

- `01_決済完了から会員状態更新_引き継ぎメモ.md`
- `01_フォーム送信からDB保存_引き継ぎメモ.md`

## 書き方のルール
- `確認できたこと / 推定 / 未確認` は章で分ける。
- 抽象語だけで終わらせず、ファイル名・条件・事故パターンまで落とす。
- `00_結論と要点.md` は概要に寄せ、`危険箇所 上位3` は1行要約で止める。詳細説明は `01_[対象フロー名]_引き継ぎメモ.md` 側へ寄せる。
- `危険箇所` は最低でも `箇所 / なぜ危険か / 触る前に見るファイル` を書く。
- `次の着手順` は最低でも `目的 / 最初に開くファイル / 確認すること / 次に進む目安` を書く。
- リポジトリ全体の仕様書にはしない。
