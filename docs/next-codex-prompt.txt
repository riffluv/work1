## 3. 現在の返信OSの主軸

現在の主軸は、ランダムに `#RE` を大量に回すことではありません。

優先順:

1. `#R` で実際にありそうな相手文へ返信候補を出す
2. ユーザーの human audit で違和感を拾う
3. 違和感を単語禁止ではなく、汎用構造へ名前づけする
4. `case_fix / preference / gold / skill / lint / soft_lens / hard_guard` に分類する
5. 再発性があり、ノイズにならず、既存骨格と衝突しないものだけ正本へ反映する
6. 採用済み構造だけ `#RE` に戻して regression / coverage 確認する
7. placement が迷う、または複数構造がたまったら Pro に critical review を依頼する

`#RE` は有料監査リソースを使うため、無目的に回し続けないこと。
`#RE` は「新しい発見の場」より「採用済み構造の回帰確認」として使う方が費用対効果が高いです。

## 4. `#R` の運用

`#R` は本丸です。
相手文に対して、外向けに送れる返信候補を作ります。

標準処理:

1. 相手文の phase を見る
   - prequote
   - quote_sent
   - purchased
   - delivered
   - closed
2. 主質問を決める
3. 明示質問を落とさない
4. service facts / price / scope / deliverable / forbidden moves を固定する
5. writer はテンプレではなく、短い packet から自然に書く
6. `japanese-chat-natural-ja` で最終自然化する
7. `runtime/replies/latest-source.txt` と `runtime/replies/latest.txt` に保存する

重要:

- final question は primary の候補であり、絶対優先ではない
- buyer の次アクションを決める問いを優先する
- 「一番困っている」「まず」「今回は」などの明示 cue は強い
- `相談できます` / `大丈夫です` / `対応できます` は blanket NG ではない
- 問題は単語ではなく、行為主体・約束レベル・phase・CTA の向き

## 7. 現在の主要構造 / lens

正式・準正式として重要なもの:

- `answer_focus_alignment`
  - buyer の主質問・明示質問に正面から答える
- `phase_evidence_alignment`
  - prequote / quote_sent / purchased / delivered / closed の証拠と返信導線を合わせる
- `promise_consistency`
  - 前段で留保した内容を後段の成果物・CTAで上書きしない
- `agency_alignment`
  - buyer / seller / third-party の行為主体をずらさない
- `deictic_agency_alignment`
  - こちら / そちら / こちらで選ぶ、などの指示語を誤らない
- `permission_benefit_alignment`
  - `大丈夫です` を許可調にしすぎない
- `positive_flow_before_refusal`
  - 軽い手順確認では通常フローを先に出し、境界突破要求では不可表明を残す
- `negative_frame_non_echo`
  - 「責めたいわけではない」「文句ではない」をそのまま復唱しない
- `pressure_word_summarization`
  - 「今日中に無料対応」「返金して」をそのまま反復せず、作業可否 / 費用や返金の扱いへ要約する
- `responsibility_admission_guard`
  - closed 後や前回ミス疑いで過失認定に見える表現を避ける
- `conversation_flow_naturalness`
  - 文をやさしくするより、会話の流れを作る
- `block_rhythm_flow`
  - 処理文・条件文・安全説明が同じリズムで並びすぎないかを見る
- `case_label_distance`
  - `〜の件ですね` が受付票・遠い案件ラベルに見えないかを見る
- `commitment_strength_calibration`
  - 時刻・納期・完了・返答の約束を強くしすぎない
- `deadline_autoinsertion_calibration`
  - buyer が質問本文をまだ送っていないのに固定時刻を約束しない
- `semantic_grounding`
  - 相手文にない技術語・原因候補・受領済み事実・作業状況を足さない
- `buyer_material_literacy`
  - 非エンジニア buyer に関係ファイル選別を返しすぎない
- `capability_declaration_drift`
  - `補足できます` で止めず、何を返すかを書く
- `post_completion_followup_scope_clarity`
  - closed 後の「関係確認」で、何を確認して何をしないかを buyer に見せる
- `commercial_temperature`
  - `この内容で問題なければ、そのままご購入ください` の強さを buyer の温度に合わせる

注意:

- lens を増やすこと自体が目的ではありません
- 1回限りの言い換えは `case_fix` または `preference`
- 再発性があり、他サービスにも効くものだけ `gold / skill / reviewer_prompt / lint / taxonomy` へ戻す

- 「単語禁止ではなく、汎用構造としてどう扱うか」を聞く

## 10. 汎用コア化の考え方

ココナラは通過点です。
ただし、ココナラは制約が厳しいため、返信OSコアを鍛える場として非常に有効です。

汎用コアに残すもの:

- service facts を読む
- phase / state に応じて発話可能範囲を固定する
- buyer の主質問を選ぶ
- buyer の材料理解度を見て、負担を返しすぎない
- action と capability を分ける
- promise consistency を守る
- semantic grounding を守る
- closed / completed 後の関係確認と実作業を分ける
- CTA の強さを調整する
- high-risk を人間確認へ上げる

ココナラ専用に閉じるもの:

- トークルーム
- 見積り提案
- 正式納品 / 承諾 / 差し戻し
- おひねり / 追加支払い
- ココナラ上の返金・キャンセル手続き

サービス専用に閉じるもの:

- 15,000円
- Next.js / Stripe / API
- 不具合1件 = 同一原因
- 48時間以内の一次結果目安
- 修正済みファイル返却
- secret 名の具体例
