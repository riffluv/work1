【最重要運用ルール】
- UI進行タグ指定がある場合は最優先で従う（`#$0`=直接購入後トークルーム初回 / `#$1`=合意後・提案送信前 / `#$2`=提案送信後 / `#$3`=提案送信後の未返信フォロー）。
- 誤記エイリアス（`#0$` / `#1$` / `#2$` / `#3$`）は、それぞれ `#$0` / `#$1` / `#$2` / `#$3` と同義で扱う。
- `#$1`/`#$2` はサービス経由・プロフィール経由・メッセージ経由の見積り提案で共通利用する。
- `#$1` を送ってから提案を送信し、送信後に `#$2` を送る（同時送信しない）。
- `#$3` は未返信フォロー時のみ使い、急かす文面を避ける。
- 文末タグ指定がある場合は最優先で従う（`#R`=送信用返信文 / `#A`=分析のみ / `#D`=下書き）。

- 対象が未指定の場合は「直前のCodex最終出力1件」を監査対象として扱う。

- `#C` はクローズ時に必ず使い、`ops/case-log.csv` へ1行追加する。

- `〜する形になります` は routing / 別件導線 / closed案内で再発しやすいので、送信前に self-check で抜き打ち確認する。
- 見積り相談では `次の3点だけお願いします` を固定で使わず、`次の3点が分かると、見立てを出しやすいです` など理由提示型を優先する。
- 見積り相談の質問数は 0〜3 点。3点は上限であり、毎回の標準ではない。今ある情報で判断できるなら質問なしで提案や見立てへ進む。
- 見積り相談はフォームではなく軽いチャットに近い場として扱い、返信構造を毎回同じ5段に固定しない。
- `.envやAPIキーの値は不要です` と `2時間以内` は必要な場面だけに使い、全件の末尾定型にしない。
- 送信前の自己確認は、まず `docs/coconala-reply-self-check.ja.md` の `最小5項目` だけを必ず回し、必要な時だけ詳細チェックへ降りる。
- 購入後や進行中の確認では `次の3点だけお願いします` の依頼型でよい。
- closed クレーム対応と納品物不満対応は、初回のみ監査前提の慎重運用にする。
- `主要1フロー整理 -> 原因判明 -> 修正` は、見積り中 / 取引中なら同じトークルーム内で追加料金対応へつなぐ表現を優先する。
- ただし内部では `25,000円の整理` と `15,000円の修正` の価格・納品物・スコープを分けて管理する。
- handoff の段階納品を説明するときは、作業説明だけでなく `引き継ぎメモを先に渡す` など納品物を明示する。
- closed 後だけは別ルールで、新規依頼 / おひねり案内の公式導線を崩さない。
- ココナラ上の場を混同しない。購入前は見積り相談 / メッセージ、購入後はトークルーム。
- 購入前の返信で `トークルーム添付` を前提に書かない。必要なら `ファイル添付でお送りください` または `ご購入後にコード一式をお送りください` と書く。
- handoff で購入前にZIPを受け取る場合も、構成確認までに止める。コード読解・原因特定・フロー整理は購入後。
- 対象フローが明確で 25,000円 で即答できる高確度 handoff は、ZIP依頼より先に `見積もり提案を送ります` を案内する。
