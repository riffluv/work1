【返信システム anti-drift】
- 今の返信システムは、live監査・少数case監査・外部レビュー・archetype fix を経て鍛えられた現行ベストであり、暫定のたたき台ではない
- 前回、返信システムを広くレビューしてさらに良くしようとした結果、layer / scenario / renderer を増やしすぎて骨格が悪化した実例がある。この事故を前提に、`広く改善` `さらに賢く` の解釈で骨格を触らないこと
- `coconala` mode では、今の理想の返信システムを壊す方向へ動かないこと
- 初見の印象やレビュー1回だけで骨格を再設計しない
- 単発の違和感に引っ張られて `scenario` や `renderer` を増殖させない
- buyer 起点 (`#R`) と seller 起点 (`#P`) を混ぜない
- 改善は、まず `reply-only / common / delivery-only` のどれかを切ってから正本へ戻す
- その場の好みより、`主質問直答 / secondary回収 / fact drift防止 / technical explanation safety / scope境界維持` を優先する
- 改善は否定しないが、導入するのは `再発性 / 実務ノイズ / 要点維持 / 既存ルールとの整合` で妥当性が確認できるものに限る
- 新しい観点や有効な方法が見つかった場合は取り入れてよいが、既存骨格の意味を理解せずに rule / layer / renderer を足して悪化させない
- 「広げる」より「締める」。既存骨格を壊す変更は、明確な再発証拠がある時だけ行う
- 新しい主軸へ移行した直後も、最初の数 batch は骨格変更を行わず、`#RE -> reviewer監査 -> 再発だけ rule へ戻す` の運用確認を優先する

【Coconala OS】
- ココナラ共通の取引仕様は `/home/hr-hm/Project/work/os/coconala/platform-contract.yaml` を正本にする
- 見積り相談の前段（prequote限定）: `coconala-prequote-ops-ja`

- 送信用返信の最終自然化は毎回 `japanese-chat-natural-ja`
- 最終自然化では、結論・質問項目数・価格・禁止事項・次アクションを増減させない
- Codex は主軸として最終判断と正本反映を担うが、外部AIの案を優先的に退ける意味ではない
- 外部AIの提案は、出所ではなく妥当性・再発性・実務ノイズ・要点維持・既存ルールとの整合で採否を決める
- 外部AIの案がより妥当なら、Codex はそれを採用して workspace に統合してよい
- 外部レビューで妥当と判断した自然化ルールは、その場の返信だけで終わらせず、関連 skill の正本へ反映する
- 反映前に、その改善が `reply-only` / `common` / `delivery-only` のどれかを必ず判定する
- `reply-only` は返信文系正本だけ、`delivery-only` は納品物系正本だけへ反映し、`common` 判定なしで相互流用しない
- 恒久反映の採用基準は `再発性がある / 実務ノイズを増やさない / 要点を薄めない / 既存ルールと衝突しない / 1回限りの好みではない`
- 直近の監査で有効だった恒久観点は、少なくとも `fact_alignment / semantic_echo / closing_complete / urgency_symmetry / unanswered_secondary / technical_explanation_safety` を含む
- `#CL` は `reply-review-prompt-ja` を使い、`runtime/replies/latest-source.txt` と `runtime/replies/latest.txt` を対象に Claude 向け監査プロンプトを作る
- `#GE` は `reply-review-prompt-ja` を使い、`runtime/replies/latest-source.txt` と `runtime/replies/latest.txt` を対象に Gemini 向け監査プロンプトを作る
- `#RE` は返信学習用ショートカットとして扱う。既存の返信骨格を使って stock から代表ケースを選び、外部監査にそのまま渡せる batch を 1 ファイルだけ更新する
- `#RE` のデフォルト対象は `bugfix-15000`。`#RE handoff` のように補足がある時だけ `handoff-25000` を対象にする
- `#RE` では生成本体の再設計を行わず、既存ルールで batch を組み直すことを優先する
- `#RE bugfix` の保存先は `/home/hr-hm/Project/work/サービスページ/rehearsal/bugfix-15000-返信学習/返信監査_batch-01.md`
- `#RE handoff` の保存先は `/home/hr-hm/Project/work/サービスページ/rehearsal/handoff-25000-返信学習/返信監査_batch-01.md`
- `#RE` の目的は「返信文を書くこと」ではなく、「外部監査に回すための batch を current rules で更新すること」
- `#R / #A / #D / #P` を優先し、相手文だけなら分析のみ
- `#R` の後ろや次行に自由文の補足がある場合は、optional な `user_override` として扱う
- `user_override` の優先順は `hard constraints > user_override > デフォルト推論` とし、tone は `temperature_plan`、ask / format / scope emphasis は `reply_contract` と `response_decision_plan` へ寄せる
- `user_override` が hard facts・公開状態・規約・最低限必要な確認と衝突する場合は hard constraints を優先し、反映できる範囲だけに留める
- 見積り相談では `coconala-prequote-ops-ja` を優先する
- skill 一覧に `/home/hr-hm/.codex/skills/coconala-*` や `/home/hr-hm/.codex/skills/delivery-pack-ja` などの custom skill が見えた場合は、その session を使わず再起動する
- 新しい文章 stock は `/home/hr-hm/Project/work/ops/tests/stock/inbox` に置く
- inbox の stock が追加された時は、「返信文を書く」ではなく「取り込み -> seed / eval / holdout / edge 仕分け -> eval-sources 接続 -> contract / renderer / lint / role suite 回帰」までを標準処理として行う
- stock 処理の結果は、単発の文面修正で止めず、再発パターンを `rule / renderer / lint` のいずれかへ戻す
- 実ストックを優先し、Claude / Gemini の生成 stock は不足領域の補完として使う
- random 5件で荒れた時は、その case だけ直さず、`case fix` か `archetype fix` かを先に切る
