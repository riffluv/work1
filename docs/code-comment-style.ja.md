# コードコメント運用ルール（確定版）

## 目的
- AI臭を抑え、保守に効くコメントだけを残す。
- 実装意図と外部制約の取りこぼしを防ぐ。

## 適用範囲
- この workspace のコードコメント全般。
- 指示がなくても常時適用（デフォルト）。

## コメントを書く判断
### 書く（4カテゴリのみ）
- WHY: なぜその実装にしたか
- 制約: 外部仕様や不変条件（署名検証、順序非保証、冪等性など）
- 副作用: 壊すと何が起きるか
- 運用条件: TODOの期限、再実行条件、監視/ログの要点

### 書かない
- WHAT/HOW の実況説明
- コードを読めば分かる説明
- 抽象語だけの注意文
- 実装とずれた古いコメント

## 文体ルール（AI臭抑制）
- 1〜2文、結論先頭。
- 日本語を基本にする（既存が英語主体なら合わせる）。
- 断定する場合は条件や理由を添える。
- 固有名詞を入れる（`event.id`, `Stripe-Signature`, `Idempotency-Key` など）。

## サービス別の濃さ
- `bugfix`: 危険箇所ごとに 0〜3 個までを目安にする。納品物で説明できる内容はコードに残しすぎない。
- `handoff`: 入口・危険箇所・境界条件だけを一段厚めに残す。全体像や未確認事項は引き継ぎメモへ逃がす。

## TODO / FIXME / NOTE
- `TODO` は期限またはチケットIDを付ける。
- `FIXME` は原則残さない。合意済みの残課題だけに限定する。
- `NOTE` は不変条件や制約の明示にだけ使う。感想メモには使わない。

## 暫定対応の書き方
- 暫定対応は「何が暫定か / なぜ必要か / いつ外すか」だけを書く。
- 経緯、会話ログ、調査過程はコードに残さず納品物へ逃がす。

## 禁止語
- `適切に`
- `必要に応じて`
- `念のため`
- `重要です`
- `〜させていただきます`
- `〜を行います`

## Stripe/Webhook の必須コメント3点
1. 署名検証で raw body が必要なこと
2. 冪等キー（例: `event.id`）で二重処理を防ぐこと
3. HTTP方針（`400` は入力不備、`500` は一時障害）を明示すること

## 実務テンプレ
- `// 署名検証は raw body が必要（json() で加工すると失敗する）`
- `// event.id で重複排除（同一イベントの二重適用防止）`
- `// 400=入力不備 / 500=一時障害（後段で再実行）`
- `// TODO(ISSUE-123, 2026-03-01): 互換層を削除`
- `// Billing API は server 経由で呼ぶ。UI 直接だと再試行時に二重実行になる`
- `// provider は 204 でも body が空とは限らない。status と payload を分けて判定`
- `// 暫定: v1 payload 互換を 2026-04-15 まで残す。移行完了後に削除`

## レビュー手順（Codex主軸）
1. Codex: 実装とコメント案を作成
2. Claude: `docs/writing-guideline.ja.md` と本規約で文章/コメントをレビュー
3. Codex: 最終判断して反映（主軸・決定責任はCodex）

## 最終チェック（Yes/No）
- コメントは4カテゴリのどれかに該当するか
- 禁止語・敬語が入っていないか
- 2行以上なら固有名詞が含まれているか
- TODOに期限またはチケットIDがあるか
- FIXMEを残す理由が納品物側でも説明できるか
