# 01_決済完了から会員状態更新_引き継ぎメモ.md

## 0. 1分要約
- このメモで、決済完了後の会員状態更新がどこで行われるかを順に追えます
- 事実・推定・未確認を分けているので、どこまで確定しているかが分かります
- 次に見るべきファイルと危険箇所を残しているため、続きの担当が入りやすい形です

## 1. 対象と目的
今回の対象は、価格ページから Stripe Checkout を開始し、決済成功後に `team` の購読状態を更新し、ダッシュボードに現在プランを反映する流れです。
このリポジトリでは、初回更新を `app/api/stripe/checkout/route.ts` が担当し、その後の購読状態同期を `app/api/stripe/webhook/route.ts` と `lib/payments/stripe.ts` が担当します。
目的は、次の担当者が「最初の更新はどこで行われるか」「後続の同期はどこで行われるか」「ダッシュボード表示までどこでつながっているか」を短時間で追える状態にすることです。

※ まず危険箇所を知りたい場合は「8. 危険箇所」からご覧ください。
※ どこから手をつければよいか知りたい場合は「9. 次の着手順」からご覧ください。
※ 関連ファイルの役割を先に押さえたい場合は「4. 関連ファイルと役割」が近道です。

## 2. このフローの全体像
- 入口は `app/(dashboard)/pricing/page.tsx` の購読ボタンです
- ボタン送信は `lib/payments/actions.ts` の `checkoutAction` を通って Stripe Checkout を開始します
- 決済成功後は `app/api/stripe/checkout/route.ts` が `teams` テーブルへ初回の購読状態を書き込みます
- その後の変更や解約は `app/api/stripe/webhook/route.ts` から `lib/payments/stripe.ts` の `handleSubscriptionChange()` へ渡されます
- ダッシュボードは `/api/team` 経由で team 情報を読み、`subscriptionStatus` と `planName` を表示します

## 3. 入口から完了までの流れ
1. Pricing 画面で価格を選び、`checkoutAction` が `createCheckoutSession()` を呼びます
2. Stripe Checkout で決済が完了すると、成功 URL から `/api/stripe/checkout?session_id=...` へ戻ります
3. `app/api/stripe/checkout/route.ts` が Stripe から `customer` / `subscription` / `product` を引き、`teams` テーブルへ保存します
4. 後続の状態変更は Webhook で `customer.subscription.updated` / `deleted` を受け、`handleSubscriptionChange()` が DB を更新します
5. ダッシュボードは `subscriptionStatus` が `active` または `trialing` の時に有料寄りの状態として表示します

## 4. 関連ファイルと役割
- `app/(dashboard)/pricing/page.tsx`
  - 役割: 価格表示と Checkout 開始ボタン
  - 備考: `priceId` を form で送ります
- `lib/payments/actions.ts`
  - 役割: `checkoutAction` と `customerPortalAction` の入口
  - 備考: Pricing 画面と Stripe 処理の橋渡しです
- `lib/payments/stripe.ts`
  - 役割: Checkout Session 作成、Billing Portal、Webhook 後続同期
  - 備考: `handleSubscriptionChange()` が後続の正本更新を担当します
- `app/api/stripe/checkout/route.ts`
  - 役割: 決済成功後の初回反映
  - 備考: `teams` テーブルへ `stripeCustomerId` / `subscriptionStatus` などを書き込みます
- `app/api/stripe/webhook/route.ts`
  - 役割: Stripe からの購読イベント受信
  - 備考: 署名検証のあと `handleSubscriptionChange()` に渡します
- `lib/db/queries.ts`
  - 役割: `getTeamByStripeCustomerId()` / `updateTeamSubscription()`
  - 備考: Webhook 側の更新条件を確認する時の基点です
- `app/(dashboard)/dashboard/page.tsx`
  - 役割: 現在プラン表示
  - 備考: 表示条件が狭いと DB 更新済みでも未反映に見えます

## 5. 確認できたこと
- Checkout 完了後の初回反映と、Webhook 側の後続同期は別経路です
- `app/api/stripe/checkout/route.ts` は `teams` テーブルへ `stripeCustomerId` / `stripeSubscriptionId` / `stripeProductId` / `planName` / `subscriptionStatus` を保存します
- Webhook 側の更新は `handleSubscriptionChange()` を通り、`getTeamByStripeCustomerId()` で team を特定してから `updateTeamSubscription()` を実行します
- ダッシュボード側は `subscriptionStatus === 'active'` または `'trialing'` の時だけ有料寄りに表示します

## 6. 推定
- 初回反映が失敗していても、後続 Webhook が正しく走れば最終的に状態は追いつく設計です
- ただし、Checkout 成功直後の表示確認だけで判断すると「未反映」に見える時間差が起こりえます
- Webhook 側で拾うイベント種別が不足していると、初回反映の取りこぼしを回収できません

## 7. 未確認
- 本番相当環境で `customer.subscription.created` がどの順序で届くか
- `customer.subscription.created` を受けていない現状で、戻り先未到達時の初回反映をどこまで回収できているか
- `/api/team` 側の取得キャッシュや再検証条件が、表示遅延へどの程度影響するか

## 8. 危険箇所
### 8-1. `app/api/stripe/checkout/route.ts`
- なぜ危ないか: 決済成功後の初回反映はここで完了します。`session_id` が無い、`session.customer` が展開されない、`subscriptionId` が取れない、`client_reference_id` から user/team が引けない、のどれかで止まると、決済だけ成功して画面は無料のままになります
- 先に見るファイル: `lib/payments/stripe.ts`, `lib/db/schema.ts`
- 起きやすい事故: Stripe ダッシュボードでは支払い成功が見えるのに、`teams.subscriptionStatus` が更新されず、ダッシュボードでは `No active subscription` のままになる

### 8-2. `app/api/stripe/webhook/route.ts`
- なぜ危ないか: ここでは `customer.subscription.updated` / `deleted` だけを受けています。`checkout.session.completed` をここでは扱っていないため、「初回反映は checkout 成功ルート、継続同期は webhook」という役割分担を見落とすと、どちらが壊れているか切り分けにくくなります
- 先に見るファイル: `lib/payments/stripe.ts`
- 起きやすい事故: 初回反映が失敗しているのに webhook 側だけ見続けて、原因の入口を取り違える

### 8-3. `lib/payments/stripe.ts` の `handleSubscriptionChange()`
- なぜ危ないか: Webhook 側の更新は `getTeamByStripeCustomerId()` で team を引き、`updateTeamSubscription()` で `subscriptionStatus` / `planName` などを更新します。初回の checkout ルートで `stripeCustomerId` が保存できていないと、後続 webhook の同期が成立しません
- 先に見るファイル: `lib/db/queries.ts`
- 起きやすい事故: webhook は受け取れているのに `Team not found for Stripe customer` が出て、購読状態が追従しない

### 8-4. `app/(dashboard)/dashboard/page.tsx`
- なぜ危ないか: 画面側は `subscriptionStatus === 'active'` または `'trialing'` の時だけ有料寄りに表示します。保存値と画面表示条件の対応を見ずに DB 側だけ追うと、「更新済みなのに未反映」に見える理由を取り違えます
- 先に見るファイル: `app/layout.tsx`
- 起きやすい事故: DB に状態は入っているが、画面では無料表示に見えて、利用者からは未反映と見なされる

## 9. 次の着手順
### 9-1. 初回反映の入口を確認する
- 目的: まず「決済成功直後の反映」がどこで完了するかを押さえる
- 最初に開くファイル: `app/api/stripe/checkout/route.ts`
- 確認すること: `session_id` 取得、`session.customer` / `session.subscription` 展開、`client_reference_id` から user/team を特定する流れ、`teams` テーブルへ保存している項目
- 次に進む目安: ここで `stripeCustomerId` / `subscriptionStatus` / `planName` を正しく保存していると確認できたら、次に Webhook 側の後続同期へ進む

### 9-2. 後続同期の入口を確認する
- 目的: 後続の購読状態変更がどこで反映されるかを押さえる
- 最初に開くファイル: `app/api/stripe/webhook/route.ts`
- 確認すること: 受けているイベント種別が `customer.subscription.updated` / `deleted` だけであること、署名検証失敗時に 400 を返すこと、`handleSubscriptionChange()` へ渡していること
- 次に進む目安: Webhook 入口が妥当なら、`lib/payments/stripe.ts` の `handleSubscriptionChange()` に進み、team 特定と更新条件を確認する

### 9-3. 表示条件とのズレを確認する
- 目的: 保存後に画面側がどう見せるかを切り分ける
- 最初に開くファイル: `app/(dashboard)/dashboard/page.tsx`
- 確認すること: `teamData.planName` と `teamData.subscriptionStatus` をどう表示しているか、`active` / `trialing` 以外の扱いがどうなっているか
- 次に進む目安: 画面表示条件に問題がなければ、`app/layout.tsx` の `/api/team` 取得と `lib/db/queries.ts` の team 取得へ進み、読み出し側のズレを確認する

## 10. 引き継ぎ時の注意
- このフローは「Checkout 成功後の初回反映」と「Webhook による後続同期」で更新元が分かれています。片方だけ見て結論を出さないでください
- `subscriptionStatus` だけではなく、`planName` と表示条件まで見て初めて「反映済み / 未反映」を判断できます
- まずは `app/api/stripe/checkout/route.ts` と `app/api/stripe/webhook/route.ts` の役割分担を押さえてから、詳細へ進むのが安全です
