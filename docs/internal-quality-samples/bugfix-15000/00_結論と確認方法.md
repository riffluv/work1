# 00_結論と確認方法.md

## 1. 今回の対象
- 対象不具合: 決済完了後に `/dashboard` の `Current Plan` が `Free` のまま残る
- 対象環境: `nextjs/saas-starter` / Next.js App Router / Stripe / 本番想定
- 今回の対応範囲: Checkout 成功後の初回反映から Webhook 同期、`/dashboard` 表示反映まで
- 今回は見ていない範囲: Billing Portal 側の操作、価格表示まわりのUI文言

## 2. 結論
- 状態: 今回の不具合は解消しました
- 主原因: `customer.subscription.created` を Webhook で処理しておらず、`/api/stripe/checkout` の到達に初回反映を依存していました
- 依頼者さまにお願いしたい確認: テスト購読を1件実行し、`/dashboard` の `Current Plan` と `subscriptionStatus` が更新されるかご確認ください

## 3. 確認手順
1. Pricing 画面からテスト購読を1件開始する
2. Checkout 完了後に `/dashboard` へ戻り、`Current Plan` が `Free` 以外へ変わることを確認する（購入したプラン名へ変わっていれば OK です）
3. Stripe ダッシュボードで `customer.subscription.created` が `200` で受信されていることを確認する（ステータスが `200` で、失敗やリトライが残っていなければ OK です）
4. 可能であれば DB 側の `subscriptionStatus` と `planName` も更新されていることを確認する（`subscriptionStatus` が `active` または `trialing` で、`planName` が購入プランに変わっていれば OK です）

## 4. 原因
この題材コードでは、Checkout 完了後に `/api/stripe/checkout?session_id=...` を通った場合は、その場で `teams` テーブルを更新します。
ただし、ユーザーが画面を閉じた、戻り先でエラーになった、ネットワークが途切れた、のどれかでこの経路に乗れないと、初回反映を Webhook 側で拾う必要があります。

修正前の Webhook は `customer.subscription.updated` と `customer.subscription.deleted` しか処理しておらず、初回作成イベントの `customer.subscription.created` を無視していました。
そのため、Checkout は成功しているのに `/dashboard` 側は `Free` のまま残ることがありました。

## 5. 修正内容
- 対象: `app/api/stripe/webhook/route.ts`
- 変更: `customer.subscription.created` でも `handleSubscriptionChange()` を呼ぶように修正
- 変更: 同期処理で例外が起きた時は `500` を返し、Stripe の再送で回復できる形に整理
- 修正方針: 初回反映をブラウザの戻り経路だけに依存させず、Webhook 単体でも会員状態をそろえる

## 6. 変更ファイル
- `app/api/stripe/webhook/route.ts`: `customer.subscription.created` を追加し、同期失敗時の `500` 応答を明示

## 7. こちらで確認した結果
- Checkout 完了後の初回更新と、Webhook 側の後続同期が別経路であることをコード上で確認しました
- Webhook 単体でも初回反映を回収できる形になり、`/api/stripe/checkout` 未到達時の未反映が残りにくくなりました

## 8. 未確認 / 依頼者環境でのみ確認できる点
- 実際の Stripe テストイベントで `customer.subscription.created` が先に届いた時の最終確認は、依頼者さま側でお願いします
- `/dashboard` の表示更新と DB 側の `subscriptionStatus` が両方そろうところまでご確認ください
