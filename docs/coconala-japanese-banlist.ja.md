# ココナラ返信 表現ガード

このファイルは、返信文の「禁止語リスト」ではなく、送信前に見る表現ガードです。
単語を見つけて機械的に消すためではなく、どの事故に接続しそうかを分類し、必要な場合だけ自然な文へ戻すために使います。

## 目的

- Codex が出しがちな機械文・AI文・内部処理語を減らす
- 再発しやすい表現を、hard accident と soft lens に分けて扱う
- `docs/reply-quality/ng-expressions.ja.md` と `docs/reply-quality/lens-taxonomy.yaml` の簡易入口にする

## 使い方

- Writer の前に「この単語は全部禁止」として読ませない
- Draft 後の reviewer / final naturalizer で、違和感の分類に使う
- 見つけた表現を消すだけで終わらせず、何が壊れているかを見る
- `相談できます` / `確認できます` / `大丈夫です` / `〜の件` / `まずは` / `はい` は blanket NG にしない
- 迷ったら、相手の主質問に短く答え、次にこちらが何をするかを具体化する

## 1. Hard Accident として扱うもの

次に接続する場合は、文体ではなく事故として見る。

- `handoff-25000` / 25,000円 / 主要1フロー整理 / 非公開サービス導線の live 漏れ
- 成功保証、今日中修正保証、返金保証、無料対応保証
- 支払い前のコード確認・ログ確認・原因確認・修正作業開始
- closed 後の旧トークルーム継続作業、修正済みファイル返却、おひねり追加前提
- 外部連絡、外部共有、外部決済、GitHub 作業面化
- 依頼者リポジトリへの直接 push、本番反映代行
- APIキー / Webhook secret / `.env` / DB接続文字列などの生値要求
- 相手文にない原因候補・技術語・作業状況の断定

## 2. 原則避ける強い組み合わせ

単語単体ではなく、場面との組み合わせで見る。

- `返金` と `承知しました` の組み合わせ
  - 返金を受けたように読めるため避ける
- `お気持ちももっともです` / `もっともです`
  - 苦情・返金・低評価圧では責任認定や迎合に見えやすい
- `返金についてすぐに断定する前に`
  - 返金判断を前提にした防御文に見えやすい
- `保証はしていません`
  - 感情注意ケースの初回返信では、冷たい拒否に見えやすい
- `返金できません` / `無料対応できません` の単独断定
  - まず確認範囲と判断順に変換する

## 3. 条件付き注意に留める表現

以下は blanket NG ではない。文脈上ずれる時だけ直す。

### 行為主体・許可の向き

- `相談できます`
- `確認できます`
- `対応可能です`
- `進められます`
- `大丈夫です`

見る点:
- buyer が「依頼できますか」と聞いているなら、標準は `ご依頼いただけます`
- buyer が「見てもらえますか」と聞いているなら、標準は `こちらで確認します`
- buyer の負担が減る場面なら `大丈夫です` は可
- 支払い・材料共有・作業開始を seller が上から許可する形になるなら避ける

### 距離感・受付票感

- `〜の件`
- `確認案件`
- `状況を整理します`
- `整理してお返しします`
- `別件として整理する`

見る点:
- 既に同じトークルームで直接やり取りしている内容を、遠い案件ラベルにしていないか
- 内部処理や受付票ではなく、相手の次行動に変換できているか

### 内部ルール語・開発者語

- `作業として発生します`
- `スコープを切って`
- `切り分けて考える`
- `対応範囲外です。` だけで終わる書き方
- `対応していません。` だけで終わる書き方

見る点:
- 断る必要がある場面でも、不可表明だけで止めず、次に取れる手順へ接続する
- buyer 向けには `今回の修正とは分けて、対応方法と費用を先にご相談します` のように訳す

### 評価語・曖昧語

- `安全です`
- `自然です`
- `合っています`
- `分かりやすいです`
- `見た目は軽く見えても`
- `気になると思いますので`

見る点:
- Codex の評価コメントが外向け返信に混ざっていないか
- `何ができるか / 何を確認するか / 何を送らなくてよいか` に置き換えられるか

### 硬さ・機械文になりやすい語

- `必要情報がそろい次第`
- `ご共有ください`
- `ご査収ください`
- `させていただきます` の連発
- `〜自体は`
- `形になります`
- `形ではありません`

見る点:
- 曖昧な待機文になっていないか
- `ご購入後にトークルームで送ってください` など、具体的な場面と言い換えられるか
- `形ではありません` は、強い境界突破要求なら可。軽い手順確認では通常フローを先に出す

## 4. 再発しやすい型

全面禁止ではないが、同じ batch や同じ返信内で繰り返すと AI 感が出る。

- `本来はどの〜想定でしょうか`
- `まず状況を確認させてください`
- `確認して、2時間以内には見立てをお返しします`
- `〜の件も含めて、まず確認させてください`
- `状況の長い復唱 -> 感情ラベル -> 大丈夫です`
- `確認しました -> はい、確認できています`
- `できること宣言 -> できること宣言 -> 返答時刻`

直す時は、次の順で考える。

1. 相手の主質問に直接答える
2. 通常フローを先に案内する
3. 必要な境界だけ短く置く
4. 次に buyer がすること、またはこちらが返すことを具体化する

## 5. どこへ戻すか

- 価格・scope・phase・secret・public/private・closed 後作業が壊れる場合: hard guard / validator
- 行為主体や許可の向きがずれる場合: `agency_alignment` / `permission_benefit_alignment`
- 軽い手順確認なのに否定から入る場合: `positive_flow_before_refusal`
- 相手の否定感情や圧力語を復唱している場合: `negative_frame_non_echo` / `pressure_word_summarization`
- 文章の塊感・処理文の連続だけが気になる場合: `conversation_flow_naturalness` / gold reply
- 1回限りの言い換え好みの場合: rule 化せず、case polish に留める

## 6. 基本姿勢

このファイルの目的は、Codex の思考を狭めることではない。
サービス事実・取引状態・相手の主質問を保ったまま、外向け返信として不自然な出方をした表現だけを戻す。
