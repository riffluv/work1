# handoff prequote mini-contract

## 目的
- `handoff` の見積り相談で、`bugfix` と同じ入口処理をしないための最小契約を固定する。
- 購入前の `見積もり判断` と、購入後の `コード受領 / 整理開始` を混同しない。
- `handoff` の高確度案件では、情報回収より先に `見積もり提案` へ進める。

## 前提
- 正本のサービス定義は [handoff service.yaml](/home/hr-hm/Project/work/ops/services/handoff-25000/service.yaml) を使う。
- 購入前の場の仕様は [platform-contract.yaml](/home/hr-hm/Project/work/os/coconala/platform-contract.yaml) を使う。
- `handoff` は private の間、外向け返信でサービス名・価格・購入導線を露出しない。

## 基本原則
- `handoff` の prequote は、購入前に `整理の着手` を始める場ではなく、`この相談をどの入口で受けるか` を決める場とみなす。
- 高確度なら、購入前にコード一式を取りに行くより先に `見積もり提案` を案内する。
- 購入前に資料依頼を出す場合も、目的は `見積もり判断` に限る。`確認後に整理します` のような購入後フェーズの動詞へ寄せない。
- `25,000円` は主要1フローの整理として扱い、作業負担が軽くても、購入前に複数フロー前提へ広げない。
- `主要1フロー` は、アプリに固定で1本だけあるものではなく、`1つの起点から1つの目的に向かう、つながった処理のまとまり` として扱う。buyer が複数の要素を挙げても、購入前に機械的に別フローと断定しない。
- ブラックボックス案件では、相手に完全に選ばせるのでも、こちらが黙って決めるのでもなく、`今回の対象フロー案` をこちらから1つ提案し、購入前に合意を取る。

## 高確度 handoff
以下をすべて満たす場合は、高確度として扱う。
- 対象フローが 1 つ具体的に書かれている
- 主目的が `修正` ではなく `整理 / 引き継ぎ / 全体把握` である
- whole repo や正式仕様書の期待が前面に出ていない
- 相手の依頼意志が明確、または整理依頼として十分具体的

この場合の既定:
- 質問を増やさず、`その流れで進められる` と返す
- closing は `見積もり提案を送ります` を先に置く
- コード共有は `ご購入後にコード一式をお願いします` と購入後手順として書く
- `コード一式をお送りください` を初手の既定にしない
- 対象フローを提案する時に `まずは` を使って後続の追加段階を含意しない。`今回はこの流れを対象にする形でよければ` のように、今回の範囲と合意対象を明示する

## 低確度 handoff
以下のいずれかに当たる場合は、低確度として扱う。
- 対象フローが曖昧
- 相手自身も何を整理したいか決め切れていない
- whole repo や仕様書化の期待が混ざっている
- 修正依頼と整理依頼が混線している
- 最小情報だけでは `handoff` として受けるか判断しにくい

この場合の既定:
- 追加確認は 1 点までを優先する
- 必要なら最小資料をお願いしてよい
- 資料依頼は任意に寄せる
- `コード一式をください` を当然の前提にしない
- `どの場面が一番困るか` のような優先順位づけを、ブラックボックス状態の相手へ返さない
- こちらから `今回は ○○ の流れを対象にする形がよさそうです` と仮の起点を置き、`問題なければこの流れで進めます` と合意を取る
- 相手が `3つほど` `まとめて` のように広く希望しているだけで、対象がまだ曖昧な時は、追加フローや追加料金の案内を先頭に出さない。先に `基本は主要1フロー` と `今回どこを対象にするか` を整える

推奨:
- `可能であれば、対象フローが分かる資料だけ先にお送りください`
- `すぐ難しければ、既存メモやスクショだけでも大丈夫です`
- `今回は ○○ の流れを対象にする形でよければ、この前提で進め方とお見積りをお送りします`

## bugfix との違い
- `bugfix` は症状・環境・エラー文・スクショで入口判定しやすい。購入前に ZIP を標準要求しない。
- `handoff` は整理対象の輪郭が曖昧だと資料が必要になることがある。ただし、それでも既定は `提案先行` であり、`資料先行` ではない。

## 購入前に書いてよいこと
- `その流れで整理できます`
- `見積もり提案をお送りします`
- `ご購入後にコード一式をお願いします`
- `必要なら、見積もり判断のために対象フローが分かる資料だけお願いします`
- `今回は ○○ の流れを対象にする形がよさそうです`
- `問題なければ、この流れを今回の対象として進めます`
- `ほかの流れも必要であれば、対象範囲を確認したうえでご案内できます`

## 明示的な複数フロー希望
- 相手が `決済` `会員登録` `メール送信` のように、複数フローを具体的に指定しており、主目的も `整理メモ` と明確な場合は、複数フロー前提の案内をしてよい
- ただし、それでも冒頭は `基本は主要1フロー` を先に置く
- 挙げられた要素が、同じ起点で連なる1つの流れとして整理できる場合は、購入前に `3フロー` と機械的に数えない
- 追加フローや追加料金は、対象が具体的に確定している時だけ後段で案内する
- `対象範囲を確認したうえで追加分もご案内できます` のように、upsell ではなく範囲確認の文として書く

## 購入前に既定化しないこと
- `コード一式をお送りください。確認後に進め方を返します`
- `一式を見てから整理を始めます`
- `受領後に最初に見る流れを決めます`
- `どの場面が止まると一番困るか教えてください`
- `今回はこの2つもまとめて見ます`
- `まずは ○○ から入ります`

上の 6 つは、相手に `もう購入後の確認フェーズに入っている`、`こちらが判断負担を返している`、`後続の追加段階が当然ある`、または `1フロー契約を曖昧にしている` 印象を与えやすい。

## closing の既定
- 高確度: `問題なければ、このまま見積もり提案をお送りします。ご購入後にコード一式をお願いします。`
- 低確度: `内容を確認したうえで進め方を判断したいため、可能であれば対象フローが分かる資料をお送りください。`
