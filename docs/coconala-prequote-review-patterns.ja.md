# ココナラ見積り返信 20件レビュー頻出修正パターン

更新日: 2026-03-24
対象:
- `ops/tests/prequote-bulk-20.txt`
- Claude 日本語品質監査 4バッチ分

## 目的
- 見積り返信20件のレビューで繰り返し出た指摘を、次回以降の固定ルールへ落とす。
- 個別修正で終わらせず、skill / Golden Replies / 仕上げチェックへ戻す。

## 結論
- 現在の返信品質は「骨格は十分通るが、日本語の初動ニュアンスと繰り返し制御で差が出る」段階。
- 量を増やす前に、頻出修正パターンを固定資産化した方が伸びる。
- 特に重要なのは次の6点。
  1. 初動で分かったふりをしない
  2. route ごとに書き出しを変える
  3. 情報不足ケースでは約束を弱める
  4. non_engineer 向けの内部語を消す
  5. 確認だけしたい相談でも公開 bugfix は 15,000円 案内を崩さない
  6. 範囲外案内は「断り1回 + 条件付き提案」で終える

## 頻出修正パターン

### 1. 初動で「分かったふり」に見える
症状:
- `状況は把握しました` が初動で強すぎる
- コードや証跡未確認なのに、全体を理解した印象になる

修正方針:
- 初動は `承りました` / `内容を確認しました` を基本にする
- かなり論点が明確な場合だけ `把握しました` を使う

固定ルール:
- 見積り初回では、証跡未確認の段階で `状況は把握しました` を使わない

### 2. 書き出しが同じになりやすい
症状:
- 同じバッチで `ご相談ありがとうございます。` が 4/5 以上出る
- route を無視して固定化しやすい

修正方針:
- route ごとに書き出しを切る

基本対応:
- `service` -> `ご相談ありがとうございます。`
- `profile` -> `ご連絡ありがとうございます。`
- `message` -> `メッセージありがとうございます。`

固定ルール:
- 同一バッチで書き出し1行目が `4/5` 以上同一にならないようにする

### 3. 結語が同じになりやすい
症状:
- `2時間以内 + 48時間以内` の骨格が毎回同じ
- 情報不足ケースでも 48時間行を機械適用しやすい

修正方針:
- 情報不足かどうかで結語を分ける
- 同一バッチで言い回しを回す

基本対応:
- 情報不足 + 金額未提示:
  - `まずは2時間以内を目安に、確認の入り口だけお返しします。`
- 情報あり + 15,000円提示済:
  - `まずは2時間以内を目安に、見立てをお返しします。`
  - `そのあと、48時間以内を目標に一次結果をご案内します。`

固定ルール:
- `確認の入り口だけお返しします` は、情報不足かつ金額未提示の初動だけ
- `.env` 注意文のあとに結語があるか必ず確認する

### 4. non_engineer 向けに内部語が混ざる
症状:
- `判断がぶれやすい`
- `保持情報`
- `確認案件`
- `修正案件`
- `時間単位の範囲で`

問題:
- 購入者に意味が伝わりにくい
- こちらの内部分類や工数感がそのまま見える

修正方針:
- 「何を見るか」「何が分かるか」に置き換える

言い換え例:
- `判断がぶれやすい` -> `何から進めるかを整理しにくい`
- `保持情報` -> `サイト側で持っている表示用の情報`
- `確認案件` -> `確認だけの対応`
- `時間単位の範囲で` -> `設定と処理の流れを見て確認します`

固定ルール:
- non_engineer 向けでは内部語をそのまま出さない

### 5. anxious ケースで、自責や怖さを拾い切れていない
症状:
- 相手が `放置してしまっていました` `怖いです` と書いているのに、受け止めが薄い

修正方針:
- 不安そのものより、まず「今から見ればよい」と返す

有効だった表現:
- `まずは今の状態を確認します。`
- `まずは1件分の情報だけで大丈夫です。`

固定ルール:
- `anxious + 自責` では、責めない一文を先頭2段落以内に入れる

### 6. 確認だけしたい相談で旧確認案内が出やすい
症状:
- `確認案件`
- `合っています`
- `時間単位の範囲で`

修正方針:
- Golden Reply §2 の型に固定する

固定文の芯:
- `15,000円には、原因切り分けと修正可否の確認を含めています。`
- `まずはStripeまわりの設定と処理の流れを見て、この内容でそのまま進められるかを確認します。`

固定ルール:
- 確認だけしたい相談でも、公開 bugfix では 15,000円 案内に統一する
- 金額は1通で1回だけ

### 7. 範囲外案内が冷たくなりやすい
症状:
- `断り -> 条件付き提案 -> 再度断り`
- `対象外としてのご案内になります`

修正方針:
- 断りは1回だけ
- その後は「この条件なら確認できる」で終える

推奨構造:
1. `正直にお伝えすると、今のサービスとは少し範囲がずれています`
2. `ただ、[条件] なら確認できます`
3. その条件に必要な3点を出す

固定ルール:
- 範囲外案内は「拒否 -> 提案」で終える
- 官僚調の締めを足さない

### 8. 相手が書いた情報とズレた質問をしやすい
症状:
- ハードコードと言っているのに環境変数だけを聞く
- 分からないならスクショ、のような条件文矛盾
- 述語欠落

修正方針:
- 質問前に、相手がすでに書いた事実を1回確認する
- 質問文は `1項目1情報` を守る

固定ルール:
- 質問に入る前に「相手の既出情報と矛盾しないか」を見る
- 条件文と述語欠落を仕上げ前に確認する

### 9. `.env` 注意文は統一した方が強い
固定文:
- `.envやAPIキーの値は不要です。キー名だけで大丈夫です。`

理由:
- 揺れが減る
- non_engineer にも通じる
- セキュリティ注意が定着する
