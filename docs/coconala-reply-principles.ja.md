# ココナラ返信 原則5行

1. 先に相手の主質問へ直接答える。
2. 次に、価格・スコープ・規約の線を崩さない。
3. 購入前と購入後の場を混同しない。
4. 質問は0〜3点の範囲で、今必要な分だけに絞る。
5. そのうえで自然な日本語に整える。

補足:
- 見積もり相談はフォームではなく、軽いチャットに近い場として扱う。
- 相手が言っていない論点をテンプレ都合で増やさない。
- Yes/No 質問には、説明より先に `可能です` / `現状では難しいです` を返す。冒頭を機械的に `はい、` に固定しない。
- 価格交渉・条件確認が主質問の時は、まずその主質問に直答する。相手の同意がまだない段階では、症状送付や購入案内など次工程へ勝手に進めない。次の案内を添えるなら、`その前提でもよろしければ、今回も対応できます。` のように同意条件と対応可否だけを置き、相手の温度が見えるまで行動要求を急がない。
- bugfix は最小情報優先、handoff は対象フローと受注確度に応じて ZIP を使い分ける。
- 返信構造を毎回同じ5段に固定せず、確度が高い案件では短く終えてよい。
- 生の秘密情報がすでに貼られた時は、`送らないで大丈夫です` だけで終わらせず、`このメッセージ内の値は削除またはマスキングしてください` を先に置く。live 値なら差し替え依頼まで含める。
- prequote / quote_sent では、`もし進める前提で問題なければ` のような同意 gate を結論より先に置かない。`直答 -> 範囲説明 -> 最小確認` の順を優先する。
- 同一原因か / 2件分になるか / 追加料金が不安という質問には、`同じ原因でも軽い修正なら基本料金内 / 同じ原因でも大きい変更が必要なら追加相談 / 別原因は追加相談 / 追加料金は自動で発生しない` を短く入れる。
- prequote の受注可否返信では、必要に応じて `15,000円では、不具合1件について原因を確認し、同じ原因の範囲で修正内容と確認手順まで対応します。` を短く添える。
- 上の説明を buyer 向けに出す時は、`修正可否確認` のような内部語や、`進められる内容かを見ます` のような曖昧な表現を避ける。何が返るかを、`原因の確認 / 修正内容 / 確認手順` の粒度で書く。
- 緊急相談では、納期確約はせずに `まず内容を優先して確認します` 程度の一文で受ける。
- 着手前の案内では、`確認中です` と言い切りすぎず、`内容を確認しています` `状況を見ています` のように現在地を弱めに返す。
- buyer 向けの初手で、`本番 / プレビュー / ローカル` のような専門語をそのまま並べない。環境確認が必要な時は、`公開中のサイトで起きる` `自分のPC上でも起きる / 起きない` のように、相手が見ている場所で聞く。
- purchased の進捗返信は、`今見ている箇所 -> 次に見る箇所 -> 返答時刻` まで置く。
- closed 後の再発相談は、料金判断より先に `再発か新規かを先に返す` を優先する。
