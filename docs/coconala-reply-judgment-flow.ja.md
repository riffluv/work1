# ココナラ返信 判断フロー

1. 相手の主質問を1文で言い直す。
  例: `15,000円で見られるか` / `まず診断から発注できるか` / `このまま続けて修正まで頼めるか`

2. 先に答えるべき結論を決める。
  - Yes/No
  - 料金内に収まるか
  - 対応可能か
  - 入口サービスはどちらか

3. 案件種別を判定する。
  - `bugfix`: 具体的な症状がある
  - `handoff`: 1フロー整理や引き継ぎが目的
  - `boundary`: bugfix / handoff / 実装支援の境界
  - `after_purchase / after_close`: 購入後 / クローズ後

4. フェーズを判定する。
  - 購入前: 見積り相談 / メッセージ
  - 購入後: トークルーム

5. 確度を判定する。
  - `高確度`: 価格と入口をほぼ即答できる
  - `低確度`: 情報が足りず、入口や価格判定に追加情報が必要

6. 返信の型を選ぶ。
  - 高確度 bugfix: `対応可能性 + 最小情報`
  - 高確度 handoff: `対応可能 + 購入導線 + 購入後コード共有`
  - 低確度 bugfix: `今の時点では + 追加1〜2点`
  - 低確度 handoff: `まずどの流れを見るか + 必要なら ZIP`
  - 希望進め方を断る時: `断る + 通常の進め方 + もしこの進め方で問題なければ`

7. 何を聞くかを最小化する。
  - デフォルトは「質問なし」。今ある情報で入口・価格・対応可否が判断できるなら、そのまま提案や見立てへ進む
  - bugfix: 期待挙動 / 実際の不具合 / 発生環境 を優先
  - handoff: 対象フロー / 目的 / 受け渡しに必要な資料 を優先
  - 相手が対象フロー・目的・資料提供可否をすでに十分具体的に書いているなら、確認質問を増やさず購入導線へ進む
  - 確認質問は「相手がまだ決めていないこと」だけに使う
  - 質問数は 0〜3 点。3点は上限であり、標準ではない

8. ZIP の扱いを決める。
  - bugfix: 原則購入後
  - handoff: 構成確認が必要な時だけ購入前に依頼可
  - 高確度 handoff: 購入導線を先に出し、コード共有は購入後

9. 最後に1文で次アクションを示す。
  - `見積もり提案を送ります`
  - `見立てを返します`
  - `このまま進められるかを返します`
  - ただし、見積もり相談はチャットに近いので、`.env注意` や `2時間以内` を毎回固定で足さない
  - 高確度案件は `あいさつ + 結論 + 購入導線` の短い構造で終えてよい
