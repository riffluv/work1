# ココナラ返信 判断フロー

1. 相手の主質問を1文で言い直す。
  例: `15,000円で見られるか` / `まず診断から発注できるか` / `このまま続けて修正まで頼めるか`

2. 先に答えるべき結論を決める。
  - Yes/No
  - 料金内に収まるか
  - 対応可能か
  - 入口サービスはどちらか
  - prequote で `15,000円で受けられるか` が主質問なら、`受ける / まだ言えないので1〜2点確認 / その料金では受けない` の3択を先に固定する

2.5 返信姿勢を決める。
  - 主質問に対して今すぐ答えるか、確認してから返すか
  - こちらが決めるか、相手に聞くか
  - 受け止めを先に置くか
  - 骨格は見積り初回か、購入後の短い返しか、途中報告か、納品前後か

3. 案件種別を判定する。
  - `bugfix`: 具体的な症状がある
  - `handoff`: 1フロー整理や引き継ぎが目的
  - `boundary`: bugfix / handoff / 実装支援の境界
  - `after_purchase / after_close`: 購入後 / クローズ後

補足:
- `handoff` は「主要1フローを深く整理する」サービス
- `bugfix` は「同一原因の不具合1件」を扱うサービスで、`1フロー / 1エンドポイント` は原因判定の補助線として使う
- bugfix の外向け説明では、`1フローのサービス` のように見せず、まず `同じ原因の範囲で見られるか` を前に出す

4. フェーズを判定する。
  - 購入前: 見積り相談 / メッセージ
  - 購入後: トークルーム

4.5 先に `platform / state / service` の噛み合わせ事故を止める。
  - `closed` 後に実作業、書き直し、追加修正を受ける話なら、前回取引の延長ではなく、改めて依頼が必要と先に返す
  - 分割払い、支払い方法変更、外部決済の相談は、こちらで決められる話ではない。ココナラ側の決済仕様として短く返す
  - `本番環境はない` `作り方が分からない` `Webhookの受け方が分からない` のような相談は、いま壊れている bugfix と同列に扱わない。boundary として止める
  - `今は問題ないが将来影響が出るかも` `知り合いにそう言われた` のような予防相談は、前回範囲や現行 bugfix に自動吸収しない
  - GitHub / Drive / Slack / 電話 / 生の秘密情報は、技術判断より先に停止と代替案内を入れる
  - `他の人は10,000円だった` `まとめて30,000円にしてほしい` のような価格比較や独自バンドル提案は、その金額に乗らない。いまの提供単位と違いだけを短く返す
  - `Zoomなら` `電話なら` `直接会って説明してほしい` のような相談は、不可だけで止めず、テキストで何を返せるかまで続ける
  - `同じ商品が2個届いた` `決済が2回入った` のような重複処理相談では、返金や原因説明より先に、同じ操作を続けるべきか止めるべきかを先に返す
  - Stripe以外の決済名が明示された時は、即断で対象外 / 対応可に寄せすぎない。決済サービス側か、Next.js / Webhook以降の後続処理かを先に切る
  - `15,000円が上限` のように予算上限が明示され、かつ別原因の可能性が残る時は、安易に `15,000円で進められます` に倒さない。最悪ケースで `15,000円では原因確認と切り分けまで` になることを先に明示し、それでも進めるか相手に選んでもらう

5. 確度を判定する。
  - `高確度`: 価格と入口をほぼ即答できる
  - `低確度`: 情報が足りず、入口や価格判定に追加情報が必要

5.5 返信契約を作る。
  - 明示された質問を `explicit_questions` として列挙し、`primary_question_id` を1つ決める
  - 各質問を `answer_map` で `今答える / 確認後に返す / 相手にだけ依頼する / 断る` のどれかへ落とす
  - 追加確認が必要なら `ask_map` を作り、何の解除条件かを明示する
  - `issue_plan` は互換要約として残してよいが、正本は `answer_map / ask_map` とする
  - `required_moves` には、その返しで必ず入れる動きだけを置く
  - `forbidden_moves` には、その返しでやってはいけない動きを置く
  - purchased の軽い追加報告では、`番号付きQ&Aで全部返す` `見積り初回へ戻る` `曖昧な保留だけで終える` を禁じる

6. 返信の型を選ぶ。
  - 高確度 bugfix: `対応可能性 + 最小情報`
  - 高確度 handoff: `対応可能 + 購入導線 + 購入後コード共有`
  - 低確度 bugfix: `今の時点では + 追加1〜2点`
  - 低確度 handoff: `まずどの流れを見るか + 必要なら ZIP`
  - 希望進め方を断る時: `断る + 通常の進め方 + もしこの進め方で問題なければ`
  - 購入後の軽い質問や追加報告は、見積り初回の骨格を持ち込まず `今答えること -> 見てから返すこと -> 最小依頼` で返す
  - `reply_skeleton` は文面の骨格を決めるが、各論点を今答えるか保留するかは `answer_map` を優先する
  - 1通に話題が3つ以上ある時は、`緊急停止が必要な話題 -> 進行中の本件 -> 追加された別件` の順で返す

7. 何を聞くかを最小化する。
  - デフォルトは「質問なし」。今ある情報で入口・価格・対応可否が判断できるなら、そのまま提案や見立てへ進む
  - 相手が `どこが大事か分からない` `おまかせで` と言っている場合は、優先順位を相手へ返さない。今ある資料で仮に最初に見る対象をこちらで決めて返す方を優先する
  - bugfix: 期待挙動 / 実際の不具合 / 発生環境 を優先
  - handoff: 対象フロー / 目的 / 受け渡しに必要な資料 を優先
  - handoff で相手が `どこが大事か分からない` `おまかせで` と言っている場合は、資料受領を主にし、フロー選定は `こちらで決めて返す` を優先する
  - 相手が対象フロー・目的・資料提供可否をすでに十分具体的に書いているなら、確認質問を増やさず購入導線へ進む
  - 確認質問は「相手がまだ決めていないこと」だけに使う
  - 質問数は 0〜3 点。3点は上限であり、標準ではない
  - こちらで最初に見る1フローを決める時は、`止まると影響が大きいので` `いま一番不安が大きいので` のように、理由を一言添える

8. ZIP の扱いを決める。
  - bugfix: 原則購入後
  - handoff: 構成確認が必要な時だけ購入前に依頼可
  - 高確度 handoff: 購入導線を先に出し、コード共有は購入後

9. 最後に1文で次アクションを示す。
  - `見積もり提案を送ります`
  - `見立てを返します`
  - `このまま進められるかを返します`
  - ただし、見積もり相談はチャットに近いので、`.env注意` や `2時間以内` を毎回固定で足さない
  - 高確度案件は `あいさつ + 結論 + 購入導線` の短い構造で終えてよい
  - 最終自然化のあとも、`required_moves` が残り `forbidden_moves` を踏んでいないかを確認する
  - 最終自然化のあとも、section order・価格・ask 数・hold reason・次アクションが壊れていないか再チェックする
