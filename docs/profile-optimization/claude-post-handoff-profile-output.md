# handoff-25000 公開後プロフィール設計（Claude 出力）

作成日: 2026-04-08
前提: bugfix-15000 で数件レビュー取得後、handoff-25000 を公開した状態

---

## 1. 総合判定

**「少しだけ変える」が正解。本文の締めに1文だけ足す。**

ひとことアピールは変えない。本文冒頭も変えない。
構造を壊さず、最後のブロックに handoff への自然な導線を1文入れるだけ。

Codex の仮説「1文だけ足すくらいがよい」は**正しい**。

---

## 2. 理由

### 読みやすさ: 変えすぎると壊れる

v2 が効いている理由は「17行で構造が一目で分かる」こと（認知的流暢性）。
ここに handoff の説明を入れると、構造が崩れる。

```
今の v2 の構造:
  Block 1: 症状の具体例（2行）← buyer の自分ごと化
  Block 2: 経歴（3行）← 信頼性
  Block 3: 進め方（4行）← Competence を行動で示す
  Block 4: 制限 + 導線 + Warmth（3行）← 締め

これを壊さずに handoff を入れる場所は「Block 4」の中しかない。
```

### 安心感: bugfix が主語のままでいい

handoff-25000 の buyer は「コードが分からない」人。
でもこの人たちは、まず**ココナラで「コード整理」「引き継ぎ」で検索する**。
その時に表示されるのは**サービスページ**であり、プロフィールではない。

プロフィールに来る動線は:
1. サービスページを見る → 「この人は他に何をしてるんだろう」→ プロフィールを見る
2. 検索結果で名前を見る → プロフィールを見る → サービス一覧を見る

どちらのパターンでも、プロフィールの役割は
「**この人は何者か**」を1秒で理解させること。

2サービスを並列に書くと「何者か」が処理しにくくなる（流暢性が下がる）。
bugfix を主軸にして「他にもやってる」を匂わせる方が、**2パターン両方**に効く:
- bugfix buyer: 「おっ、私の問題にぴったり」
- handoff buyer: 「おっ、不具合修正もできるなら整理もちゃんとやれそう」

### 説得力: 2サービス持ちの「静かな強さ」

ここが最も重要な分析。

**2サービスを持っている事実は、サービス一覧ページで勝手に伝わる。**
ココナラの出品者プロフィールには「出品サービス」セクションがある。
ここに bugfix-15000 と handoff-25000 が並ぶので、buyer は自動的に
「この人は2つのサービスを持っている＝専門性がある」と認識する。

つまり:
- プロフィール本文で2つ並べて説明する → Parvenu型（「見て見て」）
- プロフィール本文は bugfix 中心、サービス一覧で2つ見せる → Patrician型（「分かる人が分かる」）

**Brand Prominence 理論の適用: 2つ持っていることを「声高に言わない」方が強い。**

ただし、完全に無視するリスクもある:
- サービス一覧を見ない buyer がいる
- 「この人は不具合修正しかやらないのか」と思われる可能性

→ よって「1文だけ触れる」がベスト。
  存在を示すが、説明はしない。
  「ちらっと見せて、気になったらサービスページに行ってもらう」。

### 何でも屋リスク: 1文なら発生しない

「何でも屋」に見えるのは、**2つのサービスを同じ重みで並列に説明した場合**。

例（NG）:
> 不具合修正と、コード整理・引き継ぎメモ作成の2つのサービスを提供しています。
> 不具合でお困りの方も、コードの引き継ぎが必要な方も、お気軽にご相談ください。

→ これは完全に「何でもやります」に見える。Quiet Luxury が崩壊する。

例（OK）:
> コードの整理・引き継ぎが必要な方向けのサービスもあります。

→ これなら「ん？そういうのもあるのか」で終わる。
  主語は bugfix のまま。handoff は従属関係。
  営業感がない。

### サービスページとの役割分担

```
プロフィール: 「この人は何者か」を1秒で理解させる
  → bugfix 中心で、handoff の存在を1文だけ示す

bugfix-15000 サービスページ: 不具合修正の詳細・料金・進め方・FAQ
  → プロフィールと重複させない

handoff-25000 サービスページ: コード整理の詳細・料金・進め方・FAQ
  → プロフィールと重複させない

出品サービス一覧: 2つ並ぶ
  → buyer が「この人は2つ持ってる」と自然に認識する
```

## 3. 最適な設計方針

### ひとことアピール: 変えない

```
Next.js / Stripe / API連携の不具合診断・修正を中心に対応しています。
```

理由:
- 検索結果で表示されるのはこの1文
- bugfix buyer にとって最強のキャッチ
- handoff buyer もこの1文で「技術的に強い人だ」と認識する
- 「中心に対応」の「中心に」が、他にもやっていることを暗示している（← これが絶妙）

### 本文冒頭: 変えない

症状ベースの導入は bugfix / handoff 両方の buyer に刺さる。
handoff buyer も「本番の不具合」に悩んでいることが多い。

### handoff をどこに入れるか: Block 4（締め）に1文

現在の v2 の Block 4:
```
通話対応・直接push・本番デプロイ代行は行っていません。
対応範囲や料金の詳細はサービスページをご覧ください。

ご相談だけでも歓迎です。まず状況を確認し、対応できるかをお返しします。
```

ここに1文足す。位置は「サービスページ導線」の後、「Warmth」の前。

## 4. 新プロフィール案（本命1案）

```
ひとことアピール:
Next.js / Stripe / API連携の不具合診断・修正を中心に対応しています。
```

```
本文:

「決済は通るのに反映されない」「本番だけWebhookが失敗する」「どこを見ればよいか分からない」
こうした症状の原因を特定し、修正まで対応しています。

個人で設計・運用しているWebアプリ（Next.js + Firebase、約20万行）で、
本番の不具合調査と修正を日常的に行っています。
AIで作ったコードや、外注・引き継ぎのコードでも、状況を整理しながら切り分けます。

進め方
・メッセージのみで進めます（通話なし）
・症状と再現手順をもとに原因を特定します
・「何が原因か・何を直したか・どう確認するか」を整理してお返しします
・最終確認は依頼者様の環境でお願いしています

通話対応・直接push・本番デプロイ代行は行っていません。
対応範囲や料金の詳細はサービスページをご覧ください。
コードの整理・引き継ぎが必要な方向けのサービスもあります。

ご相談だけでも歓迎です。まず状況を確認し、対応できるかをお返しします。
```

### 変更点（v2 との差分）

**1行だけ追加:**
> コードの整理・引き継ぎが必要な方向けのサービスもあります。

位置: 「サービスページをご覧ください」の直後、「ご相談だけでも歓迎です」の直前。

**この1文の設計意図:**

| 観点 | 分析 |
|---|---|
| 文量 | 18行 → 19行（+1行）。流暢性への影響はほぼゼロ |
| 情報量 | 「ある」と示しているだけ。料金・範囲・内容は一切書いていない |
| 営業感 | なし。「あります」で終わる。「ぜひ」「お気軽に」を使っていない |
| Quiet Luxury | 存在を匂わせるだけで、説明しない。Patrician型の典型 |
| 導線 | 「サービスページ」導線の直後に置くことで、buyer が自然にサービス一覧を見に行く動線になる |
| Warmth の分断 | 最終行「ご相談だけでも歓迎です」の直前なので、Warmth を壊していない |

## 5. 予備案（ひとことアピールも微調整する案）

**推奨しないが、検討の余地がある別アプローチ:**

```
ひとことアピール:
Next.js / Stripe / API連携の不具合修正と、コード整理・引き継ぎメモ作成に対応しています。
```

→ **不採用理由:**
- 25文字制限に近い（文字数オーバーの可能性）
- 2つ並べると「何屋か」が処理しにくい（流暢性低下）
- 検索結果で見た時に「焦点が定まらない人」に見える
- **今のひとことアピールの「中心に対応」がすでに完璧**

→ この予備案は「不採用の理由を明示するため」に出した。
  本命案の1文追加だけで十分。

## 6. Codex への引き継ぎメモ

### 絶対に壊してはいけない点
1. **ひとことアピールの文面** — 変えない。「中心に対応」の表現が2サービス体制でも成立する
2. **本文冒頭の症状ベース導入** — buyer の「自分ごと化」を誘発する核心
3. **進め方4行** — Competence を行動で示すアンカー
4. **最終行の Warmth** — 「ご相談だけでも歓迎です」は削除・移動禁止
5. **全体の文量** — 19行以下を維持。20行を超えたら何かを削る

### ルールへ戻すべき観点
- handoff-25000 をプロフィールに出す範囲: **1文のみ。サービス名・価格・範囲は書かない**
- 2サービスを「並列に」「同じ重みで」説明する表現は禁止
- この方針は `service-registry.yaml` の `public: true` / `false` とは別の観点
  （registry が true になっても、プロフィールでの扱いは「従属」のまま）

### 今後の拡張時の注意
- 3つ目のサービスを出す場合: プロフィールの1文を「他のサービスもあります」に一般化する
  → 個別サービス名をプロフィールに並べ始めると、Parvenu型に転落する
- 実績が20件を超えた場合: ひとことアピールを「販売実績〇〇件」に変える選択肢が出てくるが、
  **「何をする人か」が消えるので非推奨**。実績は出品サービス一覧で勝手に見える

### v2 → handoff公開後への切り替えタイミング
- handoff-25000 が `public: true` に変わった日にプロフィールを更新する
- それまでは v2（現行）を維持する
- 切り替えは「1文追加するだけ」なので、1分で完了する
