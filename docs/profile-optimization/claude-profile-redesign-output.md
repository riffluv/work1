### 問題2: 技術スタック羅列が Competence を過剰にアピール

### 問題3: 開発実績が buyer の関心と不一致
「開発実績」3項目（L37-40）は自分の作品リスト。

→ **判断: 削除。実力は「進め方」セクションで行動として示す。**

### 問題4: 経歴の畳み方が中途半端
L20-22の3行（20万行のアプリ、Stripe連携、AI/外注コード対応）は、
いい情報を含んでいるが、3行に分かれていて散漫。
「個人で運用している」という事実と「AI/外注コードにも対応」の2つだけが buyer にとって重要。

→ **判断: 2文に圧縮して残す。**

### 問題5: 「ひとことアピール」と「本文冒頭」がほぼ同一
ひとことアピール（L10）と本文1行目（L17）は内容がほぼ同じ。

→ **判断: ひとことアピールはそのまま。本文冒頭は別の切り口（症状ベースの導入）にする。**

### 問題6: 全体の文字量が Thin-Slicing で不利
現在の本文は約35行。ココナラのプロフィール画面で開いた瞬間、
buyer の脳は「情報が多い」と処理し、0.1秒で「ごちゃごちゃ」と判断する。
その後は確認バイアスで「やっぱりごちゃごちゃだ」と読む。

→ **判断: 15-18行に圧縮する。**

## 2. 改稿方針

### どんな印象を作るか
「この人は話が早そう。整理してくれそう。余計なことは言わないけど、ちゃんとやってくれそう。」

具体的に:
- **整理された人**（情報の構造が見た瞬間に分かる）
- **余裕のある人**（説明しすぎない = 自信がある証拠）
- **でも冷たくない人**（最後の1文で温度感を入れる）

### どこで納得感を作るか
- **症状の具体例**（「決済は通るのに反映されない」「Webhookが失敗する」）
  → buyer が「あ、これ私の状況だ」と感じた瞬間に納得感が生まれる
- **個人でWebアプリを運用している事実**（1文だけ）
  → 「この人は自分でも使ってるんだ」= 机上の空論じゃない

### どこで安心感を作るか
- **進め方セクション**（行動を3つ示す）
  → 「どう進むか分かる」= 不安の最大の解消剤
- **制限の明示**（通話なし、push なし）
  → 「やらないこと」が明確 = scope が見える = 安心

### どこで温度感を入れるか
- **最終行のみ**
  → 「ご相談だけでも歓迎です」は Warmth として十分
  → 温度感は1文で入れるから効く。3文入れたら嘘くさい

## 5. Codex への引き継ぎメモ

### 改善点の要約

| 観点 | 現在 | 改稿後 |
|---|---|---|
| 文量 | 35行 / 約800字 | 15-17行 / 350-400字 |
| Thin-Slicing | 文字量で「ごちゃごちゃ」判定リスク | 一目で構造が見える |
| 認知的流暢性 | 処理負荷が高い | 3秒で把握可能 |
| Brand Prominence | 技術スタック・実績列挙 = Parvenu型 | 削除 = Patrician型 |
| Warmth x Competence | スキル羅列で Warmth 低下 | 行動で示す + 1文の温度感 |
| サービスページ重複 | 対応内容5項目が完全重複 | 排除。症状例だけ残す |
| ひとことアピール重複 | 本文冒頭と同一文 | 切り口を変えて回避 |

### Codex 側で最終確認すべき点

1. **アルゴリズムの最低文字数**
   - 350-400字で不利にならないか？
   - ココナラ公式は「充実度」を推奨するが、明確な最低文字数は非公開
   - 不安なら B案（400字）を採用した方が安全

2. **「20万行」を完全に外した判断**
   - 本命案では削除した。理由: Brand Prominence 理論で Competence の直接アピールは Parvenu型に映るリスク
   - ただし、「20万行」は buyer の記憶に残る可能性もある
   - Codex が残すべきと判断するなら、「個人で設計・運用しているWebアプリ（Next.js + Firebase、約20万行）で」と、経歴文の中に自然に入れる形を推奨
   - セクション見出し「開発実績」の下に置くのは避ける（列挙感が出る）

3. **サービスページとの整合性**
   - プロフィールのひとことアピール文と、サービスページのタイトル「Next.js/Stripe不具合診断・修正します」が
     表現は違うが意味が一致している → これは問題なし（むしろ補強）
   - プロフィール本文の症状例とサービスページ「こんな症状に対応」で例が重なるが、
     プロフィール側は3つに絞り、サービスページ側は4つ → 意図的な情報量差 → OK

4. **handoff-25000 が公開された場合**
   - プロフィールの冒頭に handoff を並記するのは避けた方がいい
   - 理由: 2つ並べると「何でもやります」感が出て Quiet Luxury が崩れる
   - 推奨: ひとことアピールは bugfix 中心のまま維持し、
     本文の最終ブロックに「開発中のコードを引き継ぐためのサービスもあります」程度の1文を追加する
   - これは handoff 公開時に改めて判断する

5. **ルールへ戻すべき観点**
   - 「通話対応・直接push・本番デプロイ代行は行っていません」は AGENTS.md の厳守境界と一致 → 維持必須
   - Warmth 文の「ご相談だけでも歓迎です」は temperature_plan の warm range に適合 → 維持
   - 本命案の制限表現がサービスページ「標準範囲外」の記載と矛盾しないことを確認済み
