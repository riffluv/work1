# ココナラ公式段取りメモ（guide_market準拠）

更新日: 2026-03-02
参照: https://coconala.com/pages/guide_market

## 目的
- ココナラ取引の公式段取りを、実運用で迷わない形に固定する。
- 新しいCodex/Claudeセッションでも同じ流れで返信できるようにする。
- Internal OS の `coconala -> implementation -> delivery` 切替を崩さず進める。

## 出品者側の基本フロー（公式）
1. 購入後、まず挨拶と必要情報の依頼を送る
2. トークルームでやり取りを継続する
3. 連絡が滞ったら状況確認メッセージを送る
4. 要件を満たしたら「正式な納品」を送る

## Internal OS での割り当て
- 見積り相談: `coconala`
- 購入直後から修正完了まで: `implementation`
- 納品準備と正式納品: `delivery`
- 購入後に `#S` を使ったら、以後の正本は `ops/cases/open/{case_id}/README.md`

## 運用ルール（このworkspaceの固定）
- 初回返信: 24時間以内（通常数時間以内）
- 必要情報が揃い次第、48時間以内を目標に切り分け結果と修正方針を案内する
- 送信順序: 結論 → 根拠 → 次アクション → 相手への依頼
- 購入直後に「納品形式（標準: 修正済みZIP / 任意: patch）」を合意する
- 正式な納品は「要件を満たした状態」でのみ送る
- 差し戻しは1回前提で、初回納品時に確認方法を明記する

## 特例対応の固定ルール（内部）
- 基本料金15,000円の値引きはしない。
- 特例は「同一原因の軽微な追加調整」を短時間だけ吸収する。
- 別原因/別フロー/別エンドポイントは追加見積りに切り替える。
- 上限管理・文面・記録は `docs/coconala-special-case-policy.ja.md` を正本とする。

## 見積りの別提案運用（範囲外相談）
- 見積り相談で提示した金額は「提案を購入」する形。  
  サービス本体価格に上乗せされる二重決済にはならない。
- 見積り経路では有料オプションを使えないため、必要費用は提案金額にまとめて提示する。
- 現行サービス（不具合修正）と内容/価格が近い案件は、提案時に「出品サービスに紐付ける」を使う。
- 新規実装や新機能など、現行サービスと大きく異なる案件は別提案で扱う。  
  紐付け提案は、サービス内容/価格との乖離が大きいと制限される場合がある。
- 相談が範囲外だった場合の案内順は固定する:  
  1) 今回サービス範囲外である理由  
  2) 別提案で対応可能であること  
  3) 金額と納期の提示（提案1本）

## 4フェーズ別テンプレ運用
### 1) 購入時の挨拶
- 使うテンプレ: `docs/coconala-message-templates-short.ja.md` の「初回返信（購入直後）」
- 追加テンプレ: 同ファイルの「納品形式の確認（購入直後）」
- 実運用では `#S #R` を起点にして case を開始する
- 必須要素:
1. 受領のお礼
2. 必要情報の箇条書き
3. 一次報告の時刻
4. 納品形式の合意（標準: 修正済みZIP / 任意: patch）

### 2) やり取り滞留時
- 使うテンプレ: 同ファイルの「返信が止まったとき」
- 必須要素:
1. 前回連絡日
2. 返答してほしい内容
3. 期限

### 2-1) 途中報告
- 購入後の途中報告は `#P` を正本にする
- 旧 `#R 進捗返信` は互換用としてのみ残す
- 3行で「現状 / 今見ている所 / 次の動き」を返す

### 3) キャンセルが必要な時
- 理由を明確にし、感情的な文面を避ける
- 返金はココナラ側処理である旨を簡潔に添える
- 参考: `docs/writing-guideline.ja.md`

### 4) 正式な納品時
- 使うテンプレ: 同ファイルの「正式な納品メッセージ」
- 納品準備へ入る前に `delivery` mode へ切り替える
- 必須要素:
1. 納品物一覧
2. 確認手順
3. 差し戻し時の案内
4. クローズ条件の簡潔な説明
