# ココナラ短文テンプレ（実運用）

## 使い方
- `[]` を置換して送信
- 1通は「結論→次アクション→期限」の順
- 迷ったら短くして先に返信（48時間無連絡を避ける）
- 国内向け返信では `JST` を書かない（海外在住が明確な場合のみ `日本時間` を補足）。
- 見積り初回（資料未受領）は `2時間以内には` を使ってよい。
- 資料受領後は `この返信から` を使わず、`本日[時刻]までに` で時刻を明示する。

## 時刻未確定時の定型
- まず、2時間以内には見立てをお返しします。
- 他に必要なことがあれば、こちらからお聞きします。

## 購入後フェーズの時刻未確定定型
- 本日[時刻]までに、次の確認結果をお返しします。
- 初回受領や本格着手の時だけ、そのあと48時間以内を目標に一次結果をご案内します。

## 受領後3段階の定型
- 受領直後: ありがとうございます。受け取りました。いまから確認に入りますね。
- 中間報告（30〜60分目安）: 確認を進めています。[確認ポイント]の部分を見ています。本日[時刻]までには見立てをお伝えできそうです。
- 時刻コミット: 本日[時刻]までに、見立てをお伝えします。

## `#R`ショート呼び出し（運用用）
- `#R 受領返信`: 受領直後の1通を作成
- `#R 進捗返信`: 30〜60分の中間報告を作成（旧互換。新規運用では使わず `#P` を優先）
- `#R 方向性返信`: 最初の見立て（原因候補 + 次アクション）を作成
- `#R 差し戻し返信`: 差し戻し受領後の返信文を作成
- `#R クローズお礼`: クローズ前後のお礼文を作成
- `#R` の後ろや次行に自由文の補足を書いてよい。補足は optional な override として扱い、tone / ask / 返答順へ反映する。
- 例:
  - `#R 受領返信 ここは柔らかめに`
  - `#R` の次行に `追加料金の話は今回は避けて`
- 時刻未指定なら、JST現在時刻 + 2時間を自動で挿入（`本日HH:MMまで` / 日付またぎは `明日HH:MMまで`）

## `#P`ショート呼び出し（こちら起点文面）
- `#P` は buyer の文章に返す返信ではなく、**こちら起点で buyer に送る文面** を作成する
- デフォルトの `#P` は購入後トークルームの途中報告として扱う
- 用途を明示する時は、次のように使う
  - `#P 途中報告`
  - `#P 着手報告`
  - `#P 追加情報依頼`
  - `#P 補足`
  - `#P 完了報告`
  - `#P 納品前確認`
  - `#P 正式納品案内`
  - `#P 承諾前フォロー`
- 購入後の途中報告は `#P` を正本とする
- `#R 進捗返信` は既存メモや旧会話を再現する時だけ使う
- 途中報告の骨格は原則3行
- 途中報告テンプレ本文は `## 3. 進捗報告` を正本として参照する
- 未確認の原因断定 / 進捗率 / ログ生貼りは入れない
- 技術語は最大1つまでを目安にし、非エンジニア相手には流れの言葉へ言い換える

## 末尾ローテーション定型（2時間/48時間）
- 本日[時刻]までに見立てをお伝えします。  
そのあと、48時間以内を目標に一次結果をご案内します。
- 本日[時刻]までに、まず次の確認結果をお返しします。  
続けて、48時間以内を目標に一次結果をご案内します。
- 本日[時刻]までに判定の見立てを共有します。  
その後、48時間以内を目標に一次結果をご案内します。

## 購入後の1段コミット定型
- 本日[時刻]までに、現時点の見立てをお返しします。
- 本日[時刻]までに、確認結果をお返しします。
- 本日[時刻]までに、修正版の見通しをお返しします。

## 完了確認フェーズの末尾定型（二段構成不要）
- ご返信いただければ、すぐに次の進行を確定します。
- A/Bどちらかご返信いただければ、このまま進行を確定します。

## 9. 自分用チェック（送信前）
- 依頼範囲外を無料で混ぜていないか
- 次アクションと期限が書いてあるか
- 外部連絡や秘密情報要求になっていないか
- 追加質問の前に受領確認の1文（「確認しました」など）を入れているか
- 文章が長すぎないか（要点先頭）
- 1つの質問項目に依頼を詰め込みすぎていないか（1項目1情報）
- 非エンジニア向けで「ログ」単語を単独要求していないか
- エラー文は `テキスト優先 / コピー不可ならスクショ` の順で案内しているか
- 「対応範囲外です」で終わらず、代替案を添えているか
- 「見込み + 別原因は事前相談」の文が連続で同一になっていないか
- 見積り初回で料金を並べすぎていないか（基本料金 + 追加は事前相談まで）
- セキュリティ注記が「禁止 -> 代替 -> 事後対処」の順になっているか
- 判定前に追加修正の具体金額を先走って出していないか
- 購入後フェーズの返信に、次回時刻（本日中 / [時刻]まで / 2時間以内※資料未受領時）が入っているか
- 責任/保証の回答が「保証範囲 -> ただし -> 手順」の3段構成になっているか
- 初回受領や本格着手では `本日[時刻]まで` と `48時間以内` の二段構成が入っているか。短い追加返信や再修正では1段で十分か
- 完了確認フェーズで二段構成を機械適用していないか
- 正式納品メッセージで「約3日間未操作なら自動承諾」を平易に案内しているか
- 評価依頼を送る場合、1回のみの任意トーンになっているか
- `2時間以内` が「受領確認 or 一次報告」の意味で使われ、一次結果の時刻が別で明記されているか
- 追加質問の末尾に「コピーしにくければスクショ1枚でOK」のフォールバック文を入れたか
- `対応していません。` で終わらず、柔らかい表現 + 代替案になっているか
- `補償` / `損害補償` を先に使わず、`責任範囲` / `対応方針` で書いているか
- 書き出しと末尾2行が同じ表現で連続しすぎていないか（3件連続を避ける）
- `[ ]` / `{ }` のプレースホルダーが残っていないか
- 購入者の適切な行動（バックアップ取得・情報整理など）を拾って1文で認めているか

## 10-5. 見積り判定チェック（自分用・送信前）
- サービス適合: `high / medium / low / 範囲外`
- 原因見込み: `単一 / 複数 / 不明`
- 情報充足: `判定可能 / 追加質問が必要 / 判定不能`
- 工数見込み: `基本料金内 / 範囲追加の可能性あり / 新規実装級`
- 判定が割れたら、追加質問は1回まで
- 2回目も判定不能なら、辞退または人手確認へ倒す
- 見積り経路では有料オプションを後から足せない前提で、提案金額を決める
- 原因候補の詳細説明や修正方針の提示は購入後に回す

## 14. 見積り送信前チェック（自分用）
- 依頼経路を確認したか（サービス経由 / プロフィール経由 / メッセージ経由）

- 相手が返しやすい選択肢（A/Bなど）を入れたか
