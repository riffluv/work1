今の主目的は、ココナラ返信システムのリハーサル継続です。大改造フェーズはほぼ終わっており、いまは「サービス内容とココナラ仕様と購入者心理が噛み合った返信を、実戦で崩さず出せるか」を一件ずつ確認する段階です。返信文だけでなく、必要なら再発性のあるルールを `style-rules` / `self-check` / `golden-replies` に最小で反映してください。ただし、構造を増やしすぎないこと。新しい大きな layer は、同種の失敗がまた複数回出るまでは追加しません。

前提として、外向け live は `bugfix-15000` のみです。`handoff-25000` は未公開で、外向けにサービス名・価格・購入導線を出してはいけません。ただし、ユーザーが「25000サービス開始後という設定で」と明示したリハーサルでは、その仮定で返信文を作って構いません。公開状態の正本は [service-registry.yaml](/home/hr-hm/Project/work/os/core/service-registry.yaml) です。

この workspace で既に入っている重要な構造は、`reply_stance` と `reply_contract` です。`reply_stance` は `answer_timing / burden_owner / empathy_first / reply_skeleton`。`reply_contract` は `case_type / certainty / issue_plan / required_moves / forbidden_moves` を含みます。これらは schema, intake-router, judgment-flow, reply skill, self-check に反映済みです。次の Codex は、これ以上の新設計を足す前に、まず既存の `reply_stance` と `reply_contract` が本当に返信文に効いているかをリハーサルで確かめてください。

現時点の大方針:

1. 見積もり相談では無料ラインを越えない
- prequote で深い分析や「今すぐ止めるべきこと」まで無料で返さない。
- 見積もり相談でやるのは、`15,000円で見られるか` / `まだ言えないので1〜2点確認` / `別入口` の入口判定まで。

2. 価格は曖昧にしない
- `15,000円で対応できるか` が主質問なら、3択で返す:
  - `この内容なら15,000円で進められます`
  - `この時点では15,000円で進めるとはまだ言えないので、先に1〜2点だけ確認します`
  - `15,000円ではなく別の入口です`
- `見込みです` `可能性が高いです` で金額を濁さない。
- ただし、主質問が値引き可否なら `進められます` ではなく `このケースでも15,000円です` のように価格そのものへ答える。

3. 追加料金説明は「進め方」に同意を取る
- 相手が `15,000円だけ払って終わるのが不安` と言った時だけ、強い説明を使う。
- その時は
  - `別原因ならそこから先には進めず`
  - `まず分かった内容を説明し`
  - `追加で進めるか、この時点で止めるかを相談する`
  - `自動で追加料金は発生しない`
 まで書く。
- 最後は `この進め方でよければ、見積もり提案をお送りします` で閉じる。
- これはユーザーが「魔法の言葉」と呼んでいる強い締めです。かなり重要です。

4. 感情には原則深く触れない
- `もっともだと思います` `お急ぎだと思います` `その聞き方で大丈夫です` などは避ける。
- 基本は `ご相談ありがとうございます。` `メッセージありがとうございます。` のような短い受領 + 本題。
- ただし、セキュリティ不安・苦情寄り・影響範囲不明で、受領だけだと冷たく見える場面では最小限の受け止めを入れてよい。
- 品質フィードバックは別扱い。`もっと検証してほしかった` のような指摘は感情として流さず、`前回は検証まわりの説明が不足していて、失礼しました` のように短く認めてから本題へ進む。

5. 判断を相手へ返しすぎない
- おまかせ系、どちらのサービスか分からない系は、こちらが持つべき判断を相手に返さない。
- ただし、本人しか決められないことは返してよい。
- `この進め方でよければ` は、条件を出し切った後の確認なのでOK。曖昧な丸投げではありません。

6. 不自然な日本語の再発ポイント
- `自然です` `合っています` `分かりやすいです` などの判定形容詞を使わない。
- `A の B として` の名詞詰め込みを避け、`A で B を/まで` に分解する。
- 1文に `の` が3回以上なら見直す。
- 質問リストの後を `いただければ` だけで宙に浮かせない。`教えていただければ` など動詞まで書く。
- `判断しやすいです` などこちらの内部事情で締めない。`対応できるかどうかをお返しします` など相手に返す結論で閉じる。

7. purchased / closed で特に大事なこと
- purchased で見積もり相談の骨格に戻らない。
- 追加相談には、まず短く反応してから本題。
- `ご相談します` を使う時は、何を相談するのかまで明示。
- `別途ご案内します` だけで止めない。
- closed 後は、値引き期待を先回り否定しない。

8. 保存運用を忘れない
- ユーザーが `#R` を付けて返信文を求めたら、必ず `./scripts/reply-save.sh` で
  - `runtime/replies/latest.txt`
  - `runtime/replies/latest-source.txt`
  を更新すること。
- 直近の私はここを何度か漏らしました。次では必ずセットで行ってください。

9. 外部AIレビューの使い方
- ユーザーは Claude を頻繁に使います。
- Claude は日本語の違和感や助詞、温度感、構文のバグを拾うのが強いです。
- ただし、ココナラ仕様・購入者心理・追加料金の進め方のような根本論点は、ユーザーの指摘の方が鋭いことが多いです。
- Codex は主軸。Claude 提案は、妥当性と再発性があるものだけ採用して rule に戻してください。
