# ココナラ返信 Self-Check

目的:
- 本番で Claude を毎回使わなくても、Codex が最低限の日本語品質チェックをできるようにする

## 最小5項目
- 相手の主質問に直接答えているか
- 購入前なのに `トークルーム添付` を書いていないか
- 価格・スコープ・規約の条件が崩れていないか
- 秘密情報の案内が必要な場面で抜けていないか
- Ban 表現（特に `形になります` `その代わり` `この文面だけだと`）が残っていないか

## 送信前チェック
- 相手の質問に直接答えているか
- `可能ですか` `できますか` `大丈夫ですか` のような Yes/No 質問には、流れ説明より先に結論を返しているか
- `そのまま続けて頼めますか` `別料金で修正までお願いできますか` のような継続可否質問には、先に `はい、可能です` / `現状では難しいです` を返しているか
- 相手がすでに書いた情報を聞き直していないか
- 見立て・確認観点を1文入れたか
- 質問は本当に必要か。今ある情報で答えられるのに、念のためで増やしていないか
- 質問数は 0〜3点の範囲に収まっているか。3点を機械的な標準にしていないか
- 時刻目安を入れたか
- 直近3件と同じ書き出し・末尾になっていないか
- 見積もり相談なのに、返信構造を毎回同じ5段（あいさつ / 受け止め / 見立て / 質問 / 定型フッター）で機械的に組んでいないか
- `.envやAPIキーの値は不要です` を、相手がコード・ログ・キー共有に触れていない場面まで機械的に入れていないか
- `2時間以内` を、毎回固定の定型フッターとして入れていないか
- route に合った書き出しになっているか（`service / profile / message`）
- `.env` 注意文のあとに結語が抜けていないか
- 相手が書いた内容と矛盾する質問になっていないか
- `この文面だけだと` `この内容だけだと` のように、相手の説明不足を評価する言い方になっていないか
- 選択肢型質問を使う場合、相手が使った語を優先し、相手が言っていない語を先出ししすぎていないか
- 診断上必要な質問と、テンプレ都合の先回りを取り違えていないか
- エラー文やログを、テキストで受け取れる場面なのに機械的にスクショ優先にしていないか
- UI状態の確認と、エラー内容の共有を混同していないか
- 資料依頼が `送れるか` `出せるか` のような能力確認っぽい聞き方になっていないか
- `情報` だけで依頼していないか。スクショ / 画面 / 注文番号 / エラー文 / いつ頃 など、相手がそのまま出せる具体物で聞けているか
- 相手の主質問が `料金内に収まるか` `対応可能か` の段階なのに、いきなりZIP一式の提出を前提にしていないか
- 購入前の見立てで足りる場面なのに、購入後レベルの重い解析依頼をしていないか
- 購入前にZIPを見る場合、構成確認（フォルダ構造・技術スタック・規模感・主要ファイル名）までに止める前提になっているか
- 購入前のZIP確認で、コードの中身の読解・原因特定・フロー整理まで踏み込んでいないか
- 見積り相談なのに、進行中の依頼のような `まずは次の3点だけお願いします` を機械的に使っていないか
- 高確度案件なのに、提案に進める前に習慣的な情報回収をしていないか
- 情報不足を伝える時に、相手の不足ではなく `今の時点では` `判断にもう1点必要です` のように時点や判断条件の問題として書けているか
- non_engineer 向けで内部語や強すぎる技術語が混ざっていないか
- 本文の中段も含めて Ban 表現が残っていないか
- Stripe不使用ケースなら、対応可能か対象外かを一言で明示したか
- 金額を出しているのに、同じ文面でヘッジしていないか

## Ban チェック
- `整理してお返しします` が残っていないか
- `もっともです` が残っていないか
- `ように見えた` が残っていないか
- `確認案件` が外向けに残っていないか
- `返金` と `承知しました` が近接していないか
- `安全です / 自然です / 合っています / 分かりやすいです` の判定形容詞が残っていないか
- `別件として整理する` `スコープを切って` など内部語が残っていないか
- `確認案件` `修正案件` `30分の範囲で` が外向けに残っていないか
- `整理サービス` が外向けに残っていないか
- `対象外としてのご案内になります` のような官僚調の締めが残っていないか
- `〜自体は` が残っていないか
- `〜する形になります` が残っていないか
- routing 説明で `〜する形になります` に逃げていないか。必要なら `〜で進めます` `〜が先です` `〜を先に見ます` に置き換えているか

## ケース別チェック

### 見積り初回
- 質問前に見立て1文を入れたか
- 質問の導入が `お願い` ではなく `見立てを出すための理由提示` になっているか
- 質問導入が機械的に `見立てを出しやすいです` 固定になっていないか。標準は `もう少し具体的にお答えできます` も使えているか
- `本来はどの〜想定でしょうか` の固定構文に戻っていないか
- 情報不足かつ金額未提示の時だけ `確認の入り口だけ` を使っているか
- 5,000円相談なら Golden Replies §2 の型に寄せているか
- 超短文相談では、回答例を1行入れて返しやすくしているか
- 即購入圧がある時も、必要なら `購入前に見立てだけ先に返す` 逃げ道を残せているか
- 開発中サイトなら `壊れた不具合` か `まだ作り込み中` かを分ける質問が入っているか
- 番号付きで複数質問が来ても、固定条件で即答できるもの以外まで無理に全部答えていないか
- `急ぎではないが放置しにくい` 相談に、急ぎ案件の温度感をそのまま当てていないか
- 相手が `料金内に収まるか` を聞いているときに、まず価格判定軸へ答えず、購入後前提の資料回収へ寄っていないか
- ZIPやコード一式の依頼は、相手が既に送付可能と言っているか、見積り相談段階でも必要性が高い場合だけに絞れているか
- handoff では、対象フローが具体的で受注確度が高いなら購入前にZIP依頼を出してよいことを意識できているか
- ただし、対象フローが明確で `25,000円で進められる` と即答できる高確度 handoff では、ZIP依頼より先に購入導線 (`見積もり提案を送ります`) を出せているか
- 相手が `お願いしたい` `発注したい` と明言している高確度 handoff なのに、購入導線を出さず情報回収だけに進んでいないか
- 相手がすでに対象フロー・目的・資料提供可否を十分具体的に書いているのに、価値の薄い確認質問を足していないか
- 確認質問が、相手がまだ決めていないことに向いているか
- 購入前の返信で `トークルーム添付` と書いて、まだ存在しない購入後の場を前提にしていないか
- bugfix では、症状・環境・エラー文で見積り判断できる限り、購入前にZIPを求めず `ソースコードはご購入後に` と安心材料を入れられているか
- ZIP依頼を出すときに、`どこまで整理できそうかを具体的に返しやすい` など理由を1文添えているか

### 感情注意
- 不便の認知があるか
- 感情の要約承認になっていないか
- 最初の依頼が重すぎないか
- 金額に触れた相手なら、その論点を無視していないか
- 自責が見える相手には、責めない一文があるか
- 自責には `謝らなくていい` ではなく `その聞き方で大丈夫です` のように行為肯定で返せているか
- 緊急案件では、時間を断定せず `優先して確認する` など優先度の約束で返せているか
- anxious × handoff で `自然です` に逃げていないか。必要なら `その進め方で大丈夫です` `よくあるご相談です` `そのまま進めて大丈夫です` に置き換える
- 感情応答は初手中心にし、2通目以降は要件優先に寄せているか
- 2通目以降も感情応答を繰り返して、判定や次アクションが薄くなっていないか

### quote_sent
- 聞かれたことに結論から答えているか
- ココナラ仕様の質問で `条件による` だけで終わっていないか
- 価格不安の相談では、件数判定のロジックを先に見せられているか

### closed 後再発
- 確認依頼文が毎回同じになっていないか
- 自責寄りなら、短い安心提供があるか
- 金額不安なら、方向性だけは触れているか

### 範囲外
- `別件で受けられる` と `現在は受けない` を混同していないか
- 次の行動を1文で示しているか
- `断り -> 提案 -> 再度断り` の構造になっていないか
- 秘密情報や外部共有を止める時に、止めるだけで終わらず代替の共有方法を示しているか
- できないことを伝える時に、同じ文か直後の文で「できること」も示せているか
- `その代わり` で代替案のように見せず、サービスの通常の進め方として説明できているか
- 相手が希望した進め方を断った直後に情報回収へ進む場合、`もしこの進め方で問題なければ` など受け入れ余地を1行入れているか
- 範囲不確定なら、金額を先に決め打ちせず「見られる内容かどうか」「どちらの入口が先か」を返せているか
- セキュリティ事案では、調査前に `今すぐ止められるか` を確認しているか
- 外部ツール停止では、禁止だけでなく `ここで進めた方が早い` など実務メリットに言い換えられているか
- 返金や保証の質問では、約束は保留しつつ `先に判断できること` を返せているか
- 秘密情報がすでに送信済みなら、削除/編集できるなら依頼し、できない場合はキーの無効化・ローテーションを依頼するフォールバックまで出せているか
- 秘密情報がすでに送信済みのとき、警告から入らず `詳しい情報ありがとうございます` など受け止めを1行置いてから止めているか
- 秘密情報停止の文で、backtick付きのキー名列挙が不必要に警告文っぽくなっていないか
- セキュリティ寄りに見える相談でも、即断せず `何が起きているかを優先して確認する` 形に落とせているか

### 境界ケース
- 書き出し直後に `内容を確認しました` `承りました` などの受け止め1行があるか
- 受け止め1行が `内容を確認しました` に固定しすぎていないか
- `〜の方が合っています` を使っていないか
- `自然です` `適切です` のような判定語で締めていないか
- handoff を先に勧める場合も、その後の bugfix 導線を閉じていないか
- `おまかせで` と言われた時に、まず受け止めてから具体的な進め方を示せているか
- `15Kか25Kか` を聞かれた時に、判定軸を先に示してから必要情報を回収しているか
- 複数サービス名が出ていても、そのまま返さず `メール` `認証更新` など機能に言い換えて伝えられているか

### purchased / closed
- `ついでに` 追加された複数論点を、関連あり / 範囲外で分けて返せているか
- 進捗確認 + 別件追加 + 外部連絡提案のような複数話題では、重要度に応じて返答量を配分できているか
- 2通目以降では `可能性があります` を続けず、証拠が増えた段階で判定を前に出せているか
- 2通目以降の結論が、1通目と同じ保留表現のまま停滞していないか
- 購入後や進行中の確認では、逆に遠慮しすぎず `次の3点だけお願いします` と明確に依頼できているか

### handoff 系
- `主要1フローの整理` として伝わっているか
- `修正は含まない` を必要以上に強く言いすぎていないか
- 修正依頼との混同が起きそうなバッチでは、少なくとも一部で `修正は含めず` `まずは整理に絞る` が入っているか
- `形になります` に逃げず、`〜で進めます` `〜します` で言い切れているか
- `工数` `仕様整理` などの管理用語が外向けに混ざっていないか
- `合っています` が handoff の適合説明で復活していないか
- 質問1が毎回 `いちばん整理したい流れ` に固定されていないか
- 相手がまだ flow を特定していない段階で、こちらから `決済 / 会員状態 / メール送信` のように狭い選択肢へ早く閉じていないか
- セキュリティ不安や現状把握の相談では、まず `現状整理 / セキュリティ確認 / 明確な不具合の有無` など広い切り口で聞けているか
- 選択肢型質問の語彙がテンプレ固定になっていないか。相手が使った語を優先し、相手が言っていない語は先出ししすぎない
- 結語が毎回 `進め方をお返しします` だけになっていないか
- 独自の値付け提案を断る時に、相手の意図まで否定していないか
- 2つの期待や質問が来た時に、両方を同じ強さで肯定しすぎず、明確肯定と部分肯定を使い分けられているか
- `まず整理、必要ならそのまま修正` を聞かれた場合に、別サービスの買い直し感を先に出さず `このトークルーム内で追加料金の修正として続けられる` と案内できているか
- その案内でも、`25,000円は整理まで` `修正は別料金` の境界を落としていないか
- handoff を段階的に進める説明で、作業内容だけでなく `引き継ぎメモを先に渡す` など納品物まで明示できているか
- `25,000円で全部収まるか` を聞かれたときに、`全部対応できるか` と `今回の範囲全体の目安` の両方に答えているか
- `25,000円には収まりにくい` だけで止めず、なぜ1フローを超えるのかを `別ファイル / 別連携に分かれやすい` などの具体語で説明できているか
- `どれにしますか` と迫る前に、相手が納得できる理由づけが入っているか
- 分ける理由を `安いから無理` ではなく `整理の精度を保つため` と品質理由で伝えられているか
- 相手が本当に聞いているのが `全部任せられるか` なら、`収まりません` から始めず `全部対応できます` を先に返す方が自然ではないかを確認したか
