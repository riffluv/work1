# ココナラ返信 Self-Check

目的:
- 本番で Claude を毎回使わなくても、Codex が最低限の日本語品質チェックをできるようにする
- 常時運用では `/home/hr-hm/Project/work/docs/reply-quality/self-check-core-always-on.ja.md` を先に使い、この文書はケース別リファレンスとして降りる

## 最小5項目
- 相手の主質問に直接答えているか
- 購入前なのに `トークルーム添付` を書いていないか
- 価格・スコープ・規約の条件が崩れていないか
- 秘密情報の案内が必要な場面で抜けていないか
- Ban 表現（特に `形になります` `その代わり` `この文面だけだと`）が残っていないか

## 送信前チェック
見る順:
1. primary answer integrity
2. phase integrity
3. evidence / ask integrity
4. asset reference integrity
5. compat-only residual

### primary answer integrity
- 相手の質問に直接答えているか
- `case_type / certainty / reply_stance / reply_contract / temperature_plan` の判定と、最終文面の骨格が矛盾していないか
- `reply_contract.primary_question_id` に対応する質問へ、最初の answer-bearing section で答えているか
- 最初の answer-bearing section が、primary question 以外の二次論点で始まっていないか
- 相手が価格を聞いていないのに、価格を最初の answer-bearing section へ差し込んでいないか
- 相手が別原因や追加料金を不安視していないのに、その説明を先頭へ出していないか
- `reply_contract.explicit_questions` を取りこぼしていないか
- `reply_contract.answer_map` と本文の `今答える / 確認後に返す / 依頼する / 断る` が一致しているか
- `response_decision_plan.phase_act` がある時は、`state` と `reply_skeleton` に合っているか
- `phase_act: estimate_answer / estimate_hold` なのに、購入後の確認開始報告のような現在形へずれていないか
- `phase_act: purchase_check_started / purchase_scope_recheck` なのに、見積もり相談の受付文面へ戻っていないか
- `prequote` なのに、コード・ログ・ZIP・権限を受け取って見始めた後のように読める動詞（`確認を進めます` `見ます` `原因を確認します` `コードを見て返します` など）が混ざっていないか
- `prequote` の closing が、`見ます` `原因を確認します` `進めます` のような購入後動詞で止まっていないか。今の段階で返せるのが `対応できます` `進められます` `返せます` のどれかに寄っているか
- `prequote` で `確認できます` が、見積もり判断ではなく確認済み・確認開始の意味に読めていないか
- `prequote` の追加質問が、`この情報があれば見積もれる / 対象かどうかを判断できる` ではなく、購入後の確認開始報告へずれていないか
- `purchased` なのに、見積もり相談のような `対応できるか判断します` `対象かどうかを見ます` `まず見積もりします` に戻っていないか
- 状況の受け止めと、価格・件数・範囲の判定を同じ文でつないでいないか。`急ぎの状況とのこと、15,000円で進められます` のように、別軸の情報が因果に読める並びを避けているか
- 受け止め句の直後に `不具合1件として対応できます` `15,000円です` `対象外です` を同文で続けて、意味の軸が混ざっていないか
- `user_override` がある時は、その意図が hard constraints を崩さない範囲で本文へ反映されているか
- `user_override` を十分に反映できない時は、公開状態・規約・セキュリティ・最低限必要な確認のどれが優先して残ったか説明できるか
- `temperature_plan.opening_move` が先頭1〜2文に反映されているか
- `temperature_plan.tone_constraints` にある禁止事項（`no_defense / no_internal_terms / no_burden_shift / no_negative_lead`）を本文で踏んでいないか
- `temperature_plan.ack_style = brief_specific` の時は、冒頭1文で buyer の具体的な行動・労力・結果・好意のどれかに触れられているか
- `temperature_plan.warmth_cap = one_sentence` の時は、温度のある受けが1文で止まり、本題へすぐ戻れているか
- `ここは柔らかめに` のような tone 指示がある時に、tone だけを動かして価格・可否・ask 数まで動かしていないか
- `質問は1つだけ` のような補足がある時に、実行に必要な minimum evidence を壊さない範囲で ask を最小化できているか
- `reply_stance.answer_timing` を reply-global の正本として扱わず、主質問の要約としてしか使っていないか
- `reply_contract.issue_plan` で `answer_after_check` にした論点を、本文で無理に断定していないか
- `answer_map` の `answer_after_check` に `hold_reason` と `revisit_trigger` があるか
- `fixed_slots` 相当の意味部（answer / hold / ask / next action / quoted facts）を、自然化の都合で書き換えていないか
- `required_moves` が本文に残り、`forbidden_moves` を踏んでいないか

### phase integrity
- `frontload_price_when_not_asked` があるのに、価格を主質問より先に出していないか
- `frontload_branching_risk_when_not_asked` があるのに、別原因や追加料金の話を先頭へ出していないか
- `frontload_scope_explanation_when_not_asked` があるのに、広い範囲説明を主質問より先に出していないか
- `issue_plan` や service facts にある正しい情報を、主質問より先に善意で差し込んでいないか
- `ask_map` にない質問を勝手に増やしていないか
- `可能ですか` `できますか` `大丈夫ですか` のような Yes/No 質問には、流れ説明より先に結論を返しているか
- Yes を返した直後の同じ文で、`〜なら` `〜だけなら` を付けて結論を弱めていないか。条件は次の文へ分けられているか
- `そのまま続けて頼めますか` `別料金で修正までお願いできますか` のような継続可否質問には、先に `可能です` / `現状では難しいです` を返しているか
- Yes/No へ直答する場面でも、冒頭が機械的な `はい、` になっていないか。`可能です。` `この内容なら対象になりそうです。` のように、フェーズに合う結論文そのもので始められる場面を逃していないか
- `税込みですか` `追加でかかりますか` のような Yes/No で答えられる料金質問に、制度説明だけで逃げず直接答えているか
- 料金質問で `ありきではなく` `前提ではなく` のような間接表現に逃げず、`かかりません` `かかります` `15,000円です` を先に返しているか
- 主質問に答えた直後の次アクションが、buyer の判断段階と合っているか。buyer がまだ価格や可否を決める段階なのに、こちらの情報回収フェーズへ前のめりに進めていないか

### evidence / ask integrity
- 相手がすでに書いた情報を聞き直していないか
- 相手がすでに出した技術情報や観測事実を、そのままもう一度 ask していないか。必要なら `その前後の文言` `その画面` `該当箇所` など、追加で必要な差分だけを頼めているか
- 見立て・確認観点を1文入れたか
- 質問は本当に必要か。今ある情報で答えられるのに、念のためで増やしていないか
- 任意確認を、必須質問や選択強制の形で押し出していないか
- 答えがなくてもこちらで進められる場面では、`なければこちらで〜から見ます` のような既定方針を書いたか
- 相手に優先順位づけや分岐判断を丸投げしていないか
- 相手が `自分では判断つかない` `どちらがよいか分からない` と言っている場面で、`もし進めるなら` `判断材料になると思います` のように判断を返していないか
- 不安だけでなく、`原因が分からない` の混乱や `以前お世話になった` の関係性など、相手が差し出した状況シグナルを最初の2文で拾えているか
- `迷っている` `比較して決めたい` `判断しづらい` のような buyer の判断段階表明がある時に、いきなり制度説明へ入らず1文だけ受けられているか
- 予算不安・本番障害・前任者と連絡不能のように相手のストレスが高い場面では、`確認しました` だけで受けず、`すぐ見ます` `15,000円の範囲で進める前提です。超える場合は止めて説明します` のように、行動と上限を誤解なく先に置けているか
- `answer_after_check` でも、いま見ている範囲を1文で可視化しているか
- 進捗不安への返答で、防御的な理由説明だけにならず、今後どう見える形で返すかまで書いているか
- 購入後の途中共有で、`何を見たか / 何が未確定か / 次に何を見るか / 次回時刻` のうち最低3点が入っているか
- 相手が `箇条書きで` `短く` `一言ずつ` のように返答形式を指定している時は、その形式を守っているか
- 相手が感謝や前向きな驚きを先に出している時は、主質問へ入る前に短い謝意を返せているか
- 相手が安心・好意・弱さを1行差し出している時は、本題の前に1文だけ受け止めを返せているか
- 相手が `星5` `大満足` `バッチリでした` のように強い好意を出している時は、実務案内の前に短い喜びや謝意を1文だけ返せているか
- `post_close` の再相談で、buyer が前回のお礼や遠慮を先に置いている時は、本題の前に短い受けを1文だけ返せているか
- buyer の具体的な行動や協力が見えているのに、`ありがとうございます。` だけで流していないか
- 相手が `高い` `予算が厳しい` など価格への引っかかりを率直に出している時は、説明の前に短い受け止めを1文だけ置けているか
- delivered / closed の問題報告で、`承知しました` だけで止まらず `確認します` など次の確認動作を先に出せているか
- 怒りや催促が強い途中共有で、技術的な現在地を `切り分けの軸` のような分析語で固くしすぎていないか
- 比較説明や価格差の返答で、`差額の説明だけすると` のような事務的な前置きが残っていないか
- 追加料金や追加フローの支払い方法を聞かれた時に、`追加支払いの形` や `取引状態を確認して` のような抽象案内で止まらず、buyer に見える状態語で案内できているか
- 別対応や追加料金の説明で、`勝手に進めず` のような防御的な語が残っていないか
- 根拠のない安心や、buyer の感情を言い当てる長い共感に寄っていないか
- buyer が使っていない感情語を補って、`迷いますよね` `焦りますよね` のような受けになっていないか
- 受け止めの1文を置いた直後に `ですが` で切り返して、受けた感を消していないか
- buyer の冒頭文をそのまま返して、おうむ返しの書き出しになっていないか
- 同一バッチ内で `ご連絡ありがとうございます。` `ご相談ありがとうございます。` の書き出しが完全固定になっていないか。phase は守りつつ、受領の粒度を少し散らせる場面を潰していないか
- `1行だけ` `5分だけ` `ちょっとだけ` のような作業量アピールに引っ張られず、分類で範囲判断できているか
- 低評価や不満の圧力に対して、評価ワードをなぞらず `進み方が見えにくい` など改善可能な事実へ受けを寄せられているか
- 相手がすでに共有した文脈を、別の言い方で言い直させていないか

### asset reference integrity
- facts / boundaries / decision-contract / evidence-contract / routing-playbooks / state-schema を見ずに、service 固有の価格・scope・判断・minimum ask・route・公開状態・納品物・secondary disposition を断定していないか
- bugfix / handoff の固定価格、scope unit、public/private、deliverables、repeat / refund / discount、deeper memo、repair_as_additional_work が service-pack と矛盾していないか
- bugfix の same-cause / different / undecidable の返し方が `decision-contract.yaml` と矛盾していないか。判定根拠より先に buyer 向けの結論をぼかしていないか
- bugfix の `原因不明時の扱い` と delivery gate が `decision-contract.yaml.unknown_cause_gate` と矛盾していないか。原因不明のまま正式納品へ進めるように読めていないか
- bugfix の最終確認・本番反映の責任所在が `decision-contract.yaml.final_confirmation` と矛盾していないか。こちらが最終確認や本番反映を引き受けるように見せていないか
- bugfix の `.env / secrets / prequote ask / non-Stripe / Stripe first-class evidence` の扱いが `evidence-contract.yaml` と矛盾していないか
- handoff の `one_flow / extra_flow / repair / undecidable` の判断が `decision-contract.yaml` と矛盾していないか。主目的が整理なのか修正なのか、主要1フローで止まるのか追加フローへ切るのかを buyer に見える形で返せているか
- handoff の `追加料金は自動発生しない / 整理だけで止められる / 修正は別対応` の返し方が `decision-contract.yaml.additional_work_policy` と矛盾していないか。追加対応を既定路線や買い直し前提に見せていないか
- handoff の formal delivery gate と follow-up の扱いが `decision-contract.yaml.delivery_gate` と `followup_window` に矛盾していないか。主フロー未確定のまま納品へ進めたり、deeper memo を基本料金内と読める形で出していないか
- handoff の prequote / quote_sent / purchased で聞く内容が `evidence-contract.yaml.phase_minimums` と矛盾していないか。prequote で full repo を要求したり、purchased で intended reader を落としたりしていないか
- handoff の `most_troubling_operation / intended_reader / extra_flow_or_repair_bundle / content_gap_after_delivery` の扱いが `evidence-contract.yaml` と矛盾していないか
- 外部共有・本番操作・設定変更・返金操作などの扱いが `boundaries.yaml` と矛盾していないか。誰が実行するかを曖昧にしていないか
- UI状態 / エラー共有 / scope-confirm / mixed symptom disposition / 対応可否の返し方が `routing-playbooks.yaml` と矛盾していないか
- unresolved thread や promised_next_action を state-schema 参照なしで閉じていないか
- delivered / closed で、secondary symptom・値引き不安・追加対応案内の行き先や返答タイミングを state-schema なしで曖昧にしていないか

### compat-only residual
- 時刻目安を入れたか
- 直近3件と同じ書き出し・末尾になっていないか
- 末尾の時刻コミットが `本日XX:XXまでに〜します` の同型ばかりになっていないか。時刻は残したまま、位置や言い回しを少し崩せているか
- `2時間以内` を、毎回固定の定型フッターとして入れていないか
- `.env` 注意文のあとに結語が抜けていないか
- `公開サービス` `この中で` `範囲で入る` のような内部スコープ語を、相手が聞いていない場面まで前に出していないか
- 相手が聞いていない `整理` `全体の整理` を、否定形でも持ち出していないか
- non_engineer 向けで内部語や強すぎる技術語が混ざっていないか
- 技術的に詳しい相手なのに、`timeout` `冪等処理` など相手が出した具体語を1つも拾わずテンプレだけで返していないか
- 非技術者でも、`再ログインすると直る` `朝だけ起きる` などの観察情報を1つ拾える場面で落としていないか
- 不安への返しで、相手が使っていない感情語や評価語（例: `責める`）を足していないか
- 相手が遠慮や不安を1行差し出している場面で、冒頭の受領直後に 1 拍の受け止めがあるか。事務連絡だけの場面まで機械適用していないか
- `可能性が高いです` のような確信度表現が、scope 判断の内部メモとして漏れていないか。会話として足りる場面では `よさそうです` に落とせているか
- `ここは優先して見ます` のような曖昧な指示語が残っていないか。対象が読める動詞へ落とせているか
- `流用` のような内部寄りの業務語が残っていないか。`分けて見る` `活かせる` など buyer が読める語へ言い換えられているか
- 同一バッチで `15,000円で進められます` が続きすぎていないか
- クロージングが `問題なければ` に固定しすぎていないか
- `原因を確認したうえで対応します` が機械的な標準文になっていないか
- 本文の中段も含めて Ban 表現が残っていないか
- naturalize 後にも、section order・価格・ask 数・hold reason・次アクション・forbidden claim を再チェックしたか
- 金額を出しているのに、同じ文面でヘッジしていないか

## Ban チェック
- `整理してお返しします` が残っていないか
- `もっともです` が残っていないか
- `もっともだと思います` `もっともな不安です` など、`もっとも` の変形が残っていないか
- `ように見えた` が残っていないか
- `確認案件` が外向けに残っていないか
- `返金` と `承知しました` が近接していないか
- `返金` や `減額` を、出品者がこの場で決められるように断定していないか
- `全額返金 / 全額支払いの二択` や `一部返金は不可` を platform fact のように断定していないか
- `正式な納品後は追加料金不可` を、公式仕様として断定していないか
- `安全です / 自然です / 合っています / 分かりやすいです` の判定形容詞が残っていないか
- `別件として整理する` `スコープを切って` など内部語が残っていないか
- `確認案件` `修正案件` `時間単位の範囲で` が外向けに残っていないか
- `整理サービス` が外向けに残っていないか
- `bugfix` `handoff` のような内部サービス名が外向けに残っていないか
- `現在の公開サービス` `公開中サービス` のような内部のサービス管理語が外向けに残っていないか
- `対象外としてのご案内になります` のような官僚調の締めが残っていないか
- `〜自体は` が残っていないか
- `〜する形になります` が残っていないか
- routing 説明で `〜する形になります` に逃げていないか。必要なら `〜で進めます` `〜が先です` `〜を先に見ます` に置き換えているか

## ケース別チェック

### 見積り初回
- 質問前に見立て1文を入れたか
- 質問の導入が `お願い` ではなく `見立てを出すための理由提示` になっているか
- 質問導入が機械的に `見立てを出しやすいです` 固定になっていないか。標準は `もう少し具体的にお答えできます` も使えているか
- bugfix の入口説明が、`/ops/services/next-stripe-bugfix/service-pack/facts.yaml` の `scope_unit` と `public_facts.includes` に矛盾していないか。`1フロー整理` のような別サービス化や、原因の確認と修正を不自然に分断した見せ方へずれていないか
- `原因不明でも料金だけかかるのか` の不安に答える場面で、`decision-contract.yaml.unknown_cause_gate` と矛盾していないか。`必要なら` `次の進め方` のような逃がし語へずれず、`分かったこと` と `修正まで進められるかどうか` を返す形になっているか
- buyer の恐れが `調べただけで終わる` `勝手に追加料金になる` `無料で見てもらえないか` のどれかなら、その恐れに対応する境界を直接答えているか。別の不安へすり替えていないか
- `調べただけで終わる` 不安に対して、`別対応にならない` ではなく、判断できないまま終わらせないことを返せているか
- `進められます` `このまま進められます` が、何を進めるのか読めないまま単独で残っていないか。対象や工程が近い文で明示されているか
- bugfix で複数症状や価格不安が同時にある時は、`/ops/services/next-stripe-bugfix/service-pack/routing-playbooks.yaml` の `prequote_bugfix_accept` と `prequote_success_rate_fear` に反して primary symptom や price-answer shape をぼかしていないか。primary を1つ置き、副症状は従属配置し、価格は `進められる / 1〜2点確認 / その入口ではない` のどれかに寄せられているか
- 上の3択で価格を言い切った直後の文で、`自然です` `合っています` などの判定形容詞に戻っていないか
