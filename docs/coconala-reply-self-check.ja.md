# ココナラ返信 Self-Check

目的:
- 本番で Claude を毎回使わなくても、Codex が最低限の日本語品質チェックをできるようにする

## 最小5項目
- 相手の主質問に直接答えているか
- 購入前なのに `トークルーム添付` を書いていないか
- 価格・スコープ・規約の条件が崩れていないか
- 秘密情報の案内が必要な場面で抜けていないか
- Ban 表現（特に `形になります` `その代わり` `この文面だけだと`）が残っていないか

## 送信前チェック
- 相手の質問に直接答えているか
- `case_type / certainty / reply_stance / reply_contract` の判定と、最終文面の骨格が矛盾していないか
- `reply_contract.primary_question_id` に対応する質問へ、最初の answer-bearing section で答えているか
- `reply_contract.explicit_questions` を取りこぼしていないか
- `reply_contract.answer_map` と本文の `今答える / 確認後に返す / 依頼する / 断る` が一致しているか
- `reply_stance.answer_timing` を reply-global の正本として扱わず、主質問の要約としてしか使っていないか
- `reply_contract.issue_plan` で `answer_after_check` にした論点を、本文で無理に断定していないか
- `answer_map` の `answer_after_check` に `hold_reason` と `revisit_trigger` があるか
- `required_moves` が本文に残り、`forbidden_moves` を踏んでいないか
- `ask_map` にない質問を勝手に増やしていないか
- `可能ですか` `できますか` `大丈夫ですか` のような Yes/No 質問には、流れ説明より先に結論を返しているか
- `そのまま続けて頼めますか` `別料金で修正までお願いできますか` のような継続可否質問には、先に `はい、可能です` / `現状では難しいです` を返しているか
- `税込みですか` `追加でかかりますか` のような Yes/No で答えられる料金質問に、制度説明だけで逃げず直接答えているか
- 相手がすでに書いた情報を聞き直していないか
- 見立て・確認観点を1文入れたか
- 質問は本当に必要か。今ある情報で答えられるのに、念のためで増やしていないか
- 質問数は 0〜3点の範囲に収まっているか。3点を機械的な標準にしていないか
- 時刻目安を入れたか
- 直近3件と同じ書き出し・末尾になっていないか
- 見積もり相談なのに、返信構造を毎回同じ5段（あいさつ / 受け止め / 見立て / 質問 / 定型フッター）で機械的に組んでいないか
- `.envやAPIキーの値は不要です` を、相手がコード・ログ・キー共有に触れていない場面まで機械的に入れていないか
- `2時間以内` を、毎回固定の定型フッターとして入れていないか
- route に合った書き出しになっているか（`service / profile / message`）
- `.env` 注意文のあとに結語が抜けていないか
- 質問リストの直後が `いただければ` だけで終わっていないか。`教えていただければ` `お答えいただければ` のように動詞まで書けているか
- 相手が書いた内容と矛盾する質問になっていないか
- `この文面だけだと` `この内容だけだと` のように、相手の説明不足を評価する言い方になっていないか
- 選択肢型質問を使う場合、相手が使った語を優先し、相手が言っていない語を先出ししすぎていないか
- 診断上必要な質問と、テンプレ都合の先回りを取り違えていないか
- エラー文やログを、テキストで受け取れる場面なのに機械的にスクショ優先にしていないか
- UI状態の確認と、エラー内容の共有を混同していないか
- 資料依頼が `送れるか` `出せるか` のような能力確認っぽい聞き方になっていないか
- `情報` だけで依頼していないか。スクショ / 画面 / 注文番号 / エラー文 / いつ頃 など、相手がそのまま出せる具体物で聞けているか
- 相手の主質問が `料金内に収まるか` `対応可能か` の段階なのに、いきなりZIP一式の提出を前提にしていないか
- 購入前の見立てで足りる場面なのに、購入後レベルの重い解析依頼をしていないか
- 購入前にZIPを見る場合、構成確認（フォルダ構造・技術スタック・規模感・主要ファイル名）までに止める前提になっているか
- 購入前のZIP確認で、コードの中身の読解・原因特定・フロー整理まで踏み込んでいないか
- 見積り相談なのに、進行中の依頼のような `まずは次の3点だけお願いします` を機械的に使っていないか
- 高確度案件なのに、提案に進める前に習慣的な情報回収をしていないか
- 高確度・単一質問なのに、相手がもう済ませたサービス選択や進め方の適合説明を後から足して長くしていないか
- 情報不足を伝える時に、相手の不足ではなく `今の時点では` `判断にもう1点必要です` のように時点や判断条件の問題として書けているか
- `判断しやすいです` `見やすいです` のように、こちらの内部事情で結んでいないか。相手に返す結論や次アクションで閉じているか
- non_engineer 向けで内部語や強すぎる技術語が混ざっていないか
- 相手が1テーマだけ書いているのに、`〜でしたら、この内容は` のような条件分岐の受け方をしていないか
- `〜件でしたら、不具合1件として対応できます。` が残っていないか
- 相手がすでに事実を報告しているのに、`〜なら` `〜であれば` のような仮定受けへ戻っていないか
- 保留確認で、相手がすでに答えた情報を質問へ入れ直していないか
- 2点テンプレを優先して、1点で足りる確認を増やしていないか
- 根本的な分岐が1つだけなのに、2問に増やしていないか
- 相手が使った語を、必要なく一段硬い語へ変換していないか（例: `押す -> 押下する`、`3回に1回 -> 断続的に`）
- 相手が使った動詞や症状語を、こちらの都合で別の語へずらしていないか（例: `登録できない` を `反映` に寄せていないか）
- 技術語だけ公式名へ置き換えて、相手が使った呼び方を消していないか（例: `カスタマーポータル` -> `Billing Portal`）
- `崩れる` を便利語として使って、相手の具体的な症状語（`エラーが出る` `失敗する` `止まる` `飛ばない`）を潰していないか
- 技術的に詳しい相手なのに、`timeout` `冪等処理` など相手が出した具体語を1つも拾わずテンプレだけで返していないか
- 非技術者でも、`再ログインすると直る` `朝だけ起きる` などの観察情報を1つ拾える場面で落としていないか
- `急ぎ` `至急` が明記されているのに、本文中で直接受けず作業説明だけに埋め込んでいないか
- prequote なのに、`まず確認します。` 単独で始めて購入後の進行中会話のような距離感になっていないか
- `service_mismatch` の断りが `対象外です` だけで冷たく閉じていないか
- 相手が `決済とは関係ない` と明言しているのに、`Next.jsの不具合として確認できます` のようにスコープを広く見せていないか
- 不安への返しで、相手が使っていない感情語や評価語（例: `責める`）を足していないか
- `返しやすいです` `確認したいです` のような提供者視点の語が残っていないか
- 返金・設定変更・デプロイなどの操作に触れる時、誰が行うかが曖昧なまま濁していないか
- 相手の質問を引用して、前文で答えた結論をもう一度言い直していないか（例: `対応できますか、との点には`）
- 同一バッチで `15,000円で進められます` が続きすぎていないか
- クロージングが `問題なければ` に固定しすぎていないか
- `切り分けたうえで対応します` が機械的な標準文になっていないか
- 本文の中段も含めて Ban 表現が残っていないか
- naturalize 後にも、section order・価格・ask 数・hold reason・次アクション・forbidden claim を再チェックしたか
- Stripe不使用ケースなら、対応可能か対象外かを一言で明示したか
- 金額を出しているのに、同じ文面でヘッジしていないか

## Ban チェック
- `整理してお返しします` が残っていないか
- `もっともです` が残っていないか
- `もっともだと思います` `もっともな不安です` など、`もっとも` の変形が残っていないか
- `ように見えた` が残っていないか
- `確認案件` が外向けに残っていないか
- `返金` と `承知しました` が近接していないか
- `安全です / 自然です / 合っています / 分かりやすいです` の判定形容詞が残っていないか
- `別件として整理する` `スコープを切って` など内部語が残っていないか
- `確認案件` `修正案件` `時間単位の範囲で` が外向けに残っていないか
- `整理サービス` が外向けに残っていないか
- `現在の公開サービス` `公開中サービス` のような内部のサービス管理語が外向けに残っていないか
- `対象外としてのご案内になります` のような官僚調の締めが残っていないか
- `〜自体は` が残っていないか
- `〜する形になります` が残っていないか
- routing 説明で `〜する形になります` に逃げていないか。必要なら `〜で進めます` `〜が先です` `〜を先に見ます` に置き換えているか

## ケース別チェック

### 見積り初回
- 質問前に見立て1文を入れたか
- 質問の導入が `お願い` ではなく `見立てを出すための理由提示` になっているか
- 質問導入が機械的に `見立てを出しやすいです` 固定になっていないか。標準は `もう少し具体的にお答えできます` も使えているか
- bugfix を `1フローを見ます` のサービスのように説明していないか。まず `同一原因の不具合1件として対応できるか` を前に出しているか
- ただし、`15,000円で対応可能か` への即答が主質問のときは、価格返答が `進められます / まだ言えないので先に1〜2点確認します / 別の入口です` の3択に寄っているか。`見込みです` `可能性が高いです` で曖昧にしていないか
- 上の3択で価格を言い切った直後の文で、`自然です` `合っています` などの判定形容詞に戻っていないか
- 主質問が値引き可否 (`もっと安くなりますか`) なのに、価格そのものへ答えず `進められます` `対応できます` でずらしていないか
- 状況復唱・条件分岐・価格可否を1文に詰め込みすぎていないか。見積り初回の長い条件チェーンは2文に切れているか
- `本来はどの〜想定でしょうか` の固定構文に戻っていないか
- 情報不足かつ金額未提示の時だけ `確認の入り口だけ` を使っているか
- 確認だけしたい相談でも、公開 bugfix では 15,000円 に原因切り分けと修正可否の確認を含める案内になっているか
- 超短文相談では、回答例を1行入れて返しやすくしているか
- 即購入圧がある時も、必要なら `購入前に見立てだけ先に返す` 逃げ道を残せているか
- 開発中サイトなら `壊れた不具合` か `まだ作り込み中` かを分ける質問が入っているか
- `本番環境はない` `Webhookの受け方が分からない` `作り方が分からない` の相談を、bugfix の受注前提で短絡処理していないか
- `設定だけなら自分でやる` と相手が書いている時に、設定問題かコード問題かを切る前に価格を断定していないか
- 分割払い、支払い方法変更、外部決済の相談を、こちらが決められるかのように書いていないか
- 相見積りや他者の安価実績を持ち出された時に、その比較に乗って価格交渉の文面へ崩れていないか
- 独自のバンドル価格提案を受けた時に、提供単位を崩して迎合していないか
- 番号付きで複数質問が来ても、固定条件で即答できるもの以外まで無理に全部答えていないか
- `急ぎではないが放置しにくい` 相談に、急ぎ案件の温度感をそのまま当てていないか
- 相手が `料金内に収まるか` を聞いているときに、まず価格判定軸へ答えず、購入後前提の資料回収へ寄っていないか
- `原因不明のまま15,000円だけかかるのでは` という不安に対して、`作業した分で終わり` のような読みに見える返しをしていないか
- 原因未特定・未修正のまま、こちらの判断だけで正式納品や完了へ進めるように読める文になっていないか
- 追加料金の可能性を説明するときに、`必要になることがあります` だけで終わらず、`その先の作業には進まず、まず説明して相談する` まで書けているか
- 追加料金そのものへの事前同意を取りに行っていないか。相手に確認しているのが `進め方` になっているか
- 上のフル展開を、不安表明がない高確度案件にまで機械適用して重くしていないか。コスト不安を明示したケースだけに使えているか
- セキュリティ不安や範囲未確定の見積り相談で、`今すぐ止めるべきこと` `原因の見立て` など実作業レベルの判断を、購入前に無料で返していないか
- 別原因や追加対応の説明で、`事前にご相談します` の `事前` が何に対してか曖昧になっていないか。購入後の話なら `ご購入後に` を明示しているか
- `ご購入後に` を近い2文で繰り返していないか。2文目は `別の原因が見つかった場合は` などに省いているか
- 相手がまだ言っていない広い範囲外を、見積り初回で先回り否定していないか。`伴走ではなく` などの防御表現より、まず今受けられる範囲を先に返しているか
- ZIPやコード一式の依頼は、相手が既に送付可能と言っているか、見積り相談段階でも必要性が高い場合だけに絞れているか
- handoff では、対象フローが具体的で受注確度が高いなら購入前にZIP依頼を出してよいことを意識できているか
- ただし handoff が `運用中` の時だけ、対象フローが明確で `25,000円で進められる` と即答できる高確度 handoff では、ZIP依頼より先に購入導線 (`見積もり提案を送ります`) を出せているか
- handoff が `運用中` の時だけ、相手が `お願いしたい` `発注したい` と明言している高確度 handoff なのに、購入導線を出さず情報回収だけに進んでいないか
- `docs/service-catalog.ja.md` で `登録済み・未公開` のサービス名・価格・購入導線を、外向け返信に出していないか
- `15Kか25Kか` を聞かれた時に、未公開側の価格や導線をそのまま返さず、判定軸へ言い換えて返せているか
- 相手がすでに対象フロー・目的・資料提供可否を十分具体的に書いているのに、価値の薄い確認質問を足していないか
- 確認質問が、相手がまだ決めていないことに向いているか
- 購入前の返信で `トークルーム添付` と書いて、まだ存在しない購入後の場を前提にしていないか
- bugfix では、症状・環境・エラー文で見積り判断できる限り、購入前にZIPを求めず `ソースコードはご購入後に` と安心材料を入れられているか
- ZIP依頼を出すときに、`どこまで整理できそうかを具体的に返しやすい` など理由を1文添えているか

### 感情注意
- 不便の認知があるか
- 感情の要約承認になっていないか
- 最初の依頼が重すぎないか
- 金額に触れた相手なら、その論点を無視していないか
- 自責が見える相手には、責めない一文があるか
- 自責には `謝らなくていい` のような上からの免罪や、`その聞き方で大丈夫です` のような不自然な行為肯定を機械適用していないか
- 自責が見える相手には、`まず今分かる範囲から確認します` `先に料金の話から確認します` のように、責めずに次アクションへつなげられているか
- 感情には原則深く触れない方針になっているか。受領だけで足りる場面に、余計な共感や感情説明を足していないか
- 不安や怒りが見えても、毎回そこを直接なぞらないと不自然にならないか。`メッセージありがとうございます。` のような短い受領だけで十分な場面を潰していないか
- 品質への具体的なフィードバック（検証不足・説明不足など）が来ているのに、そこを受け止めず次回案内だけに飛んでいないか
- 緊急案件では、時間を断定せず `優先して確認する` など優先度の約束で返せているか
- 緊急案件で、`お急ぎだと思います` のような緊急度ラベルだけで終わらず、先にこちらのアクションを置けているか
- 主質問が `何時間くらいで見てもらえるか` の場合、価格や範囲より先に `まず何時間で見立てを返すか` を答えているか
- 時間や原因の保留で、`現時点では` `まだ` `断定できません` を重ねて防御的になっていないか
- anxious × handoff で `自然です` に逃げていないか。必要なら `その進め方で大丈夫です` `よくあるご相談です` `そのまま進めて大丈夫です` に置き換える
- 感情応答は初手中心にし、2通目以降は要件優先に寄せているか
- 2通目以降も感情応答を繰り返して、判定や次アクションが薄くなっていないか

### quote_sent
- 聞かれたことに結論から答えているか
- ココナラ仕様の質問で `条件による` だけで終わっていないか
- 価格不安の相談では、件数判定のロジックを先に見せられているか
- 通話不可・範囲外などを断った後に、最後の判断を相手へ返していないか。`この前提で難しそうなら合わないかもしれません` のように、こちら側の条件として整理できているか
- 1文に `の` が3回以上出ていないか。名詞を詰め込みすぎているなら、`で` `を` `に` へ分解できないか確認したか

### closed 後再発
- 確認依頼文が毎回同じになっていないか
- 自責寄りなら、短い安心提供があるか
- 金額不安なら、方向性だけは触れているか
- 実作業や書き直しを受ける話なら、トークルームクローズ後は改めて依頼が必要だと明示できているか
- `追加料金の有無` `かかるかどうか` だけで止めて、無料継続の余地を残していないか
- `今は問題ない` `将来影響が出るかも` の相談を、前回範囲の bugfix として自動吸収していないか

### 範囲外
- `別件で受けられる` と `現在は受けない` を混同していないか
- 次の行動を1文で示しているか
- `断り -> 提案 -> 再度断り` の構造になっていないか
- 秘密情報や外部共有を止める時に、止めるだけで終わらず代替の共有方法を示しているか
- 電話 / Zoom / 対面希望を止める時に、不可だけで終わらず、トークルーム内でどう進めるかを続けて書けているか
- できないことを伝える時に、同じ文か直後の文で「できること」も示せているか
- `その代わり` で代替案のように見せず、サービスの通常の進め方として説明できているか
- 相手が希望した進め方を断った直後に情報回収へ進む場合、`もしこの進め方で問題なければ` など受け入れ余地を1行入れているか
- 範囲不確定なら、金額を先に決め打ちせず「対応できる内容かどうか」「どちらの入口が先か」を返せているか
- セキュリティ事案では、調査前に `今すぐ止められるか` を確認しているか
- 重複処理や二重決済の相談では、返金や原因説明より先に `同じ操作を続けない方がよいか` を返せているか
