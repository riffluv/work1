# ココナラ phase contract 素材調査（Agent-Reach / 2026-04-25）

## 位置づけ

このメモは、返信システムの「結局どうなるの？」問題を潰すための外部調査素材。
正本ではない。ここから `platform-contract / gold / reviewer_prompt / validator / #RE stock` へ落とす時は、Codex が妥当性・再発性・副作用を判定する。

## 結論

返信文の曖昧さは、日本語の弱さだけではなく、ココナラの取引 phase ごとの着地点が曖昧な時に起きる。

特に `delivered` と `closed` の扱いは、返信システム側で明確に分ける必要がある。

- `delivered`: 正式納品後でも、クローズ前ならトークルーム内でやりとりできる。購入者は承諾または差し戻しを選ぶ。
- `closed`: トークルームでの投稿・ファイル投稿・おひねり等の追加購入はできない。取引中へ戻すことや承諾取消もできない。入口はメッセージ機能へ戻る。
- `closed` 後の追加確認・再発・返金不満では、「旧トークルーム内で続ける」「キャンセルへ進める」と読ませない。まずメッセージ上で前回修正との関係を確認し、実作業が必要なら見積り提案または新規依頼へ分岐する。

## 公式情報から取れる hard facts

### メッセージ機能とトークルーム

- メッセージ機能は、購入前の事前相談や、取引終了後にトークルームがクローズした後の連絡に使う。
- ただし、正式な依頼はメッセージではなく、サービス購入後のトークルームで行う。
- メッセージは返信が任意で、送信後の取り消しは原則できない。

Source:
- https://coconala-support.zendesk.com/hc/ja/articles/218721057-%E3%83%A1%E3%83%83%E3%82%BB%E3%83%BC%E3%82%B8%E6%A9%9F%E8%83%BD%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6

### 正式な納品 / 納品確認

- 正式な納品後、購入者は承諾または差し戻しを選ぶ。
- クローズまでの間はトークルーム内でやりとり可能。
- 差し戻し後は正式納品前の状態に戻る。
- 2回目の正式な納品後は、差し戻し・承諾の表示はされない。
- 購入者が反応しない場合、正式納品から72時間後の次の00分に自動承諾扱いでクローズする。
- 納品確認は、当初合意した内容との相違確認であり、合意内容を上回る過度な修正要求には使えない。

Source:
- https://coconala-support.zendesk.com/hc/ja/articles/4402957809049-%E7%B4%8D%E5%93%81%E7%A2%BA%E8%AA%8D%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6-%E8%B3%BC%E5%85%A5%E8%80%85%E5%90%91%E3%81%91
- https://coconala-support.zendesk.com/hc/ja/articles/4403291547417-%E3%83%88%E3%83%BC%E3%82%AF%E3%83%AB%E3%83%BC%E3%83%A0%E3%81%AE%E3%82%AF%E3%83%AD%E3%83%BC%E3%82%BA%E6%9D%A1%E4%BB%B6%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6

### closed 後

- クローズ後は、トークルームでメッセージ・ファイル投稿・おひねり等の追加購入ができない。
- 一度クローズしたトークルームを取引中へ戻すこと、承諾を取り消すことはできない。
- クローズ後に追加確認が発生した場合は、メッセージ機能で出品者へ問い合わせる。
- 承諾後24時間以内かつ評価前なら、まだトークルーム内でメッセージやおひねり支払いが可能な場合がある。

Source:
- https://coconala-support.zendesk.com/hc/ja/articles/218723147-%E3%83%88%E3%83%BC%E3%82%AF%E3%83%AB%E3%83%BC%E3%83%A0%E3%81%AE%E3%82%AF%E3%83%AD%E3%83%BC%E3%82%BA%E5%BE%8C%E3%81%AB%E8%BF%BD%E5%8A%A0%E3%81%AE%E8%B3%AA%E5%95%8F%E3%82%84%E4%BE%9D%E9%A0%BC%E3%82%92%E3%81%97%E3%81%9F%E3%81%84

### キャンセル / 返金

- キャンセルは原則、出品者がトークルームからキャンセルリクエストを送る。
- 双方合意、または購入者が24時間以内に差し戻しをしない場合にキャンセル成立。
- キャンセル成立後、決済手段に応じて返金される。
- 既にトークルームがクローズしている場合、トークルームではやりとりできず、クローズ後はキャンセルできない。
- 直接口座などの個人情報や、直接の金銭のやりとりは禁止。

Source:

- https://coconala-support.zendesk.com/hc/ja/articles/218643857-%E3%83%88%E3%83%BC%E3%82%AF%E3%83%AB%E3%83%BC%E3%83%A0%E3%81%AE%E3%82%AD%E3%83%A3%E3%83%B3%E3%82%BB%E3%83%AB%E6%96%B9%E6%B3%95-%E5%87%BA%E5%93%81%E8%80%85%E5%90%91%E3%81%91
- https://coconala-support.zendesk.com/hc/ja/articles/218119178-%E3%82%AD%E3%83%A3%E3%83%B3%E3%82%BB%E3%83%AB%E3%81%AB%E3%81%AA%E3%81%A3%E3%81%9F%E5%A0%B4%E5%90%88%E3%81%AE%E8%BF%94%E9%87%91%E6%96%B9%E6%B3%95

### 見積り提案

- 見積り相談は、相談内容を踏まえて出品者が見積り提案し、buyer が購入すると取引に進む仕組み。
- メッセージ経由の見積り提案は、サービスページやプロフィールからの正式な見積り相談を buyer が送っていなくても、メッセージのやりとりを踏まえて出品者から提案できる。
- 提案の購入期限は、提案日または最終変更日から1週間。
- トークルーム経由の見積り提案は定期購入のみ可能。単発の追加依頼はメッセージ経由・サービス経由・プロフィール経由を使う。

Source:
- https://coconala-support.zendesk.com/hc/ja/articles/900004238843-%E8%A6%8B%E7%A9%8D%E3%82%8A%E6%A9%9F%E8%83%BD%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6-%E5%87%BA%E5%93%81%E8%80%85%E5%90%91%E3%81%91
- https://coconala-support.zendesk.com/hc/ja/articles/360004889793-%E8%A6%8B%E7%A9%8D%E3%82%8A%E6%8F%90%E6%A1%88%E3%81%AE%E8%B3%BC%E5%85%A5%E6%9C%9F%E9%99%90

### 外部連絡 / 外部共有 / 直接決済

- やり取りはココナラ内のみ。外部で直接連絡する、または外部連絡につながるやり取りは禁止。
- 電話、LINE、SNS、Chatwork、Discord、Zoom、Skype、Google Meet、Teams 等は使用不可として例示されている。
- GitHub、Google Drive、Dropbox 等も、設定によってアカウントやメールアドレスが見える可能性がある共有手段として禁止例に含まれる。
- ココナラ外の決済や直接返金も禁止。
- データ共有はファイル添付機能へ寄せる。

Source:
- https://coconala-support.zendesk.com/hc/ja/articles/218179168-%E3%82%B5%E3%82%A4%E3%83%88%E5%A4%96%E3%81%A7%E3%81%AE%E9%80%A3%E7%B5%A1%E3%82%84%E5%80%8B%E4%BA%BA%E6%83%85%E5%A0%B1%E3%81%AE%E5%85%B1%E6%9C%89%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6

## ユーザー実例から取れる edge 素材

### 進捗不安 / 連絡途絶

知恵袋やブログでは、出品者または購入者の連絡が遅く、キャンセルや信頼低下につながる事例が多い。
返信システムでは、購入後の進捗返信で「確認中です」だけだと不安を増やす。

返信に必要な要素:
- 今どこを見ているか
- まだ断定できない点
- 次に見る点
- 次回返答時刻

Source:
- https://detail.chiebukuro.yahoo.co.jp/qa/question_detail/q12294401862
- https://coconala-support.zendesk.com/hc/ja/articles/218179858-%E8%B3%BC%E5%85%A5%E8%80%85%E3%81%8B%E3%82%89%E9%80%A3%E7%B5%A1%E3%81%8C%E3%81%AA%E3%81%84

### 作業後キャンセル / 返金不満

作業後に buyer 都合でキャンセル希望が出るケースや、返金・評価への不安がある。
これは `返金します / 返金できません` と即断しない一方で、濁しすぎると不信感になる。

返信に必要な要素:
- その場で返金可否を断定しない
- 合意内容・作業済み範囲・現在 phase によって扱いが違う
- open talkroom ならキャンセル手続きの相談、closed ならメッセージ上で確認し、新規導線へ
- 直接返金や口座情報交換には絶対に寄せない

Source:
- https://detail.chiebukuro.yahoo.co.jp/qa/question_detail/q13325016353
- https://legal.coconala.com/bbses/41429

### 正式納品前の確認不足

ユーザー実例では、正式納品を送った後に承諾されず不安になる、または差し戻し上限を後から知るケースがある。
bugfix-15000 では、正式納品前に確認手順と修正済みファイルを渡し、buyer が確認できる状態を作る方が安全。

返信に必要な要素:
- 納品前に何を確認してほしいか明示
- 正式納品後の phase を圧として出しすぎない
- 差し戻し前提ではなく、確認手順で認識ズレを減らす

Source:
- https://detail.chiebukuro.yahoo.co.jp/qa/question_detail/q13320030953
- https://coconala-support.zendesk.com/hc/ja/articles/4402957809049-%E7%B4%8D%E5%93%81%E7%A2%BA%E8%AA%8D%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6-%E8%B3%BC%E5%85%A5%E8%80%85%E5%90%91%E3%81%91

### サービス外要望 / 後出し要件

ココナラブログでは、音信不通、サービス外要望過多、修正を装った変更・追加、着手後キャンセル希望などがトラブル型として挙がっている。
この素材は hard rule ではなく、#RE stock の edge として使う。

返信に必要な要素:
- 「修正」か「変更・追加」かを分ける
- 同じ原因 / 別原因 / 新規要件を分ける
- 別原因や追加要件なら、進める前に相談する
- buyer の怒りや不安を受け止めつつ、無償で広げない

Source:
- https://coconala.com/blogs/search?blogTagId=3157
- https://mag.coconala.com/articles/lp-measures-to-prevent-troubles

## 返信システムへの落とし込み候補

### P0: platform phase contract

`state` は最低限、次の区別を持つ。

- `prequote`: 購入前。価格・対象可否・納期・共有方法に直答する。トークルーム前提で書かない。
- `quote_sent`: 見積り提案後・購入前。提案内容への不安に直答する。購入促しより先に主質問へ答える。
- `purchased`: トークルーム open。進捗・必要情報・安全境界を返す。メッセージ機能に戻さない。
- `delivered`: 正式納品後・クローズ前。承諾/差し戻し待ち。クローズ前ならトークルーム内でやりとり可能。
- `closed`: 取引完了後。旧トークルーム継続は不可。メッセージ上で確認し、実作業は見積り提案または新規依頼へ。

### P0: closed の明示着地

closed で返金・再発・追加確認が来たら、以下を最低限返す。

1. トークルームは閉じている
2. この場で返金可否やキャンセル手続きは断定しない
3. まずメッセージ上で前回修正とのつながりを確認する
4. 前回補足で済むか、新規見積りが必要かを先に返す

### P1: delivered の明示着地

delivered で不満・追加確認が来たら、closed と混ぜない。

- クローズ前ならトークルーム内で確認できる
- 差し戻し/承諾の判断は buyer 側
- 合意内容と相違があるかを見て返す
- 同一原因なら補足確認、別原因なら追加相談

### P1: 見積り導線

closed 後の実作業は、単発ならメッセージ経由の見積り提案または新規購入へ。
トークルーム経由見積りは定期購入向けの仕様があるため、単発 bugfix の汎用導線として外向けに出さない。

### P2: edge stock に足すべき相談文

- closed 後に「返金してください」
- closed 後に「前のトークルームで続きできますか」
- closed 後に「おひねりで払えばいいですか」
- delivered 後に「承諾したらもう直してもらえませんか」
- delivered 後に「差し戻しは何回できますか」
- purchased 中に「連絡が空いて不安です」
- purchased 中に「キャンセルしたいです」
- purchased 中に「別の人に頼みたいです」
- quote_sent 後に「直らなかったら料金はどうなりますか」
- 追加料金質問で「前回の続きなのにまた15,000円ですか」

## DeepResearch / Pro に渡すべき問い

1. 上記 phase contract に、ココナラ公式仕様と矛盾する点はないか。
2. `delivered` と `closed` の返金・キャンセル・追加作業の導線を、buyer 向けに最も誤解が少ない表現へ圧縮するとどうなるか。
3. `closed` 後の「返金希望」と「再発確認」を、法務助言にならず、かつはぐらかしに見えない返信型にするには何を必ず含めるべきか。
4. 公式事実として hard rule に入れるべきものと、ユーザー実例として edge stock に留めるべきものを分けてほしい。
5. 返信システムの思考を削がないため、phase contract をどの粒度まで持たせるべきか。過剰制約になりそうな項目は何か。

## 反映しないもの

- 非公式ブログや知恵袋の法的解釈を hard rule にしない。
- 「返金できる / できない」を出品者が断定する rule を作らない。
- すべての closed 後問い合わせを即新規見積りへ飛ばさない。まず前回修正との関係確認を挟む。
- `おひねり` を closed 後の標準導線にしない。現行公式ヘルプは closed 後のおひねり不可を明示しているため、古いニュースや実サービス例より現行ヘルプを優先する。
- ココナラの返金と、依頼者サービス側の Stripe/PayPay 顧客返金を混同しない。

## 正本側へ即時ガード済み

2026-04-25 時点で、`os/coconala/platform-contract.yaml` に以下を反映済み。

- 現行の support.zendesk.com 公式ヘルプを優先し、古いニュース・サービス出品ページ・非公式ブログ・ユーザー投稿を外向け文面や validator の根拠にしない。
- closed 後のおひねり / 追加購入について、古い情報や実サービス例があっても標準導線にしない。
- closed 後は、メッセージ上で確認し、実作業が必要なら見積り提案または新規依頼へ戻す。
