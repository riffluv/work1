# ココナラ phase contract 厳密化レビュー要約（DeepResearch / 2026-04-25）

## 位置づけ

DeepResearch の `ココナラ phase contract の厳密化レビュー4-25.txt` を、返信システムへ入れやすい形に要約したメモ。
外部調査そのものではなく、Codex が最小反映するための整理。

正本へ入れる場合は、`platform-contract / gold / validator / reviewer_prompt / #RE stock` のどれに戻すかを分ける。

## 結論

`2026-04-25-coconala-phase-contract-agent-reach.ja.md` の骨子は、現行公式ヘルプと大筋で整合している。

そのまま使える主張:

- メッセージ機能とトークルームを分ける
- `delivered` と `closed` を分ける
- `closed` 後は旧トークルーム継続を前提にしない
- `closed` 後のおひねりを標準導線にしない
- 外部連絡・外部共有・直接決済を避ける

補正すべき点:

- `prequote / quote_sent / purchased / delivered / closed` はココナラ公式用語ではなく、Workspace 内部の state 写像。
- `quote_sent` は見積り提案済みだが入金完了前までを含める。トークルームが開くのは入金完了後。
- `delivered` は正式な納品後・クローズ前でよいが、一回目と二回目の正式な納品後では buyer ができることが違う。外向け文面で差し戻し可否を一般化しない。
- `closed` 後のメッセージは事実確認と切り分けの導線として妥当。ただし、新たな正式依頼や有償作業をメッセージだけで完結させない。必要なら見積り提案または新規依頼へ戻す。
- `closed` は完了としてクローズ済みのトークルームを指し、キャンセル成立済み状態とは分ける。

## すぐ platform-contract へ反映する項目

- phase 名が公式用語ではなく内部 state であること。
- `quote_sent = 見積り提案済みだが入金完了前`。
- `purchased = 入金完了後のトークルーム open`。
- `delivered = 正式な納品後・クローズ前`。
- `closed = 完了としてクローズ済み`。`cancelled` は含めない。
- `delivered` では、差し戻し可否を一般化しない。
- `closed` では、メッセージ上の事実確認と実作業を分ける。

## #RE batch で先に検証する項目

- 二回目正式納品後でも差し戻し可能と誤答しないか。
- 見積り相談経由で有料オプションを案内しないか。
- `closed` 後に100MB超の資料提出を求められた時、外部共有へ逃げないか。
- 購入者はいつでも自力キャンセル可能、と誤答しないか。
- 承諾後24時間以内で、まだ `closed` ではないケースを `closed` と決め打ちしないか。

## reviewer_prompt に留める項目

- 前回の修正と同一原因か / 別原因か / 新要件か。
- この返信で次に何が起きるかまで書けているか。
- メッセージ上の事実確認と正式依頼を混ぜていないか。
- phase 未確定なのに `delivered` / `closed` を断定していないか。
- free explanation と actual work の境界が曖昧になっていないか。

## validator 候補

Fail 寄り:

- `closed` なのに旧トークルーム継続を前提にする。
- `closed` 後におひねりを案内する。
- 直接返金・外部決済・外部共有・外部通話・外部ストレージへ誘導する。
- 未提供または不十分な状態で、一方的クローズや正式納品を促す。

Warn 寄り:

- `delivered` と `closed` の取り違え。
- 二回目正式納品後でも差し戻し可能と読める。
- 購入者がいつでも自力キャンセルできると読める。
- 正式納品後でも有料オプションを追加できると読める。
- 追加料金が自動発生すると読める。

## 入れない方がよい情報

- 古いココナラニュース。
- 個人ブログの体験談。
- 知恵袋やSNSの不満投稿。
- 他出品者の追加支払い専用サービス実例。
- 法律相談系の返金見解。
- 支払い方法別の返金 timing 詳細。
- 電話相談 / 定期購入の別系統ルール。
- 公式ヘルプにない運用慣習。

## 注意

公式には、必要不可欠な役務提供情報の共有について例外がある。
そのため、generic hard rule は「一切の個人情報・認証情報に触れない」ではなく、
「外部連絡や直接連絡につながる共有を促さない」「秘密値やログイン情報は扱わない」に留める。

ここを硬くしすぎると、本来許容される業務情報共有まで潰してしまう。

## 最小導入の要点

phase を増やさない。

`delivered` と `closed` の外向け言い切りを細くし、`closed` 後は、

1. 旧トークルーム不可
2. メッセージで事実確認
3. 必要なら見積り提案 / 新規依頼

の着地を固定する。

返金とおひねりは曖昧さを生みやすいので、

- 返金可否を断定しない
- `closed` 後のおひねりを標準導線にしない

だけは platform 側で硬くしてよい。
