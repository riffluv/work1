## 4. `#R` で重視すること

`#R` は本番寄りの返信候補です。
`#RE` よりも #R の方が、AI の自由な文生成と service facts の接続を見やすいです。

見る順番:

1. phase は何か
   - prequote
   - quote_sent
   - purchased
   - delivered
   - closed
2. buyer が次に決めたいことは何か
3. 最後の質問は primary か、単なる補足か
4. 価格・納期・成功保証・失敗時・材料共有などの明示質問を落としていないか
5. buyer に返しすぎている負担はないか
6. `こちら / そちら` の向きは合っているか
7. `できます` で止まらず、seller action が見えるか
8. 支払い前作業や closed 後実作業を匂わせていないか
9. 日本語ビジネスチャットとして不自然な距離感・皮肉感・説明者目線がないか

重要:

- final question は primary の候補でしかありません
- primary selection は、buyer の次アクション・明示 cue・phase・価格/支払い/材料共有の重さを合算します
- 「最後に書いてあるから primary」と機械的に扱わないでください

## 7. 現在の重要構造

### answer_focus_alignment

主質問と明示質問に答える。
特に複数質問では、価格・納期・失敗時・材料共有を落とさない。

### primary_decision_need

buyer が次に決めたいことを primary として扱う。
最後の文だけを機械的に primary にしない。

### phase_evidence_alignment

`見積り提案ありがとうございます` など phase の証拠を見て、prequote / quote_sent / purchased を誤らない。
`見積り提案の内容で問題なければ` は quote_sent か相手文に見積り提案済み証拠がある時だけ使う。

### promise_consistency

「必ず直るとは言えない」と言った後に、「修正済みファイルを返します」と無条件 promise しない。
修正可能時・特定できた場合・原因確認後を明確に分ける。

### agency_alignment

行為主体をずらさない。
`依頼できますか` には `ご依頼いただけます`。
`確認してもらえますか` には `こちらで確認します`。
`こちら / そちら` の向きに注意。

### capability_declaration_drift

`補足できます` / `対応できます` で止まらない。
何を返すかを書く。

例:

```text
反映する修正ファイルと順番を、承諾前の確認用に短く整理してお伝えします。
```

### buyer_material_literacy

非エンジニア buyer に、関係ファイル選別を返しすぎない。

コードが分からない / 外注コード / AI生成コード / どのファイルか分からない時:

- コード一式 ZIP
- ログ / スクショ
- secret 除外
- 必要範囲はこちらで確認

へ寄せる。

ただし、全 buyer に ZIP 一式を hard rule 化しない。

### positive_flow_before_refusal

軽い手順確認では通常フローを先に出す。
境界突破要求では不可表明を明確に残す。

例:

- 軽い手順確認: `お支払い完了後にトークルームで共有してください`
- 境界突破: `支払い前にログを見て原因判断する形では進めていません`

### purchase_before_materials_order

購入・支払い前なのに、購入後の材料案内を先に厚く出しすぎない。
CTA の後に材料案内を置く方が自然な場面がある。

### commercial_temperature

購入意思の温度に合わせる。
`この内容で問題なければ、そのままご購入ください` は強い場面もある。
柔らかくする場合は `この条件で進める形で問題なければ、見積り提案の内容でお支払いへ進んでください` など。

### semantic_grounding

相手文にない技術語・原因候補・受領済み事実・作業状況を足さない。

悪い例:

- Event ID がないのに Event ID を見ています
- ログ受領がないのにログを照合しています
- Webhook 原因と未確認で断定する

### post_completion_followup_scope_clarity

closed 後の「関係確認」で、何を返すのか buyer に見えるようにする。

例:

```text
届いた内容をもとに、前回の修正と関係がありそうか、新しい不具合の可能性が高そうかを短く見立てます。
```

「確認します。でも修正作業はしません」だけだと、buyer には何を確認してくれるのか見えにくい。

### negative_frame_non_echo

`責めたいわけではない` / `文句ではない` / `疑っているわけではない` をそのまま復唱しない。
`関係確認` / `状況確認` / `切り分け確認` へ変換する。

### pressure_word_summarization

強い圧力語をそのまま繰り返さない。
`今日中に無料対応` / `返金してほしい` は、`作業可否` / `費用や返金の扱い` / `前回修正との関係` へ要約する。

### block_rhythm_flow

処理文・条件文・安全説明が同じリズムで並びすぎると契約説明っぽく見える。
ただし高リスク場面では硬さを許容する。

### 1. 支払い前に少しだけ見てほしい

buyer の本音は「直りそうなら購入したい」。
支払い前診断は不可だが、拒否だけではなく、症状ベースで対象になり得るかを答える。

prequote:

```text
支払い前にログを見て、直せそうかだけ先に判断する形では進めていません。
ただ、Stripe決済後にDBへ保存されない不具合であれば、対象になり得ます。
この条件でよければ、15,000円で見積り提案します。
```

quote_sent:

```text
購入前にGitHubやスクショを確認して、原因や直りそうかを先に判断することはしていません。
その判断は、お支払い完了後にログやコードを確認してから行う内容になります。
ただ、決済後にアプリ側へ反映されない症状であれば、このサービスの対象になり得ます。
```

### 2. 非エンジニア / AI生成 / 外注コード

`分かる範囲の関係ファイル` は buyer に負担を返しすぎる場合がある。

安全な基本形:

```text
コードの中身が分からない状態でもご依頼いただけます。
ご購入後は、無理に関係ファイルを絞らなくて大丈夫です。
可能であれば、コード一式をZIPでトークルームに添付し、あわせてスクショやログを共有してください。
.env、APIキー、Webhook Secret、DB接続文字列などの値そのものは含めないでください。
必要な範囲はこちらで確認します。
```

### 3. 複数症状 + 優先度

`A を優先したい` が強い cue。
最後の `コードは詳しくありません` などに引っ張られすぎない。

```text
今回は、まず「決済後に注文が作られない」不具合を1件として、15,000円で確認できます。
もう一方の症状が同じ原因につながっている場合は、同じ確認範囲で見ます。
別原因と分かった場合は、進める前に対応方法と費用を先にご相談します。
```

### 4. 納品後 / 承諾前

`補足できます` で止めない。
seller が何を返すかを書く。

```text
反映する修正ファイルと順番を、承諾前の確認用に短く整理してお伝えします。
```

`専門用語を減らして` は、buyer がそう言っていない場合は嫌味・皮肉に見える可能性がある。
`確認しやすい形で` などへ寄せる。

### 5. closed 後の関係確認

`関係確認` の成果物を見せる。

```text
届いた内容をもとに、前回の修正と関係がありそうか、新しい不具合の可能性が高そうかを短く見立てます。
ここでできるのは、ログとスクショから見える範囲での関係確認までです。
コード修正や修正済みファイルの返却が必要になりそうな場合は、対応方法と費用を先にご相談します。
```

## 9. Pro を使うタイミング

Pro は常用せず、以下の時に使う。

- #R human audit で構造候補が 3件以上たまった
- 違和感の戻し先が分からない
- `case_fix` でよいか、`skill / gold / lens / lint` に戻すべきか迷う

- GitHub connector 優先
- 見られない場合の保険として公開URLも入れる
- 相手文と返信候補を必ずペアで渡す
- 「単語禁止ではなく汎用構造としてどう扱うか」を聞く
- `return_target` を必ず求める

return_target:

- `hard_guard`
- `soft_lens`
- `gold`
- `skill`
- `lint`
- `case_fix`
- `preference`

## 10. 実装・正本反映の優先順位

何か違和感が出たら、すぐ rule 化しない。

順番:

1. 実返信候補で問題箇所を確認
2. その場の言い換えで直るか見る
3. 再発性があるか見る
4. 汎用構造名をつける
5. 反映先を決める
6. 最小変更で正本へ戻す
7. `#R` で再確認
8. 必要なら `#RE` regression

正本反映先の目安:

- writer が毎回迷う -> `writer-brief.ja.md`
- bugfix 固有 -> `coconala-reply-bugfix-ja/SKILL.md`
- 汎用日本語 -> `japanese-chat-natural-ja/SKILL.md`
- 監査観点 -> `lens-taxonomy.yaml` / 監査プロンプト
- 良い型 -> `docs/reply-quality/gold-replies/`
- 表面形で高確度 -> `reply_quality_lint_common.py`
- 単発 -> batch / case_fix のみ

## 11. やってはいけないこと

- `相談できます` を blanket NG にしない
- `大丈夫です` を blanket NG にしない
- `支払い前に〜しません` を全部消さない
- `コード一式ZIP` を全ケース hard rule にしない
- `ご購入ください` を blanket NG にしない
- `関係ファイル` を blanket NG にしない
- 自然化のために phase / scope / price / secret / payment route を弱めない
- 高リスク場面の必要な硬さを、AI感として削りすぎない
- 1回限りの好みを lens / lint にしない
- #RE を費用をかけて無目的に回し続けない

## 12. 汎用コア化の方針

ココナラで見つけた違和感は、ココナラ語のまま閉じない。
汎用構造へ変換して残します。

例:

- `トークルーム` -> platform-specific message room
- `見積り提案` -> quote_sent / payment-before-work phase
- `正式納品 / 承諾` -> delivered / acceptance-before-close phase
- `おひねり` -> platform-specific追加支払い導線
- `同じ原因なら15,000円内` -> same-cause scope boundary
- `関係ファイルを選べない buyer` -> buyer_material_literacy
- `こちら / そちら` のズレ -> deictic_agency_alignment

こうしておくと、将来の SaaS / B2B / Team plan / multiple-service routing に移植できます。

## 13. 次に何をするべきか

ユーザーが「次に進んで」と言った時の優先候補:

1. #R 実返信候補の続き
2. human audit で出た違和感の構造分類
3. 反映済み構造の短い棚卸し
4. 必要なら Pro への critical prompt 作成
5. 採用済み構造だけ #RE regression

今は、無目的に #RE を何十本も回すより、#R 実返信候補でユーザーの違和感を拾う方が効果的です。
