# AI に返信を書かせたら平均1.5点だった。4.8にした全記録（2サービス対応）

## はじめに

ココナラで「Next.js / Stripe の不具合修正」サービスを運営しています。
見積もり相談から購入後のやりとり、納品、クローズまで、すべてのクライアント対応をAIに任せられないかと考え、返信システムを構築しました。

結論から言うと、**最初の出力は平均1.5点（5点満点）で使い物にならなかった。** それを4.8点まで引き上げ、ココナラのプラットフォーム仕様を踏まえた返信まで自動生成可能にした全記録です。

## 1. 最初に作ったシステムと、なぜ失敗したか

### やったこと

Codex（OpenAI）に「ココナラの返信を自動生成する skill」を作ってもらいました。

最初の設計はこうでした：

```
buyer message → scenario detection → slot-filling renderer → output
```

renderer が持っていたもの：
- `reaction_line()` — 最初の反応文
- `answer_brief()` — 回答の要約
- `bridge_line()` — つなぎの文
- `closing()` — 締めの文

**つまり「テンプレートの穴を、AIに埋めさせる」構造。**

### 何が起きたか

出力例（実際の相談に対する返信）：

```
ご連絡ありがとうございます。
ご不安な点、確認いたしました。

Stripeの不具合に関して、当サービスにて確認させていただくことが可能です。
ご購入いただけましたら、まず状況を確認させていただき、
原因の切り分けから進めさせていただきます。

ご不明な点がございましたら、お気軽にご相談ください。
```

**問題点：**
- 相手が何を聞いているかに関係なく、同じ文が出る
- 「確認いたしました」「させていただきます」の過剰敬語
- 相手が書いた技術的な詳細（Webhook、Vercel、テスト環境OK）が完全無視
- AI感が強い

**5件テストして、平均1.5点。0件が「そのまま送れる」状態。**

## 2. モグラたたきの日々

ここから修正が始まりました。

```
「反応文が硬い」→ reaction_line を修正
  → 別の場所が壊れる

「敬語が過剰」→ 語尾ルールを追加
  → 今度は答えの順序がおかしくなる

「意図が二重に出る」→ dedupe ルールを追加
  → 別のケースで必要な文まで消える
```

**直しても直しても、別の場所が壊れる。** 典型的なモグラたたき。

この時期のバッチテスト結果：

| バッチ | 平均スコア | 主な問題 |
|---|---|---|
| batch 4 | 1.5 | slot-filling の構造的失敗 |
| batch 8 | 2.5 | 同義反復、手続き文が先に来る |
| batch 9 | 3.0 | 一部改善するが複雑なケースで崩壊 |

## 3. 「構造自体がおかしい」と気づいた瞬間

何回修正しても改善しない。**プロの現場ではどうやっているのだろう？**

- そもそも返信システムなんて、一流企業くらいしか作っていない
- 日本語の情報はほぼゼロ
- ましてや「AIに返信を書かせる」設計論の情報は見つからない

そこで外部調査ツール（Agent-Reach）を使って、OpenAIのドキュメント、Anthropicのガイド、Intercom / Rasa / Dialogflow の設計を調べました。

**見つかった核心：**

> 「reasoning（思考）と realization（文章化）を分離しろ」
> 「AIの判断結果を、固定テンプレートで上書きするな」

つまり、**自分が作ったシステムの構造自体が、AIの思考を殺していた。**

## 4. 外部調査で見つけた設計思想

### scaffolding vs framework

Agent-Reach（AI エージェントにウェブ検索機能を付けるツール）の README にこう書いてありました：

> Agent Reach は scaffolding（足場）であって、framework（フレームワーク）ではない。

```
framework（フレームワーク）:
  「この枠の中で動け」
  → AI の思考を枠で制御する
  → 安定するが、柔軟性がない
  → 未知の入力に対応できない

scaffolding（足場）:
  「道具は揃えた。あとは好きにやれ」
  → AI の判断に任せる
  → 柔軟だが、設計が難しい
  → 未知の入力にも対応できる
```

**自分のシステムは完全に framework 型だった。** slot-filling renderer が AI の思考を上書きしていた。

### 業界の実装パターン

| サービス | 設計 |
|---|---|
| Intercom | knowledge sources と guidance を分離。guidance は応答前に適用 |
| Rasa | intents / entities / slots / responses / actions に分離 |
| Dialogflow | parameters / variables / knowledge sources を分離 |

**共通点：「知識」と「応答生成」を別の層にしている。**

## 5. slot-filling から scaffolding へ

### 構造の変更

```
旧（framework 型）:
  buyer message → scenario → slot-filling renderer → output
  
  skill の中身:
    reaction_line = "ご連絡ありがとうございます"
    answer_brief = "{scenario}について確認しました"
    closing = "ご不明な点がございましたら..."
  
  → AI は穴を埋めるだけ。思考がかき消される。

新（scaffolding 型）:
  buyer message → response_decision_plan → Codex が自由に書く → validator
  
  skill の中身:
    料金: 15,000円
    scope: 1件の不具合
    禁止: 秘密情報の値を受け取らない
    禁止: handoff-25000 を外向けに出さない
  
  → AI は「何を守るか」だけ受け取り、「何を書くか」は自分で判断する。
```

### 変えたもの / 変えなかったもの

```
変えたもの:
  ❌ slot-filling renderer → 廃止
  ❌ 固定本文 → 廃止
  ❌ scenario ごとの heavy paragraph template → 廃止
  ✅ response_decision_plan を追加
  ✅ validator を guardrail として再設計

変えなかったもの:
  ✅ service grounding（料金、scope、禁止事項）
  ✅ routing / scenario detection
  ✅ regression テスト基盤
```

## 7. クロスモデル監査という発見

改善過程で気づいた重要なこと：**AIの出力は、別のAIに監査させると精度が上がる。**

```
Codex（実装担当）:
  自分が作った構造を「正しい」と前提にする
  → 構造自体の問題に気づけない
  → 永遠に phrasing を修正し続ける

Claude / Gemini（監査担当）:
  構造を作っていないので、しがらみがない
  → 「この構造自体がおかしくない？」と言える
  → 出力の品質を客観的に測定できる
```

このクロスモデル体制：

| 役割 | 担当 | 何をするか |
|---|---|---|
| 実装 | Codex | skill / validator / regression を構築 |
| 品質監査 | Claude / Gemini | 出力の日本語品質を測定 |
| 外部調査 | Agent-Reach + Pro | 設計パターンの調査 |
| 最終判断 | 人間（自分） | 何を採用し、何を棄てるか |

**特に重要だったのは「Codex の案を鵜呑みにしない」判断。** Codex は優秀な実装者だが、自分の設計を疑えない傾向がある。監査を挟むことで、実装前に危険な案（tone hint、anxiety patterns）を止められた。

## 8. service understanding layer の追加

scaffolding の次に足したのは、**サービス理解の知識層。**

```
レベル1: 契約理解（最初からあった）
  「15,000円」「1件の不具合」「秘密情報は受け取らない」
  → 何をやるか / やらないか

レベル2: サービス理解（新しく追加）
  「Webhook 系はうちの strong fit」
  「ノーコード + Stripe Plugin は cautious fit」
  「機能追加は out」
  → 何に強いか / どういう証拠を先に見るか
```

これにより、quote_sent（見積もり相談）で技術的に違う相談に対して、**同じ返信に潰れる template 回帰が解消。**

### 入れたもの / 入れなかったもの

```
入れたもの:
  ✅ capability map（strong_fit / cautious_fit / out_of_scope）
  ✅ diagnostic_patterns（evidence priority のみ）

入れなかったもの（Claude のレビューで止めた）:
  ❌ tone hint（温度の指示）
  ❌ buyer anxiety patterns（不安パターンへの対応指示）
  → これらは AI がすでに自然にやっていることを構造化すると、
    逆に思考を圧縮して template 回帰を再発させるリスクがあった
```

## 9. 構造改善3本 — 「思考」「価格」「記憶」を守る

service understanding layer の次に、**1日で3つの構造改善を入れた。**

### ① Plan-Output Integrity Check（思考保全）

```
問題:
  Codex は thinking（内部思考）では正しい判断をしているのに、
  output（返信文）では templateに引っ張られて思考が消される
  
  thinking: 「この buyer は怒っている。共感してから答えよう」
  output:   「ご連絡ありがとうございます。確認します。」
  → thinking が output に反映されていない

研究では Unfaithful Chain-of-Thought と呼ばれている現象。

対策:
  validator に「plan と output の整合チェック」を追加
  → 主質問に答えているか
  → 感情を受けているか
  → 空約束（「確認します」だけで何も確認しない）がないか
```

### ② Service-Reply Binding（価格と禁止事項の構造結合）

```
問題:
  service.yaml に「15,000円」「直接pushは不可」と書いてあるのに、
  renderer がそれを参照せずに返信を書いていた
  → 価格をぼかしたり、禁止行為を引き受けてしまうことがあった

対策:
  service.yaml の base_price / hard_no を validator に直接接続
  → 価格がぼかされていたら自動検出
  → hard_no を肯定する文があったら自動検出
```

### ③ Reply Memory（会話の記憶）

```
問題:
  renderer は「今の1通」だけを見て書いていた
  → 前回「確認中です」と返したことを覚えていない
  → 3通目でも初対面のトーンで返す
  → buyer から見ると「この人、覚えてないの？」

研究では Conversation Amnesia と呼ばれている問題。

対策:
  全文履歴ではなく、低 entropy な reply memory を導入
  → 前回の約束（確認中 / 進捗共有 / 期限）
  → 前回のトーン
  → フォローアップ回数
  → 約束を果たしたかどうか
  
  案件ごとに保存（case-scoped）
  → 複数案件でも memory が混ざらない
```

### 効果

```
改善前（B5）: 平均 3.2 / 空約束 2件 / 感情無視 2件
改善後（B6）: 平均 4.2 / 空約束 0件 / 感情無視 0件 / 全件送信可
```

## 10. 2サービス同時運用への到達

構造が安定したので、**2つ目のサービス（引き継ぎメモ作成・25,000円）の同時運用テスト**に進みました。

```
テスト:
  15件のテストケース
    - 5件: bugfix（修正）が合うケース
    - 5件: handoff（整理）が合うケース
    - 5件: どちらか迷う boundary ケース

結果:
  平均 4.3 / 全件送信可 / routing（振り分け）ミス 0件

特によかったケース:
  - 一見 bugfix に見えるが、実は handoff 寄り → 正しく判断
  - API キーの生値を送ってきた buyer → 即座に止めてセキュリティ対応まで案内
  - 4つの hard_no を全部正しく断り、代替手段を提示
```

### 並列案件の運用設計

```
課題:
  購入後の案件は、buyer の確認待ちで長く open になる
  その間にも別の相談や案件が来る
  同じ workspace で安全に並行できるか？

解決:
  case-switch コマンドで明示的に案件を切り替え
  → memory は案件ごとに保存済み
  → 切り替えるだけで正しい memory が読まれる
  → 別 workspace は不要
```

## 11. Platform Contract Layer — ココナラの仕組みを知識として入れる

4.6 まで来て、品質上の構造的問題はゼロになった。残りの伸びしろは「ココナラのプラットフォーム仕様を踏まえた返信」。

### 問題

buyer が「キャンセルしたい。全額返金できますか？」と聞いた時、Codex は正しく「断定を避ける」返信を書けていた。しかし、**なぜ断定を避けるのかの根拠がなかった。** ココナラの返金の仕組みを知らないから。

同様に：
- 「見積もり提案って何？」→ 操作の意味を知らない
- 「正式な納品って普通のメッセージと違うの？」→ 区別を知らない
- 「トークルーム閉じたらどうやって連絡するの？」→ メッセージ機能の存在を知らない

### 外部調査で見つけたパターン

Agent-Reach で調査したところ、**Layered Context Architecture** というパターンが見つかった（AgentPatterns.ai）。

```
Layer 1: Institutional Knowledge（プラットフォームのルール）
  → 常に参照される。軽い guidance
  → Intercom は knowledge sources と guidance を分離
  → guidance は応答前に必ず適用される

Layer 2: Domain Knowledge（サービス固有）
  → capability map / scope / price
  → 既存の service.yaml がこれ

Layer 3: Contextual Knowledge（その場の文脈）
  → 既存の reply-memory がこれ
```

**足りなかったのは Layer 1（プラットフォームルール）だった。**

### 提案 → 即修正 → 実装

Claude（監査担当）が提案書を作成 → Codex が「方向は正しいが設計が甘い」と5箇所修正:

```
Claude の提案       → Codex の修正
ops/services/ に配置 → os/coconala/ に分離（service と混ぜない）
全部1枚の YAML     → 公式/コミュニティ/推測を分離
metadata なし       → confidence + source_kind を追加
全項目を一気に      → v1 は5項目だけ（小さく切る）
system prompt 直書き → mode=coconala の boot で発火
```

さらに Codex は自分でココナラ公式ヘルプ6ページを読み、Claude の調査の事実誤認を3箇所修正した。

### 結果

```
knowledge 層（完成形）:
  Layer 1: platform contract  ← ココナラの取引ルール（NEW）
  Layer 2: service contract   ← サービス固有の知識
  Layer 3: reply memory       ← 案件ごとの文脈
```

これにより、返信が「うまい」から「ココナラの仕組みを踏まえている」に変わった。

## 13. なぜ今この市場なのか

外部調査（Agent-Reach）で、このサービスの需要を世界的に調べた。

### AI がコードを書く → バグが増える → 直せない

```
2026年の数字:
  - AI 生成コードは人間のコードの 1.7倍 バグが多い（CodeRabbit）
  - 開発者の 45% が AI コードのデバッグに時間を取られている（Stack Overflow）
  - AI 生成コードの 45% にセキュリティ脆弱性（業界レポート）
  - 新規コードの 41% が AI 生成（GitClear）
  - 「AI-Native な開発者はデバッグできない」→ 2026年の問題
```

### 外注コードの引き継ぎは世界的な課題

```
  - 外注コードの Legacy Rewrite 失敗率: 70%（ARDURA Consulting）
  - 「Black Box Syndrome」: 外注コードが誰にも理解できない
  - 「Code Ownership Gaps」: 元の開発者がいなくなりロジック不明
  - 開発者の離職理由: レガシーコードのメンテが辛い
```

**つまり:**
- AI がコードを書く → バグが増える → bugfix-15000 の需要
- AI / 外注コードが理解不能 → handoff-25000 の需要
- **AI の普及が、AI コードを直すサービスの需要を生む** という皮肉な構造

## 14. 現在の構成

```
knowledge層（静的）
├── platform contract   ← ココナラの取引ルール（Layer 1）
├── service contract    ← 価格、scope、禁止事項（Layer 2）
├── capability map      ← strong / cautious / out fit
├── diagnostic patterns ← evidence priority
└── exceptions          ← 実案件で溜まる経験知（成長する領域）

reasoning層（Codex が判断）
├── response_decision_plan
├── fit judgment
├── service routing     ← bugfix / handoff の振り分け
└── 温度判断、回答構成

memory層（案件単位で保持）
├── reply memory        ← 前回の約束、トーン、フォロー回数
└── case-scoped で保存  ← 案件ごとに独立

guardrail層（自動チェック）
├── integrity check     ← plan と output の整合
├── service binding     ← 価格 / hard_no の照合
├── promise tracker     ← 空約束の検出
├── validator           ← 「何が足りないか」だけ見る
├── regression suite    ← template 回帰の検出
└── role suite          ← 常時監視
```

**各層のルール：**
- knowledge 層は「何を知っているか」
- reasoning 層は「何をするか」
- guardrail 層は「何を守るか」
- **どの層も他の層の仕事をしない**

---

## 15. 学んだこと

### 技術的な教訓

1. **AI の思考を slot-filling で殺すな。** テンプレートの穴を埋めさせるのではなく、ルールだけ渡して自由に書かせる方が品質が出る。

2. **scaffolding > framework。** 道具と制約だけ渡して、判断は AI に任せる。

3. **validator は writer にするな。** 「何が足りないか」だけ見て、「何を書くべきか」は見ない。

4. **検出器を先に作れ。** 予防器（validator）より検出器（regression suite）が先。検出器なしに予防器を入れると、予防器の故障に気づけない。

5. **うまくいっている部分を構造化して壊すな。** AI がすでに自然にやっていることを、ルールにすると逆効果になることがある。

### プロセスの教訓

1. **AI は自分の設計を疑えない。** Codex は優秀だが、自分が作った slot-filling の問題に自力で気づけなかった。外部の視点が必要だった。

2. **クロスモデル監査が効く。** 実装した AI とは別の AI に監査させると、「構造自体がおかしい」という指摘が出る。

3. **数字で測れ。** 「なんとなく良くなった」ではなく、5点満点のスコアと Yes/No の判定で追跡した。これがなければ改善を証明できなかった。

4. **モグラたたきは構造の問題。** 直しても直しても別の場所が壊れるなら、修正対象ではなく構造を疑え。

5. **全文履歴より低 entropy memory。** 会話の記憶は全文を渡す必要はない。前回の約束とトーンだけで十分。構造値だけ渡せば、writer 化せずに文脈を保てる。

6. **「はい / いいえ」で答える質問にも、冒頭に「はい、」を置くな。** AI は Yes/No 質問に「はい、」で始めたがるが、実務でそれをやるとbot感が出る。

7. **知識は層で分けろ。** プラットフォーム共通のルールと、サービス固有の知識を同じ場所に入れると重くなるし、古くなった時に危険。Layered Context Architecture のように、層を分けて必要な時に必要な層だけ参照させる。

8. **公式事実と運用知見を混ぜるな。** 返信の根拠に使う情報には、出典と確度（official / community / inferred）を持たせる。コミュニティ記事だけの情報を hard rule にすると事故になる。

9. **AI の提案は、別の AI に査読させろ。** Claude が提案し、Codex が修正し、公式ソースで事実確認する。この3段階で初めて「安全な knowledge」になる。

## おわりに

このシステムは、ココナラの特定サービス（Next.js / Stripe の不具合修正 + 引き継ぎメモ作成）に特化しています。そのまま他のサービスに使えるものではありません。

ただし、**設計の考え方** — slot-filling をやめる、scaffolding 型にする、クロスモデル監査をする、検出器を先に作る、全文履歴より低 entropy memory — は、どんな AI 返信システムにも応用できるはずです。

AI に返信を任せたいけど品質が出ない、という人の参考になれば幸いです。

**チーム構成:**
| 役割 | 担当 | 何をするか |
|---|---|---|
| PM + 最終判断 | 人間（自分） | 「なんか変」と感じる。判断する |
| 実装 + 設計レビュー | Codex | skill / validator / platform contract を構築 |
| 品質監査 + 提案 | Claude | 出力の監査。Layered Context Architecture を提案 |
| テスト生成 | Claude / Gemini | 実案件相当のテストケースを生成（75件） |
| 外部調査 | Agent-Reach | 設計パターン / ココナラ仕様 / 市場調査 |
| サービス設計 | ChatGPT Pro | サービス内容・差別化の検討 |

**技術者ではない自分が、AI チームを組んで、プラットフォーム仕様まで踏まえた返信システムを構築した** — というのが、この記録の本質です。

```
最終スコア: 平均 4.8 / 5.0
テスト通過: 75件 / 全件送信可
構造的問題: ゼロ
Platform 事故: ゼロ
到達期間: 約10日間
```
