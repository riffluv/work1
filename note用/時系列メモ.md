  を明確にしたことで差別化できると判断

### 4/19-4/20 — 情報密度監査の型を固定

- Pro / Claude / Codex xhigh を使い分けて、
  - 情報密度
  - strong-signal の維持
  - hidden rule の排除
  - 平均化している弱い重複の削除
  を監査する運用を作った
- `bugfix-15000` / `handoff-25000` ともに、
  - 全面改稿はしない
  - 重複だけ削る
  - 強い骨格は触らない
  が正解だと確認
- `handoff-25000` では Codex xhigh が
  - bugfix 振り分け案内が core offer より先に出ると密度が落ちる
  と指摘し、導線位置の微修正を採用

### 4/20-4/21 — 「原因だけ分かったら納品」問題を発見

- `bugfix-15000` の返信文と内部ルールを見ている中で、
  - 原因が分かった
  - でも修正済みファイル返却まで進められない
  というケースでも、理屈上は納品成立しうるように見える drift が残っていることに気づいた
- これは普通の受託の感覚では成立しても、ココナラの
  - 正式な納品
  - 購入者承諾
  - 差し戻し
  フローと相性が悪いと判断
- 結論として、
  - 原因不明なら正式納品へ進めない
  - 原因が分かっても、修正済みファイル返却まで進められないなら正式納品へ進めない
  - その場合はキャンセル含め相談
  にサービス方針を寄せた
- この時点で、以前構想にあった「原因追及だけのオプション」がココナラ初期アカウントでは危険寄りだと整理できた

### 4/21 — buyer が最も怖がる論点を潰す

- 「直してもらえないのにお金だけ取られるのでは」という buyer 不安が、
  bugfix 商品で最大の事故ポイントだと明確化
- `bugfix-15000.live.txt` の FAQ と説明文を、
  - 修正サービス
  - 修正済みファイル返却前提
  - 納品成立しない場合は無理に閉じない
  へ揃えた
- これにより、
  - Aの顔をしてBの運用（修正サービスの顔をして原因調査課金）
  だった残り香をかなり排除

### 4/21-4/22 — 返信文のモヤモヤの正体は「境界の曖昧さ」だと判明

- 返信文に感じていた違和感は、日本語の表面の問題ではなく、
  - 結局やってくれるのか
  - やってくれないのか
  - どこから追加相談になるのか
  が曖昧だったことだと判明
- これに対応して、
  - same cause / different cause
  だけでなく
  - same_cause_but_heavy
  を internal scope judge として追加
- つまり、
  - 同じ原因でも軽いなら基本料金内
  - 同じ原因でも重いなら追加相談
  - 別原因なら追加相談
  - 納品成立しないなら正式納品へ進めない
  という4分岐が見えるようになった
- ここで初めて、返信文の「やる / 追加相談 / 納品しない」の境界が buyer からも読み取りやすくなった

### 4/22 — 返信システム監査役としての Codex xhigh が想像以上に強いと判明

- Claude は
  - 日本語の自然さ
  - buyer の受け取り
  - 嫌な感じ / bot感
  に強い
- Codex xhigh は
  - 主質問直答
  - 進行制御
  - phase 整合
  - hidden rule 検出
  - 相手が既に書いた情報の聞き直し検出
  に強いことが、複数 batch 比較でかなり明確になった
- 実務上は、
  - xhigh = 主監査
  - Claude = 肌触り補助
  の役割分担が自然だと結論

### 4/22-4/23 — 返信学習の入口を `#RE` に固定

- 次の Codex が迷わないように、
  - `#R` = 返信を書く
  - `#CL / #GE` = 直近返信の外部監査プロンプトを作る
  - `#RE` = 返信学習 batch を current rules で更新する
  という役割を起動プロンプトに明記
- `#RE` の出力先を
  - `サービスページ/rehearsal/bugfix-15000-返信学習/返信監査_batch-01.md`
  - `サービスページ/rehearsal/handoff-25000-返信学習/返信監査_batch-01.md`
  に固定
- これで、
  - stock から batch 作成
  - xhigh / Claude 監査
  - 採否判定
  - rule 反映
  の loop が次の Codex でもすぐ回せる状態になった

### 4/23 — 返信システムを壊さずに強化するための「評価基盤」を追加

- 以前、生成本体に層を足しすぎて返信システムを壊した経験があったため、
  今回は生成本体ではなく review layer だけを強化する方針を採用
- 追加したのは主に以下
  - drift taxonomy
    - `nonanswer`
    - `redundant_ask`
    - `premature_progress`
    - `hidden_rule`
    - `oversell`
    - `ai_tone`
  - 固定ベンチ
    - `urgent_prequote`
    - `price_anxiety`
    - `unresolved_risk`
    - `purchased_extra_scope`
    - `secret_handling`
    - `direct_push_or_external_share`
  - 変更履歴と結果を紐づける changelog
- これにより、
  - 返信文をさらに賢くする
  ではなく
  - どの変更で何が良くなったか / 悪くなったかを追える
  方向へ返信OSを進化させ始めた

### 4/23 — buyer 入力欄の専門語を除去

- `bugfix-15000` の入力欄に残っていた
  - 本番 / プレビュー / ローカル
  という技術者向け語彙を見直し
- `購入にあたってのお願い`
  `見積りにあたってのお願い`
  の入力項目を、
  - `どこで起きているか（公開中のサイトだけか、自分のPCでも起きるか など）`
  に変更
- サービス本文の技術語は入口の絞り込みとして維持しつつ、
  buyer が「何を書けばいいか分からない」不安だけを削った

### この時期の到達点

1. `bugfix-15000` は「修正納品成立型」に揃い、buyer 最大不安をかなり潰せた
2. `handoff-25000` は非公開のまま、商品境界だけ先に sharpen できた
3. 返信システムは、文面そのものより「ズレ方を分類して止める」方向へ進み始めた
4. xhigh を主監査に置くことで、`話をちゃんと聞いて返せているか` の育成効率が上がった
5. 返信OSは「強い」段階から「壊れにくい」段階へ入った
