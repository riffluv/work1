# 2026-04-24 返信OSの骨格維持と 5.5 移行前の監査ループ固定

更新日: 2026-04-24

## この日の位置づけ

この日は、返信OSを「もっと賢くする」日ではなく、  
**骨格を壊さずに強くする運用を固定した日** だった。

1. 返信システムの骨格を壊さないことを、運用上の最優先として明文化した  
2. service page と reply system のズレをかなり広い範囲で同期した  
3. `#RE -> 外部監査 -> 再発だけ rule に戻す` という改善ループがかなり安定した

さらに後半では、

- 主軸を GPT-5.5 へ移す準備

**返信品質改善の延長線上で、運用そのものを壊れにくくした日** だった。

## まず確認された前提

この日の最初に強く確認されたのは、

- 現在の返信システムの骨格は非常に大事
- 前回の Codex はそこを知らずに触って壊した
- 今の骨格は、live 監査・外部レビュー・archetype fix を経た現行ベスト
- だから、初見の違和感や単発レビューで骨格は触らない

という点だった。

ここでの重要な整理は、

- `今は壊さない`
- `変えるのは新しい情報や再発証拠が揃った時だけ`
- `変えるとしても最小変更`

という運用方針が、かなり明確になったこと。

これは単なる慎重論ではなく、  
**骨格を守らない改善は、改善ではなく退行になる**
という経験則が共有されたことを意味していた。

## 起きていた問題

この時点では、サービスページ正本と reply system の間に、思ったよりズレが残っていた。

特に見つかったのは、

- `原因不明 / 修正済みファイル返却不可なら正式納品へ進めない`
- buyer 向けの環境確認 wording
- `quote_sent` の不要な購入促し
- purchased 後の料金不安への旧方針
- handoff-25000 の private 境界

といった領域。

重要だったのは、  
これが「骨格全体の失敗」ではなく、
**service page は先に更新されていたのに、構造化された記憶領域や renderer / guide が追随していなかった**
という種類のズレだったこと。

つまり、

- 正本はかなり正しい
- でも取り出し口が古いままの箇所があった

という状態だった。

ここが見えたのはかなり大きい。  
この問題は「全部作り直し」ではなく、**同期の問題** だと分かったから。

### 3. 返信監査 `#RE` ループの強化

この日もっとも多く回したのは `#RE` だった。

**外部監査の指摘をその場の文面修正で終わらせず、再発性があるものだけ rule に戻す**
運用がかなり定着したこと。

## 何が `reply-only` に戻ったか

この日は change-log 上でもかなり多くの `CHG` が追加された。

特に重要だったものをまとめると、次のようになる。

### hard no 後の次アクション

- 本番反映 / 直接 push / 直接操作は断る
- ただし `できます` で閉じず、相手がどこで止まっているかを1点だけ聞く

### 購入前の 2件分不安

- `ご購入後は〜を確認します` を先頭に置かない
- `この時点で2件分と決まっていない`
- `別原因でも自動加算しない`

を先に返すようにした。

### closed 後の再発

- `同じ原因なら基本料金内` を初手で言い切らない
- まず `前回とつながるか / 新しい原因か` を確認する

### 複数症状が書かれている時

- `どれを先に見ますか` と相手へ丸投げしない
- こちらで仮の起点を置く

### prequote の温度感

- `ご購入後は` を連発しない
- `進める場合は` へ散らす

### buyer 向けの内部語翻訳

- `同じ原因の範囲`
- `見られる`
- `差し戻しとして見る`

のような内部寄りの語を外向けでは避ける。

### price / boundary wording

- 価格を聞かれていない場面では `15,000円の範囲` を前に出さない
- 追加費用の金額を明示して聞かれたら、金額を先に答える

### 非Stripe 境界

- Square などが出ても、決済サービス自体まで受ける読みを出さない
- `Webhook後のサイト側処理 / DB更新` に範囲を限定して返す

### まだ不具合が起きていない「確認だけ」

- それを確認専用サービスのように受けない
- 気になる1点があるなら、不具合修正として扱えるか購入前に見立てる

これらは全部、単発修正ではなく `reply-only` のルールとして戻された。

この日は reviewer の使い分けもかなりはっきりした。

整理すると、

- `Codex xhigh`
  - 主質問への直答
  - scope / policy / secrets / phase 境界
  - 事故りやすい実務リスク

そのうえで、後半では GPT-5.5 xhigh を reviewer として導入し、

- 5.4 reviewer より hard no / phase 境界 / policy 境界が強い
- 指摘が batch や rule に戻しやすい粒度になっている

という判断に至った。

つまり、

- reviewer としては GPT-5.5 xhigh がかなり有望
- 主軸まで切り替えるのは、区切りがよい時にやる

という方針が固まった。

## 監査プロンプトも育て始めた

この日は reviewer prompt の考え方も一段進んだ。

以前は、

- batch は同じ
- prompt もほぼ固定

だったが、実際には reviewer が昔直した表現をまた resurrect させることが分かった。

そのため、監査プロンプトに

- `同じ原因の範囲` より `つながっているか`
- `このサービスで見られる範囲です` より `対応できます`
- `差し戻しとして見る` より `今回の修正とつながるか`

といった **ぶり返し防止の観点** を追加した。

ここはかなり重要だった。

主軸の記憶だけに頼らず、reviewer prompt 側にも再発防止を持たせる運用へ少し進んだと言える。

という主軸移行方針がかなり有力になった。
