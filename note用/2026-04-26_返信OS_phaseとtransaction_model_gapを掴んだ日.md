# 2026-04-26 返信OS — phase と transaction model gap を掴んだ日

更新日: 2026-04-26

## この日の位置づけ

この日は、返信システムがかなり大きく前進した日だった。

表面上は、`#RE` を回すたびに監査で大量のダメ出しが出ていて、  
一見すると「またモグラたたきに戻ったのでは」と不安になるような流れだった。

しかし実際には逆で、  
今まで名前がついていなかった返信の崩れ方に、かなり深い名前がついた。

それが、

- `phase_answer_gap`
- `unnamed_discomfort`
- `transaction_model_gap`

だった。

特に `transaction_model_gap` は重要で、  
日本語のツギハギ感や、buyer が「結局どうなるの？」と感じる違和感は、  
単なる文体の問題ではなく、

- 支払い状態
- 作業開始条件
- 見積り提案
- 購入
- 成果物返却
- closed 後の再相談導線

といった **取引構造の理解不足として表に出る** ことが分かった。

これは今日一番大きな発見だった。

## 1. 最初の流れ — #RE の監査精度が上がっていることを確認

この日は、前回から続く `#RE` ループをそのまま継続した。

最初に確認したのは、監査がかなり厳しくなっていることだった。

- lint は通る
- でも reviewer が落とす
- 形式や文法は合っている
- しかし実務返信としては危ない

という状態が何度も出た。

これは一見すると「壊れている」ように見えたが、実際には重要だった。

なぜなら、返信システムの価値は、
単に自然な日本語を出すことではなく、

**文法的には正しいが、実務上危ない返信を見抜くこと**

にあるから。

この段階で、`#RE` はただの文面修正ではなく、  
返信システムに足りない構造を発見する実験になっていることがかなり明確になった。

## 2. 「一息で言う」発見の効果

前日に見つかった大きな観点として、

> 可否・価格・方針を一息で言う

があった。

たとえば、

```text
対応できます。
この不具合なら15,000円で進められます。
```

は内容としては間違っていないが、短文が分断されていて bot 感が出る。

これを、

```text
この不具合なら15,000円で対応できます。
```

と一文にすると、人間の見積り返信にかなり近づく。

この観点はその後の batch でも効いていた。

単に言葉を短くするのではなく、  
人間が自然に一息で言う判断単位を、返信システム側も一息で出す。

この発見で、文面のぶつ切り感がかなり減った。

## 3. Pro / DeepResearch / #AR の使い分けが固まる

この日は、ChatGPT Pro / DeepResearch / #AR の使い分けもかなり明確になった。

### Pro

Pro は、返信システムの構造監査に使った。

特に、

- `#RE` の学習効率
- service state の管理
- 仕様理解と返信品質の関係
- transaction model gap の仮説

のような、抽象度が高い分析に強かった。

Pro は、こちらの仮説である

> ツギハギ感は日本語だけの問題ではなく、仕様・phase・取引構造の欠落として出る

を支持した。

これはかなり大きかった。

### DeepResearch

DeepResearch は、外部の知識や一般論を整理する用途に向いていた。

ただし、調査結果をそのまま rule に入れるとノイズになる。

そのため、DeepResearch は、

- 外部観察
- 調査メモ
- reviewer 観点

として使い、正本へ入れる時は Codex が選別する運用になった。

### #AR

#AR は、ココナラの仕様や公式情報、ユーザー側の実態に近い情報を取るのに強かった。

特に closed 後の扱いで、

- 旧トークルームが閉じている
- 旧トークルーム内での続きやおひねりを前提にしない
- メッセージ上で確認材料を見ることはあり得る
- でも実作業には見積り提案や新規依頼の導線が必要

という理解を深めるのに効いた。

## 4. phase contract の整備で返信文が一気に自然になった

この日の前半で、ココナラの phase contract をかなり厳密にした。

主に整理したのは次の state。

- `quote_sent`
- `purchased`
- `delivered`
- `closed`
- `cancelled`

特に重要だったのは、`delivered` と `closed` の違い。

最初は、

> delivered は正式納品後だから、もう閉じているのか？

という理解が曖昧だった。

しかし整理すると、

- `delivered`: 正式な納品後・クローズ前。buyer が承諾するか差し戻すかの段階
- `closed`: 取引完了としてトークルームが閉じている段階

だった。

この理解が入ると、返信文はかなり自然になった。

たとえば delivered では、

- 無理に承諾しなくてよい
- 気になる点は承諾前にトークルーム内で伝える
- 承諾後に同じ問題が出た場合は、メッセージ上で前回修正とのつながりを見る
- 実作業が必要なら見積り提案または新規依頼へ戻す

という導線が出せるようになった。

これにより、buyer が「承諾したら終わりなの？」「あとから見つかったらどうなるの？」で迷わなくなった。

## 5. `phase_answer_gap` の発見

phase contract を整えた後、次に見えてきたのが `phase_answer_gap` だった。

これは、

> 文面は安全そうだが、その phase で buyer が次に取れる行動が見えない

という失敗。

たとえば、

```text
確認します。
```

だけなら安全に見える。

しかし buyer からすると、

- 承諾していいのか
- 差し戻すべきなのか
- メッセージで送ればいいのか
- 新規購入が必要なのか
- 作業に入るのか
- 見積りを待てばいいのか

が分からない。

これが `phase_answer_gap`。

このラベルを監査プロンプトに入れたことで、  
「安全だけど導線がない返信」を reviewer が拾いやすくなった。

重要なのは、これを writer rule にしなかったこと。

毎回説明を増やすとノイズになる。
だから、`phase_answer_gap` は reviewer レンズとして置き、  
再発したものだけ gold / validator / rule に戻す運用にした。

## 6. `unnamed_discomfort` の追加

その後、ユーザー側から、

> まだ名前がついていない違和感を、AIにも拾わせられないか

という指摘が出た。

これを受けて、監査プロンプトに `unnamed_discomfort` を追加した。

これは、

- 既存ラベルにはまだ当てはまらない
- でも buyer が詰まりそう
- 逃げに見えそう
- 商売上弱そう
- 実務リスクを説明できる

という違和感を拾うための枠。

ただし、ノイズにならないように、

- 最大1〜2件まで
- 好み差では使わない
- 単発では rule 化しない

という制約をつけた。

この枠はかなり重要だった。

なぜなら、今後も新しい失敗パターンは、最初から名前付きでは出てこないから。

まず違和感として現れ、  
それが再発すると名前がつき、  
gold や validator に落ちる。

この流れが見えた。

## 7. closed 後の確認材料と実作業境界

その後の `#RE` で、closed 後のケースを重点的に回した。

ここで出た問題は、かなり大きかった。

closed 後に、

- ログやスクショを送れば見てもらえるか
- ZIPでコードを送れば修正して返してもらえるか
- Google Drive で共有していいか
- .env や Stripe キーも送るべきか
- 無料で直してもらえるか
- 見積り前に原因だけ見てもらえるか

というケースを投げたところ、r0 ではほぼ全部が generic fallback になった。

```text
今回のご相談がどの種類かを見てから案内します。
```

のような文面で、具体質問に答えていなかった。

ここで分かったのは、

**closed 後の確認材料と実作業の境界が、まだ十分に入っていなかった**

ということ。

整理後は、次のようになった。

- ログ・スクショ・最小限の関係ファイルは確認材料として見られる
- コード修正、差し替えファイル作成、成果物返却は実作業
- Google Drive など外部共有には逃がさない
- `.env` / APIキー / Webhook secret / ログイン情報は受け取らない
- 見積り前の原因調査はしない
- ただし症状概要から見立ては返せる
- 実作業が必要な場合は、見積り提案または新規依頼へ戻す

これが `Gold Reply 23` と learning-log に落ちた。

## 8. 怒り気味 buyer への closed 後無料対応要求

closed 後の中でも、特に難しかったのが、

> 前回の修正後から同じようなエラーが残っているので、無料で直してもらえますよね？また15,000円かかるのは納得できません。

というケースだった。

最初の修正版は、正しいがやや事務的だった。

Claude は、感情受け止めを強める案を出した。
しかしその案には、

- `前回の修正で解決しているべき`
- `放置するつもりはありません`

のように、無料対応や責任承認に読まれる危険があった。

Codex xhigh はそこを指摘した。

その後、ユーザーがさらに踏み込み、

> そもそも closed 後に作業するには、ココナラ上で再度取引導線に戻る必要があるのでは？

と指摘した。

ここがかなり重要だった。

単に「無料かどうか」ではなく、

- closed 後はこの場で修正作業に入れない
- まず確認材料を見る
- 前回の修正とつながる症状か確認する
- 確認しないまま通常料金の新規依頼として進めない
- ただし無料実作業も約束しない
- 作業が必要になりそうなら、ココナラ上でどう進めるかと費用が発生するかを先に相談する

という流れが必要だった。

ここで一気に文面が自然になった。

採用した方向は次のようなもの。

```text
前回の修正後から同じようなエラーが残っていて、また費用がかかるのは納得できないとのこと、確認しました。

その場合は、まず前回の修正とつながる症状かを確認するのが先です。
確認しないまま、通常料金の新規依頼として進めることはしません。

恐れ入りますが、トークルームは閉じているため、この場でそのまま修正作業に入ることはできません。
まずはエラー内容やログ、スクショを見て、前回の修正とつながる内容か、別の原因として追加の作業が必要な内容かを確認します。

コード修正などの作業が必要になりそうな場合は、作業に入る前に、ココナラ上でどう進めるかと費用が発生するかを先にご相談します。
```

この形は、かなり大きな到達点だった。

## 9. ココナラ仕様の理解が入ると文章が自然になる

この日、特に印象的だったのは、

> 仕様を理解すると、まるで言葉がするする出て自信がある人みたいな文章になる

ということだった。

逆に言うと、ツギハギ感が出る時は、

- 日本語が下手
- モデルが弱い
- テンプレが悪い

だけではなく、

**ココナラ上で実際にどう進めるのかをシステムが理解しきれていない**

ことが多かった。

たとえば closed 後の無料対応不満で、

- 旧トークルームは閉じている
- でもメッセージ上で確認材料を見ることはできる
- ただし修正作業はその場ではできない
- 実作業が必要なら新しい取引導線が必要
- その時に費用が発生するかは先に相談する

という構造が見えると、文章は急に自然になる。

これは人間の実務対応にも近い。

人間も、仕組みが分かっていない時は言葉が曖昧になる。
仕組みが分かると、自信を持って短く言える。

この日、それが AI でも同じように起きることが見えた。

## 10. `transaction_model_gap` の命名

この一連の発見を受けて、`transaction_model_gap` というラベルを作った。

定義は、

> 日本語は一見自然でも、Coconala の取引構造、支払い状態、作業開始条件、費用発生条件、成果物返却経路のどれかが曖昧で、結果として文章がツギハギ・逃げ・自信不足に見える

というもの。

このラベルは writer rule ではなく、reviewer 専用レンズにした。

理由は、毎回 writer にこの観点を常駐させると重くなるから。

ただし、deterministic に検出できる事故は validator に戻した。

具体的には、

- quote_sent で、入金前にコード/ログ/原因確認へ進ませる事故
- purchased で、追加症状の別料金不安に答えない事故

を validator guard にした。

## 12. batch-11 r1 / r2 で安定

Codex と Claude の監査結果はかなり揃っていた。

r1 では次のように直した。

### B01

購入後にトークルームが開いてから正式確認。

### B02

同じ原因なら今回内。
別原因なら先に相談。
追加料金は勝手に発生しない。

### B03

delivered では、承諾前と承諾後を分ける。

- 気になる点があるなら無理に承諾しない
- 承諾前にトークルーム内で伝える
- 承諾後に同じ問題が出たらメッセージ上で前回修正とのつながりを見る
- 実作業が必要なら見積り提案または新規依頼

### B05

購入なしでコードの直し方を具体的に教えるのは、ファイル返却がなくても修正作業の一部。

### B06

次回相談は、

1. まずメッセージで症状概要
2. 内容確認
3. 見積り提案
4. 購入

という順序。

r1 は採用可。
