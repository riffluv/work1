# phase vocabulary guard

## 目的

- prequote を「見積もり判断の場」として固定する
- purchased / closed の語彙や、実確認開始の語彙を prequote に混ぜない

## 基本

- prequote は、コード・ログ・ZIP・権限を受け取って実確認を始める場ではない
- bugfix の prequote では、こちらから ZIP を標準要求しない
- prequote の本文で、未受領のファイルやコードを前提に `確認します` `見ます` `切り分けます` `調査します` と書かない
- purchased 以降で使う `確認を進めます` `いま見ています` `コードを見て返します` は prequote に混ぜない
- prequote で追加情報を求める時は、`その情報があれば見積もれる` `対象かどうかを判断できる` と結び、確認開始報告にしない

## 直答の基本

- buyer の `直せますか` `見てもらえますか` には、`対応できます` `まず1点だけ確認します` のどちらかへ寄せる
- buyer の `15,000円で対応できますか` `15,000円で済みますか` には、`進められます / まだ言い切れないので先に1点確認します / その料金では受けません` の3択へ寄せる
- `見積もり判断できます` `対象候補です` のような内部寄りの入口語は、直答の代わりに使わない

## buyer の判断段階

- 主質問に答えた直後の次アクションは、`こちらが次に欲しい情報` ではなく `buyer が次にやる判断` に合わせる
- 価格が主質問のターンでは、価格回答の直後にスクショ・ログ・ファイル要求まで進めすぎない
- 極短文の hold でも、質問より先に `今の情報なら先に対応できるかは確認できます` など、buyer に見える1文を置く

## anxious の扱い

- anxious な prequote では、感謝だけで始めず `先に状況を絞ります` `まず見られる内容かだけ確認します` など buyer に見えるアクションを先に置く
- `まず対象かどうかを判断します` `見積もり判断を急ぎます` のような internal-ish な言い方より、buyer に見える動詞を優先する

## 禁止

- `対象候補です`
- `見積もり判断できます`
- `この不具合として` `この内容として` `この症状として`
- `確認したいので`
- 未受領なのに `確認を進めます`
