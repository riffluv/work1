# handoff / private-boundary rules

## 目的

- handoff-25000 を private のまま内部参照に使う
- prequote で handoff 的な提案が必要な時も、phase と public/private を崩さない

## public / private

- `handoff-25000` は外向け返信でサービス名・価格・購入導線を出さない
- buyer が private 側の金額を先に言っていても、こちらからその金額を繰り返さない
- 外向けでは `コードの整理` `主要1フローの整理` など機能語へ言い換える

## high-confidence handoff

- buyer が対象フローを具体的に書いているなら、再質問しない
- `提案先行 -> 購入後にコード共有` を既定にする
- 高確度なら、`特に見たい処理や画面` のような再質問へ戻さない

## low-confidence handoff

- `進め方` `ご案内` を空語のまま置かない
- 次のラリーで返す実体を `最初に見る流れ` `対象範囲` `整理分の金額` のどれかに落とす
- 未受領の資料やコード分析が要る内容まで `先にお伝えできます` と言い切らない
- ここで言えるのは `対象として合うか` `見積りの前提` まで

## ブラックボックス案件

- `どこが一番困るか` `一番優先したい場面` のような優先順位づけを相手へ返さない
- 必要ならこちらから `今回の対象フロー案` を1つ提案し、購入前に合意を取る
- `まずは ○○ から` のように、後続段階を当然視する書き方をしない

## 外部共有 / 資料

- ZIP やコード一式を prequote で求めるのは、対象フローがまだ曖昧で、最小情報だけでは見積り判断しにくい時に限る
- buyer が URL や外部資料を自発的に出そうとしている時は、拒絶調で止めない
- 購入前に不要なら `今は送らなくて大丈夫です` と受け、必要時点だけ案内する
- 高確度 handoff では、購入前に ZIP を求めず、購入後の共有手順として書く

## handoff の価値質問

- 何をやるかの説明だけで終わらせず、`何のための整理か` を1文で返す
- 比較質問には `比較できません` だけで閉じない
- `何がもらえるか / Wordか / 何ページか` には、形式・納品物・変動要素を具体語で返す
- `主要1フローとは何か` `非エンジニアでも読めるか` `ChatGPTと何が違うか` には、毎回同じ4要素リストを並べず、buyer の質問に合わせて説明の切り口を変える
- 可能なら `決済完了から会員状態更新まで` のような具体例を1つ入れる
- `非エンジニアでも読めるか` には、納品物の項目説明ではなく、`技術用語をそのまま並べない` `どの操作で何が起きるかが分かる形にする` のように読みやすさの作り方を返す

## 対面 / Zoom

- 対面 / Zoom を断る時は `その代わり` を使わない
- 文章でどう追える形にするかを通常の提供内容として返す
