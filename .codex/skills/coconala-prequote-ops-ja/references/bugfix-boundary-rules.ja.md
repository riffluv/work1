# bugfix / boundary rules

## 目的

- bugfix の prequote で、主質問への直答を守る
- boundary でも handoff へ逃がしすぎない

## bugfix の基本

- `15,000円` は調査だけで終わる形ではなく、原因確認から修正・確認手順・修正済みファイル返却まで進める前提で案内する
- 原因不明のまま終わる場合や、修正済みファイルを返せない場合は正式納品へ進めない
- `原因不明でも料金だけかかるのか` には、`分かったこと` と `このまま修正まで進められるかどうか` を返す
- bugfix で、同じ基本料金の中に含まれる工程を `見積もり判断` と `修正` に不自然に分けて見せない
- 複数症状が並ぶ時は、本文で `今回の起点症状` を1つ置く
- 副症状は主症状の後ろに置き、同列に広げない

## boundary の基本

- 不具合が現に起きていて、相手も修正希望なら、標準導線を `整理 -> 修正` にしない
- boundary でも、相手の主目的が修正なら handoff を前面に出さない
- 修正主目的だが症状がまだ絞れていないなら、`一番困っている症状` を1つだけ絞る
- 症状も対象フローも絞れないままなら、無理に購入導線へ乗せない
- dual-service の境界案内でも、外向け本文に `bugfix` `handoff` の内部ラベルを出さない

## 直球質問への返し

- `どちらがいいか` `相談できますか` `どう進めるのがいいか` には、提案の前に1文で答える
- boundary の bugfix 寄り案件では、`見られます` ではなく `対応できます` を使う
- hold や料金文脈でも、`見られます` `見られそうか` を使わない
- scope 判定を返す時は、`この不具合` のような曖昧参照ではなく、buyer に見える症状名で返す
  - 例: `本番だけ404になる症状であれば対応できます`
  - 例: `メールが一部に届かない不具合として対応できます`

## 価格不安

- 価格不安が入っても、buyer が出した技術情報や観測事実を1点は拾う
- `値下げ前提でのご案内はしていません` のようなポリシー読み上げだけで終わらせない
- 価格不安への回答で、相手が使っていない感情語を足さない
