# ココナラ出品文（日本語）

## 目的
- 機械的で硬い文を、自然で誠実な実務文に変換する。
- 読み手が「何を頼めるか」「どこまで対応か」を短時間で理解できる文にする。
- 固定条件（価格、範囲、禁止事項）を崩さない。

## このskillで扱う対象
- プロフィール文
- ひとことアピール
- 出品サービス本文
- FAQ（`Q:`/`A:` 形式）
- 見積り時の案内文

## 参照ファイル
- 固定条件: `references/fixed-facts.ja.md`
- 文体・構成ルール: `references/listing-style-rules.ja.md`

## モード判定
- 相手文や現行文の貼り付けのみ: **分析のみ**を返す。
- 「作って」「書いて」「リライトして」など明示: **送信用文面**を返す。
- 判定が曖昧: まず分析のみを返し、最後に「必要なら送信用文面を作成できます」と1行添える。

## 作成フロー
1. 依頼の対象を特定する（プロフィール / 出品本文 / FAQ / 見積り文）。
2. `references/fixed-facts.ja.md` を先に読み、固定条件をロックする。
3. トーンを決める（硬め / 標準 / やわらかめ）。指定がなければ**標準**。
4. `references/listing-style-rules.ja.md` の構成に沿って文を再構成する。
5. NG表現を言い換え、具体語（対象・範囲・期限）を補う。
6. 仕上げ前チェックを実施し、必要ならA/B 2案を返す。

## 出力ルール
- 行頭に不要なスペースを入れない（コピペ安全）。
- 絵文字と過度な感嘆符は使わない。
- です・ます調を基本にする。
- 長文を避け、見出しと箇条書きで整理する。
- FAQは原則 `Q:` / `A:` 形式。
- ユーザー指定がない限り、最終案は1本。比較希望時のみA/Bを出す。

## 固定ガード
- 価格を勝手に変更しない。
- 不具合1件の定義（同一原因 / 1フロー / 1エンドポイント）を崩さない。
- 範囲外（通話、直接push、本番デプロイ、新機能開発）を削除しない。
- 外部連絡・外部共有・外部決済への誘導を書かない。
- 秘密情報（`.env`値、APIキー値）を要求しない。

## 仕上げ前チェック
- 冒頭3行で「対象・価値・安心要素」が伝わるか。
- 固定条件（価格・範囲・禁止事項）が保持されているか。
- 抽象語だけで終わっていないか（具体語を入れたか）。
- 過剰敬語やAI臭ワードが残っていないか。
- 読み手が次に取る行動（相談・購入・返信）が明確か。

## 想定リクエスト例
- 「プロフィール文を自然に直して」
- 「出品本文を硬すぎない日本語にして」
- 「FAQをQ/A形式で作って」
- 「今の文を標準トーンでリライトして」
