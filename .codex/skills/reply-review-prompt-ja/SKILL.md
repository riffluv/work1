# 外部レビュー用プロンプト作成

## 目的
- 送信用返信文を、外部AIに日本語品質レビューさせるための監査プロンプトを安定して作る。
- 主軸は Codex のままにし、外部AIはレビュー担当に限定する。
- 単発の感想ではなく、最小修正版まで返ってくる形を作る。
- この skill は単発の返信文監査用であり、返信OSの設計相談・ロードマップ相談・Pro向け分析依頼のプロンプト作成には使わない。

## このskillを使う場面
- ユーザーが `claudeに聞いてみて` `geminiに聞いてみて` と、直近または指定の返信文監査を依頼したとき
- ユーザーが `#CL` または `#GE` を付けたとき
- 直近の送信用返信文に対する日本語監査プロンプトが欲しいとき

## このskillを使わない場面
- ChatGPT Pro に、返信OS全体・日本語自然化レイヤ・監査レンズ・ロードマップ・実装方針を分析してもらうプロンプトを作るとき
- `Proに聞くプロンプトを書いて` でも、対象が単発返信文の監査ではなく設計相談の場合
- #RE / #BR の batch 監査プロンプトや、Codex xhigh 用の監査プロンプトを更新するとき

## 対象文面の優先順位
1. `#CL` または `#GE` のときは `/home/hr-hm/Project/work/runtime/replies/latest-source.txt` と `/home/hr-hm/Project/work/runtime/replies/latest.txt`
2. ユーザーが対象文面やファイルを明示したときはその文面
3. 直前に Codex が作った送信用返信文が明確なときはその文面

対象が見つからない場合は、推測で作らず、その旨を短く伝える。

## reviewer の選び方
- `claudeに聞いてみて` または `#CL`: reviewer は Claude
- `geminiに聞いてみて` または `#GE`: reviewer は Gemini
- 指定が曖昧なら、単発返信文の監査かどうかを先に判断する。単発返信文ではない場合、この skill は使わない。

## 作成ルール
1. reviewer には「新しい設計者ではなく、レビュー担当」と明記する。
2. 主軸は Codex で、採用判断は Codex が行うと明記する。
3. 全面改稿ではなく、`どこが不自然か / なぜか / 最小修正版` を求める。
4. 対象が送信用返信なら、次を標準観点に入れる。
   - 日本人が読んで、文章がスッと入ってくるか
   - 日本語として自然か
   - AI感がないか
   - 日本のビジネスチャットを日常的に使う人が見て、ネイティブならこの場面ではあまり言わない受け方になっていないか
   - `確認しました` `受け取りました` の直後に `はい、〜確認できています` のような二重受領が出ていないか
   - `はい、` が質問への機械的な yes として付いており、会話の流れに対して不自然になっていないか
   - 内容は正しいが、bot / AI っぽく見える受領確認・相づち・つなぎ語がないか
   - 重すぎないか
   - 実務でそのまま送れるか
   - 要点が薄くなっていないか
   - 相手文に正面から答えているか
   - 相手が不安や過去の嫌な体験を明示しているとき、返信の結論が `整理してお返しします` `確認して返します` のような報告語で弱まっていないか
   - 相手が不安を明示しているとき、`対応します` `進めます` のような行動コミットで着地できているか
   - `今回の` `こちら` `その件` `この内容` などの参照語が、返信単体で何を指すか明確か
   - こちらが動くべき場面で、`ご相談ください` `ご確認ください` `ご連絡ください` のような締めに逃げていないか
   - `切り分ける` `整理する` `ハンドリングする` のような、作業者側の内部プロセス語が buyer 向けの文にそのまま出ていないか
   - buyer が知りたい結果説明より、こちらの作業工程の名前を先に見せていないか
   - `前提です` `出方` `仕様上` など、壁感や内部者目線を作りやすい語が、相手向け説明として自然か
   - 返信が引用している事実や症状名が、buyer のメッセージに実際にあるか。存在しない事実や技術語へすり替わっていないか
   - 隣り合う文が、言い換えただけで同じ意味を繰り返していないか
   - 返信の最後に、buyer が次に何をすればよいかが示されているか。値引き拒否や範囲外の説明で終わる場合も、次の入口があるか
   - buyer が急ぎ・不安・過去の嫌な経験を明示している場合、返信の冒頭2文以内でそれに触れているか
   - フェーズに合う距離感か
   - 購入後の継続会話なのに、初回ヒアリングや受付窓口の文面に戻っていないか
   - 相手が `おまかせで` `どこが大事か分からない` の温度感なら、優先順位や考える作業をまた相手へ返していないか
5. ユーザーが追加で気にしている点があれば、観点へそのまま足す。
6. 出力形式は次を基本にする。
   - 総評
   - スッと入ってこない箇所
   - 不自然・AI感のある箇所
   - 日本語ネイティブの実務者なら言わなさそうな箇所
   - その理由
   - 最小修正版
   - このまま送ってよいか
   - 恒久観点にしてよいもの
   - 好み差として流してよいもの
   - 引っかかった観点ID
   - 再発タグ
7. プロンプト本文はコードブロックで返す。

## `#CL` / `#GE` の扱い
- `#CL` は「直近の送信用返信文を対象に、Claude向けの監査プロンプトを作る」ショートカット
- `#GE` は「直近の送信用返信文を対象に、Gemini向けの監査プロンプトを作る」ショートカット
- 対象未指定なら、相手文は `/home/hr-hm/Project/work/runtime/replies/latest-source.txt`、返信文は `/home/hr-hm/Project/work/runtime/replies/latest.txt` を使う
- `latest-source.txt` が無い場合は、返信文のみを対象にし、その制約をプロンプト内に一言添える
- `#CL` / `#GE` の後ろや次行に補足があれば、重点観点としてプロンプトへ反映する

## 恒久反映の注意
- reviewer の指摘をそのまま採用しない
- workspace へ恒久反映するのは、再発性があり、実務ノイズを増やさず、要点を薄めず、既存ルールと衝突せず、1回限りの好みではないものだけ
- `はい、` 全般を禁止しない。二重受領、機械的な yes、同じ意味の近接反復など、送信用文面として実務上の違和感が再発するものだけ恒久候補にする

## 出力ルール
- 余計な前置きは付けず、コピペしやすい監査プロンプトをそのまま返す
- ユーザーが reviewer を明示している場合は、その reviewer 名を明記する
- 相手文と返信文の両方がある場合は、両方とも省略せず、プロンプト内へ全文を入れる
