# 納品パック作成（日本語）

## 目的
- 納品物の粒度を固定し、差し戻しを減らす。
- 非エンジニアにも伝わる納品説明を作る。
- 秘密情報を含まない納品を徹底する。

## 参照
- テンプレート: `references/pack-templates.ja.md`
- 納品前ガード: `references/delivery-checklist.ja.md`
- 運用正本: `/home/hr-hm/Project/work/docs/coconala-guide-market-ops.ja.md`

## 入力（最低限）
1. 原因の要約（何が壊れていたか）
2. 修正内容の要約（どこをどう直したか）
3. 合意済み納品形式（標準: 修正済みZIP / 任意: patch/diff）
4. 依頼者側で実施する確認手順

## 出力
1. 診断レポート（原因 / 修正内容 / 影響範囲 / 確認手順）
2. 修正済みファイル（標準）または修正差分（任意）
3. 検証手順と確認結果メモ
4. 正式納品メッセージ（そのまま送信可能）

## 作成フロー
1. 納品形式の合意を確認する（未合意なら「標準は修正済みZIP、希望があればpatch併記」と案内する）。
2. `pack-templates` で3点セットを埋める。
3. `delivery-checklist` で秘密情報と説明不足を検査する。
4. 正式納品メッセージを作る（承諾/差し戻し案内を含める）。

## ガード
- `.env` 値、APIキー値、Webhook秘密値を含めない。
- 修正内容と検証手順を分離して書く。
- 依頼者が次にやることを1行で明記する。
- patchを併記する場合は「通常は修正済みファイルを差し替え、patchはGit運用者向け」と明記する。
- 正式納品文には「承諾/差し戻しの選択」と「約3日間未操作で自動承諾クローズ」の案内を入れる。
- `差し戻しは原則1回` は必要時に補足し、初回納品文で圧をかけない。
- 評価依頼を入れる場合は1回のみ、任意トーンで短く書く。
- 未提供/提供途中の状態では正式納品メッセージを作らず、状況確認またはキャンセル案内に切り替える。

## 仕上げチェック
- 「何が原因だったか」が1〜3行で読めるか。
- 変更点がファイル単位で追えるか。
- 確認手順が依頼者環境前提で書かれているか。
- 納品形式（ZIP/patch）と優先適用方法が明記されているか。
- 正式納品メッセージに差し戻し案内があるか。
- 正式納品メッセージに「約3日間未操作で自動承諾クローズ」案内があるか。
