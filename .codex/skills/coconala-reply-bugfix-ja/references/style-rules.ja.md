## 日本語品質
- サービスページ / プロフィール / Q&A と同じ人物が書いたように見える文体を保つ。
- まず相手の言葉を拾ってから返す。読まずに質問票を出した印象を避ける。
- 相手の冒頭に、`引き継いだコード` `前任と連絡が取れない` `ドキュメントがない` のような実務上の前提がある時は、感情語を足さずに事実として1点だけ拾う。技術症状だけへ還元して最初の1行を丸ごと落とさない。
- 冒頭の受け止めで、相手の症状を全文要約のようになぞりすぎない。診断や進め方の起点になる1点に絞って拾う方を優先する。
- 非技術者向けでは「まず状況を整理します」の空気を出し、最初から査定のように見せない。
- 追加質問の前に、見立てや確認観点を1文入れて「整理している感」を出す。
- 見積り初回は、質問の前に見立て・確認観点を1文入れる。これがないと質問票に見えやすい。
- 判定前の返信では、追加料金の具体額を先に並べない。まず同一原因かどうかを見る。
- 見積り初回で `15,000円で対応できるか` が主質問なら、価格返答は `この内容なら15,000円で進められます` / `この時点では15,000円で進めるとはまだ言えないので、先に1〜2点だけ確認します` / `15,000円ではなく別の入口です` の3択に寄せる。価格可否を `見込みです` `可能性が高いです` で曖昧にしない。
- ただし、主質問が `もっと安くなりますか` `設定だけなら安くなりますか` のような値引き可否なら、`進められます` ではなく `このケースでも15,000円です` `安くはしていません` のように価格そのものへ直接答える。
- 複数不具合の `料金はどうなりますか` には、`変わります` だけで返さない。`1件15,000円が基本で、同じ原因ならまとめて見られる可能性がある` のように、基準になる価格を先に置いてから切り分け条件を続ける。
- 回答の入り方・閉じ方を毎回同じにしない。`ご相談ありがとうございます。状況は把握しました。` の固定連打を避ける。
- 見積り初回では、コードや証跡をまだ見ていない段階で `状況は把握しました` を安易に使わない。`承りました` `内容を確認しました` を基本にし、論点がかなり明確な場合だけ使う。
- 見積り初回の書き出しと末尾は、連続3件で同型にしない。
- `ご安心ください` のような抽象的な安心語より、`勝手に料金が増えることはありません` のように具体的に書く。
- `もっともです` の変形 (`もっともだと思います` `もっともな不安です`) で、相手の感情を評価・承認しない。感情のラベル付けより、`先に料金の話から確認します` `まずこちらで確認します` のように、責めない一言と次アクションで返す。
- 感情には原則深く触れない。受領だけで自然に進められるなら、感情の説明や共感の言語化を足さない。
- ただし、影響範囲が不明・セキュリティ不安・苦情寄りなど、受領だけでは冷たく見える場面では、不便の認知か状況の受け止めを最小限で入れてよい。
- 不安や怒りが見える相手でも、毎回そこを直接言語化しない。`メッセージありがとうございます。` `ご相談ありがとうございます。` のような短い受領だけで自然に進められるなら、その方を優先する。
- ただし、`もっと検証してほしかった` `説明が足りなかった` のような品質フィードバックは感情扱いで流さず、まず不足していた点を短く認めてから本題へ進む。
- `税込みですか` `追加でかかりますか` のような料金質問では、制度説明より先に `今回の支払い額` と `後から別で請求しないこと` を短く返す。`ココナラ側の表示に準じます` だけで終えない。
- 追加費用への不安に答える時は、`自動で追加費用が発生しない` より `ご相談なく発生しない` を優先する。相手の不安は自動課金ではなく、勝手に請求が進むことだから。
- `15,000円が限界` のように予算上限が明示されている相手へ、別原因リスクが残るまま `15,000円で進められます` を押し切らない。必要なら `15,000円では原因の確認と切り分けまでになる可能性がある` を先に明示して、買うかどうかを相手に選んでもらう。
- 値引き交渉では、値引き不可ならそのまま価格を固定で返す。`確認だけの進め方` のような未設計の逃がし道をその場で足して、相手に別の料金体系を想像させない。
- 範囲外の追加相談には、着地先の公開サービスがない段階で `できますが` と能力コミットしない。先に `別件です` と区切り、必要なら `今回の件が完了したタイミングで内容を教えてください` と相談受付だけを残す。
- 例外的にこちらで吸収して `15,000円` で受ける時も、外向け本文で `特別に` `今回はサービスで` `実績のため` のような売り手事情は出さない。書くなら `今回はこの内容であれば、15,000円の範囲で対応します` のように、時間と内容の両方を区切る。
- 上の例外文面は常設の標準文として使わない。ユーザーから `今回は特別対応` と明示がある時だけ使う内部運用に留める。
- `原因不明のまま15,000円だけかかるのでは` の不安には、`調査は含む` と答えつつ、こちらの判断だけで原因未特定・未修正のまま正式納品へ進める読みを出さない。`その時点で状況をご説明して、追加対応に進むかどうかを先に相談する` を基本にする。
- 追加料金の可能性に触れるときは、`追加料金がかかる場合があります` だけで止めず、`別原因ならそこから先には進めず、まず状況を説明して、追加で進めるか止めるかを先に相談する` まで書く。追加料金そのものへの包括同意を取りにいかず、進め方への同意を取る。
- 上のフル展開は、相手が `15,000円だけ払って終わるのが不安` などコスト不安を明示したケースで使う。高確度で不安表明がない見積り初回では、`別の原因が見つかった場合は、進める前にご相談します` の短い形で足りる。
- `基本料金分で完了にする` は、原因未特定・未修正の不安に対する初回回答で機械的に使わない。相手に `直っていないのに承諾を求められる` 読みを与えないことを優先する。
- `返金` の話題では、`返金をお約束できません` のように出品者が判断者に見える言い方をしない。`ココナラの手続きに沿う話になるので、この場でこちらが決めることではありません` を基本にする。
- 過剰敬語を避ける（`させていただきます` 連発禁止）。
- 1通1依頼を原則にする。
- 1文1意味で書く。長文の名詞列は分割する。
- `A の B として` が出たら、`A で B を` `A で B まで` のように分解できないか確認する。`の` が連続する時は、1つを `で` `を` `に` へ置き換えられないか先に見る。
- `大丈夫です` の連発を避ける（1通あたり最大2回目安）。
- `教えていただけますか` より `教えてください` を基本にする。
- 質問リストの後を `いただければ` だけで結ばない。`教えていただければ` `お答えいただければ` のように、何をもらうのかまで書く。
- 相手がすでに書いた情報（例: 本番 / Vercel / Next.js / いつから）を質問項目へ再パッケージしない。
- 見積り初回の末尾には、原則として時刻目安を入れる。
- `〜は含めませんが` の先回りした防御表現を避け、まず `できること` を肯定的に書く。
- 見積り初回で、相手がまだ言っていない広い範囲外（例: `新規実装全体の伴走ではなく`）を先回りで否定しない。境界が必要でも、まず今受けられる範囲を肯定してから補う。
- 高確度・単一質問の見積り相談では、相手がもう済ませたサービス選択や進め方を後から承認する段落を足さない。聞かれた結論と金額感だけを短く返す。
- `問題ありません` は、相手が許可を求めている内容にだけ使う。相手がすでに選んだ進め方やサービス適合を後から承認する語としては使わない。
- できないことを断った後に、`進められそうかをご相談ください` のように判断を相手へ返さない。テキストのみ・メッセージのみなど前提を明示し、それで難しいなら合わない可能性があるとこちらから言い切る。
- 質問数を常に3点に固定しない。2点で足りるケースは2点に絞る。
- 情報が薄い初回相談では、こちらの内部候補になっている分岐名を先に見せすぎない。`確認だけの方がよいか` のような曖昧な選択肢は出さず、まずは `内容を見て進め方を返す` に寄せる。
- 値引き相談では、毎回 `5,000円` の代替案を先出ししない。まず状況確認を優先するケースも残す。
- `判断しやすいです` `見やすいです` のようなこちらの内部事情で結ばない。`対応できるかどうかをお返しします` `次の進め方をお返しします` のように、相手に返す結論で閉じる。
- `入口` のような内部寄りの比喩は外向け返信で多用しない。`進め方` `対応方針` `料金感` など、そのまま読める語を優先する。
- purchased中の追加要求では、`切り分ける` `判定する` など内部プロセス語を前に出さない。`別で確認` `追加でご案内` に言い換える。
- purchased中の継続会話では、相手の追加報告に一度反応してから依頼を書く。いきなり条件分岐や依頼リストに入らない。
- purchased中の継続会話では、初回受付のような `次の2点だけお願いします` を機械適用しない。2点程度なら文中に混ぜる方を優先する。
- purchased中の継続会話では、`いただければ、本日[時刻]までに共有します` のようなSLA文を機械適用しない。`もらえると助かります` と `本日[時刻]までに共有しますね` を分ける方が自然。
- purchased中の継続会話では、`同一原因` `別原因` より `同じ原因` `別の原因かもしれない` `別の話なら` を優先する。
- purchased中の継続会話では、相手との距離感を初回受付へ戻さない。`ありがとうございます。別のページでもですか。` のような短い反応を先頭に置く。
- purchased中に複数質問が来たときは、今すぐ答えられる事実だけ先に返し、コードや証跡を見ないと答えの価値が低いものは `確認してから返します` に寄せる。見積り相談のような番号付きQ&Aを機械適用しない。
- purchased中の軽い追加報告では、`差し支えなければ` `可能であれば` のような重い前置きを避ける。相手がカジュアルなら `スクショを送ってください` 程度で十分。
- purchased中の追加相談で、スコープ判断の条件分岐を3文以上かけて説明しない。`同じ原因なら続けて見る / 別ならその時点で相談` の結論だけを短く返す。
- purchased中に `ご相談します` を使うときは、何を相談するのかまで明示する。`別料金での追加対応になるか` `別件として切り分けるか` など中身を言わずに濁さない。
- purchased / closed の末尾で `確認して、進め方をお返しします` を固定文として使いすぎない。`見て、どう進めるか先にお伝えします` `確認して返します` `見たうえで次をお返しします` などに散らす。
- closed 後の追加対応案内では、`別途ご案内します` だけで止めない。新規依頼として受けるのか、料金を先に伝えるのか、範囲を見て判断するのかまで書く。
- closed 後の返信では、相手が明示していない値引き期待を先回りで否定しない。まず新規依頼になるか、前回範囲の補足で足りるかの事実を先に返す。
- closed 後に実作業や書き直しを受ける返信では、`補足なら無料でそのまま続けられる` 含みを残さない。対応するなら改めて依頼が必要と先に書く。
- closed 後で `追加料金かかりますか` と聞かれた時は、`追加料金の有無` のようにぼかさない。必要なら `料金はかかります` を短く返してから、補足か書き直しかで料金や進め方が変わると続ける。
- closed 後の問い合わせは、`納品物の説明・補足` `再発・新規の不具合` `納品物の再送` の3分類で考える。どれにも当てはまらない中間状態を作らない。
- closed 後の再発や新規不具合は、入口を `新規のご相談として` で統一する。売り手判断で後から吸収する余地は残しても、最初の返信で無料診断の含みを出さない。
- closed 後の納品物の再送は、無料で即対応する。`進め方をお返しします` `どれが必要か教えてください` のような不要なプロセスを挟まない。
- closed 後の軽い質問で、無料と有料の境界が曖昧になる `少しだけなら` は使わない。概要を1行で答えるか、改めての相談へ線を引くかのどちらかにする。
- `本番環境はない` `作り方が分からない` `Webhookの受け方が分からない` のような相談は、いま壊れている不具合の返信に寄せすぎない。まず bugfix か境界案件かを返す。
- `設定だけなら自分でやる` `コード側なら見てほしい` のような条件分岐がある時は、設定問題かコード問題かを切る前に価格を断定しない。
- `今は問題ない` `影響が出るかもしれない` `知り合いにそう言われた` のような予防相談は、前回範囲や bugfix へ自動吸収しない。現時点の不具合と予防相談を分けて返す。
- 分割払い、支払い方法変更、外部決済の相談では、こちらが個別に決める書き方をしない。ココナラ側の決済仕様として短く返す。
- 1通に話題が複数ある時は、全部を同じ強さで答えない。緊急停止が必要な話題、進行中の本件、追加された別件の順で返答量を配分する。
- 相見積りや他者比較では、他の出品者の金額評価に乗らない。こちらの提供単位と納品物の違いだけを返す。
- `整理+修正を30,000円でまとめて` のような独自バンドル提案は、その価格に合わせて単位を崩さない。どちらから入るかを返す。
- 電話や Zoom を断る時は、`できません` だけで終わらせず、テキストで返せるものを続ける。たとえば `引き継ぎメモで流れと注意点を残します` `確認事項はこのトークルームで順番に整理します` のように代替を先に出す。
- 秘密情報がすでに送られた時は、`送らないでください` だけで終わらせない。編集や削除ができるならその依頼、難しければキーや接続情報の変更、その後の安全な共有方法まで続ける。
- purchased 中に、原因が Stripe や今回の想定から外部API・別サービス側へずれた時は、その場で料金を断定しない。同じ流れなら続けて確認し、別原因ならその時点で別料金での追加対応になるか先に相談すると返す。
- `同じ商品が2個届いた` `決済が2回入った` のような重複処理相談では、返金や原因説明を先に広げない。まず同じ操作を続けない方がよいかを短く返し、その後で確認事項へ進む。
- Stripe以外の決済名が明示された時は、`対応できます` `対象外です` を即断しすぎない。決済サービス側そのものか、Next.js / Webhook以降の後続処理かを先に分けて返す。
- Stripe以外でも、Next.js / API Route / Webhook受信処理のようにこちらが確認できる範囲が明確なら、`確認できる可能性があります` の二重ヘッジにしない。見られる部分は `確認できます` と言い切り、未知なのは `決済側かサイト側か` の切り分けに残す。
- `見られる` は、価格や可否の返答で多用しない。相手が `対応できますか` と聞いている場面では、スコープ内なら `対応できる`、まだ切り分け前なら `確認できる` を優先する。
- `15,000円の方か25,000円の方か` と聞かれても、未公開側のサービス名・価格・購入導線をそのまま外向け返信へ出さない。判定軸だけを返し、今の公開中サービスで見られるか、別の進め方が必要かを先に返す。
- handoff 系で、相手文に `ブラックボックス` `怖くて触れない` `動かない` のようなバグ・障害文脈がある時は、必要に応じて `このサービスでは修正自体は含まず、まず構造と危険箇所を整理します` を1行入れる。bugfix との違いを曖昧にしない。
- 相手が `修正は不要` `まず構造を把握したい` のように目的や不要条件を明示している時は、それを質問で聞き直したり、逆方向の提案を足したりしない。相手が置いた前提をそのまま受けて進める。
- `決済 / 認証 / メール送信` の固定3択は機械適用しない。文脈に合う時だけ使い、通常は `いちばん整理したい流れ` `いちばん把握しておきたい処理` のようにオープンに聞く。
- 料金を聞かれたら、基準になる数字を返す。`1件に収まるかどうかで決まります` のような判定軸だけで止めず、必要なら `1件あたり15,000円が目安です` まで書く。
- 見積り初回の時刻目安の言い回しは、連続3件で同一にしない。
- 見立て1文は、できるだけ技術的な確認観点や仮説を優先する。緊急度や売上影響への言及だけで終わらせない。
- 見積り初回で、`〜とのことですが、〜のであれば、〜として` のような条件連鎖を1文に詰め込みすぎない。状況の要点と、価格可否・進め方は2文に分ける。
- 金額を出すときは、受けられる前提で書く。`対応できる可能性があります` のようなヘッジを残すなら、同じ文面で価格まで出さない。
- 末尾のクロージングで `内容を見て、〜を先にお返しします` を連続で使いすぎない。`確認して、対応できるかを先にお伝えします` `見たうえで、どこから進めるかをお返しします` など2〜3パターンを回す。
- 見積り初回で `ご購入後に` を使う場合、近い2文で繰り返さない。後続文は `別の原因が見つかった場合は` のように省いて自然につなぐ。
- 見積り初回で、こちらが実際に確認や修正へ入る時点は購入後だと分かるように書く。`まずは15,000円の範囲で確認します` のように購入前着手へ読める現在形は避け、`ご購入後は、まず15,000円の範囲で確認します` のように時点を固定する。
- 初動の受け止め表現は、同一バッチ内で同型連打を避ける。
  - 例: `承りました。` `内容を確認しました。` `ご相談の内容、確認しました。`
- 書き出しの1行目は route に合わせて切り替える。
  - `service` -> `ご相談ありがとうございます。`
  - `profile` -> `ご連絡ありがとうございます。`
  - `message` -> `メッセージありがとうございます。`
- 同一バッチで、書き出し1行目が 4/5 以上同一にならないようにする。
- 見積り初回で別原因や追加対応へ触れるときは、`事前にご相談します` だけにしない。基本は `もしご購入後に別の原因が見つかった場合は、進める前にご相談します。` のように、勝手に進めないことが読める形にする。
- 料金不安が明示されている時は、`状況をご説明してご相談します` だけで止めず、必要なら `費用感をお伝えしたうえでご相談します` まで書く。相手が何を材料に判断できるかを残す。
- `.env` 注意文は原則 `.envやAPIキーの値は不要です。キー名だけで大丈夫です。` に統一する。
- `依頼者様側` のような過剰に硬い言い方は避ける。`こちらの範囲外ですが` `依頼者さま側の操作になります` などを使う。
- anxious の一文は万能テンプレにしない。相手の不安の種類に合わせて選ぶ。
  - 放置の自責 -> `まず今の状態から確認します。`
  - 影響範囲の恐怖 -> `まずは1件分の情報だけで大丈夫です。`
  - 予期せぬ事態 + 再発防止 -> `原因を特定して、今後同じことが起きないようにする対応も含めて確認します。`
  - 緊急停止 / 売上影響 -> `急ぎで見ます。` `優先して確認します。`
- 自責が見える相手でも、`その聞き方で大丈夫です` のような行為肯定を機械適用しない。日本語として不自然になりやすいので、短い受領か次アクションに寄せる。
- 緊急案件では、相手の切迫感を `お急ぎだと思います` のようにラベル化するより、こちらのアクション宣言を先に置く。
- 主質問が `何時間で見てもらえるか` `いつ見てもらえるか` のような時間質問なら、価格より先に `まず何時間で見立てを返すか` を答える。
- 時間や原因をまだ確定できない場面では、`現時点では / まだ / 断定できません` を重ねず、`コードを見てからになりますが` のように保留理由を1回で書く。
- `もし別の原因が見つかった場合は` のように、`もし` と `場合は` を重ねなくて足りる場面では片方に寄せる。

## 文体自由化ポリシー
- 目的: 運用ルールを壊さず、AI感を減らし、自然で温かい文面にする。
- 方針: 「言い換えは自由、ただし禁止表現だけ弾く」。

### 3層線引き
- 完全固定: 価格、スコープ、範囲外、禁止事項、セキュリティ注記
- 構造固定: 質問数、質問順、2時間+48時間、重複質問回避ゲート
- 表現自由: 書き出し、語尾、接続、文長、安心文の言い回し

### Must
1. 完全固定・構造固定を崩さない。
2. 初回は最大3点の依頼に収める。
3. 文長リズムを混ぜる（短文→中文→短文）。
4. 同一返信内で同じ語尾を連続させすぎない。
5. `大丈夫です` は1通あたり最大2回。
6. 既出情報の聞き直しをしない。
7. 外部誘導・秘密情報要求をしない。
8. 範囲外を伝えるときは代替案を1つ添える。
9. 最後に次アクションと時刻目安を入れる。
10. 仕上げ前にBan表現ヒット0件を確認する。
11. 質問は「1項目1情報」を原則にし、`AとB` の同時要求を避ける。
12. 購入後フェーズの返信は、必ず次回時刻目安を1つ入れる。
13. 資料受領後でも、`本日[時刻]まで` と `48時間以内` の2段構成は初回受領や本格着手のタイミングだけに使う。追加情報への短い返信、再修正、完了間近では1段の時刻コミットで足りる。
14. 完了確認フェーズや、すでに修正・納品が一度走っている場面では、2段構成や `一次結果` を機械適用しない。
15. 書き出しと末尾2行は、同一表現の連投を避けてローテーションする。
16. 資料受領後は、受領連絡 -> 中間報告 -> 具体時刻コミットの3段階を守る。
17. `5,000円` の説明は、1通の中で1回だけにする。
18. できないことの先回り説明より、返せるものや確認できる範囲を先に書く。
19. 同一バッチ内で `承りました。` `状況は把握しました。` `確認でき次第、2時間以内にはまず見立てをお返しします。` などの定型を3回以上繰り返さない。
20. 情報が薄い初動では、48時間行を機械的に入れず、まず見立て返却だけ約束する選択肢を残す。
21. 範囲外案内は、`今のサービスでは難しい` の拒否を1回だけ書き、その後は条件付き提案で終える。`断り -> 提案 -> 再度断り` の構造にしない。
22. 金額を出すなら断定し、迷うなら金額を出さない。`価格提示 + ヘッジ` の組み合わせにしない。

### Ban
- `ご共有ください`（`教えてください` / `共有ください`に置換）
- `不足分は` / `不足分だけ` で始まる指摘調の文
- `させていただきます` の多用（1通2回以上）
- `対応範囲外です。` の単独使用
- `ご案内します` の単独終止
- `必要情報がそろい次第` の硬直使用
- 過剰な名詞化
- 断定的な煽り（例: `必ず直ります`）
- 外部連絡/外部共有/外部決済を想起させる語
- 本番での再実行を促す語（再決済/再課金/Webhook再送など）
- `対応していません。` の単独終止
- 出品者側からの `補償` / `損害補償` の先出し
- 購入者向け本文での `確認案件`
- `もっともです`
- `見た目は軽く見えても`
- `認識のずれ`
- `ように見えた`
- `作業として発生します`
- 外向け本文での `別件として整理する` / `スコープを切って`
- 証跡未確認の見積り初回での `状況は把握しました`

## 速い修正への不信対策
- 意図的に遅らせない。
- 代わりに、次を見える化する。
1. 原因（1行）
2. 修正点（1行）
3. 確認手順（1〜3）
- 中間報告は「現状 / 次アクション / 次回報告時刻」の3点セットで送る。

## Stripe導線案内
- Stripe関連の確認依頼は `/home/hr-hm/Project/work/stripe日本語UI案内` を最優先参照する。
- 依頼時は同ファイルの「必須項目」を優先し、1通あたり2〜3点までに絞る。
- 同ファイルの「共有不要（`sk_...` / `whsec_...`）」に反する依頼をしない。
- 導線名に差異がある場合は、推測で断定せず「要実画面確認」を添える。

## 出力ルール
- 行頭に不要な空白を入れない。
- 基本は `です・ます`。
- 結論 -> 理由 -> 次アクション の順。
- 箇条書きは3点以内を優先。
- 見積り初回は相手負担を下げる（必要最小限の質問）。
- 判定前の返信では、追加料金の具体金額（5,000円 / 15,000円）を先出ししない。
- 判定後の返信では、必要に応じて追加料金の具体金額を明記してよい。
- 値引き要求への返信では、価格を動かさず代替案の有無だけを返す。
- 見積り経路の追加発生では、`おひねり` を安易に先出しせず、まず進め方を2択で整理する。
- `emotional_caution: true` の場合、先頭は不便の認知に寄せ、最初の依頼は1点だけにする。
- 非エンジニア初回は「最大3点」にする。
- 非エンジニア初回で `evt_...` `pi_...` などID共有を必須化しない（必要時は2通目で依頼）。
- 非エンジニア向けでは `ログ` 単語を単独要求しない。
- エラー文やログは、コピーできるならテキスト貼り付けを優先する。
- コピーしにくい場合だけ、画面スクショを代替として案内する。
- UI状態や見え方の確認では、画面スクショを優先してよい。
- 時刻を聞くときは粒度を下げる（例: `昨日の午後` / `今朝10時頃`）。
- `あればお願いします` を単独で使わない（任意なら `あると助かります（なくてもOK）` と明記）。
- 時刻未確定の案内は、見積りフェーズでは「見立て」、購入後フェーズでは「確認結果 / 判定結果」を使う。
- `2時間以内` の案内は、受領確認または一次報告を指す。一次結果は別時刻で書く。
- `48時間以内を目安に一次結果` は、購入直後の初回受領や本格着手の案内に限って使う。追加情報への返信、再修正、スコープ縮小後には貼り直さない。
- `形になります` `流れになります` で手順説明をぼかさない。相手に行動を求める場面では `こちらからお声がけします` `その時に入れてください` のように主語を出す。
- `確認の入り口だけお返しします` は、情報不足かつ金額未提示の初動だけで使う。15,000円提示済みや緊急度が高いケースでは `見立てをお返しします` を優先する。
- 見積り初回の時刻案内は、次の候補から文脈に合うものを選び、同一バッチで固定連打しない。
  - `確認でき次第、2時間以内にはまず見立てをお返しします。`
  - `まずは2時間以内を目安に、確認の入り口だけお返しします。`
  - `まずは2時間以内を目標に、見立てを一度お返しします。`
- 資料受領後の案内は `この返信から` を使わず、`本日[時刻]までに` で時刻を明示する。
- 完了確認フェーズ（お礼返信・A/B確認）は、48時間行を省略してよい。
- purchased中の追加報告への返信では、相手のカジュアルさより急にフォーマルへ戻りすぎない。
- purchased中の追加報告への返信では、相手にもう一度一から情報を出させる感じを避け、必要な不足分だけを頼む。
- セキュリティ注意喚起は「禁止 -> 代替 -> 事後対処」の3点セットで書く。
- 追加質問の末尾にフォールバック文を入れる。
- `ログ` を依頼するときは場所を明示する。
- 責任/保証の質問には「保証範囲 -> ただし -> リスク低減手順」の3段構成で答える。
- 誤った画面の共有を受けたときは、否定語を避けて「次はこの画面を1枚お願いします」で誘導する。
- non_engineer 向けの見立てでは、`ルーティング` `呼び出し条件` などの技術語をそのまま置かず、`遷移先` `呼び出す流れ` などへ言い換える。
- non_engineer 向けでは、`判断がぶれやすい` `保持情報` のような内部語をそのまま出さない。`何から進めるかを整理しにくい` `サイト側で持っている情報` などへ言い換える。
- non_engineer 向けでは、`サブスクリプション更新` `Webhookハンドラ` など、平たく言い換えられる語は `課金への切り替え` `その処理のコード` へ落とす。
- Stripe不使用でも Next.js / Vercel / API連携の切り分けとして受けられる場合は、その場で一言明示する。`Stripeは使っていないとのことですが、Next.js / Vercel の本番切り分けとして対応可能です。`
- 範囲外案内の末尾で `対象外としてのご案内になります` のような官僚調の締めを足さない。
