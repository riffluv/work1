## 日本語品質
- サービスページ / プロフィール / Q&A と同じ人物が書いたように見える文体を保つ。
- まず相手の言葉を拾ってから返す。読まずに質問票を出した印象を避ける。
- 相手の冒頭に、`引き継いだコード` `前任と連絡が取れない` `ドキュメントがない` のような実務上の前提がある時は、感情語を足さずに事実として1点だけ拾う。技術症状だけへ還元して最初の1行を丸ごと落とさない。
- 冒頭の受け止めで、相手の症状を全文要約のようになぞりすぎない。診断や進め方の起点になる1点に絞って拾う方を優先する。
- 非技術者向けでは「まず状況を整理します」の空気を出し、最初から査定のように見せない。
- 追加質問の前に、見立てや確認観点を1文入れて「整理している感」を出す。
- 見積り初回は、質問の前に見立て・確認観点を1文入れる。これがないと質問票に見えやすい。
- 見積り初回では、未受領のコードやログを前提に `確認できます` `見ます` `切り分けます` を先頭へ置かない。`対象になりそうです` `この2点が分かれば見積もれます` のように、見積もり判断の語彙を優先する。
- 見積り初回の判定文で `この内容は` `この内容なら` を固定句にしない。相談文を手元の資料のように扱う響きが出やすいので、`いただいた状況だけ見ると` `ご記載の症状だけだと` `今の情報だと` `書いていただいた状況なら` などへ散らす。
- 判定前の返信では、追加料金の具体額を先に並べない。まず同一原因かどうかを見る。
- 見積り初回で `15,000円で対応できるか` が主質問なら、価格返答は `基本料金は15,000円です` / `この時点では15,000円で進めるとはまだ言えないので、先に1〜2点だけ確認します` / `15,000円ではなく別の入口です` の3択に寄せる。価格可否を `見込みです` `可能性が高いです` で曖昧にしない。
- 価格を聞かれていない境界判定では、`15,000円の範囲で進められるか` を先に出しすぎない。`このサービスで対応できるか` `対象になるか` を優先する。
- ただし、主質問が `もっと安くなりますか` `設定だけなら安くなりますか` のような値引き可否なら、`進められます` ではなく `このケースでも15,000円です` `安くはしていません` のように価格そのものへ直接答える。
- 複数不具合の `料金はどうなりますか` には、`変わります` だけで返さない。`1件15,000円が基本で、同じ原因ならまとめて対応できる可能性がある` のように、基準になる価格を先に置いてから切り分け条件を続ける。
- 見積り初回で `2件分になりますか` `購入前に知りたいです` が主質問なら、先に `この時点で2件分と決まっているわけではない` `同じ原因の範囲なら1件` `別原因でも自動加算しない` を返す。`ご購入後は〜を確認します` から先に入らない。
- buyer 向けに `同じ原因の範囲` が重く見える場面では、`つながっている内容` `原因が見つかった場合` `今回の不具合とつながる場合` に翻訳する。
- prequote で進め方を添える時は、`ご購入後は` の連発で購入前の相手へ次工程感を強めすぎない。`進める場合は` `ご依頼後は` `確認する場合は` などに散らす。
- 回答の入り方・閉じ方を毎回同じにしない。`ご相談ありがとうございます。状況は把握しました。` の固定連打を避ける。
- 公開 bugfix は定額サービスとして扱い、価格を出したあとに `お見積りをお返しします` や `見積もり提案をお送りします` を基本形にしない。`基本料金は15,000円です` か、価格提示済みなら次アクションだけで閉じる。
- 公開 bugfix の定額案内では、`この内容なら15,000円で進められます` を常用しない。内容判定で価格が揺れるように読めるため、価格そのものを返す場面では `基本料金は15,000円です` を優先する。
- 相手がすでに `15,000円は大丈夫` など価格了承を明示している場合は、価格を重ねて2回書かない。次アクションを前に出す。
- 定額 service で、相手が価格や予算を聞いている場面では、`お見積り` を使わなくても価格未提示のまま閉じない。件数や入口を答えるなら、その段階の基本料金を先に置く。
- 見積り初回では、コードや証跡をまだ見ていない段階で `状況は把握しました` を安易に使わない。`承りました` `内容を確認しました` を基本にし、論点がかなり明確な場合だけ使う。
- 見積り初回の書き出しと末尾は、連続3件で同型にしない。
- 相手が1テーマだけを書いている見積り初回で、`〜でしたら、この内容は` のような条件分岐の受け方をしない。`〜の件、承知しました。` `〜の件、不具合1件として対応できます。` のように素直に受ける。
- `〜件でしたら、不具合1件として対応できます。` は外向け本文で使わない。`〜件、不具合1件として対応できます。` を基本形にする。
- `ご安心ください` のような抽象的な安心語より、`勝手に料金が増えることはありません` のように具体的に書く。
- 不安や焦りへの返しで、相手が使っていない感情語や評価語を持ち込まない。`責める` `怒られる` などを足さず、`その点も含めて確認できます` `今の状態から見ます` のように、事実と次アクションで安心を返す。
- `もっともです` の変形 (`もっともだと思います` `もっともな不安です`) で、相手の感情を評価・承認しない。感情のラベル付けより、`先に料金の話から確認します` `まずこちらで確認します` のように、責めない一言と次アクションで返す。
- 感情には原則深く触れない。受領だけで自然に進められるなら、感情の説明や共感の言語化を足さない。
- ただし、影響範囲が不明・セキュリティ不安・苦情寄りなど、受領だけでは冷たく見える場面では、不便の認知か状況の受け止めを最小限で入れてよい。
- 不安や怒りが見える相手でも、毎回そこを直接言語化しない。`メッセージありがとうございます。` `ご相談ありがとうございます。` のような短い受領だけで自然に進められるなら、その方を優先する。
- ただし、相手が遠慮・不安・自信のなさを1行差し出している場面では、冒頭の受領の次に 1 拍だけ受け止めを入れる。`気にしないでください` `その点は安心してください` `よくある状況です` のように短く受け、そこから本題へ入る。事務連絡だけの場面では足さない。
- 受け止めの1拍に `そこは` を足して範囲を限定しない。`そこはすみません` `そこはよかったです` より、`すみません` `よかったです` をそのまま置く。
- 受け止めの対象は不安だけに絞らない。`原因が分からない` の混乱、`前にも依頼した` の関係性、`正直よく分からない` の率直さなど、相手が差し出した状況シグナルがあれば最初の2文で1点だけ拾う。
- 予算制約・本番障害・前任者と連絡不能のようにストレスが高い相談では、`確認しました` より先に `すぐ見ます` `15,000円の範囲で大丈夫です` `コードがあればこちらで見られます` のような行動や安心を置く。
- 相手が `自分では判断つかない` `どちらがよいか分からない` と差し出している時は、`もし進めるなら` `必要なら` `判断材料になると思います` のように判断を返さない。こちらで先にざっくり方向を出し、相手には最小限の情報だけを頼む。
- ただし、`もっと検証してほしかった` `説明が足りなかった` のような品質フィードバックは感情扱いで流さず、まず不足していた点を短く認めてから本題へ進む。
- `税込みですか` `追加でかかりますか` のような料金質問では、制度説明より先に `今回の支払い額` と `後から別で請求しないこと` を短く返す。`ココナラ側の表示に準じます` だけで終えない。
- 料金質問では、`ありきではなく` `前提ではなく` のようなメタ表現でぼかさない。`追加料金はかかりません` `今回は15,000円です` のように、結論をそのまま返す。
- 追加費用への不安に答える時は、`自動で追加費用が発生しない` より `ご相談なく発生しない` を優先する。相手の不安は自動課金ではなく、勝手に請求が進むことだから。
- `15,000円が限界` のように予算上限が明示されている相手へ、別原因リスクが残るまま `15,000円で進められます` を押し切らない。必要なら `15,000円では原因の確認と切り分けまでになる可能性がある` を先に明示して、買うかどうかを相手に選んでもらう。
- 値引き交渉では、値引き不可ならそのまま価格を固定で返す。`確認だけの進め方` のような未設計の逃がし道をその場で足して、相手に別の料金体系を想像させない。
- まだ不具合が起きていない `確認だけ` 相談には、そのまま `可能です` と受けない。確認専用サービスがあるように読ませず、`気になっている箇所が1つあれば、不具合修正として扱えるか購入前に見立てる` へ寄せる。
- 範囲外の追加相談には、着地先の公開サービスがない段階で `できますが` と能力コミットしない。先に `別件です` と区切り、必要なら `今回の件が完了したタイミングで内容を教えてください` と相談受付だけを残す。
- 例外的にこちらで吸収して `15,000円` で受ける時も、外向け本文で `特別に` `今回はサービスで` `実績のため` のような売り手事情は出さない。書くなら `今回はこの内容であれば、15,000円の範囲で対応します` のように、時間と内容の両方を区切る。
- 上の例外文面は常設の標準文として使わない。ユーザーから `今回は特別対応` と明示がある時だけ使う内部運用に留める。
- `原因不明のまま15,000円だけかかるのでは` の不安には、`調査は含む` と答えつつ、こちらの判断だけで原因未特定・未修正のまま正式納品へ進める読みを出さない。`その時点で状況をご説明して、追加対応に進むかどうかを先に相談する` を基本にする。
- 追加料金の可能性に触れるときは、`追加料金がかかる場合があります` だけで止めず、`別原因ならそこから先には進めず、まず状況を説明して、追加で進めるか止めるかを先に相談する` まで書く。追加料金そのものへの包括同意を取りにいかず、進め方への同意を取る。
- 上のフル展開は、相手が `15,000円だけ払って終わるのが不安` などコスト不安を明示したケースで使う。高確度で不安表明がない見積り初回では、`別の原因が見つかった場合は、進める前にご相談します` の短い形で足りる。
- `基本料金分で完了にする` は、原因未特定・未修正の不安に対する初回回答で機械的に使わない。相手に `直っていないのに承諾を求められる` 読みを与えないことを優先する。
- `返金` の話題では、`返金をお約束できません` のように出品者が判断者に見える言い方をしない。`ココナラの手続きに沿う話になるので、この場でこちらが決めることではありません` を基本にする。
- 過剰敬語を避ける（`させていただきます` 連発禁止）。
- 1通1依頼を原則にする。
- 1文1意味で書く。長文の名詞列は分割する。
- `A の B として` が出たら、`A で B を` `A で B まで` のように分解できないか確認する。`の` が連続する時は、1つを `で` `を` `に` へ置き換えられないか先に見る。
- `大丈夫です` の連発を避ける（1通あたり最大2回目安）。
- `教えていただけますか` より `教えてください` を基本にする。
- 質問リストの後を `いただければ` だけで結ばない。`教えていただければ` `お答えいただければ` のように、何をもらうのかまで書く。
- `返しやすいです` `確認したいです` のような提供者視点の語を多用しない。相手へ返す内容は `お返しします`、依頼は `教えてください` `送ってください` のように、相手が次に何を出せばよいかまで書く。
- 保留確認で質問を出す前に、相手文の中で既に答えが出ている情報を落とす。2点テンプレを優先せず、必要なら1点だけ聞く。
- 相手がすでに書いた情報（例: 本番 / Vercel / Next.js / いつから）を質問項目へ再パッケージしない。
- 相手が使った語を、必要なく一段硬い語へ変換しない。`押す -> 押下する` `3回に1回 -> 断続的に` のような硬化は、専門性より AI 感が先に出やすい。
- 相手が使った動詞や症状語は、意味が変わらない範囲でそのまま返す。`登録できない` を `反映されない` へ寄せるなど、こちらの都合で語彙をずらしすぎない。
- 技術語でも、相手が使った呼び方を優先する。相手が `カスタマーポータル` と書いているのに `Billing Portal` へ置き換えるなど、公式名への言い換えを機械適用しない。
- `崩れる` を汎用の症状語として使わない。相手が `エラーが出る` `失敗する` `止まる` `飛ばない` と書いているなら、その語を優先する。
- 相手がすでに事実を報告している場面で、`〜なら` `〜であれば` のような仮定受けへ戻さない。`〜の状態だと` `〜の件、` のように直接受ける。
- Yes/No への返答は、結論の文を `はい、〜です。` `現状では難しいです。` で一度閉じる。同じ文の後半に `〜なら` `〜だけなら` を続けて条件付き Yes に戻さない。条件や範囲は次の文へ分ける。
- handoff 境界の説明では、`進め方です` `段階に見えます` のような抽象語だけで閉じない。相手が `何がもらえるか` を聞いているなら、受け取れるものを具体物で答える。
- 見積り初回の末尾には、原則として時刻目安を入れる。
- `〜は含めませんが` の先回りした防御表現を避け、まず `できること` を肯定的に書く。
- 見積り初回で、相手がまだ言っていない広い範囲外（例: `新規実装全体の伴走ではなく`）を先回りで否定しない。境界が必要でも、まず今受けられる範囲を肯定してから補う。
- 高確度・単一質問の見積り相談では、相手がもう済ませたサービス選択や進め方を後から承認する段落を足さない。聞かれた結論と金額感だけを短く返す。
- 技術的に詳しい相手が `timeout` `冪等処理` `payload` など具体語を出しているときは、その語を1つだけ拾って返す。テンプレを崩さず、`timeout も含めて見ます` `冪等処理も含めて確認します` 程度で十分。
- 技術的に詳しい相手への返信で、相手が出した仮説や具体語を1つも拾わずにテンプレだけで返さない。最低1箇所は `payload 構造の差異` `timeout` `キュー` `冪等処理` のような相手語をそのまま入れる。
- 非技術者でも、`再ログインすると直ることがある` `特定の時間だけ起きる` のような観察情報があれば、1フレーズだけ拾って返す。診断の手がかりとして読んでいることが伝われば十分。
- `急ぎ` `至急` が明記されている時は、作業説明に埋め込むだけで済ませず、`至急とのこと、承知しました。` `急ぎとのこと、優先して見ます。` などの短い1文で先に受ける。
- prequote では、購入後の進行中会話のように `まず確認します。` 単独で始めない。route に合う受領を1行置いてから、次の文でアクションを返す。
- prequote では、購入前に着手する読みを避けたい場面で `確認します` を前に出しすぎない。`見立てます` `対応できるかをお返しします` のように、見積もり判断の語彙を優先する。
- `問題ありません` は、相手が許可を求めている内容にだけ使う。相手がすでに選んだ進め方やサービス適合を後から承認する語としては使わない。
- できないことを断った後に、`進められそうかをご相談ください` のように判断を相手へ返さない。テキストのみ・メッセージのみなど前提を明示し、それで難しいなら合わない可能性があるとこちらから言い切る。
- `service_mismatch` や範囲外の案内で、いきなり `対象外です` とラベルで閉じない。`対応が難しいです` `このサービスの範囲外になります` など、理由つきで柔らかく返す。
- 完全に対象外で受けられない時は、必要に応じて `その領域に強い方を探されるのがよさそうです` のような1文を添えてよい。外部リンクや具体的な外部誘導には広げない。
- 相手が `決済とは関係ない` `Stripeは直接関係ない` と明言している時は、`Next.js の不具合として確認できます` のようにスコープを広く見せない。まず現行サービスの対象外になる可能性を短く伝え、そのうえで API / 状態更新 / 認証など接点があれば確認できる、の順で返す。
- 返金・設定変更・デプロイのように相手側が行う操作へ触れる時は、主語を明示する。`Stripe管理画面からお客さま側で対応できます` のように、誰の操作か分かる形にする。
- 質問数を常に3点に固定しない。2点で足りるケースは2点に絞る。
- 情報が薄い初回相談では、こちらの内部候補になっている分岐名を先に見せすぎない。`確認だけの方がよいか` のような曖昧な選択肢は出さず、まずは `内容を見て進め方を返す` に寄せる。
- 値引き相談では、未公開の旧確認プランを代替案として出さない。まず 15,000円 の固定条件と、判定前に何を確認するかを返す。
- `判断しやすいです` `見やすいです` のようなこちらの内部事情で結ばない。`対応できるかどうかをお返しします` `次の進め方をお返しします` のように、相手に返す結論で閉じる。
- `入口` のような内部寄りの比喩は外向け返信で多用しない。`進め方` `対応方針` `料金感` など、そのまま読める語を優先する。
- purchased中の追加要求では、`切り分ける` `判定する` など内部プロセス語を前に出さない。`別で確認` `追加でご案内` に言い換える。
- purchased中の継続会話では、相手の追加報告に一度反応してから依頼を書く。いきなり条件分岐や依頼リストに入らない。
- purchased中の継続会話では、初回受付のような `次の2点だけお願いします` を機械適用しない。2点程度なら文中に混ぜる方を優先する。
- purchased中の継続会話では、`いただければ、本日[時刻]までに共有します` のようなSLA文を機械適用しない。`もらえると助かります` と `本日[時刻]までに共有しますね` を分ける方が自然。
- purchased中の継続会話では、`同一原因` `別原因` より `同じ原因` `別の原因かもしれない` `別の話なら` を優先する。
- purchased中の継続会話では、相手との距離感を初回受付へ戻さない。`ありがとうございます。別のページでもですか。` のような短い反応を先頭に置く。
- purchased中に複数質問が来たときは、今すぐ答えられる事実だけ先に返し、コードや証跡を見ないと答えの価値が低いものは `確認してから返します` に寄せる。見積り相談のような番号付きQ&Aを機械適用しない。
- purchased中の軽い追加報告では、`差し支えなければ` `可能であれば` のような重い前置きを避ける。相手がカジュアルなら `スクショを送ってください` 程度で十分。
- delivered / closed で `直らない` `変わらない` `まだエラー` のような問題報告が来た時は、受け止めを `承知しました` で止めず、`確認します` `まず反映状態を見ます` のように次の確認動作を先に置く。
- purchased中の追加相談で、スコープ判断の条件分岐を3文以上かけて説明しない。`同じ原因なら続けて見る / 別ならその時点で相談` の結論だけを短く返す。
- purchased中に `ご相談します` を使うときは、何を相談するのかまで明示する。`別料金での追加対応になるか` `別件として切り分けるか` など中身を言わずに濁さない。
- purchased / closed の末尾で `確認して、進め方をお返しします` を固定文として使いすぎない。`見て、どう進めるか先にお伝えします` `確認して返します` `見たうえで次をお返しします` などに散らす。
- purchased / closed の継続可否では、`見られるか` を使わない。今回の範囲で続けられるか、前回の続きとして対応できるか、別の話か、の会話語へ置き換える。
- purchased / closed で `同じ原因かどうか` をそのまま出しすぎない。`今回の件とつながっているか` `前回と同じ話か` を優先する。
- purchased / closed の文末は `確認して、〜をお返しします` に固定しすぎない。`続けられるかをお返しします` `別件か確認してお返しします` `このトークルームで進め方をご相談します` など、会話の段階に合わせて散らす。
- `自動で続き扱い` は会話で使わない。`前回の続きになるとは限らないので、まず確認します` のように言い換える。
- closed 後の追加対応案内では、`別途ご案内します` だけで止めない。新規依頼として受けるのか、料金を先に伝えるのか、範囲を見て判断するのかまで書く。
- closed 後の返信では、相手が明示していない値引き期待を先回りで否定しない。まず新規依頼になるか、前回範囲の補足で足りるかの事実を先に返す。
- closed 後に実作業や書き直しを受ける返信では、`補足なら無料でそのまま続けられる` 含みを残さない。対応するなら改めて依頼が必要と先に書く。
- closed 後で `追加料金かかりますか` と聞かれた時は、`追加料金の有無` のようにぼかさない。必要なら `料金はかかります` を短く返してから、補足か書き直しかで料金や進め方が変わると続ける。
- closed 後の問い合わせは、`納品物の説明・補足` `再発・新規の不具合` `納品物の再送` の3分類で考える。どれにも当てはまらない中間状態を作らない。
- closed 後の再発や新規不具合は、入口を `新規のご相談として` で統一する。売り手判断で後から吸収する余地は残しても、最初の返信で無料診断の含みを出さない。
- closed 後の再発で `またお金がかかるのか` と聞かれた時も、先に `同じ原因なら基本料金内` と言い切らない。まず `前回の修正とつながる再発か / 新しい原因か` を見て、続き扱いか新しいご依頼かを返す。
- closed 後の納品物の再送は、無料で即対応する。`進め方をお返しします` `どれが必要か教えてください` のような不要なプロセスを挟まない。
- closed 後の軽い質問で、無料と有料の境界が曖昧になる `少しだけなら` は使わない。概要を1行で答えるか、改めての相談へ線を引くかのどちらかにする。
- `本番環境はない` `作り方が分からない` `Webhookの受け方が分からない` のような相談は、いま壊れている不具合の返信に寄せすぎない。まず bugfix か境界案件かを返す。
- `設定だけなら自分でやる` `コード側なら見てほしい` のような条件分岐がある時は、設定問題かコード問題かを切る前に価格を断定しない。
- `今は問題ない` `影響が出るかもしれない` `知り合いにそう言われた` のような予防相談は、前回範囲や bugfix へ自動吸収しない。現時点の不具合と予防相談を分けて返す。
- 分割払い、支払い方法変更、外部決済の相談では、こちらが個別に決める書き方をしない。ココナラ側の決済仕様として短く返す。
- 1通に話題が複数ある時は、全部を同じ強さで答えない。緊急停止が必要な話題、進行中の本件、追加された別件の順で返答量を配分する。
- 相見積りや他者比較では、他の出品者の金額評価に乗らない。こちらの提供単位と納品物の違いだけを返す。
- `整理+修正を30,000円でまとめて` のような独自バンドル提案は、その価格に合わせて単位を崩さない。どちらから入るかを返す。
- 電話や Zoom を断る時は、`できません` だけで終わらせず、テキストで返せるものを続ける。たとえば `引き継ぎメモで流れと注意点を残します` `確認事項はこのトークルームで順番に整理します` のように代替を先に出す。
- 本番反映 / 直接操作 / 直接 push のような hard no のあとに、`手順はお伝えできます` `整理して返します` だけで閉じない。相手がどの段階で止まっているか不明なら、`今どの段階で止まっているかだけ教えてください。その箇所に合わせて必要な手順をお返しします。` のように、詰まり箇所を1点だけこちらから聞く。
- 秘密情報がすでに送られた時は、`送らないでください` だけで終わらせない。編集や削除ができるならその依頼、難しければキーや接続情報の変更、その後の安全な共有方法まで続ける。
- purchased 中に、原因が Stripe や今回の想定から外部API・別サービス側へずれた時は、その場で料金を断定しない。同じ流れなら続けて確認し、別原因ならその時点で別料金での追加対応になるか先に相談すると返す。
- `同じ商品が2個届いた` `決済が2回入った` のような重複処理相談では、返金や原因説明を先に広げない。まず同じ操作を続けない方がよいかを短く返し、その後で確認事項へ進む。
- Stripe以外の決済名が明示された時は、`対応できます` `対象外です` を即断しすぎない。決済サービス側そのものか、Next.js / Webhook以降の後続処理かを先に分けて返す。
- Stripe以外の決済名が出た時は、決済サービス自体の設定や決済画面の調整まで受ける読みを出さない。`Webhook後のサイト側処理` `DB更新` `API側の反映` のように、こちらが見る範囲を先に限定して返す。
- Stripe以外でも、Next.js / API Route / Webhook受信処理のようにこちらが確認できる範囲が明確なら、`確認できる可能性があります` の二重ヘッジにしない。未知なのは `決済側かサイト側か` の切り分けに残す。
- `見られる` は、対応可否・継続可否・件数判定を含む一切の判断文脈で使わない。スコープ内なら `対応できる`、まだ切り分け前なら `確認できる`、継続可否なら `続けられる` / `前回の続きとして対応できる` を優先する。
- `15,000円の方か25,000円の方か` と聞かれても、未公開側のサービス名・価格・購入導線をそのまま外向け返信へ出さない。判定軸だけを返し、15,000円で進められる内容か、別の進め方が必要かを先に返す。
- handoff 系で、相手文に `ブラックボックス` `怖くて触れない` `動かない` のようなバグ・障害文脈がある時は、必要に応じて `このサービスでは修正自体は含まず、まず構造と危険箇所を整理します` を1行入れる。bugfix との違いを曖昧にしない。
- handoff 系が private の間も、価格やサービス名を伏せたまま商業的クロージングは残す。ただし fixed-price 前提なので、`お見積り` を基本形にしない。`内容を確認したうえで、今回整理する範囲と進め方をお返しします` を基本形にし、無料で整理を始める読みを出さない。
- handoff 系では、相手が言っていない `仕様書` `仕様書化` を先回りで否定しない。`メモ` `整理` `引き継ぎ` など相手が使った語でそのまま返す。
- handoff 系で、相手が `何を見てほしいか分からない` `1フローが分からない` と書いている時は、決済/会員登録/メール送信の選択肢を返して選ばせない。コード一式や既存メモを受けて、こちらで最初に見る流れを決める方向へ寄せる。
- 価格への反応では、`安くはしていません` のような対抗的表現を使わない。`この料金で設定しています` を基本にする。
- boundary で症状が明確な bugfix ケースは、`先に整理してから` の段階提案へ寄せすぎず、`不具合1件として対応できます` `15,000円で進められます` を基本にする。構造把握や原因確認は bugfix の調査に含める。
- bugfix 文脈で `見られます` `見られる内容です` を使わない。受ける意思は `対応できます` で返す。
- hold や料金文脈でも、サービス受け入れ判断を `見られます` `見られそうか` でぼかさない。`対象になりそうです` `対応できるか` `1件として見積もれそうか` に寄せる。
- 極短文の hold は、相手の Yes/No 質問に `今の情報だと対象候補です` などを1文返してから質問へ入る。
- hold 文脈で `確認したいです` を使わない。提供者目線より、`2点だけ教えてください` の依頼形を優先する。
- 料金不安への回答では、相手が使っていない感情語を足さない。`不安` `心配` を補足で上乗せせず、料金・範囲・正式納品条件だけで閉じる。
- 料金不安への回答では、説明が答え切れているなら余計な締め文を足さない。`〜でしたら` が入りやすい補足1文を省く。
- routing / 境界判定で事実を受けるときは、`だと` `とのこと` を優先し、`なら` `であれば` の仮定受けを避ける。
- boundary で情報を求める時は、`確認させてください` より `教えてください` を優先する。依頼文脈で提供者都合の語を残さない。
- boundary でも、受け入れ判断に `見られる` を戻さない。`対応できる内容か` `対応できる可能性がある` に寄せる。
- 特に境界ケースでは、`見られる場合もある` を使わない。揺れる場面でも `対応できる場合もある` に固定する。
- boundary の条件付き受諾で、`bugfix として見られる` `範囲で見られる` のような形にも戻さない。`bugfix として対応できる` `この範囲で対応できる` `対象になりそうです` に置き換える。
- `問題なければ` だけで閉じない。何についての同意かを明示して、`この進め方でよければ` `この内容でよければ` に寄せる。
- `.env` やキー共有を止める時は、止めるだけで終えず、`キー名だけ` `エラー文` `スクショ` など次に送ってよい代替物を必ず書く。
- 情報要求の文脈で `確認したいです` `切り分けたいです` を使わない。`確認します` `切り分けます` か、相手への依頼なら `1点だけ教えてください` に寄せる。
- 返信末尾がこちらの処理宣言だけで止まらないようにする。相手が次に送るものか、こちらが次に返すものを1つは明示する。
- handoff の価格相談では、相手が使っていない感情語を補わない。`頼みにくい` `金額を抑えたい` より、`この整理で分かること` `負担を抑えるなら` のように、判断材料と次アクションで返す。
- handoff の設計説明で `役割が違う` を使わない。`別の作業になる` `進め方が分かれる` のような会話語に置き換える。
- handoff の末尾は、同一バッチで `内容を確認したうえで、今回整理する範囲と進め方をお返しします` に固定しすぎない。`最初に見る流れをお返しします` `この場で受け取れる形をお返しします` など、質問の種類に合わせて散らす。
- 外部共有を止める時は、Drive / Notion を拒否して終わらず、この場で受け取れる代替手段まで書く。ファイル名一覧、構成、画面スクショ、コード一式の受け取り可否確認など、次に出せるものを明示する。
- `していません` は交渉や制度説明の場面では先頭でもよいが、不安の受け止めが必要な handoff では、受け止め1行の後に置く。
- handoff の提案文で進め方を説明した直後に `進めるなら` を機械的に置かない。相談段階では `この進め方でよければ` `この形でよければ` のように、相手の同意確認を先に置く。
- purchased / closed で情報を求める時は、`確認したいので` を使わない。`まずつながっているかを見たいので` `状況を整理したいので` より、`○○を送ってください` を優先する。
- 情報要求や進行案内で `進めたいので` `確認したいので` のような `〜したいので` を使わない。`進めるので` `確認するので` か、依頼なら `教えてください` に落とす。
- 情報要求や方針説明の文脈で `進めたいです` `確認したいです` `切り分けたいです` のような `〜したい` を使わない。方針は `進めます` `確認します` `切り分けます`、依頼は `教えてください` `送ってください` に落とす。
- purchased 中に別件が出たら、緊急でも `今回の件とは別の対応になる` の一言を落とさない。費用を即断しなくても、別件であることは先に伝える。
- `今の書き方だと` `今の文面だと` `今の文面だけだと` のように相手の文章自体を評価する入り方をしない。`状況を見ると` `お話を聞くと` `今の段階だと` `いただいた内容だと` で受ける。
- 相手が `修正は不要` `まず構造を把握したい` のように目的や不要条件を明示している時は、それを質問で聞き直したり、逆方向の提案を足したりしない。相手が置いた前提をそのまま受けて進める。
- `決済 / 認証 / メール送信` の固定3択は機械適用しない。文脈に合う時だけ使い、通常は `いちばん整理したい流れ` `いちばん把握しておきたい処理` のようにオープンに聞く。
- 料金を聞かれたら、基準になる数字を返す。`1件に収まるかどうかで決まります` のような判定軸だけで止めず、必要なら `1件あたり15,000円が目安です` まで書く。
- `追加でいくらかかりますか` と金額を明示して聞かれた時は、まず `追加1件15,000円です` のように金額を答えてから、`自動で発生しない / 先に相談する` を続ける。

- 外向け本文で内部ラベルの `bugfix` `handoff` をそのまま使わない。`不具合修正` `主要1フローの整理` `このサービス` に言い換える。
- 相手が `高い` `予算が厳しい` と率直に書いている時は、scope や価値説明の前に短い受け止めを1文だけ置いてよい。`率直にありがとうございます。` `そこは分かります。` までに留め、弁明には広げない。
- 相手が `安心しました` `助かりました` `自分では触れない` のように、好意・安心・弱さを1行差し出している時は、本題の前に1文だけ受けてよい。`見つけてもらえてよかったです` `お役に立てたなら嬉しいです` `正直な判断だと思います` など、短く返してから本題へ入る。
- 上の受け止めは1文で止める。評価や励ましに広げず、そのまま結論か次アクションへつなぐ。
- handoff の購入前説明で納品形式を聞かれたら、ファイル名は伏せたまま `テキスト形式（.md）` までは答えてよい。`文書` だけで曖昧に返さない。
- boundary やお断り寄りの返信では、`受けていません` `難しいです` だけで閉じない。再判定に必要な1点か、再相談できる条件を1文だけ残す。
- handoff の価値質問では、相手が出した価格をこちらから繰り返さない。価格の妥当性に答える時も `その金額に見合うか確認してから決めたい` のように受け、`25,000円の価値があるか` のような反復を避ける。
- handoff / boundary の価格不安で、相手が出した `25,000円の方` `25,000円は` をこちらから反復しない。価格帯の説明が必要でも、相手の数字をそのまま繰り返さず、判定軸や進め方から返す。
- handoff の価値質問では、相手が使っていない感情語を補完しない。`不安` `心配` を足さず、判断材料と進め方で返す。
- 見積り初回の時刻目安の言い回しは、連続3件で同一にしない。
- purchased / delivered / closed の時刻コミットも、`本日XX:XXまでに〜します` の完全定型に寄せすぎない。時刻は残しつつ `XX時ごろに一度送ります` `今日中にいったん共有します` など位置や動詞を散らす。
- 見立て1文は、できるだけ技術的な確認観点や仮説を優先する。緊急度や売上影響への言及だけで終わらせない。
- 見積り初回で、`〜とのことですが、〜のであれば、〜として` のような条件連鎖を1文に詰め込みすぎない。状況の要点と、価格可否・進め方は2文に分ける。
- 金額を出すときは、受けられる前提で書く。`対応できる可能性があります` のようなヘッジを残すなら、同じ文面で価格まで出さない。
- 末尾のクロージングで `内容を見て、〜を先にお返しします` を連続で使いすぎない。`確認して、対応できるかを先にお伝えします` `見たうえで、どこから進めるかをお返しします` など2〜3パターンを回す。
- handoff が private の間は `内容を確認したうえで、進め方とお見積りをお返しします` を使ってよいが、公開 bugfix の定額案内では `お見積り` を混ぜない。
- 見積もりクロージングは `問題なければ` だけに固定せず、`よければ` `この内容で問題なければ` `問題なければ、そのまま` を軽く散らしてよい。
- 相手の質問へ答えた直後に、`〜との点には` のような引用つき二重回答を足さない。前文で結論が出ていればそこで止める。
- 確認依頼のブロックを `1つだけ教えてください。そこから/そこを見て/そこに寄せて、〜をお返しします` に固定しない。`1つだけ` は残してよいが、接続語と末尾動詞は散らす。
- `1つだけ` 自体も固定句にしない。`これだけで大丈夫です` `一言でも助かります` `分かれば教えてください` のように、負担を軽くする言い方を散らしてよい。
- `どうなりますか` `直らなかった場合は` のような流れ確認には、否定の保証より先に `まず何をするか` を置く。`〜しません` `〜ありません` から始めると、こちらが worst case を先に提示した読みになりやすい。
- こちらが判断を返す場面では、末尾を `よさそうか` で閉じない。`どこから入るか` `修正から入るかどうか` のように、判断結果そのものを短く返す。
- `共有` `流れ` `進め方` は、トークルームでは PM 語彙として距離が出やすい。`お伝えします` `お送りします` `形です` `進めます` で言い換えられるならそちらを優先する。
- `公開サービス` `この中で` `入れる内容` のような内部スコープ語は、相手が料金や可否を聞いているだけの場面では出さない。結果だけを `対応できます` `まず見ます` で返す。
- 相手が `整理` を明示していない場面では、`全体の整理ではありません` `整理するより先に` のような否定形でも持ち出さない。bugfix で今できることだけを積極的に伝える。
- ネガティブな感想や `高かった` のようなフィードバックには、受け止めたあとで `そう見えた理由` を説明しない。こちらの課題と次回の改善だけを短く返す。
- 非技術者向けでは `切る` を便利語にしない。`見る` `分ける` `分かる` で言い換えられる場面はそちらを優先する。
- `自体は` `ことがあります` は、Yes/No を答える場面では先回りヘッジとして見えやすい。`できます` `対象外になります` で言い切れるなら先に言い切る。
- スコープ境界の説明でも、Yes の直後に条件・例外・否定を1文で重ねすぎない。`はい` -> `条件説明` -> `最後は肯定の保証` の順で短く分ける。
- `15,000円` の提示文は自然でも、同一バッチで完全一致が続きすぎないようにする。意味を変えず `15,000円で進められます` `15,000円で対応できます` を軽く散らす。
- 見積り初回で `ご購入後に` を使う場合、近い2文で繰り返さない。後続文は `別の原因が見つかった場合は` のように省いて自然につなぐ。
- 見積り初回で、こちらが実際に確認や修正へ入る時点は購入後だと分かるように書く。`まずは15,000円の範囲で確認します` のように購入前着手へ読める現在形は避け、`ご購入後は、まず15,000円の範囲で確認します` のように時点を固定する。
- `切り分けたうえで対応します` を見積り初回の標準結びにしない。まずは `切り分けます` `確認します` `見ます` を優先し、必要な場面だけ後段の `対応します` を足す。
- 相手が `箇条書きで` `短く` `一言ずつ` のように返答形式を指定した時は、その形式を守る。意味だけ拾って prose に戻さない。
- 相手が `早いですね` `助かります` `ありがとうございます` のように前向きな温度を出している時は、主質問へ入る前に短い謝意を1拍返してよい。
- 相手が `星5` `大満足` `バッチリでした` のように強い好意を出している時は、実務案内の前に `お役に立てたなら嬉しいです。` のような短い受けを1文だけ返してよい。謝意や営業を重ねすぎない。
- `もちろんです。` は無難でも固定化しやすい。肯定や受領の冒頭は、`ありがとうございます。` `承知しました。` `可能です。` `その形で大丈夫です。` などに散らす。
- 怒りや催促が強い相手への途中共有では、技術的な進捗を硬い分析語で書きすぎない。`切り分けの軸は2点です` より `だいぶ絞れてきています` のように、平易な現在地を優先する。
- 競合比較や価格差の説明で `差額の説明だけすると` のような事務的な前置きを置かない。何が含まれるかをそのまま説明する。
- 追加料金や追加フローの支払い方法を聞かれた時は、`追加支払いの形をご案内します` のような曖昧な表現を避ける。`取引状態を確認してこちらからご案内します` のように、次の動きを明確に書く。
- 別対応や追加料金の説明では、`この場で勝手に進めず` のような防御的な言い方を避ける。`先にご相談したうえで進めます` `進める前にご相談します` を優先する。
- 初動の受け止め表現は、同一バッチ内で同型連打を避ける。
  - 例: `承りました。` `内容を確認しました。` `ご相談の内容、確認しました。`
- 書き出しの1行目は route に合わせて切り替える。
  - `service` -> `ご相談ありがとうございます。`
  - `profile` -> `ご連絡ありがとうございます。`
  - `message` -> `メッセージありがとうございます。`
- 同一バッチで、書き出し1行目が 4/5 以上同一にならないようにする。
- 見積り初回で別原因や追加対応へ触れるときは、`事前にご相談します` だけにしない。基本は `もしご購入後に別の原因が見つかった場合は、進める前にご相談します。` のように、勝手に進めないことが読める形にする。
- 料金不安が明示されている時は、`状況をご説明してご相談します` だけで止めず、必要なら `費用感をお伝えしたうえでご相談します` まで書く。相手が何を材料に判断できるかを残す。
- `.env` 注意文は原則 `.envやAPIキーの値は不要です。キー名だけで大丈夫です。` に統一する。
- `依頼者様側` のような過剰に硬い言い方は避ける。`こちらの範囲外ですが` `依頼者さま側の操作になります` などを使う。
- anxious の一文は万能テンプレにしない。相手の不安の種類に合わせて選ぶ。
  - 放置の自責 -> `まず今の状態から確認します。`
  - 影響範囲の恐怖 -> `まずは1件分の情報だけで大丈夫です。`
  - 予期せぬ事態 + 再発防止 -> `原因を特定して、今後同じことが起きないようにする対応も含めて確認します。`
  - 緊急停止 / 売上影響 -> `急ぎで見ます。` `優先して確認します。`
- 自責が見える相手でも、`その聞き方で大丈夫です` のような行為肯定を機械適用しない。日本語として不自然になりやすいので、短い受領か次アクションに寄せる。
- 緊急案件では、相手の切迫感を `お急ぎだと思います` のようにラベル化するより、こちらのアクション宣言を先に置く。
- 主質問が `何時間で見てもらえるか` `いつ見てもらえるか` のような時間質問なら、価格より先に `まず何時間で見立てを返すか` を答える。
- 時間や原因をまだ確定できない場面では、`現時点では / まだ / 断定できません` を重ねず、`コードを見てからになりますが` のように保留理由を1回で書く。
- `もし別の原因が見つかった場合は` のように、`もし` と `場合は` を重ねなくて足りる場面では片方に寄せる。

## 文体自由化ポリシー
- 目的: 運用ルールを壊さず、AI感を減らし、自然で温かい文面にする。
- 方針: 「言い換えは自由、ただし禁止表現だけ弾く」。

### 3層線引き
- 完全固定: 価格、スコープ、範囲外、禁止事項、セキュリティ注記
- 構造固定: 質問数、質問順、2時間+48時間、重複質問回避ゲート
- 表現自由: 書き出し、語尾、接続、文長、安心文の言い回し

### Must
1. 完全固定・構造固定を崩さない。
2. 初回は最大3点の依頼に収める。
3. 文長リズムを混ぜる（短文→中文→短文）。
4. 同一返信内で同じ語尾を連続させすぎない。
5. `大丈夫です` は1通あたり最大2回。
6. 既出情報の聞き直しをしない。
7. 外部誘導・秘密情報要求をしない。
8. 範囲外を伝えるときは代替案を1つ添える。
9. 最後に次アクションと時刻目安を入れる。
10. 仕上げ前にBan表現ヒット0件を確認する。
11. 質問は「1項目1情報」を原則にし、`AとB` の同時要求を避ける。
12. 購入後フェーズの返信は、必ず次回時刻目安を1つ入れる。
13. 資料受領後でも、`本日[時刻]まで` と `48時間以内` の2段構成は初回受領や本格着手のタイミングだけに使う。追加情報への短い返信、再修正、完了間近では1段の時刻コミットで足りる。
14. 完了確認フェーズや、すでに修正・納品が一度走っている場面では、2段構成や `一次結果` を機械適用しない。
15. 書き出しと末尾2行は、同一表現の連投を避けてローテーションする。
16. 資料受領後は、受領連絡 -> 中間報告 -> 具体時刻コミットの3段階を守る。
17. 公開 bugfix で確認専用の別料金案内をしない。
18. できないことの先回り説明より、返せるものや確認できる範囲を先に書く。
19. 同一バッチ内で `承りました。` `状況は把握しました。` `確認でき次第、2時間以内にはまず見立てをお返しします。` などの定型を3回以上繰り返さない。
20. 情報が薄い初動では、48時間行を機械的に入れず、まず見立て返却だけ約束する選択肢を残す。
21. 範囲外案内は、`今のサービスでは難しい` の拒否を1回だけ書き、その後は条件付き提案で終える。`断り -> 提案 -> 再度断り` の構造にしない。
22. 金額を出すなら断定し、迷うなら金額を出さない。`価格提示 + ヘッジ` の組み合わせにしない。
23. 相手が返答形式を明示した時は、その形式を外さない。
24. 前向きな感謝や驚きには、主質問の前に短い謝意を返して温度を合わせる。
25. `もちろんです。` を opening の標準形にしない。
26. Yes/No へ直答する時も、冒頭を `はい、` に固定しない。
27. 外向け本文に `bugfix` `handoff` `整理サービス` のような内部ラベルを残さない。

### Ban
- `ご共有ください`（`教えてください` / `共有ください`に置換）
- `不足分は` / `不足分だけ` で始まる指摘調の文
- `させていただきます` の多用（1通2回以上）
- `対応範囲外です。` の単独使用
- `ご案内します` の単独終止
- `必要情報がそろい次第` の硬直使用
- 過剰な名詞化
- 断定的な煽り（例: `必ず直ります`）
- 外部連絡/外部共有/外部決済を想起させる語
- 本番での再実行を促す語（再決済/再課金/Webhook再送など）
- `対応していません。` の単独終止
- 出品者側からの `補償` / `損害補償` の先出し
- 購入者向け本文での `確認案件`
- `もっともです`
- `見た目は軽く見えても`
- `認識のずれ`
- `ように見えた`
- `作業として発生します`
- 外向け本文での `別件として整理する` / `スコープを切って`
- 証跡未確認の見積り初回での `状況は把握しました`

## 速い修正への不信対策
- 意図的に遅らせない。
- 代わりに、次を見える化する。
1. 原因（1行）
2. 修正点（1行）
3. 確認手順（1〜3）
- 中間報告は「現状 / 次アクション / 次回報告時刻」の3点セットで送る。

## Stripe導線案内
- Stripe関連の確認依頼は `/home/hr-hm/Project/work/stripe日本語UI案内` を最優先参照する。
- 依頼時は同ファイルの「必須項目」を優先し、1通あたり2〜3点までに絞る。
- 同ファイルの「共有不要（`sk_...` / `whsec_...`）」に反する依頼をしない。
- 導線名に差異がある場合は、推測で断定せず「要実画面確認」を添える。

## 出力ルール
- 行頭に不要な空白を入れない。
- 基本は `です・ます`。
- 結論 -> 理由 -> 次アクション の順。
- 箇条書きは3点以内を優先。
- 見積り初回は相手負担を下げる（必要最小限の質問）。
- 判定前の返信では、追加料金の具体金額を先出ししない。公開 bugfix では確認専用の別料金案内もしない。
- 判定後の返信では、必要に応じて追加料金の具体金額を明記してよい。
- 値引き要求への返信では、価格を動かさず代替案の有無だけを返す。
- 見積り経路の追加発生では、`おひねり` を安易に先出しせず、まず進め方を2択で整理する。
- `emotional_caution: true` の場合、先頭は不便の認知に寄せ、最初の依頼は1点だけにする。
- 相手が複数症状をすでに具体的に書いている時は、`一番困っている方を1つだけ送ってください` に戻しすぎない。こちらで仮の起点を1つ置き、他の症状とつながっているかを続けて返す。
- 納品後の不満やクレームで `修正ファイルがない` `期待と違う` が来た時は、buyer の `認識のずれ` より先に、送付漏れ・不足・正式納品条件との整合を確認する。
- 非エンジニア初回は「最大3点」にする。
- 非エンジニア初回で `evt_...` `pi_...` などID共有を必須化しない（必要時は2通目で依頼）。
- 非エンジニア向けでは `ログ` 単語を単独要求しない。
- エラー文やログは、コピーできるならテキスト貼り付けを優先する。
- コピーしにくい場合だけ、画面スクショを代替として案内する。
- UI状態や見え方の確認では、画面スクショを優先してよい。
- Stripe 画面のスクショを依頼する時は、必要に応じて機密情報を隠してよいことを添える。
- 時刻を聞くときは粒度を下げる（例: `昨日の午後` / `今朝10時頃`）。
- `あればお願いします` を単独で使わない（任意なら `あると助かります（なくてもOK）` と明記）。
- 時刻未確定の案内は、見積りフェーズでは「見立て」、購入後フェーズでは「確認結果 / 判定結果」を使う。
- `2時間以内` の案内は、受領確認または一次報告を指す。一次結果は別時刻で書く。
- `48時間以内を目安に一次結果` は、購入直後の初回受領や本格着手の案内に限って使う。追加情報への返信、再修正、スコープ縮小後には貼り直さない。
- `形になります` `流れになります` で手順説明をぼかさない。相手に行動を求める場面では `こちらからお声がけします` `その時に入れてください` のように主語を出す。
- 見積り初回の高確度 bugfix は、`あいさつ -> 結論 -> 金額 -> 購入後に何を見るか -> 必要なら追加相談` の5点で足りる。定額サービスなので、毎回 `見積もり提案` を挟まない。
- `確認の入り口だけお返しします` は、情報不足かつ金額未提示の初動だけで使う。15,000円提示済みや緊急度が高いケースでは `見立てをお返しします` を優先する。
- 見積り初回の時刻案内は、次の候補から文脈に合うものを選び、同一バッチで固定連打しない。
  - `確認でき次第、2時間以内にはまず見立てをお返しします。`
  - `まずは2時間以内を目安に、確認の入り口だけお返しします。`
  - `まずは2時間以内を目標に、見立てを一度お返しします。`
- 資料受領後の案内は `この返信から` を使わず、`本日[時刻]までに` で時刻を明示する。
- 完了確認フェーズ（お礼返信・A/B確認）は、48時間行を省略してよい。
- purchased中の追加報告への返信では、相手のカジュアルさより急にフォーマルへ戻りすぎない。
- purchased中の追加報告への返信では、相手にもう一度一から情報を出させる感じを避け、必要な不足分だけを頼む。
- セキュリティ注意喚起は「禁止 -> 代替 -> 事後対処」の3点セットで書く。
- 追加質問の末尾にフォールバック文を入れる。
- `ログ` を依頼するときは場所を明示する。
- 責任/保証の質問には「保証範囲 -> ただし -> リスク低減手順」の3段構成で答える。
- 予防や再発不安への返信では、`再発しにくい形` のような保証っぽい言い方を避け、`今回確認できた原因については修正済み` を起点にしつつ、設定変更や別箇所変更で似た症状が出る余地があることを短く添える。
- buyer が `教えてほしい` と明示している時は、`必要であれば` で保留せず、短い予防ポイントをそのまま返してよい。
- 誤った画面の共有を受けたときは、否定語を避けて「次はこの画面を1枚お願いします」で誘導する。
- non_engineer 向けの見立てでは、`ルーティング` `呼び出し条件` などの技術語をそのまま置かず、`遷移先` `呼び出す流れ` などへ言い換える。
- non_engineer 向けでは、`判断がぶれやすい` `保持情報` のような内部語をそのまま出さない。`何から進めるかを整理しにくい` `サイト側で持っている情報` などへ言い換える。
- non_engineer 向けでは、`サブスクリプション更新` `Webhookハンドラ` など、平たく言い換えられる語は `課金への切り替え` `その処理のコード` へ落とす。
- Stripe不使用でも Next.js / Vercel / API連携の切り分けとして受けられる場合は、その場で一言明示する。`Stripeは使っていないとのことですが、Next.js / Vercel の本番切り分けとして対応可能です。`
- 範囲外案内の末尾で `対象外としてのご案内になります` のような官僚調の締めを足さない。
