# UI進行タグとショート指示

## UI進行タグ
- 誤記エイリアス（`#0$` / `#1$` / `#2$` / `#3$`）は、`#$0` / `#$1` / `#$2` / `#$3` と同義で扱う。
- `#$0`: 直接購入でトークルームが開いた直後の初回メッセージ。
  - 文面: `ご購入ありがとうございます。さっそく進めますね。まず症状を確認して、原因の見立てをお伝えする流れです。次の3点だけ教えてください。1) いちばん困っている症状（1つ） 2) その画面のスクショ（1枚） 3) いつ頃から起きているか（だいたいでOK） 届きしだい確認に入りますね。2時間以内にはまず見立てをお返しします。`
- `#$1`: 合意後・提案送信前のひと言。
  - 文面: `ありがとうございます。内容が固まりましたので、提案をお送りします。`
- `#$2`: 提案送信後の購入案内。
  - 文面: `提案をお送りしました。内容をご確認いただき、よろしければ「提案を購入する」からお進みください。ご不明点があればお気軽にどうぞ。`
- `#$3`: 提案送信後の未返信フォロー（急かさない）。
  - 文面: `先日お送りした提案をご確認いただけましたら、都合のよいタイミングでご返信ください。ご不明点があればお気軽にどうぞ。`
- `#$1/#$2` はサービス経由・プロフィール経由・メッセージ経由の見積り提案で共通利用する。
- `#$1` を送ってから提案ボタンを押し、送信後に `#$2` を送る（同時送信しない）。
- `#$3` は提案送信後に返信が止まった場合のみ使う（購入期限の圧をかけない）。
- UI進行タグで作成した文面も、送信用として `/home/hr-hm/Project/work/runtime/replies/latest.txt` に保存する。

## `#R`ショート指示（受領後3段階）
- `#R 受領返信`: 資料受領直後の1通。受領 + いまから確認する旨を短く返す。
- `#R 進捗返信`: 受領後30〜60分の途中報告。いま見ている点 + 次回時刻を返す。
- `#R 方向性返信`: 原因候補が見えた時点の最初の見立て。候補 + 次アクション + 次回時刻を返す。
- 時刻が未指定の場合は、JST現在時刻 + 2時間を自動補完する（同日なら `本日HH:MMまで`、日付をまたぐ場合は `明日HH:MMまで`）。
- 時刻が指定されている場合は、指定時刻を優先する。
- これらは自動送信ではなく、ユーザーが明示した時だけ作成する。

## `#P`ショート指示（こちら起点で buyer に送る文面）
- `#P` は、buyer の文章に返す返信ではなく、**こちら起点で buyer に送る文面** を作る。
- デフォルトの `#P` は、購入後トークルームの途中報告として扱う。
- 目的を明示する時は、次のような形で使う。
  - `#P 途中報告`
  - `#P 着手報告`
  - `#P 追加情報依頼`
  - `#P 補足`
  - `#P 完了報告`
  - `#P 納品前確認`
  - `#P 正式納品案内`
  - `#P 承諾前フォロー`
- `#P` の骨格は「いま何が分かっているか」「次に何をするか」「buyer に何をしてほしいか」を、必要な粒度で buyer 向けに整える。
- 途中報告では、骨格は原則3行:
  - 1行目: `進捗の共有です。`
  - 2行目: `どこまでは正常で、現在どこを見ているか`
  - 3行目: `次にやること` と `次回連絡目安` または `必要情報`
- 2行目は「境界」を書く。例: `決済画面までは正常で、現在、決済後の自動反映部分を確認しています。`
- 技術語は最大1つまでを目安にし、非エンジニア相手には流れの言葉へ言い換える。
- 未確認の原因断定 / 進捗率 / ログ生貼りは避ける。
- 既存の `#R 進捗返信` は互換用として残し、今後のこちら起点文面は `#P` を優先する。

## `#R`ショート指示（納品〜クローズ）
- `#R 差し戻し返信`: 差し戻し受領後の返信文（受領 + 修正方針 + 次回時刻）を作成する。
- `#R クローズお礼`: クローズ前後のお礼文を作成する。評価依頼は1回のみ・任意トーンで入れる。クローズ後の追いDMは提案しない。
- `#R` はあくまで buyer の文章やアクションに返す返信レーンとして扱う。
