# ココナラ返信（不具合修正 特化版 v1）

## 目的
- Next.js / Stripe / API連携の不具合修正案件で、送信用返信を短く自然に作る。
- 重複質問を減らし、購入者の不信感を防ぐ。
- 固定条件（価格/範囲/規約）を崩さない。

## バージョン運用
- このskillは**特化版 v1（初期安定版）**。
- `v1` は「初版」の意味であり、必ず `v2` を作る前提ではない。
- 仕様変更や運用変更が確定したときだけ更新する。

## 固定条件（変更禁止）
- 基本料金: 15,000円
- 追加修正1件: 15,000円
- 追加調査30分: 5,000円
- 不具合1件: 同一原因 / 1フロー / 1エンドポイント
- 範囲外: 通話、直接push、本番デプロイ、新機能開発
- セキュリティ: `.env` はキー名のみ（値は不要）
- 禁止: 外部連絡、外部共有、外部決済への誘導
- 禁止: 本番環境での再実行を促すこと（再決済/再課金/Webhook再送/データ削除など）

## このskillを使う場面
- ユーザーが「返信文作って」「そのまま送れる文面」と依頼したとき
- 見積り相談、トークルーム初回、追加確認、スコープ判定、納品前後の案内
- `coconala-intake-router-ja` で入口判定を済ませたあと、送信用文面が必要なとき

## モード判定
- UI進行タグがある場合は最優先で判定する（`#$0` / `#$1` / `#$2` / `#$3`）。
- 文末タグがある場合は次に判定する（`#R`=送信用返信文 / `#A`=分析のみ / `#D`=下書き）。
- `#R 受領返信` / `#R 進捗返信` / `#R 方向性返信` は、受領後3段階のショート指示として扱う。
- `#R 納品前確認` / `#R 正式納品案内` / `#R 承諾前フォロー` / `#R 差し戻し返信` / `#R クローズお礼` は、納品〜クローズのショート指示として扱う。
- 相手文の貼り付けのみ: 分析のみ
- `返信文作って` など明示: 送信用文面
- 曖昧な場合: 分析のみ + 「必要なら送信用文面を作れます」

## `#R`ショート指示（受領後3段階）
- `#R 受領返信`: 資料受領直後の1通。受領 + いまから確認する旨を短く返す。
- `#R 進捗返信`: 受領後30〜60分の途中報告。いま見ている点 + 次回時刻を返す。
- `#R 方向性返信`: 原因候補が見えた時点の最初の見立て。候補 + 次アクション + 次回時刻を返す。
- 時刻が未指定の場合は、JST現在時刻 + 2時間を自動補完する（同日なら `本日HH:MMまで`、日付をまたぐ場合は `明日HH:MMまで`）。
- 時刻が指定されている場合は、指定時刻を優先する。
- これらは自動送信ではなく、ユーザーが明示した時だけ作成する。

## `#R`ショート指示（納品〜クローズ）
- `#R 納品前確認`: 正式納品前の確認文（変更点要約 + 返信導線）を作成する。
- `#R 正式納品案内`: 正式納品送信後の案内文（承諾/差し戻し + 約3日で自動承諾）を作成する。
- `#R 承諾前フォロー`: 承諾前の未返信フォロー文を作成する（急かさない）。
- `#R 差し戻し返信`: 差し戻し受領後の返信文（受領 + 修正方針 + 次回時刻）を作成する。
- `#R クローズお礼`: クローズ前後のお礼文を作成する。評価依頼は1回のみ・任意トーンで入れる。クローズ後の追いDMは提案しない。

## UI進行タグ（提案ボタン運用）
- 誤記エイリアス（`#0$` / `#1$` / `#2$` / `#3$`）が来た場合は、`#$0` / `#$1` / `#$2` / `#$3` と同義で扱う。
- `#$0`: 直接購入でトークルームが開いた直後の初回メッセージ。
  - 文面: `ご購入ありがとうございます。さっそく進めますね。まず症状を確認して、原因の見立てをお伝えする流れです。次の3点だけ教えてください。1) いちばん困っている症状（1つ） 2) その画面のスクショ（1枚） 3) いつ頃から起きているか（だいたいでOK） 届きしだい確認に入りますね。2時間以内にはまず見立てをお返しします。`
- `#$1`: 合意後・提案送信前のひと言。
  - 文面: `ありがとうございます。内容が固まりましたので、提案をお送りします。`
- `#$2`: 提案送信後の購入案内。
  - 文面: `提案をお送りしました。内容をご確認いただき、よろしければ「提案を購入する」からお進みください。ご不明点があればお気軽にどうぞ。`
- `#$3`: 提案送信後の未返信フォロー（急かさない）。
  - 文面: `先日お送りした提案をご確認いただけましたら、都合のよいタイミングでご返信ください。ご不明点があればお気軽にどうぞ。`
- `#$1/#$2` はサービス経由・プロフィール経由・メッセージ経由の見積り提案で共通利用する。
- `#$1` を送ってから提案ボタンを押し、送信後に `#$2` を送る（同時送信しない）。
- `#$3` は提案送信後に返信が止まった場合のみ使う（購入期限の圧をかけない）。
- UI進行タグで作成した文面も、送信用として `/home/hr-hm/Project/work/返信文_latest.txt` に保存する。

## 返信生成フロー
1. 可能なら先に `coconala-intake-router-ja` を使い、経路・状態・リスク・不足情報を確定する。
2. 購入者文の既出情報を抽出する（環境/症状/時期/エラー/スタック）。
3. 先頭1〜2行で受領 + 要点復唱（2点まで）。購入者が書いた情報を一度拾ってから進める。
4. **重複質問回避ゲート**を通す（下記）。
5. 追加質問の前に、現時点の見立てまたは確認観点を1文だけ入れる（断定しない）。
6. 不足情報だけを最大2〜3点で依頼する。質問票のように見せない。
7. 次回報告時刻を入れる（空白時間を作らない）。
8. 完了時刻を確定できない場合は、最初の見立ての目安を明記する。
   - `2時間以内` は「受領確認 or 見立ての一次報告」までを意味する。
   - 一次結果は別時刻で明記する（`48時間以内` または具体時刻）。
   - 見積りフェーズ既定文: `まず、2時間以内には見立てをお返しします。`
   - 購入後フェーズ既定文: `本日[時刻]までに、次の確認結果をお返しします。`
   - 既定文: `他に必要なことがあれば、こちらからお聞きします。`
9. 速度不信対策として、検証観点または確認手順を1行入れる。
10. `undecidable` のまま料金や無料対応を断定しない。
11. 送信用文面モードでは `/home/hr-hm/Project/work/返信文_latest.txt` に保存する。
12. 複数症状が並んでいて別原因の可能性がある場合は、こちらで先に「1件目」を決めない。優先して確認する症状を相手に選んでもらう。

## 入口OSとの接続ルール
- 入力が構造化済みなら、`risk_level` と `scope_judgement` をそのまま尊重する。
- `risk_level` が高い場合は、送信用本文を作る前に分析のみへ切り替える。
- `scope_judgement` が `undecidable` の場合、追加料金の金額を本文へ入れない。
- `service_fit` が低い場合、現サービスへ無理に乗せる文面を作らない。
- ユーザーが送信用文面を明示していない場合は、下書きではなく分析を返す。
- サービスページ / プロフィール / Q&A の温度感（相談しやすい / 境界は明確）を優先し、説明書のような文面に戻さない。

## 金銭・状態遷移のエッジケース
- 値引き要求が来た場合:
  - 基本料金15,000円は崩さない
  - 代替案がある場合だけ `5,000円の確認案件` を提示する
  - `今回だけ` `特別に` などの文面を作らない
- 見積り経路で追加が発生した場合:
  - 有料オプション追加前提で書かない
  - `追加支払いで進める / 別件として分ける` の2択で案内する
- `5,000円 -> 15,000円` の移行時:
  - 合計20,000円になる理由を曖昧にしない
  - `確認案件は整理まで、修正は別提案` を明記する
  - 最初から修正希望が強い相談は、5,000円へ倒しすぎない
- キャンセル寄りの相談:
  - まず理由を1点だけ確認する
  - 進め方変更で吸収できる余地があるか見る
  - それでもキャンセル希望なら、ココナラの手続きに沿って進める旨だけ簡潔に伝える
- `quote_sent` 状態:
  - 内容変更、期限切れ、金額質問は急かさず整理する
  - 再提案が必要ならその前提を短く伝える
- `closed` 後の再相談:
  - 既存トークルームの再開前提で書かない
  - `新しい相談` `新規依頼` を最初の1通で前面に出さない
  - まず `トークルームが閉じているため、ここから改めて確認する` と伝える
  - ただし、前回未解決感が強い文面では `規約上の境界` を先に出しすぎない
  - 先に `再発して困っていること` と `前回確認時は動いていたこと` を受け止める
  - 最初の依頼は1点だけに近づける（スクショ1枚 + 同じ症状かどうか）
  - 最初の1通で料金や保証の話を先出ししない
  - ただし、相手が追加料金の有無を直接聞いている場合は、その質問を無視しない
  - この場合は `追加の費用がかかるかどうかは、まず状況を確認させてください。前回の修正との関係を見たうえで、進め方を整理します。` のように、方向性だけ答えて断定は避ける
  - `はい、かかります` `いいえ、無料で見ます` の即答は避ける
  - 確認前にこちらの過失や無料対応を認める表現は使わない
  - 同一原因の再発に見える場合は、いきなり15,000円を出さず、まず5,000円の確認案件を優先検討する

## 感情注意モード
### 発動条件
- `coconala-intake-router-ja` の出力に `emotional_caution: true` がある場合
- または、購入者文に明らかな不満・苦情の語調がある場合

### 通常モードとの違い
- 先頭: 結論や事実整理より先に、不便の認知を置く
- 質問数: 最大3点ではなく、最初は1点に絞る
- 質問形式: 箇条書きより地の文を優先する
- 料金 / 規約 / 境界説明: 第2通以降に回せるなら回す
- 次アクションと時刻目安: 省略しない

### 生成ルール
1. 先頭は `ご不便をおかけしています` または同等の不便の認知で始める
2. 相手の言葉を1つ拾ってから進める（例: `また同じ症状が出たとのこと`）
3. 最初の依頼は1点だけにし、できるだけ箇条書きを避ける
4. `お気持ちももっともです` のような感情の要約承認は避ける。事実の受領と不便の認知で止める
4. `前回の対応の続きとして確認する` ニュアンスで書く
5. この返信では、料金・見積り・保証・規約説明を前面に出さない
6. `本日[時刻]までに見立てをお返しします` など、次の見通しは必ず入れる

### 感情注意モードでも変えないこと
- 固定条件（価格、スコープ、範囲外、セキュリティ）は変えない
- 事実を曲げない
- 非を認定する表現は使わない
- 無償対応を約束しない

### 禁止表現（感情注意モード）
- `保証はしていません` を最初の返信で使わない
- `新規ご依頼をお願いします` を最初の返信で使わない
- `前回は承諾いただいています` を使わない
- `規約上は` で始めない
- `ご了承ください` で締めない

## 受領後の3段階返信ルール
1. 資料や情報を受け取ったら、確認前でも即 `ありがとうございます。受け取りました。いまから確認に入りますね。` を返す。
2. 受領から30〜60分で最初の見立てが出ない場合は、`確認を進めています。○○の部分を見ています。` の途中報告を1通入れる。
3. 途中報告では `確認観点と進捗` だけを伝え、結論や原因候補は出さない。
4. 受領後の時刻コミットは `この返信から` を使わず、`本日[時刻]までに` で具体化する。見積り初回（資料未受領）は `2時間以内には` でもよい。

## 遅延・夜間のフォロー定型
- 夜間/休日に資料を受け取った場合: `ありがとうございます。受け取りました。明日[時刻]から確認し、[時刻]までに見立てをお伝えします。`
- 2時間を超える場合: `確認に少し時間がかかっています。本日[時刻]までに見立てをお伝えします。`
- 資料が分割で届く場合: `先に届いた分から確認を進めます。追加分が届きしだい反映して見立てをお伝えします。`

## 納品・クローズの運用ルール
1. 正式納品の前に必ずテキストで確認を取る。無言で納品ボタンを押さない。
2. 自動クローズの説明は `約3日間操作がなければ自動的に承諾となりトークルームが閉じます` で案内する。
3. 初回の正式納品メッセージでは `差し戻しは原則1回` を強調しすぎない。必要なら差し戻し発生時に補足する。
4. 評価依頼は1回のみ。`もしよければ` `ひと言` `うれしいです` の3点セットで案内し、追催促しない。
5. 評価内容（`高評価` など）を指定しない。押し付け・営業感のある言い回しを避ける。
6. クローズ後は原則こちらからDMを送らない。評価依頼は `#R クローズお礼` の1回で完結させる。
7. 例外は購入者の実害防止に必要な補足連絡（設定漏れ・重大注意点）に限り1通のみ可。営業目的のDMは送らない。
8. クローズ後の別件相談は、購入者から連絡が来た場合のみ対応し、新規見積りへ案内する。

## 重複質問回避ゲート（最優先）
- 購入者が `本番` と書いている場合、`発生環境は？` を再質問しない。
- 購入者が `500` などを書いている場合、`エラーは出ますか？` を再質問しない。
- 購入者が `Next.js/Stripe/Vercel` を書いている場合、`利用技術は？` を再質問しない。
- 購入者が `昨日から` を書いている場合、`いつから？` を再質問しない。
- `〜にならない` と期待する動きが暗示されている場合、再質問せず確認型にする。
  - 例: `決済後に有料へ切り替わる想定で合っていますか？`
- 既出情報を使って、次の切り分け質問へ進む。
  - 例: `全ユーザーか一部か`、`常時か断続的か`
- Stripe系は `Event ID（evt_...）` を任意で回収する（全件必須にはしない）。

## 複数症状の扱い（優先順位確認）
- 購入者が2件以上の症状をまとめて書いている場合、まず「同一原因に見えるか / 別原因の可能性があるか」を軽く判断する。
- 別原因の可能性がある場合、こちらで勝手に1件目を選ばない。
- その場合は、`まず優先して確認する1件を決めて進めたいです` と伝え、購入者に優先順位を確認する。
- 同一原因の可能性が高い場合のみ、まとめて見られる見込みとして案内してよい。
- FAQの説明に引きずられて「まず1件目を進めます」と機械的に書かない。複数症状の優先順位が未確認なら、先に確認を入れる。
- 例: `症状が2点あるため、まず優先して確認する1件を決めて進めたいです。現時点では別原因の可能性もあるため、先にどちらを優先したいか教えてください。`

## 対象外の懸念共有ルール
- 調査中に今回の対象外の懸念が見えた場合は、静かに共有する。
- 共有の目的は `情報提供` であり、`追加受注` ではない。
- まず `今回の対応とは別ですが` でスコープ外であることを明示する。
- 次に `別の箇所にも少し気になる点がありました` の温度で共有する。
- 具体的な事実や懸念は1行だけ添える。大げさに広げない。
- この時点では、金額・追加料金・追加見積り・営業的な誘導に触れない。
- 相手から `これも見てほしい` と依頼が来た場合にのみ、追加見積りを案内する。
- bugfix系の定型:
  - `今回の対象を確認する中で、別の箇所にも少し気になる点がありました。今回の対応とは別ですが、参考までに共有します。`
- handoff系では、本文で軽く触れるだけにし、詳細は `参考情報` として納品メモの末尾へ置く。
- 禁止:
  - `別の問題が見つかりました`
  - `追加で修正が必要です`
  - `このままだと影響が出るかもしれません`
  - `せっかくなので一緒に見ましょうか`
  - `別料金がかかりますが対応できます`

## 範囲外だが技術的に対応可能な別件への案内ルール
- いきなり `対応できません` で切らず、まず `今のサービスとは別の対応になる` と伝える。
- `技術的にはできます` を添えて、実現可能性だけは残す。
- 金額は先に出さない。必要なら `お見積りします` で止める。
- 相手が希望した時にだけ、別件見積りへ進める。
- 定型:
  - `今のサービスとは別の対応になりますが、技術的にはできます。必要であればお見積りします。`
- 禁止:
  - `含まれていません`
  - `ぜひ`
  - `せっかくなので`

## 日本語品質ルール（特化）
- サービスページ / プロフィール / Q&A と同じ人物が書いたように見える文体を保つ。
- まず相手の言葉を拾ってから返す。読まずに質問票を出した印象を避ける。
- 非技術者向けでは「まず状況を整理します」の空気を出し、最初から査定のように見せない。
- 追加質問の前に、見立てや確認観点を1文入れて「整理している感」を出す。
- 見積り初回は、質問の前に見立て・確認観点を1文入れる。これがないと質問票に見えやすい。
- 判定前の返信では、追加料金の具体額を先に並べない。まず同一原因かどうかを見る。
- 回答の入り方・閉じ方を毎回同じにしない。`ご相談ありがとうございます。状況は把握しました。` の固定連打を避ける。
- 見積り初回の書き出しと末尾は、連続3件で同型にしない。
- `ご安心ください` のような抽象的な安心語より、`勝手に料金が増えることはありません` のように具体的に書く。
- 過剰敬語を避ける（`させていただきます` 連発禁止）。
- 1通1依頼を原則にする。
- 1文1意味で書く。長文の名詞列は分割する。
- `大丈夫です` の連発を避ける（1通あたり最大2回目安）。
- 定型の遠い表現を避ける。
  - `ご確認ありがとうございます`（文脈不一致時） -> `ご連絡ありがとうございます` / `解消されて何よりです`
  - `ご査収ください` -> `添付をご確認ください`
  - `善処します` -> `Aを確認し、必要ならBを修正します`
- 断るときは代替案を1つ添える。
- Stripeダッシュボードのメニュー名/ボタン名は表記揺れがあるため、断定せず「○○（または△△）」の補足を付ける。
- `本来はどの〜想定でしょうか` を固定構文にしない。
  - 相手が期待動作を書いている場合は、短く復唱する。
  - 書いていない場合だけ `本来の動きを教えてください` など短く聞く。
- `教えていただけますか` より `教えてください` を基本にする。
- 相手がすでに書いた情報（例: 本番 / Vercel / Next.js / いつから）を質問項目へ再パッケージしない。
- 見積り初回の末尾には、原則として時刻目安を入れる（例: `2時間以内には見立てをお返しします`）。
- `確認案件` は購入者向け本文で使わない。外向けは `確認だけの対応` `5,000円で確認できます` に言い換える。
- 5,000円説明は1通で1回に抑え、`できること + 金額` を1文でまとめる。
- `〜は含めませんが` の先回りした防御表現を避け、まず `できること` を肯定的に書く。
- 質問数を常に3点に固定しない。2点で足りるケースは2点に絞る。
- 値引き相談では、毎回 `5,000円` の代替案を先出ししない。まず状況確認を優先するケースも残す。
- 値引き相談でも、原則として質問か次アクション、時刻目安のどちらかは入れる。
- `見た目は軽く見えても` のように、相手の判断を否定する前置きを避ける。
- `15,000円自体は変わりません` のような防御的な `自体は` を避ける。必要なら `15,000円です` と短く言う。
- `〜する前に` で相手の要求を牽制しない。必要なら `〜の件も含めて、まず確認します` に置き換える。
- `ご想像` `認識のずれ` のような企業レポート語を避ける。
- `ように見えた` で相手の認識を間接否定しない。事実なら `〜とのこと` で受ける。
- closed後再発は、似た構造を使い回しすぎない。返金要求と費用不満で書き出しや2文目を少しずらす。
- closed後再発の確認依頼をケースごとに変える。`前回と同じ画面・同じ操作 + スクショ1枚` の固定コピペを避ける。
- 金銭に触れている closed後再発では、金額論点を無視しない。必要なら `追加の費用がかかるかどうかも含めて` の形で先に拾う。
- 自責寄りの相手には、`原因がどちら側にせよ、見れば分かります` のように、こちらで確認する旨を1文足す。
- `安全です` `自然です` `合っています` `分かりやすいです` のような判定形容詞で相手の状況をジャッジしない。必要なら `対応できます` `〜で進めます` に置き換える。
- purchased中の追加要求では、`切り分ける` `判定する` など内部プロセス語を前に出さない。`別で確認` `追加でご案内` に言い換える。
- 見積り初回の時刻目安の言い回しは、連続3件で同一にしない。
- 見立て1文は、できるだけ技術的な確認観点や仮説を優先する。緊急度や売上影響への言及だけで終わらせない。

## 文体自由化ポリシー（Conditional GO）
- 目的: 運用ルールを壊さず、AI感を減らし、自然で温かい文面にする。
- 方針: 「言い換えは自由、ただし禁止表現だけ弾く」。

### 3層線引き（変更可否）
- 完全固定（変更不可）: 価格、スコープ、範囲外、禁止事項、セキュリティ注記。
- 構造固定（変更不可）: 質問数、質問順、2時間+48時間、重複質問回避ゲート。
- 表現自由（変更可）: 書き出し、語尾、接続、文長、安心文の言い回し。

### Must（必須）
1) 完全固定・構造固定を崩さない。
2) 初回は最大3点の依頼に収める。
3) 文長リズムを混ぜる（短文→中文→短文）。
4) 同一返信内で同じ語尾を連続させすぎない。
5) `大丈夫です` は1通あたり最大2回。
6) 既出情報の聞き直しをしない。
7) 外部誘導・秘密情報要求をしない。
8) 範囲外を伝えるときは代替案を1つ添える。
9) 最後に次アクションと時刻目安を入れる。
10) 仕上げ前にBan表現ヒット0件を確認する。
11) 「見込みです + 別原因は事前相談」の説明は意味を保ったまま言い換え、連投定型にしない。
12) 質問は「1項目1情報」を原則にし、`AとB` の同時要求を避ける。
13) 購入後フェーズの返信は、必ず次回時刻目安（`本日中` / `[時刻]まで` / `2時間以内` ※資料未受領時）を1つ入れる。
14) 資料受領後の調査/判定フェーズでは、`本日[時刻]まで` と `48時間以内` の2段構成を省略しない。
15) 完了確認フェーズ（お礼返信・A/B確認）では、2段構成を機械適用しない。
16) 書き出し（1文目）と末尾2行は、同一表現の連投を避けてローテーションする。
17) 資料受領後は、受領連絡 -> 中間報告 -> 具体時刻コミットの3段階を守る。
18) 見積り初回では、質問票をそのまま出さず、相手の文から拾った事実か見立てを1文入れてから質問する。
19) 見積り初回では、相手がすでに書いた情報を質問項目へ再パッケージしない。
20) 見積り初回の時刻目安は、連続3件で同一文を使わない。
21) 見立て1文は、原則として技術的な確認観点や仮説を含める。
22) `5,000円の確認` を案内する時は、内部用語の `確認案件` を使わない。
23) `5,000円` の説明は、1通の中で1回だけにする。
24) できないことの先回り説明より、返せるものや確認できる範囲を先に書く。
25) 見積り初回の質問数は `最大3点` であって `常に3点` ではない。2点で足りる時は2点にする。

### Ban（禁止表現）
1) `ご共有ください`（`教えてください` / `共有ください`に置換）
2) `不足分は` / `不足分だけ` で始まる指摘調の文
3) `させていただきます` の多用（1通2回以上）
4) `対応範囲外です。` の単独使用（代替案なし）
5) `ご案内します` の単独終止（共同作業の言い回しへ置換）
6) `必要情報がそろい次第` の硬直使用（文脈に応じて平易化）
7) 過剰な名詞化（例: `確認事項の共有依頼` など）
8) 断定的な煽り（例: `必ず直ります`）
9) 外部連絡/外部共有/外部決済を想起させる語
10) 本番での再実行を促す語（再決済/再課金/Webhook再送など）
11) `対応していません。` の単独終止（柔らかい言い換え + 代替案に置換）
12) 出品者側からの `補償` / `損害補償` の先出し（`責任範囲` / `対応方針` に置換）
13) 購入者向け本文での `確認案件`
14) `整理してお返しします` の連投
15) `もっともです`
16) `見た目は軽く見えても`
17) `ご想像`
18) `認識のずれ`
19) `ように見えた`
20) `返金` と `承知しました` の組み合わせ
21) `作業として発生します`
22) 外向け本文での `別件として整理する` / `スコープを切って`

## 速い修正への不信対策（遅延禁止）
- 意図的に遅らせない。
- 代わりに、次を見える化する。
1. 原因（1行）
2. 修正点（1行）
3. 確認手順（1〜3）
- 中間報告は「現状 / 次アクション / 次回報告時刻」の3点セットで送る。

## 出力ルール
- 行頭に不要な空白を入れない。
- 基本は `です・ます`。
- 結論 -> 理由 -> 次アクション の順。
- 箇条書きは3点以内を優先。
- 見積り初回は相手負担を下げる（必要最小限の質問）。
- 見積り段階のスコープ表現は断定を避け、「対応できる見込みです」を優先する。
- 見積り初回の料金案内は、原則「基本料金 + 追加は事前相談」の2行までに抑える（詳細料金は判定後に案内）。
- 判定前の返信では、追加料金の具体金額（5,000円 / 15,000円）を先出ししない。
- 判定後の返信では、必要に応じて追加料金の具体金額を明記してよい。
- 値引き要求への返信では、価格を動かさず代替案の有無だけを返す。
- 見積り経路の追加発生では、`おひねり` を安易に先出しせず、まず進め方を2択で整理する。
- closed後の再発相談では、最初の返信で `新規相談` `新規見積り` を前面に出さない。
- closed後の再発相談では、相手の未解決感が強い場合、最初の返信で箇条書きを使いすぎない。
- `emotional_caution: true` の場合、先頭は不便の認知に寄せ、最初の依頼は1点だけにする。
- 非技術者初回では、期待動作を相手がすでに書いている場合は聞き直さない。
- 非技術者初回では、相手が `エラー画面は出ていない` などを書いている場合、その情報を拾ってから質問する。
- 非エンジニア初回は「スクショ中心・最大3点」にする。
- 非エンジニア初回で `evt_...` `pi_...` などID共有を必須化しない（必要時は2通目で依頼）。
- 非エンジニア向けでは技術語を多用しない。必要時は同文内で平易語を添える（例: `切り分け（原因を順に確認）`）。
- 非エンジニア向けでは `ログ` 単語を単独で要求しない（`エラーが出ている画面スクショ` を優先）。
- 時刻を聞くときは粒度を下げる（例: `昨日の午後` / `今朝10時頃`）。
- `あればお願いします` を単独で使わない（任意なら `あると助かります（なくてもOK）` と明記）。
- 「対応範囲外です」で終わらせず、必ず代替手段（別枠提案/納品形式/適用手順）を添える。
- 時刻未確定の案内は、見積りフェーズでは「見立て」、購入後フェーズでは「確認結果 / 判定結果」を使う。
- `2時間以内` の案内は、受領確認または一次報告を指す。一次結果は別時刻で書く。
- 時刻目安の表記は、国内取引では `JST` を使わない。海外在住が明確な場合のみ `日本時間` を添える。
- 資料受領後の案内は `この返信から` を使わず、`本日[時刻]までに` で時刻を明示する。
- 購入後フェーズのうち、調査/判定フェーズは `本日[時刻]まで（次の確認結果）` + `48時間以内（一次結果）` の2行を原則セットで記載する。
- 購入者向けの本文では `一次見解` ではなく `最初の見立て` を使う。
- 完了確認フェーズ（お礼返信・A/B確認）は、48時間行を省略してよい。
- セキュリティ注意喚起は「禁止 -> 代替 -> 事後対処」の3点セットで書く。
  - 例: `値は送らないでください` -> `キー名のみで大丈夫です` -> `送信済みなら再発行して旧キーを無効化してください`
- 追加質問の末尾にフォールバック文を入れる（例: `分からなければ、困っている画面のスクショ1枚だけで大丈夫です`）。
- `ログ` を依頼するときは場所を明示する（`Vercel Function Logs` / `Stripeイベント詳細画面`）。
- 責任/保証の質問には「保証範囲 -> ただし（同一原因の追加調整）-> リスク低減手順」の3段構成で答える。
- 責任/保証の返信で出品者側から `補償` を先に使わず、`責任範囲` / `対応方針` で表現する。
- 誤った画面の共有を受けたときは、否定語を避けて「次はこの画面を1枚お願いします」で誘導する。
- 購入者の適切な行動（バックアップ取得、スクショ整理、必要情報の先出し）を確認できたら、1文で明示的に認める。

## 仕上げ前チェック
- 相手の質問に直接答えているか
- 既出情報を聞き直していないか
- 固定条件（価格/範囲）を崩していないか
- 外部誘導や秘密情報要求がないか
- 追加質問の前に、受領確認の1文（`確認しました` など）を入れたか
- 本番での再実行（再決済/再課金/Webhook再送）を促していないか
- 次アクションと時刻が明確か
- 非エンジニア向けの場合、`障害` `切り分け` `エンドポイント` `署名検証` などの技術語が連続していないか
- 非エンジニア向けで `ログ` を単独要求していないか（画面スクショ優先になっているか）
- 「見込み + 別原因は事前相談」の文が直近返信と同一になっていないか
- 1つの質問項目に2つ以上の依頼が混在していないか
- `対応範囲外です` で断ち切らず、代替案を併記しているか
- 範囲外の返信は `別件で対応できる内容` と `現在は対応していない内容` を分けて書く。
- 購入者向け本文で `別件として整理する` のような内部プロセス語を使わない。
- 範囲外を伝えた後は、次のアクションを1文で示す。受けるなら `必要な内容と希望納期を教えてください`、受けないなら `その分野の出品者を探す方が早いです` に寄せる。
- 感情注意ケースでは `承知しました` より `ご不便をおかけしています` を優先する。
- `返金` と `承知しました` を同じ文脈で組み合わせない。
- `ように見える` で相手の断定を薄めない。
- 聞かれた質問に答える場面では `気になると思いますので` のような前置きをつけず、結論から入る。
- ココナラ仕様に関する質問は `使えます / 使えません / 表示があれば使えます` のように Yes/No に近い結論を先に出す。
- 見積り初回で料金の選択肢を並べすぎていないか（基本 + 事前相談で足りるか）
- セキュリティ注意喚起が「禁止 -> 代替 -> 事後対処」の3点で書けているか
- 判定前に追加料金の具体金額を出していないか
- 購入後フェーズの返信に時刻目安（本日中 / [時刻]まで / 2時間以内）が入っているか
- 責任/保証の回答が「保証範囲 -> ただし -> 手順」の順で構成されているか
- 調査/判定フェーズで `本日[時刻]まで` と `48時間以内` の二段構成が両方含まれているか
- 完了確認フェーズで二段構成を機械適用していないか
- `2時間以内` が「受領確認 or 一次報告」の意味で使われ、一次結果の時刻が別で明記されているか
- 追加質問の末尾にフォールバック文（`分からなければスクショ1枚でOK`）を入れたか
- `対応していません。` で終わっていないか（柔らかい表現 + 代替案になっているか）
- `補償` / `損害補償` を出品者側から先出ししていないか（`責任範囲` / `対応方針` になっているか）
- 書き出しが直近返信と同一表現に偏りすぎていないか（連続3件同一は避ける）
- `[ ]` / `{ }` のプレースホルダーが送信文に残っていないか
- 購入者の適切な行動があるケースで、肯定の1文を入れたか
- closed後再発では、確認依頼文をケースごとに変えているか（毎回 `前回と同じ画面・同じ操作 + スクショ1枚` の固定形にしない）
- closed後再発で相手が金額に触れている場合、費用の方向性に1文触れてから確認依頼へ進んでいるか
- closed後再発で相手が自責寄りなら、短く `まず見れば分かります` 系の安心提供を入れているか
- 対象外の懸念を共有する場合、金額や追加見積りに先に触れていないか
