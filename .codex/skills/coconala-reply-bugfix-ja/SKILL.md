# ココナラ返信（不具合修正 特化版 v1）

## 目的
- Next.js / Stripe / API連携の不具合修正案件で、送信用返信を短く自然に作る。
- 重複質問を減らし、購入者の不信感を防ぐ。
- 固定条件（価格/範囲/規約）を崩さない。

## バージョン運用
- このskillは**特化版 v1（初期安定版）**。
- `v1` は「初版」の意味であり、必ず `v2` を作る前提ではない。
- 仕様変更や運用変更が確定したときだけ更新する。

## 固定条件（変更禁止）
- 基本料金: 15,000円
- 追加修正1件: 15,000円
- 追加調査30分: 5,000円
- 不具合1件: 同一原因 / 1フロー / 1エンドポイント
- 範囲外: 通話、直接push、本番デプロイ、新機能開発
- セキュリティ: `.env` はキー名のみ（値は不要）
- 禁止: 外部連絡、外部共有、外部決済への誘導
- 禁止: 本番環境での再実行を促すこと（再決済/再課金/Webhook再送/データ削除など）

## このskillを使う場面
- ユーザーが「返信文作って」「そのまま送れる文面」と依頼したとき
- 見積り相談、トークルーム初回、追加確認、スコープ判定、納品前後の案内

## モード判定
- 文末タグがある場合は最優先で判定する（`#R`=送信用返信文 / `#A`=分析のみ / `#D`=下書き）。
- 相手文の貼り付けのみ: 分析のみ
- `返信文作って` など明示: 送信用文面
- 曖昧な場合: 分析のみ + 「必要なら送信用文面を作れます」

## 返信生成フロー
1. 購入者文の既出情報を抽出（環境/症状/時期/エラー/スタック）。
2. 先頭1行で受領 + 要点復唱（2点まで）。
3. **重複質問回避ゲート**を通す（下記）。
4. 追加質問の前に、`確認しました / 承知しました` を1文入れる。
5. 不足情報だけを最大2〜3点で依頼する。
6. 次回報告時刻を入れる（空白時間を作らない）。
7. 完了時刻を確定できない場合は、一次見解の目安を明記する。
   - `2時間以内` は「受領確認 or 方向性の一次報告」までを意味する。
   - 一次結果は別時刻で明記する（`48時間以内` または具体時刻）。
   - 見積りフェーズ既定文: `まず、2時間以内（JST）を目標に方向性をお返しします。`
   - 購入後フェーズ既定文: `まず、2時間以内（JST）を目標に次の確認結果をお返しします。`
   - 既定文: `他に必要なことがあれば、こちらからお聞きします。`
8. 速度不信対策として、検証観点または確認手順を1行入れる。
9. 送信用文面モードでは `/home/hr-hm/Project/work/返信文_latest.txt` に保存する。

## 重複質問回避ゲート（最優先）
- 購入者が `本番` と書いている場合、`発生環境は？` を再質問しない。
- 購入者が `500` などを書いている場合、`エラーは出ますか？` を再質問しない。
- 購入者が `Next.js/Stripe/Vercel` を書いている場合、`利用技術は？` を再質問しない。
- 購入者が `昨日から` を書いている場合、`いつから？` を再質問しない。
- `〜にならない` と期待する動きが暗示されている場合、再質問せず確認型にする。
  - 例: `決済後に有料へ切り替わる想定で合っていますか？`
- 既出情報を使って、次の切り分け質問へ進む。
  - 例: `全ユーザーか一部か`、`常時か断続的か`
- Stripe系は `Event ID（evt_...）` を任意で回収する（全件必須にはしない）。

## 日本語品質ルール（特化）
- 過剰敬語を避ける（`させていただきます` 連発禁止）。
- 1通1依頼を原則にする。
- 1文1意味で書く。長文の名詞列は分割する。
- `大丈夫です` の連発を避ける（1通あたり最大2回目安）。
- 定型の遠い表現を避ける。
  - `ご確認ありがとうございます`（文脈不一致時） -> `ご連絡ありがとうございます` / `解消されて何よりです`
  - `ご査収ください` -> `添付をご確認ください`
  - `善処します` -> `Aを確認し、必要ならBを修正します`
- 断るときは代替案を1つ添える。
- Stripeダッシュボードのメニュー名/ボタン名は表記揺れがあるため、断定せず「○○（または△△）」の補足を付ける。

## 文体自由化ポリシー（Conditional GO）
- 目的: 運用ルールを壊さず、AI感を減らし、自然で温かい文面にする。
- 方針: 「言い換えは自由、ただし禁止表現だけ弾く」。

### 3層線引き（変更可否）
- 完全固定（変更不可）: 価格、スコープ、範囲外、禁止事項、セキュリティ注記。
- 構造固定（変更不可）: 質問数、質問順、2時間+48時間、重複質問回避ゲート。
- 表現自由（変更可）: 書き出し、語尾、接続、文長、安心文の言い回し。

### Must（必須）
1) 完全固定・構造固定を崩さない。
2) 初回は最大3点の依頼に収める。
3) 文長リズムを混ぜる（短文→中文→短文）。
4) 同一返信内で同じ語尾を連続させすぎない。
5) `大丈夫です` は1通あたり最大2回。
6) 既出情報の聞き直しをしない。
7) 外部誘導・秘密情報要求をしない。
8) 範囲外を伝えるときは代替案を1つ添える。
9) 最後に次アクションと時刻目安を入れる。
10) 仕上げ前にBan表現ヒット0件を確認する。
11) 「見込みです + 別原因は事前相談」の説明は意味を保ったまま言い換え、連投定型にしない。
12) 質問は「1項目1情報」を原則にし、`AとB` の同時要求を避ける。
13) 購入後フェーズの返信は、必ず次回時刻目安（`本日中` / `[時刻]まで` / `2時間以内`）を1つ入れる。
14) 時刻案内の `2時間以内` と `48時間以内` の2段構成は、調査/判定フェーズで省略しない。
15) 完了確認フェーズ（お礼返信・A/B確認）では、2段構成を機械適用しない。
16) 書き出し（1文目）と末尾2行は、同一表現の連投を避けてローテーションする。

### Ban（禁止表現）
1) `ご共有ください`（`教えてください` / `共有ください`に置換）
2) `不足分は` / `不足分だけ` で始まる指摘調の文
3) `させていただきます` の多用（1通2回以上）
4) `対応範囲外です。` の単独使用（代替案なし）
5) `ご案内します` の単独終止（共同作業の言い回しへ置換）
6) `必要情報がそろい次第` の硬直使用（文脈に応じて平易化）
7) 過剰な名詞化（例: `確認事項の共有依頼` など）
8) 断定的な煽り（例: `必ず直ります`）
9) 外部連絡/外部共有/外部決済を想起させる語
10) 本番での再実行を促す語（再決済/再課金/Webhook再送など）
11) `対応していません。` の単独終止（柔らかい言い換え + 代替案に置換）
12) 出品者側からの `補償` / `損害補償` の先出し（`責任範囲` / `対応方針` に置換）

### 段階導入（実運用）
- Step 1: Must/Banを適用し、テンプレは「参考例」として扱う。
- Step 2: 実案件5件で試行し、追質問回数・返信率・冷たい印象の反応を記録する。
- Step 3: 4/5件以上で品質維持なら継続、2件以上で悪化なら直前版へロールバック。

- `まずは1件として進められそうです。別の原因が見つかった場合は、その時点でご相談します。`

- `ご連絡ありがとうございます。`
- `ご相談ありがとうございます。`
- `共有ありがとうございます。確認しました。`

- `この返信から2時間以内（JST）を目標に方向性をお返しします。`
  `そのあと、48時間以内を目標に一次結果をご案内します。`
- `2時間以内を目標に、まず次の確認結果をお返しします。`
  `続けて、48時間以内を目標に一次結果をご案内します。`
- `まず2時間以内（JST）を目標に判定の見立てを共有します。`
  `その後、48時間以内を目標に一次結果をご案内します。`

### 完了確認フェーズの末尾候補（二段構成を使わない）
- `ご返信いただければ、すぐに次の進行を確定します。`
- `A/Bどちらかご返信いただければ、このまま進行を確定します。`

## 速い修正への不信対策（遅延禁止）
- 意図的に遅らせない。
- 代わりに、次を見える化する。
1. 原因（1行）
2. 修正点（1行）
3. 確認手順（1〜3）
- 中間報告は「現状 / 次アクション / 次回報告時刻」の3点セットで送る。

## Stripe導線案内ルール（日本語UI）
- Stripe関連の確認依頼は `/home/hr-hm/Project/work/stripe日本語UI案内` を最優先参照する。
- 依頼時は同ファイルの「必須項目」を優先し、1通あたり2〜3点までに絞る。
- 同ファイルの「共有不要（`sk_...` / `whsec_...`）」に反する依頼をしない。
- 導線名に差異がある場合は、推測で断定せず「要実画面確認」を添える。

## 出力ルール
- 行頭に不要な空白を入れない。
- 基本は `です・ます`。
- 結論 -> 理由 -> 次アクション の順。
- 箇条書きは3点以内を優先。
- 見積り初回は相手負担を下げる（必要最小限の質問）。
- 見積り段階のスコープ表現は断定を避け、「対応できる見込みです」を優先する。
- 見積り初回の料金案内は、原則「基本料金 + 追加は事前相談」の2行までに抑える（詳細料金は判定後に案内）。
- 判定前の返信では、追加料金の具体金額（5,000円 / 15,000円）を先出ししない。
- 判定後の返信では、必要に応じて追加料金の具体金額を明記してよい。
- 非エンジニア初回は「スクショ中心・最大3点」にする。
- 非エンジニア初回で `evt_...` `pi_...` などID共有を必須化しない（必要時は2通目で依頼）。
- 非エンジニア向けでは技術語を多用しない。必要時は同文内で平易語を添える（例: `切り分け（原因を順に確認）`）。
- 非エンジニア向けでは `ログ` 単語を単独で要求しない（`エラーが出ている画面スクショ` を優先）。
- 時刻を聞くときは粒度を下げる（例: `昨日の午後` / `今朝10時頃`）。
- `あればお願いします` を単独で使わない（任意なら `あると助かります（なくてもOK）` と明記）。
- 「対応範囲外です」で終わらせず、必ず代替手段（別枠提案/納品形式/適用手順）を添える。
- 時刻未確定の案内は、見積りフェーズでは「方向性」、購入後フェーズでは「確認結果 / 判定結果」を使う。
- `2時間以内` の案内は、受領確認または一次報告を指す。一次結果は別時刻で書く。
- 購入後フェーズのうち、調査/判定フェーズは `2時間以内（次の確認結果）` + `48時間以内（一次結果）` の2行を原則セットで記載する。
- 完了確認フェーズ（お礼返信・A/B確認）は、48時間行を省略してよい。
- セキュリティ注意喚起は「禁止 -> 代替 -> 事後対処」の3点セットで書く。
  - 例: `値は送らないでください` -> `キー名のみで大丈夫です` -> `送信済みなら再発行して旧キーを無効化してください`
- 追加質問の末尾にフォールバック文を入れる（例: `分からなければ、困っている画面のスクショ1枚だけで大丈夫です`）。
- `ログ` を依頼するときは場所を明示する（`Vercel Function Logs` / `Stripeイベント詳細画面`）。
- 責任/保証の質問には「保証範囲 -> ただし（同一原因の追加調整）-> リスク低減手順」の3段構成で答える。
- 責任/保証の返信で出品者側から `補償` を先に使わず、`責任範囲` / `対応方針` で表現する。
- 誤った画面の共有を受けたときは、否定語を避けて「次はこの画面を1枚お願いします」で誘導する。
- 購入者の適切な行動（バックアップ取得、スクショ整理、必要情報の先出し）を確認できたら、1文で明示的に認める。
