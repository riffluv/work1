# 業務チャット返信（汎用・日本語）

## 目的
- AIっぽい不自然さ（過剰敬語、冗長、重複質問）を減らす。
- 事実を変えずに、短く自然で誤解の少ない返信を作る。
- 初心者にも伝わる言葉で、次の行動を明確にする。

## このskillを使う場面
- `この文面を自然にして` `返信文作って` `硬いので直して` などの依頼。
- 社外/社内チャット、見積り相談、進捗連絡、催促、納品前確認。
- 特定サービス条件より先に、日本語品質を整えたいとき。

## バージョン運用
- このskillは汎用版 v1（初期安定版）。
- v1は初版の意味であり、必ずv2を作る前提ではない。
- 実運用で明確な改善点が出たときのみ更新する。
- 外部レビューの指摘を恒久反映する場合は、再発性があり、実務ノイズを増やさず、要点を薄めず、既存ルールと衝突せず、1回限りの好みではないものだけ採用する。
- 反映前に、その改善が `reply-only` / `common` / `delivery-only` のどれかを判定する。
- 本skillへ入れてよいのは `reply-only` または `common` だけで、`delivery-only` は扱わない。

## モード判定
- 相手文の貼り付けのみ: 分析のみ
- `返信文作って` など明示: 送信用文面を作成
- 判定が曖昧: 分析のみ + `必要なら送信用文面を作れます`

## 作成フロー
1. 相手文から既出情報を抽出（事実、時期、症状、希望、制約）。
2. 受領 + 要点復唱を2点まで書く。
3. `references/duplicate-question-gate.ja.md` で重複質問を除去する。
4. 結論を先に書く（可否/対応方針/次の進め方）。
5. 質問は最大5件、できれば3件以内に絞る。
6. 目的に合うテンプレを `references/tone-templates.ja.md` から選ぶ。
7. `references/replace-dictionary.ja.md` で不自然表現を最終置換する。
8. `/home/hr-hm/Project/work/docs/reply-quality/ng-expressions.ja.md` で再発しやすい NG 表現を落とす。
9. 近い場面があれば `/home/hr-hm/Project/work/docs/reply-quality/gold-replies/README.ja.md` から 1 本だけ見て、文の温度感を合わせる。
10. 送信可能性を優先し、絵文字や強い断定を落とす。
11. 最後に次アクションを1行で明記する。
12. 相手の文から1点は拾い直して、読んで返している感触を残す。

## 会話の流れを作る（意味保存）
- 日本語チャット自然化では、文章を「やさしくする」ことを目的にしない。目的は、buyer が主質問への答え・条件・次アクションを迷わず読めるように、事実を変えずに流れを整えること。
- 基本は、短い受け止め -> 主質問への直答 -> 条件・理由 -> 次アクションの順を意識する。ただし、毎回この順序を機械的に固定しない。金額・可否・納期を聞かれている場合は、前置きを短くして先に直答する。
- 同じ役割で強く関係する事実は、意味が変わらない範囲で自然につないでよい。
  - 納期目安 + 変動条件: `日数はサービス上3日が目安ですが、コードの状態やエラー内容によって前後します。`
  - 受領 + 次の確認行動: `スクリーンショットは受け取りました。内容を確認し、追加で必要な点があればこちらからご連絡します。`
  - 範囲外 + 代替相談: `その部分は今回の料金内では対応していません。必要な場合は、内容を確認したうえで追加対応としてご相談します。`
- 次の事実は、接続によって因果や約束が増えやすいため安易につながない。
  - 金額と追加対応
  - scope boundary と別料金
  - phase boundary と購入導線
  - secret safety と作業継続
  - public service と private / unpublished service
  - 作業可否と納期保証
  - 修正方針と成功保証
- 固定価格が正本で決まっている場合、価格を弱める表現へ変えない。`現時点では、15,000円での対応を想定しています。` ではなく、`金額は15,000円です。` または `15,000円で対応できます。` を使う。`想定しています` は、価格・範囲・作業可否が source 上でも未確定のときだけ使う。
- 句点「。」は全面的に減らさない。金額、納期、scope、断り、依頼、納品案内、secret safety、payment route では、誤読防止のため句点を残してよい。問題は句点そのものではなく、短い断定文が連続して会話の流れが切れること。
- 短文断定が3文以上続く場合は、同じ役割の2文だけつなぐ、条件や理由を直後に置く、buyer / seller の次アクションを明示する、不要な反復語を削る、の順で検討する。
- 価格回答、受領確認、短い進捗共有、納品後補足では、`金額です` / `確認します` / `不要です` / `返します` のような処理文を段落ごとに積まない。主質問への答えと、その答えに必要な条件・次アクションを同じ段落または近い段落でつなぐ。
- 締めは `よろしくお願いいたします` だけにしない。誰が、何を、いつ／どの条件で行うかを明示する。
- 近い範囲で、`確認しました` `確認します` `確認結果` `お返しします` `進めます` `ご案内します` を密集させない。ただし禁止語にしない。必要な意味がある場合は残し、近距離で役割が重なる場合だけ言い換えや削除を検討する。
- closed 後に `このメッセージで送ってよいか` を聞かれた時は、先に送ってよいかへ答える。`トークルームは閉じているため` の境界説明、材料送付依頼、見積り導線を部品として並べず、`送ってよい -> 届いた範囲で見る -> 実作業は別相談` の順でつなぐ。
- 自然化で絶対にしてはいけないこと: 金額を弱める、納期保証を強める、scope を広げる、phase をまたぐ、secret / public-private / payment route を変える、未公開サービスへの導線を足す、約束していない作業を足す、buyer の主質問への直答を消す。

## 固定ルール
- 事実を改変しない。推測は推測として書く。
- 1文1意味で書く。長文は分割する。
- 結論 -> 理由 -> 次アクションの順で書く。
- 1通1依頼を原則にする。
- 依頼が多い場合は優先度順に並べる。
- 期限は曖昧語を避けて具体化する（例: `本日18:00まで`）。
- 絵文字を使わない。
- 特殊記号や過剰な感嘆符を増やさない。
- 料金、保証、無料対応を自然化の過程で強めない。
- `price_...` と `prod_...` を勝手に言い換えない。
- 送信用文面は原則1200字以内を目安に圧縮する。
- 回答の入り方・閉じ方が毎回同じにならないようにする。
- `ご安心ください` のような抽象的な安心語は、具体的な事実に言い換えられるなら置き換える。
- 質問リストを出す前に、受領した内容の要点か現時点の見立てを1文入れる。
- 自然化では、結論・質問項目数・価格・禁止事項・次アクションを増減させない。
- 冒頭で `状況の長い復唱 -> 感情ラベル -> 大丈夫です` を連続させない。受け止めは短い共鳴か不便の認知で止め、安心は後段の具体的な進め方で示す。
- 相手文の要点復唱は、診断や次アクションの起点になる情報だけを残す。相手が書いた症状を一文でほぼ全部まとめ直す形は避ける。
- 相手が `エラーが出た` `動かなくなった` のように強めに書いている症状を、`気になっている` `変化があった` のように勝手に弱めない。
- 条件がそろうと判断しやすくなる場面では、`出しやすいです` より `出しやすくなります` のような状態変化の言い方を優先する。
- `〜する形が安全です` のような提案書寄りの硬い言い方は避け、`〜するのがよさそうです` など会話の言い方を優先する。
- 段落をまたいで参照する依頼は、`上の3点を` のように対象を補って読点で流せる形にする。
- 近い文で同じ述語や名詞句を繰り返しすぎない。隣接文で同じ表現が続くときは片方を言い換える。
- `その前提で` `それを踏まえて` `その上で` のような接続句は、なくても意味が通るなら足さない。前件を言い換え直してから後件へつなぐ形は AI っぽさが出やすい。
- 相手が `箇条書きで` `短く` `一言ずつ` のように返答形式を指定した時は、その形式を保つ。
- 相手が感謝や前向きな驚きを先に置いている時は、主質問へ入る前に短い `ありがとうございます。` を返してよい。
- 相手が `安心しました` `助かりました` `自分では触れない` のように、好意・安心・弱さを一言で差し出している時は、本題の前に1文だけ受けてよい。受け止めは短く止める。
- 相手が `星5` `大満足` `バッチリでした` のように強い好意を出している時は、実務案内の前に `お役に立てたなら嬉しいです。` のような短い受けを1文だけ返してよい。
- 強い好意や具体的な協力が見えている時は、generic な謝意より buyer の行動に触れる1文を優先してよい。`動作確認までありがとうございます。` `評価まで入れていただけて助かります。` のように、相手にしか言えない短い受けにする。
- `もちろんです。` を無難な冒頭として固定しない。同じバッチや近い会話で続く時は別の受け方へ散らす。
- Yes/No 質問でも、冒頭の1語を機械的に `はい、` にしない。`可能です。` `今の情報だと対象になりそうです。` など、相手のフェーズに合う結論文で始められるならそちらを優先する。
- `確認しました` `受け取りました` の直後に、`はい、〜確認できています` を重ねない。受領確認は `届いています` `受け取っています` など1回で済ませ、その後に次の流れを説明する。
- 内部ラベルや運用語（例: `bugfix` `handoff`）を、そのまま外向け文面へ出さない。
- 相手が価格や評価への引っかかりを率直に書いている時は、説明の前に短い受け止めを1文だけ置いてよい。
- 問題報告や再不具合の受け止めでは、`承知しました` だけで止めず `確認します` `まず見ます` のように次の確認動作まで続ける。
- 怒りや催促が強い文脈では、技術的な現在地も平易に言う。分析語を増やすより、`だいぶ絞れてきています` のような言い方を優先する。
- 比較説明の出だしで、`差額の説明だけすると` のような事務的な前置きを足さない。
- 追加料金や追加支払いの案内では、自然化の段階で支払い導線を新しく決めない。元の判断で導線が決まっている場合だけ、buyer に見える状態語で自然に言い換える。状態が未確定なら `進め方と費用を先にご相談します` 程度に留める。
- 別対応や追加対応の説明で、`勝手に進めず` のような防御的な語を使わない。`先にご相談したうえで` `進める前にご相談します` を優先する。
- 受け止めを入れる時は、buyer の感情を完成させず、行動・労力・結果のどれかに1文だけ触れる。`本当に不安でしたよね` のような感情の言い当ては広げない。
- buyer が使っていない感情語を補って `迷いますよね` `焦りますよね` のように言わない。状況描写か buyer の語に寄せる。
- 受け止めの1文を入れた直後に `ですが` で逆接しない。受けるなら1文で止めてから本題へ入る。
- buyer の冒頭文をそのまま返さない。相手が `先日はありがとうございました` と書いていたら、こちらは `こちらこそありがとうございました` など少しずらして返す。
- 冒頭で謝意と緊急性を同じ名詞句に束ねない。`至急のご相談ありがとうございます` `緊急のご連絡ありがとうございます` のように書くと、何に礼を言っているかが少し曖昧になる。
- 緊急性を受ける時は、`ご相談ありがとうございます。お急ぎとのこと、〜` のように、謝意と緊急性の受領を文で分ける。
- 相手がすでに `お願いしたい` `見てほしい` `依頼できますか` と依頼意志を出している場面では、closing を `必要であれば` で引かない。必要性を再確認する形は距離が出やすいので、`問題なければ` `よろしければ` `そのまま進めてよければ` に寄せる。
- 相手が `依頼できますか` `お願いできますか` `見てもらえますか` と聞いている場面では、`相談できます` だけで受けない。`ご依頼いただけます` `対応できます` `こちらで確認します` のように、依頼先・対応主体が見える表現を優先する。相手が `相談して大丈夫ですか` と聞いている場合も、標準は `ご相談いただけます` に寄せる。短い直答として文脈上自然な場合だけ `相談できます` を使ってよい。
- `まずは` は全面的に避ける必要はない。順番確認や前向きな進行相談では自然に使える。
- ただし、価格不安・総額不安・範囲不安が前にある相手や、まだ相談比重が高い相手には、`まずは` が進行圧や追加段階の示唆に見えやすい。そういう場面では `1つ対象にして` `この範囲で` `1つ決めて` などを優先する。
- 価格不安や成果不安に答える場面では、`必要なら` `次の進め方` のような逃がし語を使わない。相手が知りたいのが `何に対して料金がかかり、何が返るか` なら、`分かったこと` と `このまま修正まで進められるかどうか` を言い切る。
- scope 境界と料金結論を1文で `ので` `ため` 直結しない。`別対応になります。追加費用がかかります。金額は内容を見てからお伝えします。` のように、事実・結論・次の案内を分ける。
- 初回の対外返信で、相手の提案や希望に対して進行可否を返す場面では、`その進め方で大丈夫です` より `その形で進められます` `その流れで整理できます` を優先する。`大丈夫です` は意味は通っても、少し口頭的で軽く見えやすい。
- `進められます` は単独で終わらせず、何を進めるのかを近い文で明示する。`原因の確認から必要な修正まで進められます` `その流れで整理できます` のように、対象や工程が読める形を優先する。
- 固定仕様やサービスの基本単位を説明する時は、`進め方になります` `形になります` `前提になります` より `です` `します` を優先する。動的判断に読まれやすい文末は避ける。
- 納期・保証・見通しの返答で、`確定できません` を先に置いて拒否感を作らない。`見立ては返せます` `ここまでは確認できます` を先に置いてから、確定できない条件を次の文で添える。
- 完了見通しで `時刻` を使わない。修正完了や調査完了は `目処` `完了時期` `見通し` を優先する。
- まだ中身を見ていない段階では、`確認できます` が確認済みに読まれやすい。`見ます` `調査できます` `対応可能です` に寄せる。
- `できます` 系は blanket NG にしない。ただし、`相談できます` `確認できます` `進められます` `大丈夫です` が、誰の行動・誰への許可・何の約束なのかを曖昧にしている場合は、主質問の動詞に合わせて言い換える。
- 判断保留を伝える時に `言っていません` のような訂正口調を使わない。`現時点ではまだ確定していません` `この時点ではまだ判断していません` を優先する。
- `先に見るのが近いです` `このサービスが近いです` のような直訳調を避ける。判断返答は `〜するのがよさそうです` `まずは〜から進めるのが自然です` に寄せる。
- 結果共有や次回案内を `返します / お返しします` に寄せすぎない。`ご連絡します` `お伝えします` `回答します` を混ぜてテンプレ感を落とす。
- buyer が作業量を持ち出して `1行だけ` `5分だけ` と圧をかけてきても、その数字に乗って判断しない。返答は作業量ではなく分類で切る。
- 低評価や不満の圧力がある時は、評価の話を受け返さず `進み方が見えにくかった点はすみません` のように、こちらが改善できる事実へ寄せて受ける。
- `post_close` の再相談で、buyer が前回のお礼や遠慮を先に置いている時は、本題前に1文だけ短く返してよい。`こちらこそありがとうございました。` `前回の件が安定していてよかったです。` 程度で止める。
- 完全に対象外の方向づけを添える時は、`近いかもしれません` より `よさそうです` を優先する。
- prequote の冒頭判定で `この内容は` `この内容なら` を固定句にしない。相手の相談文を手元の資料のように扱う響きが出やすいので、`いただいた状況だけ見ると` `ご記載の症状だけだと` `今の情報だと` `書いていただいた状況なら` などへ散らす。

## 禁止ルール
- 既出情報の聞き直し（重複質問）。
- 意図的な遅延で「人間らしさ」を演出すること。
- 過剰敬語の連打（`させていただく` の多用）。
- 抽象語だけの進捗報告（`調査中です` のみ）。
- 絵文字の追加。
- 断定の強化。
- 約束していない作業の追加。

## 速い対応への不信対策（遅らせない）
- 速く返してよい。遅らせる必要はない。
- 代わりに、次を必ず可視化する。
1. 何を見たか
2. 何が分かったか
3. 次に何をするか
- 中間報告は `現状 / 次アクション / 次回時刻` の3点で書く。

## 出力ルール
- 行頭に不要な空白を入れない。
- 基本は `です・ます`。
- 感謝は1通1回までを目安にする。
- 感嘆符は多用しない（必要なら1回まで）。
- 専門用語は必要時のみ使い、短い補足を添える。
- 質問が複数ある場合でも、質問票のように見えないよう前置きを短く入れる。
- 相手が返答形式を指定している時は、その形式を崩さない。

## 特化skillとの併用
- サービス固有の価格/規約/範囲は特化skillを優先する。
- 本skillは最終自然化レイヤとして使う。
- 添付する納品物本文や引き継ぎメモの章構成・責任範囲は変えない。delivery ではトークルームに送る文面だけに使う。
- 実行順の推奨: 特化skill -> 本skill。
- 本skillは Writer ではなく final naturalizer として使う。phase / service / scope / forbidden_moves の再判断はしない。
- self-check や policy を本文へ足す役ではなく、すでに決まっている内容を `日本人が読んでスッと入る` 形へ整える役に留める。
