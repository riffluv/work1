# 整合性ガード（破綻防止）

## 目的
返信文内の宣言と実体のズレをなくす。

## 出力前チェック（必須）
1. 項目数整合:
- `3点` `次の3点` `3点だけ` と書いたら、番号付き3項目が存在すること。
- `必須5点` と書いたら、番号付き5項目が存在すること。
- 見積り初回は `必須3項目 + 任意3項目` の順を守ること。
- 見積り初回の必須3項目は `期待する動作 -> 実際の不具合 -> 発生環境` の順で統一すること。
- 見積り初回で6項目すべてを必須扱いしないこと。

2. 再現手順整合:
- `操作の順番を1,2,3で` と書いた場合、同じ文面に `例: 1) ... 2) ... 3) ...` を入れること。
- `番号で` 単独表現を使わないこと。

3. 分岐整合:
- `再現不可なら...` と書いた場合、代替入力（直前操作/最後の発生日時）を明記すること。
- `ログがなければ...` と書いた場合、スクショまたは画面録画の代替手段を明記すること。

4. 環境ラベル整合:
- bugfix の buyer 向け環境確認では `公開中のサイトだけか / 自分のPCでも起きるか` を優先すること。
- `開発環境` を単独で使わないこと。

5. 指摘反映:
- 直前ターンでユーザーが指摘した矛盾点を、次の出力で必ず解消する。
- 解消できていない場合は、その理由を1行で明記してから本文を出す。

6. 禁止:
- 宣言だけして実体がない文（例: 「例を示します」だけで例がない）
- 項目数の自己矛盾（例: 「3点」なのに4項目）
- 1文に役割の違う情報を詰め込む文（対応内容 + 納品物 + 注意事項など）

7. 明瞭性チェック:
- 1文目が質問への直接回答になっていること。
- 「何をしてくれるか」と「何を渡すか」が分離されていること。
- 初見の読み手が3秒で要点を把握できる文長であること。

8. 支払い仕様整合:
- 見積り相談経由の取引で「有料オプション追加購入可」と書かないこと。
- サービス境界と支払い導線を混ぜないこと。`整理 -> 修正` 導線でも、整理と修正のスコープ・納品物・費用を分けて説明すること。
- 見積り前 / 購入前に、同じトークルーム内追加料金・おひねり・追加支払いを前提に進めるとは約束しないこと。
- 追加支払い導線を案内する場合は、取引状態が成立しているかを確認し、必要なら `進められる場合があります` のように条件付きで書くこと。
- クローズ後は旧トークルーム内の追加購入へ誘導せず、メッセージ上で確認できる範囲と新規依頼が必要な範囲を分けること。

9. 特例対応整合:
- 同一原因の軽微調整を吸収する文面では、`無料` `特例` `サービスで対応` を使わないこと。
- 同一原因として吸収する場合は、`同一原因の範囲` の根拠を1行で示すこと。
- 別原因の場合は、追加見積りへ切り替える文面（原因が別 + 金額 + 同意確認）を入れること。
- 同一原因でも長時間化しそうな場合は、吸収案内ではなく追加見積り案内を優先すること。

## ミスを見つけたときの修正ルール
- 言い訳を先に書かない。
- 修正版を先に全文提示する。
- 変更点は最後に1行で補足する。
