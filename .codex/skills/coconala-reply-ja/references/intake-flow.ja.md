# 不具合ヒアリング分岐（実務用）

## 目的
初回返信の離脱を防ぎつつ、実装に必要な情報へ段階的に到達する。

## フェーズ1（初回の最小3点）
まずは次の3点だけを回収する。
1. 期待する動作（本来どうなってほしいか）
2. 実際の不具合（今どうなっているか）
3. 発生環境（本番/プレビュー/ローカル）

回答例を1行添える。
- 例: 決済後はDB更新される想定だが、実際は500エラー。発生は本番。

分かる範囲で、次の3点は任意で依頼する。
1. いつ頃から発生したか
2. 再現手順
3. エラーログ/エラーメッセージ

## フェーズ2（実装前の必須5点）
実装着手前に、以下を必ず揃える。
1. 期待する挙動
2. 実際の挙動
3. 再現手順（操作の順番を1,2,3で）
4. エラーログ/エラーメッセージ
5. 技術スタック/ランタイム/デプロイ先

不足がある場合は、実装ではなく診断メモと追加依頼を返す。

## 条件分岐（if/then）
- if 再現可能:
  - then 操作順1,2,3を回収し、期待結果/実際結果を併記してもらう。
  - then 発生時刻は必須にしない。
- if 再現不可または単発:
  - then 発生直前の操作、最後に発生した日時、発生頻度を回収する。
  - then 必要時のみ時刻をログ突合に使う。
- if ログあり:
  - then エラーメッセージ全文（機密を伏せた形）を優先して回収する。
- if ログなし:
  - then スクショや10〜30秒の画面録画を任意で依頼する。
  - then 追加で環境情報を聞いて切り分ける。

## 初回メッセージの型
- 最短:
  - 3点（期待する動作 / 実際の不具合 / 発生環境）だけ依頼する。
- 標準:
  - 安心文 + 3点依頼 + 回答例 + 返答期限。
- 丁寧:
  - 安心文 + 3点依頼 + 「分かる範囲でOK」 + 返答期限。

## 禁止
- 専門用語だけで質問しない（prod/devのみ等）。
- 長文で質問を詰め込まない。
- 初回から必須5点を一度に要求しない。
- APIキーや秘密情報の値を要求しない。
- `開発環境` だけの曖昧表現で聞かない。
