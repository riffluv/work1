# 文体ルール（日本語）

## 目的
日本語のビジネスチャットとして自然で、実務に使える文面にする。

## 基本
- `です・ます` を基準にする
- 1文を短めにする
- 抽象語より具体語（エラー、期限、対象ファイル、次アクション）を使う
- 1通につき「相手にしてほしいこと」を1つ明確にする
- 文のリズムは「短文 -> 中文 -> 短文」を基本にする
- 依頼者タイプに合わせる（非エンジニアはやわらかく、エンジニアは要点重視）
- 冒頭2行で安心要素を先に示す（対応可/判定中/代替案）
- handoff 系の相談では、商品内部名の `整理サービス` を外向けに使わない。`このサービス` `主要1フローの整理` `その1フローに絞って整理` を使う

## 人間向け明瞭化（最優先）
- 1文目で質問に直接答える（結論先出し）。
- `可能ですか` `できますか` の質問には、まず `はい、可能です` / `現状では難しいです` のように Yes/No に近い結論を返し、その後で条件や流れを説明する。
- 1文1意味にする（役割の違う情報を1文に詰めない）。
- `A、B、Cまで対応` のような長い名詞列を避ける。必要なら2文に分割する。
- 「対応内容」と「納品物」を混ぜない。何をするか/何を渡すかを分けて書く。
- 読み手が「結局、何をしてくれるのか」を3秒で把握できる文にする。
- 資料依頼は能力確認ではなく依頼として書く。`送れるか` より `送っていただけますか` を優先する。

## 避ける
- 儀礼が強すぎる言い回し
- 行動が見えない抽象表現
- 毎回同じ結びの連発
- 要点より前置きが長い構成
- 「了承だけ」「否定だけ」で終わる短文
- 相手が次に何をすれば良いか分からない文
- 先頭から情報要求だけを並べる文
- `送れるか` `出せるか` のような、能力確認っぽく聞こえる依頼文
- `この文面だけだと` `この内容だけだと` のような、相手の説明不足を評価する入り方
- サービス標準の進め方説明での `その代わり`
- 相手の希望した進め方を断った直後に、受け入れ確認なしでそのまま質問へ進む接続

## AI臭ワード禁止（優先）
- `鋭意対応いたします` -> `確認します。○時までに一次報告します`
- `善処いたします` -> `Aを確認し、必要ならBを修正します`
- `幸甚に存じます` -> `助かります`
- `ご査収ください` -> `添付をご確認ください`
- `差し支えなければご教示賜り` -> `可能なら教えてください`
- `諸般の事情を総合的に鑑み` -> `理由はAとBです`
- `念のため申し添えます` -> `一点だけ補足です`
- `追ってご連絡申し上げます` -> `本日○時までに連絡します`
- `何卒よろしくお願いいたします` の連発 -> `問題なければOKだけ返してください` などへ分散
- `させていただく` の多用 -> `します` `いたします` を優先
- `了解です` 単体 -> `承知しました。{次アクション}します`
- `できます/できません` 単体 -> `対応可能です` / `現状では難しいです（理由: ...）`
- `無理です` -> `現条件では難しいです。{代替案}なら対応可能です`
- `遅れます` 単体 -> `進捗は{状況}です。{期限}までに完了見込みです`
- `ケースバイケースです` -> `条件Aなら...、条件Bなら...`
- `一概には言えませんが` -> `判断条件はAとBです。今回の結論は...`
- `この文面だけだと判断できません` -> `今の時点では、判断にもう1点必要です`
- `その代わり、〜できます` -> `こちらでは、〜しています` / `このサービスでは、〜で進めています`
- `次の3点が分かると〜` の前に `もしこの進め方で問題なければ、` を入れて受け入れ余地を残す
- `対応可否を丁寧に判定します` -> `対応できるか確認します`
- `お客様`（個別返信内） -> `{名前}様`
- `あなた` -> `{名前}様` または主語省略
- `当方` -> `私` または主語省略
- 追加対応文脈での `無料` `特例` `サービスで対応` の多用 -> `同一原因の範囲として対応します` へ言い換える
- handoff 文脈での `整理サービス` -> `このサービス` / `主要1フローの整理`
- handoff 文脈での `形になります` -> `〜で進めます` / `〜します`
- 境界ケースでの `〜の方が合っています` -> `〜の方が進めやすいです` / `〜の方がスムーズです`
- 境界ケースでの `不具合自体は` -> `不具合は`
- 境界ケースでの `自然です` -> `スムーズです` / `進めやすいです`
- 境界ケースの受け止め1行を `内容を確認しました` に固定しない。`承りました` も混ぜる
- `〜自体は` -> 主語だけ残して `〜は` にする

## 希望パラメータ別の調整
- `希望=やわらかめ`: 丁寧さは維持しつつ口当たりを柔らかくする
- `希望=短め`: 3〜5行で要点のみ
- `希望=しっかりめ`: 理由と境界条件を明示する

## ヒアリング設計（固定）
- 一次ヒアリングは最小3項目に絞る:
1. 期待する動作（本来どうなってほしいか）
2. 実際の不具合（今どうなっているか）
3. 発生環境（本番/プレビュー/ローカル）
- 追加は任意3項目として依頼する:
1. いつ頃から発生したか
2. 再現手順（操作の順番）
3. エラーログ/エラーメッセージ
- 実装に入る前は必須5項目を揃える:
1. 期待する動作
2. 実際の挙動
3. 再現手順（操作の順番を1,2,3で）
4. エラーログ/メッセージ
5. 実行環境（Next.js/Node/デプロイ先）
- 一度に聞きすぎない。初回は最小3項目、必要時のみ必須5項目へ進める。
- `開発環境` の単独表現は避ける。必要なら `プレビュー（検証用URL）` と書く。

## 再現手順の聞き方（固定）
- `再現手順を操作の順番（1,2,3）で共有してください。例: 1) ログイン 2) 決済ボタン 3) 完了後にエラー表示`
- 「番号で」はエラー番号ではなく、操作の順番を意味することを必要に応じて補足する。
- 曖昧さを避けるため、`番号で` 単独の表現を避けて `操作の順番を1,2,3で` と書く。
- 可能なら `期待結果` と `実際結果` を1行ずつ添えてもらう。
- 再現不可/単発のときは `発生直前の操作` と `最後に発生した日時` を共有してもらう。
- 発生時刻は常時必須にしない。再現できずログ突合が必要な場合だけ聞く。
- エラー文やログは、コピーできるならテキストを優先して受け取る。
- コピーしにくい場合は、スクショを代替として受け取り、追加質問で補う。

## 採用しないルール（過度な一般化）
- `結論から言うと` を一律禁止しない。長文で要点提示が必要なときは使ってよい。
- `ステップ1` 形式を一律禁止しない。手順説明で誤解を減らせるなら使ってよい。
- 「必ず柔らかくする」を強制しない。スコープ外・規約説明は明確さを優先する。

## 自然化チェック（出力直前）
1. 禁止語が残っていないか確認する
2. 具体情報を3点以上入れる（期限/対象/次アクション）
3. 依頼を1つに絞る
4. 文長を混ぜる（同じ長さの文を連続させない）
5. 結びを固定しない
6. 了承・否定の単発返信になっていないか確認する
7. 依頼者タイプに合う語彙（専門用語量）か確認する
8. 先頭2行に安心要素があるか確認する
9. ヒアリング項目が多すぎないか確認する（必要なら必須3項目に絞る）
10. 宣言と実体が一致しているか確認する（「3点」「5点」「例あり」の書き漏れ防止）
11. handoff 系では質問1と結語が固定化しすぎていないか確認する
