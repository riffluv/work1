# 見積り相談の経路別テンプレ（固定）

## 目的
サービス経由・プロフィール経由・メッセージ経由のどれでも、初回返信の品質をそろえる。

## 初回返信の共通ルール
- 見積り相談は、問い合わせフォームではなく軽いチャットに近い場として扱う。
- 先頭は安心文から始める。
- 初回は必ず質問を入れるのではなく、今ある情報で判断できるなら質問なしで進める。
- 見積り相談では、`お願い` より `理由提示` を優先する（例: `次の3点が分かると、見立てを出しやすいです`）。
- 追加情報は「分かる範囲で」として任意3項目で依頼する。
- 追加質問に入る前に、受領確認の1文（`確認しました` など）を入れる。
- 1つの質問項目に依頼は1つだけ置く（`AとB` を同時に聞かない）。
- 非エンジニア向けでは `ログ` 単語を単独要求しない。
- `情報` のような抽象語だけで聞かず、スクショ / 画面 / 注文番号 / テキスト貼り付け / いつ頃 など具体物で依頼する。
- エラー文やログは、コピーできるならテキスト貼り付けを優先する。
- エラー文がコピーしにくい場合だけ、画面スクショを代替として案内する。
- UI状態や見え方の確認では、画面スクショを優先してよい。
- 見積り相談では、まず価格判定や対応可否に必要な最小情報を優先し、いきなりZIP一式の提出を前提にしない。
- 見積り相談ではファイル添付自体は使えるが、購入前の標準運用として重いコード読解を前提にしない。
- ココナラ上の場を混同しない。購入前は見積り相談 / メッセージ、購入後はトークルーム。
- 購入前の返信では `トークルーム添付` を前提に書かず、必要なら `ファイル添付でお送りください` または `ご購入後にコード一式をお送りください` と書く。
- ZIP依頼は、相手がすでに `コード一式を送れる` と言っている場合か、主要1フロー整理の購入可能性が高く、最小情報だけでは判定しにくい場合に限って使う。
- 購入前にZIPを見る場合は、構成確認（フォルダ構造 / 技術スタック / 規模感 / 主要ファイル名の確認）までに止める。
- 購入前のZIP確認で、コードの中身の読解・原因特定・フロー整理までは行わない。
- handoff は、対象フローが具体的で受注確度が高いなら、購入前にZIP依頼を出してよい。
- bugfix は、症状・環境・エラー文で見積り判断できる限り、購入前にZIPを求めず、ご購入後にコード共有を案内する。
- 時刻を聞くときは、`昨日の午後` などの回答例を添える。
- 料金案内は初手で詰め込みすぎない（原則「基本料金 + 追加は事前相談」まで）。
- 判定前は追加料金の具体金額（5,000円 / 15,000円）を先出ししない。
- 回答例を1行入れて、相手が書きやすい状態を作る。
- 秘密情報（.env値、APIキー値）は、相手がコード・ログ・キー共有に触れている場面や秘密情報を送りそうな場面でだけ明記する。
- セキュリティ注意喚起は「禁止 -> 代替 -> 事後対処」の順で書く。
- 単純な断り文（`対応していません`）で終わらせず、柔らかい表現 + 代替案を添える。
- 送信文では「必須」「任意」の語は使わず、「まず3点」「分かる範囲で」で表現する。
- `まずは2時間以内` などの時間目安は、急ぎ案件・一次報告の約束・見立て返却の必要がある場面だけに使う。
- 高確度案件では、`あいさつ + 結論 + 購入導線` の短い構造で終えてよい。
- 初回返信は以下の2パターンから選ぶ（非エンジニア向け / エンジニア向け）。
- 非エンジニア初回は `evt_...` `pi_...` などのID共有を必須化しない。
- Stripe詳細手順は必要時のみ2通目で依頼する。

## 初回返信テンプレ（2パターン固定）
### A) 非エンジニア向け（標準）
`ご相談ありがとうございます。`
`技術的なことはこちらで確認しますね。`
`エラー文がコピーできればテキストで、難しければスクショでも大丈夫です。次の3つが分かると、見立てを出しやすいです。`
`1) 本来どうなってほしいか（短く1行でOK）`
`2) エラー文（コピーできればテキスト、難しければ問題画面のスクショ1枚）`
`3) 本番とテストどちらで起きているか（不明ならそのままでOK）`
`※ .envやAPIキーの「値」は送らないでください（キー名のみ共有可）。`

### B) エンジニア向け（要点重視）
`ありがとうございます。確認しますので、次の3点が分かると見立てを出しやすいです。`
`1) 本来の動き（こうなってほしい動作）`
`2) 今の動き（実際に起きていること）`
`3) 起きている環境（本番/プレビュー/ローカル）`
`（例: 本番でcheckout後に500エラーが出る。Stripeでは決済成功だがDBに反映されない）`
`加えて、以下もあると助かります。`
`- 発生開始時期`
`- 再現手順（1,2,3）`
`- エラーメッセージ（コピーできればテキスト、難しければスクショ）`
`※ .envやAPIキーの「値」は不要です（キー名のみ共有可）。`

## サービス経由（見積り相談ボタン）
`AまたはBの本文 + いただいた内容をもとに、{期限}までに範囲判定と対応方針（必要に応じて見積り案）をお返しします。`

## プロフィール経由 / メッセージ経由
`AまたはBの本文 + 確認後、{期限}までに進め方と対応方針（必要に応じて見積り案）をご案内します。`

## handoff 系の見積り相談（主要1フロー整理）
- 価格を出すときは `25,000円で主要1フローを整理できます` と断定で書く
- `整理サービス` という内部ラベルは使わず、`主要1フローの整理` と書く
- `〜する形になります` は使わず、`〜で進めます` `〜します` で言い切る
- 修正なしは必要なときだけ短く書く。長い防御説明にしない
- 相手がまだ flow を特定していない段階では、こちらから `決済 / 会員状態 / メール送信` に早く閉じない
- その段階では、まず `現状整理が優先か` `セキュリティ確認が優先か` `すでに明確な不具合があるか` のような広い切り口で聞く
- 選択肢型質問を使う場合は、相手が使った語を優先する。相手が言っていない語を固定テンプレとして先出ししすぎない
- handoff の同一バッチでは、少なくとも一部の返信で `修正は含めず` `まずはその1フローの現状整理に絞って進めます` などを入れ、修正依頼との混同を防ぐ
- handoff でも、相手がまだ価格や入口で迷っている段階では、ZIP一式送付を先に要求しすぎない
- ZIP依頼を出す場合は、相手が `送れる` と言っているか、購入後にコード一式をお送りいただく流れを明示する
- handoff でZIPを依頼するときは、`25,000円でどこまで整理できそうかを具体的に返しやすい` など理由を1文添える
- 対象フローが明確で、`25,000円で進められる` と即答できる高確度 handoff では、購入前にZIPを求めず、先に `見積もり提案を送ります` を案内する
- `高確度 handoff` の目安は、相手が対象フローを1つ具体的に指定していて、`お願いしたい` `発注したい` など依頼意思が明確な場合
- 相手が対象フロー・目的・資料提供可否まで十分具体的に書いている高確度 handoff では、確認質問を無理に足さず、そのまま購入導線へ進めてよい
- 確認質問は、相手がまだ決めていないことや、こちらの見積り判断に本当に必要なことだけに絞る
- その場合、コード共有は `ご購入後にソースコード一式をお送りください` と購入後の手順として書く
- 高確度 handoff では、`.env注意` や `2時間以内` の定型フッターを機械的に足さず、必要時だけ入れる
- 購入前の場で `トークルーム添付` を前提に書かない。購入後に開く場だと混同させない
- 相手が `原因が分かったらそのまま修正までお願いしたい` と聞いている場合は、まず `はい、可能です` を返し、外向けには `このトークルーム内で追加料金の修正として続けられます` を優先して案内する
- その場合も内部では、`25,000円は整理とメモ作成まで` `修正は15,000円で別スコープ` を維持する
- handoff から bugfix へ進む導線で、別サービスを買い直す印象を先に出さない
- 見積り相談では `次の2点だけお願いします` より `次の2点が分かると、どちらの入口がよいか判断しやすいです` を優先する
- handoff を段階的に進める説明では、`整理して` だけで終わらせず、`その分の引き継ぎメモを先にお渡しします` まで書く
- bugfix の見積り相談では、必要なら `ソースコードはご購入後にトークルームでお送りいただけます` と1文添え、購入前にコード一式が不要だと分かるようにする
- `25,000円で全部収まるか` を聞かれた場合、先に `全部対応できます` かどうかを返し、その後で `今回は25,000円に収まりにくい / 1フローごとの整理になる` を説明する
- 総額感を聞かれている場合は、`今回の範囲全体だと {総額目安}` のように全体目安も返す
- `25,000円には収まりにくい` だけで終わらせず、`1回で全部見るより1フローずつの方が整理の精度を保ちやすい` と品質理由を添える
- `どれにしますか` と迫る前に、`まず優先度の高い1フローから始めて、残りもこのまま続けてご相談いただけます` と継続の安心感を入れる
- 質問1は次から文脈に合うものを選び、同じバッチで固定しすぎない
  - `いちばん整理したい流れが ○○ で合っているか`
  - `今回まず見たい流れが ○○ で合っているか`
  - `メモ化したい流れが ○○ で合っているか`
  - `最初に整理したい部分が ○○ で合っているか`
  - `今回先に見ておきたい流れが ○○ で合っているか`
  - `まず優先したいのが、現状整理 / セキュリティ確認 / 明確な不具合の有無 のどれか`
  - `今は全体像の把握が先か、危ない箇所の確認が先か、すでに起きている不具合の確認が先か`
- 結語も次から回す
  - `まずは2時間以内を目安に、進め方をお返しします。`
  - `まずは2時間以内を目安に、整理の進め方をご案内します。`
  - `確認でき次第、2時間以内には対象範囲の見立てをお返しします。`

## 境界ケース（bugfix / handoff の振り分け）
- 1文目で質問に答えたあと、2文目か同文内で `内容を確認しました` `承りました` を入れ、聞いた上で判断していることを見せる
- 同一バッチで境界ケースの受け止め1行が `内容を確認しました` に固定しすぎない。`承りました` も混ぜる
- 推奨表現は `こちらから入る方が進めやすいです` / `この順の方がスムーズです`
- `〜の方が合っています` は使わない
- `見積りをご案内する形が自然です` のような判定語を使わず、`見積りをご案内します` と言い切る
- handoff を先に勧めるときも、必要ならその後に bugfix へ進める道は閉じない
- handoff を先に勧める場合、見積り中 / 取引中なら `必要ならこのまま追加料金で修正へ進めます` を優先し、手続き上の分断を先に見せない

## 経路不明（迷ったとき）
`A（非エンジニア向け）を優先して送る。`

## 非エンジニア向けの2通目（Stripe詳細が必要なとき）
`ご共有ありがとうございます。確認しました。`
`もう少しだけお願いしてもいいですか。`
`次の画面スクショをお願いします（IDはコピーしなくて大丈夫です）。`
`1) 開発者 > ワークベンチ > イベント（Events）の対象イベント詳細`
`2) 開発者 > ワークベンチ > Webhooks（イベントの送信先）の配信履歴`
`3) 失敗行の表示（HTTPステータス / エラー文が見える画面）`
`途中で迷ったら、そこまでの画面をスクショで送ってください。こちらで別の確認方法を探します。`

## 運用メモ（送信文には書かない）
- プロフィール経由 / メッセージ経由で提案する場合は、提案時に必ず「出品サービスに紐付ける」。
- 経路が不明でも、送信文は中立テンプレで先に進める。
- 見積り初回（資料未受領）で完了時刻を確定できない場合は、期限の代わりに「まず2時間以内には見立て」を案内する。
- 「2時間以内」は受領確認または一次報告の時刻として使い、一次結果は別時刻で明記する。
- 資料受領後は `この返信から` を使わず、`本日[時刻]までに` で時刻を明示する。
- 資料受領後に最初の見立てが30〜60分で出ない場合は、中間報告（`確認を進めています`）を1通入れる。
- 文言の使い分け: 見積りフェーズは「見立て」、購入後フェーズは「確認結果 / 判定結果」。
- 購入後フェーズでは、毎回「本日中 / [時刻]まで / 2時間以内（資料未受領時）」のいずれかを必ず入れる。
- 調査/判定フェーズは原則 `本日[時刻]まで` と `48時間以内` の2段構成を省略しない。
- 完了確認フェーズ（お礼返信・A/B確認）は二段構成を省略してよい。
- 責任/保証の質問では、出品者側から `補償` を先に使わず `責任範囲` / `対応方針` で書く。
- 送信前に `[ ]` / `{ }` のプレースホルダー残りを必ず確認する。
- 購入者の適切な行動（バックアップ取得、情報整理、テキスト共有やスクショ共有）は1文で認める。
- 追加質問の末尾にフォールバック文を入れる（例: `コピーしにくければスクショ1枚で大丈夫です`）。
- 書き出しと末尾2行は連投固定を避ける（同一表現を3件連続で使わない）。
