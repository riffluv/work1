# ココナラ返信（日本語）

## この skill を使う場面
- ユーザーが「なんて返す？」「返信文作って」「返信相談」と依頼したとき
- ココナラ上でお客さんに送る日本語メッセージを作るとき
- AIっぽさを抑えて、短く自然な文面にしたいとき

## モード判定（重要）
- 相手文の貼り付けのみ（または質問のみ）: **分析のみ**を返す
- 「返信文作って」「返信書いて」「送信用で」「そのまま送れる文」などの明示あり: **送信用文面**を返す
- 「分析だけ」「分析のみ」明示: **分析のみ**を返す
- `#P` がある場合: **購入後の途中報告**を優先して作る（完了報告ではなく進捗共有）
- 判定が曖昧な場合: まず分析のみを返し、必要なら最後に「返信文も作れます」と1行添える

## 想定入力フォーマット
どちらでも受け付ける。足りない項目は安全側で補完する。

1行:
`返信相談: 相手文=... / 相談点=... / 希望=...`

複数行:
`返信相談`
`相手文: ...`
`相談点: ...`
`希望: ...`

## 分析のみの出力形式
- 要点（相手の主張）
- 懸念（技術/運用/規約）
- 追加で確認すべき項目（2〜4個）
- 推奨対応（次アクション）

## 返信文作成フロー
1. `相手文` `相談点` `希望` を抽出する。
2. 返信の目的を先に決める（回答 / 追加質問 / スコープ調整 / クローズ案内）。
   - `#P` がある場合は、目的を「購入後の途中報告」に固定する。
3. 相談経路を推定する（サービス経由 / プロフィール経由 / メッセージ経由 / 不明）。
4. 依頼者タイプを推定する（非エンジニア / エンジニア）。判定不能なら非エンジニア向けで作る。
5. 返信先頭に安心文を入れる（対応可 / 判定中 / 範囲外のいずれか）。いきなり要求で始めない。
6. 見積り相談の初回は「必須3項目 + 任意3項目」で回収する。最初から6項目必須にしない。
7. 実装着手前は必須5点（期待/実際/再現/ログ/環境）を揃える。
8. `references/message-patterns.ja.md` から目的と依頼者タイプに合う型を選ぶ。見積り初回は `references/estimate-reply-flow.ja.md` の経路別テンプレを優先する。
9. そのまま貼れる本文を1本作る。
   - `#P` は原則3行で作る。
   - 1行目: `進捗の共有です。`
   - 2行目: `どこまでは正常で、現在どこを見ているか`
   - 3行目: `次にやること` と `次回連絡目安` または `必要情報`
10. `references/style-rules.ja.md` の禁止語・言い換えルールを適用して下書きを整える。
11. 送信用文面は、毎回 `japanese-chat-natural-ja` で最終自然化する。
12. `references/consistency-guard.ja.md` で整合性チェックを実施する（項目数、番号例、矛盾語の有無）。
13. 支払い・仕様説明がある場合は、公式仕様と矛盾しないか確認する（見積り経由オプション不可、正式納品後はおひねり、クローズ後は新規依頼）。
14. 購入直後の初回連絡は24時間以内（通常数時間以内）を目安に送る（返信遅延リスク回避）。
15. 返信文モードでは、本文を `/home/hr-hm/Project/work/runtime/replies/latest.txt` に保存する。相手文が明確なときは `/home/hr-hm/Project/work/runtime/replies/latest-source.txt` にも保存する。
16. リスクやスコープ変更がある場合は、次アクションを1行で明記する。
17. ユーザー指定がない限り短めにまとめる。

## 出力ルール
- 行頭に不要なスペースを入れない。
- ユーザー指定がない限り、コードブロックを使わない。
- `です・ます` を基本にしつつ、チャットのテンポを維持する。
- 結論 → 理由 → 次アクション/依頼 の順で書く。
- 文のリズムは「短文 → 中文 → 短文」を基本にする。
- `本日中` `再現手順` `エラーログ` など具体語を優先する。
- 抽象的でAIっぽい定型文を避ける。
- 禁止語が出たら、必ず具体語へ言い換える。
- `対応可否を丁寧に判定します` のような硬い定型を避ける。`対応できるか確認します` を優先する。
- `〜する形が安全です` のような提案書寄りの言い回しは避ける。会話では `〜するのがよさそうです` を優先する。
- 条件がそろうと判断しやすくなる場面では、`出しやすいです` より `出しやすくなります` を優先する。
- 前段で挙げた項目を受ける依頼は、`上の3点を` のように対象を明示して省略しすぎない。
- 再現手順を依頼するときは、必ず1行の記入例を添える。
- `再現手順（番号で）` だけの表現は使わない。必ず `操作の順番を1,2,3で` と書く。
- `最小3点` と書く場合は、本文に番号付き3項目を必ず出す。
- `必須5点` と書く場合は、本文に番号付き5項目を必ず出す。
- `例` を入れると宣言した場合は、同一メッセージ内に `例: 1) ... 2) ... 3) ...` を必ず置く。
- 直前ターンでユーザーが修正指摘した点は、次の出力で最優先で反映する。
- 最終自然化では、結論・質問項目数・価格・禁止事項・次アクションを増減させない。

## ヒアリング順序（固定）
- 見積り初回（離脱を防ぐ必須3項目）:
1. 期待する動作（どうなってほしいか）
2. 実際の不具合
3. 発生環境（本番/プレビュー/ローカル）
- 見積り初回の任意3項目（分かる範囲で依頼）:
1. いつ頃から発生したか（最後に正常だった時期）
2. 再現手順
3. エラーログ/エラーメッセージ
- 実装着手前の必須5点:
1. 期待する挙動
2. 実際の挙動
3. 再現手順（操作の順番を1,2,3で）
4. エラーログ/エラーメッセージ
5. 技術スタック/ランタイム/デプロイ先
- 初回見積りでは「必須3 + 任意3」の順を崩さない。
- 実装着手前に必須5点が揃わない場合は、実装着手ではなく診断レポート案内に切り替える。
- 価格判定や対応可否が主質問のときは、見積り相談段階でZIP一式の重い解析を前提にしない。
- 購入前は最小情報で見立てを返し、ZIP依頼は相手が送付可能と言っている場合や、購入後の受領前提が明確な場合だけに絞る。

## 環境ラベルの固定ルール
- 環境の聞き方は `本番 / プレビュー / ローカル` を標準にする。
- `開発環境` は曖昧語なので、単独では使わない。
- 補足が必要な場合のみ、`プレビュー（検証用URL）` と書く。

## 再現手順の回収方式（固定）
- 相手にとって負担が少なく、こちらが切り分けしやすい形を優先する。
- 再現可能な場合は、次の最小テンプレで回収する。
1. 何をしようとしたか（期待）
2. 実際にどうなったか（結果）
3. 操作の順番（1,2,3）
- 再現不可/単発エラーの場合は、番号入力を強制しない。`発生直前の操作` と `最後に発生した日時` の共有に切り替える。
- 発生時刻は常時必須にしない。再現不可でログ突合が必要なときだけ聞く。
- 画像や動画がある場合は任意で添付を依頼してよい（10〜30秒の短い録画で可）。

## 調査結果の採用ルール
- 外部調査の結果はそのまま採用しない。`具体性` `運用整合性` `再現性` の3点で採否を決める。
- ココナラ公式仕様と固定条件（15,000円モデル、範囲外条件）に合う要素だけ採用する。
- 不採用例: 断定的すぎる文体禁止（例: 「結論から言うと」を常にNGにする、番号付き手順を常にNGにする）。
- 採用例: 抽象語の削減、具体情報（期限/対象/次アクション）の明記、1通1依頼、依頼者タイプ別テンプレ。

## 初回でも固くしない既定値
- `希望` が未指定でも、硬すぎる敬語を使わない。
- 「許可が必要な文脈」以外で `させていただく` を使わない。
- 依頼文は「1通1依頼」に絞る。
- 結びを毎回固定しない（同じ結語の連発を避ける）。

## 安心文の既定テンプレ
- 対応可能判定: `内容を確認しました。対応可能です。`
- 判定中: `内容ありがとうございます。まず原因の切り分けから進めます。`
- 範囲外だが提案あり: `ありがとうございます。現行範囲外ですが、進め方を2案で提案できます。`

## ココナラ運用の禁止ガード
- 外部連絡手段（LINE/Discord/Zoom/メール）へ誘導しない。
- 外部共有（GitHub/Drive/外部URL）へ誘導しない。
- 購入者から外部共有（GitHub/Drive/URL共有）を提案されても、原則はトークルーム添付へ戻して案内する。
- ファイル共有はまずトークルーム添付機能を案内し、容量や件数は実画面の表示に従う。
- `.env` 値、APIキー値、Webhook秘密値など生の秘密情報を要求しない。
- スコープ外は、追加対応として明確に切り分ける。
- 見積り中 / 取引中に `整理 -> 原因判明 -> 修正` へ進む場合は、外向けには同じトークルーム内で追加料金対応へ続けられるように案内する。
- ただし内部では `整理` と `修正` の価格・納品物・スコープを分けて扱う。

## 支払い仕様ガード（公式準拠）
- 見積り相談経由の取引で「有料オプション追加購入できます」と書かない。
- 見積り相談では、ファイル添付自体は可能でも、購入前からコード一式の重い解析を標準前提にしない。
- 購入前にZIPを求める場合は、その必要性が高いことを示し、必要なければ最小情報で先に見立てを返す。
- 見積り中 / 取引中の追加修正では、外向けに `別サービスを買い直してください` を先頭に出さない。まず `このまま追加料金で続けられます` を案内し、その後で手続きと境界を説明する。
- 正式納品後の追加費用案内は、クローズ前なら「おひねり」で案内する。
- クローズ後の追加対応は「新規依頼で対応」と案内する。
- 差し戻しは原則1回であることを前提に、納品メッセージには確認期限（72時間）を添える。

## 特例対応ポリシー（内部運用）
- 基本料金15,000円の値引きはしない。
- 特例は「同一原因の軽微な追加調整」のみを対象にする。
- 別原因/別フロー/別エンドポイントは追加見積りに切り替える。
- 特例の固定数値上限は設けない（案件ごとに判断する）。
- 特例を使った場合は、案件メモに理由と追加時間を記録する。
- 対外文面で `無料` `特例` `サービスで対応` を安易に使わない。
- 表現は `同一原因の範囲として対応` を優先する。
- 詳細は `docs/coconala-special-case-policy.ja.md` を参照する。

## 仕上げ前チェック
- 相手の質問に正面から答えているか
- 双方の次アクションが明確か
- 禁止事項（外部誘導/秘密情報要求）を含んでいないか
- 読んで不自然な過剰敬語になっていないか
- `references/style-rules.ja.md` の禁止語が残っていないか
- 具体情報（日付/対象/次アクション）が3点以上入っているか
