# Codex実運用ガイド

このガイドは、起動後の実務だけを最短で回すためのものです。  
起動の正本は [次セッション用_起動プロンプト.txt](/home/hr-hm/Project/work/次セッション用_起動プロンプト.txt) を使います。

## 最初に覚えること
- あなたが送る基本入力は `場 + タグ + 相手文`
- 購入前の返信は `coconala` mode
- 購入後に `#S` を打ったら、主モードは `implementation`
- 納品準備に入る時だけ `delivery`
- 実装中は返信系 docs を前面に置かない
- 送信用の外向け文面は、毎回日本語自然化してから返す

## 最小セット
- `#R`: 送信用返信文
- `#A`: 分析のみ
- `#D`: 下書き
- `#P`: 購入後トークルームの途中報告
- `#S`: 案件記録開始
- `#M`: 重要判断の保存
- `#C`: クローズ保存

## 補助ショートカット
- `#CL`: 直近の送信用返信文を対象に、Claude向け監査プロンプトを作る
- `#GE`: 直近の送信用返信文を対象に、Gemini向け監査プロンプトを作る

## これだけで回る

```text
サービス経由の見積り相談です
#R
相手文: ...
```

### 2. まず分析だけ見たい
```text
サービス経由の見積り相談です
#A
相手文: ...
```

### 3. 購入直後に始める
```text

購入後のトークルームです
相手文: ...
```

### 4. 購入後の途中報告
```text
購入後のトークルームです
#P

```

### 5. 重要判断を残す
```text
#M
決めたこと: 同一原因で進める
根拠: webhook署名検証で失敗している
次アクション: 修正差分を固める
```

### 6. クローズ保存
```text
#C
最終結果: Stripe Webhook署名エラーを修正して解消
補足: 同一原因の範囲で完了
```

### 7. 直近の返信文を Claude に見せたい
```text

補足: AI感がないかだけ見て
```

補足:
- 対象は `runtime/replies/latest-source.txt` と `runtime/replies/latest.txt`
- 補足があれば、その観点を監査プロンプトへ足す

### 8. 直近の返信文を Gemini に見せたい
```text
#GE
補足: 技術者向けとして不自然でないかを見て
```

## 場の書き方
- `サービス経由の見積り相談です`
- `プロフィール経由の見積り相談です`
- `メッセージでの相談です`
- `購入後のトークルームです`

英語タグや route 名を覚える必要はありません。  
日本語で場が分かれば十分です。

## mode の考え方
### coconala
- 見積り相談
- 購入前後の返信
- 公開中サービスの案内

### implementation
- 購入後のコード分析
- 修正
- 検証
- case README 更新

### delivery
- 納品物作成
- 正式納品前確認
- 正式納品案内
- クローズ前後

補足:
- `#S` を使うと内部で case が開始され、`implementation` が主モードになります
- 納品準備に入る段階では、Codex が `delivery` へ切り替えて進めます

## ユーザー負担を減らすコツ
- 相手文は要約せず、そのまま貼る
- 送信用が欲しい時だけ `#R`
- 途中報告は `#P` を使う
- 場を1行書く
- 補足は1行までにする

補足例:
```text
サービス経由の見積り相談です
#R
相手文: ...
補足: 料金内に収まるかを気にしていそうです
```

## `#P` の使い方
`#P` は購入後の途中報告専用です。  
骨格は次の3行に固定します。

- `進捗の共有です。`
- `どこまでは正常で、現在どこを見ているか`
- `次にやること` と `次回連絡目安` または `必要情報`

注意:
- 完了報告や納品案内には使わない
- 未確認の原因断定やログ生貼りを入れない
- 旧 `#R 進捗返信` は互換用で、新規運用では `#P` を使う
- `#P` も送信用文面なので、毎回自然化してから返す

## 実装中の考え方
- 主に見るのは `ops/cases/open/{case_id}/README.md` とコードとログ
- 返信テンプレは必要になった時だけ呼ぶ
- 実装判断は case README に寄せる
- 価格や未公開サービス情報を実装文脈に混ぜない

## フォルダ名の見方
- `os/` は `基本設定`
- `ops/` は `案件管理`
- `runtime/` は `いまの状態`

実際のフォルダ名は英語のままですが、運用上はこの日本語の意味で見れば大丈夫です。

## よくある事故
- `#R` を付けずに送って、分析だけ返ってくる
- `#R 補足` のようにタグの後ろに修飾語を付けてしまう
- 購入前なのに `トークルーム` 前提の文面を作らせる
- `#P` ではなく `#R` で途中報告を回してしまう
- `#M` を返信用タグだと勘違いする
- 場を書かずに、購入前 / 購入後を取り違える

## 困った時
- 今の状態を見たい時: `./scripts/internal-os-status.sh`
- OS 整合確認: `./scripts/os-check.sh`
- 内部OSの通し確認: `./scripts/check-internal-os-flows.sh`

## 一番大事なこと
迷ったら、まず `場 + #R か #A + 相手文` だけで十分です。  
購入後に入ったら、以後は `case README + コード + ログ` を主文脈にして進めます。
