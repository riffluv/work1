# AGENTS.md（フリーランス作業環境）

## ミッション
このワークスペースは、有償クライアント案件をテキストのみで進めるための作業環境です。  
主な提供内容は **Next.js / Stripe / API連携の不具合診断と修正（48時間目標）** です。  
強みは **Stripe Webhook 問題（署名エラー、イベント不達、重複処理）** です。

## 重要方針
- 事前知識より、実データ（ログ・コード・再現手順）を優先して判断する。
- 修正提案の前に、依頼者の証跡（ログ、コード、環境変数名、再現手順）を必ず確認する。
- スコープは固定し、意図しない拡張を避ける。

## サービスモデル
- 基本料金: 15,000円（不具合1件の原因切り分け〜修正差分/適用手順）
- 有料オプション: 追加修正1件（15,000円）
- 有料オプション: 追加調査30分（5,000円）

## 厳守する境界
- 通話・ビデオ会議は行わない。
- 契約に明記がない限り、依頼者リポジトリへの直接 push は行わない。
- 契約に明記がない限り、本番デプロイは行わない。
- チャットで生の秘密情報は扱わない。マスク済み値またはキー名のみ受け取る。
- 法務・税務の助言は行わない。

## 作業ルール
1. 初動（見積り・初回返信）は必須3点を回収する:
   - 本来の動き（期待挙動）
   - 実際の不具合
   - 発生環境（本番 / プレビュー / ローカル）
2. 実装前は必須5点を回収する:
   - 期待される挙動
   - 実際の挙動
   - 再現手順
   - エラーログ
   - 技術スタック / ランタイム / デプロイ先
3. 受領情報で再現できない場合:
   - 診断レポートと、追加で必要な情報を具体的に渡す。
4. 変更は常に可逆に保つ:
   - 小さなコミット
   - 明確な変更メモ
5. 納品は証拠付きで行う:
   - 何が壊れていたか
   - 何を変更したか
   - どう検証するか
6. 最終確認フロー:
   - 最終の実機確認は依頼者環境で実施してもらい、結果ログを受けて追加調整する。

## コミュニケーション方針
- テキストのみ、簡潔、タイムスタンプ付きで更新する。
- 実装前にスコープを確認する。
- 新要件が出た場合は一旦停止し、スコープ変更通知を出す。
- **文章ガイドライン必読**: お客さん向け文章は `docs/writing-guideline.ja.md` に従う。
- 返信文の優先順位は常に次の順にする:
  1. 相手への正確な応答
  2. スコープ・価格・規約の維持
  3. そのうえで自然な日本語
- 日本語を自然にするために、相手が言っていない論点をこちらから先回りして増やさない。
- 選択肢型質問を使う場合は、テンプレ固定語より相手の語彙を優先する。
- ただし、診断上必要な質問を出すこと自体は先回りではない。相手が言っていない論点をテンプレ都合で増やすことだけを避ける。
- 初手返信では自然さと安心感を優先してよいが、2通目以降は判定と次アクションを前に出す。
- 2通目以降で `可能性があります` を繰り返さず、証拠が増えた段階で結論を一段前へ出す。
- 感情応答は初手中心とし、継続往復では要件・判定・次アクションを優先する。
- `形になります` は routing / 別件導線 / 新規依頼案内で再発しやすいので、送信前に抜き打ち確認する。
- 見積り相談では `まずは次の3点だけお願いします` を機械的に使わず、`次の3点が分かると、見立てを出しやすいです` のような理由提示型を優先する。
- 見積り相談の質問数は 0〜3点の範囲で、判断に必要な分だけにする。3点は上限であり、毎回聞く標準ではない。
- 見積り相談はフォームではなく軽いチャットに近い場として扱い、返信構造を毎回同じ5段に固定しない。
- `.envやAPIキーの値は不要です` や `2時間以内に見立てを返します` は、必要な場面だけに入れる。全件の定型フッターにしない。
- 購入後や進行中の確認では、`次の3点だけお願いします` のような依頼型でよい。
- `主要1フロー整理 -> 原因判明 -> 修正` の流れは、見積り中 / 取引中であれば外向けには同じトークルーム内で続けられる導線を優先する。
- ただし内部では、`25,000円の整理` と `15,000円の修正` の価格・納品物・スコープを必ず分けて管理する。
- `別サービスを買い直してください` ではなく、`まず整理から入り、必要ならこのトークルーム内で追加料金の修正として続けられます` を優先表現にする。
- handoff で段階的に進める説明をするときは、`整理して` だけで終わらせず、`引き継ぎメモとして先にお渡しします` など納品物まで明示する。
- 返信判断に迷ったら、`docs/coconala-reply-principles.ja.md` の5行原則 -> `docs/coconala-reply-judgment-flow.ja.md` の順で判断してから、個別ルールを見る。
- ココナラ上の場を混同しない。
  - 購入前: 見積り相談 / メッセージ
  - 購入後: トークルーム
- 購入前の返信で、まだ開いていない場を前提に `トークルーム添付で送ってください` と書かない。
- 購入前の添付依頼は `ファイル添付でお送りください` `ご購入後にコード一式をお送りください` のように、時点に合う表現にする。
- 高確度 handoff で、対象フローが明確かつ 25,000円 で進められると即答できる場合は、ZIP依頼より先に `見積もり提案を送ります` を案内する。
- handoff で購入前にZIPを受け取る場合も、構成確認（フォルダ構造 / 技術スタック / 規模感 / 主要ファイル名）までに止め、コード読解・原因特定・フロー整理は購入後に行う。

## コンテキスト劣化の予防警告ルール
- ターン数による強制セッション切替は行わない。原則は継続運用とする。
- 次の兆候が出た場合のみ、劣化リスクとして事前警告を出す。
  - 質問と回答のズレ
  - 同じ修正のやり直し
  - 直前に合意した方針と矛盾する編集
- 兆候が1回出たら「注意喚起のみ」を行い、作業は継続する。
- 同じ兆候が2回連続で出たら、新規セッション切替を提案する。
- 重大な編集の前には「何を変更するか / どのファイルか / なぜ必要か」を短く再確認してから実行する。
- 警告は「作業後」ではなく、誤作業の前に出す。

## コードコメント方針（デフォルト）

## Codex / Claude 分業ルール
- Codex が実装とコメントの初版を作成する。
- Claude は文章とコメントの自然さをレビューする（日本語品質担当）。
- 最終判断と反映は Codex が行う（主軸は Codex）。

## 出力形式方針（コピペ安全）
- お客さん向け文面やコピペ用途の文面は、**各行の行頭に不要なスペースを入れない**。
- 新しい Codex セッションでも、このワークスペースでは同ルールを維持する。
- 既存テキスト整形が必要な場合は `文章コピー.txt` の手順（`cb`）を使う。
- 返信運用の文末タグは `#R`（送信用返信文）/ `#A`（分析のみ）/ `#D`（下書き）/ `#P`（購入後の途中報告）を使い、タグ指定がある場合は最優先で従う。
- `#P` は購入後トークルームの途中報告専用とし、完了報告や納品案内には使わない。
- `#P` の骨格は原則3行にする。
- 1行目: `進捗の共有です。`
- 2行目: `どこまでは正常で、現在どこを見ているか`
- 3行目: `次にやること` と `次回連絡目安` または `必要情報`
- `#P` では、未確認の原因断定 / 進捗率 / ログ生貼りを避ける。
- `#P` の具体性は「どこまでは正常で、どこから先を見ているか」の境界で出す。
- `claudeに聞いてみて` / `geminiに聞いてみて` / `chatgptに聞いてみて` は、直前にまとまった案を対象に「別AIへ投げる監査用プロンプト作成」のトリガーとして扱う。
- `両方に聞いてみて` の場合は Claude 用と Gemini 用を作成し、同一プロンプトで十分なら同じ内容、必要なら別プロンプトで出す。
- 対象が未指定の場合は「直前のCodex最終出力1件」を監査対象として扱う。
- 上記トリガーの後に追記された条件（例: 返信文の言い回し、特定箇所の違和感）は最優先で反映する。
- 返信文作成時（「返信文作って」「返信書いて」）は、同じ文面を `/home/hr-hm/Project/work/返信文_latest.txt` に保存する。
- `可能ですか` `できますか` のような Yes/No 質問には、手続き説明より先に結論を返す。`はい、可能です` / `現状では難しいです` を先に置く。
- 案件記録ショートカット:
  - `#S`: 購入確定後の案件記録開始。`ops/cases/open/` に案件メモを作る。
  - `#M`: 重要判断の中間保存。open案件メモへ追記する。
  - `#C`: クローズ保存。案件要約を確定し、`ops/cases/closed/` と `ops/case-log.csv` を更新する。
- `#S / #M / #C` は会話ログを丸ごと保存する指示ではなく、判断根拠を圧縮して残す指示として扱う。
