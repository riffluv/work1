今回特に見てほしいこと:

1. 日本語ビジネスチャットとして自然か
- 受付票・FAQ・規約説明のように距離が出ていないか。
- 症状を長く再構成しすぎて、AIが入力を整形している感じが出ていないか。
- `できます` `大丈夫です` `こちらで確認します` `その時点で` `この場合は` `ただ` などが、文脈上自然か。
- 商売として前のめりすぎないか。
- 逆に、曖昧すぎて buyer が次に何をすればよいか分からない箇所はないか。

2. 主質問に正面から答えているか
- buyer が次に決めたいことへ先に答えているか。
- 最後の文だけを機械的に主質問扱いしていないか。
- 複数質問で、価格・納期・保証・材料共有・追加料金不安が落ちていないか。

3. 返信OSコアとしての再発構造があるか
- 単発の言い換えではなく、汎用コアに戻すべき構造があるか。
- 逆に、ココナラ adapter に閉じ込めるべき表現を core に入れていないか。
- style setting / user customization に逃がすべき好みを、hard rule 化しようとしていないか。

4. 安全境界が崩れていないか
- 価格、scope、phase、secret、payment route、closed 後境界。
- 支払い前作業開始。
- GitHub / 外部作業面化。
- 成功保証、返金保証、無料対応断定。
- 修正できない場合の説明が曖昧すぎないか。

5. この20件から次に何をすべきか
- #R 人間監査を続けるべきか。
- Pro 構造監査をさらに増やすべきか。
- #RE regression に戻すべきケースはどれか。
- gold / skill / lens / lint / adapter / style / case_fix / preference / no_change の分類をしてほしい。

出力で必ず分類してほしい return_target:
- hard_guard: public/private, phase, secret, external share, direct push, prod deploy, closed after work, refund/free/success guarantee などの deterministic 事故
- soft_lens: 再発する構造だが文脈依存
- gold: 近い判断順序・温度感の anchor として残す
- skill: writer が毎回迷う入力処理として入れる
- lint: 表面形だけで高確度に拾えるもの
- adapter: ココナラや bugfix-15000 専用に閉じ込めるべきもの
- style: 将来ユーザー設定や文体設定に逃がすべきもの
- case_fix: そのケースだけの言い換え
- preference: 好み差、または高リスク場面で許容できる硬さ
- no_change: 直さない方がよい

注意:
- 「もっと柔らかく」だけの抽象添削は不要です。
- 危険境界を弱める自然化は不要です。
- 20件すべてに細かい添削を入れるより、再発構造を優先してください。
- ただし、日本語ビジネスチャットとして明らかに変な箇所は具体的に指摘してください。
- 返信文を直す場合は、なぜその修正文が `case_fix` なのか、`gold` なのか、`skill` なのかも示してください。
