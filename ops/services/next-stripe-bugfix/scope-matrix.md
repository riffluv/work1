# next-stripe-bugfix Scope Matrix

更新日: 2026-03-12

## 判定の基本
- まず `原因` を見る
- 次に `対象フロー` を見る
- 次に `対象エンドポイント` を見る
- 断定できなければ `undecidable` で止める

## 購入者主目的での入口判定
- 振り分け基準は `コードの複雑さ` ではなく `購入者の主目的` を優先する
- `直したい` `動かしたい` `今の不具合を止めたい` が主目的なら、コードの中身が不明でも bugfix 起点で扱う
- `まず把握したい` `引き継ぎたい` `次の担当者へ渡したい` が主目的なら handoff 起点で扱う
- `コードが分からない` は購入者の状態であって、handoff 送りの単独理由にしない
- 症状も対象フローも絞れない場合は、購入前に `一番困っている症状` または `一番止まると困る操作` を1つに絞る

## bugfix 内の最小整理
- bugfix で吸収する整理は、`対象の不具合の原因を切り分けるために必要な最小範囲` に限る
- 含むもの
  - 対象エンドポイント、対象処理の入口から出口までのコード追跡
  - 直接関連するファイル、設定、イベント線の特定
  - 症状と一次証跡を突き合わせるための最小限の構造把握
- 含まないもの
  - フロー全体の構造図作成
  - 危険箇所の網羅洗い出し
  - 引き継ぎメモ作成
  - リポジトリ全体の整理
- `整理が技術的に必要` と `整理サービスを購入者に買わせる` は別の話として扱う

## 購入後振り直し禁止
- bugfix として購入済みの案件を、途中で `先に handoff を買ってください` と振り直さない
- 想定以上の整理が必要と分かった場合も、まず bugfix の範囲で分かったことを共有する
- そのうえで
  - 現在の bugfix 範囲で区切る
  - 同一トークルーム内の追加対応として範囲を広げる
  の2択で扱い、別サービス購入へ押し戻さない

## same_cause_likely
- Webhook署名検証エラーが原因で、複数イベントが同じ受信口で失敗している
- mode の取り違えが原因で、本番だけ複数画面に反映漏れが出ている
- 同じ API 例外処理漏れが原因で、同一フロー内の表示崩れと保存失敗が連動している

## different_cause_likely
- Webhook署名エラー修正後に、別エンドポイントのDB更新不具合が見つかった
- Checkout 成功後の反映漏れとは別に、Billing Portal 側で契約解除フローが壊れている
- 同じ画面に見えても、片方は設定値不足、もう片方は別APIの認証失敗

## undecidable
- 症状は似ているが、ログがなく原因がまだ分からない
- フローが複数またがっており、どこが主因か未確定
- 依頼者の説明だけでは同一原因か別原因かを切れない

## boundary 事前判定フロー
- 症状が特定されている
  - bugfix で受ける
- 症状はあるが、どれを先に直すか曖昧
  - 購入前に `一番困っている症状` を1つに絞ってから bugfix で受ける
- 症状はまだ曖昧だが、主目的は明確に修正
  - まず bugfix 起点で扱い、購入前に対象症状を1つ特定する
- 症状の有無より、全体把握や引き継ぎが主目的
  - handoff で受ける
- どちらにも落ちず、症状も対象も絞れない
  - 無理に受けず、保留または辞退に倒す

## 文面ルール
- same のとき: `同一原因の範囲として対応します`
- different のとき: `原因が別のため、追加対応になります`
- undecidable のとき: `この時点では追加対応の要否は未確定です`
- boundary で修正主目的のとき: `まず直したい症状を起点に見ます`
- 技術的な整理が必要でも: `handoff を先に買ってください` とは言わない
