# 実ストック運用

## 目的
- 実際の相談文ストックを、返信 system 改善用の入力資産として整理する。
- 1件ずつ添削するのではなく、まとめて回して `contract -> renderer -> lint -> regression` に載せる。

## フォルダ構成
- `inbox/`
  - 未整理の生ストック置き場。
  - まずはここに月別バッチで入れる。
- `seed/`
  - 代表ケース。
  - 構造変更直後の最小確認に使う。
- `eval/`
  - 通常改善用。
  - 普段の精度改善や drift 確認で使う。
- `holdout/`
  - ブラインド検定用。
  - 改善中は正解合わせに使わず、最後にだけ回す。
- `edge/`
  - 事故になりやすい危険ケース用。
  - 規約、スコープ、秘密情報、価格、感情強め、購入後別件などを入れる。

## 置き方
- 最初は `inbox/2026-04-real-batch-01.txt` のような月別 batch でよい。
- 形式は `prequote-test-cases.txt` と同じ plain text を推奨する。
- 1 file に混在してよい項目:
  - `prequote`
  - `quote_sent`
  - `purchased`
  - `closed`
  - `delivered`
- ただし `state` は必ず入れる。

## 整理の流れ
1. 新しい実ストックを `inbox/` に置く
2. 代表性が高いものを `seed/` へ移す
3. 通常改善に使うものを `eval/` へ移す
4. 最後の検定専用は `holdout/` へ移す
5. 危険ケースは `edge/` へ移す
6. 必要なら `ops/tests/eval-sources.yaml` に active source として足す

## ルール
- `holdout/` は改善中の正解合わせに使わない。
- `rehearsal/` は文体監査用であり、主入力の正本にはしない。
- まず実ストックを優先し、Claude 生成文は不足領域の補完に限定する。
- 生の個人情報、秘密値、URL、アカウント識別子は置かない。必要なら匿名化する。

## すすめ方
- 実ストックを追加したら、まず `inbox/` に置く。
- まとまったら、こちらで `seed / eval / holdout / edge` に仕分けし、回帰へ接続する。
