# Claude バッチ監査プロンプト

以下は、ココナラ運用の返信文リハーサル結果です。
目的は、個別の添削だけでなく、「共通して出ているAI感」や「文体の癖」をまとめて見つけることです。

前提:
- 主軸は Next.js / Stripe / API連携の不具合診断・修正
- 出品ページとプロフィールは、AIっぽさを抑えた自然で直接的な文体
- 目指す人物像は「丁寧だが直接的なフリーランスエンジニア」
- 今回見てほしいのは、返信文がその人物像からズレていないか
- 個別ケースの正誤だけでなく、全体に共通する言い回しの癖を抜いてほしい
- 大改修ではなく、skill に戻せる最小限の修正方針がほしい

- /home/hr-hm/Project/work/サービスページ/bugfix-15000.live.txt

見てほしいこと:
1. このバッチの返信文に共通する AI感 は何か
2. 不自然な丁寧さ、説明しすぎ、回りくどさがどこに出ているか
3. 出品ページ・プロフィールの人物像とズレている表現は何か
4. 共通の禁止表現にした方がいいものは何か
5. 共通の置き換えルールにした方がいいものは何か
6. 個別添削が必要なケースがあれば最大3件だけ指摘してほしい
7. 最後に、skill に戻すべきルールを箇条書きで出してほしい
8. 相手が `箇条書きで` など返答形式を指定しているのに、その形式を無視していないかも見てほしい
9. 相手が感謝や前向きな驚きを出しているのに、冒頭で温度を返せていないケースがないか見てほしい
10. `もちろんです。` のような無難な冒頭が続いていないか見てほしい
11. 冒頭が `はい、` に寄りすぎて bot 的になっていないか見てほしい
12. 外向けなのに `bugfix` `handoff` `整理サービス` のような内部ラベルが漏れていないか見てほしい
13. `高い` `予算が厳しい` のような価格への引っかかりに、説明だけで返していないか見てほしい
14. `直らない` `変わらない` の報告に `承知しました` だけで止まっていないか見てほしい
15. 完全に対象外の断りで、必要なフォロー1文が抜けていないか見てほしい
16. `安心しました` `助かりました` `自分では触れない` のような好意・安心・弱さを相手が出しているのに、冒頭で1文受けずに本題へ入っていないか見てほしい
17. 怒りや催促が強いケースで、`切り分けの軸` のような分析寄りの言い回しが冷たく見えていないか見てほしい
18. 競合比較や価格差の説明で、`差額の説明だけすると` のような事務的な言い回しが出ていないか見てほしい
19. `星5` `大満足` `バッチリでした` のような強い好意があるのに、冒頭で短い喜びや謝意を返せていないケースがないか見てほしい
20. 追加料金や追加フローの支払い方法への返答で、`追加支払いの形` のような曖昧な案内になっていないか見てほしい
21. `この場で勝手に進めず` のような防御的な言い回しが残っていないか見てほしい
22. buyer の具体的な行動や協力が見えている時に、`ありがとうございます` だけで流さず、その行動に1文で具体的に触れられているか見てほしい
23. 温度を出そうとして、根拠のない安心や buyer の感情の言い当てに寄っていないか見てほしい
24. buyer が使っていない感情語を補って `迷いますよね` のような emotion_completion が出ていないか見てほしい
25. 受け止めの1文を入れた直後に `ですが` で切り返して、受けた感じを打ち消していないか見てほしい
26. 追加料金や追加支払いの案内で、`取引状態を確認して` のような内部寄りの抽象語で止まらず、buyer に見える状態語で案内できているか見てほしい
27. buyer の冒頭文をそのまま返すおうむ返しがないか見てほしい
28. 納期・保証・見通しの返答で、`確定できません` を先に置いて拒否感を作っていないか見てほしい。先に `返せること` を置けているかも見てほしい
29. `1行だけ` `5分だけ` `ちょっとだけ` という作業量アピールに引っ張られて、分類より作業量で範囲判断していないか見てほしい
30. 低評価や評価圧力のあるケースで、その言葉をなぞらず、進み方や見え方の事実へ受けを寄せられているか見てほしい
31. `post_close` の再相談で、buyer が前回のお礼や遠慮を先に置いているのに、受けなしで一般相談案内へ入っていないか見てほしい
32. 完全に対象外の方向づけで、`近いかもしれません` のような少し突き放した語尾が残っていないか見てほしい

出力形式:
1. 全体所感
2. 共通しているAI感
3. 禁止した方がいい表現
4. 置き換えルール
5. 個別で気になるケース（最大3件）
6. skill に戻すべきルール
