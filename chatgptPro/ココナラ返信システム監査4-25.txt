1. 結論
次の #RE は、**「不具合相談に見えるが、実際には buyer がサービス仕様を聞いているケース」**を最優先で回すのが最も学習効率が高いです。
batch-03 の本質は、返信品質そのものよりも 主質問タイプの取り違えです。buyer は「価格・納期・対象・保証・原因不明時の扱い」を聞いているのに、システムが「一般的な不具合相談」と解釈して、汎用ヒアリングへ流れていました。
公開 repo 側でも、#RE は「5件単位で batch を作り、監査→learning-log→必要最小限の反映」を行う運用で、恒久反映は再発性・戻し先・副作用の低さを見て判断する設計になっています。したがって、今すぐ rule / renderer を大きく触るより、次の 1〜3 batch は質問タイプ判定を狭く検査し、再発の形を集めるのが安全です。

3. 優先して検査すべき質問タイプ
P0：次 batch で必ず入れる
1. 価格不安型
理由:
buyer が「5分で分かっても15,000円？」「追加料金が怖い」と聞いている場合、汎用ヒアリングに流れると即 nonanswer になります。bugfix-15000 は 15,000円、不具合1件が基本、別原因の場合のみ追加相談、追加料金は自動発生しないという仕様が明確なので、最初に答えやすく、監査もしやすいです。
最初の直答:
「はい、基本は作業時間ではなく不具合1件として15,000円で、追加料金が自動で発生することはありません。」

2. 対象可否型
理由:
「これは対象ですか？」に対して情報収集テンプレへ流れると、buyer から見ると答えていないように見えます。対象に近いか、対象外か、判断保留かをまず返す必要があります。
最初の直答:
「Next.js / Stripe / API連携まわりの不具合であれば対象に近いです。」

3. 納期・復旧時間型
理由:
緊急性のある buyer は「必要情報をください」より先に「今日中に直るのか」「何時間で復旧できるのか」を知りたいです。ここで断定しすぎると oversell、ぼかしすぎると nonanswer になります。
最初の直答:
「復旧時間は症状と原因次第なので断定はできませんが、必要情報がそろってから48時間以内を目安に一次回答します。」

4. 原因不明・正式納品可否型
理由:
ここは buyer の不安が強く、仕様事実も明確です。「原因を特定できず、修正方針にもつながらない場合は、原因不明のまま正式納品へ進めない」という重要事実を先に出すべきです。
最初の直答:
「原因を特定できず、修正方針にもつながらない場合は、原因不明のまま正式納品へは進めません。」

P1：次か次々 batch で入れる
5. 秘密情報・共有方法型
理由:
.env や APIキー値、GitHub共有、本番権限の扱いは安全性に関わります。ここは曖昧さよりも、短く明確な境界が必要です。repo 上のサービス文面でも、.env の値や API キー値は不要で、キー名だけで足りること、可能なら ZIP 送付とすることが示されています。
最初の直答:
「.env や APIキーの値は不要で、キー名と関係ファイルのZIPだけで大丈夫です。」

6. 本番反映・直接操作型
理由:
「本番に入って直してください」「GitHubに直接 push してください」は、便利に見えるぶん oversell / hidden_rule が出やすいです。通話、直接 push、本番デプロイ代行は範囲外と明示されています。
最初の直答:
「本番デプロイ代行や直接 push は行っていませんが、修正内容と反映手順の整理までは対応できます。」

7. 保証・再発型
理由:
buyer は「また起きたら無料ですか？」を聞いているのに、一般的な安心文に流れやすいです。同一原因か別原因かで答える必要があります。
最初の直答:
「再発時は、まず同じ原因かを確認し、同一原因なら範囲内として扱える可能性があります。」

8. 購入後進捗型
理由:
batch-02 の B05 で同義反復が出た領域です。購入後は prequote の説明ではなく、「今どこまで見ているか」「次に何を見るか」が必要です。repo の failure taxonomy でも、主質問ズレ・phase 漏れ・テンプレ臭は明確な監査対象です。
最初の直答:
「進捗としては、まず現在分かっている点と次に確認する点を分けて共有します。」

P2：急がないが、回収はする
9. 不具合相談型
これは基本型なので、今すぐ重点化しなくてよいです。むしろ次 batch では混ぜすぎると、また汎用テンプレが強化されてしまいます。
10. 原因カテゴリ・自力修正可否型
「コードですか？設定ですか？」は batch-03 で見えた重要パターンですが、価格・納期・対象可否より一段低い優先度です。理由は、直答が「断定不可＋切り分けます」になりやすく、rule 化の副作用が出やすいからです。

gold は、**「この場面ではこう始めると安定する」**という少数精鋭に限定するのがよいです。repo の README でも、gold / NG は小さく curated にし、過剰追加しない方針が示されています。

validator に回す条件
validator は、機械的に検出できる危険だけに絞るべきです。自然さ・一息感・丁寧さを validator に入れると壊れます。
validator 候補は以下です。
validator候補検出したい危険bugfix-15000 で 25,000円や handoff 名が出る未公開サービス混線APIキー値 / 秘密キー値を要求する秘密情報受領の危険GitHub招待を推奨するZIP送付方針から逸れる本番デプロイ代行を受ける範囲外対応の oversell通話対応を受ける範囲外対応の oversell追加料金が自動発生するように見える価格誤認原因不明でも納品可能に見える納品条件の誤認
gold / validator にまだ回さないもの
回さないもの理由一息で言う客観検出しにくく、短文が自然な場面を壊すai_tone 全般validator 化すると文体が硬直する納期の自然な言い方状況依存が大きい進捗の温度感state と文脈依存質問タイプ分類そのものまだ観察段階。大きな層追加になりやすい

10. やらないこと
ここは強めに止めた方がいいです 🛑
やらないこと理由質問タイプ router を大規模追加するbatch-03 の観察だけで層を増やすと、現行骨格を壊すrenderer を刷新する問題は renderer 全体ではなく、主質問タイプの取り違えと直答順序rule を大量追加する単発違和感がルール化され、返信が硬くなる一息で言うを絶対 rule にする短い独立文が自然な場面まで壊すすべての返信を「はい、」で始める rule を作るbuyer の不安・断定不可・拒否場面で不自然になる納期を固定断定するoversell になりやすい保証期間を明示的に作るサービス仕様にない保証を作ってしまう危険handoff-25000 を外向け導線に混ぜる未公開サービスの hallucination になるAI tone を validator で縛る主観的すぎて副作用が大きいbatch 件数を急に増やす5〜6件単位での学習ループの利点が薄れるFAQ文面をそのまま長く貼る返信を増やす答えているようで buyer の主質問に刺さらない可能性がある不具合相談型ばかり回す今の弱点であるサービス仕様質問への nonanswer が見えにくくなる
現行ループの価値は、小さく壊して、小さく直し、再発性が強いものだけ戻す点にあります。ここを崩さない方がいいです。
