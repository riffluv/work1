再開します。結論からいうと、いまの方針はかなり良いです。特に code-comment-style.ja.md の「WHY / 制約 / 副作用 / 運用条件だけを書く」という線引きは、実務コードとして筋が通っていますし、Google のコードレビュー/文書化ガイドがいう「inline comment は code 自体が持てない情報、特に why を補うもの」という考え方とも整合しています。

いま足りないのは原則そのものではなく、運用の粒度です。つまり、「bugfix 納品コードではどのくらい薄く残すか」「handoff 前提ではどのくらい厚く残すか」「TODO / FIXME / NOTE をどこまで許すか」「コードに残す情報と納品物に逃がす情報の境界」を、もう一段だけ具体化すると完成度が上がります。bugfix サービスはすでに「原因 / 修正内容 / 確認手順」を納品物で説明する設計で、handoff サービスも「危険箇所 / 次の着手順」を別メモで渡す設計なので、コード側は“事故を防ぐための最小限の注釈”に寄せるのが自然です。

返信文まわりも方向性は良いです。writing-guideline.ja.md の「要点 → 根拠 → 次の作業 → 相手にしてほしいこと」は、日本語圏の業務チャットでかなり強い並びですし、ココナラの取引導線とも噛み合っています。トークルーム内完結や正式納品後の承諾/差し戻し導線も、現行の運用方針で問題ありません。

1. 全体所感

一番大事なのは、コードコメントは購入者への説明文ではなく、未来の保守者への局所メモだと割り切ることです。
プロっぽく見えるコメントは、読んだ瞬間に「この人はコードの事故点だけを分かって残している」と伝わるものです。逆に AI っぽく見えるコメントは、「全部説明しようとしている」「コードの横で文章を書いている」感じが出ます。

あなたのサービス設計だと、

bugfix は「今回の原因・修正・確認方法」を納品物で説明する
handoff は「全体像・危険箇所・次の着手順」を別メモで説明する
ので、コードコメントに“今回の案件の説明責任”を背負わせないのが正解です。コメントはあくまで、次にその行を触る人が誤修正しないためのものだけ残す、で十分です。
2. コードコメントの原則
自然でプロっぽく見えるコメント

自然に見えるコメントの条件は、だいたい次の4つです。

その場で必要なことしか書いていない
固有名詞がある
条件や理由がある
文体が無愛想なくらい実務寄り

日本語でいうと、**「説明文」ではなく「制約メモ」**に近いです。
event.id、Stripe-Signature、checkout.session.completed、429、raw body のような固有名詞が入ると、一気に人間の実務感が出ます。逆に「適切に」「必要に応じて」「念のため」「重要です」は、内容が薄いのに文だけあるので AI 感が出ます。

「何をしているか」より「なぜ必要か」は妥当か

inline comment については、かなり妥当です。
Google のガイドでも、通常コメントは「何をしているか」より「なぜそれが存在するか」を説明するのが有用で、コードが自分で説明できないならまず code を簡潔にすべきだとされています。例外は、正規表現・複雑アルゴリズム・状態遷移・暗黙の業務ルールのように、what を一言で補わないとかえって誤読しやすい箇所です。

ただし、関数・クラス・モジュールの doc comment は別です。
そこは「何をする API か」「どう使うか」「どんな制約や戻り値があるか」を書く領域です。Google の文書化ガイドでも、method/class comment は contract を表し、why は通常 inline comment 側に置く整理です。つまり、

行コメント: why / 制約 / 副作用
docstring / 関数ヘッダ: what / how to use / contract
の分担が自然です。
書くべき場所 / 書かないべき場所

書くべき場所

外部仕様に縛られるところ
順序保証がない・再送がある・冪等性が必要なところ
壊すと二重課金・未反映・重複更新につながるところ
暫定対応や互換層が残るところ
型や関数名では表現しづらい運用条件があるところ

書かないべき場所

if (!user) return のような自明な分岐
関数名そのままの実況
「ここで DB 更新」みたいな見れば分かる説明
その案件でしか通じない経緯メモ
調査ログや思考メモ

この線引きは、あなたの現行ルールとかなり一致しています。inline comment は code が持てない情報だけを補い、周辺説明は README・docs・handoff memo に逃がす方が保守しやすいです。

bugfix 納品コードではどの程度残すか

かなり薄くていいです。
目安としては、原因があった危険箇所ごとに 0〜3 コメントくらいで十分です。
残すのは、

今回の再発防止に直結する制約
次の人が同じ落とし穴に落ちる可能性が高い箇所
修正後の運用条件

までです。再現手順、確認手順、今回見たログ、なぜこのファイルも触ったか、は納品物へ逃がす方が見やすいです。あなたの bugfix サービスはもともと「原因 / 修正内容 / 確認手順」を納品物で説明する設計なので、source code 側は薄い方がむしろプロっぽく見えます。

diff の騒がしさという意味では、新規コメント行は変更行の 10〜15% 以内くらいを運用目安にすると、うるさく見えにくいです。これは公式ルールではなく実務目安ですが、Google も「機能変更と大きな style/comment 変更を混ぜるとレビューしにくい」としています。

handoff 前提のコードではどの程度残すか

handoff では bugfix より一段だけ厚くていいです。
ただし厚くする場所は限定で、入口・危険箇所・境界条件だけです。

フロー入口の関数や route
状態更新の正本が決まっている場所
外部 API / Webhook / queue 境界
「ここを変えると他に効く」共有状態

一方で、ファイル一覧、処理の全体像、未確認事項、次の着手順は handoff memo の役目です。サービス説明でも handoff の納品物は 00_結論と要点.md と 01_[対象フロー名]_引き継ぎメモ.md を前提にしているので、source code は流れの全説明書ではなく、危険箇所の標識だけ残すのが自然です。

日本語コメントと英語コメント

結論は、既存コードベース優先です。
現行ルールの「日本語を基本にする（既存が英語主体なら合わせる）」は正しいです。加えて、実務では次のように切ると迷いません。

既存コメントが英語中心 → source code は英語に寄せる
既存コメントが日本語中心、チームも日本語圏 → 日本語でよい
API フィールド名・ヘッダー名・イベント名は 原文の英語を残す
同じファイル内で「英語コメント群の中に日本語 1 行だけ」は避ける

Google のコードレビューガイドも、他ルールがなければ既存コードとの一貫性を保つべきだとしています。英語主体 repo に日本語コメントを混ぜると、それだけで“納品のためにあとから書いた感”が出やすいです。日本語で書く場合も、Stripe-Signature や event.id はそのまま残した方が grep と照合に強いです。

行コメントとブロックコメントの使い分け

基本は行コメント優先です。

行コメント: その直下の 1〜5 行に効く why / constraint
短いブロックコメント: 1つの処理ブロック全体にかかる前提
docblock / docstring: exported function, module, public utility の contract

中腹の /* 長文 */ はだいたいうるさく見えます。
自然に見えるのは、

route の入口に 2〜3 行の block comment
危険な分岐の直前に 1 行 comment
の組み合わせです。これは Google の「inline comment は why」「method/class comment は contract」の分担と同じです。
TODO / FIXME / NOTE は残すべきか
TODO: 条件付きで可
FIXME: 納品コードでは基本避ける
NOTE: 不変条件や制約なら可、感想メモは不可

特に TODO は、Google の style guide でも「検索できる一貫した形式」「文脈にたどれる情報」が必要とされ、bug reference や owner/identifier を付けることが推奨されています。あなたの TODO(ISSUE-123, 2026-03-01) はかなり良いです。

一方で paid delivery の source code に FIXME が残ると、購入者には「未解決不具合が source に明記された」ように見えやすいです。
残すなら、

依頼者と合意済みの残課題
納品説明にも同内容を明記
ticket / date / removal 条件あり

のときだけに絞るのが安全です。

一時しのぎ / 暫定対応はどこまで説明するか

暫定対応のコメントは、3点だけあれば十分です。

何が暫定か
なぜ今はこれが必要か
いつ / 何を条件に外すか

たとえば、
// 暫定: v1 payload 互換を 2026-04-15 まで残す。移行完了後に ISSUE-123 で削除
くらいで十分です。

逆に、

何月何日に何件起きたか
誰との会話でそう決めたか
どのログを見たか

は code ではなく納品物に逃がす方が自然です。

3. AIっぽく見えるコメントの特徴

AI っぽく見えるのは、文体より配置と密度の問題が大きいです。典型は次です。

コードの各行を実況している
1つの修正に対してコメントが多すぎる
抽象語だけで中身がない
例: 適切に, 必要に応じて, 重要です
丁寧語・敬語が混ざる
例: 〜します, 〜させていただきます
「念のため」「ここでは」「まず」「次に」が多い
コメント単体では具体性がなく、コードを読まないと意味が分からない
逆に、コードを読めば 100% 分かることしか書いていない
文章が均一に長く、全部同じトーン
“作者の思考メモ”が残っている
例: 一応この形にしています

現行の禁止語リストと、メッセージ文面で避けている AI 臭い表現は、そのまま code comment にも効きます。特に「敬語」「抽象語」「説明しすぎ」は source code だと一気に浮きます。

4. 良いコメント / 悪いコメント例
Stripe / Webhook
悪い: // StripeのWebhookを適切に処理する
良い: // 署名検証は raw body 必須。JSON に触ると失敗する
悪い: // 同じイベントが来る可能性があるので注意
良い: // event.id で重複排除。Stripe は同一イベントを再送することがある
悪い: // invoice.created のあとに invoice.paid が来る
良い: // Stripe は配信順を保証しない。invoice.paid 単体で再計算できる前提にする
悪い: // エラー時は 500
良い: // 署名不正は 400。再試行で回復しうる障害だけ 500

これらは、Stripe の official docs にある「raw body が必要」「duplicate event がありうる」「ordering は保証されない」「retry がある」という前提に沿う形です。

Next.js / React
悪い: // 初回レンダリング時に実行
良い: // session 未確定で購読開始しない。匿名状態の二重購読を防ぐ
悪い: // データを取得して state を更新
良い: // 楽観更新しない。Stripe 側反映前に UI だけ先行すると状態がずれる
悪い: // ここで API を呼ぶ
良い: // UI から直接 Billing API を叩かない。再試行時の二重実行を避けるため server 側に寄せる
API 連携 / 暫定対応
悪い: // TODO: あとで直す
良い: // TODO(ISSUE-123, 2026-04-15): v1 互換分岐を削除
悪い: // 念のため null チェック
良い: // provider は 204 を返しても body が空とは限らないため、status と payload を分けて判定
悪い: // 一時的にこの処理を追加
良い: // 暫定: 旧レスポンス形式を 2026-04-15 まで受ける。移行完了後に削除

英語ベースの repo なら、上の意味をそのまま英語にして統一するのが自然です。日本語 1 行だけを差し込むより、// Deduplicate by event.id; Stripe may redeliver the same event. のように揃えた方がきれいです。

5. 実務で使えるコメントルール
名前で言えることはコメントで書かない。
コメントは why / constraint / side effect / ops だけ。
1 コメント 1 論点。
固有名詞を入れる。 (event.id, Stripe-Signature, 429)
結論先頭で 1〜2 文まで。
敬語・抽象語を入れない。
TODO は issue / owner / date / removal 条件を付ける。
今回の案件説明は納品物へ、未来の事故防止だけ code に残す。

この8本でかなり実務運用できます。いまの code-comment-style.ja.md は方向性が合っているので、足すならこの粒度です。

6. コメントを残すべき時 / 納品物に逃がすべき時

判断基準はシンプルで、その情報が「次にこの行を触る人」に必要かどうかです。

code に残すべき時
無いと誤修正の確率が上がる
型や関数名だけでは伝わらない
外部仕様に縛られている
副作用や再実行条件がある
暫定対応が残る
その場の 3〜10 行を守るために必要
納品物に逃がすべき時
今回の不具合の経緯
見たログ / 再現条件 / 切り分け過程
依頼者環境での確認手順
変更ファイル一覧
未確認事項
次の担当者への着手順
購入者との合意内容

これは、あなたの納品パックと handoff サービスの役割分担と一致しています。00_結論と確認方法.md や 01_..._引き継ぎメモ.md がある前提なら、source code は薄くて問題ありません。むしろその方が「相手のコードに残して違和感がない」状態になります。

7. 返信文の見やすさルール

ここは今の writing-guideline.ja.md がかなり良いです。運用ルールとしては次で十分です。

結論先出し + 補足 を維持する
1メッセージ1目的にする
1段落1役割にする
質問は 3〜5 件まで
期限と返答形式を明記する
柔らかさは冒頭 1 行で足す
結びの定型文を毎回固定しない

見た目の細部では、

空行は 意味の切れ目ごとに 1 行
箇条書きは 本当に答えてほしい項目だけ
1文は 1論点
余計な空白は入れず、Event ID や .env のような識別子の周辺だけ整える

が効きます。今のガイドにある「要点 → 根拠 → 次の作業 → 相手にしてほしいこと」は、まさに日本語圏の業務チャットで読みやすい並びです。トラブル対応では特に有効です。

「見やすいが冷たくない」ためには、礼儀は冒頭 1 行、具体性は本文が基本です。
ありがとうございます。状況は把握しました。 までで柔らかさは足ります。そのあとに長いクッション言葉を重ねず、観測・次の作業・依頼事項を切る方が、むしろ安心感が出ます。現行の回答例もこの方向にかなり近いです。

8. 良い返信例 / 悪い返信例
初回返信

良い例

ありがとうございます。状況は把握しました。
「本番のみ / Webhook署名検証失敗 / checkout.session.completedで500」が主症状ですね。

まず、署名検証とWebhook設定を切り分けます。
追加で、次の3点をお願いします。
1) 失敗した Event ID を 2〜3件
2) エラーログ原文
3) Webhook URL と購読イベント一覧

本日18:00までに一次見解を返します。

悪い例

ご相談ありがとうございます。鋭意確認させていただきますので、差し支えなければ可能な範囲で必要情報一式をご共有いただけますと幸いです。確認後、追ってご連絡申し上げます。何卒よろしくお願いいたします。

違いは、良い例は「現状認識」「次にやること」「お願い」「期限」が見えること、悪い例は礼儀はあるのに中身が見えないことです。

進捗報告

良い例

進捗です。
原因候補は2つまで絞れています。現時点では raw body を JSON 化している可能性が高いです。

次に、署名検証前の body 読み出し箇所を確認します。
18:00までに、修正案か追加で必要な情報かを返します。

悪い例

進捗についてご共有いたします。現在、総合的に確認を進めており、いくつかの可能性が考えられます。引き続き確認いたしますので、今しばらくお待ちください。

良い例は短くても状況が進んでいることが分かります。悪い例は“調査中”以外の情報がなく、AI っぽく見えます。

9. 今のワークスペース方針で足りない点

追加するなら、次の6点です。

bugfix / handoff 別のコメント量ルール
bugfix: 危険箇所ごとに 0〜3 コメント
handoff: 入口 + 危険箇所 + 境界条件まで可
行コメント / ブロックコメント / docblock の使い分け
行 = 局所制約
block = 処理ブロック前提
docblock = API contract
TODO / FIXME / NOTE の出荷ポリシー
TODO は条件付き許可
FIXME は原則避ける
NOTE は不変条件のみ
暫定対応コメントのテンプレ
例: // 暫定: [理由]。[削除条件]。ISSUE-123
言語統一ルールの強化
既存 English repo に日本語を混ぜない
日本語コメントでも API 名は原文維持
コードと納品物の境界ルールの明文化
code = 未来の保守者向け
納品物 = 今回の購入者 / 次の担当者向け

補足で言うと、delivery-pack 側の「約3日間未操作で自動承諾クローズ」は現在のココナラヘルプと整合しているので、その運用はこのままで大丈夫です。むしろ足すべきなのは、コメント規約側の運用細則です。

一言でまとめると、短く具体的で AI 感のないコメントは、
「未来の保守者がそこで事故らないための制約だけを、無愛想なくらい局所的に残す」
ものです。購入者への説明責任は、すでに整っている納品物とメッセージ側で担えば十分です。
