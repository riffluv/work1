1. 結論

🧭 判定：bugfix-15000 の返信コアは「v1 completion candidate」と言ってよいです。
ただし、「v1 complete / release sealed」と言い切るにはまだ早いです。

理由は明確です。

返信文そのもの、phase 境界、public/private 境界、secret 境界、closed 後境界はかなり安定しています。
completion checklist も「もっと #RE を回す」ではなく「何を検証済みとみなすか」に軸足を移しており、この判断方向は妥当です。
completion shelf も bugfix-15000 を v1 完成候補 / 機能コア完成圏 と明示しつつ、real_stock、app_memory、multi_service、email_channel を未検証として残しています。これは現状認識として正しいです。

ただし、辛口に言うと、今の弱点は返信文ではありません。弱点はこの4つです。

領域	現状評価	問題
返信品質	高い	同型 #RE は飽和気味
completion 定義	良い	candidate と sealed v1 の境界がまだ少し曖昧
contract packet	有望	sample と shape check 止まり。runtime 入力としては未封印
service truth	かなり良いが危険	canonical facts と legacy service.yaml が二重正本化しうる
app / 汎用コア骨格	ある	まだ移植性・入力 schema・resolver が弱い

最終判断：

bugfix-15000 は v1 completion candidate として棚に上げてよい。
ただし次にやるべきは #RE 追加ではなく、service truth の単一化、contract packet の実行可能化、checker の強化、synthetic / real stock の区別明文化です。
この4点を潰す前に「完成」と呼ぶと、将来アプリ化時に設計負債になります。⚠️

completion checklist / completion shelf の妥当性

core-completion-checklist.ja.md の完成条件は、方向としてかなり妥当です。特に、完成条件を「返信文をさらに磨く」ではなく、Service truth、Phase contract、Reply decision、Evidence、Secret safety、Writer preservation、Naturalizer safety、Evaluation coverage、Learning loop に分解しているのは正しいです。

また、completion shelf が bugfix-15000 の対象範囲を、

bugfix-15000
prequote / quote_sent / purchased / delivered / closed
Next.js / Stripe / API 不具合修正

に限定しているのも健全です。

ただし足りない完成条件

現状の完成条件に、次の4つを追加した方がいいです。

1. v1 candidate と v1 sealed を分ける

今の文書は v1 完成候補 / 機能コア完成圏 という表現が適切ですが、実装者が読み進めると「もう完成でよい」と誤読しやすいです。

おすすめは次の3段階です。

ラベル	意味
functional_core_complete	返信文・境界・主要 phase は安定
v1_completion_candidate	synthetic rehearsal では高飽和。meta layer も揃い始めた
v1_release_sealed	contract packet / service truth / #R-#RE 経路 / real stock 受け入れ方まで封印済み

今は 2段階目です。

2. synthetic_high と real_high を分ける

coverage map 上では主要 family が saturation: high ですが、全て source_type: synthetic_rehearsal です。

なので、今後は saturation: high だけではなく、

saturation:
  synthetic: high
  real_stock: low

のように分けた方が安全です。

今のままでも v1 candidate は名乗れます。
ただし、real stock が少ないまま v1 complete と呼ぶのは早いです。

3. #R と #RE が同一 contract packet から作られている の証跡が必要

completion checklist では「#R と #RE の writer candidate が同じ contract packet から作られていること」が完成判定ゲートに入っています。

しかし現物を見る限り、contract packet sample は存在するものの、runtime の #R / #RE がそこから実際に生成されていることを証明する pipeline まではまだ見えていません。contract packet README も「実装前に見える形で確認するためのサンプル」と位置づけています。

ここは candidate から sealed へ進むための最大ゲートです。

4. service truth の canonical resolver が必要

この後詳しく書きますが、service-pack/facts.yaml を canonical にしたい意図はある一方で、registry の facts_file は旧 service.yaml を指しており、render scripts もそこを読んでいます。

これは将来サービス追加時に破綻しやすいです。

3. contract packet 評価
総評

✅ 方向はかなり良いです。
memory_packet / phase_contract / reply_contract / response_decision_plan / writer_notes の分離は、#R / app 化の入力単位としてかなり近いです。

特に、5 phase の sample があるのは強いです。prequote / quote_sent / purchased / delivered / closed それぞれで、buyer 最新文、known commitments、received materials、forbidden assumptions、answer_map、ask_map、direct_answer_line が具体化されています。

ただし、現時点では **「runtime packet」ではなく「設計サンプル」**です。

良い点
1. memory と reply_contract の責務分離は正しい

reply-memory-schema.yaml は、保持すべき取引記憶を facts として定義しており、writer へテンプレートではなく事実制約として渡す方針が明確です。

これは良いです。

特に、

known_commitments
received_materials
pending_actions
forbidden_assumptions

は app 化時にも必要になります。

2. phase_contract は安全境界として使える

phase-contract-schema.yaml は phase ごとの allowed / forbidden / preferred next actions が分かれており、prequote、quote_sent、purchased、delivered、closed の境界がかなり明確です。

特に closed の、

promise_free_repair 禁止
return_fixed_files_in_closed_message 禁止
実作業が必要なら費用相談

は強いです。

3. reply_contract の answer_map / ask_map は実行契約として良い

output-schema.yaml で answer_map と ask_map を executable source of truth と位置づけているのは正しいです。

これは将来アプリ化するときも使えます。

問題点
問題1：checker がまだ「形だけ」

check-contract-packets.py は現状、

required fields の存在
phase の一致
hidden service が visible_public_services に入っていないこと
primary_question_id が explicit_questions に存在すること
answer_map / ask_map の参照先が存在すること
direct_answer_line / response_order の存在

を見るに留まっています。

これは最低限としては良いですが、v1 completion gate としては弱いです。

追加すべき検査は次です。

追加検査	理由
全 explicit_questions が answer_map か ask_map によって処理されている	複合質問落ちを防ぐ
primary_question_id の answer が direct_answer_line と矛盾しない	主質問直答の担保
answer_after_check に hold_reason / revisit_trigger がある	保留の空文化防止
ask_map が blocking_missing_facts と対応する	不要質問防止
forbidden_moves / required_moves が output-schema enum に含まれる	typo / 独自増殖防止
hidden services / 25,000円 / handoff 導線が packet 全文字列に出ない	public false 漏洩防止
writer_notes.use が exact phrase 強制になりすぎていないか warning	Writer preservation 保護
absolute path が入っていないか warning	app 化移植性
問題2：direct_answer_line が完成文に近すぎる

response_decision_plan.direct_answer_line は便利ですが、今の sample では完成文にかなり近いです。

これは二面性があります。

良い面：
Writer が主質問から逃げにくい。

悪い面：
Writer が「この文を少し変えるだけ」になりやすい。

対策として、field を分けるとよいです。

direct_answer_intent: "原因不明でも15,000円で依頼可能。ただし成功保証はしない"
suggested_answer_line: "原因が分からない状態でも、15,000円の不具合修正としてご依頼いただけます。"
preserve_policy: intent_not_exact_wording

v1 では direct_answer_line を残してよいですが、checker で **「verbatim copy required ではない」**ことを明記した方がいいです。

問題3：writer_notes.use / avoid が少し危ない

sample の writer_notes は実用的ですが、use に具体文が入りすぎると、Gold テンプレ化と同じ方向に寄ります。

特に危ないのはこれです。

use:
  - "ご依頼いただけます"
  - "コードとエラー内容を見る前に、必ず直せるとはお約束できません"
avoid:
  - "相談できます"

この粒度は、単発 case では有効です。
ただし app 化時は Writer の自由度を狭めます。

おすすめはこうです。

writer_notes:
  semantic_preferences:
    - "依頼可否への返答なので、相談可否ではなく依頼可能性を明示する"
  risky_wording:
    - phrase: "相談できます"
      reason: "この buyer 文では依頼可否への答えとして主体が弱い"
      blanket_ng: false

つまり、phrase ではなく判断理由を渡す方がよいです。
