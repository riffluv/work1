1. 総評
結論から言うと、現行 repo は「かなり良い骨格に入っている」が、まだ “主質問未回答・不要な聞き返しを構造的に潰し切る” には一段足りないです。
ただし、弱さの主因は scenario や renderer 分岐が足りないことではありません。むしろ分岐はすでにかなり増えています。いま残っている弱さは、主に次の層です。
領域評価AI の思考を潰さないかなり改善済み。ただし renderer 側の文面分岐がまだ強く、decision が最終出力に必ず反映されたかの保証が弱いサービス内容を把握するservice-registry.yaml と service grounding は良い。ただし絶対パス依存と final policy validator の中央集約がやや弱い主質問に正面から答えるreply_contract / answer_map / response_decision_plan でかなり進んでいる。ただし全 renderer で同じ強さの coverage check にはまだなっていない
特に重要なのは、quote_sent / post_purchase / delivered / closed では response_decision_plan が入り、primary_concern、facts_known、blocking_missing_facts、direct_answer_line、response_order のような「思考を潰さないための中間表現」がすでに実装されています。これは、参考資料で示されている「自由生成に寄せず、分類・判断・最終検証を分ける」方向と一致しています。
一方で、prequote は reply_contract が強いものの response_decision_plan 形式としては他 renderer より薄く、delivered / closed は renderer 内の final validator が弱めです。ただし統合側の render-coconala-reply.py と lane validator / common lint では、direct_answer_first、buyer_questions、known_info_reask、template_leakage、hallucination などを見ており、repo 全体としては validator/eval の方向にかなり寄っています。
なので今やるべきことは、大規模再設計ではなく、既存骨格を “decision-locked renderer” に少し寄せることです。つまり、「分類した」「答えるべき主質問を決めた」「直接回答文を作った」だけで終わらせず、その直接回答が実際に冒頭付近に出ているか、既知情報を聞き返していないか、公開サービス範囲を破っていないかを共通 property として落とすのが最小で効きます。

2. 今の状態で強いところ
2.1 reply_contract の骨格は正しい
render-prequote-estimate-initial.py は、明示質問を explicit_questions に分解し、primary_question_id、answer_map、ask_map、issue_plan、required_moves、forbidden_moves を持たせています。これは「相手の文章から主質問を拾い、答える / 保留する / 最小限聞く」を renderer 前に固定する設計です。見積り初回では特に、この構造があることで、自由文生成が先に走って主質問を落とすリスクをかなり下げています。
2.2 response_decision_plan が入っているのは大きい

特に direct_answer_line があるのは重要です。文章を自然にする前に、「主質問へ最初に何と答えるか」を短い文として持てるため、後段で文章が丸まっても検証しやすくなります。

さらに shared 側では service registry から service record を解決し、facts.yaml、evidence-contract.yaml、routing-playbooks.yaml を読む形になっています。quote_sent では capability / diagnostic patterns も service pack から引く構造があり、「サービス内容を覚えたふりで答える」よりかなり安全です。
2.4 統合 validator / common lint の方向が良い

3. 今の状態でまだ弱いところ
3.1 弱さは「scenario 不足」ではなく「decision → output の保証不足」
いちばんの論点はここです。
今の renderer 群は、かなり多くの scenario を持っています。quote_sent、post_purchase、delivered、closed は特に、scenario 分岐がすでに大きいです。なので、ここからさらに「このケースが抜けたから scenario を足す」を続けると、モグラたたきに戻るリスクが高いです。
不足しているのは、scenario の数よりも、


direct_answer_line が本当に出力冒頭に出たか


answer_map の各 answer_now が実際に本文に反映されたか


blocking_missing_facts がないのに聞き返していないか


facts_known に入っている内容を再質問していないか


private service / external contact / raw secret / refund promise などを最終文で破っていないか


を 全 renderer 共通の property として必ず検査する層です。
すでに common validator はありますが、まだ marker ベースの個別対策が多く、「contract coverage を共通仕様として機械的に見る」段階までは行き切っていません。
3.2 prequote と他 renderer の中間表現が少し揃っていない
prequote は reply_contract が強い一方、response_decision_plan 形式は他 renderer ほど前面に出ていません。逆に quote_sent 以降は response_decision_plan が強い。ここは揃えたほうがよいです。

post_purchase には validate_render_payload() があり、違反時に rendered_reply_validator_mode = "guardrail_violation" を付けます。prequote にも payload validation があります。一方で、delivered / closed は renderer 内に同等の validate_render_payload() が見当たりません。もちろん統合側や lane validator では検査されていますが、renderer 単体の安全性はやや薄いです。
ここは live を壊さないためにも、「renderer 内では記録だけ、regression では strict fail」という二段構えがよいです。
3.4 service grounding は良いが、絶対パス依存が運用リスク
service-registry.yaml の service_pack_root と facts_file は /home/hr-hm/Project/work/... の絶対パスです。実運用環境では問題ない可能性がありますが、repo を別環境で回す regression / 外部監査 / CI では壊れやすいです。
さらに shared loader では routing-playbooks.yaml 読み込み失敗時に一部 RuntimeError を握りつぶして route lookup を続行する箇所があります。live 安全性としては理解できますが、eval では「service pack が読めていないのに generic に落ちた」ことを見逃す可能性があります。ここは --strict-service-grounding 的な eval 専用チェックを追加したほうがよいです。

generic_quote_sent、generic_followup、generic_delivered、generic_closed のような fallback は必要です。ただし、「具体的な主質問があるのに generic fallback が生き残る」ことは、主質問未回答の温床になります。
すでに check-rendered-quote-sent-followup.py では generic_quote_sent fallback survived a concrete quote_sent question のような発想が入っており、方向は良いです。これを全 lifecycle に広げると効きます。

新 scenario を足す条件は、新しい lifecycle / 公開範囲 / policy 境界 / 入力型が明確に違う場合だけでよいです。
5.2 「文章を自然にする」後処理を強くする
render-coconala-reply.py の apply_semantic_safe_naturalization() が identity-only になっているのは正しいです。意味を変えない typed naturalizer がない段階で自然化を強くすると、せっかく作った reply_contract / direct_answer_line を後段で壊します。
5.3 外部AI監査を live 判定に直結する
外部AI監査は有効ですが、live の送信可否を毎回 AI judge に委ねるのは危険です。使い方は、「失敗ケースを見つけて eval fixture に戻す」です。運用ループに入れるべきで、送信直前 gate の主役にしないほうがよいです。
5.4 聞き返しを安全策として増やす
このシステムの目標は「相手の文章に対して主質問へ正面から答えること」です。聞き返しは安全に見えますが、実際には主質問未回答の逃げ道になります。
聞き返しは、
blocking_missing_facts があるanswer_after_check の理由が明示されているask_text が最小である既知情報を再質問していない
を満たす場合だけに寄せるべきです。

6. 文章学習の運用方針
6.1 live 確認用ケース
live 確認用は、件数を増やしすぎないほうがよいです。目的は「本番で壊していないか」の確認です。
見るべき項目は文体ではなく、次です。


主質問に冒頭で答えているか


価格 / 返金 / 追加料金 / 対応可否を逃げていないか


不要な聞き返しがないか


既知情報を再質問していないか


公開サービス範囲を破っていないか


ココナラ外誘導・raw secret・直接 push・本番 deploy を許可していないか


complaint / anxiety の冒頭受け止めが崩れていないか


live 確認用は seed / renderer_seed に近い役割で、少数・固定・毎回必ず通すがよいです。
6.2 learning / stress test 用 batch
learning / stress test は、ケース数を増やしてよい場所です。
ここでは、


価格不安


返金質問


追加料金不安


「対応できますか」


「このまま購入していいですか」


「何を送ればいいですか」


「前にも送ったのですが」


「本番キーを貼っていいですか」


「Zoom できますか」


「外部で送っていいですか」


「前回と同じ症状です」


「別原因ならどうなりますか」


のような、主質問未回答・不要な聞き返しが起きやすいケースを batch に寄せるべきです。
現在の role suites は edge、eval、holdout を持っているので、このまま活かせます。edge は既知の地雷、eval は改善確認、holdout は過学習検出に使うのがよいです。
6.3 外部AI監査
外部AI監査に聞くべきことは、「この文章はうまいか」ではありません。
監査プロンプトは、例えばこういう観点に固定するのがよいです。
1. 相手の主質問は何か2. 返信はその主質問に最初の2段落以内で答えているか3. 答えていないなら、どの質問を落としているか4. 既に相手が出した情報を聞き返していないか5. 聞き返しは blocking missing facts に限られているか6. サービス範囲・公開範囲・運用ルールを破っていないか7. 価格・返金・追加料金への回答が曖昧に逃げていないか
監査結果は、その場で prompt を増やすのではなく、fixture / expected marker / property validator に戻すのがよいです。参考資料でも、小さな curated set、golden set、post-check を育てる方向が推奨されています。
6.4 classifiers / correctness / policy adherence / evals の使い方
この repo では、次の分担にするときれいです。
層役割具体化classifiers入力理解lifecycle、scenario、buyer_questions、known_facts、emotion、service_hint、riskcorrectness主質問回答primary_question_id、answer_map、direct_answer_line、coveragepolicy adherence破ってはいけない運用public only、no raw secret、no external contact、no direct push、no prod deploy、no refund promiseevals退行検出role suites、holdout、property buckets、failure report
特に「主質問未回答や不要な聞き返しを減らす」なら、classifiers は scenario 名を当てるためだけではなく、buyer_questions と known_facts を抽出する層として使うべきです。そこから correctness validator が「答えたか」「聞き返していないか」を見るのが一番効きます。

ただし、現行骨格の上に置くなら、明確に上回る最小案はあります。名前を付けるなら、Decision-locked renderer です。

8. 返信システム強化メモに残すべき原則
残すべき原則は、次の8つです。


主質問は scenario より上位に置く。
scenario が合っていても、主質問に答えていなければ失敗。


聞き返しは blocking missing facts があるときだけ。
不安だから聞く、ではなく、答えるために不可欠だから聞く。


direct_answer_line は最終文に出るまで完了ではない。
plan に存在するだけでは不十分。冒頭付近に出ていることを検査する。


既知情報の再質問は品質事故として扱う。
facts_known / reply_memory / buyer-provided context を再質問したら fail。


service grounding は文面生成前ではなく、最終文でも検査する。
public/private、価格、scope、hard-no は final output policy。


scenario 追加より property 追加を優先する。
新しい地雷は、まず coverage、ask_gating、policy、known_fact のどれで落とせるかを見る。


live と learning を混ぜない。
live は少数固定で退行検出。learning / stress / holdout は失敗を集めて育てる場所。


外部AI監査は文章添削ではなく、未回答検出器として使う。
「自然さ」ではなく、「主質問・既知情報・不要質問・policy」を判定させ、結果を eval に戻す。


最終判断としては、現行骨格は捨てないほうがいいです。方向は合っています。次に触るべき最小点は、scenario 追加ではなく、reply_contract / response_decision_plan と final output を結ぶ共通 validator です。これがいちばん「AI の思考を潰さず、サービス内容と運用ルールを踏まえ、主質問に正面から答える」目的に効きます。
