残存リスク
purchased_handoff_progress_update が state 側に閉じていない — 今すぐ直すべき。 routing-playbooks.yaml には purchased_handoff_progress_update があり、regression もその route を参照していますが、state-schema.yaml の current_route.values にはまだその route が入っていません。つまり routing と regression では存在するのに、state continuity の正本では未登録です。これは silent drift になりやすいです。
その route の return_timing が schema 外の語彙 — 今すぐ直すべき。 purchased_handoff_progress_update.state_contract.promised_next_action.return_timing は same_day_or_next_reply ですが、state-schema.yaml の promised_next_action.return_timing は immediate|next_reply|before_delivery|with_delivery|before_close|with_close で、共通 runtime interface もその閉じた timing vocabulary 前提で書かれています。今のままだと route が state 契約の外側に 1 個だけはみ出しています。
回答例2の「危険箇所」ニュアンスは少し弱く圧縮されている — 運用監視で十分。 公開文面の回答例2は、どこまで確認できたか / どこが危険箇所になりそうか / 次にどこを見るか / いつ返すか まで含みますが、現在の route と regression は主に 確認範囲 / 次に見る点 / 返却タイミング に寄せています。実運用で進捗共有がやや平板になるならここを must_say 化すればよいですが、現時点では即修正必須までは言いません。

今は触らない方がよいもの
same_day_or_next_reply を共通 timing enum として docs / schema 全体に増やすこと。 これは今回の残件を直す方法としては重すぎます。共通 runtime interface と state-schema の語彙を増やすより、今回の 1 route を既存の next_reply に寄せる方が v1 fixed に合います。
facts_file など legacy metadata を今すぐ registry から消すこと。 service_pack_root はすでに入口として入り、README も runtime がそこから読む前提に揃っています。ここで legacy cleanup まで進めると、未知の参照先を壊す可能性の方が高いです。今は fidelity 直結ではありません。
回答例2の「危険箇所」まで今すぐ first-class contract 化すること。 たしかに公開文面にはありますが、今の route でも progress update の骨格は保持できています。ここは実返信で薄さが再発したら上げればよく、先に直すべきは route-state の未同期です。
route/state 整合専用の新しい lint や仕組みを足すこと。 役には立ちますが、いま必要なのは設計追加ではなく 1 箇所の同期修正です。v1 後は monitoring 主軸という repo の方針にも、その方が合っています。
