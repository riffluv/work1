1. 結論

判定：一部できている。総合 8.0 / 10 です。
内訳としては、bugfix-15000 live の公開運用準備は 8.4 / 10、将来アプリ化・他サービス展開できる返信判断コアとしては 7.3 / 10 くらいです。

かなり良いです。単なるテンプレ集ではなく、service registry / public-private gate / platform-contract / renderer / validator / regression / #RE / #BR が接続された「返信判断OS」になりつつあります。特に、bugfix-15000 は public:true、handoff-25000 は public:false と registry で分離され、通常 live では public:true のみ外部返信に出す、public:false はサービス名・価格・購入導線を出さない、というルールが明文化されています。

ただし、完成ではありません。主な未完成点は 3つ です。

renderer にまだ第二正本化リスクがある。
監査文書自身も、renderer 内に 15,000 系のリテラルが残っており、完全には動的化されていないと認めています。
service page → service facts → service-pack → renderer の意味一致は、まだ完全な semantic diff ではない。
check-service-pack-fidelity.py は facts / boundaries / decision-contract / evidence-contract / routing-playbooks / state-schema などへの参照解決を見ていますが、実質は「参照が存在するか」に寄っており、サービスページ文面と返信文の意味一致までは十分に保証していません。
将来コアは育っているが、まだ抽出されていない。
#RE / #BR は core training として有効ですが、現状の実装単位はまだ coconala / bugfix-15000 / Stripe / Next.js / 日本語ココナラ文体に深く結びついています。#BR は shadow と明記され、通常 live への漏れを禁止していますが、同じ repo・近い docs・近い renderer 周辺に語彙が存在するため、物理隔離までは弱いです。

なお、私はリポジトリ上の公開ファイルを読んで監査していますが、ローカル clone して regression を実行したわけではありません。pass 19 / fail 0 などは、監査文書に記録された結果として評価しています。

2. 良い構造：何が返信コアとして育っているか

一番良い点は、返信品質を「文体」ではなく 判断の層 として育てているところです。#RE は bugfix-15000 の live 返信品質確認、#BR は future dual-service の boundary routing 訓練、と目的が分かれています。#RE では price-only、vague相談、secret 値、購入前作業開始、複数不具合、purchased / closed 境界などを確認し、#BR では bugfix / handoff / neither / payment route / public leak / delivered / closed を分けています。

これはココナラ専用の表面訓練だけではありません。以下は将来サービス横断で使える core です。

育っている core	現状での現れ
service visibility 判定	public:true のみ外部返信、public:false は名前・価格・CTA禁止。
phase / state 判定	prequote / quote_sent / purchased / delivered / closed / cancelled を platform-contract が内部状態として定義。
route taxonomy	bugfix-first / handoff-first / neither / payment route separation を #BR で訓練。
allowed facts / forbidden moves	secret 値禁止、外部連絡禁止、直接 push 禁止、本番 deploy 禁止などを service facts と renderer が扱う。
reply quality lenses	主質問への直答、状態理解、buyer の次アクション、context bleed、direct answer first、closing present などが lint / changelog に入っている。
regression feedback loop	seed / eval / holdout / edge / regression_seed が eval-sources に整理され、active cases に public-private leakage、delivery completion、post-BR boundary、docs cleanup noise、live human review、public launch practical が並んでいる。

特に良いのは、#BR の学びをそのまま live に戻さない設計 です。handoff-25000 / 25,000円 / 主要1フロー は #BR 内では扱えるが、通常 live に戻す時は危険語彙として扱う、という線引きが明記されています。

3. 危ない構造
ココナラ専用に閉じすぎている部分

現状の core は、判断構造としては横展開可能ですが、実装の肉付けはまだかなりココナラ寄りです。platform-contract.yaml は、見積り提案、トークルーム、正式納品、承諾、クローズ後、追加支払い、おひねり、新規依頼など、ココナラ固有の transaction model を詳細に扱っています。

これは今の目的には必要です。むしろ live 安定化には強みです。ただし、将来アプリ化するなら、ここを core に上げてはいけません。core に上げるべきなのは closed そのものではなく、たとえば以下です。

ココナラ語彙	core 化するなら
トークルーム	transaction channel
正式納品	delivery submitted
承諾	buyer acceptance
クローズ	transaction closed
おひねり / 追加購入	in-transaction extra payment
見積り提案 / 新規依頼	new transaction route
ココナラメッセージ	out-of-transaction contact channel
将来展開時に足を引っ張りそうな部分

最も危ないのは、renderer が判断と文言を両方背負い続けること です。現状でも unified pipeline は state から lane を選び、lane ごとの drafter / validator を呼び、common validator と public leakage validator を通しています。これは良い骨格です。

一方で、post-purchase renderer には same_cause_rule、talkroom_only_rule、direct_push_rule、deploy_boundary_rule、secret_boundary_rule のような buyer-facing ルール文が直接入っています。これらは有用ですが、将来は service facts / platform adapter / renderer phrase のどれなのか分解しないと、renderer が第二正本になります。

もう1つ危ないのは、時間コミットの扱い です。post-purchase renderer には datetime.now(JST) + timedelta(hours=hours) から「本日HH:MMまでに」を作る関数があり、closed renderer 側にも「本日HH:MMまでに、見立てをお返しします」という生成があります。
実運用で現在時刻から生成するなら成立しますが、rehearsal / regression に固定時刻が残ると、stale deadline の事故になります。#RE batch 44 の B07 / B08 にも 本日15:02、本日15:15 のような具体時刻が出ています。

4. サービス記憶領域と返信システム接続の評価
十分な接続

接続はかなり強いです。service-registry.yaml は bugfix-15000 と handoff-25000 を分け、各サービスに facts_file、service_pack_root、source_of_truth を持たせています。さらに public replies use public_true services only、public_false services must not emit service name / price / purchase CTA、registry source_of_truth overrides pack assets というルールを持っています。

bugfix-15000.live.txt 側の本丸情報も、返信判断に必要な粒度で揃っています。15,000円、1不具合・同一原因、原因確認・修正作成・確認手順・修正済みファイル返却、複数原因なら事前相談、secret 値禁止、外部連絡・直接 push・本番 deploy 禁止が明記されています。

service facts 側にも、価格、scope unit、in-scope / out-of-scope、Stripe / Next.js / Webhook 関連 marker、必要情報、deliverables、fee policy、human review 条件が整理されています。

さらに recent audit で、prequote renderer が registry から public bugfix service grounding を読み込み、service_id / public_service / source_of_truth / facts_file / pack_root / display_name / base_price / fee_text / scope_unit / hard_no / out_of_scope を持つようになっています。

弱い接続

弱いのは 意味一致の検知 です。service-grounding audit は、接続自体はできているが、service-pack fidelity は「参照の存在確認」に寄っていて semantic proof ではない、と明示しています。

実装側でも、check-service-pack-fidelity.py は fidelity 対象ファイルとして facts / boundaries / decision-contract / evidence-contract / routing-playbooks / state-schema を許可し、参照がその pack 内で解決できるかを確認しています。これは重要ですが、「サービスページに書いた 原因だけ安く不可 が rendered reply で本当に守られているか」までは別テストが必要です。

また、registry の service id は bugfix-15000 ですが、facts file 内の service id は next-stripe-bugfix です。これは「registry id」と「internal facts id」を分けているなら問題ありませんが、alias として明文化しないと、将来サービス増加時に接続バグの温床になります。

第二正本化リスク

第二正本化リスクは 中程度 です。危険度が高い順に並べるとこうです。

価格・scope・deliverable の renderer 直書き
audit は 15,000 リテラルが残っていると明記しています。
post-purchase / closed renderer 内の buyer-facing rule 文
direct_push_rule、deploy_boundary_rule、secret_boundary_rule、closed_room_boundary_rule は有用ですが、facts / platform-contract との責務境界を崩す可能性があります。
#BR の shadow 語彙
#BR では 25,000円 や 主要1フロー整理 を使ってよいとされていますが、通常 live では出してはいけないと明記されています。論理分離はできていますが、語彙が近い場所にあるため、live renderer / skill / docs cleanup の sentry は継続必須です。
追加すべき最小テスト

追加は大きくしなくてよいです。最小で十分です。

追加テスト	目的
service-page-semantic-smoke	bugfix-15000.live.txt の価格、1不具合、同一原因、deliverables、secret 禁止、原因だけ安く不可が service facts と rendered reply に一致するかを見る。
