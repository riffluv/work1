1. 結論
promise_consistency は、正式な監査レンズとして入れる価値があります。
ただし、最初から hard validator / renderer rule / Writer 常駐ルールにするのは危険です。まずは #RE / #BR の外部監査プロンプトに soft lens として追加し、再発した subtype だけを ng-expressions / gold / lint へ落とすのがよいです。
今回の問題は、厳密には単純な local_semantic_contradiction ではありません。

「確実に直ることは約束できません」
→ 「修正済みファイルの返却まで進められます」

これは論理的には「不可能な矛盾」ではなく、buyer が読むと、後段の成果物表現が前段の留保を上書きして見える問題です。つまり本体は promise_consistency、下位現象として local_promise_shadow / success_guarantee_shadow と呼ぶのが適切です。
現行リポジトリ上でも、#RE は live bugfix-15000 の品質確認、#BR は handoff-25000 を future-dual simulation として分ける構造になっており、public leak / secret / phase / price・scope 崩れは candidate_source に関係なく必須修正扱いです。したがって、新レンズもこの hard boundary を弱めず、「文ごとは安全だが連続で読むと未約束の約束に見える」ものだけを拾う soft lensとして入れるのが自然です。
将来コアとしても、promise_consistency はかなり強いです。メール返信、SaaSサポート、受託案件、問い合わせ対応、カスタマーサポートのすべてで、**「未確定と言った直後に確定成果を約束する」「条件付きと言った直後に無条件に戻る」**は事故になります。ドメイン非依存レイヤになり得ます。
ただし、名前は promise_consistency を正式名にし、local_semantic_contradiction は内部 subtype 程度に留めるのがよいです。local_semantic_contradiction は広すぎて、「文体の不自然さ」「単なる接続の好み」「自然化の言い換え」まで巻き込みやすいです。⚠️

2. 今回の例の評価
今回の候補は hard fail ではなく、fix recommended に近いです。
理由はこうです。
観点評価成功保証を明示しているか「確実に直ることまでは約束できません」と否定しているので、明示保証ではないただし後段が保証っぽく見えるかはい。「原因確認から修正済みファイルの返却まで進められます」が、buyer には「直して返すところまで約束」と読めるservice fact と矛盾しているか直せる場合に修正済みファイルを返すこと自体は問題ない問題の本質無条件成果物表現が、直前の成功保証否定を弱めているseverityfix recommendedrule 化いきなり rule / lint ではなく、まず監査 prompt + gold 反映
元文の問題点

Stripe決済は成功しているのに注文が作られない症状であれば、15,000円の不具合修正として原因確認から修正済みファイルの返却まで進められます。

ここで危ないのは、原因確認 と 修正済みファイルの返却 が同じ強さで並んでいることです。


原因確認：購入後に始められる作業


修正済みファイル返却：修正箇所が特定でき、修正可能だった場合の成果物


この2つは promise level が違います。
それを「〜まで進められます」で一本化したため、「保証はしない」と言った直後に「でも直して返します」と見えるわけです。
暫定修正案の評価

ただ、Stripe決済は成功しているのに注文が作られない症状であれば、15,000円の不具合修正として、まず原因確認から進められます。修正できる箇所が特定できた場合は、修正済みファイルをお返しします。

これは適切です。かなり良いです。✅
さらに少しだけ強くするなら、こうです。
ただ、Stripe決済は成功しているのに注文が作られない症状であれば、15,000円の不具合修正として、購入後にまず原因確認から入れます。修正箇所を特定でき、こちらで直せる範囲であれば、修正済みファイルをお返しします。
差分は2点です。


**「購入後に」**を入れて phase を固定する

ただし、ここまで条件を足すと少し重くなるので、通常返信では暫定案のままでも十分です。

3. promise_consistency の定義案
正式定義は、こう置くのがよいです。
promise_consistency:  1通の返信内、または直近の会話文脈内で、  先に明示した「未確定・不可・条件付き・購入後・値不要・範囲外・先に相談」と、  後段の「対応します・進めます・直します・返します・確認します・送ってください・追加なし・原因です」などの promise-bearing 表現が、  buyer にとって同じ約束レベルとして整合して読めるかを見る監査レンズ。  1文ごとの事実誤りではなく、前後接続によって  成功保証、購入前着手、closed後作業、secret値要求、無条件追加対応、支払い導線、scope拡張などを  暗黙に約束していないかを確認する。
重要な分解
promise_consistency で見る対象は、単なる「矛盾」ではなく、次の promise ledger です。
種類例outcome promise直ります、解消できます、成功しますdeliverable promise修正済みファイルを返します、原因を特定しますphase promise購入前に見ます、購入後に始めますscope promise2件まとめて対応します、同じ原因なら対応しますpayment promise追加料金なし、そのまま対応しますevidence promiseログを見て判断します、コード確認後に返しますsecret promise値は不要です、.env を送ってくださいproduction promise本番反映します、反映後に確認しますsupport-state promiseclosed 後も直します、この場で続けますtimeline promise48時間以内に一次結果、今日中に修正完了
この lens の目的は、すべてを弱くすることではありません。
固定してよい約束は強く言う。
未確認の約束だけ条件を付ける。
これが中心です。

5. hard fail / soft lens / lint の分類基準
hard fail にすべき矛盾
次は hard fail でよいです。既存の acceptance gate でも、事実 gate / phase gate / prohibited action / scope 事故 / public-private 事故は major fail 扱いです。
hard fail例成功保証の明示購入すれば確実に直ります未確認なのに原因断定コードを見る前ですがWebhookが原因ですprequote で作業開始購入前にログを見て原因確認しますsecret 値要求sk_... の値を送ってください / .envをそのまま送ってくださいclosed 後の旧トークルーム継続作業クローズ後ですが、この場に送っていただければ直しますpublic:false サービスの導線漏れ通常 live / #RE で 25,000円のhandoffを購入してください本番反映・直接 push を約束こちらで本番に反映します追加料金・返金・無料対応の断定追加料金なしで全部対応します / 全額返金します同一原因未確認なのに複数件を定額確約2つまとめて15,000円で対応できます
fix recommended に留めるべき矛盾
これは今回の中心です。
fix recommended例成功保証は否定しているが、後段の成果物表現が無条件に見える保証はできません。ただ、修正済みファイルの返却まで進められます未確定と言った後、可能性表現なしで原因候補を強めすぎるまだ断定できません。Webhook周りが原因です購入後作業のつもりだが、文面上は購入前にも読めるログを見て原因を確認します追加料金相談のつもりだが、後段で無料吸収っぽい先に相談します。そのまま対応します本番反映はしないが、反映後確認の主体が曖昧本番反映はしません。反映後に確認します
preference として扱うもの
preference理由条件は明確だが、もっと自然な順序にできる実務リスクは低いまず の有無だけpromise ではなく語感対応できます vs 対応できそうです の好みservice facts 上、対応可能なら強く言ってよい文が少し硬いjp_business_native_naturalness 側で見る句点や段落の好みhard boundary と無関係
lint 化してよい高精度パターン
高精度 lint は、**「単語」ではなく「危ない組み合わせ」**で拾うべきです。


保証できません + 無条件の 修正済みファイルを返します


値は不要 + .envを送ってください


購入後に始めます + 購入前に原因確認します


closed / クローズ後 + 送っていただければ直します


本番反映はしません + こちらで本番反映します


追加料金は先に相談 + 追加分もそのまま対応します


lint 化すると危険なパターン
これは入れない方がよいです。🚫
危険な lint理由進められます 全般正しい service 可否まで落とす修正済みファイル 全般bugfix サービスの成果物説明を弱める対応できます 全般価格・範囲の明確さを失うログを見ます 全般purchased では正常反映後に確認します 全般「お客様側で反映後に確認」は正常原因です 全般購入後・証拠ありでは正常購入前に見ます 全般「相談文だけ見ます」は正常

6. 監査プロンプトへ入れる実文面
#RE / #BR の Codex xhigh 監査プロンプトには、以下をそのまま入れられます。
### promise_consistency（soft lens）1通の返信内で、先に置いた留保・不可・条件付き回答と、後段の作業可否・成果物・納期・料金・次アクションが、同じ promise level として整合しているかを見てください。1文ごとの事実が正しくても、前後をつなげて読むと buyer に次のような未約束の約束として見える場合は `promise_consistency` として指摘してください。- 成功保証はしないと言った直後に、無条件で「直します」「修正済みファイルを返します」と読める- 原因は未確定と言った直後に、特定原因を断定している- 購入後に始めると言った直後に、購入前のログ確認・原因確認・コード確認を約束している- secret 値は不要と言った直後に、`.env` や API キー値の送付を求めている- closed 後は旧トークルームで作業しないと言った直後に、送れば直すと読める- 追加料金は先に相談すると言った直後に、そのまま追加作業を進めると読める- 本番反映や直接 push はしないと言った直後に、こちらが反映・push するように読めるただし、この lens は hard fail ではありません。次の場合だけ hard fail に昇格してください。- public/private 事故- phase drift- secret 値要求- 成功保証・返金保証・無料対応の断定- scope / price / payment route の事実変更- 外部共有、直接 push、本番デプロイ誘導- closed 後の旧トークルーム継続作業の約束それ以外で、無条件に読めるが明示断定ではないものは `fix recommended` としてください。単により自然な順序にできる、語感を整えられる、という程度なら `preference` に留めてください。修正提案では、料金・scope・phase・secret・payment route・public/private 境界を弱めないでください。成功保証を避けるために、固定価格・対応可能範囲・購入後に開始できる作業まで曖昧にしないでください。望ましい修正は、曖昧化ではなく、条件・順序・対象を最小限足すことです。例:- `まず原因確認から進められます。修正できる箇所が特定できた場合は、修正済みファイルをお返しします。`- `購入後にログと関連コードを確認します。`- `.env は値を伏せ、キー名だけ分かる形で送ってください。`- `本番反映はお客様側でお願いします。反映後に出た結果は、この場で確認します。`出力時は、可能なら subtype を添えてください。subtype:- success_guarantee_shadow- deliverable_promise_shadow- diagnosis_assertion_drift- phase_promise_drift- secret_request_contradiction- closed_work_promise- scope_bundle_promise- payment_scope_promise- production_action_shadow
この文面のポイントは、hard fail 化しすぎないことです。現行プロンプトでも hard fail と soft lens を分け、conversation_flow_naturalness は hard fail ではないが、自然化によって price / scope / phase / payment route / secret handling / public-private boundary が変われば deterministic 事故として扱う、という整理になっています。この既存思想と同じ置き方で追加できます。

9. やらない方がいいこと
かなり重要です。ここを間違えると、返信OSの骨格が崩れます。
1. promise_consistency を hard fail にしすぎない
今回の例は hard fail ではありません。
hard fail にすると、Writer が全部を弱くしてしまいます。
悪い方向：
直るかは分かりません。対応できるかも確認後になります。金額も状況によります。
これは fixed price / service scope / buyer の判断材料を壊します。
2. 修正済みファイル を禁止しない
bugfix サービスでは成果物説明として必要です。
禁止すべきなのは、未確認・保証不可の直後に無条件成果物として出すことです。
3. 対応できます を弱めすぎない
15,000円で対応できます が正しい場面では、はっきり言うべきです。
すべてを 対応できる可能性があります にすると、商売上も UX 上も弱くなります。
4. renderer に全面実装しない
renderer は guardrail であり、全文テンプレ生成器に戻さない方がよいです。現行 skill でも、文面は Codex が freeform に近い形で下書きし、renderer / validator は guardrail として使う設計です。
5. lint を broad にしない
保証できません と 進めます が同じ返信にあるだけで警告する、みたいな lint は危険です。
正しい例まで落ちます。
確実な修正保証はしていませんが、購入後に原因確認から進められます。
これは正常です。
6. 自然化レンズへ吸収しきらない
conversation_flow_naturalness や jp_business_native_naturalness で拾える部分はありますが、今回の問題は「自然かどうか」ではなく、約束の整合性です。自然な日本語に直しても、約束が強すぎれば事故は残ります。
7. 成功保証回避のためにサービス内容まで弱くしない
この制約はかなり大事です。
promise_consistency は、固定事実を弱めるレンズではありません。
強く言ってよいもの：
15,000円です。購入後に原因確認から進めます。不具合1件として対応できます。
条件を付けるもの：
修正できる箇所が特定できた場合は、修正済みファイルをお返しします。同じ原因と分かれば、同じ不具合として扱えます。
