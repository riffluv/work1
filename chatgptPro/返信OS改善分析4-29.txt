入れるなら、既存 SKILL.md に 「会話の流れを作る（意味保存）」 という節を追加するのがよいです。
位置は、既存の fixed rules / NG rules の前後ではなく、自然化の実行手順の後、NG表現の前がよいです。理由は、これは禁止語ではなく、自然化時の判断 lens だからです。
追記案
## 会話の流れを作る（意味保存）日本語チャット自然化では、文章を「やさしくする」ことを目的にしない。目的は、buyer が主質問への答え・条件・次アクションを迷わず読めるように、事実を変えずに流れを整えること。### 基本順序原則として、次の流れを意識する。1. 短く受け止める2. buyer の主質問に直答する3. 条件・理由・変動要因を置く4. buyer または seller の次アクションを明示するただし、毎回この順序を機械的に固定しない。buyer が金額・可否・納期を聞いている場合は、前置きを短くし、先に直答する。### つないでよい文次のように、同じ役割で強く関係する事実は、意味が変わらない範囲で自然につないでよい。- 納期目安 + 変動条件  - OK: `日数はサービス上3日が目安ですが、コードの状態やエラー内容によって前後します。`- 受領 + 次の確認行動  - OK: `スクリーンショットは受け取りました。内容を確認し、追加で必要な点があればこちらからご連絡します。`- 範囲外 + 代替ルート  - OK: `その部分は今回の料金内では対応していません。必要であれば、進める前に追加対応としてご相談します。`### つないではいけない文次の事実は、接続によって因果や約束が増えやすいため、安易につながない。- 金額と追加対応- scope boundary と別料金- phase boundary と購入導線- secret safety と作業継続- public service と private / unpublished service- 作業可否と納期保証- 修正方針と成功保証NG:`範囲外なので別料金で対応します。`OK:`その部分は今回の料金内では対応していません。必要な場合は、内容を確認したうえで追加対応としてご相談します。`### 固定価格の扱いbase price が固定されている場合、価格を弱める表現に変えない。NG:`現時点では、15,000円での対応を想定しています。`OK:`金額は15,000円です。`OK:`15,000円で対応できます。``想定しています` は、価格・範囲・作業可否が source 上でも未確定のときだけ使う。### 句点の扱い句点「。」は全面的に減らさない。金額、納期、scope、断り、依頼、納品案内、secret safety、payment route では、誤読防止のため句点を残してよい。問題は句点そのものではなく、短い断定文が連続して会話の流れが切れること。短文断定が3文以上続く場合は、次のいずれかを検討する。- 同じ役割の2文だけつなぐ- 条件や理由を直後に置く- buyer / seller の次アクションを明示する- 不要な反復語を削る### 次アクションの書き方締めは `よろしくお願いいたします` だけにしない。誰が、何を、いつ／どの条件で行うかを明示する。OK:`この内容で進める場合は、そのままご購入ください。`OK:`ご購入後、コードとエラー内容を確認できた時点で、より正確な見通しをお返しします。`OK:`追加で必要な点があれば、こちらからご連絡します。`### 反復の扱い近い範囲で同じ語尾・同じ動詞を繰り返しすぎない。注意語:- `確認しました`- `確認します`- `確認結果`- `お返しします`- `進めます`- `ご案内します`ただし、禁止語にしない。意味が必要な場合は残し、近距離で密集している場合だけ言い換えや削除を検討する。例:`確認しました。確認します。確認結果をお返しします。`のような連続は避ける。### 自然化で絶対にしてはいけないこと- 金額を弱める- 納期保証を強める- scope を広げる- phase をまたぐ- secret / public-private / payment route を変える- 未公開サービスへの導線を足す- 約束していない作業を足す- buyer の主質問への直答を消す
この追記は、既存 skill の「自然化は facts を変えない」「service/scope/phase 再判断をしない」という方針と整合します。

## conversation_flow_naturalness（soft lens）この lens は、返信の意味・価格・scope・phase・secret・public/private・payment route を変えずに、buyer が自然に読めて次に動ける流れになっているかを見る。優先順位:1. service / scope / phase / price / secret / public-private / payment route の保持2. buyer の主質問への直答3. 次アクションの明示4. 日本語チャットとしての自然さ見る観点:- 短い受け止め → 主質問への直答 → 条件・理由 → 次アクションの流れがあるか- 短文断定が3文以上続き、マニュアル的・AI的に見えていないか- 関係が強い情報だけ自然につながっているか- 金額・納期・scope・作業可否が、自然化によって曖昧になっていないか- `確認しました` / `確認します` / `確認結果` / `お返しします` / `進めます` が近距離で密集していないか- 句点「。」そのものを悪いと判断していないか- buyer 固有の情報を1点拾えているか。ただし、拾うために事実を創作していないか- チャットなのにメール文体へ寄りすぎていないかHard fail:- 自然化により price / scope / phase / payment route / secret handling / public-private boundary が変わった- 未公開サービス、handoff、private 導線が live 返信に漏れた- 固定価格を `想定しています` などで不当に弱めた- buyer の主質問への直答が消えた- 次アクションが誤った導線になった- 約束していない作業、無料対応、返金、保証、直接作業、外部共有を追加したFix recommended:- 料金・納期・条件が `Aです。Bです。Cです。` のように短文断定で3文以上続き、少しつなぐだけで意味保存したまま自然になる- 受領確認や進捗報告で `確認しました` / `進めます` が密集し、何が終わって次に何が起きるか見えにくい- 最後が `よろしくお願いいたします` だけで、buyer または seller の次アクションが見えない- 段落がすべて1文ずつで、説明書のように切れている- `ご案内します` などが抽象的で、何を返すのか分かりにくいPreference only:- 句点の有無だけの好み- 1文がやや硬いが、意味・導線・安全境界が明確- 同じ構成に見えるが、今回のケースでは buyer が迷わない- より自然な言い換え候補はあるが、修正すると price / scope / phase が揺れそうな場合修正提案のルール:- 修正は最小差分にする- price / scope / phase / secret / payment route の語句は原則そのまま保持する- つなぐのは、同じ役割で強く関係する文だけにする- `やさしくするための曖昧化` を禁止する

gold 例：進捗報告
進捗をご共有します。現在、エラーの再現確認までは完了しています。修正箇所の切り分けにもう少し時間が必要です。次回は明日18時までに、確認できた内容と見通しをご連絡します。
gold 例：納品後の補足
修正版をお送りしました。今回の対応内容は、〇〇の修正と△△の調整です。表示崩れや想定と違う動きがあれば、このトークルームでそのままお知らせください。
gold 例：クローズ後の再相談
このたびはご依頼ありがとうございました。クローズ後に新しい不具合や追加のご相談がある場合は、まず症状と該当箇所をメッセージでお送りください。内容を確認したうえで、新規のお見積りとして対応可否をお返しします。
クローズ後だけは、Deep Research の「このトークルームでそのまま」系をそのまま使わない方がよいです。closed renderer は closed room boundary を明確に持っているため、phase をまたぐ表現は特に慎重に扱うべきです。
