## 背景

私はココナラで、Next.js / Stripe / API 連携の不具合修正サービスを運用しています。
理想は、

- 相手の文章を貼る
- AI がサービス内容・価格・スコープ・規約を踏まえて整理する
- 相手の主質問に直接答える
- 必要なときだけ最小限の追加質問をする
- 購入後は進捗返信や受領返信も安定して返せる

という状態です。

一番の課題は、日本語の微妙ないい回しではなく、

- 今あるサービス内容をちゃんと把握したうえで
- 相手の文章にしっかり critical に返しているか
- 答えるべき質問に答え、保留すべきものは保留理由つきで返せているか

です。

## 現在のサービス前提

現時点の公開サービスは 1 つだけです。

- `bugfix-15000`
  - 価格は 15,000 円
  - 対象は Next.js / Stripe / API 連携まわりの不具合修正
  - 基本は「不具合1件（同一原因 / 1フロー / 1エンドポイント）」の範囲
  - 原因切り分けと修正可否の確認を含む
  - 範囲外は、新機能開発、通話、直接 push、本番デプロイ、外部連絡誘導、外部決済誘導
  - `.env` や API キーの生値は扱わない

重要なのは、このサービス内容を踏まえずに、一般的なカスタマーサポート論だけで答えてほしくないことです。

## 現在の返信システムの構造

ざっくり 4 段です。

### Stage 1: intake / routing
相手文から次を判定します。

- どこから来た相談か
- 状態は何か
  - `prequote`
  - `quote_sent`
  - `purchased`
  - `predelivery`
  - `delivered`
  - `closed`
- 主力サービスに合うか
- リスクは高いか
- スコープは同一原因っぽいか / 別原因っぽいか / 未判定か
- case_type
  - `bugfix`
  - `handoff`
  - `boundary`
  - `after_purchase`
  - `after_close`
- certainty
  - `high`
  - `low`

### Stage 2: reply stance / contract
返信文を作る前に、返信姿勢と契約を決めています。

`reply_stance`:
- `answer_timing`
  - `now`
  - `after_check`
- `burden_owner`
  - `us`
  - `client`
  - `shared`
- `empathy_first`
  - `true`
  - `false`
- `reply_skeleton`
  - `estimate_initial`
  - `estimate_followup`
  - `post_purchase_receipt`
  - `post_purchase_quick`
  - `post_purchase_report`
  - `delivery`

`reply_contract`:
- `issue_plan[]`
  - `issue`
  - `disposition`
    - `answer_now`
    - `answer_after_check`
    - `ask_client`
    - `decline`
- `required_moves[]`
- `forbidden_moves[]`

### Stage 3: reply writer
上の判定を使って返信文を作ります。

### Stage 4: naturalize
最後に日本語を自然に整える工程があります。
ただし、ここでは本来

- 結論
- 質問項目数
- 価格
- 禁止事項
- 次アクション

を変えてはいけない想定です。

## 現在ある問題

仕組みはある程度ありますが、まだ次のズレが起きます。

1. 相手の主質問に直接答えず、説明や保留に逃げることがある
2. 上流で決めた stance / contract が、下流の writer で再解釈されて崩れることがある
3. purchased なのに、見積もり初回のような受付テンプレに戻ることがある
4. unnecessary ask が出る
5. vague hold が出る
6. 文として自然でも、「相手が本当に聞きたいことに答えていない」ことがある

## 現在すでにあるもの

次は既にあります。

- intent / state / scope の routing
- reply_stance
- reply_contract
- issue_plan
- NG 表現リスト
- self-check
- 最終自然化

つまり、完全なゼロベースではありません。
この状態で、なお足りない構造が何かを見たいです。

## 私の仮説

今の不足は、新しい言い回しや文体ではなく、

1. `answer-contract layer`
   - 相手の主質問・暗黙質問・保留対象を明示的に持つ層
2. `grounding packet layer`
   - service facts / state / scope / guard / forbidden claim をまとめた根拠パケット
3. `critic gate layer`
   - writer 後、naturalize 前後に「主質問へ答えているか」「禁止を踏んでいないか」を止める層

ではないかと考えています。

## あなたにしてほしいこと

この仮説を、かなり批判的にレビューしてください。

特に知りたいのは、

- 既にある `reply_contract` で本来足りるのか
- それでもズレるなら、schema が足りないのか、writer が自由すぎるのか
- 一番効く1手は `grounding_packet` なのか `reply_critic` なのか `section renderer` なのか
- 実務上、人間の微修正を減らすにはどこを deterministic にすべきか

## 出力形式

1. 総評
- 方向性の妥当性
- 一番危ない failure mode

2. 結論
- 最優先で足すべきもの 上位3
- 第一推奨案 1つ
- やらなくてよいこと

3. 構造レビュー
- 既にあるもの
- 既にあるが効いていないもの
- 新規追加が必要なもの

4. 具体提案
- 中間表現
- critic / validator
- renderer
- naturalize 前後のチェック

5. 実装順
- 今すぐやること
- 後回しでよいこと

6. eval 設計
- 最小セット
- 回帰テスト化すべき failure mode

7. 参照ソース
- 一次情報の URL
- どの結論がどのソースに依っているか

8. 最終判断
- もしあなたがこの案件の実装責任者なら、最初に 1 つだけ着手するものは何か
- それを選ぶ理由
- それ以外を後回しにする理由

## 制約

- きれいな理想論で終わらせない
- 第一推奨案を曖昧にしない
- 日本語の美しさより「主質問に正面から答える構造」を優先する
- 「すでにある仕組み」を見落として、的外れな追加提案をしない
- なるべく最小変更で効く案を優先する
- 出力は冗長にしすぎず、実装判断に使える密度で返す
