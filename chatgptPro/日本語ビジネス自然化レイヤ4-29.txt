2. 現状評価
強い点
今の構成は、かなり良い方向です。特に強いのは、安全境界と自然さを同じ場所で解こうとしていない 点です。
reply-quality の運用ループは、個別添削ではなく batch → 外部監査 → taxonomy → adoption policy → rule / validator / gold へ戻す形になっており、1回限りの好みを正本化しない設計がすでにあります。学習ループでも、case_fix / pattern_candidate / rule_return_candidate / preference / reject を切り、再発するものだけ skill / rule / validator / gold へ戻す方針になっています。
adoption-policy もかなり健全です。監査で見つかった違和感を毎回 rules に昇格させず、再発性・戻し先・副作用を見て、1回の改善では原則1論点だけ動かす方針になっています。自然化レイヤを強める時に一番危ない「気になる言い回しを全部NG化する」事故を、すでに制度として止めようとしています。
acceptance-gate も、事実 gate → phase gate → 会話 gate → 日本語 gate の順番になっており、日本語 gate が最後に置かれています。これは正しいです。自然さは大事ですが、public/private、scope、phase、secret より上に置いてはいけません。
危ない点
一方で、危ない点もはっきりあります。
1つ目は、自然さルールの戻し先が増えすぎて、同じ論点が skill / NG / lint / renderer / audit prompt に分散しやすいことです。
今回の 確認しました → はい、確認できています は、renderer 修正、NG追加、skill追加、lint追加まで行っています。これは今回のような再発性が高いものなら妥当ですが、この運用を全パターンに広げると、すぐにノイズ化します。
2つ目は、lint が一部広すぎる可能性です。
現行 linter では 進め方 を PM-style wording として拾う rule が見えますが、buyer が「この後の流れ」を聞いている場面では、進め方 や 流れ は自然にもなります。進め方 を blanket hard error にすると、今回のような「次の流れを教えてください」文脈で誤検知が出やすいです。
3つ目は、tone-template 側に古い硬さが残っています。
tone-templates.ja.md の標準テンプレには 確認でき次第、[日時]までに次の進め方をお返しします。、硬めテンプレには ご共有ください や 進め方は[手順]で進めます が残っています。今の自然化ルールから見ると、この参照ファイルは “現行 few-shot” としてはやや危ないです。
4つ目は、gold replies が増えてきていることです。
Gold の方針は「近い1本だけ見る」「完全テンプレではなく温度感・依頼数・既出情報の受け方・次アクションを合わせる」です。これは正しいですが、現行 README では初期セットが 35 本まで並んでいます。ここから先は、追加よりも current / eval_only / superseded の棚卸しが重要です。

3. jp_business_native_naturalness lens 案
追加すべきです。
ただし、hard fail lens ではなく soft lens です。
既存の監査プロンプトには、hard fail と soft lens を分ける思想がすでにあります。response_weight_mismatch や buyer_state_ack_gap は hard fail ではなく soft lens とし、hard fail は phase drift、非公開サービス漏れ、返金・無料・保証断定、外部共有・直接 push・本番デプロイ誘導など deterministic な事故に限る、とされています。この方針に jp_business_native_naturalness を足すのが自然です。
監査プロンプトへ入れる文面案
そのまま貼れる形で書きます。
### jp_business_native_naturalness日本の実務チャットとして、内容が正しいだけでなく、担当者が普通に書いたように読めるかを確認してください。これは hard fail 用の採点軸ではなく、soft lens です。全面改稿、好みの言い換え、過剰な共感、営業っぽい提案を増やすためには使わないでください。価格・scope・phase・secret・public/private・支払い導線・作業可否は変えず、既存の返信骨格を保ったまま、最小修正で改善できる違和感だけを見ます。特に見る点:- 同じ受領確認、同じ謝意、同じ可否、同じ結論を隣接文で二重に言っていないか- Yes/No 質問に機械的に毎回 `はい、` で入っていないか- `確認しました` の直後に `はい、確認できています` のような、受領確認の重ね書きがないか- `確認します / 確認できます / 確認結果` など、同じ語根が短い範囲に密集していないか- `確認材料` `進め方` `その時点で` `この中で` `公開サービス` など、内部処理語・PM語彙が buyer 向け本文で浮いていないか- buyer が聞いた主質問に答える前に、説明・手順・安全条件が並んでいないか- 受け止めを入れる場合、1文で止めているか。謝罪・過失認定・返金断定・無料対応約束へ広げていないか指摘する場合は、次の形式にしてください。- label: `jp_business_native_naturalness`- severity: `fix_recommended` / `preference` / `gold_candidate` / `lint_candidate` / `skill_rule_candidate` / `lens_only`- issue: 何が不自然か- buyer impact: buyer からどう見えるか- minimal patch: 意味を変えない最小修正- return target: `case_fix` / `ng-expressions` / `japanese-chat-natural-ja` / `lint_warning` / `lint_error` / `gold` / `none`- hard fail?: 原則 no。hard fail にする場合は、phase drift、scope事故、価格・保証・無料対応の誤約束、secret要求、public/private漏れ、既出情報の聞き直しなど deterministic な事故と結びつく場合だけ yes。

5. skill / ng-expressions に戻す候補
japanese-chat-natural-ja に入れるべきルール
skill は、lint より広く、ただし 意味を変えない自然化原則 として書くのがよいです。
追加・調整案
- 受領確認は1回で済ませる。`確認しました` `受け取りました` `届いています` のどれかを使ったら、直後に同じ意味の `はい、〜確認できています` を重ねない。
これは既に入っていますが、維持でよいです。
- Yes/No 質問でも、毎回 `はい、` で始めない。`届いています` `対応できます` `対象になります` のように、結論文だけで自然に答えられる時は `はい、` を省く。
これも既に方向性は入っています。強めすぎず、「毎回」「機械的に」だけ避ける が大事です。
- 同じ短い範囲で `確認` が続く場合は、片方を `見ます` `お伝えします` `届いています` `判断します` など、役割の違う語に置き換える。
これは skill 向きです。lint では誤検知が多いです。
- buyer が送ったものに触れる時は、generic な `ありがとうございます` より、可能なら `ログとスクショありがとうございます` のように対象を短く入れる。ただし謝意は1通1回までを目安にする。
今回の修正後の「ログとスクショありがとうございます。」は良い方向です。
- `確認材料` `その時点で` `進め方` など、内部処理やPM文脈に見える語は、buyer が聞いた言葉に寄せる。例: `確認材料` → `見る情報`、`その時点で` → `必要になれば`、`進め方` → `この後の流れ`。
ただし、進め方 は全面禁止にしない方がよいです。
ng-expressions に入れるべきもの
NG 集は、過去に直したのに再発しやすい表現 に限定するべきです。現行 NG 集もその目的で設計されています。
追加するなら、このくらいが安全です。
### 受領確認の二重化- 問題: `確認しました` の直後に `はい、確認できています` を重ねると、受領確認を二度言っており、機械的に見える。- 避けたい例: `昨日のログとスクショの件、確認しました。はい、昨日のログとスクショは確認できています。`- 推奨: `昨日のログとスクショは届いています。`- 推奨: `追加で必要なものがあれば、こちらからお願いします。`
### 謝意の近接重複- 問題: 冒頭と次段落で `ありがとうございます` が続くと、文面の型だけを当てた印象が出やすい。- 避けたい例: `ありがとうございます。ログありがとうございます。`- 推奨: `ログありがとうございます。`- 注記: generic 謝意と具体的謝意が離れていて、意味が違う場合は必ずしもNGにしない。
### `確認` の密集- 問題: `確認します / 確認できています / 確認結果` が短い範囲で続くと、意味は通ってもテンプレ感が出る。- 避けたい例: `確認しました。確認できています。確認結果をお返しします。`- 推奨: `届いています。内容を見て、結果をお返しします。`- 注記: 技術的に確認対象を明示する必要がある場合は許容する。

6. やらない方がいいこと
自然化レイヤを強める時に、特に避けるべき実装です。
1. 自然化レイヤに scope / price / phase を再判断させる
これは一番危険です。
自然化は、15,000円、不具合1件、public:false、secret 値不要、直接 push しない などを変えてはいけません。現行 skill にも、料金・保証・無料対応を自然化の過程で強めない、phase / service / scope / forbidden_moves の再判断をしない、とあります。
2. soft lens を hard fail にする
jp_business_native_naturalness を入れた直後に CI hard fail 化すると、確実にノイズになります。
最初は fix_recommended / preference / lint_candidate / gold_candidate を分けて、2回以上の再発または高リスク事故だけ昇格させるべきです。
3. はい、 や まずは を全面禁止する
これは過剰矯正です。
はい、 は自然な直答にも使えます。まずは も順番を示す時には自然です。問題は 機械的な固定化・近接重複・不安場面での進行圧 です。
4. 進め方 を全件 lint error にする
これはノイズ化しやすいです。
buyer が「この後の流れ」を聞いている時、進め方 は文脈次第で自然です。進め方になります、進め方を返します、その前提で進め方を のような PM 語彙化した組み合わせだけ拾うのが安全です。
5. human audit の指摘を即 rule 化する
人間監査は違和感発見に強いですが、rule 化判断とは別です。
現行 adoption policy にある通り、まず分類し、再発性・副作用・戻し先を確認してから反映すべきです。
6. gold をテンプレ化する
gold は「温度感・情報量・質問数・次アクションの基準」であって、文面のコピペ元ではありません。Gold README でも、近い1本だけ見て、内容をそのまま流用しない方針になっています。
7. 文体設定を先に広げる
今の段階で「やわらかめ」「かなり丁寧」「事務的」などを先に作ると、標準文体が固まる前に drift が増えます。
順番は 標準 = 日本のビジネス実務チャットとして自然 を固める → その差分として文体設定、が正しいです。
