9. やらない方がいいこと
1. 「最後の文を優先する」ルール
これは危険です。
最後の文が主質問とは限りません。
悪い設計:
final sentence = primary question
良い設計:
final sentence is one cue.primary question is selected by decision need + buyer priority + phase + explicit question.
2. 選び切る必要はありません の禁止語化
これは禁止語ではなく、構造の問題です。
禁止すると、判断し切らなくて大丈夫です のような別の教材語が出ます。
3. 毎回コード一式 ZIP を要求する
bugfix-15000 では有効ですが、prequote で前のめりに出すと重いです。
また、secret 混入リスクもあります。
4. echo_to_working_angle を hard lens 化する
まだ observe で十分です。
復唱が必要な場面もあります。禁止すると、逆に「読んでいない」返信になります。
5. reviewer prompt を Writer に渡す
これは Writer preservation を壊します。
Writer には短い brief、Reviewer には監査観点、lint には deterministic guard という分離を維持すべきです。
6. Gold reply を増やし続ける
Gold は selector がないとテンプレ化します。
今後は「増やす」より「近い1本だけ選ぶ」運用が重要です。
7. 同型 #RE の無駄回し
今回の発見を見る限り、自然化・コア化に効く発見は #RE より #R 実例から出ています。
同型 #RE を回すより、real_stock / #R / human audit を主軸にした方が費用対効果が高いです。

今は candidate です。
completion docs も scope seal も、sealed ではなく candidate として扱っています。
