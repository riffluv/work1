前提として、この実行環境では ChatGPT コネクター用の repo 直結ツールが露出していなかったため、公開 repo 側を参照しました。見た範囲では、現行方針は「骨格を壊さず、#RE → reviewer 監査 → 再発だけ戻す」にかなり明確に寄っています。公開中サービスは bugfix-15000、handoff-25000 は private / 非公開として扱う前提も、repo 側で一貫して確認できます。
1. 結論
今の #RE は学習ループとして成立しているか
成立しています。
ただし、「自動で賢くなる仕組み」ではなく、人間が gate を握る、監査型の学習ループとして成立しています。
成立している理由は明確です。


#RE の目的が「良い返信を1通作る」ではなく、既存 stock から batch を作り、外部監査で弱点を見つけ、再発するものだけ戻す運用になっている。


reviewer prompt 側でも、全面改稿ではなく、既存骨格を保ったまま監査する役割が明示されている。


batch 側に nonanswer / redundant_ask / premature_progress / hidden_rule / oversell / ai_tone の監査ラベルと、直答・scope・情報節約・自然さ・次アクションの採点軸がある。


直近の運用でも、必須修正を batch 上で潰し、再監査して、rule / renderer / 骨格には戻さず閉じた。この動きは、現行 adoption policy とかなり整合しています。


厳しめに言うと、今の #RE の価値は「返信生成の改善」よりも、「どの違和感を rule に戻さないかを判定する訓練」にあります。
ここを守れている限り、モグラたたきではありません。
モグラたたき化のリスク
一番危ないのは、reviewer の 必須修正 をそのまま rule 追加候補 と誤読することです。
必須修正 は「この返信は直すべき」という意味であって、「恒久ルールを増やすべき」という意味ではありません。
モグラたたき化の主な入口はこの5つです。
リスク起きること何が壊れるかlint 通過を品質通過と誤認文法・禁止語は通るが、主質問未回答を見逃すreviewer の存在意義が薄れるreviewer の趣味を rule 化自然さ・語感の好みがルールへ混入Writer が萎縮するrandom だけで batch を組む弱点軸が偶然に依存する再発判定が鈍る既存 rule hit を確認せず rule 追加同じ趣旨の rule が重複するself-check / skill が肥大化service facts と common core を混ぜるbugfix 固有の境界が汎用ルール化されるhandoff / common へ漏れる
現行 repo には「Writer に長い self-check を読ませない」「renderer は guardrail に留める」「自然化で意味・価格・禁止事項を変えない」という抑制線が既にあります。ここを崩さないことが、#RE を学習ループとして保つ条件です。

1. batch manifest を追加する
何を変えるか
返信監査_batch-XX.md の冒頭に、次のような短い manifest を置きます。
batch_id: RE-2026-04-XX-01目的: 主質問直答 / scope 境界 / secret handling の再確認件数: 6選定: curated 4 / random 2state_mix: prequote 2 / quote_sent 1 / purchased 2 / delivered_or_closed 1risk_axes: nonanswer, scope, secret, external-share, price-anxiety今回 rule 追加は原則しない。P0 safety / service facts mismatch のみ例外。
骨格を壊さない理由
batch の説明を増やすだけで、Writer・renderer・validator・service facts に触りません。
期待効果
reviewer の指摘が「この batch の狙いに対して重要か」を判断しやすくなります。後で「この batch は何を検査したのか」が残るので、学習ログとしての価値も上がります。
副作用リスク
curated 意図が強すぎると、未知のズレを拾いにくくなります。対策として、各 batch に random を1〜2件残します。
導入優先度: S

2. reviewer 出力に rule 戻し判断用の最小欄を足す
何を変えるか
現在の出力形式は大きくは良いです。足すなら、各 issue に次の4項目だけで十分です。
- 直す単位: case_fix / pattern_candidate / rule_return_candidate / preference / reject- 再発根拠: なし / 同 batch 内2件以上 / 過去 batch と同型 / 既存rule未実装疑い- 既存rule hit: あり（場所）/ なし / 不明- 戻し先候補: batch / reviewer_prompt / gold / rule / service_facts / validator / none
骨格を壊さない理由
reviewer prompt の出力形式に metadata を足すだけです。新しい layer でも self-check 追加でもありません。
期待効果
必須修正 = rule 化 の誤変換を防げます。特に今回の B02〜B05 のように、「既存 rule にあるが最終文に出なかった」タイプを切り分けやすくなります。
副作用リスク
reviewer が分類に意識を取られて、日本語の違和感検出が弱くなる可能性があります。項目は4つで止めるべきです。
導入優先度: S

3. learning log に “戻さなかった理由” を残す
何を変えるか
採用した修正だけでなく、採用しなかった reviewer 指摘も短く残します。
case_id: B04label: premature_progress修正: case fixrule戻し: なし理由: 既存 rule で quote_sent の押し感は既に扱われている。今回は文面上の適用漏れ。次回確認: quote_sent deadline / approval 待ち case を sentinel に1件入れる。
骨格を壊さない理由
学習ログの情報量を増やすだけです。生成系には触りません。
期待効果
「前も同じ話をした気がする」が可視化されます。将来、rule 化すべきだったかを再判定する時に、過去の保留理由が資産になります。
副作用リスク
ログが日記化すると重くなります。1 issue 5行以内を上限にした方がよいです。
導入優先度: S

4. 既存 rule hit check を必須化する
何を変えるか
rule に戻す前に、必ず次を確認します。
既存rule確認:- 同趣旨の rule はあるか- あるなら、今回の失敗は rule 不在ではなく適用漏れか- どこで落ちたか: classifier / writer / naturalizer / reviewer / batch facts
骨格を壊さない理由
新規 rule を増やす前のブレーキです。むしろ骨格肥大化を防ぎます。
期待効果

5. sentinel case を1件だけ混ぜる

前 batch で必須修正が出た archetype を、次 batch に1件だけ再投入します。

副作用リスク
sentinel を増やしすぎると、新規弱点探索ではなく過去問演習になります。1 batch 1件までが上限です。
導入優先度: A

危険な誘惑 ⚠️
一番危ない誘惑は、**「reviewer が賢いから、指摘を全部 rule にすればもっと良くなる」**です。
それをやると、前回壊れた方向に戻ります。
今の正解は、広げることではなく、弱点を見つける精度と、戻さない判断の精度を上げることです。
今回の #RE は、その方向にかなり近いです。骨格は触らず、batch・reviewer 出力・learning log の3点だけ締めるのが、いちばん強い改善です。
