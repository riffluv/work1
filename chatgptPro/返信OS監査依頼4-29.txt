1. 結論

現行OSは「完成に近い」が、まだ完成扱いにはしない方がいいです。
理由は、返信判断コアの骨格そのものはかなり正しく、壊すべきではない一方で、完成判定に必要な source-of-truth 整合 / shadow asset 漏れ防止 / delivery-completion 境界 / handoff-only public 化前検査 がまだ残っているためです。

特に critical に見ると、現時点の主リスクは「レンズ不足」ではありません。むしろ、レンズは足りてきているが、運用ファイル・gold・renderer・validator・batch mode の接続が一部まだ事故り得る という状態です。bugfix-15000 が public:true、handoff-25000 が public:false、通常 live / #RE では handoff 名・25,000円・購入導線を出さない、#BR だけ future-dual shadow rehearsal として扱う、という前提は service-registry / next prompt / AGENTS の方針と一致しています。

一番強く指摘したいのは、handoff-25000 の service page 側に、payment route と service boundary を混ぜる表現がまだ残っていることです。たとえば handoff ページ案には「追加1フローは15,000円」「修正が必要な場合は、同じトークルーム内で続けてご案内できます」という趣旨の記述があり、現行の platform-contract が定める「サービス境界と支払い導線を分ける」方針と緊張します。これは public:false の今は P0 事故ではありませんが、public:true 前の blocker です。

したがって結論はこうです。

大規模 redesign は不要。
ただし、完成判定には「レンズ追加」よりも、source-of-truth 整合、mode leakage 防止、delivery/completion regression、handoff-only rehearsal が必要。

2. 完成条件
2.1 実運用可能と言える条件

実運用可能の条件は、単に「返信が自然」ではなく、以下が揃っている状態です。

A. 外向けに出してよいサービスだけが出る

bugfix-15000 は public:true、handoff-25000 は public:false なので、通常 live / #RE では次が 0件 である必要があります。

handoff-25000
25,000円
handoff サービス名
主要1フロー整理サービスへの購入導線
bugfix と handoff の比較による誘導
#BR 用の future-dual 文脈

これは soft lens ではなく、deterministic validator / lint でよい領域です。service-registry.yaml も public:false service の名称・価格・購入CTAを outbound に出さないルールを持っているため、ここは hard gate にできます。

B. bugfix-15000 単体で、状態別返信が安定する

bugfix live の完成条件は、以下の state で hard fail が出ないことです。

state	完成条件
prequote	15,000円 / 1不具合 / 同一原因 / 購入前は見立てまで、を過不足なく言える
quote_sent	見積もり提案後だが、作業開始ではないことを崩さない
purchased	作業中・追加症状・別原因・新機能を混同しない
delivered	納品済みだがクローズ前の補足・確認・次アクションを誤らない
closed	旧トークルームでの追加作業・おひねり誘導を出さない
cancelled	作業継続・納品・追加導線を混ぜない

bugfix-15000 自体の service facts は、Next.js / Stripe / API連携の不具合1件、同一原因、原因特定・修正内容・確認手順・修正済みファイル返却、直接 push / 本番デプロイ / 外部連絡 / 秘密情報共有なし、という形でかなり明確です。

C. 「主質問直答 → 境界 → 次アクション」が自然に出る

過去の問題だった、

ぶつ切り感
宙ぶらりん感
取引状態を理解していない感じ
buyer が次に何をすればいいか分かりにくい感じ
安全だが重い・硬い返信

は、現在の writer-brief / scorecard / gold / learning-log でかなり対策されています。特に Writer に大量ルールを常駐させず、「主質問に先に答える」「buyer の言葉を拾う」「次アクションを1つに絞る」「状態認識を1文だけ置く」という設計は妥当です。

完成判定では、返信の構造をこう見るのがよいです。

1. buyer の主質問に直接答えている
2. 現在の取引状態を誤っていない
3. サービス境界を必要最小限だけ示している
4. buyer が次に何をすればよいか迷わない
5. platform route / payment route / service scope を混ぜていない
6. 温度感が過剰に硬くない
D. #RE と #BR の成果物が混線しない

#RE は bugfix-15000 単体の返信品質確認、#BR は future-dual shadow rehearsal です。この分離は完成条件に含めるべきです。#BR の学びを通常 live に戻す場合も、「汎用の境界学習」だけを戻し、handoff 名・25,000円・dual service routing 文は戻さない必要があります。

2.2 何が残っていると危険か

危険なのは次です。

handoff public:false 情報の外向け漏れ
#BR の文体・分類語彙が live / #RE に混ざること
handoff service page 内の payment route 表現が public:true 時にそのまま出ること
delivered / closed / formal delivery / completion gate の誤判定
buyer の主質問に答えず、安全境界だけ述べる返信への回帰
response_weight_mismatch を validator 化して、必要な安全境界まで削ること
platform 固有仕様を common core に入れて、他サービス展開時に壊れること
2.3 完成前に最低限通すべき rehearsal / regression

最低限はこの 5 系統です。

種別	最低限見るもの
public/private regression	通常 live / #RE で handoff 名・25,000円・購入導線が 0 件
bugfix-only #RE	prequote / quote_sent / purchased / delivered / closed / cancelled
delivery-completion regression	納品前後、正式納品、承諾前、クローズ後、追加相談
payment-route regression	same talkroom / おひねり / 有料オプション / 新規相談を state 別に分離
shadow leakage regression	#BR の dual-service 語彙が live renderer に流入しない
2.4 これ以上やっても学習効率が低い領域

以下は、今すぐ増やしても効率が低いです。

bugfix vs handoff の基本分類を同じ形で何十回も回す
既に通っている neither ケースの類似追加
response_weight を細かく採点軸化する
buyer_state_ack を文言テンプレ化する
Writer brief に reviewer レベルの観点を追加し続ける
gold を増やすだけで、superseded / current 管理をしない
3. 現行骨格の評価
3.1 維持すべき骨格

現行骨格はかなり正しいです。特に次は維持した方がいいです。

骨格	評価
Writer / Reviewer / renderer / validator / gold / learning-log 分離	✅ 維持。返信品質と安全性の両立に効いている
Writer に大量ルールを置かない	✅ 維持。自然な返信を守っている
validator は deterministic 事故だけ	✅ 維持。soft lens の硬直化を防いでいる
buyer_state_ack / response_weight は reviewer / gold 側	✅ 維持。hard validator 化すると壊れる
service facts を common core に混ぜない	✅ 維持。複数サービス展開の前提になる
#RE / #BR を生成モードではなく学習・境界検査として扱う	✅ 維持。mode separation として重要
rule 化は再発性・戻し先・副作用の低さ確認後	✅ 維持。過剰ルール化を抑えている

AGENTS.md でも、role separation、mode separation、runtime Writer の簡素化、learning-log / gold / validator の役割分担が明確にされています。これは返信OSの骨格としてかなり健全です。

3.2 現行骨格の弱点

弱点は、骨格そのものではなく、接続部です。

弱点A: source-of-truth の微妙なズレ

platform-contract.yaml は「service boundary と payment route を分ける」方針を明確にしており、closed talkroom / おひねり / 追加支払い / 正式納品前後の扱いもかなり丁寧です。

一方、handoff-25000.ready.txt には、追加1フローや同一トークルームでの続行案内がサービスページ本文に残っています。これは public:false の今は表に出ないため即時事故ではありませんが、将来 public:true にする前には必ず整理すべき source drift です。

弱点B: gold / renderer / first-pass の接続

learning-log では、soft lens の配置後も、初回案で古い三点セット・context bleed・purchased template misfire・delivered fallback drift が出た近接事故が記録されています。これは「レンズが足りない」というより、既存 gold へ戻る導線が first-pass でまだ弱い という問題です。

弱点C: delivery/completion 系の境界がまだ完成判定に足りない

platform-contract には delivered / closed / formal delivery / additional payment の扱いがかなり書かれていますが、完成判定としては、ここを batch regression として固定化した方がよいです。

4. 監査レンズの十分性
4.1 現在の lens 構成は足りているか

会話品質 lens と service boundary lens は、ほぼ足りています。
今足りないのは「新しい会話 lens」ではなく、以下のような meta / artifact / mode leakage 検査 です。

source-of-truth drift
shadow asset leakage
handoff public:true 前の service-page contract drift
delivery/completion regression density
first-pass gold connection

既存の scorecard では、transaction_model_gap、work_payment_boundary、free_support...、buyer_state_ack_gap、surface_overexposure、response_weight_mismatch などが補助軸として扱われ、点数化しすぎない設計になっています。これは妥当です。

4.2 既存 lens の配置評価
lens	現配置の評価	コメント
transaction_model_gap	✅ 妥当	hard validator ではなく、上位の状態理解 lens として残すのがよい
buyer_state_ack_gap	✅ 妥当	reviewer / gold / writer-brief が適切。validator 化は危険
route_clarity	✅ 妥当	buyer が次に迷わないために必要
buyer_not_lost	✅ 妥当	route_clarity と近いが、心理的迷子防止として残す価値あり
response_weight_mismatch	✅ 妥当	hard fail にせず、reviewer / gold 側に留めるべき

response_weight_mismatch と buyer_state_ack_gap は、learning-log でも reviewer / gold / writer-brief 側に置く方針で、validator / renderer へ入れない判断になっています。この判断は正しいです。

4.3 重複・肥大化・ノイズ化の危険

重複しやすいのはこのあたりです。

transaction_model_gap
  ├─ route_clarity
  ├─ work_payment_boundary
  ├─ buyer_not_lost
  └─ phase_payment_mismatch

transaction_model_gap は上位概念として残し、下位 lens は具体的な失敗分類として使うのがよいです。これをすべて個別 hard validator にすると、返信が「安全確認表」みたいになって、過去に問題だった硬さが戻ります。

また、buyer_state_ack_gap と温度感、response_weight_mismatch と surface_overexposure も近いです。これらは採点表で増やすより、gold のアンカー例で制御する方がよいです。Gold README でも gold はテンプレではなくアンカーとして扱う設計になっています。

4.4 既存 lens では拾いきれない blind spot

主な blind spot は 4 つです。

1. source-of-truth drift

返信そのものではなく、サービスページ / service-registry / platform-contract / gold の間で矛盾が起きる問題です。handoff ページの payment route 表現はこの典型です。

2. shadow asset leakage

#BR の shadow 文脈が通常 live / #RE へ混入する問題です。public_leak と近いですが、artifact の昇格経路まで見るなら別メタ検査が必要です。

3. first-pass gold connection gap

レンズはあるのに、初回返信案が古い型へ戻る問題です。これは新 lens より、gold family / renderer selection / reviewer prompt の接続改善で見るべきです。

4. delivery-completion micro-state

delivered、formal delivery、accepted、closed、cancelled、second formal delivery unclear の粒度です。platform-contract には材料がありますが、regression density はもう少し必要です。

5. 追加 lens candidate の有無

結論として、会話 lens の大幅追加は不要です。
追加するなら、通常の返信 lens ではなく、meta lens / artifact guard として最小限に留めるべきです。

Candidate 1: source_of_truth_drift

6. 不足している境界検査
6.1 route

不足しているのは「分類そのもの」より、分類後に何を言わないかの検査です。

分類	足りない検査
bugfix-first	buyer が整理・調査・仕様把握と言っても、実態が明確な不具合なら bugfix に戻せるか
handoff-first	buyer が「修正も」と言った瞬間に bugfix へ吸われすぎないか
bridge	まず修正か整理かを buyer に選ばせず、こちらで自然に境界を置けるか
neither	対象外でも冷たくならず、次に出せる情報を絞れるか

BR14 / BR15 でこの領域はかなり鍛えられていますが、今後は重複ケースより state と組み合わせた route を見る方がよいです。

6.2 state

不足している state 検査は以下です。

quote_sent だが作業開始前
purchased で追加症状が出たが、同一原因か未判定

delivered 後に別件を聞かれた
closed 後に同じトークルーム継続を求められた

7. #BR / #RE / handoff-only rehearsal の次アクション
7.1 #BR は今後どこまで回すべきか

BR13〜BR15 の結果を見る限り、basic boundary routing はもうかなり十分です。neither / payment route / service boundary は安定してきています。BR13〜BR15 では neither-heavy、handoff-positive、payment route separation が順に扱われ、BR15 では prequote / purchased / delivered / closed の payment route 境界が中心になっています。

次は量ではなく、以下の 高密度 edge だけでよいです。

same input, different mode
#BR では handoff を出す
#RE/live では handoff を出さない

handoff public:true 前提の negative rehearsal
ただし handoff service page cleanup 後
cheap-route mismatch
「まず安い方で調べて、違ったら高い方へ」圧力
bundle pressure
bugfix + handoff + 追加フロー + 修正をまとめたがるケース
もう十分な領域
明らかな新機能 neither
明らかな full repo / 仕様書化 neither
明らかな bugfix-first
明らかな handoff-first
基本的な 15,000円 / 25,000円 分離
closed で旧トークルームのおひねりを出さない基本形

#BR から通常 live / #RE に戻してよい学び

8. 壊してはいけない骨格

ここは強く維持です。

8.1 Writer を軽く保つ

Writer に大量の lens を持たせると、返信がまた硬くなります。現在の writer-brief は、主質問直答、buyer 語、状態認識、次アクション、重さ調整を短くまとめており、ここは良いです。

壊れるパターンはこれです。

Writer が毎回:
- state を説明する
- scope を説明する
- payment route を説明する
- public/private を説明する
- buyer state を拾う
- response weight を調整する
- naturalize する

これをやると、返信は安全でも「人間が読んで重い」状態に戻ります。

8.2 validator は hard accident 専用にする

validator に入れてよいものは、だいたい次です。

逆に、以下は validator に入れない方がいいです。

自然さ

宙ぶらりん感

これらは reviewer / gold / learning-log の領域です。

8.3 gold をテンプレにしない

Gold は anchor であって、返信テンプレではありません。この設計は維持すべきです。Gold README でも current / eval / superseded の管理と family index が置かれており、これは gold 肥大化の副作用を抑えるために重要です。

8.4 platform layer と common core を混ぜない

ココナラの talkroom / おひねり / 有料オプション / 正式納品 / クローズは platform layer に置くべきです。common core は、

platform-contract も、サービス境界と payment route を分ける方針を持っています。これは複数サービス展開時の中核になります。

Writer brief は「毎回の返信で思い出すこと」だけに留めるべきです。

9.3 gold を増やしすぎる危険

Gold を増やすほど、以下が起きます。

古い gold が復活する
current / superseded が曖昧になる
似たケースで異なる文体が混ざる
renderer がどれを anchor にすべきか迷う
reviewer が gold 準拠を過剰要求する
返信がテンプレ臭くなる

増やすなら、family 単位で統合し、superseded を明示するべきです。

9.4 rule 化してはいけない reviewer 指摘

以下は rule 化しない方がよいです。

「もう少し柔らかく」
「少し短く」
「不安に寄り添う」
「文頭を自然に」
「理由を添える」
「buyer の言葉を拾う」

これらは状況依存です。rule にすると過剰適用されます。

9.5 platform 固有仕様を common core に混ぜる危険

common core に おひねり、トークルーム、正式納品、見積り提案 を直接入れると、他サービス展開時に壊れます。

common core ではこう持つべきです。

state:
  - pre_contract
  - quote_sent
  - active_order
  - delivered_not_closed
  - closed
  - cancelled

payment_route:
  - not_allowed
  - only_if_buyer_asks
  - allowed_after_scope_confirmation
  - unavailable_after_close

ココナラ固有語は platform adapter で変換するのが安全です。

9.6 サービス facts を返信判断コアへ混ぜる危険

bugfix-15000 と handoff-25000 の価格・納品物・範囲・除外事項を common core に混ぜると、将来サービスが増えた時に判断コアが肥大化します。

core は、

service.public?
service.scope_match?
service.deliverable_match?
service.price_visible?
service.payment_route_visible?

だけを見て、facts は service pack から注入する方がよいです。

9.7 自然化ルールを増やしすぎる危険

自然化 rule を増やすと、次が起きます。

安全境界が薄まる
必要な断りが曖昧になる
buyer が次に何をすればよいか分からなくなる
「やわらかいが判断していない返信」になる
最後に reviewer が境界を足し戻して硬くなる

自然化は renderer / reviewer の微調整で十分です。
