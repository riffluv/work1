1. 結論
判定: かなり良い。
現行のサービス状態・記憶領域管理は、骨格としてかなり強いです。service-registry.yaml を公開状態の正本にし、外向け案内は public: true のみ、現時点の live は bugfix-15000 のみ、handoff-25000 は内部準備用という線引きが Core / AGENTS / Coconala boot / next prompt に重複して入っています。
ただし、事故入口はあります。最大の入口は、handoff-25000.ready.txt 自体が外向けサービスページ文面として完成度が高く、25,000円・追加1フロー15,000円・見積り回答例まで含む点です。public: false のガードを踏み外すと、そのまま外向け返信に混ざります。
広い再設計は不要です。
必要なのは、router / renderer / layer 追加ではなく、既存の service-registry → source_of_truth → service-pack → Writer / Reviewer の流れに、数個の最小ガードを足すことだけです。

2. 現行構造の強い点
公開状態管理
強いです。AGENTS.md と next-codex-prompt.txt の両方で、公開状態の正本は os/core/service-registry.yaml、外向け案内は public: true のみ、現時点の live は bugfix-15000 のみ、handoff-25000 は外向けでサービス名・価格・購入導線を出さない、と明記されています。
service-registry.yaml 側でも、bugfix-15000 は public: true、handoff-25000 は public: false、さらに public_reply_must_use_public_true_only と private_services_are_internal_reference_only が rules として置かれています。これはかなり良い正本配置です。
source_of_truth 管理
service-registry.yaml に source_of_truth があり、bugfix-15000 は bugfix-15000.live.txt、handoff-25000 は handoff-25000.ready.txt に紐づいています。AGENTS.md では、mirror / 同期用ファイルは補助にとどめ、公開判断や文言監査の基準にしない、と明記されています。
また、service-pack 内の source-of-truth policy では、公開サービスページを最上位、次に facts.yaml / boundaries.yaml、その下に decision / evidence / routing / seeds を置く優先順位が定義されています。これは「契約事実」と「返信運用」の分離として妥当です。
Core OS / mode 分離
強いです。Core OS では常時読むものを policy.yaml、service-registry.yaml、code-comment-style に絞り、mode ごとに coconala、implementation、delivery へ分岐しています。
Implementation OS は、購入後のコード分析・修正・実装専用で、返信系 docs を読まない、未公開サービス情報を参照しない、実装判断の正本は case README と一次証跡、と明記されています。これは返信文脈が実装判断に混ざる事故をかなり抑えています。
reply-only / common / delivery-only 分離
かなり良いです。next-codex-prompt.txt では、改善反映前に reply-only / common / delivery-only を必ず判定し、common 判定なしで相互流用しない、と固定されています。
guidance-scoping.ja.md でも、反映レーン、route、state、service / case_type の4軸で切れない改善は恒久反映しない、という判断基準があります。これは「よい改善が別サービスや別状態へ漏れる」事故に強いです。
#RE 学習ループ
かなり良いです。#RE は「返信を書く」ではなく、外部監査に回す batch を current rules で更新するショートカットとして定義され、デフォルトは bugfix-15000、handoff-25000 は #RE handoff のような明示時のみです。
さらに、reviewer 指摘は case_fix / pattern_candidate / rule_return_candidate / preference / reject などに分類し、必須修正をそのまま rule 化候補にしない、という導線があります。これは外部AIレビューの過剰採用を抑えています。
外部AI / Deep Research の扱い
強いです。外部調査ツールの取得情報はそのまま正本へ入れず、要約・選別した調査メモ経由で持ち込む、最終判断と正本反映は Codex が行う、という線引きがあります。
adoption-policy.ja.md でも、監査で見つかった違和感を毎回 rules に昇格させず、再発性・戻し先・副作用を見て採用する方針になっています。
Codex の推論保全
良いです。writer-brief.ja.md は、Writer に渡す既定をまず10行に絞り、self-check や ban list、platform 契約全文、監査議論を Writer に渡さないとしています。Reviewer は phase、public / private 境界、価格 / scope facts、forbidden moves などを見る役割に分離されています。
つまり、service facts は強く、本文生成は自由という設計になっています。これはかなり良いです。

事故

重大度
High


発生確率
Medium


最小対策

5. Codex の思考を殺していないか
Writer に渡す情報量は適切か
現行方針どおりなら適切です。
writer-brief.ja.md は、Writer へ渡す既定を10行だけにし、service facts、required / forbidden moves、few-shot 2〜3件までに抑えています。self-check 全文、ban list 全文、platform 契約の細部全文、監査議論は渡さない設計です。
危険なのは、運用で「念のため全部読ませる」に戻ることです。設計ではなく、実行時の読み込み過多がリスクです。
self-check や reviewer 観点が Writer に混ざりすぎていないか
混ざりすぎていません。Reviewer は phase、public / private 境界、価格 / scope facts、forbidden moves、ban 表現、closing、主質問への直答を見る役割として分離されています。
ただし、coconala-reply-self-check.ja.md は詳細リファレンスとしてかなり重いので、Writer に直接読ませた瞬間に文体が固くなります。これは避けるべきです。

10. 最後に
次の Codex は、まず service-registry.yaml を見て public:true/false を確定する。
外向けに出せる live service は、registry が許可したものだけ。
handoff-25000 は ready でも public ではない。価格・導線・サービス名を外へ出さない。
価格、範囲、納品物、FAQ、公開約束は source_of_truth が最上位。
Writer には最小 facts と短い brief だけを渡し、Reviewer で強く止める。
#RE は返信を書く運用ではなく、current rules で監査 batch を組み直す運用。
外部AIの指摘は、即 rule 化せず、case / pattern / rule / preference に分ける。
改善は大きく作り直さず、再発した事故だけを最小単位で戻す。
現行骨格は十分強い。壊さず、公開状態ガードだけをさらに硬くするのが正解です。
