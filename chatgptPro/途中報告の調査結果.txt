ココナラ向けの途中報告は、「原因確定」や「修正方針」を出す場面ではなく、「放置していない」「今どこを見ている」「次に何をする」を短く伝える場面です。公式上も、購入後はトークルームでやり取りし、出品者は納品時期や必要な追加情報に返信しながら進めます。評価項目には「コミュニケーション」と「納期/スケジュール」があり、トークルームには定型文機能があり、送信済みメッセージは原則取り消せません。なので、同じ骨組みを使い回しつつ、毎回1つだけ確認済み事実を差し込む設計がいちばん回しやすいです。

途中報告に最低限入れるべき要素

・今見ている箇所（例: 決済後の画面遷移 / Stripe webhook / API応答） — 着手済みが伝わり、放置感が消える
・現時点で確認できた事実 — 「調査中」ではなく「どこまでは分かったか」になる
・次に確認すること — 相手が待ち時間の意味を理解しやすくなる
・次の連絡目安、または必要情報 — いつ動きがあるか分かり、無駄な往復も減る

入れない方がいい要素

送信後は原則取り消せないので、途中報告は「薄い」より「言い切りすぎる」ほうが危険です。

・未確認の原因断定や「すぐ直ります」 — 外れた瞬間に信頼が落ちる
・進捗率だけ（50%、80% など） — 不具合修正では意味が薄く、何が進んだか分からない
・ログや技術用語の生貼り — 非エンジニアには重く、エンジニアにも要点が伝わりにくい

途中報告のタイミング

一番効果的なのは、初回挨拶の直後ではなく、「着手して一つ確認が取れた時点」です。再現確認・ログ確認・接続確認のどれか1つが終わり、「今見ている箇所」「今わかった事実」「次にやること」が1セットで言える瞬間がベストです。購入後は納品時期や必要な追加情報への返信が前提で、納品予定日もトークルーム上で共有できるので、次の連絡目安や依頼事項が見えた段階で出すと自然です。

出さない方がいいタイミングは、
・まだ何も触っていない直後
・一次報告がもう出せる直前
・推測しかなく、確認済み事実がゼロの状態
です。

推奨テンプレート

同じ骨組みを毎回使うこと自体は不自然ではありません。ココナラ公式も投稿文テンプレートを用意しており、トークルームには定型文機能があります。

汎用版
ご依頼の件、現在 [確認している箇所] を確認しています。
現時点で確認できているのは [確認できた事実] です。
この後は [次に確認する内容] を進めます。
[次回連絡の目安] に改めて状況をご共有します。
まずは進捗共有です。

追加情報を依頼する版
ご依頼の件、現在 [確認している箇所] を確認しています。
現時点で確認できているのは [確認できた事実] です。
続けて確認するため、[必要な情報 / 権限 / 再現手順] をお願いできますでしょうか。
いただけ次第、[次に進める作業] を行い、[次回連絡の目安] にご共有します。
分かる範囲で大丈夫です。

置き換えやすい語
[確認している箇所] = 決済後の画面遷移 / Stripe webhook / API応答 / 外部連携結果
[確認できた事実] = 決済完了までは正常 / webhook受信までは確認済み / API呼び出しまでは到達済み

悪い途中報告の例

例1
「現在調査中です。分かり次第ご連絡します。」
理由: どこを見ているのか、何が分かったのか、次にいつ連絡が来るのかが抜けていて、放置とほぼ同じに見える

例2
「たぶんStripe側の問題です。すぐ直ると思います。」
理由: 未確認の原因断定と楽観見積もりがセットになっており、外れたときに一気に不信感へ変わる

よくある失敗パターン

・テンプレを使うこと自体が目的になり、2行目まで毎回ほぼ同じで中身がない
・追加情報が必要と分かったのに、遠慮して聞かずに抱え込む
・1回だけ途中報告して、その後また長く無言になる

補足

1行だけ具体性を足すなら、「どこまでは正常で、どこから先を見ているか」という境界を書くのが最も有効です。Stripe では webhook の署名確認で raw body・署名ヘッダー・endpoint secret の整合が重要で、Next.js でも API Routes と Route Handlers の受け口を見分けて確認する場面があるので、技術名を増やすより“境界”を示したほうが伝わります。

使いやすい1行例
「現時点では、決済画面までは正常で、決済後の反映処理（Webhook〜API連携）のどこで止まっているかを確認しています。」

非エンジニア相手に重く見せないコツは、「Stripe webhook」より「決済後の自動反映部分」、「外部APIのレスポンス」より「連携先からの返り値」のように、技術名を“利用者が見ている流れ”へ言い換えることです。
