1. 総評

方向性はかなり近いです。
ただし、本丸は grounding_packet を厚くすることでも、reply_critic を増やすことでもありません。いま壊れているのは、上流で決めた判断が下流で再解釈可能なまま prose 化されていることです。OpenAI / Anthropic / Google の一次情報は、共通して「明示的な出力契約」「構造化出力」「タスク単位の eval」を重視しており、自由文のうまさに依存する設計を勧めていません。

一番危ない failure mode は、答えられる主質問を答えず、purchased なのに intake に巻き戻した、丁寧だが実質無回答の返信です。実務ではこれが一番痛いです。Intercom と Zendesk も AI の運用品質を、resolution / handoff / understood / automated resolution のような指標で見ており、文章美では見ていません。

第一推奨案

section renderer に絞ります。
ただし実装上は、reply_contract をそのまま使うのではなく、reply_contract v2（主質問と回答 disposition を持つ executable contract）を入力にした renderer にしてください。

Q2 の本丸

根本原因は schema 欠落 です。順位は次です。

schema 欠落
writer の自由生成度
上流判定と下流骨格の結線不足
naturalize での破壊

ここは少し重要です。
最初に着手する実装は section renderer ですが、一番支配的な原因は schema 欠落です。今の reply_contract は policy sheet であって answer plan ではありません。

やらなくてよいこと
answer-contract layer を reply_contract と別の真実源として増やすこと
単一サービス段階で、大きな grounding_packet layer を新設すること
free writer の上に reply_critic を先に載せること
semantic freeze 前に naturalize を強くすること

OpenAI / Anthropic / Google が structured outputs を推す理由は、「後から critic で叩く」より「最初から出力契約で縛る」方が安定するからです。

4. 具体提案
中間表現

最小構成はこれで足ります。

primary_question_id
explicit_questions[]
answer_map[]
ask_map[]

implicit_questions[] は今は要りません。
ここを先に入れると、勝手な善意解釈が増えて unnecessary ask が悪化します。必要なら後で latent_decision を 1 個だけ持てば十分です。

最小 schema
{
  "primary_question_id": "q1",
  "explicit_questions": [
    {
      "id": "q1",
      "text": "15,000円で対応可能ですか？",
      "priority": "primary",
      "source_span": "..."
    },

      "priority": "secondary",
      "source_span": "..."
    }

      "question_id": "q1",
      "disposition": "answer_now",
      "answer_brief": "bugfix-15000の範囲なら15,000円で対応可能",
      "evidence_refs": ["svc.price", "svc.scope", "conv.state"]
    },
    {
      "question_id": "q2",

ポイントは、defer_map[] を別に作らず、既存 enum を使って answer_map[].disposition に統合することです。
新しい層を増やすより、reply_contract をこれに進化させる方が安全です。

critic / validator

順番は deterministic lint → LLM judge です。

deterministic に検査すべきもの
state と skeleton の整合
例: purchased 以降で estimate_initial 禁止
価格 15,000円 の改変有無
scope 定義の改変有無
required_moves[] の欠落
forbidden_moves[] の発生
ask 数上限
answer_after_check に hold_reason と revisit_trigger があるか
.env / API キー生値要求の有無
外部連絡 / 外部決済誘導の有無
primary_question_id に対応する section が存在するか
LLM judge が必要なもの
主質問に正面から答えているか
ask が最小限か
hold が vague でないか
答え方が scope を曖昧化していないか
直接答えるべき所で逃げていないか

OpenAI / Anthropic / Google は eval を task-specific / rubric-based / automated-first に寄せています。judge も自由作文ではなく、binary or small-scale rubric に寄せるべきです。

結論として、writer を自由生成のまま改善するのは限界があります。
必要なのは reply_skeleton ごとの section renderer です。

naturalize 前後のチェック

naturalize は最後に残してよいです。
ただし条件があります。

raw の相手文を渡さない

8. 最終判断

私が実装責任者なら、最初に 1 つだけ着手するのは estimate_initial と post_purchase_quick の section renderer 化です。
正確には reply_contract v2 を作って、その入力から deterministic に section を組むところまでを 1 着手とみなします。

それを選ぶ理由
いまの主要 failure mode に最短で効く
purchased → intake 回帰を止めやすい
主質問を先頭で答える制約を入れられる
unnecessary ask を ask_map ベースで抑制できる
existing skeleton / required_moves / forbidden_moves をそのまま活かせる
それ以外を後回しにする理由
grounding_packet は今は大きな layer ではなく、小さい typed facts で足りる
reply_critic は free writer の誤りを後追いするだけで、根治ではない
eval dataset はすぐ作るべきだが、runtime を直接改善しない

要するに、あなたの仮説は「構造が足りない」という点では正しいです。
ただし、最初にやるべきは layer を増やすことではなく、既存の reply_contract を answer-executable にして writer から意思決定権を剥がすことです。
