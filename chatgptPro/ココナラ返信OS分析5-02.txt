4. #R と #RE の運用方針
#R 単発監査の役割
#R は、実務チャット上の違和感の発見装置です。
特に今回のようなものは #R が向いています。
- 商談温度が合っているか- 購入導線が押し込みに見えないか- buyer / seller の主体がズレていないか- 親切が見下しに見えないか- 材料案内が買う前提に見えないか- closed 後に「何をしてくれるのか」が分かるか
#R の監査では、毎回これを記録するとよいです。
audit_record:  case_id:  phase:  buyer_temperature:  primary_decision_need:  issue:  candidate_lens:  severity:    - hard_by_underlying_guard    - fix_recommended    - acceptable_as_is    - observe_only    - preference  return_target:    - oneoff_polish    - gold    - reviewer_prompt    - soft_lens    - lint    - pro_consult  recurrence_evidence:  reason_not_rule:
重要なのは、#R で見つかった違和感を即 rule 化しないことです。
#R は発見に強い一方、1件単位の文脈差・好み差も拾います。

#RE 回帰テストの役割
#RE は、採用済みルールと hard guard の退行検知です。
#RE が見るべきものはこうです。
- public/private 漏れ- bugfix-15000 / handoff-25000 の露出事故- 価格ミス- phase 崩れ- secret 値要求- external contact / external share / direct push- success / refund / free guarantee- closed 後作業 promise- primary answer 欠落- adopted subtype の構造退行
service-registry.yaml では、bugfix-15000 は public true、handoff-25000 は public false であり、public false はサービス名・価格・購入CTAを出してはいけないルールになっています。
このような事故は #RE の役割です。
一方で、#RE に「この言い方が少し押し込みに見える」「専門用語を減らすが見下しっぽい」まで背負わせると、テストが好み判定化します。

今後の回し方
おすすめの運用はこれです。
1. #R で 10〜20件の単発実務ケースを回す2. human audit で違和感をタグ化する3. その場で rule 化せず、return_target を決める4. 再発 2〜3件、または高重大度なら gold / reviewer / subtype 化5. 採用したものだけ #RE fixture に入れる6. #RE は hard guard と採用済み構造だけを見る7. change log に「何を変えたか / なぜ / 何で確認したか / ruleにしなかった理由」を残す
現行 change log は、変更ごとに分類・レイヤ・きっかけ・確認を残す形になっており、この運用と相性が良いです。

5. 未発見リスク候補
今回の10件から見ると、次に出そうな危険はこのあたりです。
1. sample_deliverable_vs_pre_diagnosis
サービスページ上は「購入前に納品物の見本を送れる」一方で、購入前に実コード診断や原因特定はしない前提です。service facts でも、サンプルは generic であり actual diagnosis ではないと整理されています。
危険文：
購入前に見本として、今回のコードを軽く見て原因候補をお返しします。
安全文：
購入前にお送りできるのは、納品物の形式イメージです。実際の原因確認は、お支払い完了後にログやコードを確認して進めます。

2. external_share_refusal_trust_gap
GitHub 招待や Drive 共有を断ったあとに、急に「ZIPで送ってください」と言うと、buyer には「なぜGitHubはだめでZIPはいいの？」が残ります。
安全文：
作業場所はココナラ上のやり取りに揃えています。お支払い完了後は、GitHub招待ではなく、必要なファイルをZIPまたは添付でトークルームに共有してください。
ただし、外向けに「ココナラ仕様でGitHubが常に不可」と断定しない方がよいです。platform contract でも、外部共有を避けるのは Workspace の安全運用であり、ココナラ仕様として断定しない方針です。

3. production_retry_prompt_risk
Stripe / Webhook / 本番だけ不具合で、buyer に再決済・Webhook再送・本番操作を促す危険です。
危険文：
もう一度本番で決済して、ログを送ってください。
安全文：
本番での再決済や再送は、こちらからはお願いしません。今あるログやスクショ、Stripe の Event ID が分かれば、その範囲から確認します。

4. closed_file_receive_vs_work_start
closed 後に「ファイルを送ってよい」と返すと、buyer は「修正までやってくれる」と読みやすいです。
安全順序：
送ってよい→ 見える範囲で関係確認する→ 実作業は別相談
これは relation_check_scope_clarity と同じ系列です。

5. ai_code_overcollection_secret_risk
AI生成コード・コード不明 buyer に「コード一式ZIP」を求めると、.env や APIキーをそのまま含めるリスクが上がります。
安全文：
コード一式をZIPで送る場合も、.env、APIキー、Webhook Secret、DB接続文字列などの値は含めないでください。.env ファイルは外すか、値を伏せた状態で共有してください。

6. acknowledgement_as_blame_shift
納品後・再発・不満相談で、
認識のずれかもしれません
のように言うと、buyer 側の誤解扱いに見えます。
安全文：
まず、こちらからお渡しした内容と、今起きている状態の差分を確認します。

7. quick_fix_or_light_check_scope_creep
buyer が「軽く見てほしい」「少しだけ確認してほしい」と言う時、seller が軽く受けると、購入前作業や無料確認に滑ります。
安全文：
購入前に原因確認だけを先に進める形ではありません。この症状自体は対応範囲なので、購入後に原因確認から進める流れでよければご購入ください。

8. primary_decision_need_hidden_by_last_question
最後の質問が「何を送ればいいですか？」でも、buyer の実質主質問は「購入してよいか」「15,000円で収まるか」「直らない時どうなるか」の場合があります。
