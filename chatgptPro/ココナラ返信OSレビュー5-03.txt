4. 構成順序の推奨
現状の、
状況要約↓対応可否↓条件↓購入後材料↓注意↓購入導線
は、安全性を守るには強いですが、低〜中リスクの初回相談では重くなりやすいです。
今回のような通常 prequote では、こちらがよいです。
主質問への直答↓必要な context anchor↓不安の核への返答↓購入後材料↓secret 注意↓次アクション
phase / risk 別の使い分けは以下です。
phase / risk推奨冒頭理由prequote / low risk / 依頼可否・価格質問直答から入るbuyer は買えるか・頼めるかを決めたいprequote / buyer が長文で不安多め直答 → 1文 anchor全要約すると bot 感。拾うのは核だけprequote / 支払い前診断・GitHub先見要求境界 → 対応範囲 → 条件付き CTA購入前作業開始を防ぐ必要がある。Gold 44 の領域 quote_sent / 購入判断中提案内容・前提への直答 → 購入後材料材料案内を先に厚くすると買う前提に見えるpurchased / 材料受領後受領確認 → こちらの確認動作ご購入後は に戻らない。phase drift 防止delivered / 軽い補足何を返すかを先に言う補足できます だけで止めないclosed / 関係確認送ってよい → 見る範囲 → 実作業は別相談closed 後境界が強いので、必要な summary / boundary を置くsecret / 外部共有 / 直接push / 本番反映境界を早めに置く自然さより hard guard 優先返金・無料・保証・責任認定不便認知 → 判断条件 → boundarysummary を削りすぎると危険。ただし否定語の復唱は避ける
重要なのは、**「状況要約を置くか」ではなく、「主質問に答える前に、どの anchor まで必要か」**です。

以下は単語禁止にしない方がよいです。
- `〜状態でも`- `この状態でも`- `その状態でも`- `ご記載の状況でも`- `今の状況でも`- `〜症状として`- `ご依頼いただけます`- 長い文そのもの
理由は、どれも自然に使える場面があるからです。
今回の問題は語ではなく、冒頭で複数 anchor を条件句に詰め、回答を後ろへ押し出す構造です。

10. やってはいけないこと
この改善は効きますが、過剰矯正すると別の事故が起きます。
1. 相手文を拾わなさすぎる
悪い例です。
ご相談ありがとうございます。15,000円でご依頼いただけます。ご購入ください。
これは短いですが、何に対して15,000円なのか、AI/外注コードや関係ファイル不明の不安に答えているのかが見えません。
2. price / scope / phase を弱める
自然化のために、以下へ逃げない方がよいです。
15,000円で対応できると思います。
固定価格・公開条件が決まっている場面では、思います は弱すぎます。サービスページと facts では、bugfix-15000 は public true、15,000円、不具合1件、原因確認・修正・修正済みファイル返却が公開条件として置かれています。  
3. ZIP一式 を全ケース hard rule にする
今回の buyer には適切です。
しかし engineer buyer が app/api/stripe/webhook/route.ts などを特定している場合は、関係ファイルで足ります。Gold 43 も 毎回コード一式 ZIP を hard rule にしない としています。
4. 購入前診断へ滑る
自然な材料案内を急ぐと、
購入前にZIPを送ってください。確認します。
のように崩れるリスクがあります。
prequote では、購入前にコード・ログを見て原因診断を始めない方針を維持する必要があります。facts でも prequote は対応可否・進め方の見立てまでで、コード一式の前のめり要求は避ける扱いです。
5. input_summary_overfit を hard fail にする
これは危険です。
返金・closed・secret・外部共有・複数症状のような高リスク場面では、短い状況整理が必要なことがあります。
6. 状態でも を禁止する
禁止すると、自然な返答まで壊れます。
原因が分からない状態でも相談できます。
これは自然な場面があります。問題は、長い条件句に複数事実を詰めることです。
7. handoff-25000 側へ逃がす
今回の相談は bugfix-15000 の公開条件内です。
AI生成コード・外注コード・コードの中身が分からないことは、bugfix-15000 サービスページ上でも対応可能とされています。handoff-25000 は public false なので、通常 live / #R / #RE へ出してはいけません。 
8. 人間っぽさのために根拠のない安心を足す
よくある症状なので大丈夫です。
これは避けた方がよいです。
自然さは、安心語を足すことではなく、答え・条件・次アクションを人間の順番で置くことです。
