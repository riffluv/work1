1. 結論
起きています。
現在の返信SYSTEMは、安全境界そのものはかなり守れています。一方で、#RE batch の B01/B03/B04/B08 では、外向け返信が 「buyer の質問に答える会話文」ではなく、「内部監査項目を順に満たす説明文」 に寄っています。
主原因は、writer-brief 単体ではなく、renderer / validator / gold / learning loop の合成圧です。
特に大きいのはこの構造です。

内部では
transaction_model_gap / completion_gate_gap / fix_vs_structure_first を reviewer lens として持つ
しかし実際の生成では
「同一原因」「別原因」「15,000円」「追加料金なし」「止めて説明」「正式納品しない」「購入後」「closed後は作業不可」などが、本文に複数個そのまま並ぶ

文書上は、transaction_model_gap や completion_gate_gap を reviewer-only / internal model として扱う設計が明記されています。completion_gate_gap も、6点すべてを外向けに出すのではなく、「15,000円の範囲で進める」「超える場合は止めて説明」「勝手に追加作業へ進まない」程度に圧縮する方針です。
ただし、現在の batch ではその分離が完全には機能していません。batch 自体も「completion_gate_gap を長くしすぎず答える」「transaction_model_gap を writer rule にせず lens として保つ」「handoff-25000 / 25,000円を漏らさない」ことを狙っていますが、B01/B03/B04/B08 では安全条件の積み上げが本文に残っています。
一言でいうと：

安全性は落ちていない。
ただし、安全条件を「内部で保持する」よりも「本文に出して証明する」方向へ寄っている。
その結果、自然な返信よりも監査通過文に見える場面がある、という状態です。


2. 重大リスク Top 5
優先リスク箇所何が起きているか影響1validator / lint「必要要素が入っているか」の検査が強く、短い自然文を落とす圧になっているchecklist 化の最大要因2renderer直答 + boundary + stop + fee + ask + next action + deadline のように部品が積み上がるツギハギ感、説明過多3gold replies「判断順序 anchor」のつもりでも、長めの gold がそのまま模倣されやすいbatch 文面が長文テンプレ化4compression rules の適用範囲prequote 向けの圧縮思想は良いが、post-purchase / delivered / closed まで十分に後段圧縮できていないB07/B08 で長くなる5learning loop の反復圧危険ケースを増やすたびに、本文へ安全条件を追加する方向へ学習しやすい回を追うごとに「監査項目列挙」化
2-1. validator / lint が「入っているか」寄り
check-rendered-quote-sent-followup.py や check-rendered-post-purchase-quick.py では、追加料金不安・completion gate・同一原因/別原因などに対して、複数の安全要素を明示的に満たす方向の検査が強くなっています。特に quote_sent 側の extra_fee_fear 系は、「追加作業」「事前相談」「完了条件」「キャンセル系の断定回避」などが束になりやすい構造です。
これは安全上は有効ですが、短い一文で安全境界を満たす回答を不利にします。
たとえば、

15,000円内で直し切れないと分かった場合は、追加作業へ進まず、そこでご説明します。

この1文で十分な場面でも、validator 側が複数要素を別々に要求すると、本文が長くなります。
2-2. renderer が guardrail ではなく説明部品を積み上げる
renderer は、buyer の質問、phase、service facts、closed/delivered/purchased 状態を見て、安全な返信を構成しています。ただし、共通 renderer 側では purchased / delivered などで closing を要求しやすく、closed 側でも直接回答・境界・確認依頼・見立て返答時刻が連結されます。
この設計だと、1つ1つの文は正しくても、最終出力では次のようになります。

直答


範囲説明


同一原因説明


別原因説明


追加料金説明


closed後の作業不可説明


資料依頼


見立て時刻



B08 がまさにこの状態です。
2-3. gold replies が長文テンプレとして作用しやすい
Gold 24 / Gold 25 は、文書上は「そのままテンプレ化しない」「判断順序を揃える」用途になっています。Gold 25 でも、completion gate の6点すべてを外向けに出すのではなく、必要部分だけ出す方針が明記されています。
ただし、実際の gold 例文は安全条件をかなり丁寧に並べています。3 API や closed 後の相談などは、batch の B04/B08 に近い長さ・構造で、そのまま「正解文」として吸われる危険があります。
gold は残すべきですが、gold body を正解にするのではなく、gold decision order を正解にする扱いへ戻した方がよいです。
2-4. compression rules は良いが、後段で負けている
prequote-compression-rules.ja.md はかなり良いです。主質問を先に置く、理由は1つ、next action は1つ、聞かれていない内部判断や例外条件を出しすぎない、という方向が明確です。
ただ、現在の問題は 「最初に短く書く」だけでは足りない ところにあります。
renderer が安全部品を積み、validator が要素不足を嫌い、gold が長めの例を与えると、compression rule が前段にあっても最終文は長くなります。
必要なのは、prequote だけでなく全 phase にかかる post-render compression です。
2-5. learning loop が安全条件を本文へ押し出しやすい
learning log では、「全部を renderer rule にしない」「dangerous boundary は常に本文へ出さない」「deterministic に検出できるものだけ validator 化する」という良い方針が残っています。
ただ、active case が増えるほど、batch 側では「この危険も避ける」「この境界も明示する」という圧が増えます。特に completion-gate 系と transaction-model 系の active cases は、いずれも安全上は重要ですが、本文が監査答案化しやすいケースです。

4. システム改善案
4-1. writer-brief に足すべきこと / 削るべきこと
writer-brief は、現時点でもかなり正しい方向です。
「主質問を先に答える」「buyer fact は1つ」「next action は1つ」「内部ラベルを出さない」「長い self-check や ban list を writer に渡さない」という設計になっています。
なので、writer rule を増やしすぎる必要はありません。足すなら 1つだけ でよいです。
追加案：

外向け本文に出す安全条件は、buyer の主質問に直接必要なものを最大2個までにする。
transaction_model_gap / completion_gate_gap の判断は内部で保持し、本文は原則 直答 → 必要な境界 → 次アクション に圧縮する。

削る / runtime から外すべきもの：


value proposition 系の長い説明


内部判断の理由説明


service facts の網羅的説明


FAQ 的な補足


writer-brief は「増やす」より、runtime で渡す情報を薄く保つのが重要です。

4-2. prequote-compression-rules に足すべきこと
現行の compression rules は良いです。
特に「理由は1つ」「next action は1つ」「聞かれていない補足や内部判断を削る」「FAQをそのまま貼らない」という思想は、そのまま維持すべきです。
足すべきなのは、prequote 専用ではなく、全 phase 共通の post-render compression です。
追加案：
Post-render compression rule:1. 主質問への直答は 1〜2文まで。2. 安全境界は原則 1〜2文まで。3. next action は 1文まで。4. 同じ安全意味を2回言わない。5. buyer が聞いていない返金・キャンセル・正式納品・別料金条件は出さない。6. closed / delivered / quote_sent でも、内部判断を本文に全部出さない。
さらに、boundary atom を分けるとよいです。
hidden:  reviewer が内部で見るだけ。本文に出さない。one_line:  必要なら一文で出す。  例: 15,000円内で直し切れない場合は、追加作業へ進まずそこで説明します。explicit:  buyer が直接聞いた場合だけ詳しく出す。  例: キャンセル、返金、正式納品、closed後作業、別原因の費用。

4-3. renderer で直すべきこと
renderer は、安全な返信を作る骨格としては有効です。
ただし、今は safety fragments を積み上げる方向が強いです。requires_closing や closed 後の direct answer / boundary / ask / deadline が重なることで、自然文よりも手続き文に寄ります。
直すべき方向：
A. safety atom を本文出力レベルで制御する
今のように、
同一原因別原因追加料金止めて説明closed後作業不可資料依頼時刻
を順に出すのではなく、
buyer question type = free_closed_followupsurface atoms:  - same_cause_check: one_line  - closed_no_work: one_line  - fee_path: hidden unless buyer asks price directly  - deadline: hidden unless already promised / purchased active work
のように、出すものを2個程度に絞るべきです。
B. post-render shrinker を入れる
renderer の最後に、以下を消す処理を入れるのが効果的です。


2つ目の next action


同じ意味の fee boundary


buyer が聞いていない cancel/refund/formal delivery


「まず」「この場合」「前提」だらけの内部判断文


deadline が不要な場面の時刻


C. B08 の closed angry/free 系は専用に短くする
closed 後の怒り文は、長い説明をするとさらに契約的に見えます。
default は次の程度でよいです。

まず前回の修正とつながる症状か確認します。

4-4. validator で直すべきこと
validator は safety を守るために残すべきです。
ただし、「全要素が入っているか」だけでなく、「余計な要素を出しすぎていないか」を見る必要があります。

6. まだ直さなくてよいもの
6-1. transaction_model_gap / completion_gate_gap の削除
削除しない方がよいです。
これらは reviewer lens として有効です。問題は lens の存在ではなく、lens の判断要素が本文に全部出ることです。failure taxonomy 側でも reviewer-only として整理されています。
6-2. writer rule の大量追加
writer-brief はすでに「短く、主質問から、内部ラベルを出さない」方向です。
ここにルールを増やしすぎると、さらに checklist 化します。
足すなら「本文に出す安全条件は最大2個」程度で十分です。
6-3. renderer / scenario の大量追加
scenario を増やすと、一見正確になりますが、長期的にはまた安全文の積み上げになります。
必要なのは scenario 追加ではなく、既存 scenario の出力を圧縮する後段処理です。
6-4. handoff-25000 / 25,000円側の外向け調整
ここは現状維持でよいです。
service registry では handoff-25000 が public: false で、public reply では public_true のサービスだけを使うルールになっています。
batch でも handoff-25000 / 25,000円の外向け漏れは見当たりません。
6-5. サービス facts の大幅変更
bugfix-15000 の公開 facts は、15,000円 / 不具合1件 / 同一原因 / 追加料金自動発生なし / 購入前は範囲・進行の見立てまで、という現在の安全境界と合っています。
変えるべきは facts ではなく、facts を本文に何個出すかです。

最終判断
現在のSYSTEMは、安全境界を守る能力はかなり高いです。
ただし、その安全性を「外向け本文に全部出す」方向へ寄せると、buyer からは親切ではなく 規約説明・契約説明・監査答案 に見えます。
直すべき中心は、writer の人格や文体ではありません。
最優先で直すべき設計点は、以下です。


validator を presence-only から、surplus warning 付きにする


renderer に post-render compression を入れる


gold を長文完成例ではなく、minimal outward line / expanded only if asked に変える


completion_gate_gap の外向け標準を1文にする


batch B01/B03/B08 を短縮して、今後の学習見本を短くする


この方針なら、
15,000円 / 不具合1件 / 同一原因 / 追加料金自動発生なし / 購入前調査不可 / closed後作業不可 / 非公開サービス非露出
を維持しながら、buyer には短く自然に答えられます。
