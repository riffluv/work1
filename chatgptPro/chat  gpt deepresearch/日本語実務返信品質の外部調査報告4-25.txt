
日本語実務返信品質の外部調査報告
結論
今回の調査は、公開リポジトリ内で直接確認できた正本・監査文書と、文化庁、消費者庁、Nielsen Norman Group、Microsoft、Google、Intercom、Zendesk、Stripe、OpenAIなどの公開ガイド・研究を照合してまとめたものです。公開リポジトリ側では、bugfix-15000.live.txt が外向け正本として扱われ、価格・範囲・保証まわりの事実が固定されており、監査文書・強化メモ・レビュー記録でも「現行骨格はかなり良い」「広い再設計は不要」「監査観点を強くする方向」が一貫しています。外部調査も、この方針をほぼ補強しており、結論は「大きな再設計は不要。#RE に入れる観察候補を増やし、reviewer で再発性を見極めるべき」です。 

#RE に追加すべき観察候補は、次の 8 個に絞るのが妥当です。いずれも「いきなり rule 化」ではなく、まず batch archetype と reviewer_prompt で再発確認するのが安全です。 

主質問直答の空白: 最初の 1〜2 文で相手の主質問に答えているか。説明や前置きが先に来ていないか。 
一息回答の分断: 可否・価格・範囲・次方針を、本来ひと続きで出すべき場面で不自然に分割していないか。とくに「対応できます。」単独から始まる bot 感を監査する。 
仕様質問のテンプレ流し: 価格、納期、保証、1件定義、原因不明時の扱いなどを聞かれているのに、不具合ヒアリングや資料送付依頼へ逃げていないか。 
保留だけ返して終わる弱さ: 「確認します」「見ます」だけで終わらず、今言えること・返答時期・必要な追加入力の最小単位があるか。 
既出情報の聞き直し: 相手が既に書いた症状・環境・不安を聞き直していないか。不要な 1 ラリーを増やしていないか。 
内部語・一般論化・同義反復: 「切り分け」「整理して返す」「対応範囲として見る」などの内部処理語、抽象語、意味の重複で AI 感・事務感を出していないか。 
事実上書き・仮説先走り: 相手が書いていない原因、固有名詞、環境名、既知障害らしき断定を持ち込んでいないか。 
安全境界の返し方: 秘密情報、外部連絡、外部決済、本番環境、直接操作、GitHub 共有のような境界質問に対して、禁止だけでなく安全な代替共有方法まで返せているか。 
大きな変更が必要かどうかについては、不要です。公開文書側でも「scenario 増殖」「新層追加」「renderer 増殖」は避けるべきだと明示されており、外部調査でも、効果が高いのは“答えを先に、内部語を減らし、価格・条件・次アクションを透明にすること”であって、分類器や新 router を増やすことではありません。今回の成果は、#RE の観察ラベル、batch archetype、reviewer_prompt 候補、learning-log 候補として扱うのが最も整合的です。 

外部調査で見つかった重要パターン
冒頭の結論空白

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 受け手が知りたいのはまず「それは頼めるのか」「いくらか」「どうなるのか」なのに、前置き・背景説明・運用説明から入ると、答えているようで答えていない印象になります。外部ガイドでは「最重要情報を先に」「実際の answer がない応答は解決とみなさない」が一貫しており、仕様質問や見積り相談では特に違和感が強くなります。 
悪い例: 「ご相談ありがとうございます。状況に応じて確認方法が変わるため、まずは症状やログを共有いただけますと幸いです。」
良い返し方の方向: 先頭で主質問に短く答え、その後に必要最小限の条件や補足を続ける。価格質問なら価格、保証質問なら保証、可否質問なら可否を最初に置く。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: primary_q_before_context。nonanswer, oversell。反映先は batch archetype, reviewer_prompt, learning-log。今すぐ rule 化はしない。まず「どの質問型で再発するか」を batch で確認するのが先です。 
一息回答の分断

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 可否・価格・条件が本来ひとつの返答なのに、「対応できます。」「15,000円です。」「詳細は確認後です。」のように細切れにすると、情報は合っていても bot 的に見えます。リポジトリの監査プロンプトでも、可否だけを独立文にしすぎない点が明示的に重視されており、外部の明快な文章ガイドとも整合します。 
悪い例: 「対応できます。15,000円です。まず内容を見てから判断します。」
良い返し方の方向: 「対応できます。今回の内容が不具合1件の範囲なら 15,000円で、まず原因確認から進めます。」のように、同一答えの要素は一息でまとめる。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: one_breath_answer_packaging。one_breath_failure, ai_tone。反映先は batch archetype, reviewer_prompt。今すぐ rule 化はしない。絶対規則にすると不自然な圧縮を産むので、監査観点として薄く持つ方が安全です。 
仕様質問のテンプレ流し

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 価格、納期、保証、範囲、1件定義、原因不明時の扱いを聞かれているのに、「まず症状を教えてください」「ログを送ってください」で返すと、相手は“質問を読んでもらえていない”と感じます。公開正本には FAQ がかなり明確にあり、強化メモでも「答えられることを答える前に聞き返しへ逃げない」が原則です。 
悪い例: 「まずエラー内容と関係ファイルを ZIP で送ってください。確認後に料金や範囲をお伝えします。」
良い返し方の方向: まず既知の仕様事実で答える。その後で、判断に必要な追加確認が本当にある時だけ、追加質問を 1〜2 個に絞る。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: service_spec_answer_before_intake。nonanswer, redundant_ask, premature_progress。反映先は batch archetype, reviewer_prompt, gold。rule 化は条件付き。同型再発が 2 batch 以上で出たら候補になります。 
保留だけ返して終わる

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 「確認します」だけの返答は、誠実そうに見えて、実際には相手の不安をほとんど減らしません。日本語のビジネスチャット実務でも、すぐ答えられない場合は一報に加えて回答時期や次動作を添えるのが望ましいとされます。顧客対応ガイドでも、すぐ解決できなくても personalized な acknowledgment が必要だとされています。 
悪い例: 「確認します。少々お待ちください。」
良い返し方の方向: 「確認を進めます。現時点では X までは見ています。追加で必要なら Y をお願いします。今日中に一次結果を返します。」のように、状態・次手・時期のうち最低 2 つを入れる。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: ack_without_commitment。nonanswer, premature_progress。反映先は batch archetype, reviewer_prompt, learning-log。今すぐ rule 化はしない。state ごとの差があるため、まず post-purchase 系 batch で観察すべきです。 
既出情報の聞き直し

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 相手がすでに「本番だけ」「昨日から」「決済は通るが反映されない」と書いているのに再度聞くと、読み飛ばし感が強くなります。リポジトリ側ではこれを品質事故として扱う方向が明示されており、日本語チャット運用でも「今いいですか？」のような不要な 1 ラリーはストレスになるとされています。 
悪い例: 「どこで起きていますか？　毎回ですか？　何の不具合ですか？」
前便で全部書かれているケース。
良い返し方の方向: 既知 fact を短く回収してから、不足している blocking fact だけ聞く。聞き返す数は最小限にする。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: known_fact_reask。redundant_ask, ai_tone。反映先は batch archetype, validator, reviewer_prompt。rule 化の候補ですが、まずは warn-only が安全です。 
内部処理語の露出

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 「切り分け」「ハンドリング」「スコープとして見る」「対象候補」「レンダリング」など、作業者側の整理語が表に出ると、受け手には“あなたの都合の言葉”に見えます。外部 UX ガイドは、利用者の言葉で話すこと、内部 jargon や曖昧語を避けることを強く推奨しています。 
悪い例: 「まず切り分けの上で、今回のスコープに入るかをハンドリングします。」
良い返し方の方向: 内部語を、相手が受け取れる結果語へ置き換える。「まず何を見るか」「今回の修正とつながるか」「対応できるか」に変換する。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: internal_process_exposure。ai_tone, hidden_rule。反映先は reviewer_prompt, learning-log, gold。今すぐ rule 化はしない。語そのものではなく、受け手作用で見るべきです。 
定型句と同義反復の連打

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 「承知しました」「確認します」「整理してお返しします」「ご相談ください」「対応可能です」が毎回似た順番で並ぶと、単体では問題なくても batch で bot 感が出ます。外部 CS ガイドでも、stock answers や overused phrases は見抜かれやすく、AI 文章研究でも“より形式的・構造的・反復的”な傾向が指摘されています。 
悪い例: 「承知しました。確認します。整理してお返しします。ご相談ください。」
良い返し方の方向: 相手が今ほしいことに合わせて、定型句を削る。繰り返し効かせたいのは礼儀ではなく意味で、同じ意味の文は 1 回で十分です。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: semantic_echo_template_loop。ai_tone, one_breath_failure。反映先は batch archetype, reviewer_prompt, learning-log。rule 化は原則しない。語単位 NG ではなく、再発パターンとして監査するのが妥当です。 
顧客が書いていない事実の上書き

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 相手がまだ言っていない環境名、原因名、イベント名、既知障害らしき推定を持ち込むと、親切どころか“勝手に話を作っている”と受け取られます。支援 AI の外部ガイドでも、明確に分からない時は分からないと言い、利用可能な根拠に基づくべきだとされています。具体語は、相手の言葉を鏡のように受ける方が信頼に寄与します。 
悪い例: 「Webhook 署名検証エラーの可能性が高いです。」
相手は単に「決済後に反映されない」としか書いていない。
良い返し方の方向: 事実と推定を分ける。推定するなら断定せず、「まずは A と B を確認したいです」と確認方針へ寄せる。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: fact_overwrite_or_guess。hallucination, scope_drift, ai_tone。反映先は batch archetype, reviewer_prompt, validator。rule 化は慎重。安全寄りで warn-only から始めるべきです。 
価格・保証・返金・キャンセル条件の濁し

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 価格・保証・キャンセル・返金条件は、消費者側にとって最重要情報です。これを小さく、後ろに、曖昧に、もしくは口頭の雰囲気で済ませると、一気に不信感が出ます。公的ガイドでも、価格や返品条件は見やすく明瞭で、埋没しない形で示す必要があるとされています。 bugfix 正本も、原因不明で修正方針につながらない場合は正式納品へ進めず、キャンセルを含めて相談すると明確化しています。 
悪い例: 「内容次第では別途ご相談になります。」
何が別途か、いつ決まるかがない。
良い返し方の方向: 既知の条件を先に明示し、未確定部分だけを未確定として残す。「15,000円固定」「別原因だけ追加相談」「自動加算なし」「原因不明のまま正式納品にしない」を曖昧にしない。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: commercial_anxiety_answering。hidden_rule, nonanswer, oversell。反映先は batch archetype, reviewer_prompt, service_facts, gold。rule 化候補ですが、正本にある事実の再現に限るべきです。外部知見そのものを rule へ入れるのは避けるべきです。 
境界質問で安全と利便が混線する

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: .env、API キー、GitHub 招待、本番環境、Vercel/Stripe 権限、通話などの質問で、「大丈夫です」だけでも「できません」だけでも弱いです。安全制約と代替手段を同時に返せないと、不親切か、逆に無警戒に見えます。ココナラの公式ルールでは外部連絡先・外部決済・外部誘導が禁じられ、Stripe も secret key の共有を避けるよう明示しています。 bugfix 正本では、.env はキー名のみ、外部共有や本番反映はしない、と明文化されています。 
悪い例: 「APIキーを送ってください。こちらで本番も確認します。」
良い返し方の方向: 禁止 / 不要 / 代替共有方法を 1 セットで返す。「値は不要」「キー名のみ」「必要ファイルは ZIP」「本番反映は依頼者側」を一息で返す。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: boundary_safe_alternative。scope_drift, hidden_rule, premature_progress。反映先は batch archetype, reviewer_prompt, gold, validator。rule 化は一部可。秘密情報と外部誘導は安全性が高いので、正本事実に沿う範囲で戻せます。 
マニュアル敬語と過剰な丁寧語

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 日本語では、丁寧だから良いとは限りません。文化庁系の説明でも、これからの敬語は必要以上に煩雑でなく、平明・簡素であることが望ましいとされ、最近の実務解説でも“マニュアル敬語”の画一性は相手に不快感を与えうると整理されています。チャット文脈でも、必要以上の体裁や挨拶の持ち込みは本題到達を遅らせます。 
悪い例: 「ご確認のほど、何卒よろしくお願い申し上げます。恐れ入りますが、ご共有いただけますと幸いでございます。」
良い返し方の方向: 「です・ます」を基調にしつつ、相手を立てるより意味を通す。丁寧さは維持し、語尾の重ね掛けを減らす。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: manual_keigo_chat_mismatch。ai_tone。反映先は batch archetype, reviewer_prompt, learning-log。rule 化はしない。文脈依存が強く、好み差との境界が近いからです。 
逃げた締めと閉じの不完全

何が問題か / なぜ AI っぽいか / どの場面で起きやすいか: 本文ではある程度答えていても、最後が「ご相談ください」「ご共有ください」「必要であればお知らせください」だけだと、受け手は“で、今どう動けばいいのか”が残ります。外部 CS ガイドでは、会話の終わりに invite だけでなく、解決に向かう入口を残すことが重視されています。リポジトリの live summary でも closing_complete が恒久観点として採用済みです。 
悪い例: 「必要でしたらご相談ください。」
良い返し方の方向: 最後は、相手が次に送るべきもの・待てばよいのか・こちらがいつ返すのか、のいずれかを残す。締めは丁寧語ではなく進行補助で考える。 
#RE archetype / failure label / 反映先候補 / 今すぐ rule 化: closing_complete_or_weak。ai_tone, premature_progress。反映先は batch archetype, reviewer_prompt, learning-log。rule 化は様子見。state ごとの差が大きいからです。 
#RE batch 案
batch_id案: re_spec_prequote_01

目的: サービス仕様質問に対して、主質問へ直答したうえで price/scope/guarantee を透明に返せるかを見る batch です。特に「価格を聞かれたのにヒアリングへ流す」「可否と条件が分断する」「FAQ 既知事実を曖昧化する」を潰すのが狙いです。 
state_mix: prequote x6
risk_axes: direct_answer_first, one_breath_packaging, commercial_clarity, no_hidden_rule, no_premature_intake
入れるべき相談 archetype 6件
何時間で復旧できますか
5分で分かっても15,000円ですか
これは不具合1件に入りますか
コード問題か設定問題か分かりますか
原因が分からなかったらどうなりますか
保証期間はありますか
reviewer が見るべき観点: 先頭 2 文以内に主質問の答えがあるか。FAQ の既知事実で答えられるのに「送ってください」へ逃げていないか。価格・追加料金・キャンセル条件の出し方が濁っていないか。可否だけ独立して bot 感が出ていないか。 
rule に戻す条件: 同じ仕様質問型で nonanswer または hidden_rule が 2 batch 以上継続し、しかも公開正本に答えがある場合。
rule に戻さず観察で止める条件: 問題が語尾や段落切りの好み差に近く、service facts の誤りや buyer 不利益がない場合。 
batch_id案: re_boundary_share_01

目的: 秘密情報・共有方法・本番環境・外部接触まわりの返答を、安全だけでなく「代替共有方法まで含めて自然に返せるか」で監査する batch です。禁止だけを返す壁感と、利便優先で境界を緩める事故の両方を避けます。 
state_mix: prequote x4, purchased x2
risk_axes: boundary_safety, alternative_path, platform_compliance, no_secret_request, no_external_routing
入れるべき相談 archetype 6件
.env と APIキーを送ればよいですか
GitHubに招待した方が早いですか
本番環境に入って直してもらえますか
Vercel / Stripe の権限を渡してよいですか
通話で説明できますか
ZIPだと何を入れればよいですか
reviewer が見るべき観点: 禁止事項だけで終わっていないか。値は不要・キー名のみ・ZIP 代替・本番反映は依頼者側、のような安全代替が一息で出ているか。ココナラ外の連絡・決済・誘導に触れていないか。 
rule に戻す条件: 秘密情報要求、外部連絡誘導、外部決済示唆、本番直接操作の約束が 1 件でも出た場合。これは安全・規約・期待値のリスクが高いからです。
rule に戻さず観察で止める条件: 代替共有方法の説明順や、GitHub と ZIP のどちらを先に言うかなどが好み差にとどまる場合。 
batch_id案: re_postpurchase_progress_01

目的: 購入後・納品後の「今どこまで進んでいるか」「まだ直っていない」「同じエラーが再発した」「追加ログで見られるか」に対して、保留だけで終わらず、状態・方針・範囲を自然に返せるかを見る batch です。 
state_mix: purchased x3, delivered x3
risk_axes: status_visibility, closing_complete, same_cause_vs_new_issue, no_reset_to_prequote, no_weak_ack_only
入れるべき相談 archetype 6件
進捗どうですか
まだ直っていないように見えます
納品後に同じエラーが出ました
別のページでも似たエラーが出ています
このログも追加で見てもらえますか
原因が分からない場合は納品になりますか
reviewer が見るべき観点: prequote に巻き戻らず、今の state で答えているか。進捗質問に actual status があるか。「今回の修正とつながるか」の見立てがあるか。納品後の再発で、保証っぽい約束を勝手に増やしていないか。 
rule に戻す条件: purchased や delivered で prequote 口調に戻る phase 崩れ、または「確認します」だけの弱い保留が複数再発する場合。
rule に戻さず観察で止める条件: 進捗表現の言い回し差や、若干の温度差だけで、相手の次アクションと範囲説明は成立している場合。 
取り込むと危険なノイズ
過剰な共感テンプレート: 共感そのものは有効ですが、決まり文句が続くと本題より感情処理が前に出て、かえって「テンプレ対応」に見えます。過剰共感は batch で薄く観察し、恒久 rule にはしない方が安全です。 
長すぎる謝罪・前置き: 謝罪は必要でも、主質問より先に長く置くと nonanswer になります。謝罪量ではなく、直答への到達速度を見るべきです。 
「必ず Yes/No で始める」固定ルール: answer-first は重要ですが、可否だけ先に投げると一息回答が壊れる場面があります。絶対ルール化すると分断を増やします。 
AIっぽさ検出の validator 化: 研究でも人間の AI 判定ヒューリスティックはかなり誤りやすく、公式にも AI text classifier は短文や非英語で不安定とされています。ai_tone は soft 観察に留めるべきです。 
単語単位の NG リスト増殖: 監査メモ側でも「単語レベルで NG 語を増やすな」と整理されています。問題は単語ではなく、受け手作用と構造です。 
「一息で言う」を絶対 rule にすること: これは有効な観察観点ですが、絶対化すると逆に詰め込みすぎ・不自然な長文を招きます。 reviewer の観察に留めるのがよいです。 
返信を常に短くすること: 短さ自体は目的ではありません。短くしすぎると actual answer や条件説明が欠けます。必要なのは“短いが答えている”です。 
営業色を強めること: 追加提案や別サービス導線を前に出すと、聞かれていない売り込みになり trust を落とします。現行 repo の oversell 観点を強めるだけで十分です。 
社内チャットの作法をそのまま顧客対応へ移植すること: LINE WORKS 的な「挨拶不要」「今いいですか？は不要」は社内 chat には有効でも、対顧客ではそのまま適用すると温度感を崩します。社内/外向けを分けて扱うべきです。 
新しい router / layer / renderer 刷新: 公開文書の方針と今回の外部調査の両方から見て、そこは問題の主因ではありません。今やると正答率より変動要因が増えます。 
最終提案
すぐ #RE の batch archetype に入れる

service_spec_answer_before_intake
one_breath_answer_packaging
commercial_anxiety_answering
boundary_safe_alternative
known_fact_reask
closing_complete_or_weak
fact_overwrite_or_guess
これらは、公開正本と external guidance の両方で再発リスクが高く、buyer 側の不信感に直結しやすい観点です。まずは batch で同型再発を見るのがよいです。 
reviewer_prompt に薄く入れる候補

「最初の 2 文で主質問に答えているか」
「可否だけを独立文にしすぎていないか」
「内部語・抽象語を buyer 語へ変換できているか」
「既知 fact を回収してから聞いているか」
「具体語で相手の文面を受け、勝手な固有名詞を足していないか」
「最後に次アクションまたは返答時期があるか」
「観察は observable behavior で書き、推測で怒らないか」
この層は validator ではなく reviewer の観察精度を底上げする用途に向いています。 
learning-log に観察メモとして残すだけ

manual_keigo_chat_mismatch
catchphrase の鮮度低下
同義反復の出方
参照語の曖昧さ
共感の量感
可否文の切り方
これらは再発しても、しばしば service risk より体感差に寄ります。監査メモでは残す価値がありますが、すぐ rule へ戻すと過学習しやすい領域です。 
今は捨てる / ノイズとして扱う

AI っぽさの自動判定
すべて Yes/No 先頭ルール
単語 NG の増殖
常に短文化
常に強い共感を入れる
営業導線を強くする
新規 router / layer / renderer 刷新
外部知見をそのまま正本 rule に移植すること
ここは、現行骨格を壊す割に、再発防止の精度が上がりにくい領域です。 
何をしないべきか

既存骨格を守るために、今回は次をやらない方がよいです。新しい大規模 router を足さない。renderer をケース別に増やさない。ai_tone を hard-fail validator にしない。単語単位の禁止表現ルールを増やさない。外部調査の一般論を service facts より上位に置かない。好み差を再発パターンと誤認して rule 化しない。やるべきことは、#RE で同型再発だけを観察し、reviewer で「危険度」と「再発性」を分け、再発だけを最小限 rule に戻すことです。これは公開リポジトリ自身の運用方針とも、外部の QA/UX ガイドとも整合しています。
