
ココナラ phase contract の厳密化レビュー
結論
アップロード済みの調査メモの骨子は、現行の公式ヘルプと大筋で整合しています。特に、メッセージ機能 と トークルーム を分けること、delivered と closed を分けること、closed 後は旧トークルーム継続を前提にしないこと、closed 後のおひねりを標準導線にしないこと、外部連絡・外部共有・直接決済を避けること、という主張はそのまま使えます。
 

明確に補正すべき点は四つです。第一に、prequote / quote_sent / purchased / delivered / closed はココナラ公式の用語そのものではなく、公式の 見積り / 取引中 / 完了 / キャンセル とその下位ステータスを内部都合で写像したものです。第二に、quote_sent は「見積り提案済みだが入金完了前」までを含めるのが安全で、トークルームが開くのは入金完了後です。第三に、delivered は「正式な納品後・クローズ前」でよいですが、一回目の正式納品 と 二回目の正式納品 では購入者ができることが異なるため、外向け文面では必要以上に言い切らない方が安全です。第四に、closed 後のメッセージは事実確認と切り分けの導線としては妥当ですが、新たな正式依頼や有償作業の受領導線としてメッセージだけで完結させるのは危険で、必要なら見積り提案または新規依頼へ戻す方が、公式仕様との衝突が少ないです。

返金とキャンセルについては、closed 後に出品者が「返金できます」「返金できません」と断定するのが最も危険です。公式が言っているのは、トークルームがクローズした後はトークルーム上でキャンセルできないこと、直接返金や直接決済は禁じられていること、返金はトークルームでキャンセルが成立した場合の公式フローとして案内されていることです。したがって、closed 後の返信 contract は「まず事実確認を行う」「この場で返金可否は断定しない」「直接返金に寄せない」が正解です。

以下の整理は、通常の受託トークルームを主対象にしたものです。電話相談と定期購入は、「正式な納品」の有無やクローズ条件が別なので、この phase contract にそのまま混ぜない方がノイズが少ないです。

Phase contract 表
前提として、ここでの phase は公式 UI の名前ではなく内部写像です。安全な写像は、prequote = 見積り提案前の購入前相談、quote_sent = 見積り提案済みで未購入または要支払い、purchased = 入金完了後の取引中、delivered = 正式な納品後かつクローズ前、closed = 完了としてクローズしたトークルーム です。closed は キャンセル を含めない方が contract として安定します。

prequote
項目	内容
その phase の意味	購入前、かつ見積り提案送信前の相談段階です。やり取りはメッセージ機能で行い、正式な依頼はまだ始まっていません。正式な依頼は購入後のトークルームで行う必要があります。 
できること	価格・納期・対応可否・必要資料・共有方法の確認、必要に応じたメッセージ経由の見積り提案です。メッセージ上の相談を踏まえて、出品者側から見積り提案を送れます。 
できないこと	取引中前提の案内、トークルーム前提の指示、購入済み扱い、外部連絡・外部共有・直接決済・直接返金の示唆です。 
外向けに言い切ってよいこと	「購入前なので、まずメッセージ上で条件確認を行う」「正式な依頼は購入後のトークルームになる」は言い切ってよいです。 
外向けに断定しない方がよいこと	「簡単だから必ず安くなる」「直らなければ必ず返金になる」「この内容で確実に直る」などは断定不可です。まだ取引が始まっておらず、金額も成果も確定前だからです。 
購入者に示すべき次導線	不足情報があるならメッセージで追加確認し、条件が固まったら見積り提案または通常購入へ進む導線を示すのが適切です。 
返信文で曖昧になりやすいポイント	「相談」と「正式依頼」、「概算」と「確定額」、「資料確認」と「着手開始」が混ざりやすいです。 
validator / reviewer で見るべき危険パターン	トークルームでお願いします と書く、外部ストレージや外部通話に誘導する、未購入なのに返金や修正保証を断定する、が危険です。 

quote_sent
項目	内容
その phase の意味	見積り提案済みで、まだ入金完了していない段階です。内部 contract では、少なくとも公式の 未購入 と 要支払い をここに含めるのが安全です。トークルームは入金完了後に開きます。 
できること	提案内容の説明、範囲のすり合わせ、提案の修正・再送、購入期限の案内です。購入期限は提案日または最終変更日から 1 週間です。 
できないこと	取引開始済みとして扱うこと、トークルーム前提で話すこと、進行中の作業として案内すること、追加オプションやおひねりの話をすることです。まだトークルームがありません。 
外向けに言い切ってよいこと	「入金完了後にトークルームが開いて取引開始になる」「期限内であれば提案内容の確認や修正ができる」は言い切ってよいです。 
外向けに断定しない方がよいこと	「未購入でも席は確保済み」「直らなかったら自動で返金」「この提案なら追加料金は今後一切ない」などは断定不可です。購入未成立であり、かつ範囲変更の可能性も残っています。 
購入者に示すべき次導線	提案内容に不安があれば質問または修正、内容に問題がなければ購入・入金完了でトークルームへ進む導線を示すのが適切です。 
返信文で曖昧になりやすいポイント	「購入ボタンを押した」だけで purchased と誤認すること、未購入 と 要支払い を区別せずにトークルーム文面を出してしまうことです。 
validator / reviewer で見るべき危険パターン	トークルームで続けます、納品後は のような purchased/delivered 文面、購入未成立なのにキャンセル・返金を既定事実として説明する文面が危険です。 

purchased
項目	内容
その phase の意味	入金完了後にトークルームが開いた、取引中の段階です。公式ステータスでは 購入済み から 取引中 に入った状態に相当します。 
できること	トークルームでの要件確認、進捗連絡、必要資料の受領、トークルーム内ファイル添付、必要に応じたキャンセル相談、購入者側の追加支払い操作です。ファイル添付はトークルームなら 200MB まで可能です。 
できないこと	メッセージ機能を本線に戻すこと、外部共有や外部通話に逃がすこと、直接決済・直接返金を案内すること、追加料金を出品者側が自動発生させること、未提供のまま正式な納品で押し切ることです。 
外向けに言い切ってよいこと	「取引中なので、確認や作業のやり取りはトークルームで行う」「追加支払いが必要な場合でも、支払い操作は購入者側で行う」は言い切ってよいです。 
外向けに断定しない方がよいこと	「購入者はいつでも自分でキャンセルできる」「有料オプションは常に追加できる」は断定不可です。購入者側のキャンセルボタンは限定条件つきで、見積り相談経由の取引では有料オプションを買えず、正式な納品後も有料オプションは購入できません。 
購入者に示すべき次導線	まずトークルームで必要情報と現在の確認点を明示し、キャンセル希望ならトークルーム上で理由と方針を整理し、追加費用が必要なら先に相談してから購入者側の支払い導線へ案内するのが適切です。 
返信文で曖昧になりやすいポイント	取引中 と 正式な納品後 の境界、追加料金が「相談」なのか「確定請求」なのか、見積り経由取引か通常購入か、が混ざりやすいです。 
validator / reviewer で見るべき危険パターン	「勝手に追加料金が発生する」と読める文面、外部ストレージや外部通話の提案、購入者がいつでも自力キャンセルできると書く文面、未完成での正式納品が危険です。 

delivered
項目	内容
その phase の意味	内部的には「正式な納品後・クローズ前」です。公式では少なくとも一回目の正式な納品後は 要検収 に相当し、差し戻しがあると 再納品待ち に戻ります。二回目の正式な納品後は、評価入力または 72 時間経過でクローズに向かいます。 
できること	クローズ前である限り トークルーム内で確認・やり取りが可能です。一回目の正式な納品後は購入者が承諾または差し戻しを行え、差し戻しは 1 回のみです。承諾後も、評価入力または 24 時間経過まではクローズしません。正式な納品後でも、クローズ前ならおひねりの支払いは可能です。 
できないこと	closed 扱いすること、差し戻し回数を無制限として扱うこと、正式な納品後でも有料オプションを追加できると案内すること、合意内容を超える過度な修正要求を当然の権利として扱うことです。 
外向けに言い切ってよいこと	「まだクローズ前なので、確認はトークルームで行える」「合意内容との差異があるかを見てほしい」は言い切ってよいです。 
外向けに断定しない方がよいこと	「承諾した瞬間に何もできない」「今は必ず差し戻しを選べる」は断定不可です。承諾後でも 24 時間はクローズ前のことがあり、二回目の正式な納品後は承諾・差し戻しの選択ではなく、評価入力または 72 時間経過でクローズします。 
購入者に示すべき次導線	一回目の正式な納品なら、合意内容との差異確認のうえ承諾または差し戻しに進む導線を示すのが自然です。二回目の正式な納品かどうか不明なら、「クローズ前なのでまずトークルームで事実確認する」とだけ書く方が安全です。 
返信文で曖昧になりやすいポイント	delivered だけでは、一回目の正式な納品なのか、差し戻し後の二回目なのか、承諾済みで 24 時間以内なのかが分かりません。ここで「差し戻しできます」と一般化すると誤答になります。 
validator / reviewer で見るべき危険パターン	delivered を closed と書く、二回目の正式な納品後でも差し戻し可能と書く、正式な納品後に有料オプション購入を案内する、差し戻しを無制限に見せる、が危険です。 

closed
項目	内容
その phase の意味	完了としてトークルームがクローズした段階です。内部 contract では、closed = 完了のクローズ済みトークルーム とし、キャンセル は別物として扱う方が安全です。クローズ条件には承諾後、72 時間自動承諾、二回目の正式な納品後、120 日経過などがあります。 
できること	連絡窓口はメッセージ機能へ戻ります。メッセージではファイル添付が可能で、1 回あたり最大 5 ファイル、100MB まで、zip も送れます。条件が固まれば、メッセージ経由の見積り提案も可能です。 
できないこと	旧トークルームでのメッセージ投稿・ファイル投稿・おひねりなどの追加購入、一度クローズしたトークルームを取引中に戻すこと、承諾の取り消し、クローズ済みトークルームのキャンセル、直接返金・直接決済・外部共有・外部連絡の誘導はできません。 
外向けに言い切ってよいこと	「旧トークルームは使えないので、ここではメッセージで状況を確認する」「必要なら見積り提案または新規依頼へ戻す」は言い切ってよいです。前半は公式事実、後半はそれに沿った安全な運用判断です。 
外向けに断定しない方がよいこと	「返金できます」「返金できません」「前のトークルームで続けます」「closed 後はおひねりで払ってください」は断定不可または誤りです。返金はこの場で出品者が断定すべきではなく、おひねりは closed 後には使えません。 
購入者に示すべき次導線	まずメッセージ上で、前回の依頼内容・今回の症状・再現条件・確認資料を受け取り、そのうえで「前回の補足で済むか」「新たな見積り提案や新規依頼が必要か」を返す導線が最も誤解が少ないです。公式は粒度を明示していないため、ここは運用判断です。 
返信文で曖昧になりやすいポイント	closed と cancelled を混ぜること、メッセージ上の事実確認と新規の正式依頼を混ぜること、返金希望に対して結論を先に言い切ることです。公式上、返金ページは「キャンセル成立」を前提にしています。 
validator / reviewer で見るべき危険パターン	旧トークルーム継続、closed 後おひねり誘導、closed 後キャンセル案内、直接口座・PayPay・外部ファイル共有・外部通話の提案、返金可否の即断が危険です。 

delivered と closed の混同防止
delivered は終わった状態ではなく、正式な納品は済んだが、まだトークルームが生きている状態として扱うべきです。公式の取引管理画面でも、正式な納品後は 要検収、差し戻し後は 再納品待ち としてなお 取引中 に含まれており、完了 は別です。さらに、承諾後も評価入力または 24 時間経過までクローズしません。したがって、返信システム上で delivered = 取引完了 に寄せるのは誤りです。 

逆に closed は、旧トークルームを使えない状態であり、メッセージ・ファイル投稿・おひねりなどの追加購入ができず、再オープンも承諾取消もできません。ここで delivered と同じノリで「トークルームで確認します」と返すと、返金・差し戻し・追加作業・再発確認の導線が全部あいまいになります。closed では旧トークルームではなく、メッセージ機能が入口です。 

観点	delivered	closed
内部意味	正式な納品後・クローズ前。 
完了としてクローズ済み。旧トークルーム不可。 
連絡チャネル	まだトークルーム。 
メッセージ機能へ戻る。 
ファイル受領	トークルーム添付可、200MB。 
メッセージ添付可、100MB、zip 可。 
購入者のアクション	一回目の正式な納品なら承諾 / 差し戻し。二回目の正式な納品後は評価入力または 72 時間経過でクローズ。 
承諾・差し戻し・再オープンは不可。 
追加費用の扱い	正式な納品後は有料オプション不可だが、クローズ前までおひねりは可能。 
旧トークルームでのおひねり・追加購入不可。 
キャンセル / 返金文言	まだトークルームが生きている以上、即 closed 前提の断定はしない。 
旧トークルームでのキャンセルは不可。返金可否の即断も不可。直接返金は禁止。 

delivered を一つの phase に留めるなら、外向け文面は「まだクローズ前なので、まずトークルーム内で確認できます」までに留め、承諾 / 差し戻しの可否は「一回目の正式な納品だと確認できた場合だけ」出すのが最も安全です。ここを最小補正にすれば、新しい phase を増やさなくても実運用はかなり安定します。 

closed 後の重点整理
公式事実と contract 上の答え
争点	公式事実	contract 上の答え
closed 後に「前回の修正がまた壊れました」と言われたら、どこまで確認してよいか	クローズ後の連絡窓口はメッセージ機能です。メッセージは取引終了後の連絡に使えますが、正式な依頼は購入後のトークルームで行う建て付けです。 
症状、再現条件、環境差分、いつから起きたか、前回修正との関係、スクリーンショットやログの有無まではメッセージで確認してよいです。ただし、実際の修正着手、成果物の再納品、納期約束、追加費用の確定まで入るなら、見積り提案または新規依頼へ戻すのが安全です。どこからが「正式依頼」かの厳密線引きは公式にないため、その境界自体は断定不可です。 
closed 後にスクショ・ログ・ZIP など確認材料を受け取れるか	メッセージ機能には添付機能があり、1 回あたり最大 5 ファイル、100MB まで、zip を含む複数形式が送れます。クローズ後の追加連絡もメッセージ機能を使います。 
可能です。ただし旧トークルームではなくメッセージで受け取る前提です。100MB を超える場合でも、外部ストレージや外部リポジトリ共有には逃がさず、分割・縮小・必要部分のみ提出の方向で案内する方が安全です。外部共有を許す公式例外はこの用途では見当たりません。 
closed 後、実作業へ入る前に見積り提案 / 新規依頼へ戻す境界はどこか	クローズ後の連絡はメッセージ機能、正式な依頼は購入後のトークルームです。メッセージ経由の見積り提案は公式に可能です。 
説明・切り分け・証跡確認だけならメッセージ継続でよいです。実際に手を動かす、成果物を変える、納期を置く、費用を決める段階に入るなら、見積り提案または新規依頼へ戻すべきです。ここは公式の明文化ではなく、返信文を曖昧にしないための運用境界です。断定不可ではあるものの、この線引きが最も副作用が少ないです。 
closed 後に「返金してほしい」と言われた場合、どう返すべきか	クローズ後はキャンセルできません。直接の金銭のやりとりは禁止です。返金ヘルプは「トークルームでキャンセルが成立した場合」を前提に支払い方法別の返金フローを説明しています。 
返信 contract は、「旧トークルームはクローズ済みで、トークルーム上のキャンセル手続きには進めない」「まず今回の不具合内容と当初合意との関係を確認したい」「返金可否はこの場で断定しない」「直接返金や外部決済には進まない」です。ここで 返金できます / できません の即断は避けるべきです。 
closed 後に「おひねりで払えばよいか」と言われた場合	クローズ後はトークルームでおひねりなどの追加購入ができません。おひねりはトークルーム内で購入者側のみが支払い操作できます。 
closed なら答えは 不可 です。返信 contract は、「closed 後の旧トークルームではおひねりは使えない」「追加作業が必要なら、メッセージで内容確認後に見積り提案または新規依頼へ戻す」です。 おひねり を closed 後の標準導線にしない方がよいです。 
closed 後に「前のトークルームで続けられるか」と言われた場合	一度クローズしたトークルームを取引中に戻すことはできず、旧トークルームでのメッセージ・ファイル投稿もできません。承諾後 24 時間以内で評価前なら、まだクローズ前である可能性だけはあります。 
phase が本当に closed なら答えは 不可 です。返信 contract は、「もしまだクローズ前ならトークルームで続けられるが、closed なら旧トークルームは使えない。今はメッセージで確認する」にすべきです。 phase 未確定なのに closed と決め打ちするのも危険です。 

結局どうなるのかを消すための返信型
以下は実送信用の文面ではなく、返信システムが保持すべき型です。公式事実を踏まえつつ、曖昧な着地を消す形に寄せています。 

ケース	返信型
closed 後の再発相談	[旧トークルーム不可明示] → [メッセージで症状・再現条件・前回との関係確認] → [確認資料の依頼] → [前回補足か / 新規作業かを後返し]
closed 後の返金希望	[closed でトークルームキャンセル不可明示] → [不備内容と当初合意の確認] → [返金可否は断定不可を明示] → [ココナラ上で取れる対応を後返し]
closed 後の追加作業希望	[旧トークルーム不可明示] → [メッセージで内容整理] → [実作業なら見積り提案または新規依頼へ戻す]
delivered 後の差し戻し不安	[クローズ前であることを明示] → [合意内容との差異確認へ案内] → [一回目納品か二回目納品か不明なら差し戻し可否は言い切らない]
purchased 中のキャンセル希望	[取引中であることを明示] → [理由と現状整理] → [通常はトークルーム上のキャンセルリクエスト導線] → [直接返金や外部決済には寄せない]
quote_sent 後の「直らなかったら料金は？」	[まだ購入完了前であることを明示] → [提案がカバーする範囲を明示] → [返金を先に断定しない] → [不安があれば提案修正または見送りへ分岐]
prequote の「簡単なら安くなりますか？」	[現時点では概算であることを明示] → [簡易対応で済むなら見積りを下げうると示す] → [購入前に金額確定へ導く]
追加料金不安「勝手に追加料金が発生しますか？」	[自動課金を否定] → [追加費用が必要なら先に相談・合意] → [支払いは購入者側の操作で行うことを明示]

この表で重要なのは、どのケースでも返信が 相談ください で終わらず、この次に何が起きるか を型の中に入れていることです。特に closed 系は、メッセージで確認します だけでは不十分で、確認後に旧トークルームへ戻らないこと、必要なら見積り提案または新規依頼へ戻ること まで含める方が、はぐらかしに見えにくいです。 

hard rule / gold / validator / reviewer_prompt / edge stock / 入れない の分類
以下の分類は公式事実そのものではなく、上の事実を既存返信システムへ最小反映するための実装判断です。

置き場所	入れてよい内容
hard rule	prequote / quote_sent はメッセージ系、purchased / delivered はトークルーム系、closed はメッセージ系 というチャネル境界。closed では旧トークルーム投稿不可、再オープン不可、承諾取消不可、おひねり不可。外部連絡・外部共有・直接決済・直接返金を勧めない。quote_sent は入金完了前まで含め、入金完了後にトークルーム開始とする。 返金できます / できません の無条件断定を禁止する。 
gold reply にした方がよいもの	closed 後の再発相談、closed 後の返金希望、closed 後の追加作業希望、delivered 後の差し戻し不安、purchased 中のキャンセル希望、quote_sent 後の no-fix 不安、prequote の価格不安、追加料金不安。これらは「結局どうなるか」が最も崩れやすく、gold 化の費用対効果が高いです。 
validator で fail した方がよいもの	closed なのに旧トークルーム継続を前提にする、closed 後におひねりを案内する、直接返金や外部決済や外部共有を案内する、外部通話や外部ストレージに誘導する、未提供・不十分な状態での一方的クローズや正式納品を促す、は fail が妥当です。 
validator で warn した方がよいもの	delivered と closed の取り違え、二回目正式納品でも差し戻し可能と書く、購入者がいつでも自力でキャンセルできると書く、正式納品後でも有料オプションを追加できると書く、追加料金が自動発生すると読める文面、は warn が妥当です。 phase 粗さによる誤答 を拾う目的で十分です。 
reviewer_prompt に留めるべきもの	前回の修正と同一原因か / 別原因か / 新要件か、今どこを見ていて、次にどうなるか、free explanation と actual work の境界、phase が未確定なら言い切りを下げる といった観点です。これは hard rule 化すると過剰拘束になりやすく、監査観点に置く方がよいです。 
#RE の edge stock に回すべきもの	二回目の正式納品後、見積り相談経由で有料オプション不可、closed 後 100MB 超の資料提出問題、出品者無返信に伴う購入者側キャンセルボタン、120日クローズ後の再発相談、承諾後 24 時間以内でまだ closed ではないケース です。再発しやすいが phase contract に直書きすると重くなります。 
返信システムに入れるとノイズになるので入れない方がよいもの	支払い方法ごとの返金 timing 詳細、部分減額の細かな再購入フロー、電話相談・定期購入の例外、評価公開の細則、必要不可欠な役務提供情報共有の例外詳細です。これらは公式には存在しますが、phase contract の本体に入れるとシステムの思考を削ぎやすいです。必要なら個別 edge や reviewer へ回す方がよいです。 

ノイズ判定
情報種別	判定	理由
古いココナラニュース	原則入れない	特に closed 後のおひねり や過去の運用は現行ヘルプとズレやすく、正本根拠として使うと誤誘導の元になります。
 
個人ブログの体験談	hard rule には入れない	感度のよい edge 探索には使えても、仕様根拠にはなりません。再発パターンとして #RE stock に留めるのが適切です。
知恵袋や SNS の不満投稿	hard rule には入れない	トラブルの語り口は拾えても、現行の公式仕様確認には使えません。 edge stock までです。
他出品者の「追加支払い専用サービス」実例	原則入れない	サービスページ実例はヘルプ仕様ではなく、現行ヘルプと衝突しうります。特に closed 後の追加購入 を誤学習しやすいです。
 
法律相談系の返金見解	入れない	system が扱うべきなのは一般論の法解釈ではなく、ココナラ上での現行手続と禁止行為です。返信内で法務断定を強めると危険です。
公式ヘルプにない運用上の慣習	reviewer まで	公式にない慣習は drift しやすく、hard rule 化しない方が安全です。 reviewer の観点か #RE stock に留めるのがよいです。

補足すると、必要不可欠な役務提供情報の共有には公式上の例外があります。したがって generic hard rule は「一切の個人情報・認証情報に触れない」ではなく、外部連絡や直接連絡につながる共有を促さない に留める方が正確です。ここを硬くしすぎると、本来許容される業務情報共有まで誤って潰します。 

最小反映案
大規模再設計なし、既存骨格維持の前提で、最小反映は次の順序が最も安全です。

反映先	最小案
すぐ platform-contract に反映すべきもの	quote_sent = 見積り提案済みだが入金完了前、purchased = 入金完了後のトークルーム open、delivered = 正式な納品後・クローズ前、closed = 完了としてクローズ済みでキャンセルは含めない を明文化する。併せて closed では旧トークルーム不可、closed 後のおひねり不可、返金断定不可、外部連絡 / 外部共有 / 直接決済不可を一行 contract に落とす。 
#RE batch で先に検証すべきもの	二回目正式納品でも差し戻し可と誤答しないか、見積り相談経由で有料オプションを案内しないか、closed 後 >100MB 資料で外部共有に逃げないか、購入者はいつでも自力キャンセル可能と誤答しないか を batch で先に見るのがよいです。phase を増やさずに危険 edge だけ拾えます。 
reviewer_prompt に足すだけでよいもの	前回の修正との関係を聞いたか、この返信で次に何が起きるかまで書いたか、message 上の事実確認と正式依頼を混ぜていないか、phase 未確定なのに delivered/closed を断定していないか の四観点です。これは hard rule ではなく監査向きです。 
validator に入れるなら warn にすべきもの	closed なのに「トークルームで」、purchased/delivered なのに「メッセージで続けます」、delivered で差し戻しを一般化、追加料金が発生します とだけ書いて buyer 操作を欠く、のような wording drift を warn で拾うのがよいです。 fail を増やしすぎるより、まず phase 取り違え検知に寄せる方が副作用が少ないです。 
入れない方がよいもの	古いニュース、体験談、SNS 不満投稿、他出品者の個別運用、法律相談の返金見解、支払い方法別返金 timing の詳細、電話相談 / 定期購入の別系統ルールは、現段階の phase contract 本体には入れない方がよいです。必要なら reviewer や edge stock に逃がす方が、返信システムの思考を削ぎません。 
 

最小導入の要点を一行で言うと、phase を増やす必要はなく、delivered と closed の外向け言い切りを細くし、closed 後は「旧トークルーム不可 → メッセージで事実確認 → 必要なら見積り提案 / 新規依頼」の着地を固定する、これだけでかなり安定します。返金 と おひねり はその着地を曖昧にしやすいので、返金断定不可 と closed 後おひねり不可 だけは platform 側で先に硬くしてよいです。
