1. Executive Summary

結論から言うと、現設計の骨格は壊れていません。ただし、いま r0 で崩れている型の多くは、もう「gold を増やせば自然に直る段階」を超えています。特に 非Stripe scope / 緊急復旧時間 / 保証期間 / 仕様か不具合か / purchased 進捗 / quote_sent 語彙 / closed 後導線 は、r1 監査で直せるだけでは不十分で、renderer の初手判断順序またはdeterministic validatorへ昇格すべきです。

今の skill は、設計思想としては Codex の自由判断を塞いでいません。coconala-reply-bugfix-ja も coconala-prequote-ops-ja も、skill は思考の代替ではなく、Writer は freeform に書く前提だと明記しています。これは健全です。
一方で、返信 skill 本体に “ケース別の禁止・注意・例外” が増え続けており、今後さらに r0 失敗型を skill に直接追記すると、template-first / rule-dump / 長文化に寄ります。ここはかなり危ないです。

最も重要な診断はこれです。

r1 で直るが r0 で繰り返し崩れる型は、学習ループが効いている証拠ではあるが、初回生成側への接続はまだ弱い。
特に同じ型が複数 batch で r0 fail → r1/r2 pass しているものは、learning-log/gold 止まりではなく、renderer または validator へ戻すべきです。

実際、learning-log では bugfix28〜31 にわたり、非Stripe、緊急復旧、browser / scope、仕様境界、closed discount、怒り気味 buyer、保証、quote_sent phase wording などが r0 で崩れ、r1/r2 で安定したことが記録されています。各 log 自身も「r0 が続くなら renderer/validator 検討」と書いており、今回の状況はその条件を満たしています。

2. 現設計の健全性評価
2.1 骨格はかなり健全

良い点は、Classifier / Writer / Reviewer の分離が明確なことです。coconala-prequote-ops-ja では、Classifier が phase / service / scope / forbidden_moves を固定し、Writer は最小 facts packet を受けて自然文を書く、Reviewer が phase/public-private/price/scope/forbidden/next action を見る構造になっています。

writer-brief.ja.md も、Writer に full self-check や長い audit history を渡さず、buyer 文、route/state、最小制約、近い few-shot だけを渡す方針です。これはかなり正しいです。Writer に全ルールを背負わせない設計なので、自由文生成の余地があります。

また docs/reply-quality/README.ja.md の #RE 方針も、「再発性があるものだけ skill/facts/lint へ」「一回限りは preference/case_fix」という運用になっており、これは過剰反映を防ぐ上で妥当です。

2.2 ただし r0 崩れは “文体” ではなく “初手分岐” の問題

r0 未安定型を見ると、崩れ方はだいたい次のどれかです。

崩れの種類	実態
service spec 質問を通常の不具合相談として処理	緊急復旧時間、保証、返金、キャンセル
scope boundary を price/intake に流す	非Stripe、browser-specific、仕様/不具合境界
phase vocabulary を誤る	quote_sent で「トークルーム内」、closed 後の旧トークルーム継続
purchased の進捗が薄い	「確認中です」だけに戻る
感情温度を普通の受付文に戻す	怒り気味 buyer に「内容ありがとうございます」系

これは「言い回しの好み」ではなく、primary_question_type / phase / boundary_kind / buyer_next_action の初期選択ミスです。したがって、gold を増やしても r0 では毎回 retrieval や判断の揺れに依存します。

2.3 validator の適用範囲に穴がある

render-coconala-reply.py の common validator は、quote_sent / purchased / delivered / closed には走りますが、prequote には原則走らない構造です。collect_common_validator_errors が対象 state をこの4つに限定しています。

しかし、今回 r0 で崩れやすいものの多くは prequote です。

非Stripe scope
緊急復旧時間
保証期間
仕様/不具合境界
返金/キャンセル
同義反復テンプレ

つまり、r0 崩れの主戦場に対して、共通 validator が薄い可能性があります。ここは接続不良です。大規模再設計ではなく、prequote 専用の deterministic guard を薄く足すのがよいです。

2.4 platform / service fact の分離は正しいが、返信側でまだ漏れるリスクがある

service-registry.yaml では、外向け public は bugfix-15000 のみ、handoff-25000 は public: false の private service と定義されています。public reply では public false service の name/price/purchase CTA を出してはいけない、という契約も明記されています。

platform-contract.yaml では、quote_sent は payment 前、purchased は payment 後に talkroom が開く状態、closed は closed talkroom であり、closed 後は旧 talkroom に戻せず、message 経由や正式導線が必要だと定義されています。

この contract は強いです。問題は、この情報が r1 reviewer では効いているが、r0 renderer の語彙選択にまだ十分効いていない箇所があることです。

4.2 validator に入れない方がよいもの

これは入れるとノイズになります。

入れないもの	理由
怒り気味 buyer の温度不足全般	主観が大きい。validator ではなく reviewer_prompt。例外的に complaint 冒頭の「内容ありがとうございます」だけ deterministic。
保証/返金説明がないこと	buyer が聞いていないなら不要。常時要求すると返信が怖く長くなる。
Gold と違う言い回し	gold は template ではない。README も「コピーではなく近い anchor」と定義。
質問数が多い/少ないの一律判定	Writer brief では質問は原則少なくするが、case による。hard fail ではなく reviewer。
仕様/不具合を断定していないこと	断定しないのが正しい case も多い。validator は “根拠なし断定” だけ落とす。
短くないこと	長文化は重要だが deterministic ではない。文字数だけで落とすと必要説明も落ちる。compression review へ。
非Stripe word が出たら全部 fail	Next.js 側の Webhook/API 実装確認は scope 内になり得る。provider 全体対応だけ落とす。
platform 仕様説明不足	buyer に見せる必要がない内部構造もある。surface_overexposure とセットで reviewer が見る。
5. renderer に戻すべきもの

validator は「落とす」ものですが、今回の r0 崩れの本丸は「最初に何を答えるか」です。したがって、次は renderer に戻すべきです。

5.1 primary_question_type を先に確定する

今の r0 失敗は、buyer の主質問が service spec なのに、通常 bug intake として返すケースが多いです。renderer 側で最低限この分類を持たせるべきです。

primary_question_type:
  - normal_bug_intake
  - scope_boundary
  - emergency_time
  - guarantee
  - refund_or_cancel
  - spec_vs_bug
  - purchased_progress
  - quote_sent_material_or_scope
  - closed_new_route

prequote-compression-rules.ja.md も、価格・可否・納期・返金・保証などの service spec 質問は、通常 bug template に流さず、主質問へ先に答える方針を持っています。

5.2 direct_answer_line を renderer の必須出力に近づける

skill 側にはすでに response_decision_plan と direct_answer_line があります。
ただ、r0 で崩れるなら、これは reviewer 用の観点ではなく、renderer の初期 skeleton に近づけた方がよいです。

特に必要な型は以下です。

型	direct answer の例
非Stripe	「決済サービス自体の管理画面操作は対象外ですが、Next.js 側の Webhook/API 処理であれば確認できます。」
緊急復旧	「原因確認前に復旧時間は約束できませんが、まず一次確認の範囲ならこの情報で判断できます。」
保証	「固定の保証期間としては約束していません。同一原因の再発かどうかは確認できます。」
返金/キャンセル	「こちらで返金可否や金額を断定する形ではなく、ココナラ上の手続きに沿う形になります。」
purchased 進捗	「いま見ている箇所は◯◯で、次に△△を確認しています。」
quote_sent	「追加ファイルは、購入後に開くトークルームで確認できます。」
closed	「確認材料はメッセージで見られますが、実作業が必要なら新しい正式導線になります。」
5.3 purchased 進捗は minimum content を renderer に持たせる

japanese-chat-natural-ja では、進捗返答は「何を見ているか / 何が分かっているか / 次に何を見るか / 次の目安」を出す方針です。
これは reviewer では遅いです。post_purchase renderer の skeleton に入れるべきです。

最小条件はこの程度で十分です。

purchased_progress_minimum:
  require_any_3:
    - current_target
    - observed_fact
    - remaining_uncertainty
    - next_check
    - next_update_timing

ただし、毎回全部出すと長くなるので、3/5で可くらいがよいです。

5.4 phase vocabulary は renderer の state-aware surface map に寄せる

quote_sent と closed の誤語彙は、本文生成後に validator で落とせますが、r0 で毎回落ちるなら遅いです。

surface_words_by_phase:
  prequote:
    avoid: ["作業を進めます", "確認を進めます", "トークルーム", "納品"]
  quote_sent:
    avoid_current: ["トークルーム内", "作業中", "納品"]
    allowed_future: ["購入後にトークルームが開いたら"]
  purchased:
    allowed: ["トークルーム", "確認中", "次に確認"]
  closed:
    avoid: ["旧トークルームで続けます", "追加購入で", "納品します"]

platform contract 上、quote_sent は payment 前、purchased で talkroom が開き、closed は talkroom が閉じた状態なので、これは renderer の語彙制約として置く価値があります。

6. gold の整理方針

Gold 26〜33 は内容としては有効です。ただし、このまま増やし続けると retrieval ノイズになります。Gold README は、gold は template ではなく reusable anchor であり、新規追加は「ほぼそのまま再利用できる」「一回限り preference ではない」ものに限る方針です。
現状 33 本まで増えているので、追加より整理の段階です。

Gold	判断	理由 / 処理
26 concrete symptom pickup	残す。ただし validator seed を抽出	旧三点セット検出の根拠として強い。本文例としても有効。ただし、毎回参照するより validator に昇格した方が r0 に効く。
28 price/scope boundaries	残すが参照頻度を下げる	discount / add-on / new feature split は重要。ただし 29/32/33 と重なるので、price_scope_boundary tag にまとめる。

gold_index:

今やるべきは Gold 34 を増やすことではなく、26〜33 から renderer/validator へ抽出することです。

7. reviewer_prompt に置くべき観点

本文生成側へ入れると長文化・説明過多になるが、監査では非常に効くものです。これは reviewer_prompt に残すべきです。

reviewer 観点	renderer/validator に入れない理由
buyer が迷子になっていないか	「次に何をすればいいか」は文脈依存。validator で一律判定すると false positive。
責任を認めすぎていないか	怒り気味 buyer や返金文脈で重要。ただし wording は多様。
無料対応期待を作っていないか	closed / warranty / discount で重要。deterministic phrase だけ validator、全体姿勢は reviewer。
surface_overexposure	phase, transaction_model_gap, closed_talkroom_locked など内部構造が表に出ていないか。本文生成へ入れると逆に露出しやすい。
buyer の感情温度に合っているか	怒り気味 buyer に通常受付文はまずいが、感情表現を template 化すると不自然。
技術見立てを出しすぎていないか	Stripe/Next.js で推測しすぎると責任が重くなる。validator では拾いにくい。
境界説明が多すぎないか	返金/保証/追加料金などを全部説明すると buyer に不安を作る。compression review が適切。
主質問に先に答えているか	direct-answer-first は一部 validator 化できるが、自然な順序は reviewer が見るべき。

self-check-core-always-on.ja.md は、Reviewer の always-on core を「直接回答、phase、質問過多、price/scope/public、closed、NG、一つの next action」などに絞っています。これはよい線引きです。

8. ノイズ化リスク

ここはかなり重要です。今後の #RE で改善しようとして、逆に壊しやすい箇所です。

8.1 skill へケース知識を足しすぎる

coconala-prequote-ops-ja は「skill に新しい判断ルールを足さない」「厳密固定は facts/phase/scope/forbidden_moves、本文は natural」と書いています。
ここを破って、非Stripe、保証、怒り buyer、closed discount などを全部 skill 本体に足すと、Writer が “全部説明しなきゃ” になり、r0 は短く自然になるどころか長文化します。

方針:
skill 本体には戻さない。戻すなら renderer/validator/platform_contract/gold index。

8.2 返金・保証・追加料金を常に説明する

buyer が聞いていないのに返金/保証/追加料金を出すと、逆に不安・交渉・責任範囲を広げます。prequote-compression-rules.ja.md でも、未質問の返金・キャンセル・正式納品・追加費用説明は削る候補です。

方針:
asked only。保証/返金/キャンセルは primary_question_type が該当するときだけ direct answer。

8.3 ココナラ仕様の説明過多

phase contract は内部判断に必要ですが、buyer に quote_sent や closed_talkroom_locked 的な説明を出すと不自然です。platform contract は内部 source of truth として使い、外向けは「購入後に開くトークルーム」「この前のトークルームは閉じているため」程度に抑えるべきです。

8.4 技術見立てを出しすぎる

Stripe / Next.js / API は原因推測がしやすい領域ですが、ログ前に「Webhook が原因です」「PaymentIntent の処理です」と寄せすぎると、責任が重くなります。特に怒り気味 buyer・返金・緊急復旧では危険です。

方針:
「可能性」より「確認対象」を出す。
例: “Webhook 周りの可能性があります” より “Webhook 受信後の処理とログを確認します”。

8.5 gold が増えすぎる

Gold 26〜33 は有効ですが、数が増えるほど r0 の retrieval が “近いけど違う例” を拾いやすくなります。Gold README も gold は少数の reusable anchor として扱う前提です。

方針:
Gold 追加停止。26〜33 の tag/index 化、重複 family 化、再発部分の renderer/validator 抽出。

8.6 validator を主観判定に使う

怒り buyer の温度、buyer迷子、説明過多、人間っぽさは reviewer/naturalizer です。validator に入れると、良い返信も落ちます。

方針:
validator は phase/secret/private/external/overpromise/repetition/old-template のみに絞る。

9. 次の #RE 設計案

優先順はこうです。

Priority 1: r0未安定型だけの regression batch

まずこれです。通常 batch に混ぜると、r0 がどの型で崩れているか見えづらくなります。

batch_name: r0-unstable-regression-bugfix-15000
cases:
  - non_stripe_scope_paypay_or_gmo
  - urgent_recovery_time
  - warranty_period_question
  - refund_cancel_question
  - spec_vs_bug_boundary
  - purchased_progress_anxiety
  - quote_sent_additional_file_phase_wording
  - closed_discount_or_new_request
  - angry_buyer_temperature
metrics:
  - r0_pass
  - r1_pass
  - failure_layer: [renderer, validator, gold, reviewer_prompt, facts, case_only]
  - direct_answer_present
  - phase_vocab_ok
  - private_service_leak_none
  - overpromise_none

ここでは r1/r2 の改善能力ではなく、r0 単体の分岐安定性を見るべきです。

Priority 2: purchased 進捗 batch

購入後の進捗は、buyer 信頼とキャンセルリスクに直結します。platform contract でも購入後の無応答・キャンセルリスクに関する規定があります。

作るべきケース:

「今どこまで進んでいますか？」
「まだですか？急いでいます」
「返金してほしいです」
「前回から進捗が見えません」
「ログを追加しました。見ましたか？」

評価軸:

「確認中です」だけで終わっていないか
見ている対象があるか
未確定点が明確か
次の確認/次回共有があるか
謝りすぎ/責任認めすぎになっていないか
Priority 3: non-Stripe / browser-specific / 仕様か不具合か batch

これは scope boundary の r0 安定化用です。

PayPay / GMO / Square / PayPal
Stripe + 別 payment provider 混在
Safari だけ落ちる
本番だけ起きる
仕様か不具合かわからない
CSS/見た目/並び順なのか API 不具合なのか曖昧

目的は「受ける/受けない」ではなく、どの面なら見られるかを r0 が言えるかです。

Priority 4: refund / cancel / warranty / emergency sensitive batch

この batch は validator の false positive/false negative を見るために使います。

聞かれているときは direct answer
聞かれていないときは出さない
seller が返金可否/金額/キャンセル可否を断定しない
復旧時間を保証しない
保証期間を勝手に作らない
Priority 5: quote_sent + closed phase routing batch

quote_sent/closed は phase contract の誤用が出ると、取引導線が壊れます。

quote_sent で追加ファイルを送りたい
quote_sent で「もう作業開始できますか？」
closed 後に同じ不具合かも
closed 後に別件を安くやってほしい
closed 後にファイルを送りたい
closed 後に返金/保証を聞かれる
Priority 6: 短文化 / 人間の担当者っぽさ batch

これはまだ主戦場にしない方がいいです。やるなら上記 regression が r0 で安定してからです。

理由は、短文化を先にやると、r0 がまた「確認します」「大丈夫です」「見ます」だけに戻る危険があるからです。

11. 最重要懸念 3つ
1. r0 の初手分岐ミスを gold/reviewer で吸収し続けている

これが最大の懸念です。

r1 で直るのは良いことですが、同じ型が r0 で繰り返し崩れるなら、gold だけでは接続が弱いです。特に 非Stripe、緊急復旧、保証、仕様境界、purchased 進捗、quote_sent/closed phase は renderer の判断順序へ戻すべきです。

このまま #RE を続けると、r1 品質は上がりますが、r0 は「近い gold を引けたときだけ良い」状態が残ります。

2. prequote の deterministic guard が薄い

多くの危険型は prequote にありますが、common validator は主に quote_sent/purchased/delivered/closed 側です。
そのため、prequote で起きる以下が r0 で抜けやすいです。

旧三点セット
非Stripe broad acceptance
緊急復旧 overpromise
保証/返金/キャンセル overclaim
秘密値/外部共有/内部サービス leak
同義反復

ここは大規模再設計ではなく、prequote 専用 validator を薄く足せばよいです。

3. skill/gold の増殖で template-first に戻るリスク

現在の skill は思想としては freeform を守っています。しかし、ここに r0 失敗型を全部追記し始めると、Writer はまた「全部の注意を説明する」方向へ寄ります。

危険な未来像はこれです。

validator は増えず、renderer の判断順序も増えず、gold と skill だけが増える。
その結果、r0 は長く、保険が多く、内部構造が透け、buyer に関係ない保証/返金/追加料金説明まで出る。

今やるべきなのは、Gold 34 を追加することではなく、Gold 26〜33 から再発部分を renderer/validator/platform_contract/service_facts へ抽出し、gold を参照 anchor に戻すことです。
