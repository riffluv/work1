1. 結論
6 lens は、全部を独立 top-level lens にしない方がよいです。
ただし、agency_alignment は今回の「15,000円で相談できます」問題を説明する力が強く、jp_business_native_naturalness 配下の正式 named sub-lens として昇格する価値があります。
現状の設計はすでに、hard fail と soft lens の分離、#RE only と #R reproduced の切り分け、writer_candidate_manual を #R 相当候補として監査する運用がかなり整理されています。監査プロンプトでも、非公開 service leak・phase drift・secret 値要求・保証断定などだけを deterministic hard fail にし、jp_business_native_naturalness や conversation_flow_naturalness は soft lens として扱う方針が明記されています。
今回の B01 は、すでに 15,000円で相談できます から 15,000円でご依頼いただけます に修正されており、これは単なる語感ではなく、buyer の「依頼できますか？」に対して seller 側が「ご依頼いただけます」と返すことで、相談相手・依頼先・対応主体を一致させた修正です。change log でも、この修正は「返信者自身の対応ではなく第三者への相談にも見える余地」を消すためと整理されています。
私の判断はこうです。
lens扱いpromise_consistency独立 soft lens 維持。自然さではなく、約束レベルの意味整合。hard fail へ昇格する条件もある。conditional_scope_claritypromise_consistency / transaction_model_gap の subtype として当面運用。単独正式化はまだ早い。ack_to_answer_bridgeconversation_flow_naturalness の subtype。独立 lens 化は不要。permission_benefit_alignmentagency_alignment の近縁 subtype。ただし 大丈夫です 問題が再発しやすいので named sub-lens として残す価値あり。unnecessary_refusal_frameconversation_flow_naturalness / jp_business_native_naturalness の subtype。hard 化しない。agency_alignment正式 named sub-lens 候補。かなり重要。 permission_benefit_alignment より上位の「主体・行為・向き」問題として扱う。
つまり構造としては、hard guard は増やさず、jp_business_native_naturalness の中に named sub-lens を足すのが一番安全です。
agency_alignment は “AIっぽい” の深い原因を説明できますが、広すぎる validator にすると過剰矯正になるので、reviewer prompt / gold / human audit 中心、lint は狭く warning のみがよいです。

4. agency_alignment の位置づけ
価値はかなり高いです
agency_alignment は、AI返信の弱点として本質的です。理由は、LLM が日本語の「主語省略 + 可能表現 + 丁寧語」を便利に使いすぎるためです。
相談できます、確認できます、進められます は文法的には自然でも、取引上の主体が曖昧になります。
今回の例では、buyer は「依頼できますか？」と聞いています。
ここで seller が「相談できます」と返すと、次の3つがぼやけます。
buyer が seller に依頼できるのかbuyer がどこかに相談できるのかseller が正式に受けるのか、入口相談だけなのか
だから修正文は、単に丁寧にしたのではなく、行為を buyer → seller の依頼として戻しているのが本質です。
他の方で原因が分からなかった状態でも、Stripe決済後に注文が作られない不具合として、15,000円でご依頼いただけます。
この修正は change log でも、buyer の主質問が 依頼できますか？ なので返答も ご依頼いただけます とする方が対応主体と依頼可否がずれない、と整理されています。
permission_benefit_alignment とは別だが、親子関係に近い
permission_benefit_alignment は、agency_alignment の一部と見てよいです。
ただし運用上は分けた方がよいです。
agency_alignment  └ permission_benefit_alignment


agency_alignment: 誰が何をするか。依頼先・確認主体・作業主体・第三者感。


permission_benefit_alignment: その表現が buyer の負担を軽くしているのか、seller が buyer の行動を許可しているように見えるのか。


ログや関連コードは、お支払い完了後で大丈夫です は、大丈夫です の語感問題だけではなく、buyer が支払い後に材料を送る負担を seller が許可しているように見える問題です。change log でも、支払い後材料共有で 大丈夫です を使うと seller 側の許可調に見えるため、buyer の負担を軽くする用法と、支払い・購入・材料送付で許可調に見える用法を分けて観察すると整理されています。
severity はこう置く
場面severity依頼可否・価格・支払い・作業開始・納品・返金・無料対応・closed 後導線で主体ズレがあるfix recommended。誤読が強ければ hard fail に昇格。主体ズレによって public/private、phase、secret、payment route、promise が壊れるhard fail。ただし hard の理由は agency_alignment ではなく壊れた boundary 名で付ける。文法上は自然で、取引主体も誤読されないが、よりよい表現があるpreference。「相談できます」などが buyer の質問と一致している問題なし。

基本基準

7. 監査プロンプト文案
以下は Codex xhigh の監査プロンプトへそのまま入れられる形です。
既存の Pro後の補助監査レンズ の後、または jp_business_native_naturalness の直後に追加するのがよいです。
追加 soft lens: `jp_business_native_naturalness`日本の実務チャットとして、内容が正しいだけでなく、担当者が送信直前に普通に書いたように読めるかを見てください。目的は「やさしくする」ことではありません。価格、scope、phase、secret、public/private、payment route、作業可否、成果物 promise は変えないでください。見る対象は、主質問への直答、buyer/seller の行為主体、支払い・材料共有・確認開始の導線、受け止めから主回答への接続、不要な拒否フレーム、`できます` 系の主体ぼやけです。この lens は hard fail ではありません。ただし、自然化によって price / scope / phase / payment route / secret handling / public-private boundary が変わった、buyer の主質問への直答が消えた、未約束の作業・無料対応・返金・保証・外部共有を足した場合は deterministic な事故として扱ってください。preference only:- 句点の有無だけ- やや硬いが意味・導線・安全境界が明確- より自然な言い換えはあるが、修正すると price / scope / phase が揺れそう- `はい、` `まずは` `大丈夫です` `確認できます` `相談できます` などの語が、文脈上自然に機能している
追加 soft lens: `agency_alignment`返信内で、誰が何をするのか、誰に向けて許可・依頼・案内しているのかが、buyer の主質問とズレていないかを見てください。特に `相談できます` `確認できます` `対応可能です` `進められます` `大丈夫です` `ご案内します` のように主語が省略されやすい表現で、buyer / seller / 第三者、依頼者 / 作業者 / 確認者の向きが曖昧になっていないかを確認してください。例:- buyer が `依頼できますか？` と聞いている場合、`相談できます` だけで返すと、seller へ依頼できるのか、第三者へ相談できるのかがぼやける可能性があります。- この場合は `15,000円でご依頼いただけます` `この不具合として対応できます` のように、依頼先と対応主体が見える表現を優先してください。- buyer が `相談して大丈夫ですか？` と聞いている場合は、`相談できます` は自然なことがあります。blanket NG にしないでください。hard fail 条件:- agency のズレによって、支払い前作業開始、未約束の成果物返却、外部共有、返金/無料対応、closed 後作業、public/private leak などの deterministic 事故に接続している場合。- この場合、hard fail 名は `agency_alignment` ではなく、実際に壊れた boundary 名で付けてください。fix recommended 条件:- 価格、依頼可否、作業開始、納品、追加料金、closed 後導線など取引判断に関わる箇所で、主体・依頼先・行為の向きが曖昧な場合。preference 条件:- 誰が何をするかは明確で、より自然な言い換えがあるだけの場合。
追加 soft lens: `permission_benefit_alignment``大丈夫です` `送っていただいて大丈夫です` などの許可・安心表現が、buyer の負担を実際に軽くしているのか、seller が buyer の支払い・作業・材料送付を許可しているように見えていないかを見てください。自然な用法:- `値そのものは送らなくて大丈夫です`- `ログの意味が分からない状態でも相談できます`- buyer の心理負担または実務負担を実際に下げている要注意:- `ログや関連コードは、お支払い完了後で大丈夫です`- `お支払い後に送っていただいて大丈夫です`- 支払い、購入、材料送付、作業開始の導線で、seller が buyer の負担を許可しているように見える場合推奨:- `お支払い完了後に、トークルームで共有してください`- `共有いただければ、そこから確認に入ります`- `zip か関係ファイルをまとめた形で進められます`hard fail 条件:- `大丈夫です` によって、支払い前作業開始、外部共有、secret 値送付、未約束の対応可否などを誤って許可している場合。preference 条件:- buyer の負担を下げる自然な `大丈夫です` で、意味・導線・安全境界が明確な場合。
追加 soft lens: `unnecessary_refusal_frame`協力的な buyer に対して、必要以上に拒否・否定から入っていないかを見てください。phase boundary や外部共有禁止などの境界は必ず守りますが、buyer が単に共有タイミングや進め方を確認している場合は、拒否文より手順案内を先に置いた方が自然なことがあります。要注意:- `支払い前にスクショを送っていただいて、こちらで確認を先に始める形ではありません`- `先に確認はしません`- buyer が強い要求をしていないのに、防御的・規約説明的に見える否定先行推奨:- `スクショやログは、お支払い完了後にトークルームで共有いただければ進められます`- `今の段階で先に送っていただく必要はありません`- `原因確認や作業は、お支払い完了後に始めます`hard fail 条件:- 不要な拒否フレームを避けるために、支払い前作業、secret 値、外部共有、直接 push、本番反映、closed 後作業などを許可してしまった場合。preference 条件:- 境界説明として必要で、少し硬いが buyer が次に何をすればよいか明確な場合。
追加 soft lens: `promise_consistency`返信内で、前段の留保・不可・条件付き回答を、後段の成果物・納期・次アクションが上書きして見えないかを確認してください。1文ごとの事実が正しくても、buyer が通しで読んだ時に、未約束の修正完了・原因確定・購入前確認・無料対応・返金・closed 後作業を約束しているように読める場合は指摘してください。特に見る例:- `必ず直せるとはお約束できません` の後に、`修正済みファイルの返却まで進められます` と無条件に書いていないか- `購入後に確認します` と言った後に、購入前の原因確認開始を約束していないか- `secret 値は不要です` と言った後に、値入り `.env` や API キー送付を求めていないか- `closed 後は実作業しない` と言った後に、この場で修正や修正済みファイル返却を約束していないか- 複数症状で、`同じ原因なら範囲内 / 別原因なら相談` の条件が購入導線で消えていないかhard fail 条件:- public/private 事故- phase drift- secret 値要求- 成功保証・返金保証・無料対応の断定- scope / price / payment route の事実変更- 外部共有、直接 push、本番デプロイ誘導- closed 後の旧トークルーム継続作業の約束fix recommended 条件:- 明示断定ではないが、buyer には未約束の約束として読める場合。- 修正例は、固定してよい約束を弱めず、未確認の成果だけ条件付きにしてください。preference 条件:- 条件・約束レベルは明確で、順序や語感を整える程度の場合。
