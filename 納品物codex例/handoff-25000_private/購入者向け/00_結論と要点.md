# 00_結論と要点.md

## 1. 今回の対象フロー
- 今回見た処理: 公開 embed で概算を表示し、最後のステップで lead を保存する流れ
- 対象範囲: `/embed/[publicId]` 表示から `/api/embed/[publicId]/lead` 保存と通知まで

## 2. 結論
- このフローは `web/src/app/embed/[publicId]/page.tsx` から始まり、実体は `web/src/components/embed-runtime.tsx` で進みます
- まず見るべき場所: `web/src/components/embed-runtime.tsx` と `web/src/app/api/embed/[publicId]/lead/route.ts`
- いちばん危険な箇所: `lead/route.ts` に保存・利用上限・通知が集まっており、後段失敗時の挙動を誤解しやすい点
- 今すぐの次の1手: `embed-runtime.tsx` の最終ステップ条件と、`lead/route.ts` の保存順序を先に確認してください

## 3. 危険箇所 上位3
1. `embed-runtime.tsx` のステップ制御
   最終ステップ前に unlock が出ると、lead 保存前に詳細結果が見えてしまいます
2. `lead/route.ts` の保存順序
   lead 保存後に usage 更新と通知が続くため、部分成功を見落としやすいです
3. `calculate.ts` と `lead-brief.ts`
   見えている概算と保存される案件概要が、設定変更でずれるおそれがあります

## 4. 次の着手順
1. `embed-runtime.tsx` で最終ステップ到達前に lead gate が出ないことを確認する
2. `lead/route.ts` で `insert -> usage更新 -> event記録 -> 通知` の順を確認する
3. LP テンプレを1本決めて、表示結果と `lead_brief` の内容が噛み合うか確認する

## 5. 今回の範囲外
- calculator 作成・編集UI
- publish 制限や billing 画面
- Stripe Checkout / Webhook の課金フロー
