## 1. 対象と目的
今回の対象は、価格ページから Stripe Checkout を開始し、決済成功後に `team` の購読状態を更新し、ダッシュボードに現在プランを反映する流れです。  
このリポジトリでは、初回更新を `app/api/stripe/checkout/route.ts` が担当し、その後の購読状態同期を `app/api/stripe/webhook/route.ts` と `lib/payments/stripe.ts` が担当します。  
目的は、次の担当者が「最初の更新はどこで行われるか」「後続の同期はどこで行われるか」「ダッシュボード表示までどこでつながっているか」を短時間で追える状態にすることです。

## 8. 危険箇所
### 8-1. `app/api/stripe/checkout/route.ts`
- 危ない理由: 決済成功後の初回反映はここで完了します。`session_id` が無い、`session.customer` が展開されない、`subscriptionId` が取れない、`client_reference_id` から user/team が引けない、のどれかで止まると、決済だけ成功して画面は無料のままになります
- 先に見るファイル: `lib/payments/stripe.ts`, `lib/db/schema.ts`
- 典型的な事故: Stripe ダッシュボードでは支払い成功が見えるのに、`teams.subscriptionStatus` が更新されず、ダッシュボードでは `No active subscription` のままになる

### 8-2. `app/api/stripe/webhook/route.ts`
- 危ない理由: ここでは `customer.subscription.updated` / `deleted` だけを受けています。`checkout.session.completed` をここでは扱っていないため、「初回反映は checkout 成功ルート、継続同期は webhook」という役割分担を見落とすと、どちらが壊れているか切り分けにくくなります
- 先に見るファイル: `lib/payments/stripe.ts`
- 典型的な事故: 初回反映が失敗しているのに webhook 側だけ見続けて、原因の入口を取り違える

### 8-3. `lib/payments/stripe.ts` の `handleSubscriptionChange()`
- 危ない理由: Webhook 側の更新は `getTeamByStripeCustomerId()` で team を引き、`updateTeamSubscription()` で `subscriptionStatus` / `planName` などを更新します。初回の checkout ルートで `stripeCustomerId` が保存できていないと、後続 webhook の同期が成立しません
- 先に見るファイル: `lib/db/queries.ts`
- 典型的な事故: webhook は受け取れているのに `Team not found for Stripe customer` が出て、購読状態が追従しない

### 8-4. `app/(dashboard)/dashboard/page.tsx`
- 危ない理由: 画面側は `subscriptionStatus === 'active'` または `'trialing'` の時だけ有料寄りに表示します。保存値と画面表示条件の対応を見ずに DB 側だけ追うと、「更新済みなのに未反映」に見える理由を取り違えます
- 先に見るファイル: `app/layout.tsx`
- 典型的な事故: DB に状態は入っているが、画面では無料表示に見えて、利用者からは未反映と見なされる

## 9. 次の着手順
### 9-1. 初回反映の入口を確認する
- 目的: まず「決済成功直後の反映」がどこで完了するかを押さえる
- 最初に開くファイル: `app/api/stripe/checkout/route.ts`
- 確認すること: `session_id` 取得、`session.customer` / `session.subscription` 展開、`client_reference_id` から user/team を特定する流れ、`teams` テーブルへ保存している項目
- 次に進む目安: ここで `stripeCustomerId` / `subscriptionStatus` / `planName` を正しく保存していると確認できたら、次に webhook 側の後続同期へ進む

### 9-2. 後続同期の入口を確認する
- 目的: 後続の購読状態変更がどこで反映されるかを押さえる
- 最初に開くファイル: `app/api/stripe/webhook/route.ts`
- 確認すること: 受けているイベント種別が `customer.subscription.updated` / `deleted` だけであること、署名検証失敗時に 400 を返すこと、`handleSubscriptionChange()` へ渡していること
- 次に進む目安: Webhook 入口が妥当なら、`lib/payments/stripe.ts` の `handleSubscriptionChange()` に進み、team 特定と更新条件を確認する

### 9-3. 表示条件とのズレを確認する
- 目的: 保存後に画面側がどう見せるかを切り分ける
- 最初に開くファイル: `app/(dashboard)/dashboard/page.tsx`
- 確認すること: `teamData.planName` と `teamData.subscriptionStatus` をどう表示しているか、`active` / `trialing` 以外の扱いがどうなっているか
- 次に進む目安: 画面表示条件に問題がなければ、`app/layout.tsx` の `/api/team` 取得と `lib/db/queries.ts` の team 取得へ進み、読み出し側のズレを確認する
