# 00_結論と要点_購入前見本.md

## 1. 今回の対象フロー
- 題材にした公開 repo: `nextjs/saas-starter`
- 今回見た処理: Stripe で決済完了したあとに、アプリ側の `team subscriptionStatus` とプラン表示を更新する流れ
- 対象範囲: 価格ページの Checkout 開始から、成功後の更新、Webhook による購読状態同期、ダッシュボード表示まで

## 2. 結論
- このフローは、`app/(dashboard)/pricing/page.tsx` から Checkout を開始し、`app/api/stripe/checkout/route.ts` が最初の会員状態更新を担当し、その後 `app/api/stripe/webhook/route.ts` と `lib/payments/stripe.ts` が購読状態の同期を継続する構成です
- まず見るべき場所: `app/api/stripe/checkout/route.ts` と `app/api/stripe/webhook/route.ts`
- いちばん危険な箇所: 決済成功直後の更新が `checkout` 成功ルートと `webhook` 同期の二段構えになっているため、どちらかが崩れると「決済は成功しているのにダッシュボードは無料のまま」に見える点です
- 今すぐの次の着手順: まず `app/api/stripe/checkout/route.ts` で `teams` テーブルに何を書き込むかを確認し、その次に `lib/payments/stripe.ts` の `handleSubscriptionChange()` で後続同期の条件を確認してください

## 3. 危険箇所 上位3
1. `app/api/stripe/checkout/route.ts`
   `session_id` 取得、Stripe からの `customer` / `subscription` / `product` 取得、ユーザーと team の特定までを一気に行っています。ここで 1 つでも失敗すると、決済成功後の最初の更新が止まります
2. `app/api/stripe/webhook/route.ts` + `lib/payments/stripe.ts`
   Webhook 側は `customer.subscription.updated` / `deleted` を受けて `handleSubscriptionChange()` に渡しています。初回更新と後続同期の更新元が分かれているため、どちらを基準に状態を見るかが曖昧になると追跡が難しくなります
3. `app/(dashboard)/dashboard/page.tsx`
   画面側は `subscriptionStatus === 'active'` / `'trialing'` の時だけ有料寄りに見せています。DB 保存値は正しくても、表示条件の解釈が狭いと「反映されていない」と見えるおそれがあります

## 4. 次の着手順
1. `app/api/stripe/checkout/route.ts` を開いて、`teams` テーブルに保存している `stripeCustomerId` / `stripeSubscriptionId` / `stripeProductId` / `planName` / `subscriptionStatus` を確認する
2. `lib/payments/stripe.ts` の `handleSubscriptionChange()` を開いて、Webhook 到着後に同じ項目をどう上書きするかを確認する
3. `lib/db/queries.ts` と `app/(dashboard)/dashboard/page.tsx` を見て、保存値と画面表示がどの条件でつながっているかを切り分ける
