# bugfix-15000 返信監査 batch

## 対象

RE-2026-04-28-bugfix-38-post-fix-r0-stability-r1

CHG-068〜073 で戻した r0 安定化候補（旧三点セット、purchased 分岐、delivered yes/no、context bleed、response weight）を、近縁ケースで再確認する batch です。r0 監査で指摘された B02 の返金/キャンセル回答不足と B07 の症状受領前の時刻約束を r1 として反映しています。

## 目的

- 非Stripe決済名が出ても、決済サービス全体ではなく Next.js 側 Webhook/API 受信処理へ閉じられるかを見る。
- purchased の遅延/返金不満、新機能追加混入を、進捗・返金・scope に分けて返せるかを見る。
- delivered の承諾 yes/no を、汎用 poststate fallback に流さず短く直答できるかを見る。
- closed 後の「このメッセージで症状を送ってよいか」に、浮いた「はい」や旧トークルーム継続前提を出さないかを見る。
- context bleed、旧三点セット、同義反復、phase drift が戻っていないかを見る。

## 構成

- B01: prequote / PAY.JP + 料金質問
- B02: purchased / 遅延不満 + 返金希望
- B03: prequote / フロント範囲 + 2症状

- B05: purchased / 新機能追加混入
- B06: delivered / 承諾前確認 yes/no
- B07: closed / 症状をこのメッセージで送ってよいか

## 監査で見てほしい点

- 旧三点セット（`内容ありがとうございます` / `この不具合なら15,000円` / `どこで止まっているか確認`）が残っていないか。
- 非Stripe決済で、PAY.JP 側そのものの仕様調査まで受けるように見えないか。
- 料金質問に対して、15,000円固定と対象範囲が buyer に見えるか。
- purchased で、進捗/返金/新機能追加を混同していないか。
- delivered で、承諾前確認の yes/no に短く答えているか。
- closed で、確認材料の受領と修正作業の正式導線を分けているか。
- response weight が過剰に重くなっていないか。ただし安全境界を削りすぎていないか。
