- `handoff-25000` や 25,000円側の外向け導線が出ていないか確認する
- delivered / closed の generic fallback が再発していないか確認する

監査で見てほしい点:

- 主質問に最初に答えているか
- `15,000円 / 不具合1件 / 同一原因 / 追加料金は自動発生しない` の軸が崩れていないか
- 購入前・見積り提案後・購入後・納品後・closed 後の phase が混ざっていないか
- 購入前に原因調査・コード確認・具体的な修正手順へ入っていないか
- delivered 後に、承諾前確認 / 承諾後メッセージ確認 / 実作業継続不可を分けているか
- closed 後に確認材料と実作業の境界が曖昧になっていないか
- キャンセル/返金を保証していないか。ただし未完成納品へ押し切る文になっていないか
- 秘密情報、外部共有、外部連絡、直接 push、本番デプロイ、非公開サービス導線が出ていないか
- 日本語がぶつ切り、AI 的な丸引用、過剰防御文になっていないか

戻し先の判定方針:

- `validator`: deterministic に再発し、機械的に落とせる事故
- `gold`: 判断順序として残す価値があり、テンプレ丸写しではなく参照 anchor にするもの
- `reviewer_prompt`: 常時本文に出すと長文化するが、監査レンズとして有効なもの
- `case_fix`: その case だけの文体・温度・言い換え

出力形式:

1. まず一文で結論
   - `採用可 / 条件付き採用 / 要修正` のどれかを最初に明記してください
   - そのうえで、「今回はそのまま使ってよいか / どこを直してから使うべきか」を1〜2文で書いてください
2. 共通して良い点
   - 2〜4点まで
3. 危ない点 / 共通 drift
   - 本当に再発しそうなものだけ
   - 細かい好みより、実務リスクと buyer の違和感を優先してください
4. ケース単位の必須修正
   - 必須修正がなければ `必須修正なし` と明記してください
   - 必須 / 軽微 を分けてください
5. 学習判定
   - 必須 / 軽微 の指摘ごとに、必要な場合だけ短く添えてください
   - `直す単位`: `case_fix / pattern_candidate / rule_return_candidate / preference / reject`
   - `再発根拠`: `なし / 同batch内 / 過去batch同型 / 既存rule未反映疑い`
   - `既存rule hit`: `あり / なし / 不明`
   - `戻し先候補`: `batch / reviewer_prompt / gold / rule / service_facts / validator / none`
6. rule に戻すべき再発パターン
   - `reply-only / common` を分けてください
   - 再発性が弱いものは無理に rule 化しないでください
7. 採点
   - 必要なら `直答 / scope順守 / 情報節約 / 自然さ / 次アクション明確さ` を短く評価してください
8. 最後に短くまとめ
   - `今回は止めてよい / 1〜2点だけ直す / まだ危ない` のどれかを、人間向けに短くまとめてください

まだ `rule / validator / gold / reviewer_prompt` には反映しないでください。監査結果を見て、再発性があるものだけ最小反映します。
