# bugfix-15000 返信監査 batch

対象: `RE-2026-04-28-bugfix-35-buyer-state-weight-near-miss-r3`

目的:

- CHG-070 / CHG-071 後に、`buyer_state_ack_gap` が必要なケースで効くかを見る
- 他者修正失敗・再発疲弊・購入後進捗不安で、症状や料金だけを受けて buyer の状態を落としていないかを見る
- 通常技術相談や短い相談で、状態受けが過発火して長文化していないかを見る
- `response_weight_mismatch` は今回は観察のみとし、相手の文量や温度に対して返信が重くなりすぎていないかを見る
- purchased / delivered / closed の phase 境界、確認材料と実作業の導線、追加料金不安が崩れていないかを見る
- `handoff-25000` や 25,000円側の外向け導線が出ていないか確認する

今回の構成:

- B01: closed / 再発疲弊 + 無料対応期待 + 費用不満
- B02: prequote / 他者修正失敗 + 不信 + 15,000円対応可否
- B03: purchased / 進捗不安 + 短い返答希望
- B04: prequote / 通常技術症状 + 短い相談
- B05: prequote / 設定かコードか分からない困惑型
- B06: delivered / 短い承諾確認
- B07: closed / 通常再相談 + まずメッセージで送ってよいか

監査で見てほしい点:

- 主質問に最初に答えているか
- buyer が書いた具体事情・不安・比較軸を1〜2個拾えているか
- 怒り・疲弊・不安・焦り・不信・困惑などの状態シグナルを、症状・価格・手順だけで受け流していないか
- `buyer_state_ack_gap` を使う場合、状態受けが謝罪・過失認定・返金断定・無料対応約束に広がっていないか
- 通常の技術相談で、不要な共感文や長文化が出ていないか
- 短い buyer 文に対して、返信が契約説明のように重くなりすぎていないか
- `bugfix-15000` 以外の未公開サービス名・価格・購入導線を出していないか
- 購入後の進捗不安で、空の `確認中です` や追加症状テンプレ誤発火に戻っていないか
- delivered / closed で、承諾前後・旧トークルーム継続・実作業導線を混同していないか
- `transaction_model_gap` を使う場合は、必要に応じて `transaction_clarity / phase_route_clarity / work_payment_boundary_clarity / buyer_not_lost / free_support_expectation_risk / responsibility_over_admission_risk / request_minimality` のどれが崩れているかを短く特定してください
- 文面がぶつ切り、丸引用、テンプレ反復、内部分類語、surface_overexposure になっていないか

戻し先の判定方針:

- `validator`: deterministic に再発し、機械的に落とせる事故
- `gold`: 判断順序として残す価値があり、テンプレ丸写しではなく参照 anchor にするもの
- `reviewer_prompt`: 常時本文に出すと長文化するが、監査レンズとして有効なもの
- `renderer`: 同じ r0 崩れが複数 batch で再発し、case_fix では止まらないもの
- `case_fix`: その case だけの文体・温度・言い換え
- `preference`: 好み差で、今回の batch 採用を止めるほどではないもの

出力形式:

1. まず一文で結論
   - `採用可 / 条件付き採用 / 要修正` のどれかを最初に明記してください
   - そのうえで、「今回はそのまま使ってよいか / どこを直してから使うべきか」を1〜2文で書いてください
2. 共通して良い点
   - 2〜4点まで
3. 危ない点 / 共通 drift
   - 本当に再発しそうなものだけ
   - 細かい好みより、実務リスクと buyer の違和感を優先してください
4. ケース単位の必須修正
   - 必須修正がなければ `必須修正なし` と明記してください
   - 必須 / 軽微 を分けてください
5. 学習判定
   - 必須 / 軽微 の指摘ごとに、必要な場合だけ短く添えてください
   - `直す単位`: `case_fix / pattern_candidate / rule_return_candidate / preference / reject`
   - `再発根拠`: `なし / 同batch内 / 過去batch同型 / 既存rule未反映疑い`
   - `既存rule hit`: `あり / なし / 不明`
   - `戻し先候補`: `batch / reviewer_prompt / gold / rule / renderer / service_facts / validator / none`
6. rule に戻すべき再発パターン
   - `reply-only / common` を分けてください
   - 再発性が弱いものは無理に rule 化しないでください
7. 採点
   - 必要なら `直答 / scope順守 / 情報節約 / 自然さ / 次アクション明確さ / 状態受け` を短く評価してください
8. 最後に短くまとめ
   - `今回は止めてよい / 1〜2点だけ直す / まだ危ない` のどれかを、人間向けに短くまとめてください

まだ `rule / validator / gold / renderer / reviewer_prompt` には反映しないでください。監査結果を見て、再発性があるものだけ最小反映します。
