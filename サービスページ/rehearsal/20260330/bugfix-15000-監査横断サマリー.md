# bugfix-15000 監査横断サマリー

対象:
- M01-M08: 1件ずつ個別監査
- M09-M20: batch-01
- T01-T10: batch-02
- T11-T20: batch-03
- C01-C10: batch-04

結論:
- 構造の大枠はかなり安定している
- 残ったズレは、ほぼすべて `再発パターン` として rule に戻せる種類だった
- `1件ずつの微修正` から、`batch で再発 drift を取る` 進め方へ切り替えたのは正しかった

## 全体の到達点

- スコープ境界はかなり明確
  - 本番デプロイ代行を受けない
  - Stripe 以外でも Next.js / API Route 側なら切り分ける
  - 新規実装と不具合修正を分ける
- 価格説明はかなり安定
  - `1件15,000円が基本`
  - 値引き不可は曖昧にしない
  - 追加費用は `ご相談なく発生しない`
- 日本語の drift も主要なものは吸収済み
  - `形になります`
  - `見込みです`
  - `その前提で`
  - `入口`
  - `確認できる可能性があります`

## batch で見えた共通 drift

### 1. `見られる` の多用
- `対応できる / 確認できる` に寄せるべき場面で、受動的な `見られる` が残りやすかった
- これは全batchで出たので、rule 化済み

### 2. クロージングの固定化
- 見積もりでは `内容を見て、〜を先にお返しします`
- 購入後では `そのあと、48時間以内を目安に一次結果をご案内します`
- closed後では `進め方を先にお返しします`
- いずれも batch で並べるとテンプレ感が強く出た
- これも rule 化して、用途ごとにローテーション・段数可変へ修正済み

### 3. 抽象複合語
- `入口`
- `確認の進め方`
- `この観点になります`
- `どの形で`
- こうした語は、1件だと流れても batch で見ると再発していた
- 具体語へ置換する方針に整理済み

### 4. 購入後の時刻コミットの過剰固定
- `本日[時刻]まで` + `48時間以内`
- これを全購入後メッセージに貼るのは不自然
- 初回受領 / 本格着手だけ2段
- 追加情報返信 / 再修正 / 完了間近は1段
へ整理済み

### 5. closed後の無料中間状態
- 再発を `まだ決めません` で保留
- 再送を `進め方を返します` で複雑化
- `少しだけなら` で無料と有料の境界を曖昧化
- これは closed 後特有の最大リスクだった
- `説明・補足 / 再発・新規 / 再送` の3分類に整理済み

## 重要だった個別論点

### M13 値引き交渉
- `確認だけの進め方` のような未設計の逃がし道を出すとサービス設計と矛盾する
- 値引き不可なら、そのまま価格を固定で返す方が安全

### M15 新規実装と修正の境界
- `受信処理が空っぽ` は修正ではなく新規実装
- 相手の言い換えに引っ張られず、範囲外の理由を短く言うのがよかった

### T13 範囲外の機能追加
- `できますが` で始めると、着地先のない期待を作る
- 未公開サービスがある状況では特に危険
- `別件です` から入る rule に修正

### C01 / C05 / C08 closed後
- 無料診断
- 再送の過剰複雑化
- `少しだけなら`
- この3つが closed 後の典型事故として明確になった

## いまの判断

- bugfix-15000 の返信 OS は、実務投入可能な水準まで来ている
- ただし、今後の改善は `1件ずつ全文を詰める` より、`batch 監査で再発 drift を拾って rule に戻す` 方が効率がいい
- 単発レビューは、重要案件か、batch で浮いたケースだけに絞るのがよい
