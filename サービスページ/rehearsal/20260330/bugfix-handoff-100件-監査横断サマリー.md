# bugfix-15000 + handoff-25000 監査横断サマリー

## 総評

100件を通した結論は、`1件ずつのモグラたたき` より `batch単位で再発 drift を炙る` 方が明確に有効でした。

- `bugfix-15000`: 返信文の自然さより先に、`価格 / スコープ / closed後 / 無料の中間状態` の事故を潰せた
- `handoff-25000`: `整理であって修正ではない` という軸を保ったまま、購入後・クローズ後で品質が大きく安定した

特に効いたのは、個別文面の採否ではなく、再発パターンを `rule / golden / batch` に戻したことです。

## 主要な改善点

### bugfix-15000

- `closed後` に無料の中間状態を作らない
- `1件15,000円が基本` を、料金質問には明示する
- `確認だけ` `入口` `見られる` のような曖昧語を減らす
- `48時間以内の一次結果` を、購入後の初回受領だけに限定する
- ファイル再送は `再送します。少しお待ちください。` の即対応にそろえる
- `15,000円が上限` かつ別原因の可能性が高い時は、安易に販売せず最悪ケースを開示する

### handoff-25000

- バグ文脈があっても、`修正は含まない` を必要箇所で明示する
- 相手が `修正不要` `まず構造把握したい` と書いている場合は、それを読み飛ばさない
- 金額を聞かれたら `追加1件あたり15,000円目安` のように具体数字を返す
- 固定3択の聞き返しをやめて、相手が整理したい流れをオープンに聞く
- クローズ後は `説明・補足 / 再送 / 新規相談` の3分類を一貫して守る

## 横断で見えた再発 drift

### 1. 曖昧語

以下は、全体を通して再発しやすい語でした。

- `見られる`
- `入口`
- `確認だけ`
- `進め方`
- `その前提で`
- `形になります`
- `見込みです`

対処:

- `対応できる`
- `確認できる`
- `進める前にご相談します`
- `1件15,000円が基本`

のように、相手の判断に必要な情報へ戻す。

### 2. テンプレ末尾の反復

- `内容を見て、〜を先にお返しします`
- `48時間以内を目安に一次結果をご案内します`

のような固定末尾は、単体では無難でも batch で見ると drift になりました。

対処:

- 見積もり前: 相手が今送るべき情報だけを具体化する
- 購入後初回: 時刻コミットを入れてよい
- 追加情報返信 / 完了間近: 1段コミットか、不要なら省略
- クローズ後: 中間状態を作らず即答する

### 3. 相手の明示意図の読み飛ばし

典型例:

- `修正は不要`
- `まず構造を把握したい`
- `料金はどうなりますか`

これらを読み飛ばして別の質問を返すと、AI感より前に実務上の不信感を生みます。

対処:

- 相手がすでに決めていることは受ける
- 足りない情報だけ追加で聞く
- 金額質問には、判断軸だけでなく数字を返す

## batch 監査で見えた構造差

### bugfix

- 見積もり: 価格と範囲の出し方に drift が出やすい
- 購入後: 時刻コミットの貼りすぎが出やすい
- クローズ後: 無料診断や曖昧な無料境界を作りやすい

### handoff

- 見積もり: `整理` と `修正` の境界が最重要
- 購入後: 読み手レベル調整や、危険度の整理は強い
- クローズ後: 3分類が最も安定していた

## 現時点の到達点

- `bugfix-15000` は、実務で事故りやすい返信パターンがかなり潰れた
- `handoff-25000` は、未公開の内部 rehearsal として十分に安定した
- 100件を通して、`局所改善` ではなく `OS の癖` を可視化できた

次に同様の rehearsal を回す時は、1件ずつ全部詰めるより、

1. 草案を一気に作る
2. 5〜10件単位で監査する
3. 再発 drift だけ rule に戻す
4. 個別修正は最後に絞る

の流れを標準にするのが最も効率が良いです。
