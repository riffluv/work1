目的:

- いまの `bugfix-15000` / `handoff-25000` の次に、何をどう出すかの判断材料を残す
- 返信OSを、`テンプレ商品` ではなく `導入型サービス` として成立させる前提を固める
- 外向けには、内部語の `組み込み` ではなく、`あなたのサービス内容に合わせた返信システムを作ります` という売り方へ寄せる

前提:
- 現在の主力は `bugfix-15000`
- `handoff-25000` は準備済みで、近いうちに公開予定

## 次の本命候補

位置づけ:
- `bugfix-15000` と `handoff-25000` の上位に置く、コード側の中間高単価商品
- `今すぐ1件だけ直す` でも `整理メモだけ返す` でもなく、決済まわりを安心して触り始められる状態まで整える商品
- 返信システムより先に出しやすく、今の buyer 文脈から自然に上位提案しやすい

本質:
- 売るのは `小修正1件` ではなく、`主要フローの安定化の初動まで含めた整理`
- `handoff` より一歩深く入り、
  - 主要フロー整理
  - 危険箇所の優先順位
  - どこから直すべきか
  - 修正方針の具体化
  まで返す
- ただし、大規模改修や新規実装に広げず、`最優先の安定化範囲` に絞る

想定 buyer:
- AI で作ったアプリが動いているが、怖くて触れない人
- 外注コードの決済まわりがブラックボックスな人
- bugfix だけだと狭く、handoff だけだと浅いと感じる人
- `まずどこが危ないか知りたい / できれば優先度の高いところだけ前に進めたい` 人

含めるもの:
- 決済まわりの主要フロー整理
- 危険箇所トップ 3〜5
- 関連ファイル一覧
- どこから直すべきかの優先順位
- 修正方針の具体化
- 確認手順
- 引き継ぎメモ

含めないもの:
- CSS / デザイン調整
- 新規 Stripe 実装一式
- 本番デプロイ
- 直接 push
- APIキーや秘密値の受け取り
- リポジトリ全体の仕様書化
- 複数機能の全面改修

方向メモ:
- `優先度1位の小修正1件だけ` だと中途半端に見えやすい
- なので、商品としては `整理 + 危険箇所 + 優先順位 + 安定化の初動まで` を返す形に寄せた方が強い
- 修正を主役にせず、あくまで `安定化の初動まで含む整理商品` として見せる

価格感:
- 初回 3件: `79,800円`
- 通常: `119,800円`
- 実績後: `148,000〜198,000円`

判断メモ:
- `15,000 / 25,000` だけでは手取りが薄くなりやすい
- 返信システムは本命だが、buyer 層が少し変わる
- その前に、いまの技術資産を最も自然に高単価化できるのがこの商品
- 順番としては `bugfix-15000 -> handoff-25000 -> 安定化パック -> 返信システム` が一番勝率が高い

外向け表現の候補:
- AI/外注コードの決済まわりを安定化します
- Next.js / Stripe の保守前整理と安定化をします
- 決済まわりの危険箇所と修正優先度を整理します

### AI返信OS / 顧客対応運用設計

位置づけ:
- 今の返信システムを、他人のサービス運用へ移植する上位商品
- `skillを作る` のではなく、`AIを実務で事故らせない顧客対応構造を設計する` 商品
- テンプレ販売や軽いコンサルではなく、導入型の高単価商品として考える
- 外向けの一番自然な言い方は、`あなたのサービス内容に合わせた返信システムを作ります`

本質:
- AIに返信を書かせること自体ではなく、
  - AIの思考を削らない
  - サービス内容を理解した状態で返す
  - platform 仕様を踏まえて事故らない
  - モグラたたきではなく、再発パターンを構造で止める
 という状態を作ることが価値

強み:
- ココナラの仕様を記憶領域として持たせられる
- サービス理解と価格・禁止事項を構造結合できる
- routing / scope / state / memory がある
- purchased 後の並列案件運用まで視野に入る
- reply lane と implementation lane を分けて、Codex の思考を殺さない
- `15000 / 25000` の実運用で見つけた再発パターンがすでに資産化されている

## この商品が「汎用品」ではなく「導入型」である理由

### 共通資産として持てるもの
- `Classifier -> Writer -> Reviewer` の三層構造
- Writer Brief
- reviewer / lint の基本骨格
- platform knowledge
- acceptance gate の考え方

### 毎回の導入で必要なもの
- その人のサービス理解
- service facts の定義
- route / scope / forbidden moves の定義
- few-shot 例 2〜3件
- 初回バッチ監査と微調整

### つまり何を売るのか
- 売るのは `prompt` ではない
- 売るのは `あなたのサービス内容に合わせた返信システムを作る作業`
- 内部的には返信OSへ落とし込むが、外向けには `返信システムを作る` と表現した方が自然
- だから高単価が成立する

## 売り方の前提

やらない:
- 単なる `返信テンプレ集`
- 単なる `prompt 販売`
- 骨格そのものの安売り

やる:
- `あなたのサービス内容に合わせた返信システムを作ります`
- `platform仕様を踏まえた顧客対応のAI運用を設計します`
- `顧客対応の属人化と炎上リスクを減らす返信の土台を作ります`

外向け表現の候補:
- あなたのサービス内容に合わせた返信システムを作ります
- あなたのサービス向けの自然な返信ルールを作ります
- AIが相手の文章にちゃんと返せる返信システムを作ります
- 顧客対応の返信設計を作ります

※ 内部では `返信OS` や `組み込み` と呼んでよいが、外向けでは `返信システムを作ります` に寄せた方が伝わりやすい

### 外向けで避けたい言い方
- `組み込み`
- `導入支援`
- `診断`
- `コンサル`

理由:
- `何に組み込むのか` が相手には見えにくい
- `導入支援` や `コンサル` は、ラリーや面談が多い重いサービスに見えやすい
- 実際に売りたいのは `材料を受け取って、あなた向けの返信システムを作って返すこと`

## 商品構成の考え方

### 基本の受け取り方
- 面談中心にはしない
- 相手からは、サービスページ、FAQ、よくある相談、断ること、困った返信例などを受け取る
- こちらは、その材料をもとに返信システムを作って返す
- 追加確認は不足がある時だけ最小限にする

### 入口商品
- `返信システム適合チェック`
- いまのサービス内容で返信システム化できるか
- 足りない材料は何か
- bugfix寄りか、handoff寄りか
- どのプランが向くか

判断メモ:
- `診断` という言い方は重い
- 既に相手が高度な返信システムを持っている前提になりやすい
- 実態は `受け入れ判定 + 材料確認` なので、`適合チェック` の方が近い

### 本体
- 1サービス分
- platform 仕様の反映
- サービス理解
- 基本返信導線
- 購入前 / 購入後 / 納品後の phase 判定
- scope と禁止事項の反映
- 思考を落とさない返信運用
- 初回の監査 / 微調整まで含む

### オプション
- purchased memory
- case切替
- 複数サービス対応
- delivery / close 導線
- workspace 全体実装
- 文章学習 / トーン寄せ

## いま分かっている現実

## 注意

- 薄い商品にすると価値が落ちる

- ただし、骨格をそのまま見せると分析・転用されやすい
- そのため、本丸の設計思想や深いレイヤーは隠し、利用面を完成品として渡す方向がよい
