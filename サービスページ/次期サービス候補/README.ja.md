## 次の本命候補

### AI返信OS / 顧客対応運用設計

位置づけ:
- 今の返信システムを、他人のサービス運用へ移植する上位商品
- `skillを作る` のではなく、`AIを実務で事故らせない顧客対応構造を設計する` 商品
- テンプレ販売や軽いコンサルではなく、導入型の高単価商品として考える

本質:
- AIに返信を書かせること自体ではなく、
  - AIの思考を削らない
  - サービス内容を理解した状態で返す
  - platform 仕様を踏まえて事故らない
  - モグラたたきではなく、再発パターンを構造で止める
 という状態を作ることが価値

強み:
- ココナラの仕様を記憶領域として持たせられる
- サービス理解と価格・禁止事項を構造結合できる
- routing / scope / state / memory がある
- purchased 後の並列案件運用まで視野に入る
- reply lane と implementation lane を分けて、Codex の思考を殺さない
- `15000 / 25000` の実運用で見つけた再発パターンがすでに資産化されている

## この商品が「汎用品」ではなく「導入型」である理由

### 共通資産として持てるもの
- `Classifier -> Writer -> Reviewer` の三層構造
- Writer Brief
- reviewer / lint の基本骨格
- platform knowledge
- acceptance gate の考え方

### 毎回の導入で必要なもの
- その人のサービス理解
- service facts の定義
- route / scope / forbidden moves の定義
- few-shot 例 2〜3件
- 初回バッチ監査と微調整

### つまり何を売るのか
- 売るのは `prompt` ではない
- 売るのは `あなたのサービスに合わせて、AIが事故らず返せる状態まで組み込む作業`
- だから高単価が成立する

## 売り方の前提

やらない:
- 単なる `返信テンプレ集`
- 単なる `prompt 販売`
- 骨格そのものの安売り

やる:
- `あなたのサービスに合わせて、AIが事故らず返せる状態まで組み込みます`
- `platform仕様を踏まえた顧客対応のAI運用を設計します`
- `顧客対応の属人化と炎上リスクを減らす返信OSを作ります`

外向け表現の候補:
- AI返信OS
- 顧客対応運用設計
- AI顧客対応導入
- 返信システム導入支援

※ 内部では `返信OS` と呼んでよいが、外向けでは `運用設計` や `導入支援` に寄せた方が伝わりやすい

## 商品構成の考え方

### 本体
- 1サービス分
- platform 仕様の反映
- サービス理解
- 基本返信導線
- 購入前 / 購入後 / 納品後の phase 判定
- scope と禁止事項の反映
- 思考を落とさない返信運用
- 初回の監査 / 微調整まで含む

### オプション
- purchased memory
- case切替
- 複数サービス対応
- delivery / close 導線
- workspace 全体実装
- 文章学習 / トーン寄せ

## 価格感

### 本来価値

1サービス分の導入:
- `150,000〜300,000円`

複数サービス / memory / phase 管理込み:
- `300,000〜500,000円`

workspace レベル:
- `500,000〜1,000,000円+`

判断メモ:
- `3万〜8万` は安すぎる
- その価格だと、骨格価値に対して割が合わない
- 1サービス分をちゃんと入れる時点で、かなり骨格寄り
- 本当に売るなら、相手専用に組み込み、使える完成品として渡す方がよい

### ココナラ初出しの現実解

本体:
- `79,800円`
- 1サービス分
- platform 仕様の反映
- サービス理解
- 購入前 / 購入後の基本返信導線
- 購入後〜納品の phase 判定
- 範囲外・追加対応の基本判断
- 自然な日本語化
- 初回の監査と微調整

オプション:
- 文章トーン寄せ: `+25,000円`
- purchased memory / 継続文脈: `+45,000円`
- 追加サービス1件: `+65,000円`
- 文章学習 / 既存返信への寄せ: `+75,000円`
- case切替 / 複数案件運用: `+70,000円`

補足:
- ココナラの一覧で見える価格帯を踏まえると、初出しは `79,800円` が現実的
- これは `本来価値` より低めだが、初回の成約とレビュー獲得を優先した価格
- 最初の数件で評価が付いたら、段階的な値上げを前提にする

## いま分かっている現実

- `#AR` の外部調査だけでは、サービス固有の崩れまでは分からない
- 実戦テストで初めて見えるものがある
  - 値引き交渉の返し
  - delivered 後の価値不満
  - purchased での外部共有 / 秘密値 / 直接操作の断り方
  - handoff の説明を、buyer の質問ごとに変える必要
- ただし、これは `商品として厳しい` という意味ではなく、`共通骨格` と `個別導入` を分けて売るべきという裏付けになった

## 注意

- 薄い商品にすると価値が落ちる
- ただ冊子や方針メモを渡すだけでは弱い
- 相手が喜ぶのは、サービス理解が入り、実際に返せる状態になった時
- ただし、骨格をそのまま見せると分析・転用されやすい
- そのため、本丸の設計思想や深いレイヤーは隠し、利用面を完成品として渡す方向がよい

## 次にやること

前提:
- 返信OSは、`15000 / 25000` 向けの主要パターンではかなり安定してきた
- ただし、日本語の返信システムとして何を共通化できて、何をサービス固有に残すべきかは、まだ完全には詰め切っていない
- つまり今は、商品化に進む前に、返信OSのブラックボックスをさらに減らす段階

そのため、商品化の作業は、返信OSに十分な自信が持てた時点で次の順に進める。

1. `service facts 導入テンプレ` を作る
- 新しいサービスへ入れる時に
  - 何をヒアリングするか
  - 何を facts 化するか
  - 何を forbidden にするか
  を定型化する

2. `acceptance gate` を明文化する
- `相手にちゃんと返せているか`
- `phase / scope / platform を外していないか`
- `buyer が次に何をすればよいか見えるか`
  を、感覚ではなく通過条件として固定する

3. `導入フロー` をまとめる
- どの順で
  - サービス理解
  - facts 組み込み
  - few-shot 作成
  - 初回バッチ監査
  - 微調整
 まで進めるかを、商品として渡せる形にする

判断メモ:
- 今は、返信OSそのものをさらに詰める方が優先
- 上の3つは、返信OSに十分な自信が持てた段階で着手する
