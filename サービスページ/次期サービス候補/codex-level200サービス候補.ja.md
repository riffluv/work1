# Codex Level200 サービス候補メモ

目的:
- `bugfix-15000` / `handoff-25000` の次に育てる、高単価寄りサービス候補を3本に絞って残す
- 売るのは `AIそのもの` ではなく、内部で鍛えた skill / rule / review loop を使って仕上げた成果物であることを明確にする
- いまの workspace 資産をどう次期サービスへ流用するかを整理し、後でブラッシュアップできる土台にする

前提:
- 内部では `Codex level200` のように、素の Codex より高い品質で案件に挑むイメージを持つ
- ただし外向けには `AIでやります` とは言わず、出品者が高品質に仕上げるサービスとして見せる
- 差別化の中心は、実装力そのものよりも `品質差が出る内部OS` を持っていること
- すでに使い回せる資産として、返信システム、handoff 納品構造、scope 判定、review loop、`#AR` による下調べの型がある

---

## 先に結論

今すぐ着手しやすい順:

1. コード特化の整理メモ上位版
2. 設計監査 / セカンドオピニオン
3. AIっぽくない高品質 UI 改善実装

意味合い:
- `1` は、今の `handoff-25000` を一段上位化する方向で、一番すぐ現実にできる
- `2` は、読む力・切る力・返す力を高単価化しやすい
- `3` は、最もやりがいと伸びしろが大きいが、内部 skill の育成がやや必要

## 1. コード特化の整理メモ上位版

位置づけ:
- `handoff-25000` の上位版
- 単なる `主要1フロー整理` ではなく、保守・改修・引き継ぎの前に必要な判断材料を揃える商品
- `直す前に、何が危ないか / どこから触るべきか / 何を確認すべきか` を高い粒度で返す

外向けで売るもの:
- AI / 外注 / 引き継ぎコードの保守前整理
- 危険箇所の優先順位
- 関連ファイルの整理
- 次の着手順
- 引き継ぎメモ
- 追加で、保守前に見落としやすい注意点

返す成果物イメージ:
- `00_結論と要点.md`
- `01_[対象フロー名]_引き継ぎメモ.md`
- 必要なら `02_危険箇所と優先順位.md`
- 必要なら `03_次に直すべき候補.md`

価値の中心:
- 次の担当者が迷わず触れる
- どこが危ないかを先に把握できる
- 全体を読む前に、直近で重要な範囲だけ掴める
- 改修前に事故る確率を下げられる

今の資産が効くところ:
- `handoff` の納品構造をそのまま伸ばせる
- `確認できたこと / 推定 / 未確認` の分け方がそのまま使える
- 返信システムで鍛えた `何を返すと安心になるか` の感覚が効く
- scope 判定や範囲線引きの文面も流用しやすい

高単価にしやすい理由:
- 実装はしなくても、判断材料が増えることで意思決定コストを大きく下げられる
- 特に `触るのが怖い既存コード` は、整理された情報の価値が高い
- ただのメモではなく、保守や改修の前提を整えるドキュメントに寄せると単価理由が立つ

難所:
- `handoff-25000` と何が違うかを明快にする必要がある
- 1フロー整理だけでなく、危険箇所優先順位や着手順まで返すことを前に出す必要がある
- `仕様書化` と誤読されないようにしつつ、浅い整理に見せないバランスが難しい

向いている buyer:
- 前任者が抜けて止まっている
- 外注コードのまま保守に入りたい
- AIで作ったコードが動いてはいるが怖い
- いきなり大改修より、まず危険箇所を見たい

現実的な価格帯イメージ:
- 初回トライ: `39,800〜59,800円`
- 標準: `69,800〜99,800円`
- 実績後の上位版: `120,000円+`

今後の詰めどころ:
- `handoff-25000` との差分を定義する
- 返却物を2本にするか3本にするかを決める
- `危険箇所トップ3〜5` を商品価値の中心に置くか判断する

## 2. 設計監査 / セカンドオピニオン

位置づけ:
- 実装代行ではなく、既存方針・既存コード・既存フローを読んで、妥当性とリスクを返す商品
- `これで進めていいのか` を判断したい人向け
- 実装前レビュー / 方針レビュー / 危険箇所レビューに寄せる

外向けで売るもの:
- 実装方針の監査
- 既存コードのリスク整理
- 危険箇所・見落としやすい分岐の指摘
- 代替案の提示
- 次の一手の優先順位

返す成果物イメージ:
- `00_結論と判断材料.md`
- `01_リスクと懸念点.md`
- `02_推奨方針と次アクション.md`

価値の中心:
- その実装方針で進めてよいかが分かる
- 自分たちでは見えていないリスクを先に潰せる
- 「何が危ないか」「今どこを直すべきか」が明確になる
- 外部の実装会社やAI生成コードの妥当性チェックにも使える

今の資産が効くところ:
- `handoff` の読む力をそのまま使える
- `bugfix` で鍛えた原因確認・分岐確認の感覚が効く
- 返信システムで育てた `相手にどう返すと納得感が出るか` がレポート品質に効く
- `#AR` と外部調査を前処理として足せば、根拠の厚い提案にしやすい

高単価にしやすい理由:
- 実装するより前に、やり直しコストや事故コストを下げる
- 1つの判断で後続コストが変わる案件ほど価値が高い
- `セカンドオピニオン` は、フル実装より軽く見えて、実は責任のある判断仕事

難所:
- 何を監査して何をしないかを明確にしないと、`何でも相談` に見えやすい
- 実装しない分、返却物の質で価値を感じさせる必要がある
- `コンサル` と誤読されると重くなるため、成果物主導で見せた方が良い

向いている buyer:
- 今の方針で進めてよいか不安
- 外注会社や別担当の提案を第三者目線で見てほしい
- AIが出したコードや設計案が妥当か見てほしい
- 既存システムに手を入れる前に、リスクを知りたい

現実的な価格帯イメージ:
- 初回トライ: `29,800〜49,800円`
- 標準: `59,800〜89,800円`
- 上位監査パック: `120,000円+`

今後の詰めどころ:
- 監査対象を `決済まわり / API連携 / 会員状態更新` に寄せるか、もう少し広げるか
- レポート形式を `厳しめ` にするか `実務寄り` にするか
- `コードを読む監査` と `方針を読む監査` を分けるか

## 3. AIっぽくない高品質 UI 改善実装

位置づけ:
- AI生成感の強い画面や、安っぽく見える画面を、実務で使えるUIへ整えて実装する商品
- 売るのは `AI排除` ではなく、`安っぽく見えない / 使いやすい / 情報が整理された UI` への改善と実装
- レビュー単体は入口にできても、本体は `改善案 + 実装` に置く

外向けで売るもの:
- 既存画面のUI改善
- 情報設計の整理と再設計
- 余白・階層・視線誘導・コンポーネント整理
- AIっぽく見える箇所の洗い出しと修正
- 実際に使う画面への改善実装

返す成果物イメージ:
- `00_結論と変更方針.md`
- `01_画面ごとの改善要点.md`
- 実装済みコードまたは差分
- 必要なら annotated screenshot / mock / Figma的補助資料

価値の中心:
- AI丸出しの雑な見た目から脱却できる
- どこが安っぽく見えるかが言語化される
- デザインの方向性が揃う
- 実際に触った時の見やすさ・整い方まで改善される
- レビューだけで終わらず、仕上がった画面として返る

今の資産が効くところ:
- 返信システムで育てた `質の低い出力をどう改善するか` の考え方がそのまま使える
- 今後育てる `UI禁止パターン / 参照知識 / skill` が大きな差別化になる
- `#AR` や外部観察で、一流プロダクトや有名デザイナーの共通項を抽出し、内部ルール化できる
- 画面改善を文章で返す構造は、今の納品文書文化と相性が良い
- その上で、改善内容を実装まで持っていける

高単価にしやすい理由:
- 見た目は成果が分かりやすい
- しかも `高品質にする内部skill` は真似されにくい
- ただのレビューより、実装を含む方が満足度と差別化が上がる
- 管理画面 / SaaS / 業務ツールなどは、見た目と使いやすさに直接価値がある

難所:
- 今のままだと内部skillがまだ未成熟
- 参照元・禁止パターン・実務での再現ルールを整理する必要がある
- 外向けには `人が作った感じ` ではなく、`実務レベルの見やすいUI` として見せる方が伝わりやすい
- 実装まで含めると、受ける画面数と範囲をきちんと切らないと重くなりやすい

向いている buyer:
- AIで画面を作ったが安っぽい
- 管理画面が見づらい
- 情報はあるが整理されていない
- LPや業務画面を、もっと実務で使える見た目にしたい

現実的な価格帯イメージ:
- 小さめの改善実装: `59,800〜89,800円`
- 主要画面改善パック: `120,000〜180,000円`
- 管理画面 / SaaS まとめ改善: `200,000円+`

今後の詰めどころ:
- 参照対象を何にするか（一流SaaS、著名デザイナー、UI監査記事など）
- `AIっぽさ` を禁止パターンとしてどう定義するか
- レビュー入口商品を置くか、本体を最初から改善実装にするか
- 1画面単位か、画面群単位か、管理画面単位かで切るか

## どれが一番現実的か

### すぐ始めやすい
- `1. コード特化の整理メモ上位版`

理由:
- いまの `handoff` 資産を最も流用できる
- 返却物の型がすでにある
- buyer の困りごとも想像しやすい

### 収益化と差別化のバランスが良い
- `2. 設計監査 / セカンドオピニオン`

理由:
- 実装まで抱え込まずに価値を出せる
- 判断の質で勝てる
- 既存コード / AIコード / 外注コードとの相性が良い

### 一番おもしろく、伸びしろがある
- `3. AIっぽくない高品質 UI 改善実装`

理由:
- 見た目に成果が出る
- 内部 skill の育成がそのまま差別化になる
- 長期的に一番 `Codex level200` 感が出やすい

- 売るのは内部 skill ではなく、その skill で仕上がった成果物

- 実績が増えるまでは、価格を下げるより `対象範囲を絞る` 方が安全
