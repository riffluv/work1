==================================================

3. ユーザーが最後の5件を送る

- **5件ごとに生成**
- **15件で一括監査**

==================================================

- 各ケースは独立した synthetic test として扱う
- reply memory を通さない
- purchased memory を使わない
- active case を読まない

==================================================
【5件ずつの出力ルール】
==================================================

- 返信文は **5件ずつ** 出す
- そのターンで受け取った5件にだけ答える
- ユーザーが次の5件を送るまで、勝手に 6〜10 / 11〜15 を作らない
- 各ケースは番号付きで返す
- そのまま送れる日本語の完成文を出す
- 必要なら短い補足は付けてよいが、主出力は送信用文面

==================================================
【15件目でやること】
==================================================

15件目まで出し終えたら、そのターンで続けて:

1. 11〜15件の返信文を出す
2. その15件全体を対象にした **Claude監査プロンプト** を出す

この監査プロンプトでは、以下を必ず入れること:
- Claude 側の memory / 長期記憶は **オフ前提** で見ること
- 15件を横断して、傾向と再発パターンを拾うこと
- 各ケース単体の自然さだけでなく、15件全体での癖を監査すること

==================================================
【Claude監査プロンプトで必ず見る観点】
==================================================

最低限入れる観点:
- 日本語として自然か
- AI感がないか
- buyer の主質問に正面から答えているか
- 相手がすでに書いた情報を無視していないか
- 温度感が重すぎないか、軽すぎないか
- generic fallback の匂いが残っていないか
- near-echo が残っていないか
- direct answer の順番が自然か
- buyer の語彙を勝手に弱めていないか
- `件ですね` `大丈夫です` `そのままご購入` の多用がないか
- `はい、` のような bot 的な文頭がないか
- まだコードやログを見ていないのに調査済みに読める表現がないか
- semantic のズレがないか
- 実務でそのまま送れるか
- 15件横断で再発している悪癖がないか
- 5件単位だと見えにくい偏りがないか
